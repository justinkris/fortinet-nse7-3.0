💭 THINK FIRST · TOPOLOGY DESIGN

Before a single phase 1 proposal is chosen, the shape of the network decides how many tunnels you will maintain, where traffic hairpins, and how much every branch upgrade will hurt. Site-to-site, hub-and-spoke, dual-hub, and full mesh each solve a different problem — and each introduces a different tax.

  • Why does adding more branches make a full mesh painful long before it becomes insecure?
  • What does dual hub-and-spoke buy you that a single hub never can?
Q1 — MODEL ANSWER
A full mesh grows as N(N-1)/2 tunnels. Ten branches means forty-five permanent tunnels each firewall must renegotiate and rekey. Branch models often can't hold that many phase 1 SAs at once, so the operational ceiling appears long before any single tunnel gets weak.
Q2 — MODEL ANSWER
A second hub gives you a failover path for the spokes and lets you spread aggregate throughput across two devices. If the first hub is saturated or offline, spokes still reach the other sites — no single bottleneck decides whether the whole enterprise stays online.
SECTION 01

Picking a topology before you pick a proposal

IPsec design starts long before the CLI. The choice of topology decides how many tunnels each firewall carries, how traffic flows between branches, and how gracefully the network absorbs new sites. Fortinet groups the common shapes into four families: site-to-site, hub-and-spoke, dual hub-and-spoke, and full mesh. Each solves a different pressure — reach, scale, resilience, or latency — and each carries a different cost.

Site-to-site is the simplest: two devices, one tunnel, symmetric configuration. It is the right answer when you have two offices and no plans to grow. Hub-and-spoke centralises tunnels on a single firewall so branches only maintain one connection each, but every spoke-to-spoke conversation must transit the hub, and the hub becomes both a bottleneck and a single point of failure. Dual hub-and-spoke splits that risk across two hubs so spokes keep working when one hub is down or over-subscribed. Full mesh gives every site direct connectivity to every other site — great latency, awful scaling — and is where ADVPN eventually takes over.

When you sketch a topology you should also weigh the type of firewall at each site, the volume of traffic likely to cross the tunnel, and how many phase 2 selectors each device can hold. Branch appliances support far fewer concurrent SAs than campus-class firewalls, so a hub sized for headquarters can support dozens of spokes that would each be crushed if you tried to connect them directly.

  • Reliable algorithms — pick proposals that all sites can honour, so key negotiation never fails on the weakest branch.
  • Scalability — plan for future growth; adding a spoke should not force you to re-key every existing peer.
  • Robust authentication — pre-shared keys are convenient at two sites, but certificates scale better once you cross ten.
Design heuristic: a hub-and-spoke topology with ADVPN is the closest thing to a full mesh you can operate at branch scale. Design your tunnels statically; let ADVPN build shortcuts dynamically.
🧠
MENTAL NOTE
Every extra tunnel costs memory, CPU, and rekey time on every peer — pick the smallest topology that carries the traffic you actually have.

💭 THINK FIRST · IKEv2 & KEY EXCHANGE

IKEv2 was not a cosmetic upgrade. It rewrites how peers authenticate, how they trade keys, and how they recover from packet loss. If you inherited an IKEv1 deployment, the question is not whether IKEv2 is better; it's whether every peer can handle it and whether the migration is worth the maintenance window.

  • Why do modern deployments prefer AES-GCM and ECP groups over 3DES-CBC and MODP 1024?
  • What does perfect forward secrecy in phase 2 actually protect you from?
Q1 — MODEL ANSWER
AES-GCM combines encryption and authentication in one pass — fewer instructions, less latency, and integrity built in. ECP groups reach the same security level as MODP with shorter keys, so the phase 1 exchange completes faster and eats less CPU on branch hardware.
Q2 — MODEL ANSWER
PFS forces phase 2 to derive a brand-new Diffie-Hellman secret for every rekey. If an attacker later compromises the phase 1 key, they still cannot decrypt any historic phase 2 traffic — each session is cryptographically independent.
SECTION 02

The security features that make IKEv2 the default

IKE — internet key exchange — is the protocol that negotiates the shared secrets IPsec later uses to encrypt traffic. IKEv1 supports older algorithms like DES, 3DES, and AES-CBC and pairs them with the smaller Diffie-Hellman groups (768–1536 bit MODP). IKEv2 supports AES-GCM for authenticated encryption, ChaCha20-Poly1305 for platforms without AES acceleration, larger MODP groups, and elliptic-curve (ECP) groups that reach the same security level with dramatically shorter keys.

Beyond raw crypto, IKEv2 offers real functional wins: EAP integration for external authentication, HMAC-SHA2 / AES-XCBC integrity checks, and a built-in mode config exchange that carries client parameters inside the IKE conversation itself. Every example in this chapter uses IKEv2 because those advantages compound in ADVPN, mesh, and SD-WAN deployments.

Diffie-Hellman groups — the key-exchange primitive underneath IKE — come in two flavours. MODP uses large prime numbers and exponents; its security rests on the difficulty of the discrete logarithm problem. ECP uses points on an elliptic curve; it reaches equivalent security with much smaller keys. Enabling perfect forward secrecy in phase 2 tells FortiGate to rerun a fresh Diffie-Hellman exchange for every new IPsec SA, so compromising one key never compromises the others.

💭 THINK FIRST · TUNNEL BEHAVIOUR

A working tunnel is not a static object. Peers vanish, links flap, subnets overlap, and traffic patterns swing across the day. FortiOS gives you handles for each of those problems — DPD modes, multiple phase 2 selectors, aggregate interfaces, route-overlap, and forward error correction — and picking the right one depends entirely on what your network actually looks like.

  • When is on-demand DPD the right choice, and when is on-idle better?
  • Why would you enable multiple phase 2 selectors on a single phase 1?
Q1 — MODEL ANSWER
On-demand fires DPD only when there is unanswered traffic — ideal for unpredictable branches where you need fast reaction without constant background chatter. On-idle sends periodic probes even when the tunnel is quiet, giving you steady visibility on regular-traffic sites like data centres.
Q2 — MODEL ANSWER
Multiple phase 2 selectors let one authenticated phase 1 channel carry several distinct network segments, each with its own local/remote subnet pair. This lets you segment traffic on the tunnel — declaring exactly which subnets are allowed through — without opening a wildcard 0.0.0.0/0 pair that would swallow every route.
SECTION 03

Keeping tunnels healthy, redundant, and honest

Dead peer detection (DPD) is how FortiGate discovers that the far side has quietly disappeared. It runs in one of three modes. On-demand probes only after traffic has gone unanswered — ideal when traffic patterns are unpredictable and you want minimal background overhead. On-idle probes at a steady interval regardless of traffic, which suits data-centre tunnels with regular flows. Disable is defensible only in tightly controlled stable environments where the overhead of any probe is unwelcome and the tunnel is monitored externally.

A single phase 1 SA can carry multiple phase 2 selectors. Each selector declares one pair of local and remote subnets that are allowed to encrypt over the tunnel. Explicit selectors are safer than wildcard 0.0.0.0/0 pairs — they document exactly which segments belong on the VPN and avoid nasty surprises when peers advertise overlapping routes.

For redundancy, FortiOS supports IPsec tunnel aggregation. You bind two or more tunnels to a virtual aggregate interface — like LACP for VPNs — and pick a load-balancing algorithm: round-robin, L3, L4, redundant, or weighted-round-robin. When one member drops, the others carry the traffic. When two remote sites present overlapping subnets, the phase 2 route-overlap setting decides who wins: use-new disconnects the old dial-up tunnel, use-old keeps the existing peer, and allow lets both run in parallel with ECMP load balancing. On SD-WAN hubs with multiple egress interfaces, forgetting allow often causes tunnels to flap between active and dead states — a signature failure worth memorising.

Finally, forward error correction (FEC) — a phase 1 option — adds redundant packets so the receiver can rebuild lost or corrupted ones. FEC costs bandwidth but recovers voice and video quality over lossy WAN links, which is the exact reason branch VPNs enable it for VoIP.

FortiOS CLI · overlapping routes on dial-up phase 2
config vpn ipsec phase1-interface
  edit Phase1
    set type dynamic
end

config vpn ipsec phase2-interface
  edit Phase2
    set phase1name Phase1
    set route-overlap 
end

MENTAL NOTE
DPD keeps the tunnel honest, aggregation keeps it redundant, and explicit phase 2 selectors keep it well-behaved when your routes collide.

💭 THINK FIRST · MTU & FRAGMENTATION

Every IPsec tunnel carries an encapsulation tax — extra headers stacked on the original packet. If nobody accounts for that tax, the router quietly fragments; if fragmentation is forbidden, it silently drops. Understanding MTU, PMTUD, and TCP MSS is the difference between a fast tunnel and a mysteriously broken application.

  • Why does FortiOS honour the DF bit by default, and what happens when it does?
  • How does lowering TCP MSS at the firewall avoid IP-level fragmentation?
Q1 — MODEL ANSWER
Fragmentation is expensive and unreliable — reassembly happens only at the destination and any missing fragment loses the whole packet. Honouring the DF bit forces the network to drop oversized packets, which pushes the sender to run PMTUD and adjust its own packet size instead of relying on midpath fragmentation.
Q2 — MODEL ANSWER
TCP MSS is negotiated during the three-way handshake. By clamping tcp-mss-sender and tcp-mss-receiver on the firewall policy, FortiGate rewrites the MSS options so the endpoints agree on a segment size small enough to fit through the tunnel — no IP-layer fragmentation needed.
SECTION 04

MTU, PMTUD, and the tax IPsec adds to every packet

IP fragmentation happens when a packet is larger than the outgoing interface's MTU. The router splits it, the destination reassembles it, and any lost fragment kills the whole datagram. Typical Ethernet MTU is 1500 bytes; if the far side offers only 1400, the difference either gets fragmented or dropped depending on the DF (don't-fragment) bit. FortiOS defaults to honor-df enable, meaning the firewall will not fragment packets larger than the interface MTU — it drops them and lets the sender retry with a smaller size.

Path MTU discovery (PMTUD) is how a host learns the smallest MTU on the whole path. The sender sets the DF bit on outgoing packets; any hop with a smaller MTU replies with an ICMP fragmentation needed message, and the sender adjusts. You can trigger PMTUD manually with ping -f -l 1472 fortinet.com on Windows or execute ping-options df-bit yes / data-size 1472 on FortiOS. The 1472 bytes of payload plus 28 bytes of header exactly fills a 1500-byte MTU.

Encapsulation adds noticeable overhead. VXLAN adds 50 bytes; GRE adds 24; MPLS adds 4 per label; PPPoE adds 8; VLAN Q-in-Q adds 8; 6in4 and 4in6 add 20 and 40 bytes respectively. When any of these ride inside your IPsec tunnel, you need to react — either raise the interface MTU, adjust the firewall policy's TCP MSS, or set the fragmentation MTU in phase 1.

  • Physical interface MTU — configurable from 68 to 65 535 bytes, default 1500.
  • IPsec VPN interface MTU — configurable from 68 to 1700 bytes, default 1420.
  • TCP MSS — payload only, no TCP header. Set via tcp-mss-sender / tcp-mss-receiver on the policy.
Diagnostic reflex: when an application works over the LAN but hangs over VPN, the first suspect is MTU. Ping with the DF bit set at both 1472 and 1500 bytes — if 1500 fails, you've found the culprit.
🧠
MENTAL NOTE
IPsec headers eat into every packet — plan for the tax, or the tax will show up as mysterious application timeouts.

💭 THINK FIRST · FORTIMANAGER TEMPLATES

Building one tunnel by hand is tolerable. Building forty is a full-time job — and each one is a fresh chance to typo a proposal or a phase 2 subnet. FortiManager IPsec templates and VPN Manager both attack that problem, and choosing between them isn't about which is more powerful. It's about what kind of deployment you're actually running.

  • What jobs does VPN Manager do that IPsec templates can't?
  • Why are IPsec templates the preferred choice for zero-touch provisioning?
Q1 — MODEL ANSWER
VPN Manager can configure SSL VPN, register non-managed gateways (including third-party devices), and drive an assisted setup wizard. It's the right tool for large or heterogeneous deployments where the VPN spans devices FortiManager doesn't directly manage.
Q2 — MODEL ANSWER
IPsec templates support metadata variables and configuration import/reuse, so the same template drives many branches with per-device values injected at install time. That is exactly what zero-touch provisioning needs — a single reusable definition that adapts to each new box as it registers.
SECTION 05

Scaling IPsec with FortiManager templates

FortiManager offers two paths for centralised IPsec: IPsec templates (found in Device Manager > Provisioning Templates) and VPN Manager (a top-level module). Templates are lean and reusable; VPN Manager is a broader central-management interface for VPN communities. They overlap heavily, but the details of which one to pick depend on scale, provisioning model, and whether SSL VPN is in scope.

Before installing an IPsec template for the first time, sketch the topology: which interfaces participate, which remote networks must be reachable, which metadata variables to use (remote gateway IP, outgoing interface, local ID, remote subnet, tunnel interface setup), which IKE version, which DH groups for phase 1 and phase 2, and which proposals you plan to negotiate. The workflow itself is short: create and assign IPsec templates for the hub and spokes → install device settings → build a zone with normalised interfaces → create the policy package and install it.

When you assign a template, both the device database and the provisioning template status flip to modified. That is expected — assignment is not installation. Have your device and policy layers synchronised beforehand or you'll end up chasing pending configurations. On the hub, group all the spoke tunnels under a single zone interface so one firewall policy can govern the whole spoke traffic; on the spokes, one normalised interface per tunnel is usually enough.
