Chapter 10 · NSE7 EF 7.6

Silicon That Does the Firewall's Heavy Lifting — Hardware Acceleration on FortiGate

How Fortinet's NP, CP, and SP processors let a FortiGate move traffic at line rate without asking the CPU for permission on every packet.

The CPU is a generalist. The SPU is a specialist. Enterprise-grade throughput is a story about knowing which packets need which brain.

Master this chapter and you can predict — for any given policy, profile, or tunnel — whether the traffic will fly on the fast path or crawl through the slow one.

💭 THINK FIRST · SPU ARCHITECTURE

A FortiGate has a general-purpose CPU, but Fortinet also solders on specialized chips called SPUs. If the CPU can already run FortiOS and forward packets, why bother with dedicated silicon at all — and why split it into NP and CP flavors instead of one bigger chip?

  • What is the practical difference between what an NP7 accelerates and what a CP9 accelerates, and how does that split map to real workloads?
  • Why do entry-level and mid-range FortiGates ship an SP (SoC4/SP5) instead of separate NP and CP chips?
Q1 — MODEL ANSWER
The NP works at the interface level and accelerates network functions: session fast-pathing, IP integrity header checking, and IPsec VPN encrypt/decrypt. The CP is application-focused and accelerates security-inspection work — flow-based IPS pattern matching, AV, application ID, and SSL/TLS deep-inspection crypto. NP moves packets; CP inspects them.
Q2 — MODEL ANSWER
Cost, power, and PCB real estate. Entry-level and mid-range boxes don't need separate high-throughput NP and CP dies, so Fortinet fuses "light" versions of both onto a single system-on-chip (SoC4, SP5). You get most of the acceleration benefits at lower cost and lower throughput ceilings — appropriate for branch and small-office roles.
SECTION 01

NP, CP, and SP — the three flavors of Fortinet silicon

Fortinet hardware acceleration refers to the use of specialized chips — collectively called Security Processing Units (SPUs), sometimes ASICs — that offload resource-intensive tasks from the main CPU. Instead of the general-purpose CPU handling every packet's crypto, pattern match, and header check in software, dedicated silicon executes those operations in hardware at wire speed. The net effect is higher throughput, lower latency, and headroom for security features that would otherwise crush a software-only firewall.

An SPU can be a network processor (NP), a content processor (CP), or both — depending on the FortiGate model. On larger boxes NP and CP are separate chips working in parallel with the CPU. On smaller models the SPU is a system-on-chip that packages a light NP and a light CP together.

SPUWhat it acceleratesLatest versionExample FortiGate
NPInterface-level offload of CPU-bound traffic; IPsec VPN encrypt/decrypt; IP integrity header checksNP7FortiGate 900G
CPCommon security workloads: application ID, IPS, flow-based AV, SSL/TLS deep-inspection cryptoCP10FortiGate 200G
SP (SoC)Combined light NP + CP on one die for entry/mid-range platformsSP5FortiGate 90G

The NP7 can hold up to 12 million concurrent sessions, limited by on-chip memory. Once an NP7 hits its session limit, additional sessions spill back to the CPU — which is why, on multi-NP platforms, session distribution across NPs matters. You need to know which physical interfaces map to which NP, and design traffic paths so no single processor is overloaded.

To identify what silicon a given FortiGate ships with, the CLI gives you two commands. diagnose hardware lspci | grep 1a29 lists Fortinet PCI devices (vendor ID 1a29) — an NPID of 4e36 means NP6, 4e37 means NP7. For CP and SP identification, get hardware status exposes an ASIC version: line — CP9, CP10, SoC4, or SP5.

FortiOS CLI · SPU identification
# diagnose hardware lspci | grep 1a29
9b:00.0 Class 1000: Device 1a29:4e37 NPID     # → NP7

# get hardware status
Model name: FortiGate-900G
ASIC version: CP9
...

On larger platforms, Fortinet also offers NP Direct architecture, available on FortiGates with two or more NP6 processors or on models where ports X5–X8 are directly wired to an NP7. NP Direct eliminates the internal switch fabric (ISF) between the front ports and the NP, cutting forwarding latency to the absolute minimum. The catch: traffic must enter and exit through interfaces attached to the same NP for the low-latency path to work.

MENTAL NOTE
NP moves packets; CP inspects them. Big FortiGates split them into separate chips; small FortiGates fuse a lite version of each onto one SoC.

💭 THINK FIRST · NTURBO & IPSA

Ordinary NP offload works beautifully — until a firewall policy attaches a flow-based UTM profile. The NP can't inspect payloads, so those sessions would drop back to the CPU. That would be a performance cliff. Fortinet's answer is two features working together: NTurbo and IPSA.

  • Why can't a straight NP offload accelerate a session that has a flow-based IPS or AV profile attached?
  • What does NTurbo actually do differently, and how does IPSA extend that acceleration into the CP?
Q1 — MODEL ANSWER
The NP fast-paths a session by installing the session key and forwarding future packets without CPU involvement. But if a policy attaches a security profile, every packet must reach the IPS engine for inspection — and the IPS engine runs on the CPU. So plain NP offload has to punt.
Q2 — MODEL ANSWER
NTurbo creates a dedicated data path from the NP to a driver that hands packets directly to the IPS engine and returns them to the NP after inspection — avoiding the kernel's slow path. IPSA (IPS acceleration) then offloads the actual pattern-matching work from the IPS engine to the CP, so the flow inspection itself runs partly in hardware.
SECTION 02

NTurbo and IPSA — accelerating traffic the NP can't offload alone

Standard NP offload works for sessions that don't need Layer-7 inspection. The moment a firewall policy carries a flow-based security profile — IPS, application control, flow-based AV, web filter — the NP can no longer fast-path those packets on its own, because the payload has to reach the IPS engine running on the CPU. Without help, throughput on such policies would collapse to CPU-bound rates.

NTurbo is Fortinet's workaround. It offloads traffic that the NP alone cannot: it creates a special data path that redirects packets from the ingress interface to the IPS engine. For a new session, the NTurbo driver sends packets to the IPS engine, the engine performs its security action, and then hands the packet back to NTurbo, which forwards it back to the NP. Anything the IPS engine can handle — IPS, flow-based UTM, NGFW policies — can ride the NTurbo path. Proxy-based inspection stays on the CPU.

IPSA (IPS acceleration) is the CP-side partner. When enabled, flow-based pattern-matching databases are compiled and pushed to the content processors. The IPS engine then offloads the pattern-matching work to CP hardware, dramatically reducing CPU load and lifting inspection throughput. Together, NTurbo and IPSA let flow-based UTM policies run at rates close to bare NP offload.

FortiOS CLI · NTurbo / IPSA acceleration modes
config ips global
    set np-accel-mode {none | basic}
    set cp-accel-mode {none | basic | advanced}
end

# Per-policy override — disable NTurbo only for troubleshooting
config firewall policy
    edit <policy_id>
        set np-acceleration {disable | enable}
    next
end
  • np-accel-mode basic — default, enables NTurbo distribution across IPS engine processes.
  • cp-accel-mode basic — default on devices with a single CP8; offloads basic pattern matching.
  • cp-accel-mode advanced — default on newer platforms; offloads more pattern-matching types for larger throughput gains.
  • set np-acceleration disable — per-policy kill switch, used only for troubleshooting; leave enabled in production.
Common misconception: NTurbo does not accelerate proxy-based inspection. Only flow-based UTM/NGFW and IPS ride the NTurbo path. If your policy mixes flow-based and proxy-based profiles, the whole session drops to the CPU.
🧠
MENTAL NOTE
NTurbo builds a shortcut between the NP and the IPS engine; IPSA drops pattern matching onto the CP. Together they turn "flow-based UTM" from a CPU tax into an accelerated workload.

💭 THINK FIRST · LIFE OF A PACKET

If offload were unconditional, every packet would take the fast path. Instead, the CPU always inspects the start of a session — TCP three-way handshake or first UDP packet — before deciding what runs on the CPU, the NP, the CP, or a combination.

  • Why does FortiGate always send the first packets of a session through the CPU, even on sessions that will eventually offload to the NP?
  • What is the difference between the "slow path" and the "fast path," and which scenarios stay entirely on the slow path?
Q1 — MODEL ANSWER
The CPU has to establish session state — do policy lookup, apply NAT rules, decide which security profiles apply, and negotiate whether the session even qualifies for offload. Only after that decision is made can the CPU install a session key in the NP so subsequent packets bypass it. FIN and RST also stay on the CPU so it can tear session state down cleanly.
Q2 — MODEL ANSWER
Slow path = through the CPU (full processing, routing lookup, session helpers). Fast path = through the NP after the session key is installed, bypassing the CPU entirely. Sessions with proxy-based UTM stay on the slow path for their entire life; sessions with session helpers, tunnels (except IPsec), or policies that explicitly disable ASIC offload also can't fast-path.
SECTION 03

Life of a Packet — slow path, fast path, and what runs where

Every session on a FortiGate follows the same opening act. For TCP, the CPU processes the three-way handshake. For UDP, the CPU processes the first packet. During those first packets the CPU runs the full pipeline — quarantine, FortiTelemetry, captive-portal auth, destination NAT, routing/RPF, stateful policy lookup, session helpers, SSL VPN, source NAT, IPsec VPN encryption, traffic shaping, WAN optimization — and decides what the session's future will look like. FIN and RST packets also always ride the slow path, so the CPU can retire session state cleanly.

Once the session is established, one of three trajectories takes over:

  • Offloaded non-UTM sessions — the session key is installed in the NP, and subsequent packets are handled entirely by the NP: IP integrity, IPsec crypto, traffic shaping, egress interface. The CPU never sees them. This is the fastest possible fast path.
  • NTurbo / flow-based UTM sessions — subsequent packets skip routing/kernel but still visit the IPS engine on the CPU, with IPSA offloading pattern matching to the CP. Much faster than pure CPU, still not as fast as pure NP.
  • Proxy-based UTM sessions — the NP does not offload. Every packet passes through the FortiGate proxy for antispam, DLP, antivirus, web filtering, and VoIP inspection. The IPS engine still handles single-pass IPS/app-control/botnet/SSL inspection alongside. SSL deep inspection leverages the CP for TLS crypto but the flow is otherwise CPU-bound.
Exam tip: When SSL deep inspection is configured, the flow is modified — the IPS engine sends the packet to the proxy, the proxy offloads TLS decrypt to the CP, decrypted content goes back to the IPS engine for inspection, back to the proxy for proxy-based inspection, then re-encrypted by the CP. Five hops, two CP round-trips per packet.

Virtual FortiGates (FortiGate-VM) follow the same pipeline logic, except there is no NP or CP hardware — every step runs on the vCPU. All the software-level optimizations still apply, but you should not expect ASIC-level throughput on a VM.

🧠
MENTAL NOTE
The first packets are always the CPU's — that's how the FortiGate decides who else gets to touch the session. After that, offload is a design decision the policy and profile mix already made for you.

💭 THINK FIRST · CONFIG BY OSI LAYER

Hardware acceleration is not a single switch you flip. Different features hook into different layers of the packet's journey — physical, data link, IP, and application — and each layer has its own set of knobs.

  • What does a "VDOM link" over an NP actually offload, and why do VLANs over the NP VDOM link matter?
  • How does FortiGate offload IPsec, and what does the npu_flag field on an IPsec SA tell you at a glance?
Q1 — MODEL ANSWER
A VDOM link (npu_vlink0/npu_vlink1) is a pair of interfaces on an NP that lets inter-VDOM traffic ride the NP fast path instead of looping through the CPU. You assign one endpoint to each VDOM. VLANs layered on NPU VLINKs let you create additional accelerated inter-VDOM links beyond the fixed pair the NP provides — same VLAN ID on both NPU VLINKs, different VDOMs.
Q2 — MODEL ANSWER
FortiGate can offload the encrypt/decrypt operation itself to the NP for supported ciphers/HMACs. The npu_flag field on an IPsec SA tells you the offload state: 00 = both SAs still in the kernel; 01 = outbound copied to NPU; 02 = inbound copied; 03 = both copied (fully offloaded); 20 = unsupported cipher, cannot offload.
SECTION 04

Configuring Acceleration at Every OSI Layer — from VDOM links to IPS

Acceleration decisions attach to different features at different layers. From lowest to highest: inter-VDOM links at the physical/interface layer, DVLAN and VXLAN at the data-link layer, IPsec at the IP layer, and traffic handled by IPS engines at the application layer.

Inter-VDOM link acceleration. Every NP has a fixed pair of VDOM link interfaces named npu_vlink0 and npu_vlink1, both initially assigned to the root VDOM. To offload traffic between two VDOMs, assign one VLINK to each VDOM and route between them. Traffic then rides the NP instead of the CPU.

To scale beyond the built-in pair, layer VLAN interfaces over the NPU VLINKs. Create a VLAN interface with the same VLAN ID on both npu_vlink0 and npu_vlink1, associate each with a different VDOM, and you've built an additional accelerated inter-VDOM path.

FortiOS CLI · VLAN over NPU VDOM link
config system interface
    edit "VLANvlink1Sales"
        set vdom "Sales"
        set ip 10.10.10.10 255.255.255.252
        set interface "npu_vlink1"
        set vlanid 10
    next
end

To verify inter-VDOM acceleration, run diagnose sys session list and check the npu_info line. The offload=9/9 value indicates NP7 is doing the forwarding. Per-policy set auto-asic-offload disable forces a session to the CPU — useful for packet capture on VLINK interfaces during troubleshooting, but performance-costly and to be re-enabled ASAP.

DVLAN and VXLAN. NP7 and SP5 (also called NP7lite) can offload 802.1ad and 802.1Q double-VLAN (Q-in-Q), as well as VXLAN. Default DVLAN mode is 802.1AD (S-Tag 0x88a8 + C-Tag 0x8100). Change it per NP ID or globally with diagnose npu np7 dvlan-mode <802.1Q|802.1AD|Disable> <dev-id>.

IPsec encryption offload. FortiGate offloads IPsec encrypt/decrypt to the NP for supported algorithms. It is on by default; you can disable it per tunnel with set npu-offload disable under config vpn ipsec phase1-interface. The npu_flag field in diagnose vpn tunnel list shows offload state:

npu_flagMeaning
00Both IPsec SAs loaded to the kernel; no NPU offload (or no traffic yet)
01Outbound SA copied to NPU
02Inbound SA copied to NPU
03Both inbound and outbound SAs copied to NPU (fully offloaded)
20Unsupported cipher or HMAC — SA cannot be offloaded

The IPsec Diffie-Hellman key exchange for ESP traffic is also accelerated by default via the CP. Disable globally with config system global / set ipsec-asic-offload disable — again, troubleshooting only.

Strict header checking. By default the NP verifies L4 header length, IP header length, IP version, IP checksum, and IP options. Setting check-protocol-header strict under config system global adds ESP sequence number, SPI, and data-length checks — and disables hardware acceleration on all NPs, CPs, and SPs. Use it only for short troubleshooting windows.

🧠
MENTAL NOTE
Every layer of the stack has its own acceleration switch. If a session isn't offloading the way you expect, walk the layers: VDOM link, VLAN, IPsec SA, IPS mode, per-policy override.

💭 THINK FIRST · BEST PRACTICES

Every architecture decision — inspection mode, SSL policy, tunnel design — either enables acceleration or defeats it. The best-practice list is really a list of decisions that either preserve the fast path or knock traffic off it.

  • Why is a firewall policy with a mix of flow-based and proxy-based security profiles a performance anti-pattern?
  • List at least three situations where an NP-capable FortiGate still won't offload a session, even though acceleration is nominally enabled.
Q1 — MODEL ANSWER
Mixed-mode policies never offload to the NP — they are always fully processed by the CPU. Even the flow-based portion loses its acceleration path, so you pay the proxy-based penalty on every packet. If you need proxy features, put them on a separate policy from the flow-only traffic.
Q2 — MODEL ANSWER
(1) auto-asic-offload disabled on the policy; (2) proxy-based security profiles attached; (3) session requires a FortiOS session helper; (4) tunneled traffic other than IPsec VPN (SSL VPN, GRE, etc.); (5) mixed flow-based + proxy-based profiles on the same policy; (6) unsupported IPsec cipher/HMAC combination.
SECTION 05

Hardware Acceleration Best Practices — designing for the fast path

Choosing inspection modes has a direct impact on resources. Flow-based inspection is the default and has hardware-offload paths (NTurbo + IPSA). It identifies and blocks threats in real time, typically uses fewer resources, and is recommended for high-throughput traffic. Proxy-based inspection buffers traffic and examines it as a whole before deciding — heavier, but supports advanced features (category usage quota, web-profile override). Proxy inspection runs on the CPU/proxy path; no NP offload.

Encrypted traffic requiring SSL deep inspection is the second big throughput sink. Every session needs decrypt → inspect → re-encrypt. Best practices to keep the impact manageable:

  1. Limit the number of policies that allow encrypted traffic. Fewer inspected policies = less crypto load.
  2. Gradually deploy SSL inspection using device security policies rather than flipping it on globally at once.
  3. Exempt known-trusted traffic (banking, healthcare, well-known SaaS) from SSL deep inspection.
  4. Use CP hardware for SSL content scanning (CP9/CP10) whenever the platform supports it.

Cases where sessions will not offload even with NP available:

  • NP acceleration is disabled on the firewall policy (set auto-asic-offload disable).
  • The firewall policy includes proxy-based security profiles.
  • The session requires a FortiOS session helper — protocol helpers for FTP, SIP, PPTP, and similar aren't NP-friendly.
  • Tunneling is enabled — traffic to or from SSL VPN, GRE, and similar tunnel interfaces. IPsec VPN is the exception; it can be offloaded.
  • Mixed flow-based + proxy-based profiles on the same policy — the whole policy becomes CPU-bound.
Remember: a firewall policy with a mix of flow-based and proxy-based security profiles never offloads to the NP and is always processed by the FortiGate CPU. Split the workloads onto separate policies if you need proxy features on some flows but want to keep others on the fast path.
🧠
MENTAL NOTE
Every acceleration decision is really a design choice. Flow-based, split policies, targeted SSL inspection, and CP-offloaded crypto — those four habits keep the fast path fast.
