Letting the Network Route Itself

OSPF as a link-state map, BGP as an autonomous-system negotiator, and the FortiGate role in both.

Static routes describe a network you already understand. Dynamic routing discovers the one that keeps changing.

OSPF for the interior, BGP for the exterior — and a common toolkit of filters, route maps, and convergence knobs sits on top of both.

💭 THINK FIRST · OSPF FUNDAMENTALS

OSPF asks every router to hold an identical map of the network and compute its own best path. That design decision — full-topology awareness instead of hearsay — is why OSPF scales, converges quickly, and stays quiet when nothing is changing.

  • What does "link-state" mean in practice, and why does it converge faster than a distance-vector protocol like RIP?
  • What are the three router roles in OSPF — backbone, ABR, ASBR — and how does each shape what gets advertised?
Q1 — MODEL ANSWER
Link-state means every router within an area holds an identical database of every link in that area (the LSDB). Each router runs Dijkstra's algorithm against that database, computing its own best-path tree with itself at the root. Because every router already has the topology, when a link fails, only the change needs to be flooded — everyone recomputes independently and quickly. RIP, by contrast, relies on neighbors telling it what they know; convergence propagates hop-by-hop with holddown timers.
Q2 — MODEL ANSWER
Backbone routers sit in area 0.0.0.0 — the mandatory core that every other area must connect to. Area border routers (ABRs) have interfaces in the backbone plus at least one other area; they summarize prefixes between areas using config range. Autonomous system boundary routers (ASBRs) inject external, non-OSPF routes (e.g. RIP, BGP, static) into OSPF via redistribution; they can summarize externals with config summary-address.
SECTION 01

OSPF — every router builds its own topological map

OSPF is a link-state routing protocol. Every router in the same area maintains an identical link-state database (LSDB) and runs Dijkstra's algorithm against it to build a shortest-path tree with itself at the root. Because the database is shared, convergence is fast — a topology change only needs to be flooded, not walked hop-by-hop. And because updates only fire when something actually changes (with a 30-minute periodic refresh as a heartbeat), OSPF is remarkably quiet in a stable network.

OSPF is organized into areas. Every OSPF network must have area 0.0.0.0 — the backbone — and every non-backbone area must physically or virtually attach to it. Routers with interfaces exclusively inside area 0 are backbone routers. Routers that straddle area 0 and another area are area border routers (ABRs), and they can summarize prefixes across the boundary with config range. Routers that inject non-OSPF routes into OSPF (RIP, BGP, static, connected) are autonomous system boundary routers (ASBRs), and they can summarize external routes with config summary-address.

Area types matter for what gets propagated. The default type is regular. A stub area excludes external LSAs — the ABR replaces them with a default route. A totally stubby area replaces even inter-area prefixes with a default. A not-so-stubby area (NSSA) is a stub that is allowed to host an ASBR, so external routes can enter through it and reach the backbone as type-7 LSAs (which the ABR converts to type-5).

Network types on FortiGate default to point-to-point, appropriate for directly-cabled routers. In a hub-and-spoke IPsec VPN where net-device is disabled and the hub shares one virtual interface across multiple dial-up spokes, you must set the network type to point-to-multipoint.

FortiOS CLI — OSPF area types
config router ospf
    config area
        edit 0.0.0.1
            set type stub
        next
        edit 0.0.0.3
            set type nssa
        next
    end
end
Filter with the right tool: Access lists filter by prefix. Prefix lists add ge/le logical operators for length matching. Route maps layer conditional logic on top and can also modify OSPF attributes — the most powerful of the three, and the one you apply with distribute-route-map-in.

🧠
MENTAL NOTE
In OSPF, every router keeps the same map and computes its own directions. That is why it converges fast, and why the LSDB — not any single router — is the source of truth.

💭 THINK FIRST · OSPF USE CASES

Three real-world scenarios push OSPF beyond textbook basics: ECMP for external routes, tunneled OSPF over IPsec, and virtual links that stitch a stranded area to the backbone. Each has a specific CLI knob or design constraint.

  • Why do you need to enable rfc1583-compatible to get true ECMP on external OSPF routes?
  • When would you configure a virtual link, and what does it actually do at the LSA level?
Q1 — MODEL ANSWER
By default OSPF operates under RFC 2328's path preference rules, which prefer intra-area paths over the backbone even for external routes — that can defeat ECMP because paths through different areas are not treated as equal-cost. Enabling rfc1583-compatible switches to the older behavior where path selection is based solely on cost, so two equal-cost external paths through different areas both get installed. It is the recommended setting when you specifically want to load balance across externals.
Q2 — MODEL ANSWER
A virtual link stitches a remote area to the backbone through a transit area — you configure it on both the ABR facing the backbone and the ABR of the stranded area (typically an NSSA), specifying each other's router IDs. At the LSA level it behaves as if the far-side area were physically adjacent to area 0, so its routes get propagated backbone-wide without physical cabling changes.
SECTION 02

Three OSPF patterns you will actually configure in production

OSPF ECMP lets FortiGate install multiple equal-cost paths to the same destination in the routing table. By default OSPF applies RFC 2328 path preference rules — intra-area beats inter-area, non-backbone intra-area beats backbone — which can prevent load balancing across externals. To force equal-cost multipath treatment of external routes based purely on cost, enable rfc1583-compatible:

FortiOS CLI — ECMP for external OSPF routes
config router ospf
    set rfc1583-compatible enable
end

# Verify — expect both paths in the routing table:
get router info routing-table all
# O E2  10.2.0.0/24 [110/10] via 10.1.1.100, port4
#                  [110/10] via 10.1.0.1,   port3

OSPF over IPsec protects OSPF adjacencies with encryption. Two common patterns: site-to-site, where you build a route-based IPsec tunnel with phase 1 and phase 2 completed, assign tunnel-end IPs to the tunnel interfaces, and add the tunnel interface to the OSPF network configuration; and dial-up (hub-and-spoke), where the hub uses mode-config to assign IP ranges to spokes and redistributes static plus connected routes into OSPF so spokes can learn corporate prefixes.

Virtual links solve the "orphan area" problem. If you have an NSSA connected to another area which itself connects to the backbone, the NSSA is not directly adjacent to area 0. Configuring a virtual link on the ABRs facing each other — specifying the peer's router ID and the transit area — makes the remote area behave, at the LSA level, as if it were connected to area 0 directly. The transit area cannot itself be a stub.

FortiOS CLI — Virtual link for a stranded NSSA
# On R2 (backbone ABR)
config router ospf
    set router-id 2.2.2.2
    config area
        edit 0.0.0.3
            config virtual-link
                edit VLINK_area
                    set peer 4.4.4.4
                next
            end
        next
    end
end

# On R4 (NSSA ABR)
config router ospf
    set router-id 4.4.4.4
    config area
        edit 0.0.0.3
            config virtual-link
                edit VLINK_area
                    set peer 2.2.2.2
                next
            end
        next
    end
end
Diagnostic set: get router info ospf neighbor for adjacency states, get router info ospf interface for per-interface DR/BDR IDs, get router info ospf status for the whole router summary, get router info ospf database brief for the LSDB ordered by LSA type.
🧠
MENTAL NOTE
ECMP for externals needs RFC 1583. OSPF over IPsec needs the tunnel interface in the OSPF network config. Virtual links resurrect a stranded area without new cabling.

💭 THINK FIRST · BGP FUNDAMENTALS

OSPF is what you run inside a network you fully control. BGP is what you run between networks that must negotiate. That difference — trusted vs. contractual — shapes every BGP design choice.

  • What is the practical difference between EBGP and IBGP, and why does that matter for full-mesh peering?
  • Why is a loopback interface the preferred BGP source in a production peering, and what extra setting must you enable when you use one?
Q1 — MODEL ANSWER
EBGP runs between routers in different autonomous systems — it is the protocol that ISPs use to exchange routes with each other and with their customers. IBGP runs within a single AS, and its default rule that IBGP-learned routes are not re-advertised to other IBGP peers forces you to build a full mesh: every IBGP router must peer with every other IBGP router. That is why route reflectors exist — they concentrate updates and eliminate the mesh.
Q2 — MODEL ANSWER
A loopback interface stays up regardless of physical link state, so a BGP session sourced from a loopback survives failures of any single physical link — as long as an alternate path to the peer exists. Because the loopback is not the physical next hop, you must enable ebgp-enforce-multihop so the TTL is not decremented to zero. You also need a firewall policy allowing TCP/179 between the loopback and the physical interface the BGP packets actually traverse.
SECTION 03

BGP — the protocol behind every peering agreement

BGP is the routing protocol of the internet. It uses autonomous systems (AS) as its unit of trust: routers in the same AS speak IBGP; routers in different AS speak EBGP. IBGP has an important quirk — an IBGP-learned route is not re-advertised to other IBGP peers, which forces you to build a full mesh of peering sessions between every IBGP-speaking router. That is where route reflectors come in (Section 4).

BGP does not advertise anything by default — you must tell it what to inject with either network statements or config redistribute. Redistribution supports connected, static, RIP, OSPF, and ISIS routes. Filter what BGP advertises using access lists, prefix lists (with ge/le), or route maps; route maps additionally allow you to modify BGP attributes and are applied with route-map-in or route-map-out.

Three use cases you should recognize in production:

  • BGP ECMP: When multiple equal-cost paths exist to a BGP next hop, enable ebgp-multipath or ibgp-multipath to let FortiGate install all of them and load balance outgoing traffic. Without this setting, only one path is used.
  • Loopback as BGP source: A loopback interface is always up. Setting update-source Loopback_Interface for a neighbor makes BGP peering survive any single physical link failure — but you must also set ebgp-enforce-multihop because the loopback is not the immediate next hop, and you must add a firewall policy allowing TCP/179 between the physical interface and the loopback.
  • neighbor-group with neighbor-range: Instead of enumerating each spoke neighbor individually, define a neighbor group with common settings (interface, remote-as) and a neighbor range (prefix, group name) that dynamically pulls in any peer whose IP falls within that subnet. Extremely useful in SD-WAN overlay designs with many dial-up spokes.
FortiOS CLI — Loopback-sourced BGP peer
config router bgp
    set as 65100
    set router-id 172.16.1.254
    config neighbor
        edit 100.64.1.254
            set remote-as 65101
            set update-source Loopback_Interface
            set ebgp-enforce-multihop enable
        next
    end
end
Diagnostic set: get router info bgp summary shows session state and packet counters. get router info bgp neighbors drills into a peer. get router info bgp network reveals the BGP database. get router info routing-table bgp shows only BGP-installed routes.
🧠
MENTAL NOTE
EBGP is between autonomous systems. IBGP is inside one, and its no-re-advertise rule is why route reflectors exist. Loopback sources plus enforce-multihop keep peerings alive when physical links fail.

💭 THINK FIRST · BGP CONVERGENCE

BGP is famously slow to converge — that is a feature, not a bug, because BGP is designed to be conservative when accepting new paths from strangers. But inside your own AS, and around HA events, that caution costs you seconds you may not have.

  • Why do route reflectors improve BGP convergence, and what's the trade-off you accept by using them?
  • What does graceful-restart solve, and why does it matter specifically during FGCP failover?
Q1 — MODEL ANSWER
Route reflectors act as concentrators for IBGP routers, replacing an N-squared full mesh with a hub-and-spoke pattern where clients only peer with the RR. Fewer sessions means less CPU, less memory, faster updates — and the RR is the one that redistributes routes to other clients, effectively bending the IBGP "don't re-advertise" rule under controlled circumstances. The trade-off is that the RR becomes a single point of failure for your IBGP topology, so production designs use pairs of RRs.
Q2 — MODEL ANSWER
In an FGCP cluster the BGP daemon only runs on the primary unit. When failover happens, BGP sessions have to be reestablished from scratch — during a reboot or upgrade window, that looks like a route flap to every neighbor, and traffic gets rerouted and then rerouted back. graceful-restart lets the primary advertise "I'm going offline, hold my routes" so neighbors keep the routes in their tables while the new primary comes up and reestablishes peering. No route flap, no traffic interruption.
SECTION 04

BGP convergence — the difference between working and working fast enough

BGP convergence has multiple moving parts: installing a new path in the routing and forwarding tables, processing candidate paths, and detecting failure quickly. Each has a knob you can turn on FortiGate.

  • Filter aggressively to reduce RIB size. Route processing is CPU-bound; the smaller the RIB, the faster convergence.
  • Use route reflectors to eliminate IBGP full-mesh peering. RRs forward IBGP-learned routes to their clients, so you get near-full-mesh reachability without the session explosion.
  • Reduce keepalive timers or enable BFD for sub-second failure detection.
  • Configure graceful-restart to survive HA failover without a route flap.

Route reflectors address the IBGP full-mesh problem. In an IBGP mesh, each router must peer with every other router because IBGP-learned routes are not re-advertised. In an N-router network that means N(N-1)/2 sessions — administratively brutal at scale. An RR bends the rule: it re-advertises IBGP routes to its clients, so clients only need to peer with the RR. On each RR you set route-reflector-client enable on the neighbor definition. Production designs deploy pairs of RRs for redundancy.

BFD (Bidirectional Forwarding Detection) is an out-of-band probe that verifies a peer is alive independent of the BGP keepalive. It detects failures in under a second — faster than any reasonable BGP keepalive would. Configure BFD per neighbor with set bfd enable; for multihop peers (like loopback-sourced BGP), define a BFD multihop template with source, destination, and optional MD5 authentication.

graceful-restart matters specifically because the BGP daemon in FortiGate only runs on the primary FGCP unit. During HA failover the daemon must restart on the new primary and reestablish peering with every neighbor. Without graceful-restart, neighbors treat the disappearance as a link failure, remove the routes, and traffic re-routes elsewhere — then re-routes back when peering is re-established. Enable graceful-restart and capability-graceful-restart on both sides so the primary announces "I'm going offline briefly, please hold my routes." Traffic keeps flowing on cached forwarding tables until the new primary is up.

FortiOS CLI — graceful-restart for HA-friendly BGP
config router bgp
    set as 65100
    set graceful-restart enable
    set router-id 172.16.1.254
    config neighbor
        edit 100.64.1.254
            set capability-graceful-restart enable
            set remote-as 200
        next
    end
end
🧠
MENTAL NOTE
Filters shrink the work. Route reflectors kill the mesh. BFD notices failure fast. graceful-restart hides your HA failover from the rest of the world.

💭 THINK FIRST · FORTIMANAGER & BGP

Configuring BGP on one FortiGate is a CLI exercise. Configuring it on fifty branches that are supposed to be nearly identical but each have their own AS number and router-ID is where FortiManager BGP templates earn their keep.

  • What are metadata variables in a FortiManager BGP template, and what problem do they solve?
  • Why would you use the "Import BGP Template" option instead of building one from scratch?
Q1 — MODEL ANSWER
Metadata variables — written as $(variable_name) in the template — are per-device placeholders that FortiManager substitutes at install time from a mapping table. So one BGP template can carry set as $(AS_Spokes) and router-id 172.16.1.$(BGP_Number_Spokes), and each device gets its own AS number and last IP octet without requiring a separate template.
Q2 — MODEL ANSWER
Import lets you take an already-working BGP configuration from a device or VDOM and turn it into a reusable template. That is far faster than authoring from scratch and it captures every setting you have already validated in production — you just adjust for metadata variables and install to other devices.
SECTION 05

Deploying BGP at scale with FortiManager templates

When your BGP topology spans a hub and many spokes, hand-configuring each device becomes error-prone and un-auditable. FortiManager provides BGP templates under Device Manager > Provisioning Templates > BGP Template. A template captures the shared shape of your BGP configuration — set as, router-id, neighbor definitions, features like ibgp-multipath — and then you assign it to devices or groups.

The critical scaling feature is metadata variables. Any field can hold a placeholder like $(AS_Spokes) or $(BGP_Number_Spokes). FortiManager keeps a mapping table of per-device values for each variable and substitutes them at install time. That means one template covers fifty spokes that each get their own AS number and last-octet router-ID without touching the template itself.

FortiManager ships two Fortinet Recommended templates — HUB_BGP_Recommended and BRANCH_BGP_Recommended. They come with the best-practice settings pre-enabled: ibgp-multipath, graceful-restart, capability-graceful-restart, link-down-failover, soft-reconfiguration, and typical advertisement intervals. Use them as your starting point and adjust rather than authoring from scratch.

The Import workflow lets you pick a device or VDOM whose BGP is already tuned and generate a template from its live configuration. Combined with metadata variables added afterward, this is often the fastest path to a working multi-device deployment.

  1. Create or import a BGP template.
  2. Assign the template to a device or device group.
  3. Run the Install Wizard — FortiManager substitutes metadata variables and pushes the configuration.
  4. Verify with get router info bgp summary on the target device.
Where the CLI meets the GUI: The template's "Preview CLI Configuration" pane shows exactly the FortiOS CLI commands that will be executed on the device. Read it before hitting install — it is the last chance to catch a wrong metadata mapping before it hits production.
🧠
MENTAL NOTE
One template, many devices, per-device values. Metadata variables are what turn "configure BGP on N branches" from a script into a checkbox.
