Session 62 · Two Roads, One Brain
Foundations
What roads exist, and how healthy are they?
  • A member is interface + gateway. Zone, cost and priority are optional.
  • The gateway is what lets FortiGate auto-build the member's static route.
  • Member without a gateway → probes have no path: dead, 100% loss, 0.000 latency. That signature means broken config, not a bad carrier.
"A thermometer in another room" — a probe with no gateway measures nothing real.
  • Purpose 1 — group members so rules can target the group.
  • Purpose 2 — firewall policies reference the zone; membership changes underneath without touching a single policy. Add port3 to the zone, every policy covers it.
  • SD-WAN rules can target a zone, not just members — this matters later for multi-hub.
  • Probes are generated per member, stapled to that member's interface and gateway, and pushed out directly — they never consult the routing table.
  • Protocols: ping (default), tcp-echo, udp-echo, http, https, dns, twamp — probe the protocol your traffic actually uses; carriers queue ICMP differently.
  • Probe a server beyond the ISP gateway, or you only measure the first metre of road.
  • diagnose sys sdwan health-check status — alive/dead, latency, jitter, loss, sla_map.
The health check is the sensor. The SLA target is the judge. The rule is the actuator.
  • link-cost-factor latency jitter packet-loss — every listed factor must pass. One breach = out of SLA. AND logic.
  • failtime × interval = seconds to declare out-of-SLA. recoverytime is deliberately slower.
  • Out-of-SLA does not remove the route. It changes the member's eligibility inside rule selection. The route stays in the table.
  • sla_map is a bitmap: bit per target, 1 = passed. sla(0x0) while alive = the brownout signature.
"Quick to distrust, slow to forgive."
  • For DIA: no backhaul latency, no hub link load, no hub inspection load.
  • Against DIA: one enforcement chokepoint becomes eight; eight must agree forever; drift is invisible.
  • The one that hides: return path and NAT identity. Each DIA site NATs to its own public IP — partner allowlists and "we only accept your Sydney range" agreements break.
Session 63 · Rules of the Road
Rules
Where would a new session go right now?
  • Static routes match destination only. SD-WAN rules match src / users / groups, dst, ISDB, app-ctrl, protocol + ports, ToS, input device.
  • Matching never includes health — a rule still matches even when every member is out of SLA.
"A policy route picks an interface. An SD-WAN rule picks from a group of interfaces — and keeps picking."
  • Rules are read top-down by sequence; first match wins, nothing below is read.
  • A catch-all in seat one turns every later rule into dead config that looks valid.
  • Specific first, general last. Delete the catch-all, don't demote it.
  • Manual — no SLA at all; steers into sick links.
  • Best Quality (priority) — ranks by measured link-cost-factor; no threshold, no concept of "acceptable"; churns paths for 1 ms prizes.
  • Lowest Cost SLA (sla) — SLA as an eligibility gate, then lowest cost, then config order. Stable by default, decisive on breach.
  • Maximize Bandwidth (load-balance) — SLA gate + hash-mode spread.
  • All members out of SLA in mode sla → best-effort fallback in configured order. The rule is not skipped. Skipped only when selection is impossible (all dead / no route).
Sick is not dead.
  • Not really a rule — it is the global set load-balance-mode under config system sdwan.
  • Balances across all members with a valid route to the destination.
  • Consults no SLA — out-of-SLA members still receive implicit traffic, because their routes remain.
  • Self-originated traffic never ingressed, so rules have nothing to match — it bypasses SD-WAN and uses a plain routing lookup.
  • Fix is per feature: set interface-select-method sdwan under syslogd, DNS, FortiGuard, SNMP, etc. No global switch.
  • ISDB = destination lookup (IP/proto/port) → matches on the first packet.
  • app-ctrl identifies via SNI / certs, several packets in → the first session escapes to fall-through, then the mapping is cached so later sessions match on packet one.
  • app-ctrl steering requires application control on the firewall policy — or the rule matches nothing forever while looking valid.
Session 64 · Sessions Have Memory
Sessions & routing
But where is this session actually going?
  • Rules are consulted on the first packet only. Every later packet consults the session table.
  • SLA failover moves new sessions. Existing sessions stay pinned to the browned-out link — Jam's ocean-sounding call.
"The rule answers where a new session would go. The session table answers where this session is going."
  • may_dirty = permission — this session can be re-evaluated. dirty = event — a routing change happened; next packet gets a full re-check.
  • An SLA brownout is not a routing change — routes stay intact, so the dirty machinery never fires. That's why 09:15 stranded the old calls.
  • Manual fix ritual: filter → list → clear → filter clear. List first to see the blast radius; the filter stays armed until you disarm it.
The gym card: swipe = session table; may_dirty = the card can be revoked; dirty = flagged for a full re-check at the desk.
  • Moving a SNAT session changes its egress → new public IP → the far end never negotiated it → silent drop. The session cannot survive the move; only the application can re-establish.
  • set snat-route-change disable (default) — SNAT sessions exempt from the dirty stamp: "let them finish". enable — stamped like everyone else: "rip the bandaid".
  • Fires only on routing changes — irrelevant to brownouts. It would not have saved Jam.
"Escorting dead men to a new door" — moving a SNAT session buys nothing.
  • diagnose firewall proute list — policy routes and SD-WAN rules share one table. IDs 1–65535 = admin-made; above 65535 = SD-WAN or ISDB born.
  • vwl_service / vwl_mbr_seq appear only on SD-WAN-rule routes — the tell.
  • Lowest wins: a hand-made policy route steals traffic upstream of the whole SD-WAN layer — no SLA, no failover, dashboard shows nothing wrong.
"Absence is evidence" — a session with policy_id but no sdwan fields has been stolen.
  • diagnose sys sdwan service — per member: alive (probes returning), sla(0xN) (bitmap of targets passed), selected (current choice — for new sessions only).
  • alive + sla(0x0) = the brownout signature: the road exists, the road is bad.
Seam one · Foundations → Rules
Up ≠ eligible ≠ preferred

A member can be up, in the routing table, and passing every probe — and still be ineligible, simply because it isn't in this rule's member list. Health and eligibility are judged by different courts. Out-of-SLA never removes the route; it only changes standing inside rule selection.

Seam two · Rules → Sessions
The rule swings, the session stays

Failover is a promise made only to new sessions. Existing sessions forward from the session table and never look back at the rule — unless a routing change stamps them dirty, or you clear them yourself. An SLA breach with routes intact stamps nothing.
