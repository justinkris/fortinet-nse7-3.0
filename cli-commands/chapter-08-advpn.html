Chapter 08 · CLI Reference

ADVPN, By Command

Every auto-discovery phase 1 knob, BGP wiring, and diagnose view from Chapter 8 — the commands that turn a hub-and-spoke into a full mesh on demand.

Three settings turn a hub-and-spoke into a mesh: auto-discovery-sender, auto-discovery-receiver, auto-discovery-forwarder.

Config commands wire the shortcut roles; BGP carries the overlay reachability; diagnose commands prove the shortcut ever fired.

ADVPN Phase 1 Overlay Interfaces BGP for ADVPN Mixed & Non-ADVPN Spokes Verify Shortcuts
💭 THINK FIRST · ADVPN PHASE 1

ADVPN adds three role knobs to phase 1 (auto-discovery-sender, auto-discovery-receiver, auto-discovery-forwarder). Which one you enable depends purely on the device's role in the topology.

  • Which of the three auto-discovery-* settings goes on the hub, the spoke, and a hub that connects to another hub?
  • Why is set net-device disable paired with set add-route disable on the hub's dial-up phase 1?
Q1 — MODEL ANSWER
Hub → auto-discovery-sender enable (sends the shortcut offer). Spoke → auto-discovery-receiver enable (accepts the offer and starts the shortcut). Hub-to-hub → auto-discovery-forwarder enable on the inter-hub tunnel so shortcut messages cross into the other overlay.
Q2 — MODEL ANSWER
net-device disable stops FortiOS from creating a dynamic per-spoke child interface — you want the spokes to share the same parent VPN interface so a dynamic routing protocol can peer across all of them. add-route disable stops FortiOS from injecting IKE-mode-config routes; you'll learn spoke prefixes over BGP or OSPF instead, which is exactly what ADVPN needs to steer shortcut traffic.
SECTION 01

ADVPN Phase 1

The three auto-discovery flags turn a normal hub-and-spoke phase 1 into an ADVPN one. Everything else — dynamic routing, dst-subnet in phase 2 — is the plumbing that makes the shortcut usable.

CONFIG
Hub # config vpn ipsec phase1-interface edit "VPN1" set type dynamic set interface "port1" set ike-version 2 set net-device disable set add-route disable set auto-discovery-sender enable next end

Hub-side ADVPN phase 1. type dynamic is required — the hub accepts inbound spoke connections. auto-discovery-sender enable makes the hub send a shortcut offer when it sees transit traffic between two spokes. net-device disable + add-route disable keep the interface single and let a dynamic routing protocol learn spoke prefixes.

What to look for

IKEv2 is required — ADVPN shortcut messaging piggybacks on IKEv2 information exchanges. Without ike-version 2, the offer never gets sent. You still need a firewall policy allowing spoke-to-spoke traffic across the hub, otherwise there's no transit for the hub to offer a shortcut for.

CONFIG
Spoke # config vpn ipsec phase1-interface edit "HUB1-VPN1" set ike-version 2 set auto-discovery-receiver enable set net-device enable next end

Spoke-side ADVPN phase 1. auto-discovery-receiver enable tells the spoke to accept shortcut offers and initiate the on-demand tunnel. net-device enable lets the spoke create dynamic interfaces for the shortcuts it forms with other spokes.

What to look for

The spoke keeps its static tunnel to the hub. When it receives a shortcut offer, it queries, gets the peer's WAN IP, and initiates a new on-demand IKE negotiation directly to that peer. If the shortcut fails, traffic just keeps flowing through the hub — no outage.

CONFIG
HubA # config vpn ipsec phase1-interface edit "Hub2Hub" set auto-discovery-forwarder enable next end HubA # config vpn ipsec phase1-interface edit "ADVPN" set auto-discovery-sender enable set network-id 1 next end

Dual-region ADVPN. Hub2Hub is the tunnel between HubA and HubB — auto-discovery-forwarder enable makes it relay shortcut messages across overlays. ADVPN is the tunnel accepting spokes, and network-id tags it so multiple IKEv2 tunnels between the same gateway pair stay distinguishable.

What to look for

network-id is Fortinet-proprietary — it's how ADVPN steers shortcut messages when a spoke could be reachable through more than one overlay. Set a different network-id per overlay. Not needed for single-region deployments.

🧠
MENTAL NOTE
Sender = hub. Receiver = spoke. Forwarder = hub-to-hub. IKEv2 is not optional — ADVPN messaging rides on IKEv2 informational exchanges.

💭 THINK FIRST · OVERLAY INTERFACES

ADVPN needs each IPsec virtual interface to have an overlay IP address so a dynamic routing protocol can peer across it. That address is separate from the underlay WAN IP — the underlay is what carries the ESP; the overlay is the routing fabric.

  • Why do you configure set ip with a /32 mask and set remote-ip on the same IPsec virtual interface?
  • Phase 2 selector is set to 0.0.0.0/0 on both sides. What does that buy you in ADVPN?
Q1 — MODEL ANSWER
set ip is this device's overlay address (the "loopback-like" address on the tunnel interface), set remote-ip is the address of the far end / broadcast domain over the ADVPN overlay. Both together let the two ends look like L3 neighbours to BGP/OSPF even though they're really over IPsec.
Q2 — MODEL ANSWER
Wide-open selectors mean the tunnel doesn't reject any encrypted traffic based on subnet — routing is what decides what's actually forwarded across it. Combined with dynamic routing, this gives ADVPN its flexibility: you can add or remove spoke subnets without editing phase 2 selectors on the hub.
SECTION 02

Overlay Interfaces

The IPsec virtual interface is where the overlay IP lives — that's what BGP or OSPF peers see. Phase 2 stays wide open; routing does the steering.

CONFIG
Hub # config system interface edit "VPN1" set ip 172.16.1.1 255.255.255.255 set remote-ip 172.16.1.254/24 next end

Assigns the hub's overlay IP on the ADVPN tunnel interface. The /32 is the hub's own address; remote-ip defines the broadcast domain the spokes will use — every spoke's overlay IP lives in that /24.

What to look for

The overlay subnet (here 172.16.1.0/24) is the ADVPN broadcast domain. All spokes get a /32 inside it (172.16.1.2, 172.16.1.3, …) with set remote-ip 172.16.1.1/24 pointing back at the hub. BGP or OSPF then peers using those addresses.

CONFIG
Spoke1 # config system interface edit "HUB1-VPN1" set ip 172.16.1.2 255.255.255.255 set remote-ip 172.16.1.1/24 next end

Spoke's overlay IP. /32 as the local address, remote-ip pointing at the hub with the /24 mask that defines the ADVPN broadcast domain.

What to look for

Every spoke needs a unique overlay IP. When ADVPN builds a shortcut between two spokes, next-hop for the far spoke's subnets becomes that spoke's overlay /32. If two spokes share an overlay IP, the shortcut works but the routes collide.

CONFIG
Hub # config vpn ipsec phase2-interface edit "VPN1" set phase1name "VPN1" set proposal aes256-sha256 set src-subnet 0.0.0.0 0.0.0.0 set dst-subnet 0.0.0.0 0.0.0.0 next end

Wide-open phase 2 for ADVPN. Setting both src-subnet and dst-subnet to 0.0.0.0/0 tells IPsec to accept any traffic — routing decides what actually goes through.

What to look for

Quick-mode selectors on both sides must be 0.0.0.0/0 — otherwise the IPsec engine rejects packets that don't fit the tightened selector even if the routing table would send them there. Same proposal on hub and spoke; a mismatch here shows up in diagnose vpn ike log-filter.

🧠
MENTAL NOTE
Underlay carries ESP. Overlay carries routing. Spokes get unique /32 overlay IPs so the shortcut's next-hop resolves cleanly through the tunnel.

💭 THINK FIRST · BGP FOR ADVPN

ADVPN needs a dynamic routing protocol so spokes can learn each other's subnets — otherwise, even after a shortcut forms, the hub is still the only known next hop. IBGP with a route reflector on the hub is the canonical choice.

  • Why do you need set route-reflector-client enable on the hub's neighbor group and not on the spokes?
  • For EBGP between two hubs in different regions, why is set ebgp-enforce-multihop enable required?
Q1 — MODEL ANSWER
By default IBGP does not re-advertise routes learned from one IBGP peer to another IBGP peer (loop prevention). Route reflection breaks that rule for its clients — the hub, acting as reflector, forwards Spoke1's prefixes to Spoke2 and vice-versa. Without it, spokes never learn each other's subnets and shortcuts have nowhere to route.
Q2 — MODEL ANSWER
EBGP peers are normally expected to be directly connected (TTL=1). Between two hubs the BGP session is over the ADVPN overlay, which is at best two hops away (physical + tunnel + physical). ebgp-enforce-multihop tells BGP to accept the session even though the peer isn't a direct neighbour.
SECTION 03

BGP for ADVPN

Single-region deployments use IBGP with the hub as route reflector. Multi-region deployments use EBGP between hubs plus IBGP inside each region.

CONFIG
Hub # config router bgp set as 65100 set router-id 172.16.1.1 config neighbor-group edit "Overlay1" set remote-as 65100 set route-reflector-client enable end config neighbor-range edit 1 set prefix 172.16.1.0 255.255.255.0 set neighbor-group "Overlay1" end config network edit 1 set prefix 10.0.12.0 255.255.255.0 end end

Hub BGP config. neighbor-group defines the shared settings for every spoke (all in the same AS 65100). route-reflector-client enable makes the hub reflect prefixes between spokes. neighbor-range accepts any peer coming from the overlay subnet, so you don't have to define each spoke individually.

What to look for

neighbor-range matches spokes by the source IP of their BGP session — which will be their overlay /32. Adding a new spoke is now zero-touch on the hub side: as long as the spoke's overlay IP falls inside the range, the hub accepts the neighbor.

CONFIG
Spoke1 # config router bgp set as 65100 set router-id 172.16.1.2 config neighbor edit "172.16.1.1" set remote-as 65100 end config network edit 1 set prefix 172.20.2.0 255.255.255.0 end end

Spoke BGP config. Peers only with the hub (172.16.1.1) — the hub reflects other spokes' prefixes back. Advertises the local LAN subnet (172.20.2.0/24) so the hub can share it with peers.

What to look for

Spokes never need to know about each other in BGP config — the hub is the only neighbor. All learned routes come back with the origin spoke's overlay IP as next-hop, which is exactly what ADVPN needs to steer post-shortcut traffic.

CONFIG
HubA # config router bgp set as 65100 set router-id 172.16.1.1 config neighbor edit "10.255.255.2" set attribute-unchanged next-hop set ebgp-enforce-multihop enable set remote-as 65200 end end

EBGP session between HubA (AS 65100) and HubB (AS 65200) for multi-region ADVPN. ebgp-enforce-multihop is required because the session rides across the ADVPN overlay, not a direct link. attribute-unchanged next-hop preserves the original next-hop so cross-region shortcuts can resolve.

What to look for

Without attribute-unchanged next-hop, EBGP would rewrite the next-hop to the advertising hub — killing ADVPN's ability to build spoke-to-spoke shortcuts across regions. This is the setting that makes cross-region shortcuts actually work.

🧠
MENTAL NOTE
Single region = IBGP + route reflector on hub. Multi region = EBGP between hubs (ebgp-enforce-multihop + attribute-unchanged next-hop) + IBGP inside each region.

💭 THINK FIRST · MIXED & NON-ADVPN SPOKES

ADVPN is Fortinet-proprietary. A third-party spoke (or a legacy IKEv1 spoke) can still participate in the topology — it just won't form shortcuts. To make it announce its subnets you need a second phase 2 with the overlay prefix as dst-subnet.

  • The extra phase 2 for non-ADVPN spokes must be named so it's matched before the wildcard Spoke phase 2. Why does the name matter?
  • What does set add-route enable on that overlay phase 2 do that plain phase 2 doesn't?
Q1 — MODEL ANSWER
Phase 2 lookup happens in alphabetical order. Overlay_advertisement starts with "O", Spoke starts with "S" — the more specific overlay phase 2 gets matched first, so its dst-subnet prefix is what the non-ADVPN spoke sees in the SA proposal. Without the alphabetical trick, the wildcard Spoke phase 2 would win and the overlay prefix would never be advertised.
Q2 — MODEL ANSWER
It installs an "IKE route" for the phase 2 destination subnet into the FortiOS routing table — a static route derived from the IKE selector. That's how the hub learns to return traffic to the non-ADVPN spoke's overlay without needing a dynamic routing session, because the third-party device may not speak BGP or OSPF.
SECTION 04

Mixed and Non-ADVPN Spokes

ADVPN is Fortinet-proprietary — third-party spokes can join the topology but won't form shortcuts. The workaround: give them an extra phase 2 whose dst-subnet covers the overlay, and lean on add-route to inject IKE routes.

CONFIG
Hub # config vpn ipsec phase1-interface edit "Spoke" set type dynamic set interface "port2" set proposal aes256-sha256 set add-route disable set auto-discovery-sender enable set psksecret <secret> next end

Phase 1 config stays unchanged from a normal ADVPN hub. auto-discovery-sender is still on — ADVPN-aware spokes will use it, non-ADVPN spokes will just ignore the shortcut messages.

What to look for

The single phase 1 accepts both ADVPN and non-ADVPN peers. Nothing to change here — the trick is at phase 2.

CONFIG
Hub # config vpn ipsec phase2-interface edit "Spoke" set phasename "Spoke" set proposal aes256-sha256 next edit "Overlay_advertisement" set phasename "Spoke" set proposal aes256-sha256 set add-route enable set comments "Used by legacy Spokes to advertise their overlay IP" set dst-subnet 172.16.1.0 255.255.255.0 next end

Two phase 2 selectors on the same phase 1. Overlay_advertisement is the extra selector for non-ADVPN spokes — its dst-subnet covers the overlay, and add-route enable injects the IKE route so the hub learns how to reach the third-party device's overlay IP.

What to look for

Naming matters — phase 2 lookup is alphabetical, so Overlay_advertisement matches before Spoke. The third-party spoke gets the specific overlay subnet in the selector; ADVPN-aware spokes still fall through to the wildcard Spoke selector. Verify with diagnose vpn ike config list.

🧠
MENTAL NOTE
Name phase 2 selectors alphabetically. Specific-first-alphabetical, wildcard-last-alphabetical — that's the reliable order for mixed ADVPN / non-ADVPN hubs.

💭 THINK FIRST · VERIFY SHORTCUTS

Two questions come up every time ADVPN misbehaves: "did the shortcut ever form?" and "if it did, why isn't my traffic riding it?" The routing table and the loaded phase 2 config together answer both.

  • Which command shows the next-hop for a remote spoke's subnet — and how do you tell whether traffic is going via the hub or through a shortcut?
  • Where do you look to prove the extra phase 2 (Overlay_advertisement) is actually loaded in the IKE process?
Q1 — MODEL ANSWER
get router info routing-table details <remote-subnet>. If the next-hop is the hub's overlay IP, traffic still hairpins through the hub. If the next-hop is the remote spoke's overlay IP (like 172.16.1.3) and the interface is a shortcut tunnel, the shortcut is up and routing across it.
Q2 — MODEL ANSWER
diagnose vpn ike config list. Phase 2 selectors currently loaded appear under phase2s:. If Overlay_advertisement is there with the right dst-subnet and add-route flag, it's live; if it's absent, the config didn't parse or wasn't named alphabetically before the wildcard.
SECTION 05

Verify Shortcuts

Two views tell you everything: get router info routing-table (is traffic going via a shortcut?) and diagnose vpn ike config list (are the selectors loaded?). Everything else is drill-down.

GET
Spoke1 # get router info routing-table details 172.20.3.62

Prints the routing entry for a specific host — including the next hop and the tunnel interface that carries it. In ADVPN, watching this before and after generating traffic is how you confirm a shortcut has replaced the hub as the next-hop.

Sample output
Routing entry for 172.20.3.0/24
  Known via "bgp", distance 200, metric 0, best
  * vrf 0 172.16.1.3 priority 1 (recursive is directly connected, HUB1-VPN1_0_0)
What to look for

The next-hop 172.16.1.3 is the remote spoke's overlay /32, and the recursive resolution shows it's now directly connected via HUB1-VPN1_0_0 — a dynamically-created shortcut child interface. Before the shortcut, the same route would have resolved through the hub's overlay IP.

DIAGNOSE
# diagnose vpn ike config list

Shows the phase 2 selectors currently loaded by the IKE process — the runtime view, not the config file. Essential for confirming that mixed-ADVPN selectors (like Overlay_advertisement) landed in the right alphabetical order.

Sample output
Hub # diag vpn ike config list
...
phase2s:
  Overlay_advertisement proto 0 src 0.0.0.0/0.0.0.0:0 dst 172.16.1.0 255.255.255.0:0 dhgrp 14 5 add-route
  Spoke proto 0 src 0.0.0.0/0.0.0.0:0 dst 0.0.0.0/0.0.0.0:0  dhgrp 14 5
What to look for

Overlay_advertisement listed before Spoke = alphabetical ordering worked. If a selector isn't listed, it never loaded — check the config and reload. add-route flag proves the IKE route injection is enabled for that selector.

DIAGNOSE
# diagnose vpn tunnel list

Lists every IPsec SA on the box. On a spoke that has formed shortcuts, you'll see the static hub tunnel plus one dynamically-created child tunnel per active shortcut.

What to look for

Shortcut tunnels appear as child interfaces named like HUB1-VPN1_0_0 (parent name + suffix). Their rxp/txp counters should increment as traffic flows spoke-to-spoke. If the child SA is present but counters aren't moving, the shortcut formed but routing hasn't picked it up — go back to the routing table.

DIAGNOSE
# diagnose vpn ike log-filter dst-addr4 <spoke-wan-ip> # diagnose debug application ike -1 # diagnose debug enable

Watches the IKE exchange scoped to a single peer. In ADVPN debugging, this is how you catch the six shortcut messages (Offer → Query → Forward → Reply → Forward → Negotiation).

What to look for

You want to see the hub send the shortcut offer, the initiating spoke query for the peer's WAN IP, and the receiving spoke reply. If the offer never appears, phase 1 is missing auto-discovery-sender. If the reply never comes back, the far spoke is missing auto-discovery-receiver — or IKEv1 is still in the way.

🧠
MENTAL NOTE
A shortcut is "working" when: routing-table shows the far spoke's overlay /32 as next-hop, vpn tunnel list shows a child SA with incrementing counters, and ike config list shows both phase 2 selectors loaded.
