Dynamic Routing, By Command

Every OSPF and BGP command shown in the Dynamic Routing supplement — redistribution, network advertisement, prefix-list filtering, and the OSPF/BGP verification you'll run every time.

Adjacencies fail on tiny mismatches — get router info tells you which one.

OSPF areas, BGP route-selection tiebreakers, and prefix-list defaults all bite the same way: the config looks right, but a subtle default makes routes vanish.

💭 THINK FIRST · OSPF VERIFICATION

OSPF adjacencies have a strict checklist — same subnet/mask, same area, matching hello and dead intervals, matching MTU, unique router IDs, matching authentication. Any single mismatch stops the state machine before it reaches Full. Verification commands exist so you can see exactly where it stopped.

  • Which single command shows this router's OSPF area membership, ABR/ASBR flags, LSA counts, and neighbour count in one screen?
  • You suspect a Type 3 (Summary) LSA is missing from a stub-adjacent area. Which OSPF database command narrows it down?
Q1 — MODEL ANSWER
get router info ospf status. It prints the router ID, the ABR/ASBR flags, the SPF timers, the LSA type counts, and — per area — the number of interfaces and fully adjacent neighbours. That's the one-liner you paste in when someone says "OSPF isn't working."
Q2 — MODEL ANSWER
get router info ospf database — that's the LSDB dump, broken out by LSA type per area. Filter by network for Type 2, summary for Type 3, asbr-summary for Type 4, or external for Type 5. Missing Type 3 = the ABR didn't generate/leak it, which usually means area 0 adjacency isn't Full or the area is stub with summary suppression.
SECTION 01

OSPF Status and Database

Every router in an area maintains an identical LSDB, built from the LSAs its neighbours flood. The two get router info ospf variants let you see the local view (status, neighbour count) and the raw database (every LSA type, per area).

GET
# get router info ospf status

One-shot OSPF health check. Prints the router ID, whether this router is an ABR/ASBR, SPF timers, LSA counts, and per-area neighbour statistics.

Sample output
Routing Process "ospf 0" with ID 0.0.0.3
Process uptime is 1 hour 12 minutes
...
This router is an ABR, ABR Type is Standard (RFC2328)
...
Number of areas attached to this router: 2
    Area 0.0.0.0 (BACKBONE)
        Number of interfaces in this area is 1(1)
        Number of fully adjacent neighbors in this area is 2
        Area has no authentication
        SPF algorithm executed 7 times
        Number of LSA 5. Checksum 0x03bd6a
    Area 0.0.0.1 (Stub)
        Number of interfaces in this area is 1(1)
        Number of fully adjacent neighbors in this area is 1
What to look for

"Number of fully adjacent neighbors" per area — if it's zero, the adjacency isn't Full. Check the area type (Stub/normal/NSSA) matches on both ends. Router ID must be unique. If the router shows is an ABR, it has interfaces in more than one area — one of which must be 0.0.0.0.

GET
# get router info ospf database # get router info ospf database {router | network | summary | asbr-summary | external}

Dumps the OSPF LSDB. Without a filter you get every LSA type per area; with a filter you narrow to one of the five common types.

What to look for

router = Type 1 (every router in the area). network = Type 2 (only DR-generated, one per multiaccess network). summary = Type 3 (ABR-generated inter-area). asbr-summary = Type 4 (ABR-generated, describes the path to an ASBR). external = Type 5 (ASBR-generated, redistributed routes). Type 5 is never seen in stub or NSSA areas.

GET
# get router info ospf neighbor

Lists every OSPF neighbour with its state (Down / Init / 2-way / ExStart / Exchange / Loading / Full) and the interface where it was learned.

What to look for

Full = adjacency succeeded, LSDBs synced. 2-way on multiaccess = normal for non-DR/BDR pairs; on point-to-point it's a problem. Stuck in ExStart/Exchange almost always means MTU mismatch on the interfaces. Stuck in Init means hellos are one-way (return path or ACL problem).

GET
# get router info ospf interface

Per-interface OSPF settings: area, cost, network type (broadcast/point-to-point/point-to-multipoint), hello/dead intervals, DR/BDR, priority.

What to look for

Hello and dead intervals must match on both peers or the adjacency fails. Network type must match. Cost (1–65535) drives route selection — the sum of costs to the destination wins. On multiaccess, priority 0 means "never become DR."

🧠
MENTAL NOTE
Adjacency requirements: same subnet + mask, same interface type + area, matching hello/dead, unique router ID, matching MTU, matching authentication. If get router info ospf neighbor shows anything other than Full, one of those six is wrong.

💭 THINK FIRST · BGP ORIGINATION

By default, FortiGate BGP originates no prefixes at all. You have two ways to change that — protocol redistribution (from connected/static/OSPF/RIP into BGP) or the network command (explicit prefix list). Each has a subtle default trap.

  • You typed config router bgp / config network / edit 1 / set prefix 10.10.0.0/16 and BGP still isn't advertising it. What's the most likely reason?
  • Why does redistributing static routes into BGP sometimes advertise more than you intended, and how would you filter that?
Q1 — MODEL ANSWER
The network command only advertises a prefix when the exact prefix is active in the routing table. If 10.10.0.0/16 isn't a real active route (only, say, 10.10.1.0/24 is), it's suppressed. Fix: either add a matching static route, or set network-import-check disable under config router bgp to skip the check and advertise regardless.
Q2 — MODEL ANSWER
Because config redistribute "static" pushes every static route into BGP — including ones you'd rather keep internal. Attach a route-map on the redistribution, or use a prefix-list on the outbound side of each neighbour, to filter the set. Route maps are the general-purpose filter/attribute-modify tool; prefix lists are the narrower "which prefixes are allowed" list.
SECTION 02

BGP Prefix Origination

Two ways to make FortiGate BGP say something. config redistribute pulls another protocol's routes into BGP. config network lists prefixes explicitly. Both default to advertise nothing extra.

CONFIG
# config router bgp config redistribute "static" set status enable end end

Redistributes the named non-BGP source (static, connected, ospf, rip, isis) into BGP. Once status enable is set, every matching route in the routing table becomes a BGP-originated prefix on this router.

What to look for

Redistribution is all-or-nothing unless you add a route-map. Attach one with set route-map <name> under the same redistribute block to selectively permit prefixes or rewrite BGP attributes as they enter BGP. Default is status disable — no redistribution at all.

CONFIG
# config router bgp config network edit <id> set prefix <prefix> next end end

Explicit list of prefixes for BGP to advertise. Cleaner than redistribution because you name the exact prefixes — but by default each one is advertised only when a matching active route already exists in the routing table.

What to look for

Set set prefix to the exact CIDR you want in BGP. If the routing table doesn't contain that exact prefix (mask included), FortiGate suppresses it — this is the network-import-check in effect. See the next card for the override.

CONFIG
# config router bgp set network-import-check disable end

Disables the "prefix must match an active route" check. After this, every prefix in the config network list is advertised regardless of what the routing table looks like.

What to look for

Default is enable — safer, because you can't advertise a prefix you can't actually route. Turn it off only when you deliberately want to originate an aggregate that no interface holds (a summary announcement from an edge router, for example).

🧠
MENTAL NOTE
"BGP is broken and no prefixes appear" → default: BGP originates nothing. Add config redistribute or config network. "One prefix from the network list doesn't show up" → network-import-check is enforcing an exact match against the RIB.

💭 THINK FIRST · PREFIX LISTS & FILTERING

By default, every route redistributed or in config network is announced to every neighbour, and every prefix received from every neighbour is accepted. Prefix lists let you tighten both ends — but they carry an invisible implicit deny at the end.

  • Your prefix list has one rule permitting 10.0.0.0/8. A neighbour advertises 172.16.5.0/24. What happens, and why?
  • What's the difference between applying a prefix list with set prefix-list-in versus set prefix-list-out on a neighbour?
Q1 — MODEL ANSWER
The 172.16.5.0/24 is denied. Prefix lists have an implicit deny at the end — anything that doesn't match a rule is dropped. So a list with just one permit rule for 10.0.0.0/8 permits only 10.0.0.0/8 and denies everything else.
Q2 — MODEL ANSWER
prefix-list-in filters what this router accepts from that neighbour (populates RIB-in → local RIB after the filter). prefix-list-out filters what this router advertises to that neighbour (RIB-out). Same list can be applied in one, the other, or both — each direction is independent.

SECTION 03

Prefix Lists and Neighbor Filters

Prefix lists live under config router prefix-list. They're evaluated top-down, first match wins, with an implicit deny at the end. Apply them per-neighbour, per-direction.

CONFIG
config router prefix-list edit filter-subnets config rule edit 1 set prefix 10.1.0.0/16 set action deny next edit 2 set prefix 10.0.0.0/8 set action permit next end next end

Creates a named prefix-list filter-subnets. Rules are evaluated by ID, ascending — first match wins. Here, 10.1.0.0/16 is denied before the broader 10.0.0.0/8 permit, so that one specific subnet is filtered while the rest of 10/8 passes.

What to look for

Rule order matters — put the specific deny before the broader permit or the broader match will hit first. Implicit deny at the end: any prefix that doesn't match a rule is denied. So if the list is meant to allow "everything except X," you need an explicit permit-all rule at the bottom.

CONFIG
config router bgp config neighbor edit 10.3.1.254 set prefix-list-in filter-subnets next end end

Applies the prefix list to the inbound direction from neighbour 10.3.1.254. Every prefix the neighbour advertises is checked against filter-subnets; denied prefixes are dropped before they hit the local RIB.

What to look for

Same list can be applied outbound with set prefix-list-out. Route-maps (set route-map-in / set route-map-out) are the more powerful cousin — they can also rewrite BGP attributes (local-pref, MED, communities). Use prefix lists when you just want a filter; use route maps when you also need to shape the route's attributes.

🧠
MENTAL NOTE
Prefix list = who's allowed through the door. Route map = who's allowed through the door and what stamp we put on their forehead. If you're only filtering, prefix list. If you're changing local-pref, MED, or community, route map.

💭 THINK FIRST · BGP VERIFICATION

A BGP router stores routes in three RIBs — RIB-in (raw from peers), local RIB (after inbound filters), RIB-out (what we advertise). Each has its own verification command, and the ten-step route-selection tiebreaker is what picks the winner in the local RIB.

  • Two BGP routes to the same destination arrive with identical local preference and same AS-path length. Which attribute breaks the tie next?
  • You want to see every prefix a specific neighbour advertised to you, even the ones your inbound filter dropped. Which command?
Q1 — MODEL ANSWER
Origin type (IGP < EGP < incomplete — lowest wins), then MED (lowest), then EBGP over IBGP, then lowest IGP metric to the BGP next hop, then (with ECMP) up to 10 parallel routes, else the lowest router ID. Order of the ten tiebreakers is: weight → local-pref → locally-originated → AS-path length → origin type → MED → IGP-to-next-hop → EBGP-over-IBGP → ECMP → router ID.
Q2 — MODEL ANSWER
get router info bgp neighbors <ip> received-routes. That's the RIB-in view for that peer — everything they sent you, filter-free. Compare against routes (post-filter local RIB) to see what the inbound filter dropped.
SECTION 04

BGP Neighbors and Routing Table

Three commands cover 90% of BGP troubleshooting — one for peer state, one for the RIB-in view per peer, one for the merged routing table.

GET
# get router info bgp summary

One-line-per-neighbour summary. Local AS, router ID, and — per peer — the remote AS, uptime, and state or number of prefixes received.

What to look for

The State/PfxRcd column. A number = adjacency is Established and that's how many prefixes came in from RIB-in. A word (Idle, Active, Connect) = the session is not up. Active means "actively trying and failing" — check TCP 179, source IP, source-interface, or authentication mismatch.

GET
# get router info bgp neighbors <ip> # get router info bgp neighbors <ip> received-routes # get router info bgp neighbors <ip> routes # get router info bgp neighbors <ip> advertised-routes

Per-neighbour detail. Without a suffix you get the peer state, capabilities, and counters. With a suffix you get: received-routes (RIB-in, before local inbound filter), routes (local RIB after filter), advertised-routes (RIB-out to this peer).

What to look for

Difference between received-routes and routes = what your inbound filter dropped. Difference between what you expect and advertised-routes = what your outbound filter dropped or what your redistribute/network failed to originate. That's the debugging triangle.

GET
# get router info routing-table all # get router info routing-table bgp

Full RIB (all protocols) or just BGP-installed routes. Each entry shows the source code letter (B for BGP, O for OSPF, S for static), the administrative distance and metric in [AD/metric], and the next-hop.

Sample output
Codes: K - kernel, C - connected, S - static, R - RIP, B - BGP
       O - OSPF, IA - OSPF inter area
       N1 - OSPF NSSA external type 1, N2 - OSPF NSSA external type 2
       E1 - OSPF external type 1, E2 - OSPF external type 2

S*    0.0.0.0/0 [10/0] via 100.65.5.35d, port2, [1/0]
O    10.0.12.0/24 is directly connected, port5, [1/0]
C    172.16.1.1/32 is directly connected, HUB1-VPN1
B    172.20.3.0/24 [200/0] via 172.16.1.2 (recursive is directly connected, HUB1-VPN1_0), 00:02:56, [1/0]
What to look for

The prefix's leading letter tells you which protocol won the AD contest. BGP AD defaults to 200 for IBGP and 20 for EBGP — connected (0) and static (10) beat BGP by default, so redistribution loops can silently disappear behind a static route. Also look for recursive next-hops — a BGP next-hop that resolves through a tunnel or an IGP-learned route.

🧠
MENTAL NOTE
Ten-step BGP route selection: Weight → Local-pref → Locally-originated → AS-path length → Origin type → MED → IGP metric to next-hop → EBGP over IBGP → ECMP → Router ID. Memorise the order or you'll be second-guessing every close race.
