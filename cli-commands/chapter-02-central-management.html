Chapter 02 · CLI Reference

Central Management, By Command

The FortiGate-side CLI for FortiManager registration, FGFM tunnel troubleshooting, FortiGuard/FDS behaviour, and the CLI-script contexts (device DB, policy package, remote FortiGate) that FortiManager uses to push change.

When install fails, you'll live inside diagnose fdsm and diagnose debug application fgfm.

Registration is easy. The pain begins when the FGFM tunnel keeps flapping or a policy package refuses to install — and both are diagnosed from the FortiGate CLI, not the FortiManager GUI.

💭 THINK FIRST · CENTRAL-MGMT REGISTRATION

FortiManager doesn't just show up. Somewhere, either on the FortiGate or on FortiManager, the pairing was declared. On the FortiGate side it's one config block: config system central-management.

  • What does set type fortimanager plus set fmg <ip> actually cause the FortiGate to do?
  • Why does FortiManager still show the device as "unauthorised" after the FortiGate config is correct?
Q1 — MODEL ANSWER
The FortiGate opens an outbound FGFM tunnel on TCP/541 (or TCP/443 for FMG-Access) to the specified FortiManager IP, presenting its serial number. This is what a ZTP/LTP device does the moment it boots and gets network reachability — it phones home to the pre-declared FortiManager.
Q2 — MODEL ANSWER
Because pairing is a two-sided handshake. The FortiGate is now knocking at the door, but on FortiManager the administrator still has to authorise the device (or the FortiGate had to be pre-registered by serial number as a model device). Until then it sits in the unauthorised queue.
SECTION 01

Central-Mgmt Registration

Two commands do the whole pairing: config system central-management to declare the FortiManager, and execute central-mgmt register-device to trigger the handshake explicitly.

CONFIG
config system central-management set type fortimanager set fmg <fortimanager-ip> set serial-number <fmg-serial> end

Declares the target FortiManager on the FortiGate side. Once committed, the FortiGate opens the FGFM tunnel outbound to the FortiManager IP.

What to look for

set type fortimanager is what tells the FortiGate to speak FGFM; without it, this block does nothing. The serial-number field binds the tunnel to a specific FortiManager, preventing a rogue FMG from hijacking the connection. The pairing still needs the FortiManager admin to authorise the device — or the FortiGate must already exist as a model device.

EXECUTE
# execute central-mgmt register-device <fmg-serial> <registration-password> <fgt-user> <fgt-password>

Explicitly triggers the FGFM registration handshake using a pre-shared key (the LTP flow — Low-Touch Provisioning). Alternative to serial-number-only ZTP when your process requires a shared secret.

What to look for

The pre-shared key must match the one the FortiManager admin configured on the model device. Wrong key = registration fails silently; check FortiManager's Device Manager log or run the FGFM debug on the next card.

EXECUTE
# execute central-mgmt unregister # execute central-mgmt update

unregister tears down the tunnel and clears the pairing. update forces the FortiGate to re-sync its state with the FortiManager immediately, rather than waiting for the next heartbeat.

What to look for

Use unregister before you rebuild the pairing — leaving a half-broken FGFM state around is a common cause of "install succeeds on FMG but nothing changes on the FortiGate." update is the "please pick up my most recent state now" nudge.

🧠
MENTAL NOTE
Central-mgmt is a two-sided handshake: the FortiGate opens the FGFM tunnel (config system central-management), FortiManager authorises the serial or matches it to a model device. Neither side alone is enough.

💭 THINK FIRST · FGFM TUNNEL DIAGNOSTICS

The FortiGate–FortiManager Protocol (FGFM) is the tunnel every install, every retrieve, every quick-install rides over. When it flaps, nothing FortiManager does actually reaches the FortiGate.

  • You install a policy package on FortiManager, it says "installed successfully", but the FortiGate hasn't received the change. Where do you look first?
  • Why does the FGFM debug need to be filtered? What happens on a busy box if you don't?
Q1 — MODEL ANSWER
The FGFM tunnel status on the FortiGate side: diagnose fdsm central-mgmt-status and then the FGFM debug application to see the tunnel state. If FGFM is down, FortiManager thinks it installed but the changes are queued and never actually push.
Q2 — MODEL ANSWER
Because FGFM debug is verbose — every heartbeat, every keepalive, every state change gets printed. On a device with active traffic, unfiltered debug will scroll faster than you can read and add measurable CPU load. Always end a debug session with diagnose debug disable and diagnose debug reset.
SECTION 02

FGFM Tunnel Diagnostics

When a policy package or device-settings install doesn't take effect, the FGFM tunnel is the first suspect. These commands sit on the FortiGate side.

DIAGNOSE
# diagnose fdsm central-mgmt-status

Prints the current central-management state: the configured FortiManager IP, tunnel status (up/down), and the last-seen heartbeat time.

What to look for

Tunnel up means installs will actually reach this device. Tunnel down means every install on FortiManager gets queued locally — the GUI can still say "installed successfully" while the FortiGate never sees the change. Also confirms which FortiManager IP the FortiGate is trying to reach — a common bug when the FMG IP was changed after registration.

DIAGNOSE
# diagnose debug application fgfm -1 # diagnose debug enable

Turns on live FGFM protocol debug output. Prints every FGFM message exchanged with FortiManager — heartbeats, install pushes, retrieve requests, auth handshakes.

What to look for

The -1 is "all levels" — you can pass a smaller integer for less noise. Look for auth failures (wrong pre-shared key or serial mismatch), TLS errors (certificate expired), and install responses (success/failure with reason). Always end with diagnose debug disable and diagnose debug reset.

DIAGNOSE
# diagnose test application fgfmd 1

Runs the FGFM daemon's built-in self-test: prints internal state, the configured FortiManager list, and the current tunnel session ID.

What to look for

Different integer arguments (1, 2, 3…) dump different internal structures — 1 gives you the state summary. When the tunnel keeps flapping, comparing the state before and after a flap here tells you which side dropped it (FortiGate NAT/firewall vs. FortiManager rejected the auth).

EXECUTE
# execute fgfm reconnect

Forces the FortiGate to tear down and rebuild the FGFM tunnel to FortiManager. Faster than a reboot when the tunnel is stuck.

What to look for

Use this after a firewall or NAT change on the path between FortiGate and FortiManager — the existing tunnel may be holding stale state. Also useful right after you fix a config issue on FortiManager; without a reconnect, the FortiGate may wait up to a minute before the next heartbeat.

🧠
MENTAL NOTE
FGFM tunnel down = FortiManager's "install successful" is a lie. Always check central-mgmt-status on the FortiGate before you trust the FMG install log.

💭 THINK FIRST · FORTIGUARD & FDS

The chapter's ADOM architecture slide shows FortiGuard as the source of the global database. On the FortiGate side, a companion service handles the actual signature pulls — either from the public FortiGuard cloud or from a FortiManager acting as a local FortiGuard Distribution Server (FDS).

  • In a closed network, FortiManager doubles as an FDS. Which FortiGate CLI setting is what actually redirects the signature pull to FortiManager instead of the internet?
  • Why does the FortiGate FortiGuard status show "unreachable" even when the FortiGate can ping FortiManager fine?
Q1 — MODEL ANSWER
config system central-management → set include-default-servers disable → config server-list with the FortiManager IP and set server-type update rating. This tells the FortiGate to pull antivirus / IPS updates and web/DNS rating queries from FortiManager instead of the public FortiGuard servers.
Q2 — MODEL ANSWER
FortiGuard connectivity uses specific TCP/UDP ports (443 for updates/rating in 7.6). Ping proves L3 reachability but proves nothing about port filtering. And in multi-VDOM setups (Ch03), FortiGuard traffic sources from the management VDOM — if that VDOM has no internet or its default route is broken, updates fail even though other VDOMs work fine.
SECTION 03

FortiGuard and FDS

The FortiGate pulls signatures and rating info from either public FortiGuard or a FortiManager acting as a local FDS. These commands cover both the config and the diagnostics for that pull.

CONFIG
config system central-management set include-default-servers disable config server-list edit 1 set server-type update rating set server-address <fmg-ip> next end end

Redirects FortiGuard pulls to FortiManager acting as a local FDS. include-default-servers disable stops the FortiGate from also trying the public FortiGuard cloud as a fallback.

What to look for

set server-type update rating covers both signature pulls (update) and web/DNS category queries (rating). In a fully closed network you always want include-default-servers disable — otherwise the FortiGate wastes cycles trying to reach FortiGuard cloud on every update and generates alarming logs.

DIAGNOSE
# diagnose debug rating

Prints the current rating-server list and the last query results — used to confirm the FortiGate is actually reaching FortiManager (or FortiGuard) for web filter / DNS category lookups.

What to look for

Each server entry shows an RTT and a "packets lost" counter. Rising loss = the FortiGate is timing out on the rating queries — check management-VDOM reachability first, then check port 443 filtering on the path.

DIAGNOSE
# diagnose test application fgtserverd 3 # diagnose test application forticldd 2

Internal daemon self-tests. fgtserverd handles the FortiGate side of FGT-to-FortiGuard communication; forticldd is the FortiCloud daemon. Different integer args dump different internal structures.

What to look for

Arg 3 on fgtserverd is the "print current state" call — use it to verify what the FortiGate believes the FortiGuard update servers are right now. If that list is stale, FortiManager as an FDS won't be tried at all.

EXECUTE
# execute update-av # execute update-ips # execute update-now

Forces an immediate signature pull for antivirus, IPS, or all FortiGuard services. Useful after you fix an FDS/FortiManager path — no need to wait for the next scheduled poll.

What to look for

Watch diagnose debug application update -1 in a second window while these run — you'll see the actual HTTPS pull, the version compare, and the "no new signatures" or "downloaded" line. That's the fastest way to prove the FDS path is working end-to-end.

🧠
MENTAL NOTE
FortiGuard traffic sources from the management VDOM. In multi-VDOM, a broken management VDOM = broken updates for every VDOM. Setting FortiManager as an FDS only fixes half the problem if the management VDOM still can't reach the FMG IP.

💭 THINK FIRST · SCRIPT CONTEXTS

FortiManager runs FortiOS CLI scripts against three different targets — remote FortiGate directly, the FortiGate device database, or a policy package / ADOM database. Same syntax, three very different behaviours.

  • You run a CLI script that creates a firewall address. It runs against the device database. What has to happen next before the address exists on the actual FortiGate?
  • Why does running a script directly on the remote FortiGate (via CLI) trigger an auto-retrieve, but running it against the device database doesn't?
Q1 — MODEL ANSWER
An install device settings (or quick install) is required. The device database is FortiManager's staging area — the script only marks the device as "Modified" until an install pushes the changes over FGFM to the real FortiGate.
Q2 — MODEL ANSWER
A direct-CLI script bypasses FortiManager's staging — the changes happen on the FortiGate first, then FortiManager auto-retrieves the new revision so its device database stays in sync. A device-database script goes the other direction: FortiManager stages, then pushes on install. The direction determines whether an auto-retrieve is needed.
SECTION 04

Script Contexts

The same FortiOS CLI syntax runs in three FortiManager contexts. Understanding which context you're in is how you avoid "the script ran but nothing changed" surprises.

CONFIG
config firewall address edit "DMZ_Address" set subnet 10.10.10.0 255.255.255.0 next end

A representative FortiOS CLI script — creates a firewall address. FortiManager runs this same block against one of three targets: remote FortiGate directly (via CLI), device database, or policy package / ADOM database.

What to look for

The syntax doesn't change. What changes is the target selector at the top of the FortiManager script editor. Remote FortiGate directly (via CLI): runs on the real device, then FortiManager auto-retrieves. Device Database: stages the change; requires install-device-settings or quick-install to push. Policy Package / ADOM Database: stages in the policy layer; requires install-policy-package.

CONFIG
config system global set vdom-mode multi-vdom end

A FortiManager pre-run CLI template example — a one-time bootstrap script that runs during ZTP/LTP before the FortiGate joins the device database. Auto-detaches after the quick install so it doesn't run again.

What to look for

Pre-run templates are the ZTP/LTP mechanism for "do this exactly once at first-boot." After the FortiGate connects with Automatically Link to Real Device enabled, the template applies then unassigns itself. Use it for VDOM-mode changes, mgmt-interface pre-config, or admin-account bootstrap.

CONFIG
config firewall address edit "LAN" set subnet $(CLI_LAN_Address) next end

CLI template with a FortiManager metadata variable. The $(...) reference is resolved per-device at install time — different FortiGates get different subnets from the same template.

What to look for

Metadata variables are defined under Policy & Objects → Advanced → Metadata Variables with a default value and optional per-device overrides. When a CLI template using a variable is assigned, the FortiGate's status shows Modified with the exclamation triangle until the next install.

CONFIG
config vpn certificate ca edit "Custom-CA" set ca "-----BEGIN CERTIFICATE----- ... -----END CERTIFICATE-----" next end

Representative "remote FortiGate directly" script — pushes a CA certificate straight to the FortiGate via FGFM. After the push, FortiManager auto-retrieves the new revision so the device database stays synchronised.

What to look for

Configuration Revision History on FortiManager should show the new revision as Retrieved, and the device's Config Status should read Synchronized. If it doesn't, the auto-retrieve failed — check FGFM tunnel status. Avoid modifying objects used in firewall policies via direct-CLI scripts — that creates conflicts between the device layer and the policy layer.

🧠
MENTAL NOTE
Three script contexts, three behaviours. Direct = "runs now, auto-retrieve syncs FMG." Device DB = "stages, needs install-device-settings." Policy package = "stages, needs install-policy-package." Pick the wrong context and the change either doesn't apply or applies twice.
