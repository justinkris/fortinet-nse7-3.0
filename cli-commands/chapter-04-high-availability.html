Chapter 04 · CLI Reference

High Availability, By Command

Every FGCP, FGSP, and VRRP command from Chapter 4 — the config knobs, the diagnose views, and the field to actually look at.

When the cluster misbehaves, you'll live inside diagnose sys ha.

Config commands stand up the cluster; diagnose commands tell you why the election went the way it did or why a member won't sync.

FGCP Config Election & Failover Virtual MAC & Heartbeat Virtual Cluster & Signaling FGSP & VRRP
💭 THINK FIRST · FGCP CONFIG

Before you paste a big config system ha block onto both units, ask what the cluster actually needs to agree on. Two things must match: the identity of the group (so members recognise each other) and the heartbeat plumbing (so they can talk).

  • Which of the settings inside config system ha must be identical on both cluster members, and which are per-unit?
  • Why is it a best practice to specify at least two heartbeat interfaces in set hbdev?
Q1 — MODEL ANSWER
Group ID, group name, HA password, and the list of heartbeat interfaces (with their priorities) must match on every member — that's what tells the cluster "you and I are the same cluster." Priority and override are per-unit and are how you steer which member wins the election.
Q2 — MODEL ANSWER
Redundancy. If the single heartbeat link flaps you'll get split-brain or a bogus failover. Two heartbeat interfaces let the cluster fall back to the next-highest-priority link if the primary drops, and the numbers after each port name ("port3" 10 "port4" 20) control that priority order.
SECTION 01

FGCP Basic Config

The foundation of an FGCP cluster. Every command below is entered inside config system ha unless otherwise noted.

CONFIG
config system ha set mode a-p set group-id 10 set group-name "Training" set password <password> set hbdev "port3" 10 "port4" 20 end

The canonical FGCP bootstrap. Sets the cluster to active-passive, gives it a group identity, and nominates two heartbeat ports with priorities 10 and 20 (higher number = higher priority for heartbeat use).

What to look for

Every one of these settings — group-id, group-name, password, and the hbdev list — must be identical on every cluster member. Any drift and the members won't recognise each other. Also: use static addresses on cluster ports during initial setup — DHCP/PPPoE can hand out different IPs mid-election.

CONFIG
config system ha set schedule <none|leastconnection|round-robin|weight-round-robin|random|ip|ipport> end

Active-active load balancing method. Only the primary distributes sessions; secondaries never make load-balancing decisions themselves.

What to look for

Default is round-robin. Pick ip or ipport when you need session stickiness. none means the primary handles everything — useful when you want A-A for redundancy but not load-spread. Note: by default only proxy-inspection sessions get distributed; to load-balance flow-inspection or unscanned traffic you also need set load-balance-all enable.

CONFIG
config system ha set link-failed-signal enable | disable end

After a failover, some upstream switches don't clear their MAC tables fast enough and keep forwarding to the old primary. Enabling this bounces every non-heartbeat interface on the former primary for one second to force the switches to relearn.

What to look for

Default is disable. Turn it on when you see traffic being blackholed to the former primary after a failover — the gratuitous ARP alone isn't being honoured by your L2 fabric. Heartbeat and reserved-management interfaces are excluded from the bounce.

🧠
MENTAL NOTE
Group ID + group name + password + hbdev list are the "same cluster?" question — mismatched, no cluster. Priority and override are per-unit and answer "who wins the election?"

💭 THINK FIRST · ELECTION & FAILOVER

The primary election has four tiebreakers (monitored ports → HA uptime → priority → serial number), but their order depends on one setting: override. Understanding why you'd flip it is the whole game.

  • With override disable (the default), you want to manually fail over — which diagnose command is the intended way?
  • Why is set override enable a double-edged sword? What extra failover event does it introduce?
Q1 — MODEL ANSWER
diagnose sys ha reset-uptime on the current primary. With override disabled, HA uptime beats priority, so resetting the primary's HA uptime to zero makes the other member win the next election. It only resets HA uptime, not system uptime.
Q2 — MODEL ANSWER
With override enable, priority is checked before HA uptime, so you can guarantee "unit X is always primary as long as it's up." The downside: when the preferred primary comes back after a failure, it triggers a second failover to reclaim its role. Every recovery causes a service blip. Also, override and priority are not synchronised between members — you must set them per unit.
SECTION 02

Election and Failover

Two commands do all the election work: one to trigger a controlled failover, one to see why the current primary won.

DIAGNOSE
# diagnose sys ha reset-uptime

Zeros the HA uptime on the current member. Combined with the default override disable, this is the intended way to hand primary role to the other member without pulling cables.

What to look for

Run this on the current primary. The other member's HA uptime is now higher by at least 5 minutes, so it wins the next election. It resets HA uptime only — the system uptime is untouched.

DIAGNOSE
# diagnose sys ha dump-by vcluster

Prints per-member HA uptime and reset counters — the raw data behind the election. Use it to confirm which unit has the higher uptime and whether uptime has been reset (either manually or by an interface flap / reboot).

Sample output
...
FGVMxx92:...uptime/reset_cnt=7814/0
FGVMxx93:...uptime/reset_cnt=0/1
What to look for

The uptime value (in seconds) — the higher one wins the HA-uptime tiebreaker. The reset_cnt tells you how many times that unit's HA uptime has been zeroed. In the sample above, unit 93 has been reset once and now has 0s uptime, so unit 92 is the primary.

CONFIG
config system ha set override enable end

Reorders the election criteria: priority is now checked before HA uptime. This makes one member the "always primary as long as it's up" — but at the cost of a second failover event when it recovers.

What to look for

CLI-only setting — GUI can't touch it. Not synchronised — set it on each member. To force failover with override on, just adjust set priority up on the secondary or down on the primary.

🧠
MENTAL NOTE
Default election order: connected monitored ports → HA uptime → priority → serial. Override flips uptime and priority. Reset-uptime is the "please fail over now" button when override is off.

💭 THINK FIRST · VIRTUAL MAC & HEARTBEAT

HA traffic doesn't ride the IP stack — it uses raw ethertypes. Knowing which ones (and how to sniff them) is the difference between "the cluster is broken" and "the switch between them is filtering."

  • You suspect heartbeats aren't reaching the peer. What sniffer filter proves whether the frames are leaving the port?
  • How do you verify that an interface is using its virtual MAC and not the burned-in hardware MAC?
Q1 — MODEL ANSWER
diagnose sniff packet any "ether proto 0x8890" 4. HA heartbeats use ethertype 0x8890 (and 0x8891). Config sync uses 0x8893. If you don't see 0x8890 frames leaving the port, the FortiGate isn't the problem — check the switch or the cable.
Q2 — MODEL ANSWER
diagnose hardware deviceinfo nic <port> | grep addr — the output shows both the current HWaddr (what the port is transmitting now) and the Permanent Hwaddr (the burned-in MAC). If they differ, the virtual MAC is active.
SECTION 03

Virtual MAC and Heartbeat

FGCP uses three ethertypes on the wire: 0x8890 / 0x8891 for heartbeats and election, 0x8893 for configuration sync. When something looks off, sniff the wire.

DIAGNOSE
# diagnose sniff packet any "ether proto 0x8890" 4

Sniffs HA heartbeat frames across every interface. Verbosity 4 gives you the interface name so you can tell which port the frame is coming from.

What to look for

Frames arriving on the heartbeat interfaces on a steady cadence. No frames = the peer isn't sending, or something between them is dropping the ethertype. Change the filter to 0x8893 to sniff config-sync traffic instead.

DIAGNOSE
# diagnose hardware deviceinfo nic <port> | grep addr

Shows the current transmitted MAC and the permanent hardware MAC for a given port. Used to confirm whether a virtual MAC is in effect.

Sample output
Hwaddr:            06:d5:90:04:f8:9c
Permanent Hwaddr:  04:d5:90:04:f8:9c
What to look for

Compare Hwaddr with Permanent Hwaddr. Different = the virtual MAC is in use. Same = you're using the burned-in MAC (either HA isn't active on this port, or you've configured a manual per-interface override).

DIAGNOSE
# diagnose sys ha dump-by debug-zone

Prints the FortiGate's internal HA debug zone including every interface's ifindex and phyindex. Those are the raw inputs to the virtual-MAC assignment logic (which of the four logics applies depends on group ID, vcluster ID, and phyindex).

Sample output
...
Port1  ifindex=3  phyindex=0  mac=...
Port7  ifindex=9  phyindex=6  mac=...
What to look for

The phyindex value drives which vMAC logic gets picked. Different HA clusters sharing an L2 domain should use different HA group IDs — same group ID + overlapping phyindex + overlapping vcluster ID = vMAC collision.

CONFIG
config system interface edit <interface> set virtual-mac <mac_address> next end

Manual per-interface virtual MAC override. Highest-priority assignment method — beats automatic per-interface and group-ID-based assignment.

What to look for

Only reach for this when the four automatic logics collide with another cluster or with a legacy static ARP somewhere upstream. If you're manually setting vMACs on every port, you probably want set auto-virtual-mac-interface <interface> instead.

🧠
MENTAL NOTE
HA rides three ethertypes: 0x8890/0x8891 for heartbeat and election, 0x8893 for config sync. When in doubt, sniff.

💭 THINK FIRST · VIRTUAL CLUSTER & SIGNALING

Virtual clustering is how you get load sharing out of an active-passive cluster: partition VDOMs so unit A is primary for VDOM1 and unit B is primary for VDOM2. The config is a matched pair — both units need it, with the priorities flipped.

  • Which two set priority values need to be flipped between the two members for VDOM partitioning to actually distribute load?
  • Why is set vcluster-status enable required before the config vcluster block does anything?
Q1 — MODEL ANSWER
On NGFW-1: vcluster 1 priority=200, vcluster 2 priority=50 → NGFW-1 wins VDOM1. On NGFW-2: vcluster 1 priority=100, vcluster 2 priority=200 → NGFW-2 wins VDOM2. Each unit is primary for the vcluster where its priority is higher.
Q2 — MODEL ANSWER
It's the gate — without it, FortiOS ignores everything inside config vcluster and treats the cluster as a plain single-vcluster A-P setup. Virtual clustering also requires HA mode to be a-p; it doesn't work with active-active.
SECTION 04

Virtual Cluster and Failover Signaling

Virtual clustering partitions VDOMs across cluster members. Each vcluster has its own priority, so each member can be primary for a different VDOM set.

CONFIG
NGFW-1 # config system ha set mode a-p set vcluster-status enable config vcluster edit 1 set override disable set priority 200 set vdom "VDOM1" "root" next edit 2 set override disable set priority 50 set vdom "VDOM2" next end end

Virtual clustering config on member 1. This member is primary for vcluster 1 (VDOM1 + root) because its vcluster-1 priority is 200. It's the secondary for vcluster 2 (VDOM2) because its priority there is only 50.

What to look for

The pair of set priority values — one high (200), one low (50). On member 2 you flip them: 100 in vcluster 1, 200 in vcluster 2. Now each unit is the primary for one vcluster, and traffic for each VDOM lives on a different physical box.

CONFIG
NGFW-2 # config system ha set mode a-p set vcluster-status enable config vcluster edit 1 set override disable set priority 100 set vdom "VDOM1" "root" next edit 2 set override disable set priority 200 set vdom "VDOM2" next end end

Virtual clustering config on member 2. Mirror of member 1 with the priorities inverted — this member becomes the primary for VDOM2, and secondary for VDOM1 and root.

What to look for

The set vdom lists must match on both members — the same VDOM lives in the same vcluster on every member. Only the priorities differ.

CONFIG
config system ha set link-failed-signal enable end

Post-failover switch-recovery kick. When enabled, the former primary shuts down every non-heartbeat / non-reserved-management interface for one second, forcing upstream switches to flush their MAC tables and re-learn.

What to look for

Symptom that tells you to enable this: after a failover, traffic still gets blackholed to the former primary for tens of seconds because a switch upstream ignored the gratuitous ARP. The bounce forces a link-down event that every switch honours.

🧠
MENTAL NOTE
Virtual clustering = two A-P clusters overlaid on the same two boxes, each carrying different VDOMs. Priorities flipped per vcluster is what makes it "load sharing" without needing active-active.

💭 THINK FIRST · FGSP & VRRP

FGSP isn't a full HA solution — it's just session sharing between peers that a load-balancer distributes traffic to. VRRP is even lighter — no session sync, just gateway redundancy at the IP layer.

  • By default FGSP syncs TCP and IPsec sessions only. Which set session-pickup-* settings expand that coverage, and to what session types?
  • Why does the VRRP config block sit inside config system interface rather than under a global config system vrrp?
Q1 — MODEL ANSWER
set session-pickup-connectionless enable adds UDP and ICMP. set session-pickup-expectation enable adds firewall-opened pinholes (SIP, FTP data channels). set session-pickup-nat enable adds NAT sessions. They're not free — every extra session type adds sync bandwidth and CPU on both peers.
Q2 — MODEL ANSWER
Because VRRP is per-interface — you can have different VRRP virtual routers on different interfaces, each with its own virtual IP, priority, and monitored destination. The config vrrp block sits inside edit <port> because that's the scope where a VRRP virtual router lives.
SECTION 05

FGSP and VRRP

FGSP shares sessions between peers that an external load balancer distributes traffic to. VRRP just handles first-hop gateway redundancy with no session sync at all.

CONFIG
config system standalone-cluster config cluster-peer edit 1 set peerip 10.10.10.1 next end set standalone-group-id 1 set group-member-id 1 end

Minimum FGSP peer setup on Peer_1. Points at Peer_2's L3 IP, tags this member as ID 1 in group 1. The mirror config on Peer_2 uses set peerip 192.168.26.1 (Peer_1's IP) and set group-member-id 2.

What to look for

By default, session sync rides L3 between the two peerip addresses — no L2 required. The standalone-group-id and group-member-id are what pair the two peers together.

CONFIG
config system ha set session-pickup enable set session-pickup-connectionless enable set session-pickup-expectation enable set session-pickup-nat enable end

Expands FGSP session-sync coverage. By default only TCP and IPsec sessions are synced — these four toggles cover UDP/ICMP (connectionless), pinhole/expectation sessions (SIP, FTP data), and NAT sessions.

What to look for

Every extra session-pickup setting adds sync bandwidth on the heartbeat / session-sync link. Enable only what your traffic actually needs. If you want a UDP-heavy workload (DNS, RTP) to survive failover, you need session-pickup-connectionless on.

CONFIG
config system standalone-cluster set session-sync-dev port4 end

Nominates a dedicated interface for session sync. If the two peers are directly connected on this port, sync happens at L2; if that link fails, FortiOS falls back to L3 sync via the peerip.

What to look for

Direct L2 cable between peers on port4 = fastest sync. Any switch in between still works but degrades the "layer 2" advantage. Session sync only — configuration sync uses its own path.

CONFIG
config system standalone-cluster set encryption enable set psksecret fortinet end

Encrypts FGSP session-sync traffic by auto-creating an IPsec tunnel between the peers. Only supported on L3 sync connections.

What to look for

Use this when the two peers live in different data centers and the sync traffic crosses untrusted infrastructure. The IPsec tunnel and firewall policies get created automatically — you'll see them appear in VPN > IPsec Tunnels and Policy & Objects > Firewall Policy.

CONFIG
config system standalone-cluster set standalone-group-id 1 set group-member-id 1 set layer2-connection available set session-sync-dev port1 end

FGSP with asymmetric-traffic support at L2. Traffic is always forwarded back to the session owner (the peer that saw the first packet) for UTM inspection. Requires the internal/outgoing interfaces on both peers to share a subnet with L2 reachability.

What to look for

layer2-connection available is the key toggle — it tells FortiOS to bounce return packets over L2 to the session owner using the owner's MAC. In cloud environments where L2 isn't available, set it to unavailable and unset session-sync-dev; FortiOS will encapsulate the return traffic in UDP instead.

CONFIG
config global config system ha set group-id 1 set group-name "Cluster1" set mode standalone set password fortinet1 set hbdev "port2" 50 "port3" 50 set standalone-config-sync enable set override disable set priority 200 end end

Enables standalone configuration sync for FGSP peers (mode=standalone, standalone-config-sync enable). Uses the same election machinery as FGCP but the elected "primary" only owns config sync — traffic is still distributed by the external load balancer.

What to look for

mode standalone is the distinguisher — the FortiGate is not an FGCP member, just an FGSP peer. Requires L2 adjacency between hbdev interfaces on the two peers, and multiple hbdev entries for redundancy.

CONFIG
config system interface edit port10 config vrrp edit 200 set vrip 10.10.101.250 set priority 200 set adv-interval 250 next end next end

Adds a VRRP virtual router on port10 with virtual IP 10.10.101.250, priority 200, and advertisement interval 250 (centiseconds = 2.5s). Higher priority = more likely to be primary. The backup routers have the same VRID (200 here) but lower priority.

What to look for

VRRP is per-interface, so this whole block sits inside edit port10. VRID must match on every device in the VRRP domain; priority is what breaks the tie. FortiOS supports VRRPv2 and v3, for both IPv4 and IPv6.

CONFIG
edit port10 config vrrp edit 200 set vrdst 8.8.8.8 set vrdst-priority 30 next end

Adds a VRRP destination monitor. If the primary VRRP router loses reachability to 8.8.8.8, its effective priority drops by 30 (from 200 → 170), letting the backup with priority 100 take over.

What to look for

Up to two vrdst targets per virtual router. Pick remote reachable addresses (a public DNS resolver like 8.8.8.8) rather than something local — you want the failover to trigger when the WAN is broken, not just when a local switch reboots.

🧠
MENTAL NOTE
FGCP = full HA (sync + failover). FGSP = session sync only, external load-balancer does distribution. VRRP = gateway redundancy at the IP layer, no session sync at all. Pick the lightest one that meets the requirement.
