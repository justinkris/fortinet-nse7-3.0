Chapter 05 · CLI Reference

Dynamic Routing, By Command

Every OSPF and BGP command from Chapter 5 — the area knobs, the neighbor plumbing, and the get commands that tell you why an adjacency won't come up.

When a prefix isn't in the routing table, you'll live inside get router info.

Config commands stand up the protocol; the get and diagnose commands tell you why the LSDB is empty or the BGP session is stuck in Idle.

💭 THINK FIRST · OSPF CORE

OSPF is a link-state protocol — every router in an area holds an identical LSDB. Before you paste an ospf block, ask what identity the router uses on the wire (router ID) and which area each interface joins. Area type and network type dictate what LSAs get flooded.

  • What is the default OSPF area type on a FortiGate, and what CLI keyword changes it to a stub or NSSA area?
  • Why would you set an interface's OSPF network-type to point-to-multipoint instead of the default point-to-point?
Q1 — MODEL ANSWER
Default is regular. Change it under config area > edit <area-id> with set type stub or set type nssa. A stub blocks type-5 external LSAs (the ABR injects a default instead); an NSSA is like a stub but allows an ASBR inside via type-7 LSAs.
Q2 — MODEL ANSWER
Dial-up hub-and-spoke IPsec, where the hub uses net-device disable and shares one virtual tunnel interface for many spokes. Default point-to-point assumes exactly two neighbors on the segment; point-to-multipoint lets the hub form adjacencies with every spoke over that single interface.
SECTION 01

OSPF Core Config

The commands that stand the process up: the router-id, the areas, the networks, and the per-interface network type. Every command here lives inside config router ospf.

CONFIG
config router ospf set router-id 2.2.2.2 config area edit 0.0.0.1 set type stub next edit 0.0.0.3 set type nssa next end end

Bootstrap OSPF with a router ID and two non-backbone areas — one stub, one NSSA. Every OSPF network must have at least one backbone area (0.0.0.0); ABRs bridge non-backbone areas to it.

What to look for

Default area type is regular and doesn't need to be declared. Stub areas block type-5 external LSAs and see external destinations only through a default route from the ABR. NSSA is a stub that does allow an ASBR — external routes ride type-7 LSAs and get translated to type-5 by the ABR.

CONFIG
Hub # config router ospf config ospf-interface edit "Hub" set interface To-Spokes set network-type point-to-multipoint next end end

Per-interface OSPF network type. Default is point-to-point. Change to point-to-multipoint on a dial-up hub with net-device disable, where one virtual IPsec interface is shared by every spoke and the hub must form multiple adjacencies over it.

What to look for

Directly connected routers with one neighbor per link should stay on point-to-point. Reach for point-to-multipoint when the interface has more than one neighbor and you don't want a DR election (as you would on broadcast).

CONFIG
R2 # config router ospf config redistribute bgp set status enable end config redistribute connected set status enable end config redistribute static set status enable end end

OSPF redistribution knobs. By default OSPF does not advertise routes learned from other protocols, or even directly connected networks that aren't declared inside config network. Flip status to enable for each source protocol you want an ASBR to inject.

What to look for

Available sources: bgp, connected, isis, rip, static. All default to disable. Injecting the wrong set is how routing loops start — always pair a redistribute with a route-map or prefix-list to bound what gets leaked.

CONFIG
R2 # config router ospf set router-id 2.2.2.2 config area edit 0.0.0.3 config virtual-link edit VLINK_area set peer 4.4.4.4 next end next end end

Virtual link. Used when a non-backbone area (here, 0.0.0.10) has no physical connection to area 0 — you tunnel it through a transit area (0.0.0.3) to a router that is in the backbone. Configure it on both ends, each pointing at the other's router-id.

What to look for

Transit area cannot be a stub. Both routers need matching edit <area> and set peer <other-router-id>. The pair extends area 0 across the transit area — a stopgap when your topology grew out of order.

🧠
MENTAL NOTE
Regular → stub → NSSA is a spectrum of "how much external stuff do I want to see." Default is regular. Redistribute is off by default on every source — that's why "my connected net isn't advertised" is the #1 OSPF new-user question.

💭 THINK FIRST · OSPF FILTERING & GET

Filtering OSPF has three flavors — access-list, prefix-list, and route-map — each strictly more powerful. When something isn't showing up in the routing table, five get router info ospf commands cover 90% of the diagnosis.

  • What's the difference between a prefix-list and an access-list when used with distribute-list-in?
  • Which get router info ospf command tells you whether an adjacency reached FULL, and what does the output look like?
Q1 — MODEL ANSWER
Access-list matches purely on prefix + mask. Prefix-list adds ge and le logical operators, so one rule can match "any /24 or longer inside 10.0.0.0/8." Both are only usable on the inbound side (distribute-list-in) — for outbound OSPF filtering or attribute rewrite, you need a route-map with distribute-route-map-in.
Q2 — MODEL ANSWER
get router info ospf neighbor. The State column shows the FSM state — Full/DR, Full/BDR, or Full/DROther means the LSDB exchange completed. If it's stuck in Init, 2-Way, or ExStart, the adjacency never formed — usually mismatched hello/dead intervals, mismatched area, or an MTU mismatch.
SECTION 02

OSPF Filtering and Get Commands

Route-maps, access-lists, prefix-lists — and the get router info ospf family that tells you what OSPF thinks is happening.

CONFIG
config router ospf config distribute-list edit 1 set access-list <name> set protocol connected | static | rip next end end

Access-list-based inbound filter (distribute-list-in). Blocks external routes from other protocols before OSPF injects them into the routing table.

What to look for

Simple, prefix-only match. Sits at the front of the OSPF-to-RIB path. Use when you just need to say "these subnets never make it into the table." For anything more nuanced, reach for a prefix-list or route-map.

CONFIG
config router ospf config distribute-list edit 1 set prefix-list <name> next end end

Prefix-list variant of the same inbound filter. Adds ge/le range operators for "everything of length X or longer inside prefix Y."

What to look for

Still only inbound filtering — that's the entire capability of access-list / prefix-list in OSPF. If you need outbound, or if you want to rewrite metrics or tags, jump to a route-map with distribute-route-map-in or config redistribute … set route-map.

SHOW / GET
# get router info ospf neighbor

Status of every OSPF neighbor and the FSM state of each adjacency. First stop when something isn't converging.

What to look for

The State column — you want Full/DR, Full/BDR, or Full/DROther. Anything else (Init, 2-Way, ExStart, Exchange, Loading) is a stuck adjacency; usually mismatched hello/dead timers, MTU, area ID, or auth.

SHOW / GET
# get router info ospf interface

Per-interface OSPF status: area, network type, hello/dead intervals, number of adjacencies, plus the current DR and BDR router IDs on that segment.

What to look for

Check hello/dead — mismatched values silently kill adjacencies. Verify network type is what you set (point-to-point vs point-to-multipoint). DR/BDR fields are meaningful only on broadcast/NBMA segments.

SHOW / GET
# get router info ospf status

The overall OSPF process view: this router's ID, the OSPF capabilities negotiated, the number of areas attached, and the count of fully adjacent neighbors.

What to look for

Confirm the router ID is what you configured (it can be auto-picked from an interface if you didn't set router-id). Confirm the areas attached match your intent — a missing area means the interface wasn't picked up by config network.

SHOW / GET
# get router info ospf database brief # get router info ospf database router lsa

Dumps the OSPF LSDB. brief is the whole database ordered by LSA type; router lsa zooms into type-1 LSAs (each router's own advertisement of its links).

What to look for

Every router in the same area must have an identical database. If two routers disagree on any LSA, there's a flooding problem. Type-1 LSAs are the "who am I connected to" record — missing a link there is a symptom of an interface not participating in OSPF.

SHOW / GET
# get router info bfd neighbor

BFD (Bidirectional Forwarding Detection) neighbor status. When BFD is enabled on OSPF neighbors it accelerates failure detection from ~40s (default OSPF dead) to sub-second.

What to look for

Session state Up and the detect timer around 150-300ms. If BFD isn't up, OSPF still runs but you'll wait for the OSPF dead interval on a failure — much slower failover.

🧠
MENTAL NOTE
Filter power ladder: access-list < prefix-list (adds ge/le) < route-map (adds outbound, tags, metrics). Diagnose ladder: get router info ospf neighbor → interface → status → database brief. Fix in that order.

💭 THINK FIRST · OSPF USE CASES

Three canonical use cases: ECMP for load-spreading external routes, OSPF over IPsec for encrypting the routing plane, and virtual links for reaching an area that grew out of the backbone.

  • By default, OSPF uses RFC 2328 path preference — why would you enable RFC 1583 compatibility with set rfc1583-compatible enable?
  • In an OSPF-over-site-to-site-IPsec design, which two things must be configured for the tunnel interfaces?
Q1 — MODEL ANSWER
RFC 2328 prefers intra-area non-backbone paths over intra-area backbone paths (to reduce backbone load), which means two equal-cost external routes through different areas don't tie under 2328 — one wins by area. RFC 1583 picks strictly on cost, so equal-cost external routes across different areas both install and you get real ECMP. Recommended when you want ECMP to external destinations.
Q2 — MODEL ANSWER
(1) Route-based IPsec tunnels with explicit tunnel-end IPs assigned on the phase1 config so the tunnel interfaces have IPs to talk over. (2) The tunnel subnets must appear in the OSPF config network stanza (or the interfaces must be declared in config ospf-interface) so OSPF forms adjacencies across the tunnel.
SECTION 03

OSPF Use Cases

ECMP, OSPF-over-IPsec, and virtual links — the three configurations you're most likely to see on a real Fortinet exam question.

CONFIG
R2 # config router ospf set rfc1583-compatible enable end

Enables RFC 1583 path preference. Selection then depends purely on cost, so equal-cost external routes learned through different areas will both install — real OSPF ECMP for external destinations.

Sample output
R2 # get router info routing-table all
O E2  10.2.0.0/24 [110/10] via 10.1.1.100, port4, 00:00:21, [1/0]
                  [110/10] via 10.1.0.1,   port3, 00:00:21, [1/0]
What to look for

Two via lines for the same prefix = ECMP installed. Under default RFC 2328 you'd only see one because 2328's path-preference rules kick in before cost. Turn RFC 1583 on when you want ECMP to external routes.

SHOW / GET
# get router info routing-table all

Full routing table with source-protocol codes: O = OSPF intra-area, O IA = inter-area, O E1/E2 = external type-1/type-2. The [110/x] is [admin-distance/metric].

What to look for

OSPF's admin distance is 110. A prefix in OSPF's LSDB that isn't in the routing table means something with a lower distance beat it — usually a static (10) or a directly connected (0). Confirm before you blame OSPF.

CONFIG
R2 # config router ospf set router-id 2.2.2.2 config area edit 0.0.0.3 config virtual-link edit VLINK_area set peer 4.4.4.4 next end next end end R4 # config router ospf set router-id 4.4.4.4 config area edit 0.0.0.3 config virtual-link edit VLINK_area set peer 2.2.2.2 next end next end end

Full virtual link config — both ends of the transit area (0.0.0.3) declare each other by router-id. After both apply, area 0.0.0.10 (behind R4) is virtually attached to backbone area 0.

What to look for

Router IDs must match what's configured on the peer (set router-id). Transit area cannot be a stub or NSSA. Verify with get router info ospf neighbor — you'll see a virtual neighbor listed once it's up.

🧠
MENTAL NOTE
RFC 1583 = strict cost, enables external ECMP. RFC 2328 = area-aware preference, no ECMP for externals. Virtual links = "I forgot to make sure every area touches area 0" duct tape. OSPF-over-IPsec = tunnel interfaces need IPs and must appear in the OSPF network stanza.

💭 THINK FIRST · BGP CORE

BGP is path-vector, not link-state. It runs over TCP 179 to explicit neighbors you declare by IP. The choice between EBGP and IBGP is decided by whether the neighbor's AS matches yours — no separate toggle.

  • Why does BGP not advertise anything by default, and which command controls what does get injected?
  • When you peer BGP to a loopback (rather than a physical interface), which two settings are mandatory?
Q1 — MODEL ANSWER
BGP is a policy protocol — advertising a prefix implies you take responsibility for the traffic. So by default connected, static, rip, ospf, and even bgp (into other AFIs) all sit at disable. The command is config redistribute <protocol> > set status enable, and each source has its own toggle. Alternatively use config network to advertise a specific prefix explicitly.
Q2 — MODEL ANSWER
(1) set update-source <loopback> — tells BGP to source the TCP session from the loopback IP so the peer's expected source matches. (2) set ebgp-enforce-multihop enable (or ebgp-multihop) — because the loopback is one hop away from any physical interface, the default TTL=1 for EBGP would drop the session. Loopbacks are used because they stay up even if one physical link fails.
SECTION 04

BGP Core Config

The AS, the router-id, the neighbor block. Everything lives inside config router bgp. Neighbors are explicit — no auto-discovery.

CONFIG
config router bgp set as 65100 set router-id 172.16.1.254 config neighbor edit 100.64.1.254 set remote-as 65101 next end end

Minimum BGP config: local AS, router-id, one neighbor with its IP and its AS. If remote-as = local AS, this is IBGP; otherwise EBGP.

What to look for

Neighbor IPs are TCP endpoints — they must be reachable via the routing table before BGP will come up. TCP 179 must not be blocked by any policy between the two speakers. Router-id is a 32-bit identifier — normally an IPv4, but doesn't need to be assigned to an interface.

CONFIG
config router bgp config redistribute bgp set status enable end config redistribute connected set status enable end config redistribute static set status enable end end

Same redistribution pattern as OSPF but for BGP. By default nothing gets advertised — flipping status to enable injects the named source into BGP's outbound updates.

What to look for

Sources: bgp, connected, isis, rip, static. Alternative to redistribution: config network announces one prefix explicitly and is safer for well-known aggregates you own.

CONFIG
config router bgp set as 65100 set router-id 172.16.1.254 config neighbor edit 100.64.1.254 set remote-as 65101 set update-source Loopback_Interface set ebgp-enforce-multihop enable next end end

Loopback-sourced EBGP session. update-source forces the BGP TCP session to originate from the loopback IP; ebgp-enforce-multihop raises TTL above the default 1-hop EBGP limit because the loopback is not on the wire between the two boxes.

What to look for

Loopbacks stay up even when a physical link fails, so BGP peering survives link flaps as long as another path exists. Firewall policy note: TCP 179 flows through a physical port even though it's sourced from the loopback — you need a policy that permits it between the loopback IP and the physical interface.

CONFIG
config router bgp set as 65100 set router-id 172.16.1.254 set ibgp-multipath enable config neighbor-group edit SpokeISP1 set interface ISP1 set remote-as 65100 next end config neighbor-range edit 1 set prefix 10.1.0.0 255.255.255.0 set neighbor-group SpokeISP1 next end end

Neighbor group + neighbor range. Used in SD-WAN hub configs: instead of explicitly declaring each spoke as a neighbor, the hub applies the group's common settings to any peer whose IP falls in the declared prefix. Spokes are dial-up BGP clients.

What to look for

Any peer inside 10.1.0.0/24 that opens a BGP session to this hub inherits the SpokeISP1 group's settings — interface ISP1 and AS 65100. Because the remote AS matches the local AS, this is IBGP; that's why ibgp-multipath enable is turned on to allow ECMP over the multiple underlays.

CONFIG
config router bgp set ebgp-multipath enable end

Enables ECMP for EBGP-learned routes. When there are multiple equal-cost paths to a BGP next hop (e.g. dual-ISP), BGP will install all of them for load balancing.

What to look for

Same-cost EBGP paths only. For IBGP-learned same-cost paths, use set ibgp-multipath enable instead. Both default to disabled — without either, BGP picks the single best path per prefix.

SHOW / GET
# get router info bgp summary

Per-neighbor summary: peer IP, remote AS, uptime, packet counters, and current session state.

What to look for

Look at the "up/down" time and the FSM state. Anything other than Established (or a big prefix count in the last column) means the session never fully came up — check TCP reachability, AS numbers, and the neighbor's own view.

SHOW / GET
# get router info bgp neighbors # get router info bgp network # get router info routing-table bgp

Three levels of BGP detail: full neighbor detail (peer IP, router ID, remote AS, BGP state, negotiated capabilities); the BGP RIB (every prefix BGP knows about, best path or not); and the subset of BGP prefixes that made it into the actual routing table.

What to look for

Prefix visible in bgp network but not in routing-table bgp = BGP received it but a better source won. Prefix visible on the peer's side but not here = an inbound filter (route-map, prefix-list) dropped it before it entered your local RIB.

🧠
MENTAL NOTE
BGP defaults are quiet: no auto-neighbors, no redistribution, no ECMP. Every one of those requires an explicit toggle. IBGP vs EBGP is decided by remote-as vs local as — nothing else.

💭 THINK FIRST · BGP CONVERGENCE

BGP convergence has three dials: RIB size (filters), update propagation (route reflectors), and failure detection (keepalive tuning or BFD). Plus one HA-specific dial: graceful restart for keeping routes during a controlled failover.

  • Why does an IBGP full mesh become a problem at scale, and what BGP construct concentrates the updates?
  • What does set graceful-restart enable plus set capability-graceful-restart enable do during an HA failover, and why does it matter?
Q1 — MODEL ANSWER
IBGP has a rule that a route learned from an IBGP peer will not be re-advertised to another IBGP peer (loop prevention). So every IBGP speaker must peer with every other IBGP speaker — an n(n-1)/2 mesh. Route reflectors (RRs) are IBGP speakers explicitly allowed to relay IBGP-learned routes to a list of "client" peers. Spokes only need to peer with the RR, not each other.
Q2 — MODEL ANSWER
Only the primary FortiGate runs the BGP daemon in an HA cluster. Normally a failover would tear down every BGP session (peer sees the TCP session drop, invalidates all routes) — a big blip. Graceful-restart advertises capability to peers so they preserve the received routes during a brief BGP restart window; the new primary comes up with a fresh usable routing table, no traffic drop, no route flap.
SECTION 05

BGP Convergence and HA

Route reflectors, BFD, and graceful-restart — the three levers between "BGP takes 3 minutes to reconverge" and "sub-second failover."

CONFIG
config router bgp config neighbor edit 100.64.1.254 set next-hop-self enable set route-reflector-client enable next end end

Marks the neighbor as a route-reflector client. This BGP speaker will now relay IBGP-learned routes to this client — breaking the full-mesh requirement for scale. Configure on the RR, not the client.

What to look for

Configure this on the RR, not on the client. The client has a normal neighbor block with no special setting. next-hop-self is usually paired with RR configs so the client resolves next-hop to the RR itself rather than to an external EBGP peer it can't reach.

CONFIG
config router bgp config neighbor edit 100.64.1.254 set bfd enable set ebgp-enforce-multihop enable next end end config router bfd config multihop-template edit 1 set src 100.64.2.0 255.255.255.0 set dst 100.64.1.0 255.255.255.0 set auth-mode md5 set md5-key password next end end

BFD-accelerated BGP. Under router bgp the neighbor gets bfd enable; under router bfd a multihop-template defines source/dest ranges and authentication. Now instead of waiting for BGP hold time (~180s default) after a failure, BFD flags the drop in under a second.

What to look for

Must be configured on both connected routers. Multihop path uses a template (source/dest range + auth); single-hop uses no template. Confirm operational status with get router info bfd neighbor.

CONFIG
config router bgp set as 65100 set graceful-restart enable set router-id 172.16.1.254 config neighbor edit 100.64.1.254 set capability-graceful-restart enable set remote-as 200 next end end

Global graceful-restart for BGP, plus per-neighbor advertisement of the graceful-restart capability. Used to prevent traffic interruption during HA failover — the peer keeps the received routes marked as "stale" instead of tearing them down, so packets keep flowing while the new primary reconverges.

What to look for

Only the primary FortiGate in an HA cluster runs BGP. Without graceful-restart, a failover flaps every BGP session — routes disappear, traffic drops. With it, the cluster advertises "I'm going offline briefly," peers hold onto the routes, and the new primary comes up with a usable table. Both sides must support and enable graceful-restart for it to work.

EXECUTE
# execute router clear bgp all # execute router clear bgp ip <neighbor-ip> soft in

Force a BGP session reset. The all form is a hard reset of every session; the soft in form just re-processes inbound updates from one neighbor without dropping the TCP session — useful after changing an inbound route-map.

What to look for

Hard resets flap the session — every route gets withdrawn and re-advertised. Use soft whenever you can. For a soft reset to work on the receive side, set soft-reconfiguration enable must be configured on the neighbor beforehand (or the peer must support route-refresh, which most modern implementations do).

🧠
MENTAL NOTE
Speed levers: filters trim the RIB, route reflectors collapse IBGP fan-out, BFD kills the 180s hold-time wait, graceful-restart keeps routes alive across HA. Use as many as your peers support — every one shaves seconds off convergence.
