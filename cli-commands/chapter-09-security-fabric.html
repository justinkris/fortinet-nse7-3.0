Chapter 09 · CLI Reference

Security Fabric, By Command

Every CSF, external-connector, automation-stitch, and quarantine command from Chapter 9 — the CLI behind the Security Fabric that the GUI hides.

Almost everything the Fabric GUI clicks is really a config system csf or config system automation-* change.

Learn the CLI shapes and you can debug fabric issues even when the GUI reports "synchronised" but the leaves don't agree.

CSF Fabric External Connectors Automation Stitches Quarantine & Diagnose
💭 THINK FIRST · CSF FABRIC

The Security Fabric is really a tree: one root FortiGate, downstream leaves that authorise upward, and a small ring of Fortinet appliances (FortiAnalyzer, FortiNDR, FortiNAC) that plug in as fabric members. All of the wiring lives under config system csf.

  • Which single set status enable or set status disable flag decides whether a FortiGate is participating in the Fabric at all?
  • On a leaf FortiGate, what tells FortiOS where to find the upstream root — and why does that not go on the root itself?
Q1 — MODEL ANSWER
set status enable inside config system csf. That single flag switches CSF on. Without it, the Fabric menus in the GUI still render, but the FortiGate won't advertise itself upstream or accept downstream joins.
Q2 — MODEL ANSWER
set upstream (the root's IP address). The root has no upstream — it's the top of the tree — so it doesn't need the setting. Leaves point at the root's fabric-facing IP, submit a join request, and wait for the root's admin to accept.
SECTION 01

CSF Fabric

The Security Fabric configuration lives in one command tree: config system csf. Root and leaf devices share the same command block — a couple of settings differ.

CONFIG
Root # config system csf set status enable set group-name "Acme_Corp_SF" set group-password <shared-secret> set configuration-sync default set fabric-object-unification default end

Minimum root FortiGate setup. status enable turns CSF on; group-name + group-password identify this Fabric so leaves know which root they're joining. The configuration-sync and fabric-object-unification defaults let downstream FortiGates share objects with the root.

What to look for

Group name and password must match on every member — leaves that get either wrong will silently fail authorisation. The root does not set upstream; only leaves do.

CONFIG
Leaf # config system csf set status enable set upstream 10.0.12.253 set group-name "Acme_Corp_SF" set group-password <shared-secret> end

Leaf FortiGate joining the Fabric. upstream is the root's fabric-facing IP; the group name and password have to match the root exactly. After this, the leaf's join request shows up in the root's top-bar alarm badge waiting for authorisation.

What to look for

The join is not automatic — the root admin has to accept it. Until they do, the leaf shows as pending and doesn't yet appear in the Logical or Physical Topology views. Wrong group-password = silent reject.

CONFIG
Root # config system csf config trusted-list edit "downstream-1" set serial "FGVM01T*" set action accept next end end

Pre-authorises downstream FortiGates by serial number (or wildcard prefix). If the joining leaf matches a trusted-list entry with action accept, it's admitted automatically instead of waiting for manual approval.

What to look for

Serial wildcards (FGVM01T*) are handy for large fleets, but they're also blast-radius — any device whose serial matches gets in. Prefer exact serials in production. action deny lets you explicitly block a serial that would otherwise be admitted.

DIAGNOSE
# diagnose sys csf global # diagnose sys csf downstream # diagnose sys csf upstream

Runtime view of the Fabric from this FortiGate's perspective. global shows the local CSF status; downstream lists every leaf that has joined this device; upstream shows the root that this leaf is connected to.

What to look for

If diagnose sys csf downstream is empty but the GUI says a leaf joined, that leaf never actually completed the authorisation — the top-bar badge is still waiting on a click. If upstream reports a disconnected state, check the leaf's upstream IP is reachable and the group-password matches.

🧠
MENTAL NOTE
Root sets status + group-name/password. Leaf sets those plus upstream. Join isn't automatic unless you populate trusted-list.

💭 THINK FIRST · EXTERNAL CONNECTORS

Fabric connectors are for Fortinet gear (FortiAnalyzer, FortiClient EMS, FortiNDR, FortiNAC). External connectors are for the rest of the world — public/private SDN, threat feeds, SSO. Both create dynamic objects that policies can reference.

  • Which command family creates an external threat feed of malicious MAC addresses that can then be used as a source in a firewall policy?
  • Why is the FortiAnalyzer connector considered a fabric connector and not an external connector?
Q1 — MODEL ANSWER
config system external-resource with set type mac-address, a URL pointing at the plain-text feed, and an update interval. Once it's active, a resource object with the same name shows up under Policy & Objects — you can drop it into a firewall policy as the source.
Q2 — MODEL ANSWER
Because it's Fortinet-native and part of the tightly-coupled Fabric integration — logs from every FortiGate in the Fabric consolidate at that one FortiAnalyzer, and the connector is what enables UTM and traffic log correlation across the whole Fabric. External connectors are for third-party feeds or multivendor cloud environments.
SECTION 02

External Connectors

Fabric connectors integrate Fortinet appliances. External connectors bring in third-party context — SDN, SSO, threat feeds — as dynamic objects usable in policy.

CONFIG
config log fortianalyzer setting set status enable set server 10.0.12.100 set upload-option realtime set reliable enable end

Configures the FortiAnalyzer fabric connector — the log consolidator for the whole Security Fabric. upload-option realtime streams logs immediately; reliable enable uses TCP/OFTPS for guaranteed delivery.

What to look for

Only the first FortiGate to handle a session logs it; upstream Fabric members skip the duplicate. FortiAnalyzer then correlates UTM events with the original traffic log automatically. If an upstream FortiGate performs NAT, it logs its own translated version — that's expected.

CONFIG
config system external-resource edit "blocked_MACs" set type mac-address set resource "https://example.com/badmacs.txt" set refresh-rate 5 set username "fortinet" set password <api-token> next end

Defines an external threat feed of malicious MAC addresses. FortiOS re-fetches from the URL on the refresh-rate interval (in minutes) and creates a matching resource object under Policy & Objects > Addresses that policies can reference.

What to look for

Feed types include ip, domain, mac-address, malware-hash, and category. Alternate delivery: switch to Push API (FortiGate REST) where a webhook updates entries in real time instead of polling a URL. Verify with diagnose test application external-resource.

CONFIG
config user fortinac edit "NAC1" set server 10.0.12.90 set password <shared-secret> next end

Adds a FortiNAC fabric connector so FortiGate can consume the dynamic firewall addresses FortiNAC creates when users log in — the device IP plus FortiNAC group names and firewall tags become a live source/destination object in policy.

What to look for

Requires two user login events on FortiNAC before the dynamic address is trusted. Verify the object landed with the FortiNAC Tag (IP Address) section under Policy & Objects > Addresses. Missing or empty = FortiNAC never sent a REST-API update — check the connector token.

CONFIG
config user saml edit "SF_SSO" set entity-id "https://root-fgt/saml" set single-sign-on-url "https://root-fgt/saml/login" set idp-cert "root-cert" next end

Wires Security Fabric SAML SSO. In a Fabric, the root FortiGate is the identity provider (IdP); downstream FortiGates, FortiAnalyzer, and FortiManager register as service providers so an admin can move between devices without re-logging in.

What to look for

Only the root can be IdP. Once SP registration is accepted, the top-bar Security Fabric dropdown lets you click through to any registered device — same browser session, no second login. Session ends when the browser session ends.

🧠
MENTAL NOTE
Fabric connector = Fortinet gear. External connector = the rest (SDN, feeds, SSO providers). Both create dynamic objects — the value is being able to reference them in normal firewall policy.

💭 THINK FIRST · AUTOMATION STITCHES

Every stitch is three things: a trigger (the event), one or more actions (the response), and the stitch that binds them and decides sequential vs parallel. All three live under config system automation-*.

  • Between the High CPU trigger and the "collect diagnostics + email" actions, why does the stitch use sequential execution with a 30-second delay?
  • Where do you set the CPU threshold that defines "high CPU" — the trigger, or somewhere else?
Q1 — MODEL ANSWER
The email action needs to reference %%results%% from the CLI-script action's output. In parallel mode both actions launch at once, so the email would go out before the diagnostics ran. Sequential + a 30-second gap gives the CLI commands time to finish before the email is composed and sent.
Q2 — MODEL ANSWER
Elsewhere — config system global; set cpu-use-threshold <n>; end. The trigger is preconfigured (High CPU) and just watches for the global threshold to be crossed. Default is 90%.
SECTION 03

Automation Stitches

Automation stitches are administrator-defined if/then workflows. Configure them on the root FortiGate and they get pushed to leaves; the root doesn't need to be up for previously-configured stitches to keep firing.

CONFIG
config system global set cpu-use-threshold 90 end

Sets the global CPU threshold that the preconfigured High CPU automation trigger watches. Any process exceeding this percentage causes FortiOS to fire the trigger.

What to look for

Default is 90. Set it lower (e.g. 75) if you want early warning during peak, higher if a well-tuned box is legitimately busy. This is the only place the threshold lives — it isn't configurable inside the trigger itself.

CONFIG
config system automation-trigger edit "High_CPU" set trigger-type event-based set event-type high-cpu next end

Defines the event-based trigger for high CPU. Other trigger types include scheduled (cron-like), fabric-connector-event (e.g. compromised host, HA failover), and incoming-webhook (external system posts JSON to FortiOS).

What to look for

Event types are preconfigured — you pick from the list, you don't invent new ones. The trigger fires as soon as the underlying condition is true. Multiple stitches can share one trigger.

CONFIG
config system automation-action edit "Collect_Diagnostics_Action" set action-type cli-script set script "get system status get system performance status diagnose sys top 5 20 2" set minimum-interval 0 next edit "Email_Diagnostics_Action" set action-type email set email-to "fgt-admins@acme.com" set email-from "fgt@acme.com" set email-subject "FGT High CPU – Diagnostics Attached" set message "%%results%%" next end

Two actions: a CLI script that gathers diagnostic output, and an email that sends the results. The %%results%% variable in the email body inherits the previous action's output when the stitch is sequential.

What to look for

%%results%% only works in sequential stitches — in parallel mode the email runs at the same time as the CLI script, and %%results%% is empty. minimum-interval lets you rate-limit — set it to 300 seconds if you don't want the trigger firing every minute during a genuine sustained-high event.

CONFIG
config system automation-stitch edit "CPU_Monitor_Stitch" set status enable set trigger "High_CPU" set action "Collect_Diagnostics_Action" "Email_Diagnostics_Action" set action-execution sequential config actions edit 1 set action "Collect_Diagnostics_Action" set delay 0 next edit 2 set action "Email_Diagnostics_Action" set delay 30 next end next end

Binds the trigger to both actions in sequence, with a 30-second delay before the email so the CLI-script output has time to complete. action-execution parallel would fire both simultaneously — useful when actions don't depend on each other.

What to look for

Delay unit is seconds. Sequential + delay is the pattern any time an action consumes the previous action's %%results%%. Parallel is cheaper but breaks that data-flow. You can also scope the stitch to a subset of FortiGates in the Fabric via set fortigates.

CONFIG
config system automation-trigger edit "Frequency" set trigger-type scheduled set trigger-frequency once set trigger-datetime 2025/03/08 09:26:10 next end config system automation-action edit "Automatic_Backup" set action-type auto-config-backup next end

Scheduled backup pattern. Frequencies: once, hourly, daily, weekly, monthly. The preconfigured auto-config-backup action writes to the Configuration > Revisions table so you can roll back to any snapshot.

What to look for

Look for a Autod backup config by stitch: <stitch-name> entry in the Revisions list — that proves the stitch ran. Time zone is the root FortiGate's local timezone. To back up the whole Fabric, target all FortiGates in the stitch scope.

CONFIG
config system automation-stitch edit "Compromised_Host_Quarantine" set trigger "Compromised_Host" set action "FortiClient_Quarantine" next end

IoC-detection-to-quarantine stitch. The preconfigured Compromised Host trigger fires when FortiAnalyzer reports an IoC; FortiClient Quarantine tells FortiClient EMS (via the fabric connector) to isolate the endpoint.

What to look for

Requires the full loop: FortiGate + FortiAnalyzer + FortiClient EMS + FortiClient on the endpoint. FortiClient sends logs to FortiAnalyzer, FortiAnalyzer flags the IoC, FortiGate tells EMS to quarantine, FortiClient enforces. Not supported on FortiClient (Linux).

🧠
MENTAL NOTE
Trigger + action + stitch. Sequential when actions share data via %%results%%; parallel when they're independent. Configure once on the root; the leaves inherit.

💭 THINK FIRST · QUARANTINE & DIAGNOSE

The endpoint quarantine flow is a chain of REST calls between FortiGate, FortiClient EMS, and the endpoint agent. When it doesn't work, you need to know which link in the chain to poke.

  • Which diagnose command manually triggers a quarantine on a specific endpoint IP, useful for testing when the automation stitch isn't firing?
  • What's the FortiOS command that shows all detected IoCs and the endpoints they came from, before you write a stitch to auto-respond?
Q1 — MODEL ANSWER
diagnose endpoint fctems queue-complete-calls Q-<endpoint_ip_address>. This queues a quarantine request to FortiClient EMS for that IP as if the automation stitch had fired — perfect for testing the FortiGate → EMS → endpoint chain end-to-end without waiting for a real IoC.
Q2 — MODEL ANSWER
diagnose user-device-store list plus diagnose endpoint fctems list. Together they show what FortiOS knows about the endpoints (from FortiClient telemetry) and what EMS is reporting back — cross-check against FortiAnalyzer's IoC list before automating.
SECTION 04

Quarantine and Diagnose

The IoC-and-quarantine path spans four devices. These commands are how you test each hop.

DIAGNOSE
# diagnose endpoint fctems queue-complete-calls Q-<endpoint_ip_address>

Manually enqueues a quarantine action against a specific endpoint IP through the FortiClient EMS connector. The endpoint's FortiClient then blocks all its network traffic and notifies FortiGate + EMS of the status change.

What to look for

Use this when the automation stitch's quarantine action doesn't seem to reach the endpoint — running it manually isolates the failure to either "FortiGate can't reach EMS" (fabric connector) or "EMS can reach the endpoint but FortiClient isn't enforcing" (agent). Not supported on FortiClient (Linux).

DIAGNOSE
# diagnose endpoint fctems list # diagnose endpoint fctems test-connectivity

Lists the FortiClient EMS servers this FortiGate is registered with and tests the connector. Use before enabling any quarantine stitch — if this fails, the stitch will silently do nothing.

What to look for

A healthy connector shows the EMS FQDN/IP, the token as authenticated, and a recent last-poll timestamp. test-connectivity failures usually mean the EMS IP config on FortiGate is wrong, the admin token expired, or a firewall between FortiGate and EMS is blocking HTTPS.

DIAGNOSE
# diagnose sys csf downstream # diagnose sys csf authorization pending-list

On the root FortiGate, lists downstream FortiGates that have completed authorisation plus the ones still waiting for approval. Use this whenever the Physical/Logical Topology view shows a device missing.

What to look for

pending-list non-empty = someone is waiting to be authorised (accept them in the GUI top-bar or via CLI). downstream empty with no pending entries = the leaf never even reached out — check group-name/password and the leaf's upstream IP for reachability.

DIAGNOSE
# diagnose test application external-resource 2 <feed-name>

Forces an immediate refresh of an external resource feed. Useful for validating that the feed URL, credentials, and parser are all working without waiting for the next scheduled interval.

What to look for

A successful fetch reports the entry count; a bad URL or auth failure prints an HTTP error code. If the count is zero, the feed file exists but no entries parsed — usually a format mismatch (line-terminated plain text is required).

🧠
MENTAL NOTE
When an automation stitch "doesn't work", isolate the chain: connector alive? (test-connectivity) → downstream present? (csf downstream) → feed populated? (test application external-resource) → endpoint reachable? (queue-complete-calls).
