IPsec, By Command

When the tunnel is up but nothing passes, live inside diagnose vpn ike and diagnose vpn tunnel.

💭 THINK FIRST · PHASE 1 INTERFACE

Phase 1 (IKE) is where the two peers agree on identity and set up the authenticated channel that phase 2 will ride on. Almost every IPsec failure resolves back to a phase-1 proposal or authentication mismatch.

  • Why does the lesson recommend IKEv2 over IKEv1 for every new IPsec deployment?
  • Under config vpn ipsec phase1-interface, which single setting controls whether the peer's identity travels in the clear or encrypted?
Q1 — MODEL ANSWER
IKEv2 negotiates in four messages instead of six or nine, supports modern ciphers (AES-GCM, ChaCha20P), stronger DH groups (14–20, ECP), EAP authentication, and integrated Mode Config. Its identities are always encrypted and it's more efficient under packet loss, making it the recommended choice unless a peer literally can't speak v2.
Q2 — MODEL ANSWER
set ike-version. IKEv1 aggressive mode sends peer IDs in cleartext; IKEv1 main mode and IKEv2 both encrypt them. That single decision — set ike-version 2 — buys you identity protection and better ciphers in one line.
SECTION 01

Phase 1 Interface

Phase 1 establishes the authenticated IKE channel between two peers. All the knobs that decide how they authenticate and which crypto they'll agree on live here.

CONFIG
config vpn ipsec phase1-interface edit "VPN1" set type dynamic set interface "port1" set ike-version 2 set proposal aes256-sha256 set psksecret <shared-secret> next end

Minimal dial-up (dynamic) phase 1 on a hub. type dynamic accepts inbound connections from any spoke, ike-version 2 switches to IKEv2, and proposal pins the crypto suite both sides must agree on.

What to look for

The proposal string must overlap with what the peer offers — mismatch here shows up in diagnose vpn ike log-filter as no proposal chosen. On IKEv1 pick between aggressive and main mode; on IKEv2 there's only one negotiation flow so the choice is gone.

CONFIG
config vpn ipsec phase1-interface edit "VPN1" set authmethod signature set certificate "MyCert" set peer-id-type peer set peer "PeerCA" next end

Switches phase 1 from pre-shared key to digital signature authentication. Loads a local certificate and pins the accepted peer certificate / CA, giving you stronger identity than a shared password.

What to look for

Use a valid CA — self-signed is fine only if both sides trust the same root. Combined with ike-version 2, digital signatures make phase 1 both encrypted and cryptographically identity-checked. This is the recommended posture for permanent site-to-site tunnels.

CONFIG
config vpn ipsec phase1-interface edit "VPN1" set dpd on-demand set dpd-retrycount 3 set dpd-retryinterval 20 next end

Configures Dead Peer Detection. on-demand only probes when there's outbound traffic and no reply, on-idle probes on a regular cadence, disable turns DPD off entirely.

What to look for

Pick on-demand for unpredictable / full-mesh traffic — no overhead until you actually need to know. Pick on-idle when you have regular traffic and want fast detection. Disable is only safe on very stable point-to-point links; in a full mesh it will hide dead peers.

CONFIG
config vpn ipsec phase1-interface edit "VPN1" set fec-egress enable set fec-ingress enable next end

Enables Forward Error Correction on the tunnel. FortiGate adds redundant packets so the receiver can reconstruct lost or errored packets without a retransmit — costs bandwidth, buys reliability.

What to look for

Turn this on only when the underlay is lossy (bad WAN, satellite) and the traffic is loss-sensitive (voice, video). It burns extra bytes even on a clean link, so it's not a default-on setting.

🧠
MENTAL NOTE
IKEv2 + AES-GCM + DH group 14+ + digital signatures is the modern default. Everything else is a compatibility compromise.

💭 THINK FIRST · PHASE 2 INTERFACE

Phase 2 (IPsec SA) negotiates the encryption for actual user traffic and defines which subnets are allowed through the tunnel. One phase 1 can carry many phase 2 selectors — each selector = one protected network pair.

  • What are the three values set route-overlap can take, and which one is the default?
  • When would you enable Perfect Forward Secrecy on phase 2, and what's the cost?
Q1 — MODEL ANSWER
use-new (default) drops the existing dialup VPN and accepts the new one. use-old keeps the existing and rejects the new. allow keeps both active and load-balances traffic via ECMP. In SD-WAN hubs with multiple egress interfaces you often want allow — otherwise interfaces will flap between active and dead as competing routes fight.
Q2 — MODEL ANSWER
Enable PFS when you need each phase 2 rekey to derive independent keys via a fresh DH exchange — so compromising one session key doesn't unlock past or future sessions. The cost is CPU: every rekey triggers a new DH computation. Disabling PFS lets rekeys reuse a simpler derivation, saving cycles but weakening the security posture.
SECTION 02

Phase 2 Interface

Phase 2 rides inside the authenticated phase 1 channel. Each phase 2 entry defines a selector pair — the local and remote subnets allowed through — plus its own crypto proposal.

CONFIG
config vpn ipsec phase2-interface edit "Phase2" set phase1name "Phase1" set proposal aes256-sha256 set src-subnet 172.16.16.0 255.255.255.0 set dst-subnet 192.168.1.0 255.255.255.0 next end

A single phase 2 selector bound to Phase1. The src-subnet / dst-subnet pair is what the two peers exchange as their quick-mode selectors and what determines which traffic gets encrypted.

What to look for

The two sides must have mirror selectors — your source is their destination and vice-versa. Selectors are matched on packet ingress; if the packet doesn't fit a selector, it doesn't get encrypted (or the tunnel refuses to establish). Multiple selectors under the same phase 1 is normal — that's how you segment traffic per subnet.

CONFIG
config vpn ipsec phase2-interface edit "Phase2" set phase1name "Phase1" set route-overlap <use-new | use-old | allow> next end

Controls what happens when a new dialup peer arrives advertising a subnet that overlaps an existing tunnel's subnet. Determines which route wins in the central FortiGate's routing table.

What to look for

Default is use-new. Switch to allow in SD-WAN hubs so both routes can coexist via ECMP — without it, interfaces can falsely toggle between active and dead when two spokes overlap. NAT is the cleaner long-term fix for genuinely overlapping remote subnets.

CONFIG
config vpn ipsec phase2-interface edit "Phase2" set phase1name "Phase1" set pfs enable set dhgrp 14 next end

Enables Perfect Forward Secrecy on phase 2 and pins the Diffie-Hellman group. Group 14 (MODP 2048-bit) is the modern minimum; 19/20 are elliptic-curve alternatives with shorter keys and less CPU.

What to look for

PFS forces a new DH exchange at every phase 2 rekey — expensive on CPU, but every rekey gets an independent key. Groups 1/2/5 are legacy and considered weak; groups 14/15/16 are MODP strong; groups 19/20 are ECP with equivalent strength at a lower cost.

🧠
MENTAL NOTE
Phase 1 = who you are. Phase 2 = what you carry. Multiple phase 2 selectors ride one phase 1 — that's how you segment tunnel traffic per subnet.

💭 THINK FIRST · TUNNEL AGGREGATE

IPsec aggregate interfaces are the tunnel-layer equivalent of LACP — you take two or more phase 1 tunnels and bind them into one virtual pipe with a selectable load-balancing algorithm.

  • Which of the five aggregate algorithms guarantees stickiness for the same 5-tuple flow?
  • Why can't you form an aggregate from a single tunnel?
Q1 — MODEL ANSWER
L4 — it hashes on Layer 4 header info (source/dest port plus IP), which yields the same tunnel choice for every packet in a given flow. round-robin and weighted-round-robin spread packets across all tunnels regardless of flow; L3 is coarser than L4; redundant only ever uses the primary until it fails.
Q2 — MODEL ANSWER
An aggregate needs at least two tunnels to serve its purpose: redundancy needs a backup member, and load balancing needs somewhere to spread packets. A single-tunnel aggregate is just a plain tunnel with extra config overhead — FortiOS requires two or more members.
SECTION 03

IPsec Tunnel Aggregate

Aggregate interfaces bind multiple phase 1 tunnels into one virtual pipe for redundancy or load-spreading. Think LACP, but for IPsec.

CONFIG
config system ipsec-aggregate edit "IPSEC_AGG" set member "Tunnel1" "Tunnel2" set algorithm weighted-round-robin next end

Creates an IPsec aggregate IPSEC_AGG spanning Tunnel1 and Tunnel2, spreading traffic in weighted round-robin (each tunnel carries a share proportional to its configured weight).

What to look for

Members must be phase 1 tunnels flagged as aggregate member in their own config. Algorithm choices: round-robin (per-packet), L3 (Layer-3 hash), L4 (Layer-4 hash — flow-sticky), redundant (primary/backup), weighted-round-robin (per-tunnel weight). Pick L4 when apps hate reordering, redundant when you want strict active/standby.

CONFIG
config vpn ipsec phase1-interface edit "Tunnel1" set aggregate-member enable set aggregate-weight 1 next end

Flags a phase 1 tunnel as an aggregate member and sets its weight (only meaningful when the aggregate uses weighted-round-robin). Higher weight = larger share of traffic.

What to look for

A tunnel that's an aggregate member no longer carries its own selectors — routing and policy point at the aggregate interface, not the individual member. Weights are relative — 3 and 1 mean the first member carries 3× the traffic of the second, not any absolute rate.

🧠
MENTAL NOTE
L4 = flow-sticky, round-robin = per-packet, redundant = active/standby, weighted = tune the share. Pick the algorithm for the traffic, not the topology.

💭 THINK FIRST · MTU & FRAGMENTATION

IPsec wraps every packet in extra headers. Whenever the encapsulated packet exceeds the underlay MTU, something has to give — you either fragment (which every intermediate device might handle differently) or you preempt the fragmentation with a smaller MSS.

  • Between set mtu and set tcp-mss-sender / tcp-mss-receiver, which one preempts the problem before packets are ever built, and where does that command live?
  • Why does set honor-df enable mean a 1600-byte packet gets dropped on a 1500-byte interface instead of fragmented?
Q1 — MODEL ANSWER
set tcp-mss-sender and tcp-mss-receiver live inside config firewall policy — they rewrite the MSS in the TCP handshake so both endpoints agree to build smaller segments up front. That prevents fragmentation from ever happening. set mtu under config system interface only tells FortiGate what it can transmit locally.
Q2 — MODEL ANSWER
With honor-df enable, FortiOS respects the sender's DF (Don't Fragment) bit. If the packet has DF set and doesn't fit the outgoing MTU, FortiGate is not allowed to fragment it, so the only legitimate action is to drop it and send back an ICMP "fragmentation needed" message. This is what PMTUD relies on.
SECTION 04

MTU and Fragmentation

IPsec ESP + IP + optional NAT-T all add bytes on top of the payload. When the sum exceeds the underlay MTU you either fragment, drop, or preempt with a smaller MSS. These are the commands that decide.

CONFIG
config system interface edit "wan2" set vdom "root" set mtu-override enable set mtu 1700 next end

Overrides the default MTU on an interface. Physical interfaces default to 1500, IPsec VPN interfaces default to 1420. Range is 68–65535 for physical and 68–1700 for IPsec.

What to look for

Only useful on internal / data-center networks where you actually control every hop and can guarantee jumbo frames. On the internet you can't push MTU higher than what the ISPs allow, so this is a data-center trick, not an internet trick. Confirm with diagnose hardware deviceinfo nic <interface>.

CONFIG
config firewall policy edit <policy id> set tcp-mss-sender <mss value> set tcp-mss-receiver <mss value> next end

Clamps the TCP maximum segment size on a firewall policy. FortiGate rewrites the MSS option in the TCP SYN so both endpoints agree to a smaller segment before the tunnel encapsulation is added.

What to look for

MSS is payload only — it excludes TCP and IP headers. A safe rule of thumb over IPsec is interface MTU − 40 (IP+TCP) − ESP overhead. Lowering MSS too far raises packet count and header overhead; setting it too high triggers fragmentation you were trying to avoid.

CONFIG
config system global set honor-df enable end

Tells FortiOS to respect the Don't Fragment bit in the outer IP header. When enabled (default), packets larger than the outgoing interface MTU that have DF set are dropped, not fragmented.

What to look for

Leave it on for correct PMTUD behaviour — hosts learn the real path MTU from the ICMP unreachables FortiGate sends back. Turning it off makes FortiGate fragment anyway, which "works" but hides the real MTU problem and can double the packet count downstream.

EXECUTE
# execute ping-options df-bit yes # execute ping-options data-size 1472 # execute ping google.com

FortiOS Path MTU Discovery. Sets the DF bit so intermediate routers can't fragment, then specifies a payload size. If the ping succeeds the path can carry that size; if it fails with sendmsg failed: Message too long the packet exceeded some link's MTU.

Sample output
PING google.com (142.251.33.78): 1472 data bytes
76 bytes from 142.251.33.78: icmp_seq=0 ttl=114 time=4.7 ms
...
PING google.com (142.251.33.78): 1500 data bytes
sendmsg failed: 90(Message too long)
What to look for

Data-size 1472 + 28 bytes IP/ICMP header = 1500 bytes = standard MTU. Data-size 1500 fails because the total 1528 exceeds MTU with DF set. Walk the size down until the ping succeeds — that's your effective path MTU. Then set MSS to PMTU − 40 in policy.

DIAGNOSE
# diagnose hardware deviceinfo nic <interface>

Prints the driver-level info for a NIC, including the currently applied MTU. Use it to confirm your set mtu actually took effect and matches what the interface is transmitting.

What to look for

Look for the Current_HWaddr and MTU lines. If the MTU shown doesn't match what you set, an ADOM override or config-sync from a manager may have re-applied a default; check where the config actually lives.

🧠
MENTAL NOTE
Interface MTU = what you can transmit locally. Firewall-policy MSS = what the two TCP endpoints agree to build. IPsec adds bytes — clamp MSS on the policy that rides the tunnel, not on the internal LAN policy.

💭 THINK FIRST · DIAGNOSE & VERIFY

When a tunnel misbehaves, don't guess. There are three views to reach for: the IKE log (for phase 1/2 negotiation issues), the tunnel list (for the runtime SA), and the config list (for what was actually loaded from the phase 2 selectors).

  • Which diagnose command scopes the IKE log to a single peer so you're not drowning in noise?
  • Where would you go to see the negotiated phase 2 selectors in memory — not what's in the config, but what the tunnel actually agreed on?
Q1 — MODEL ANSWER
diagnose vpn ike log-filter dst-addr4 <peer-ip> then enable diagnose debug application ike -1 and diagnose debug enable. Now the IKE debug is scoped to messages to/from that peer only — critical when a busy hub has hundreds of tunnels.
Q2 — MODEL ANSWER
diagnose vpn ike config list — it shows the phase 2 selectors currently loaded in the IKE process, including the selector name, proto, src/dst subnets, DH group, and any add-route flags. That's the source of truth for what got installed, versus what you typed in the config.
SECTION 05

Diagnose and Verify

Four commands cover almost every IPsec triage: diagnose vpn ike log-filter to scope, diagnose vpn tunnel list for the runtime SA, get vpn ipsec tunnel summary for the fleet view, and diagnose vpn ike config list for the loaded selectors.

DIAGNOSE
# diagnose vpn ike log-filter dst-addr4 <peer-ip> # diagnose debug application ike -1 # diagnose debug enable

Scopes the IKE debug log to a single peer and enables full-verbosity IKE debugging. Without the filter you'll see every tunnel negotiate — on a hub with hundreds of spokes that's unreadable.

What to look for

Filter options include dst-addr4, src-addr4, name (phase 1 name), and clear to reset the filter. Always diagnose debug disable and diagnose vpn ike log-filter clear when done — leaving debugs on floods the console log.

DIAGNOSE
# diagnose vpn tunnel list # diagnose vpn tunnel list name <tunnel-name>

Prints every active IPsec SA — phase 1 identity, phase 2 selectors, current SPIs, packet counters, and last error. Adding name narrows it to one tunnel.

Sample output
name=VPN1 ver=2 serial=3 100.65.0.101:0->100.65.2.112:0 dst_mtu=1500
bound_if=3 lgwy=static tun=intf mode=auto/1 encap=none/8888
proxyid_num=1 child_num=0 refcnt=13 ilast=2 olast=2 ad=/0
stat: rxp=48 txp=48 rxb=6720 txb=6720
dpd: mode=on-demand on=1 idle=20000ms retry=3 count=0 seqno=0
What to look for

rxp/txp counters incrementing = traffic is actually passing. ilast/olast = seconds since last inbound/outbound packet — big values mean idle. dpd count rising = the peer's not answering DPDs. If proxyid_num=0, phase 1 came up but phase 2 never did.

GET
# get vpn ipsec tunnel summary

Fleet-view summary of every IPsec tunnel — one line per tunnel with name, remote gateway, uptime, and status. Use it as the first check before drilling into diagnose vpn tunnel list.

What to look for

Tunnels in up state should have non-zero traffic counters. Any tunnel repeatedly cycling between up and down is a rekey or DPD problem — jump to diagnose vpn ike log-filter on that specific peer.

DIAGNOSE
# diagnose vpn ike config list

Prints the phase 2 selector definitions currently loaded in the IKE process — the runtime view of what got installed, not the config file view. Critical for debugging quick-mode selector mismatches.

Sample output
Hub # diag vpn ike config list
...
phase2s:
  Overlay_advertisement proto 0 src 0.0.0.0/0.0.0.0:0 dst 172.16.1.0 255.255.255.0:0 dhgrp 14 5 add-route
  Spoke proto 0 src 0.0.0.0/0.0.0.0:0 dst 0.0.0.0/0.0.0.0:0  dhgrp 14 5
What to look for

Phase 2 lookup is alphabetical. If you have two selectors and the more specific one must match first, name it earlier in the alphabet — that's why the study guide names it Overlay_advertisement so it precedes Spoke. If a selector isn't listed here, the config wasn't successfully loaded.

🧠
MENTAL NOTE
Filter first, then debug. Every IKE debug session should start with log-filter dst-addr4 — otherwise you're reading noise from every tunnel on the box.
