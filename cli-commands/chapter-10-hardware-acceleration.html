Hardware Acceleration, By Command

Every SPU-identification, NP/CP offload, NTurbo, IPSA, DVLAN, IPsec-offload, and header-check command from Chapter 10 — with the field to actually look at.

When the box says "slow", you'll live inside diagnose sys session list.

Config commands enable the offload; diagnose commands prove whether a given session is on the fast path or stuck in software.

💭 THINK FIRST · SPU IDENTIFICATION

Before you troubleshoot offload behaviour, ask what silicon you're actually running on. A 200E has two NP6s; a 900G has NP7s and a CP9; a 90G is a single-chip SP5. Every subsequent diagnose command changes shape based on that answer.

  • Which two CLI commands together tell you the NP version, the CP version, and (on entry-level boxes) the SoC/SP version?
  • In diagnose hardware lspci | grep 1a29, what does the hex value 4e37 after the vendor ID represent?
Q1 — MODEL ANSWER
diagnose hardware lspci | grep 1a29 shows the NP model (via the NPID hex after Fortinet's vendor ID 1a29); get hardware status shows the CP model in ASIC version (e.g. CP9) and, on entry-level FortiGates like the 91G, the SoC/SP version (SOC5). Together they give you the full SPU picture.
Q2 — MODEL ANSWER
It's the NPID — the identifier of the network processor chip. 4e36 = NP6, 4e37 = NP7. That single hex byte decides which family of diagnose npu commands actually work on the box.
SECTION 01

SPU Identification

Fortinet's specialized acceleration silicon lives in three flavours: the NP (network processor, offloads networking), the CP (content processor, offloads security scanning), and the SP/SoC (combined chip on entry-level boxes). Before doing anything else, confirm what's actually inside your FortiGate.

DIAGNOSE
# diagnose hardware lspci | grep 1a29

Lists every Fortinet PCI device and lets you read the NP model out of the NPID. 1a29 is Fortinet's PCI vendor ID; the hex value in the same line is the NPID.

Sample output
...
9b:00.0 Class 1000: Device 1a29:NPID
What to look for

The NPID after 1a29:. If it's 4e36 the FortiGate has NP6; if it's 4e37 the FortiGate has NP7. Multiple lines means multiple NPs — each one has its own np6_x or np7_x device name.

GET
# get hardware status

Prints the model name and the ASIC version — where the CP or SoC/SP model lives.

Sample output
Model name: FortiGate-900G
ASIC version: CP9
...

Model name: FortiGate-91G
ASIC version: SOC5
What to look for

ASIC version. On mid-range/high-end boxes it names the content processor (CP9, CP10). On entry-level boxes it names the SoC/SP (SOC4, SOC5 — the SP5 is also called NP7lite). Only the SP boxes will combine NP and CP into one chip.

🧠
MENTAL NOTE
NP = network offload (interface-level, IPsec crypto, IP header checks). CP = content offload (IPS pattern matching, SSL crypto). SP/SoC = both, on one chip, for entry-level. NP7 supports up to 12 million sessions per processor — over the limit, sessions fall back to the CPU.

💭 THINK FIRST · NTURBO & IPSA

Plain NP offload can't accelerate sessions that hit a security profile — the packets need to visit the IPS engine. That's what NTurbo is for. IPSA is the sidekick that pushes the pattern-matching work down onto the CP.

  • What does setting np-accel-mode none under config ips global do, and when would you use it?
  • Why is disabling np-acceleration per firewall policy a useful troubleshooting move, and what's the trade-off if you leave it disabled?
Q1 — MODEL ANSWER
It globally disables NTurbo — flow-based UTM traffic no longer gets a special NP → IPS data path and falls back to the CPU. Default is basic. You'd set it to none only for troubleshooting, e.g. to sniff or isolate a problem to the IPS engine without the NTurbo path in the way.
Q2 — MODEL ANSWER
It forces the session down the CPU-only path so you can sniff npu_vlink or debug the traffic without acceleration muddying the picture. Trade-off: all traffic through that policy is now handled in software, which costs CPU and throughput. Set it back to enable as soon as debugging is done.
SECTION 02

NTurbo and IPSA

NTurbo builds a fast path from the ingress NP straight to the IPS engine (and back) for flow-based UTM sessions. IPSA pushes the flow-based pattern matching down to the content processor. Together they turn "flow-based UTM = CPU" into "flow-based UTM = mostly hardware."

CONFIG
config ips global set np-accel-mode {none | basic} set cp-accel-mode {none | basic | advanced} end

Global knobs for NTurbo (np-accel-mode) and IPSA (cp-accel-mode). NTurbo distributes flow-based sessions across IPS engine processes; IPSA offloads pattern matching to the CP.

What to look for

Default np-accel-mode is basic — NTurbo is on. Default cp-accel-mode is advanced on boxes with more than one CP8 (offloads more pattern types) and basic on single-CP8 devices. Set either to none only for troubleshooting; leaving them off costs measurable CPU.

CONFIG
config firewall policy edit <policy_id> set np-acceleration {disable | enable} next end

Disables NP offload for a single firewall policy — the session gets processed by the CPU instead. Default is enable.

What to look for

Use this to sniff or debug traffic on a specific policy without disabling acceleration globally. Because every session going through that policy is now on the CPU, revert to enable the moment the debug session ends — else you're paying real throughput for a diagnostic.

🧠
MENTAL NOTE
NP offload = no security profile. NTurbo = flow-based security profile (NP → IPS → NP). Full CPU path = proxy-based, or mixed flow+proxy on one policy, or tunnelling (except IPsec), or a session-helper is required.

💭 THINK FIRST · VDOM LINK & DVLAN

Inter-VDOM traffic normally traverses the CPU. If both VDOMs live on interfaces served by the same NP, you can shove that traffic straight through the NP using an npu_vlink — no CPU touched. And on NP7/SP5, you can even 802.1ad-tag it.

  • What field in diagnose sys session list tells you whether a session is being NP-offloaded, and what does offload=9/9 mean?
  • Why does the DVLAN mode default to 802.1AD on NP7, and when would you switch to 802.1Q?
Q1 — MODEL ANSWER
The npu info line, specifically the offload=X/Y value. Code 9 represents NP7 — so offload=9/9 means both directions of the session are on the NP7 fast path. If the field reads no_offload with no_ofld_reason: disabled-by-policy, the session is stuck on the CPU because auto-asic-offload is disabled on that policy.
Q2 — MODEL ANSWER
Because 802.1AD (Q-in-Q) is the more general case — it supports both the provider S-Tag (ethertype 0x88a8) and the customer C-Tag (ethertype 0x8100). Switch to 802.1Q only when you have a plain single-tag L2 domain and want strict single-tag handling; disable turns DVLAN offload off entirely on that NP7 ID.
SECTION 03

VDOM Link Acceleration and DVLAN

Two related NP tricks. npu_vlink makes an inter-VDOM link that stays inside the NP. DVLAN offloading lets NP7 (and SP5/NP7lite) handle 802.1ad / Q-in-Q on the wire without punting to the CPU.

CONFIG
config system interface edit "VLANvlink1Sales" set vdom "Sales" set ip 10.10.10.10 255.255.255.252 set interface "npu0_vlink1" set vlanid 10 next end

Creates a VLAN interface on the pre-existing npu0_vlink1 to offload inter-VDOM traffic. You must configure a matching VLAN with the same VLAN ID on the paired NPU VLINK in the other VDOM (npu0_vlink0).

What to look for

interface "npu0_vlink1" is what makes this offloaded — plain inter-VDOM links go through the CPU. The vlanid must be identical on both NPU VLINKs (the two ends of the same virtual link). Assign one VLINK to each VDOM; without a firewall policy, no traffic passes.

DIAGNOSE
# diagnose sys session list

Prints every session with its NP offload state. Look for the npu info and npu_state lines.

Sample output
...
npu info: flag=0x81/0x91, offload=9/9, ips_offload=0/0, epid=128/166, ipid=166/128, vlan=0x0000/0x0000
...
# session on a policy with auto-asic-offload disabled:
npu_state=0x000001 no_offload
no_ofld_reason: disabled-by-policy
What to look for

offload=9/9 — both flow directions are on NP7 (code 9). no_offload with a no_ofld_reason line tells you why the session is stuck on the CPU. disabled-by-policy means someone set auto-asic-offload disable; other reasons include proxy-based profiles, session helpers, or tunnelled traffic.

CONFIG
config firewall policy edit <id> set auto-asic-offload {enable | disable} next end

Per-policy toggle for NP offload. Default enable; disabling forces the session onto the CPU. Useful when you need to sniff npu_vlink or vlan-over-vlink traffic that would otherwise never reach a software packet capture.

What to look for

The corresponding session in diagnose sys session list will now show no_ofld_reason: disabled-by-policy. Because this affects all traffic on that policy, treat it as a temporary diagnostic — revert to enable immediately after.

DIAGNOSE
# diagnose npu np7 dvlan-mode-list # diagnose npu np7 dvlan-mode {802.1Q | 802.1AD | disable} <dev-id> # on NP7lite / SP5: # diagnose npu np7lite dvlan-mode-list # diag npu np7lite dvlan-mode {802.1Q | 802.1AD} <dev-id>

Reads or changes the DVLAN (double-VLAN / Q-in-Q) offload mode on NP7 or NP7lite. Applies per NP7 device ID, or to all if you omit the ID.

Sample output
FG3K2F-1 # diagnose npu np7 dvlan-mode-list
Device_name    Dvlan_mode
np7_0          802.1AD-0x88a8
np7_1          802.1AD-0x88a8

FG-91G # diagnose npu np7lite dvlan-mode-list
Device_name    Dvlan_mode
np7lite_0      802.1AD
What to look for

The mode string. Default is 802.1AD (S-Tag ethertype 0x88a8 + C-Tag 0x8100) — that's Q-in-Q. Switch to 802.1Q only in single-tag environments. disable turns DVLAN offload off entirely for that NP7 ID. NP7 and SP5 also offload VXLAN, no separate config needed.

🧠
MENTAL NOTE
The two NPU VLINKs are always there — npu0_vlink0 and npu0_vlink1. Assigning one end to VDOM A and the other end to VDOM B is what turns them into an offloaded inter-VDOM path. VLAN on top of the vlink multiplies that budget.

💭 THINK FIRST · IPSEC OFFLOAD

IPsec crypto is expensive, so FortiGate offloads both encryption/decryption and Diffie-Hellman. Every tunnel exposes an npu_flag that tells you exactly what's on the NP and what's still in software.

  • You see npu_flag=03 in diagnose vpn tunnel list. What's on the NP, and what would you see if the tunnel's cipher wasn't supported?
  • The default is set ipsec-asic-offload enable. When would flipping it to disable be the right call?
Q1 — MODEL ANSWER
03 = both inbound and outbound SAs are copied to the NP — full crypto offload. If the cipher/HMAC were unsupported you'd see 20, meaning the SA can't be offloaded and the CPU does the work. 01/02 mean only outbound or only inbound is offloaded so far; 00 means the SAs are in kernel only (no traffic yet, or offload disabled).
Q2 — MODEL ANSWER
For debugging when you suspect the NP crypto path is causing the problem, e.g. one tunnel misbehaves while others work. Setting npu-offload disable on that phase1 forces its ESP through software so you can attach normal packet-level tools. Any tunnel with an unsupported algorithm will get npu_flag=20 anyway — no config change needed.
SECTION 04

IPsec Encryption & Diffie-Hellman Offload

Every supported cipher/HMAC combo is offloaded by default. Two config knobs and one diagnose flag are all you need to remember.

CONFIG
config vpn ipsec phase1-interface edit <tunnel_name> set npu-offload {enable | disable} next end

Per-tunnel NP offload for IPsec encryption/decryption. Default enable. Disabling forces the software crypto path for that tunnel only — everything else keeps hardware acceleration.

What to look for

Use disable only when troubleshooting one specific tunnel. Which algorithms actually offload depends on the NP model — a phase1 with an unsupported cipher will show npu_flag=20 even with npu-offload enable.

CONFIG
config system global set ipsec-asic-offload {enable | disable} end

Global toggle for offloading the IPsec Diffie-Hellman key exchange to the CP hardware. Default enable.

What to look for

Disable only for debugging — the DH computation then runs in software, costing CPU during phase1/rekey. This setting is global, so it affects every IPsec tunnel on the box.

DIAGNOSE
# diagnose vpn tunnel list name Hub2Spoke1 # (or: list ipsec tunnel by names in vd 0)

Prints every IPsec SA with its NP offload state in the npu_flag field.

Sample output
npu_flag=03 npu_rgwy=10.10.2.2 npu_lgwy=10.10.1.1 npu_selid=1
What to look for

The npu_flag value. 00 = both SAs still in kernel only (no traffic yet, or offload off). 01 = outbound SA copied to NP. 02 = inbound SA copied to NP. 03 = both copied to NP (fully offloaded). 20 = unsupported cipher/HMAC, IPsec SA cannot be offloaded.

🧠
MENTAL NOTE
npu_flag is the one-glance tunnel health check. 03 = good. 00 = no traffic yet (fine) or offload disabled (fix). 20 = cipher isn't supported by this NP — change the algorithm or accept CPU cost.

💭 THINK FIRST · HEADER CHECK & GLOBAL

One global switch — check-protocol-header strict — tightens header checking and, as a side effect, disables all hardware acceleration on NPs, CPs, and SPs. Powerful for debugging, expensive as a permanent setting.

  • What's the practical difference between the default loose mode and strict, beyond the header-field list?
  • You've got a policy with mixed flow-based and proxy-based security profiles. Will the NP offload it? Why or why not?
Q1 — MODEL ANSWER
In loose, FortiGate does basic header checks (L4 length, IP length/version/checksum/options) with NP assistance. In strict, FortiGate additionally checks ESP sequence number, SPI, and data length — and, because those need software inspection, the setting disables all hardware acceleration globally. So the practical difference is: strict is a CPU-only mode. Use it briefly for debugging, then revert.
Q2 — MODEL ANSWER
No — a firewall policy with mixed flow-based and proxy-based profiles is never offloaded. The proxy path needs the CPU end-to-end, so the whole session is processed in software. If you want NP acceleration, split the profiles: pure flow-based policies get NP (via NTurbo when needed); pure proxy-based policies stay on the CPU by design.
SECTION 05

Strict Header and Global Best Practices

Two related knobs that decide how paranoid FortiGate is about packet headers — and, indirectly, whether hardware acceleration stays on at all.

CONFIG
config system global set check-protocol-header {loose | strict} end

Tightens packet-header validation. loose (default) checks L4 header length, IP header length, IP version, IP checksum, and IP options with NP assistance. strict additionally checks ESP correct sequence number, SPI, and data length — and disables all hardware acceleration on NPs, CPs, and SPs.

What to look for

Failing header check = dropped packet. Setting strict is a heavy troubleshooting hammer: it forces the whole box into software processing so you can inspect traffic that would normally never touch the CPU. Revert to loose the moment you're done — throughput collapses under strict.

🧠
MENTAL NOTE
Reasons a session won't offload even when NP is available: (1) auto-asic-offload disabled on the policy, (2) the policy has any proxy-based profile, (3) the session requires a FortiOS session helper, (4) the session is inside a tunnel other than IPsec (SSL VPN, GRE). Memorise the four — every session-list debug points to one of them.
