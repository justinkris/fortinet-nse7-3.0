Chapter 03 · CLI Reference

VLANs and VDOMs, By Command

Every command from Chapter 3 that carves the box into logical pieces — VLAN sub-interfaces, hardware and software switches, VDOM enable/type/mode, and the inter-VDOM link that stitches them back together.

Segmenting the box begins with set vdom-mode multi-vdom and ends with an inter-VDOM link.

VLANs partition L2. VDOMs partition L3 and above — a separate routing table, policy set, and admin space per virtual firewall. Neither works right until the CLI reflects the intent exactly.

💭 THINK FIRST · VLAN SUB-INTERFACES

A FortiGate VLAN interface is a child of a parent physical port, tagged with a specific VLAN ID. The parent stays the trunk; the child is where you put the IP and the policies apply.

  • Which two knobs inside config system interface — interface and vlanid — turn a plain port into a dot1q sub-interface?
  • Why can you reuse the same VLAN ID on different parent interfaces and end up with completely different broadcast domains?
Q1 — MODEL ANSWER
set interface <parent-port> nominates the physical port that carries the tagged trunk; set vlanid <id> tells FortiOS which 802.1Q tag to match. Together they turn the child interface into a VLAN sub-interface tied to that parent.
Q2 — MODEL ANSWER
Because a VLAN sub-interface is scoped to its parent — port1.vlan5 and port2.vlan5 live on different L2 segments, since the tag is only meaningful inside its parent port's link. Different parent + same VLAN ID = totally different broadcast domain.
SECTION 01

VLAN Sub-Interfaces

All VLAN sub-interfaces on FortiGate live under config system interface. The parent physical port becomes the dot1q trunk automatically the moment you create the first VLAN child on it.

CONFIG
config system interface edit "VLAN_5" set vdom "root" set ip 192.168.5.254 255.255.255.0 set allowaccess ping https ssh set type vlan set interface "internal1" set vlanid 5 next end

Canonical VLAN sub-interface. Type vlan, parent port internal1, dot1q tag 5. IP and admin access sit on the sub-interface — the parent stays untagged.

What to look for

set interface = parent port, set vlanid = tag. Both are mandatory. VLAN IDs are per-parent — you can have VLAN_5 on internal1 and a totally separate VLAN_5 on internal4. The VDOM assignment is per interface: a VLAN sub-interface can live in a different VDOM from its parent.

CONFIG
config system interface edit "VLAN_5_QinQ" set type vlan set interface "internal1" set vlanid 5 set vlan-protocol 8021ad next end

Switches the VLAN protocol from the default 802.1Q (0x8100) to 802.1ad / QinQ (0x88A8). Used when the FortiGate is acting as an S-tag boundary in a service-provider setup.

What to look for

802.1Q supports 4,094 VLANs. 802.1ad (QinQ) stacks two tags (S-tag + C-tag) for up to 16 million combinations. Only turn this on if the upstream device is actually double-tagging — mismatched vlan-protocol on either side = frames dropped silently.

CONFIG
config system interface edit "Aggregate" set type aggregate set member "internal4" "internal5" set lacp-mode active next end config system interface edit "Vlan_5" set type vlan set interface "Aggregate" set vlanid 5 set ip 10.5.0.254 255.255.255.0 next end

LACP (802.3ad) aggregate as a parent, VLAN child on top. The aggregate bundles multiple physical ports for bandwidth + redundancy; VLAN sub-interfaces then ride the bundle as if it were a single port.

What to look for

Recommended sizes: 2, 4, or 8 physical ports per aggregate. VLAN children reference the aggregate by name, not by a physical port. This is the standard shape for connecting a FortiGate to a switch pair with a dot1q trunk on top of an LACP.

CONFIG
config system interface edit "VXLAN_1000" set type vxlan set vxlan-id 1000 set remote-ip 10.0.0.2 set interface "port2" next end

VXLAN tunnel interface. Encapsulates L2 frames in UDP (default port 4789) for tunnelling across an IP network. 24-bit VNI supports up to 16 million virtual networks — where 802.1Q hits the 4,094-VLAN ceiling.

What to look for

set remote-ip is the far end of the VXLAN tunnel. set interface is the local underlay port. Used in data-centre / cloud environments and inside the LAN Extension VDOM (last section) to stretch a LAN across sites.

🧠
MENTAL NOTE
VLAN sub-interfaces = parent port + vlanid pair. Parent can be a physical port, an aggregate, a hardware switch, or (rarely) a software switch. VLAN IDs are only unique within a parent.

💭 THINK FIRST · HARDWARE & SOFTWARE SWITCHES

FortiGate normally treats each port as its own broadcast domain. If you want multiple ports to share one L2 segment, you need a switch — hardware (offloaded) or software (CPU-processed).

  • What's the CLI difference between config system virtual-switch and config system switch-interface?
  • Why is a software switch the only option if you want to bridge wireless SSIDs and physical ports into the same broadcast domain?
Q1 — MODEL ANSWER
Hardware switch uses config system virtual-switch with a nested config port / edit <port> sub-block. Software switch uses config system switch-interface with a flat set member "port1" "port2" line. Hardware = offloaded to the switching ASIC (specific models only). Software = every frame handled in CPU (any model).
Q2 — MODEL ANSWER
Hardware and VLAN switches don't support wireless SSIDs as members. Only software switches do — because bridging an SSID into an L2 segment is a software operation the CPU has to mediate. That's the whole reason to accept the CPU cost of a software switch.
SECTION 02

Hardware and Software Switches

Two ways to bridge multiple ports into a single L2 segment on FortiGate. Hardware is offloaded; software runs in CPU but supports wireless SSIDs and can enforce inter-member policies.

CONFIG
config system virtual-switch edit "Hardware Switch" set physical-switch "sw0" config port edit port1 next edit port2 next edit port3 next end next end

Hardware switch — offloaded to the underlying switching silicon. Ports listed here are hardware-bridged; intraswitch traffic doesn't touch the CPU.

What to look for

set physical-switch "sw0" nominates the underlying switch ASIC. config port / edit <port> is the sub-block that lists members. Only supported on specific models with an internal switch fabric. Supports STP; does not support wireless SSIDs as members.

CONFIG
config system interface edit "Hardware Switch" set vdom "root" set ip 10.10.10.254 255.255.255.0 set allowaccess ping https next end

The L3 side of the hardware switch. Once the virtual-switch object exists, an interface with the same name becomes the SVI — IP address and admin access live here, exactly like a plain physical port.

What to look for

The name has to match the virtual-switch name exactly. Devices connected to any member port share this broadcast domain and use this IP as their default gateway. Intraswitch traffic is allowed by default.

CONFIG
config system switch-interface edit "Software Switch" set vdom "root" set member "port1" "SSID" "port3" set intra-switch-policy implicit next end

Software switch — bridges the listed members in the CPU. Can mix physical ports and FortiGate wireless SSIDs in the same L2 segment. Available on every FortiGate model.

What to look for

intra-switch-policy implicit allows traffic between members without policy inspection (default). explicit forces you to write firewall policies for member-to-member traffic — useful when you want UTM inspection between wired and wireless clients. Software switches do not support STP.

CONFIG
config system interface edit "Software Switch" set vdom "root" set ip 10.10.10.254 255.255.255.0 set allowaccess ping https next end

L3 side of the software switch. Same shape as the hardware-switch SVI: interface name matches the switch-interface object name; IP address and admin access live on the interface.

What to look for

Because software switches process every frame in CPU, they cost more per-Mbps than hardware switches. Use them for wireless bridging and small port counts; use hardware switches when the model supports them and you don't need SSID membership.

🧠
MENTAL NOTE
Hardware switch = offloaded, STP-capable, no SSIDs, specific models only. Software switch = CPU-processed, no STP, supports SSIDs, any model. Both share one broadcast domain across their members.

💭 THINK FIRST · ENABLE & TYPE VDOMS

A single FortiGate becomes many virtual firewalls when you flip vdom-mode multi-vdom. From then on, every command lives inside either config global or edit <vdom> — the two contexts are strictly separate.

  • Which single command enables multi-VDOM mode, and which two VDOMs are automatically created?
  • What's the difference between set vdom-type admin, traffic, and lan-extension, and which one only ever has one instance per FortiGate?
Q1 — MODEL ANSWER
config system global / set vdom-mode multi-vdom / end. That creates the Global VDOM (super-admin only) and the root VDOM. Neither can be deleted.
Q2 — MODEL ANSWER
admin = management-only, no user traffic (Dashboard, Network, System, Security Fabric, Log & Report modules only). traffic = a regular firewall (NAT or Transparent operation mode). lan-extension = a VDOM dedicated to extending a LAN over VPN IPsec + VXLAN. Admin type is limited to one instance per FortiGate; traffic and lan-extension VDOMs go up to 10 by default (more with licensing).
SECTION 03

Enable and Type VDOMs

Three commands do the whole VDOM setup: enable multi-VDOM mode, create a VDOM, choose the VDOM's type (admin, traffic, lan-extension) and — for traffic VDOMs — pick NAT or transparent operation mode.

CONFIG
config system global set vdom-mode multi-vdom end

Enables multi-VDOM mode on the FortiGate. Automatically creates the Global VDOM (super-admin only) and the root VDOM. Neither can be deleted.

What to look for

After this runs, every subsequent config command has to be scoped: either config global for box-wide settings, or edit <vdom> for VDOM-local ones. Confirm with get system status — the Virtual domain configuration line should now read multiple.

CONFIG
config vdom edit "VDOM1" config system settings set vdom-type admin end end

Creates VDOM1 as an admin VDOM — management only, no user traffic. Has only Dashboard, Network, System, Security Fabric, and Log & Report modules. Used for FortiGuard updates and licence validation.

What to look for

Only one admin VDOM per FortiGate. Changing an existing VDOM from traffic to admin deletes its firewall policies, security profiles, wifi controllers, user & auth objects — the GUI prompts for confirmation before proceeding.

CONFIG
config vdom edit "VDOM1" config system settings set vdom-type traffic set opmode nat end end

Traffic VDOM in NAT mode — the most common VDOM. Behaves like a regular firewall: interfaces have IPs, routing table, firewall policies, NAT, UTM inspection.

What to look for

opmode must be set via CLI — the GUI's Create New form doesn't expose it directly on modern versions. Up to 10 traffic VDOMs by default; more with a licence.

CONFIG
config vdom edit "VDOM2" config system settings set vdom-type traffic set opmode transparent set manageip 172.16.16.1/255.255.255.0 end end

Traffic VDOM in transparent mode — L2 firewall with UTM inspection but no NAT, no routing, no per-interface IPs. Sits invisibly in the L2 path.

What to look for

manageip is required — without it, FortiOS throws "failed node check object for 'manageip'". This is the only IP a transparent VDOM has, and it's just for admin access. All interfaces in a transparent VDOM share one broadcast domain unless you subdivide with forward-domain.

CONFIG
config vdom edit "VDOM3" config system settings set vdom-type lan-extension end end

LAN Extension VDOM — dedicated to stretching a LAN over VPN IPsec + VXLAN, with FortiExtender / FortiGate connectors on one side and a FortiGate access controller on the other. Only Network and VPN modules are available.

What to look for

Purpose-built for extending a local subnet across sites — the FortiExtender phones home over IPsec and the LAN traffic rides VXLAN inside the tunnel. Cannot be used as a general firewall.

🧠
MENTAL NOTE
Multi-VDOM = set vdom-mode multi-vdom. Types: admin (1 per FortiGate, mgmt only), traffic (NAT or Transparent, up to 10 default), lan-extension (VPN + VXLAN stretch). Transparent traffic VDOMs must have a manageip.

💭 THINK FIRST · VDOM CONTEXT & GLOBAL

Once multi-VDOM is on, every command is either global or VDOM-scoped. Learning to move between the two contexts fluently is what separates smooth VDOM operations from broken installs.

  • Inside config global vs. edit <vdom>, which context owns interface creation, and which owns firewall policies?
  • Why is diagnose sys vd list the fastest way to figure out which VDOM your CLI session is currently in?
Q1 — MODEL ANSWER
Interfaces are created in config global — the physical hardware is a box-wide resource. Assigning an interface to a VDOM (set vdom) also happens in global. Firewall policies, routing, security profiles, UTM — all live inside edit <vdom>. Global-object security profiles (g-default) are an exception.
Q2 — MODEL ANSWER
diagnose sys vd list prints every VDOM plus which one your current CLI session is attached to. Handy when you lose track after a bunch of config global / edit <vdom> transitions.
SECTION 04

VDOM Context and Global

Every command in multi-VDOM mode has a scope: global (box-wide, super-admin only) or VDOM-local. Getting the scope wrong is the most common source of "I typed it and nothing happened" bugs.

CONFIG
config global ... end config vdom edit "VDOM1" ... next end

The two scope roots. config global holds box-wide settings — physical interfaces, HA, FortiGuard, admin accounts. config vdom / edit <vdom> enters a specific VDOM's config space.

What to look for

Interface creation and set vdom always happen inside config global. Policies, routing, security profiles, VPN — all inside a VDOM. The prompt tells you where you are: FGT (global) # vs. FGT (VDOM1) #.

EXECUTE
# config global # ... # end # config vdom # edit VDOM1 # ... # end

The alternate "temporary switch" pattern. Instead of nesting an interactive scope with config vdom / edit, some operators just type this shell-style to hop between VDOMs during troubleshooting.

What to look for

Always end a global block before starting a VDOM block. Nested is fine, but half-open blocks (leaving config global without an end) confuse the CLI parser and produce cryptic errors.

DIAGNOSE
# diagnose sys vd list

Prints every VDOM defined on the box, plus flags for admin/traffic/lan-extension type and which one your CLI session is currently attached to.

What to look for

Look for the "current-vdom" marker to confirm which VDOM you're editing right now. Also useful during multi-VDOM troubleshooting to confirm all expected VDOMs are actually created (a missed next during creation can leave a partial one).

CONFIG
config global config system interface edit "port2" set vdom "VDOM1" next end end

Reassigns a physical port from one VDOM to another. Must be run from config global — VDOM-scoped edit of an interface can't change the interface's VDOM assignment.

What to look for

Before moving an interface, remove all references to it from the source VDOM: policies, static routes, VIPs, IP pools, DHCP servers. FortiOS refuses the move if references remain — and the error message tells you which reference is blocking.

🧠
MENTAL NOTE
Global scope = box-wide. VDOM scope = firewall-instance-local. Interfaces are created global and assigned to a VDOM. Policies, routing, and UTM live inside the VDOM. Look at the prompt to know where you are.

💭 THINK FIRST · INTER-VDOM & TRANSPARENT

VDOMs are supposed to be isolated — but a VDOM without internet is useless. The inter-VDOM link is a pair of virtual back-to-back interfaces that lets traffic cross between two VDOMs without leaving the box.

  • How does FortiOS name the two ends of a VDOM link, and which side of the link lives in each VDOM?
  • In a transparent VDOM, what does set forward-domain do, and when do you need to change it from the default?
Q1 — MODEL ANSWER
FortiOS appends 0 and 1 to the link name — VDOM_Link0 and VDOM_Link1. You assign each side to a different VDOM (side 0 in root, side 1 in VDOM1, for example). To actually pass traffic, both sides need IPs, static or dynamic routes must point across the link, and firewall policies on each VDOM must permit the direction.
Q2 — MODEL ANSWER
All interfaces in a transparent VDOM share one broadcast domain by default. set forward-domain <id> subdivides the VDOM into multiple broadcast domains — using the VLAN ID as the domain ID is the convention. Only interfaces sharing a forward-domain ID exchange traffic in transparent mode.
SECTION 05

Inter-VDOM and Transparent Mode

Two shapes for connecting VDOMs. Inter-VDOM link stitches routed VDOMs together on the same box. Transparent-mode forward-domain subdivides a transparent VDOM's single broadcast domain.

CONFIG
config global config system interface edit "VDOM_Link0" set vdom "root" set ip 172.16.1.213/30 set type vdom-link set allowaccess ping next edit "VDOM_Link1" set vdom "VDOM1" set ip 172.16.1.214/30 set type vdom-link set allowaccess ping next end end

The two ends of an inter-VDOM link. Each side lives in a different VDOM; the pair shares a small transit subnet (a /30 is typical). Once IPs are on, routes and policies decide whether traffic actually flows.

What to look for

FortiOS names the two ends by appending 0 and 1 to the parent VDOM_Link name — this is convention, not something you set. Static or dynamic routes on each VDOM must point across the link, and firewall policies on each side must permit the direction. Ping alone won't work without both.

CONFIG
config vdom edit "VDOM1" config router static edit 1 set device "VDOM_Link1" set gateway 172.16.1.213 next end end

Default route from VDOM1 across the inter-VDOM link to root VDOM. Combined with a firewall policy on root that allows VDOM1's subnet outbound, this gives VDOM1 internet access via root.

What to look for

The gateway is the far end of the link (root's side). The device is the local end (VDOM1's side). Symmetric routing back from root to VDOM1's networks is also needed — otherwise return traffic gets dropped.

CONFIG
config system interface edit <interface> set forward-domain <domain_ID> next end

Subdivides a transparent VDOM's single broadcast domain. Interfaces with the same forward-domain ID exchange traffic; different IDs are isolated at L2.

What to look for

The convention is to use the VLAN ID as the forward-domain ID — makes it obvious which L2 domain each interface belongs to. Only relevant in transparent VDOMs; ignored in NAT mode. Without this, every interface in a transparent VDOM is in the same broadcast domain, which is rarely what you want.

CONFIG
config global config system global set management-vdom "VDOM1" end end

Changes which VDOM sources FortiGuard updates, licence validation, and other box-wide outbound traffic. Default is root.

What to look for

The management VDOM must have internet reachability — or reach a FortiManager acting as an FDS (Ch02). If a downstream VDOM has internet but the management VDOM doesn't, every VDOM experiences stale signatures. GUI equivalent: Global VDOM > System > VDOM > Switch Management.

DIAGNOSE
# diagnose sys vd stats

Prints per-VDOM resource statistics — session counts, policy counts, current usage against limits. Useful when a VDOM starts failing and you suspect it's hit a per-VDOM resource cap.

What to look for

Watch for VDOMs at or near their Override Maximum for sessions or policies. Per-VDOM caps are set at VDOM Global > System > VDOM > Edit VDOM. The override max must be higher than or equal to current usage, or the change is rejected.

🧠
MENTAL NOTE
Inter-VDOM link = two ends (_Link0 and _Link1), one in each VDOM. Both need IPs, routes, and policies. Transparent VDOMs subdivide their single broadcast domain with forward-domain — use the VLAN ID as the domain ID by convention.
