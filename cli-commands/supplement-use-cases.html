Use Cases, By Command

Every CLI knob from the three lab exercises — VDOM-link segmentation with OSPF, ADVPN hub-and-spoke with IBGP, and the Security Fabric automation stitch that quietly backs up your config.

Three exercises, one lesson — defaults decide most of the outcome.

Every stanza on this page has a default you need to know: OSPF area type, ADVPN discovery direction, BGP next-hop-self on the hub, and the automation trigger that fires the backup.

💭 THINK FIRST · SEGMENTATION & OSPF

Exercise 1 puts an HR VLAN in a separate VDOM (Zone2), then bridges Zone2 back to root over an inter-VDOM link, then puts OSPF area 0.0.0.1 across that link. Everything rides on the VDOM link being the only path between the two OSPF speakers.

  • Why is area 0.0.0.1 configured as a stub in this exercise, and what does that mean for the LSAs Zone2 receives?
  • Once the OSPF link is up, which single command on HQ-DCFW proves the 10.0.3.0/24 HR subnet actually arrived?
Q1 — MODEL ANSWER
Because Zone2 only needs a summary view of the world, not every external prefix. Stub areas don't receive Type 5 (external) LSAs — the ABR replaces them with a default route. That keeps Zone2's LSDB small and means changes outside area 1 don't cause SPF churn inside HR.
Q2 — MODEL ANSWER
get router info routing-table all on HQ-DCFW. Look for the line starting with O IA 10.0.3.0/24 — the IA code means it came via inter-area (from area 0.0.0.1 via the ABR into area 0). If the code is missing, either the adjacency didn't reach Full or the ABR isn't leaking a summary Type 3.
SECTION 01

Segmentation with VDOM Links + OSPF

Two-VDOM design: root holds the fabric-facing interfaces, Zone2 holds the HR VLAN. A VDOM link joins them. OSPF area 0.0.0.1 runs across the link.

CONFIG
config system interface edit "VLAN102" set vdom "Zone2" set vlanid 102 set interface "port5" set ip 10.0.3.254 255.255.255.0 set allowaccess ping https next end

Creates the HR VLAN (VLAN 102) on port5 inside the Zone2 VDOM with the gateway address 10.0.3.254/24. PING and HTTPS allowaccess so HR endpoints can reach the FortiGate for policy hits and management if needed.

What to look for

set vdom "Zone2" — the VLAN interface itself has to live in the target VDOM, not root. VLAN ID must be unique per physical parent. Change allowaccess to a stricter list if HR shouldn't manage the box.

CONFIG
config global config system vdom-link edit "C" next end end config system interface edit "C0" set vdom "root" set ip 172.16.1.249 255.255.255.252 set allowaccess ping set type vdom-link next edit "C1" set vdom "Zone2" set ip 172.16.1.250 255.255.255.252 set allowaccess ping set type vdom-link next end

Creates VDOM link C with two ends — C0 in root, C1 in Zone2 — on a shared /30. This is the L3 pipe OSPF will neighbour across.

What to look for

Vdom-link is not offloaded (that's npu_vlink — see the Hardware Acceleration page). This is a plain kernel-forwarded link, fine for routing-protocol traffic. Each end lives in a different VDOM; the /30 is only used inside the FortiGate. allowaccess ping lets you troubleshoot with execute ping.

CONFIG
# in Zone2: config router ospf set router-id 0.0.0.4 config area edit 0.0.0.1 set type stub next end config network edit 1 set prefix 172.16.1.248 255.255.255.252 set area 0.0.0.1 next edit 2 set prefix 10.0.3.0 255.255.255.0 set area 0.0.0.1 next end end

Enables OSPF on Zone2 with router ID 0.0.0.4, defines area 0.0.0.1 as stub, and advertises the vdom-link transit subnet plus the HR subnet into that area. The root VDOM does the mirror config with router ID 0.0.0.3 and the transit prefix in area 0.0.0.1 (the root VDOM is the ABR into area 0).

What to look for

Stub area means no Type 5 (external) LSAs enter — the ABR generates a default route into the stub instead. Both sides must agree the area is stub or the adjacency won't reach Full. Router IDs must be unique across the whole OSPF domain, not just per VDOM.

GET
# get router info ospf status # get router info routing-table all

Verifies the adjacency (ospf status) and confirms Zone2's HR subnet made it into HQ-DCFW's RIB via inter-area routing.

What to look for

In get router info ospf status: "Number of fully adjacent neighbors in this area is 1". In the routing table: O IA 10.0.3.0/24 [110/2] via … — the IA code confirms it's an inter-area Type 3 summary; the 110 is OSPF's default administrative distance.

EXECUTE
# execute ping 10.0.3.254 # execute ping 10.0.5.11

Data-plane sanity: HR PC pings its VLAN gateway (10.0.3.254) and the internal web server (10.0.5.11). If ping to the gateway works but ping across to HQ-Web-1 doesn't, routing or policy is the issue.

What to look for

Gateway-only success → firewall policy is dropping the transit. Full success → OSPF injected the route and policy accepts the flow. Failing gateway ping from HR → VLAN ID mismatch or wrong VDOM binding on VLAN102.

🧠
MENTAL NOTE
Segmentation = new VDOM + vdom-link + OSPF area. The vdom-link carries the routing protocol. The stub area declaration must match on both ends. The ABR (root VDOM) is what makes area 1's subnets visible to area 0.

💭 THINK FIRST · ADVPN HUB & SPOKE

Classic hub-and-spoke means spoke-to-spoke traffic tromboned through the hub. ADVPN fixes that: the hub advertises shortcut information so spokes can build on-demand direct tunnels to each other. Two flags — auto-discovery-sender on the hub, auto-discovery-receiver on the spokes — control the whole thing.

  • Why does the hub set auto-discovery-sender enable but the spokes set auto-discovery-receiver enable? What happens if you swap them?
  • Why is Mode Config disabled on both sides of the overlay, and why does the exercise use static overlay IPs (172.16.1.1/32 hub, 172.16.1.2/32 spoke) instead?
Q1 — MODEL ANSWER
The hub is the one that knows both spokes and can tell one about the other, so it's the sender. Each spoke needs to receive that shortcut info to know it should build a direct tunnel to a peer instead of routing through the hub. Swap them and no shortcut information flows — you're stuck with normal hub-and-spoke.
Q2 — MODEL ANSWER
Mode Config is for dial-up VPN where the hub hands out addresses to unknown spokes. ADVPN with BGP over the overlay wants deterministic tunnel-interface IPs so BGP can peer stably — hence static /32s. Metadata variables ($(Tunnel_IP), $(Router_ID)) let one FortiManager template render per-device values without repeating the config.
SECTION 02

ADVPN Hub & Spoke Overlay

Same IPsec phase1-interface CLI as any hub-and-spoke, plus two auto-discovery knobs and a couple of transport tweaks to keep the shortcut plumbing happy.

CONFIG
# on the hub (HQ-NGFW / Core1 VDOM): config vpn ipsec phase1-interface edit "HUB1-VPN1" set interface "port2" set mode-cfg disable set transport udp set auto-discovery-sender enable set localid "fortinet" set psksecret 123456789 next end config system interface edit "HUB1-VPN1" set ip 172.16.1.1 255.255.255.255 set remote-ip 172.16.1.254 255.255.255.0 next end

Hub-side phase1 with ADVPN sender enabled. mode-cfg disable because we're using static overlay IPs. transport udp is the default; explicit here so troubleshooting is unambiguous. Tunnel interface HUB1-VPN1 gets the overlay IP 172.16.1.1/32 and treats the 172.16.1.254/24 as its remote-side.

What to look for

auto-discovery-sender enable is the ADVPN switch on the hub — without it, the hub is just a regular hub-and-spoke concentrator. Local ID fortinet must match the set peer-id fortinet or equivalent identifier the spokes use to authenticate.

CONFIG
# on each spoke (BR2-FGT-1, BR3-FGT-1): config vpn ipsec phase1-interface edit "HUB1-VPN1" set interface "port2" set mode-cfg disable set transport udp set auto-discovery-receiver enable set remote-gw 100.65.0.101 set localid "fortinet" set psksecret 123456789 next end config system interface edit "HUB1-VPN1" set ip $(Tunnel_IP) # e.g. 172.16.1.2/32 set remote-ip 172.16.1.1 255.255.255.0 next end

Spoke-side phase1 with ADVPN receiver enabled and the hub's public IP as remote-gw. The tunnel-interface IP comes from a FortiManager metadata variable so BR2 gets 172.16.1.2/32, BR3 gets 172.16.1.3/32 — from the same template.

What to look for

auto-discovery-receiver enable is what lets the spoke listen for shortcut messages from the hub and build direct tunnels on demand. If a spoke never gets a shortcut, it keeps sending spoke-to-spoke traffic through the hub — check that auto-discovery-sender is on at the hub end.

EXECUTE
# on BR2-FGT-1: # execute ping-options source 172.20.2.254 # execute ping 172.20.3.254

Data-plane test that spoke-to-spoke works. Sourced from BR2's LAN interface so the ping actually rides the overlay (not the WAN). Target is BR3's LAN gateway.

What to look for

First few packets may show latency ≈ hub path RTT (traffic tromboning through the hub). Once ADVPN kicks in, latency should drop — you're on a direct BR2↔BR3 tunnel. Look for a new tunnel appearing in diagnose vpn tunnel list on the spoke after the first ping.

DIAGNOSE
# diagnose vpn tunnel list # (or on Core1: diagnose vpn tunnel list in vd 2)

Shows every phase1 and its child phase2 SAs, including nexthop, encryption/decryption counters, and the npu_flag offload state (see the Hardware Acceleration reference).

What to look for

Two tunnels expected on each spoke after a successful ADVPN shortcut: the always-on tunnel to the hub, and a dynamically-created tunnel to the other spoke. npu_flag=03 on both = fully hardware-accelerated crypto. bytes/pkts counters growing on the shortcut tunnel = ADVPN is actually being used.

🧠
MENTAL NOTE
ADVPN = sender on the hub, receiver on the spokes, static overlay IPs so BGP has stable next-hops. First inter-spoke packet still goes via the hub; then a shortcut tunnel is built and subsequent packets go direct.

💭 THINK FIRST · IBGP WITH ROUTE REFLECTOR

IBGP has a "don't propagate what you learned from an IBGP peer" split-horizon rule. The classic workaround is full mesh — but that's N² and doesn't scale. Route reflectors punch a hole in the rule: the RR (the hub) is allowed to re-advertise IBGP prefixes to its clients (the spokes).

  • Why does the hub set both route-reflector-client enable AND next-hop-self enable on the neighbor group?
  • What does neighbor-range with max-neighbor-num 2 get you that a plain neighbor stanza doesn't?
Q1 — MODEL ANSWER
Route-reflector-client turns split horizon off for that peer, so the hub will re-advertise prefixes it learned from spoke A to spoke B. Next-hop-self rewrites the BGP next-hop to be the hub's own overlay IP, so spoke B's IGP/tunnel-routing can actually reach the next-hop. Without next-hop-self, spoke B would see a next-hop pointing to spoke A's overlay IP, which it might not have a route to yet (before ADVPN kicks in).
Q2 — MODEL ANSWER
Dynamic acceptance: any spoke whose overlay IP falls inside the range 172.16.1.0/24 can peer without a per-neighbour stanza on the hub. That's how you onboard spokes at scale — provision the spoke, it dials in, the hub accepts. max-neighbor-num 2 caps the acceptance at two peers (matching the two branch offices in this exercise).
SECTION 03

IBGP with Route Reflector

Same AS number (65100) everywhere — that's IBGP. Hub is the route reflector; spokes are the clients. Neighbor groups + neighbor ranges keep the hub config short.

CONFIG
# on the hub (HQ-NGFW / Core1 VDOM): config router bgp set as 65100 set router-id 172.16.1.1 config neighbor-group edit "advpn" set remote-as 65100 set route-reflector-client enable set next-hop-self enable next end config neighbor-range edit 1 set prefix 172.16.1.0 255.255.255.0 set neighbor-group "advpn" set max-neighbor-num 2 next end config network edit 1 set prefix 10.0.12.0 255.255.255.0 next end end

Hub BGP: AS 65100, router-id 172.16.1.1 (the overlay IP), a neighbor-group advpn where every member is treated as an RR client with next-hop-self, and a dynamic neighbor-range that auto-accepts up to 2 peers from the overlay /24. The hub also originates its LAN subnet.

What to look for

Same-AS on both sides = IBGP (split horizon in effect). route-reflector-client enable lifts split horizon for those peers. next-hop-self enable is essential when the RR's clients don't share L3 reachability to each other's overlay IPs — which is exactly the pre-ADVPN state.

CONFIG
# on each spoke (BR2-FGT-1, BR3-FGT-1): config router bgp set as 65100 set router-id $(Router_ID) config neighbor edit "172.16.1.1" set remote-as 65100 set connect-timer 120 next end config network edit 1 set prefix $(LAN_BR) next end end

Spoke BGP: same AS, router-id from a metadata variable (per-device), one static neighbor pointing at the hub's overlay IP, and one network stanza with the local LAN subnet (also per-device). No route-reflector-client on the spoke — that's a hub-side concept.

What to look for

connect-timer 120 speeds up initial peering (default is 120s already — explicit here for consistency). Because the spoke doesn't know the other spoke's overlay IP directly, it relies on the hub's RR + next-hop-self to learn other LAN prefixes. Once BGP converges, the FIB has the other spoke's LAN reachable through the hub; ADVPN then rebuilds the data path direct.

GET
# get router info bgp summary # get router info routing-table all

Verifies BGP peer state and BGP-learned routes. On BR3, look for BR2's LAN prefix marked with a B code.

Sample output
Codes: K - kernel, C - connected, S - static, R - RIP, B - BGP
       O - OSPF, IA - OSPF inter area

S*    0.0.0.0/0 [10/0] via 100.65.5.35d, port2
C     172.16.1.1/32 is directly connected, HUB1-VPN1
C     172.16.1.3/32 is directly connected, HUB1-VPN1
B     172.20.2.0/24 [200/0] via 172.16.1.2 (recursive is directly connected, HUB1-VPN1_0), 00:02:56, [1/0]
C     172.20.3.0/24 is directly connected, port4
C     192.168.0.0/16 is directly connected, port1
What to look for

B 172.20.2.0/24 [200/0] via 172.16.1.2 — the B code proves BGP delivered BR2's LAN to BR3. [200/0] is IBGP admin distance 200. recursive means the next-hop 172.16.1.2 resolved through the overlay tunnel to the spoke — which after the first packet becomes a direct ADVPN shortcut.

🧠
MENTAL NOTE
IBGP split horizon = don't re-advertise learned IBGP prefixes. RR = allowed to. Combine RR + next-hop-self so the client can actually reach the advertised prefix. Neighbor-range = dynamic peer acceptance without per-spoke config.

💭 THINK FIRST · SECURITY FABRIC + AUTOMATION

Fabric root + fabric members lets one FortiGate speak for the group. On top of that, an automation stitch (trigger → action) can do things like back up the config every time it changes — without any external scheduler.

  • Why must the root FortiGate authorise each fabric member in System > Firmware & Registration before the topology view shows them?
  • The stitch has trigger Configuration Change and action Backup Config Disk. Where does the backup file end up, and what log confirms it fired?
Q1 — MODEL ANSWER
The fabric root has to explicitly trust every member — otherwise anyone with the fabric name and a reachable network could join and pull telemetry / config. The member sends a registration request; the root sees it as "pending" until an admin approves. Only after authorisation does the member show up in the topology and start reporting fabric telemetry.
Q2 — MODEL ANSWER
On the FortiGate's own disk (that's what Backup Config Disk means — local disk, not FortiManager, not FTP). You can retrieve it from System > Configuration > Revisions. The confirming log is a System Events entry with message stitch:Config_Change is triggered. Any admin-generated CLI or GUI change fires it.
SECTION 04

Security Fabric and Automation Stitch

Root elects the fabric name and interface. Members join it. An automation stitch marries a trigger to an action so config-changes automatically produce a local backup.

CONFIG
# on the root (HQ-NGFW-1 global): config system csf set status enable set group-name "fortinet" set upstream-ip 0.0.0.0 set configuration-sync default set fabric-object-unification default config trusted-list edit "1" set authorization-type serial next end end

Enables Security Fabric with fabric name fortinet. upstream-ip 0.0.0.0 means "no upstream" — this box is the root. Trusted-list is where the root records which downstreams are authorised to join.

What to look for

status enable plus a fabric name is the minimum. The GUI wraps this in Serve as Fabric Root. Downstream members set upstream-ip to this root's IP instead. Both ends must reach each other on TCP 8013 for fabric telemetry.

CONFIG
# on each fabric member (HQ-ISFW, HQ-DCFW): config system csf set status enable set group-name "fortinet" set upstream-ip 10.0.12.254 end

Member config: same group name, points upstream at the root's IP (10.0.12.254 in this exercise). Once this is committed, the member appears as pending on the root and stays that way until the root admin authorises it.

What to look for

Group name typo = silent fail (the member never registers). upstream-ip must be reachable and speaking fabric on TCP 8013. Under-provisioned allowaccess on the root's port5 (the fabric-facing port) will also block registration — check allowaccess fabric or equivalent.

CONFIG
config system automation-trigger edit "Config_Change_trigger" set trigger-type event-based set event-type config-change next end config system automation-action edit "Backup_Config_Disk" set action-type cli-script # or backup-config-disk in the GUI mapping next end config system automation-stitch edit "Config_Change" set trigger "Config_Change_trigger" set action "Backup_Config_Disk" next end

Three-step stitch. Trigger fires on any configuration change. Action performs the disk backup. Stitch binds them into a runnable pipeline. The GUI shortcut is HQ-NGFW > Security Fabric > Automation Stitch > Create New with trigger Configuration Change and action Backup Config Disk.

What to look for

Trigger + action together = a stitch. A stitch is what actually runs. Multiple actions can be chained under one stitch (e.g. backup + email notification). Stitches live at the fabric-root level and can target any authorised member.

EXECUTE
# on a member (HQ-DCFW), touch the config: config system interface edit port2 set description modification next end

Any admin change fires the Configuration Change trigger. Setting a harmless description is the standard smoke test.

What to look for

Then check System > Configuration > Revisions for a new entry with comment Autod backup config by stitch: Config_Change, and Log & Report > System Events > Logs for a Notice-level entry stitch:Config_Change is triggered. Both together = the automation ran end-to-end.

🧠
MENTAL NOTE
Fabric = one root + authorised members over TCP 8013. Automation = trigger (event-based or scheduled) + action (backup, script, email, webhook) + stitch (the binding that runs). Configuration Change trigger + Backup Config Disk action = free, automatic, per-change local backups.
