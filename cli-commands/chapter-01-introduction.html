Chapter 01 · CLI Reference

Introduction, Baseline CLI

Chapter 1 is mostly architecture, not command syntax — but the handful of commands you meet here (status, packet flow, sniffer, backup) are the ones you'll type before every deeper troubleshoot.

Every deeper chapter starts with get system status and a sniffer.

The commands below are the baseline "what am I looking at?" toolkit — used constantly before FGCP, VLAN, or VDOM troubleshooting even begins.

System Status Packet Flow & Offload Sniffer & Debug Backup & Restore
💭 THINK FIRST · SYSTEM STATUS

Before you touch anything on a strange FortiGate, you want two answers: what model / version am I on, and which VDOM mode am I in. Both come from one command.

  • Which single command tells you the firmware version, model, licence, and whether multi-VDOM is enabled?
  • Why does knowing the FortiGate model matter before you plan HW SSL inspection, hyperscale, or NP offload?
Q1 — MODEL ANSWER
get system status. It prints version, build, serial number, model, licence, and — importantly for Ch03 — the Virtual domain configuration line (standalone / multi-vdom / split-task).
Q2 — MODEL ANSWER
The model decides everything downstream. VMs handle every packet in CPU. Physical models have NP7 for network offload and CP for content-processor offload. Hyperscale, HW SSL inspection, and low-latency features exist only on specific hardware families (1000–7000 series for DCFW; 100–900 for ISFW/campus).
SECTION 01

System Status

The first commands you type on any FortiGate. They tell you the model, the firmware, and the VDOM mode — the three things every later troubleshooting step assumes you know.

GET
# get system status

The canonical "where am I" command. Prints version, build, serial, model, licence status, current time, and — critically — the Virtual domain configuration line (standalone, multi-vdom, or split-task).

Sample output (fields you care about)
Version: FortiGate-VM64 v7.6.x,build...
Serial-Number: FGVMxxxxxxxxxxxx
Virtual domain configuration: multiple
Max number of virtual domains: 10
Current HA mode: standalone
System time: ...
What to look for

The Virtual domain configuration field tells you whether you're in multi-VDOM mode. The Max number of virtual domains line answers "how many VDOMs does this box support?" (Ch03 uses both.) Also check version — a 7.4 policy package won't install cleanly onto a 7.2 ADOM (Ch02).

GET
# get sys status | grep "Max number of virtual domains"

Filtered version of the above — pulls just the maximum VDOM count line. Useful when you're on a session-restricted terminal and don't want the full status dump.

What to look for

Most models support up to 10 VDOMs by default. Need more? You buy a VDOM licence. This line is what tells you whether you've already hit the limit.

GET
# get system performance status

Snapshot of CPU load (per-core and total), memory usage, session counts, and uptime — the "is this box actually busy?" view.

What to look for

High CPU with low session count → likely UTM/proxy inspection saturating cores. High CPU with high session count → the box is close to its throughput ceiling; check the model's rated throughput against the DCFW/ISFW/NGFW/DEFW table before blaming the config.

🧠
MENTAL NOTE
On any unfamiliar FortiGate, always run get system status first. Version + model + VDOM mode are the three assumptions every later command implicitly makes.

💭 THINK FIRST · PACKET FLOW & OFFLOAD

The chapter's "Life of a Packet" slide has one big lesson: the first packet takes the long path (CPU), and every subsequent packet in that session may ride the NP7 fast path. If offload isn't happening, throughput collapses — and one diagnose command tells you why.

  • A session shows high CPU and low pps. Which command shows whether NP offload is actually engaging?
  • Why does the same firewall config give very different throughput on a physical FortiGate vs. a FortiGate-VM?
Q1 — MODEL ANSWER
diagnose sys session list and look at the npu_state / offload flags on the session. If offload is disabled you'll see a reason (asymmetric routing, flow inspection features enabled per policy, or the traffic type isn't NP-eligible).
Q2 — MODEL ANSWER
Physical models offload to NP7 (network) and CP (content) processors — flow-based IPS pattern matching, IPsec crypto, SSL decrypt. FortiGate-VM has no NP/CP: every packet is handled by CPU, so throughput is bounded by vCPU count and hypervisor overhead.
SECTION 02

Packet Flow and Offload

Two commands cover the "why is this packet doing what it's doing?" question — a session list to see the flow, and the debug flow when the session list isn't enough.

DIAGNOSE
# diagnose sys session list

Prints every active session in the session table with source/destination IPs, ports, policy ID, protocol, and offload state. This is the most-used session diagnostic on FortiGate.

What to look for

The npu_state (or state) line tells you whether the session is offloaded to NP. The policy_id confirms which firewall policy matched. On busy boxes narrow the output first with diagnose sys session filter.

DIAGNOSE
# diagnose sys session filter src <ip> # diagnose sys session filter dst <ip> # diagnose sys session filter dport <port> # diagnose sys session list

Applies a filter to the session table before listing. Essential on a production box where list alone would print thousands of sessions.

What to look for

Filters are AND-combined. Clear them with diagnose sys session filter clear before running your next unrelated query — old filters silently stay in place and hide sessions you thought should be there.

DIAGNOSE
# diagnose debug flow filter addr <ip> # diagnose debug flow show function-name enable # diagnose debug flow trace start 100 # diagnose debug enable

Turns on the packet-flow debugger — the tool for "the session isn't being created, why?" It prints, for each packet, every stage of the Life-of-a-Packet flow: ACL, HPE, IP integrity, IPsec decrypt, routing/policy lookup, UTM inspection, forwarding.

What to look for

Match the flow output against the "Life of a Packet" slide: the packet should traverse ACL → HPE → IP integrity → routing → session/policy lookup → UTM → forwarding. Wherever it stops printing, that's where the packet was dropped. Always follow with diagnose debug disable when you're done — flow debug is heavy on CPU.

🧠
MENTAL NOTE
Session list = "what's happening?" Debug flow = "why isn't anything happening?" Physical box offloads to NP/CP; a VM does everything in CPU.

💭 THINK FIRST · SNIFFER & DEBUG

When the flow debugger and session list both come up empty, you drop to the wire. The FortiGate sniffer isn't tcpdump — but the filter syntax is deliberately similar, and the verbosity number changes what you see.

  • What does the verbosity level (1, 2, 3, 4, 5, 6) actually control in diagnose sniffer packet?
  • Why is any as the interface both the most useful and the most dangerous option?
Q1 — MODEL ANSWER
1 = header only, 2 = header + IP data, 3 = header + Ethernet data, 4 = header + interface name (this is the one you want most days — it tells you which port the packet arrived on), 5 = header + IP data + interface, 6 = header + Ethernet data + interface. Verbosity 4 or 6 is what the HA chapter uses to prove which port heartbeat is arriving on.
Q2 — MODEL ANSWER
any tells the sniffer to listen on every interface — perfect when you don't know which port the traffic is on. Dangerous because on a busy production box you'll swamp the console. Always combine any with a tight BPF-style filter (host/port/proto) and a packet count.
SECTION 03

Sniffer and Debug

When flow debug can't explain it, sniff the wire. The FortiGate sniffer uses tcpdump-style BPF filters and takes a verbosity number that controls how much of the frame it prints.

DIAGNOSE
# diagnose sniffer packet <interface> '<filter>' <verbosity> <count>

The canonical shape. Interface = a port name or any. Filter = tcpdump-style BPF. Verbosity = 1–6 (see the socratic answer above). Count = how many packets to capture before stopping — always set it so you don't flood the console.

What to look for

Verbosity 4 is the sweet spot for most troubleshooting — you get the packet header plus the interface name, so on any you can see which port the packet came in on. Verbosity 6 when you actually need the payload bytes.

DIAGNOSE
# diagnose sniffer packet any 'host 10.0.0.5 and port 443' 4 20

A working example: sniff any interface, only TCP/443 to/from 10.0.0.5, verbosity 4 (with interface name), stop after 20 packets.

Sample output shape
interfaces=[any]
filters=[host 10.0.0.5 and port 443]
1.234567 port1 in 10.0.0.5.51234 -> 1.1.1.1.443: syn ...
1.234789 port2 out 10.0.0.5.51234 -> 1.1.1.1.443: syn ...
What to look for

The in/out marker plus the interface name tell you the packet's path across the FortiGate. If you see port1 in but no port2 out, the packet was dropped inside the FortiGate — go back to flow debug.

DIAGNOSE
# diagnose debug reset # diagnose debug disable

The "please stop everything" pair. reset clears every debug filter you may have accidentally left on; disable turns off the debug output stream.

What to look for

Always run both when you leave a troubleshooting session. Leftover debug filters + enabled output is a common way to make an idle FortiGate look like it has a CPU problem.

🧠
MENTAL NOTE
Sniffer verbosity: 4 for "which port?", 6 for "what payload?" Always end with diagnose debug reset + diagnose debug disable.

💭 THINK FIRST · BACKUP & RESTORE

The execute family covers commands that do something (as opposed to configuring or showing). Backup and restore live here — plus reboot, factoryreset, and the update pulls.

  • What's the difference between the config backup full-config takes and one from a specific VDOM (Ch03)?
  • Why is execute factoryreset the one command you almost never want to run without console access?
Q1 — MODEL ANSWER
A global (super-admin) backup with full-config captures both config global and every VDOM. A VDOM-scoped backup only contains that VDOM's config — useful when you're delegating a VDOM to another team and don't want to hand over the whole box's config.
Q2 — MODEL ANSWER
Because it wipes the config to defaults, including the management interface's IP. If you ran it over SSH, you're now locked out and the box comes back on 192.168.1.99/24 reachable only from a direct-connected laptop. Console cable required.
SECTION 04

Backup and Restore

The execute family — commands that perform an action rather than configure or query. These are the ones you type at the end of a maintenance window.

EXECUTE
# execute backup config <destination> <filename> [<server>] [<user>] [<password>]

Backs up the FortiGate configuration. Destination is one of tftp, ftp, usb, sftp, or scp. When run from global (super-admin), captures the entire config; when run inside a VDOM, captures only that VDOM.

What to look for

To back up everything on a multi-VDOM box, run this from config global as a super-admin — otherwise you get a per-VDOM slice. Add full-config before the destination if you want the full config including defaults (much larger file, but reproducible).

EXECUTE
# execute restore config <source> <filename> <server> [<user>] [<password>]

Restores a previously backed-up config file. The FortiGate reboots after the restore completes.

What to look for

The restore will fail if the source config was taken from a different model or major version. It will work across minor versions of the same model. Always test restores on a lab box before you promise a customer you can recover their prod device from a backup.

EXECUTE
# execute reboot # execute shutdown

Reboots or gracefully shuts the FortiGate down. Both prompt for confirmation before executing.

What to look for

Both commands halt traffic. In an HA cluster (Ch04), reboot on the current primary triggers a failover — plan for the brief service blip. shutdown is the graceful equivalent when you're physically pulling the box.

EXECUTE
# execute update-now

Forces an immediate FortiGuard update pull — antivirus signatures, IPS signatures, application-control database, web-filter categories. Useful when a new IPS signature was just published and you don't want to wait for the scheduled poll.

What to look for

Requires FortiGuard reachability — the management VDOM (Ch03) must have internet access, or a FortiManager acting as a local FDS must be reachable. If you're in a closed network and neither is true, this command will fail silently and every VDOM inherits stale signatures.

EXECUTE
# execute factoryreset

Wipes the config back to factory defaults. The FortiGate reboots to a bare state with the default management IP.

What to look for

Only run this when you have console access. The default management IP is 192.168.1.99/24 — reachable only if you're on that subnet or physically wired to the mgmt port. Running this over SSH is a guaranteed lockout.

🧠
MENTAL NOTE
The execute family is the "do it now" family — backup, reboot, update, factoryreset. Anything destructive needs console access before you type it.
