Security Profiles, By Command

💭 THINK FIRST · SSL/SSH INSPECTION

Before deciding "certificate inspection or full inspection," ask what you actually need to see. Cert inspection reads the SNI and server cert; full inspection MITMs the session and reads the payload. Full inspection costs CPU and RAM but is the only mode that catches an attack hidden inside HTTPS.

  • Which two CLI settings on an ssl-ssh-profile control the minimum accepted TLS version and what happens to deprecated versions?
  • Why does protocol port mapping (e.g. SMTPS on 10587 instead of 587) matter, and where is it configured?
Q1 — MODEL ANSWER
set min-allowed-ssl-version tls-1.2 sets the floor (nothing older than TLS 1.2 is accepted). set unsupported-ssl-version block tells the profile to drop any session using a version below the floor. Together they block TLS 1.0 / 1.1 outright while letting 1.2 and 1.3 through. Applies per-protocol under the https, smtps, pop3s, imaps, ftps sub-blocks.
Q2 — MODEL ANSWER
Because attackers use non-standard ports to bypass inspection — running SMTPS on 10587 rather than 587 means the firewall would only recognise 587 as SMTPS traffic and miss the 10587 flow. Under the SSL/SSH profile you can enable protocol port mapping and set each protocol's port list (HTTPS 8443, SMTPS 10587, etc.) so the firewall knows to inspect those alternate ports. Only works in proxy-based inspection with full SSL and multi-client-to-multi-server mode.
SECTION 01

SSL/SSH Inspection Profiles

The ssl-ssh-profile is where you decide whether the firewall is a spectator (cert inspection) or a MITM (full inspection). Every knob below lives inside config firewall ssl-ssh-profile.

CONFIG
config firewall ssl-ssh-profile edit "ssl_ssh_profile" config https set unsupported-ssl-version block set min-allowed-ssl-version tls-1.2 end next end

Enforces a TLS floor of 1.2 on HTTPS traffic and drops sessions that fall below it. The two knobs work together: min-allowed-ssl-version declares the floor; unsupported-ssl-version block is what makes the firewall enforce it rather than log-and-allow.

What to look for

This is CLI-only — no GUI equivalent. Apply the same pattern under smtps, pop3s, imaps, and ftps sub-blocks for the other TLS-wrapped protocols. TLS 1.0 and 1.1 are deprecated; 1.2 is the minimum broadly compatible; 1.3 is the current default for HTTP/3.

CONFIG
config system global set admin-https-ssl-versions tlsv1-2 tlsv1-3 end

Controls which TLS versions the FortiGate's own management interface (admin HTTPS) accepts. Different setting from the SSL/SSH profile — this protects the management plane, not the transit traffic.

What to look for

Confirm both the FortiGate and the admin's browser support the versions you list. Older browsers stuck on TLS 1.0/1.1 will lose GUI access if you tighten this without checking. Verify with show full | grep -f tls.

SHOW / GET
# show full | grep -f tls

Filters the running config for every TLS-related line — useful for auditing which protocols are on which floor across an SSL/SSH profile.

What to look for

Make sure both endpoints negotiating a session support the same versions — asymmetric floors cause silent handshake failures. If the admin says "the website works from my laptop but not through the firewall," this is the first grep to run.

CONFIG
config firewall ssl-ssh-profile edit "protect_ssl_server" set server-cert-mode re-sign config https set ports 443 set status deep-inspection end next end

Full SSL inspection profile in "Protecting SSL Server" mode. FortiGate presents its own certificate (self-signed or CA-signed) to the client and decrypts the flow to the server. Used when the FortiGate sits in front of your web server and needs to inspect inbound HTTPS.

What to look for

Three certificate options: (1) certificate signed by an internal CA — lab use; (2) FortiGate self-signed — lab use; (3) certificate from a global CA — recommended for production. Certificate inspection alone will let HTTPS attacks through untouched, hence full inspection for this scenario.

🧠
MENTAL NOTE
Certificate inspection = read SNI/cert only, cheap, misses payload-level attacks. Full inspection = MITM, sees payload, catches injections, costs CPU and RAM. Pick the lightest mode that catches the threat you actually care about.

💭 THINK FIRST · WEB FILTER · APP CONTROL · ISDB

Three overlapping tools, three different layers. Web filter is layer-7 URL/domain filtering. Application control is layer-7 pattern-matching for apps regardless of URL. ISDB is a layer-3/4 IP/port database — no inspection, just address groups you use in a policy.

  • Which of these three features doesn't inspect traffic at all, and where is it used in a policy?
  • How do you check whether the AC signature and ISDB databases on this FortiGate are current, and where does the web-filter database actually live?
Q1 — MODEL ANSWER
ISDB (Internet Service Database). It's a policy/object — a curated database of IP addresses and TCP/UDP ports for well-known services (Office 365, AWS, etc.) — used as a source or destination object in a firewall policy, an SD-WAN rule, or an internet policy. No inspection is performed; it's an address group with a friendly name.
Q2 — MODEL ANSWER
diagnose autoupdate versions — look under the Application Definitions section for AC signature version and expiry; look under Internet-service Full Database for ISDB. Web-filter database is not hosted on the firewall — FortiGate queries the FortiGuard Distribution Center (or a local FortiManager) for URL rating lookups on demand, then caches the result.
SECTION 02

Web Filter · Application Control · ISDB

The three UTM staples for content policy. Different config trees, different behaviors — mixing them up is the fastest way to build a policy that "looks right" but doesn't actually block what you think.

CONFIG
config webfilter profile edit "profile" config web set urlfilter-table 1 end next end

Skeleton of a web-filter profile. Inspects URLs, domains, SNI/SAN in flow or proxy mode. The urlfilter-table reference is how you attach a specific list of URL rules to this profile.

What to look for

Web filter can run in either flow-based or proxy-based inspection. Flow mode is faster; proxy mode gives more visibility (headers, cookies, safe-search) but adds latency. URL types supported: Simple, Wildcard, Regex.

CONFIG
config application list edit "profile" next end

Skeleton of an application-control profile. Operates only in flow-based inspection. Identifies apps by transmission patterns using application signatures, protocol decoders, and rate-based IPS signatures — regardless of the URL or port.

What to look for

Because AC uses signatures shipped by FortiGuard, keep the AC database current — see diagnose autoupdate versions below. Enable "Application Detection-Based SD-WAN" under System > Feature Visibility if you want to use AC inside SD-WAN rules.

CONFIG
config firewall internet-service-name edit "service" next end

ISDB (Internet Service Database) entry — a named group of IPs + TCP/UDP ports for a specific internet service. Used as a policy object (source or destination) in firewall policies, internet policies, and SD-WAN rules. No traffic inspection is performed.

What to look for

ISDB entries are shipped by FortiGuard — the FortiGate doesn't build the database itself. Reference an entry as internet-service-name in a firewall policy destination, or as internet-service-src-name for source-based matching.

DIAGNOSE
# diagnose autoupdate versions

Dumps the version, expiry date, last update attempt, and last result for every FortiGuard-updated database on the FortiGate — AC signatures, ISDB, IPS, AV, and more.

Application Definitions = AC signatures. Internet-service Full Database = ISDB. Confirm the Contract Expiry Date is in the future, the Version is recent, and Result was successful. Web-filter data isn't hosted here — it's queried on-demand from FortiGuard/FortiManager and cached locally.

Web filter = URL/domain inspection (flow or proxy). App control = flow-only pattern matching by signature. ISDB = policy object, no inspection. Web-filter DB lives on FortiGuard; AC and ISDB live on the FortiGate and need FortiGuard updates.

💭 THINK FIRST · IPS CONFIG

An IPS profile is a list of rules ordered top-down. Each rule has a Type (Filter or Signature), an Action (Block, Monitor, Default, Reset), and a target scope (Client, Server, both). Filters match by category and severity; Signatures match by name.

  • What's the practical difference between a Filter rule and a Signature rule, and why must a Signature-based exception be placed above a broad Filter?
  • Why is Action: Monitor the right choice when you're not sure whether a rule will cause false positives?
Q1 — MODEL ANSWER
A Filter matches many signatures by category, severity, OS, protocol, and target — good for broad "block everything critical on client and server" coverage. A Signature matches exactly one named signature — good for exceptions. Rules are evaluated top-to-bottom, so a Signature-based exception (e.g. Monitor a specific ICMP.Oversized.Packet) must sit above the broad Filter (Block Critical, Client+Server) or the Filter fires first and the exception never runs.
Q2 — MODEL ANSWER
Monitor lets the traffic through but writes an IPS log entry — so you get to observe what would have been blocked without actually blocking legitimate work. Pair with Packet logging enable during triage to capture the actual packet for tuning. Once you've confirmed nothing legitimate is being flagged, flip to Block and disable packet logging (it's expensive over time).
SECTION 03

IPS Profile Config

Every IPS command lives inside config ips sensor, then config entries. Rules inside a sensor are evaluated top-to-bottom — the first matching rule wins.

CONFIG
config ips sensor edit "IPS test" config entries edit 1 set severity critical high medium low info set protocol all set os all set application all set location client server set action block set log-packet enable set status enable next end next end

Filter-type entry that maximally covers threats. Every severity, every protocol, every OS, both client and server as targets — around 10,320 signatures activated by this one rule. Action is Block; packet logging is on for triage.

What to look for

Broad filters catch a lot but risk false positives — an app using unusual packet sizes might get flagged. Use Monitor first, watch the logs, tune, then move to Block. Rule ordering matters: a broad Block Filter will fire before a Monitor Signature exception if the exception is below it.

CONFIG
config ips sensor edit "IPS test" config entries edit 2 set rule 12345 set action monitor set log-packet enable set status enable next end next end

Signature-type entry — exactly one named signature (identified by rule ID). Sits above a broad Filter to carve out an exception: allow this specific signature to trigger only Monitor even though the Filter below would block it.

What to look for

Rule IDs come from the FortiGuard IPS encyclopedia — look up the signature by name (e.g. ICMP.Oversized.Packet) to find the ID. Signature exceptions must be higher priority than the Filter that would otherwise block the traffic.

CONFIG
config firewall policy edit <id> set utm-status enable set ips-sensor "IPS test" set ssl-ssh-profile "certificate-inspection" next end

Attaches the IPS sensor and an SSL/SSH profile to a firewall policy. The SSL/SSH profile mode dictates what IPS can see — cert inspection = HTTP payload but no HTTPS payload; full inspection = both.

What to look for

For a scenario detecting a PHP code injection in HTTP, certificate inspection is enough. For the same attack in HTTPS, you need a full-inspection profile (ssl-ssh-profile "protect_ssl_server") — otherwise IPS just sees an encrypted blob.

CONFIG
config ips sensor edit "IPS test" config entries edit 3 set rule <DNSExfiltrator.Data.Exfiltration id> set action block set status enable next end next end

Named FortiGuard signature DNSExfiltrator.Data.Exfiltration — used to detect and block DNS exfiltration (where an infected client encodes stolen data into DNS queries and an attacker-controlled name server decodes them).

What to look for

DNS exfiltration signature triggers require the traffic to actually be inspected — for DNS-over-TLS (DoT) or DNS-over-HTTPS (DoH) you need SSL/SSH inspection enabled in full mode ("Multiple Clients Connecting to Multiple Servers"). Otherwise the DNS payload is opaque and the signature never fires.

🧠
MENTAL NOTE
Rule order = policy: Signature exceptions above, broad Filters below. Action: Monitor first, Block second. Packet logging on during triage, off in production. For HTTPS attacks, IPS is only as good as the SSL/SSH profile in the policy.

💭 THINK FIRST · IPS DIAGNOSE

When the IPS log doesn't match your expectation — a rule you thought would fire didn't, or one you didn't expect did — the answer is almost always in three places: the signature version, the rule ordering, and what the SSL/SSH profile actually inspected.

  • Which single diagnose command tells you the current IPS engine version and signature database version on this FortiGate?
  • An attacker uses hping3 to simulate an attack. What does each of these do: hping3 -S -p 80 -c 5 <target>, hping3 --traceroute -V -1 -p 33434 <target>, hping3 -1 -d 654001 -c 8 <target>?
Q1 — MODEL ANSWER
diagnose autoupdate versions — same command as for AC and ISDB. Look for the IPS Definitions and IPS Attack Engine sections. Confirm contract expiry, version, and last update result. An old signature database is why the "brand-new CVE" isn't being caught.
Q2 — MODEL ANSWER
(1) -S -p 80 -c 5 = 5 TCP SYN packets to port 80 (a TCP ping); (2) --traceroute -V -1 -p 33434 = traceroute using ICMP (-1) with verbose mode to dest port 33434; (3) -1 -d 654001 -c 8 = 8 ICMP packets each 654001 bytes (way above MTU, forces fragmentation) — the classic "oversized ICMP" test that fires the ICMP.Oversized.Packet signature.
SECTION 04

IPS Diagnose and Attack Simulation

The commands that tell you whether the IPS engine is current and whether a rule actually fired. Plus the hping3 recipes used in the study guide to reproduce attacks safely in a lab.

DIAGNOSE
# diagnose autoupdate versions

Same command from Section 2 — but for the IPS use case, look at the IPS Attack Engine and IPS Definitions blocks in the output. Version and contract expiry are the two you check first when a signature isn't firing.

What to look for

If the IPS engine or definitions are more than a few weeks old, or contract expiry is past, FortiGuard updates are broken — check WAN reachability, DNS, and licence status before you spend time tuning rules.

EXECUTE
host $ hping3 -S -p 80 -c 5 www.example.com

TCP-SYN ping. Sends 5 (-c 5) TCP SYN packets (-S) to port 80 (-p 80) of the target. Like ICMP ping but proves TCP reachability instead of ICMP reachability.

What to look for

Useful when ICMP is blocked but you want to prove TCP hits. If the port is open you get SYN/ACK; closed = RST; filtered = no reply. Runs from an external test host, not the FortiGate CLI.

EXECUTE
host $ hping3 --traceroute -V -1 -p 33434 www.example.com

Traceroute using ICMP (-1), verbose output (-V), destination port 33434 (-p 33434). Same idea as classic traceroute but crafted with hping3 to control every field.

What to look for

Use when standard traceroute is filtered but you want to see which hop drops the packet. Port 33434 is the historical default for UDP-based traceroute; the ICMP variant just borrows the port number.

EXECUTE
host $ hping3 -S -p 22 -c 1 www.example.com

Single TCP SYN to port 22 — a one-shot port-check. If open you get SYN/ACK, closed = RST, filtered = silence. Cleaner than nmap for a specific port verification.

What to look for

Read the reply flag — SA = open, RA = closed. Use for firewall testing where you want to verify one specific port behaves as configured. This is one of the three canonical hping3 uses in the study guide.

EXECUTE
host $ hping3 -1 -d 654001 -c 8 10.0.3.55

Sends 8 ICMP packets (-1) each 654001 bytes (-d 654001) — way above the MTU, forcing fragmentation. This is the study guide's canonical trigger for the ICMP.Oversized.Packet IPS signature.

What to look for

On the FortiGate, check Log & Report > Security Events > Intrusion Prevention — you should see an ICMP.Oversized.Packet event with action dropped if the IPS Filter is set to Block, or detected if you carved out a Signature exception with action Monitor. This is the exact command that demonstrates a false-positive scenario in Use Case 1.

🧠
MENTAL NOTE
Signature not firing? Check three things: is the database current (diagnose autoupdate versions), is the rule ordering correct (Signature exceptions above broad Filters), and is the SSL/SSH profile letting IPS see the payload. hping3 is the safe way to prove any of these in a lab.
