The Ladder

Five questions, asked in order. Riley's ticket gives you the symptom — never the rung.

Why a ladder at all: five completely different failures all produce the same sentence from a user — "it's slow." A catch-all eating the match, a member with no gateway, a breached SLA, a member missing from a rule's list, and a losing preference contest are indistinguishable from the help desk. The ladder is how you find out which one without guessing — start at the top, climb down, stop at the first rung that fails.
1

Does a rule MATCH? New in S63

show system sdwan  ·  diagnose sys sdwan service

Rules are read top-down, first match wins, nothing below is ever consulted. Is the traffic hitting the rule you think it's hitting — or is a catch-all in seat one eating everything? Check src/dst ranges in the service output against the actual flow. Unmatched traffic falls to the implicit rule (the global load-balance-mode) — which consults no SLA, ever.

Fails like: every rule below the catch-all is dead config that looks valid. J.R.'s Monday.

2

Is it UP?

get router info routing-table all

Reachability. Does the member have a route — including the auto-built member route that only exists if the member has a gateway? Only a connected route for that interface = no gateway was ever set.

Fails like: port3 — link light on all weekend, zero packets. Up is a fact; it is not a route.

3

Is it IN-SLA?

diagnose sys sdwan health-check

Quality. Read the shape of the numbers: real bad numbers = carrier problem. dead + 100% loss + latency/jitter 0.000 = nothing ever left = configuration problem wearing a carrier costume.

Fails like: Baz's daily brownout — alive, valid route, and 400ms of lying.

4

Is it ELIGIBLE?

(read the rule's priority-members + confirm a route to THIS destination)

Two sub-checks: is the member in this rule's list at all, and does it hold a valid route to this destination? A rule never creates reachability — it only chooses among members that already have it.

Fails like: a healthy member that simply isn't invited — the spare chair at a full table.

5

Is it PREFERRED?

diagnose sys sdwan service

The contest itself. Which strategy is this rule running — and is the SLA a gate (sla, load-balance), a ranking (priority), or not consulted at all (manual)? Check which members show selected and whether the health check even appears in the output — its absence is evidence too.

Fails like: mode sla with all members out of SLA — the rule is NOT skipped; it falls back to configured member order, best effort. Sick ≠ dead.

6

Is the SESSION pinned? Session 64

The exit is stamped into the session; existing flows don't re-ask. "New calls are fine. Old calls sound like the ocean." — next episode.

Exam framing: Fortinet writes questions that hand you rung-5 evidence for a rung-2 fault (or vice versa). Identify the rung the output belongs to before choosing an answer — a health-check output cannot prove anything about rule matching, and a routing table cannot prove anything about SLA state.
Symptom shape is evidence. Before touching a single command: "everyone, occasionally" smells like per-flow balancing (round-robin); "the same four people, always" smells like deterministic hashing (source-ip-based). The ticket pattern told you the algorithm before the config did.
