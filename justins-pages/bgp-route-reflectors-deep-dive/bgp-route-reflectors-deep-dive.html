The three drawers, the rules that guard them, and the mirror that lets one router speak for nine.
"Believing and telling are different drawers. Almost all of BGP policy is the door between them."

Section 01 · The Filing System

RIB-in → Local RIB → RIB-out

Every BGP question on this page — and most BGP questions on the exam — comes down to naming the drawer before you diagnose. A BGP router does not have "a routing table" for BGP. It has three distinct stores per neighbour relationship, and a route must pass through a gate to move between them.

Drawer 1
RIB-in
"What I was told."
Everything received, per neighbour.
· Raw updates, exactly as the peer sent them
· One RIB-in per neighbour
· Contains valid AND invalid routes
· Loop checks happen at the door — a route that fails ORIGINATOR_ID / CLUSTER_LIST / AS_PATH never even lands here
➜
Gate 1
Validity check + inbound policy + best-path selection
Drawer 2
Local RIB
"What I believe."
The winners. One best path per prefix.
· Only valid routes compete here
· Best-path algorithm picks one winner
· The winner is offered to the routing table
· Seeing a route ≠ using it — a route can sit in RIB-in and never make it here
➜
Gate 2
Advertisement rules: split horizon, reflection, outbound policy
Drawer 3
RIB-out
"What I'm willing to say."
Per neighbour, after policy.
· One RIB-out per neighbour
· Only best paths are candidates
· Split horizon, RR reflection rules, and prefix-lists all live at this gate
· Attributes get stamped here (ORIGINATOR_ID, CLUSTER_LIST, next-hop-self)
Received → Believed → Willing to say. A route must win Gate 1 before it is even a candidate at Gate 2.
Almost every "the route is missing" ticket is really the question: which drawer is it stuck in, and which gate refused it?
Verification maps directly onto the drawers:
get router info bgp neighbor <ip> routes → that neighbour's RIB-in (post-policy)
get router info bgp network → the Local RIB view (look for the > best marker and * valid marker)
get router info bgp neighbor <ip> advertised-routes → that neighbour's RIB-out
get router info routing-table bgp → what actually made it into the forwarding decision
🧠
Mental Note
Name the drawer before you diagnose. Say it aloud: "the route is in the Local RIB, so this is a RIB-out question." Selection problems live at Gate 1. Advertisement problems live at Gate 2. Reflection is a Gate 2 mechanism — it never fixes a Gate 1 problem like an unresolvable next hop.
Section 02 · The Law Inside the AS

IBGP Rules — and the Relaxation

Rule 1 — IBGP split horizon

Exact wording matters, because the exam's favourite distractor swaps it: a route learned from an IBGP peer is never re-advertised to any other IBGP peer. Not "not back to the sender" — to anyone inside the AS. IBGP carries no AS_PATH change within the AS, so there is no loop detector inside the borders. Split horizon is the substitute: if nobody relays, nothing can loop.

Rule 2 — Therefore, full mesh

If nobody relays, everybody must hear everything first-hand. Every IBGP speaker peers with every other IBGP speaker: n(n−1)/2 sessions. Nine LoFinance sites is 36 sessions, and adding Adelaide meant touching all nine firewalls — an eleven-hour change window for one new branch.

Every safety rule has a price, and the price is usually scale. Split horizon buys structural loop-freedom and charges n(n−1)/2. Engineering is choosing which bill to pay.

Rule 3 — The route reflector relaxation

A route reflector (RR) is a router that is permitted to break split horizon for a defined set of peers — its clients. The client set is defined on the RR, under each client's neighbour block. The client itself carries an ordinary IBGP config; nothing on the client says "client." Everything inside config neighbor / edit <ip> describes the peer at that address — same shape as set remote-as and set next-hop-self.

HQ-FGT — promoting to Route Reflector (configured on the RR, per client)
config router bgp
    set as 65001
    set cluster-id 0.0.0.1        // optional; defaults to the router-ID
    config neighbor
        edit "10.0.1.2"           // MEL-FGT
            set remote-as 65001
            set next-hop-self enable
            set route-reflector-client enable
        next
        edit "10.0.4.2"           // PER-FGT — ordinary IBGP + one extra line
            set remote-as 65001
            set next-hop-self enable
            set route-reflector-client enable
        next
    end
end

Rule 4 — The three relay cases

When a best path sits in the RR's Local RIB and reaches Gate 2, the RR classifies where it came from and applies one of three behaviours:

Route learned from…RR advertises to…Why
A clientEveryone — other clients, non-client IBGP peers, EBGP peersThe client cannot tell anyone itself. If the RR doesn't relay, nobody hears it.
A non-client IBGP peerClients onlyNon-clients are ordinary IBGP peers of each other — original split horizon still binds sideways.
An EBGP peerEveryoneSplit horizon never applied to EBGP-learned routes in the first place.
🧠
The Compressed Rule
Downward (to clients): always. Sideways (non-client to non-client): never. And its shadow: clients are mute. A client that learns a route via IBGP is still bound by ordinary split horizon — it advertises it to no IBGP peer, not even back to the RR as a relay.
"The reflector speaks freely to its own. To strangers, it only repeats what its own told it."— Señor, taping the diagram back into the drawer
Section 03 · Gate 1

What Makes a Route Valid

Validity is the entry ticket to the best-path contest. An invalid route can sit in RIB-in forever — visible in get router info bgp neighbor routes, missing the * marker, and never competing. Valid ≠ best ≠ installed. Three separate hurdles.

A route is valid when all of the following hold:

  • The NEXT_HOP is resolvable. The router must already have a route (IGP, static, or connected) to the next-hop address. This is the classic IBGP failure: EBGP routes arrive with an external next hop nobody inside the AS can reach. Reflection does not fix this — next-hop-self on the RR is almost always still required.
  • It passes the loop checks. Three nested rings, one question at every scale — "did this come from me?" (next section). A failure here is a silent drop.
  • It survives inbound policy. Prefix-lists, route-maps, and filters at Gate 1 can discard or mark it before it ever competes.
The trap pairing: "valid but not best" (lost the contest — look at attributes) versus "not valid at all" (usually an unresolvable next hop, or a loop-check drop). A route that is present-but-unusable and a route that was silently rejected produce the same business symptom and completely different fixes. Name the drawer first.
🧠
Mental Note
In get router info bgp network: * = valid, > = best. A line with * and no > is a route the router believes but did not choose. A prefix with no * line at all was refused at Gate 1 — and if it never appears, suspect a silent loop-check drop. Absence of errors is not evidence of correctness.
Section 04 · The Replacement Guards

ORIGINATOR_ID & CLUSTER_LIST

Split horizon was a structural guarantee: loops were impossible because relaying was impossible. The route reflector relaxes that structure — so something must replace it. The replacement is detection: two new optional attributes that a route reflector stamps onto every route it reflects, so any router can later recognise its own work coming back around.

Ring 1 · Between ASes
AS_PATH
"Is my AS in it?"
The original guard. Prepended at every EBGP hop. Useless inside one AS — which is why IBGP needed split horizon in the first place.
Ring 2 · Between Clusters
CLUSTER_LIST
"Is my cluster-ID in it?"
Each reflecting RR prepends its cluster-ID as it reflects. If an RR receives a route already carrying its own cluster-ID, it drops it. This is the multi-RR / inter-cluster guard. Checked by reflectors.
Ring 3 · Within a Cluster
ORIGINATOR_ID
"Is my router-ID it?"
Set once by the first RR to the router-ID of the router that originally injected the route. If any router sees its own router-ID here, it drops the route — silently, no log. Protects the originator from being fed its own route.
🧠
Mental Note
Same question at every scale: "did this come from me?" Scope tells them apart under exam pressure — ORIGINATOR_ID names the originating router (router-scope, set once); CLUSTER_LIST names every reflecting RR (cluster-scope, grows at each reflection).
Reading the guards on a reflected route (Perth's view)
PER-FGT # get router info bgp network 10.99.60.0/24
BGP routing table entry for 10.99.60.0/24
Paths: (1 available, best #1, table Default-IP-Routing-Table)
  Local
    10.0.0.1 from 10.0.0.1 (172.16.1.1)
      Origin IGP metric 0, localpref 100, valid, internal, best
      Originator: 172.16.3.1          // MEL-FGT's router-ID — the true source
      Cluster list: 0.0.0.1           // one reflection, by cluster 1 (HQ)
Section 05 · The Worked Trace

Melbourne → HQ → Perth

The setup: MEL-FGT and PER-FGT are both route-reflector clients of HQ-FGT (cluster-ID 0.0.0.1). Neither client peers with the other — the hub-and-spoke replaced full mesh. Melbourne originates 10.99.60.0/24 via the network command. Trace it into Perth's routing table: which drawer, on which router, and which rule fires, at each of the three steps.

🤔Think First — before scrolling to the answer
There are exactly three routers and exactly three gate decisions between Melbourne's network statement and Perth's routing table. A wrong trace almost always makes one of two errors: giving a client a voice it doesn't have, or expecting reflection to solve a next-hop problem.
  • Melbourne learned this route from nobody — it originated it. Does IBGP split horizon restrict what Melbourne may say about it, and why not?
  • When the route reaches HQ's Gate 2, which of the three relay cases applies — and what two attributes does HQ stamp before the route leaves?
  • At Perth's Gate 1, three checks run before the route can enter the Local RIB. What are they, and which config decision on HQ makes the next-hop check pass?
  • Step four doesn't exist: why does the route stop at Perth and go no further?
Q1 — Origination and split horizon
No restriction. Split horizon binds routes learned from an IBGP peer. An originated route was learned from nobody — Melbourne may tell every IBGP peer it has. It only has one: HQ.
Q2 — HQ's relay case
Learned from a client → reflect to everyone. HQ stamps ORIGINATOR_ID = Melbourne's router-ID and prepends its cluster-ID (0.0.0.1) to CLUSTER_LIST. next-hop-self rewrites the next hop to HQ.
Q3 — Perth's Gate 1
Loop checks (ORIGINATOR_ID isn't Perth's router-ID; Perth isn't a reflector so CLUSTER_LIST doesn't apply to it), inbound policy, and next-hop resolution — which passes because HQ set next-hop-self, not because of reflection. Different drawers, both needed.
Q4 — Why it stops
Perth learned the route via IBGP and Perth is a client, not a reflector. Ordinary split horizon binds it. Clients are mute.

The trace, step by step

1
MEL-FGT — the originator
Local RIB → RIB-out (toward HQ)Rule: origination is exempt from split horizon

The network 10.99.60.0/24 statement injects the prefix into Melbourne's Local RIB (it must already exist in Melbourne's routing table — the network command advertises, it does not create). At Gate 2, split horizon has nothing to say: it only restricts routes learned from an IBGP peer, and this one was learned from nobody. The route is placed into the RIB-out for Melbourne's only IBGP neighbour, HQ, and sent. No ORIGINATOR_ID, no CLUSTER_LIST yet — clients stamp nothing.

2
HQ-FGT — the route reflector
RIB-in (from MEL) → Local RIB → RIB-out (toward PER)Rule: from a client → reflect to everyone

The update lands in HQ's per-neighbour RIB-in for Melbourne. Gate 1: next hop resolvable (Melbourne's tunnel address, reachable), loop checks pass, no filter blocks it — valid; it's the only path, so best; into the Local RIB and the routing table. Gate 2: HQ classifies the source. Melbourne is a client, so the first relay case fires: reflect to everyone — Perth, the other clients, and any EBGP peers (subject to outbound filters).

As it reflects, HQ stamps the guards: ORIGINATOR_ID = 172.16.3.1 (Melbourne's router-ID) and CLUSTER_LIST = 0.0.0.1 (prepends its own cluster-ID). Because next-hop-self is enabled toward clients, the next hop is rewritten to HQ's own address. The stamped route enters the RIB-out for Perth and is sent. Note what reflection did not do: it never touched the next-hop problem — that was next-hop-self, a separate decision at the same gate.

3
PER-FGT — the receiving client
RIB-in (from HQ) → Local RIB → routing tableRule: loop checks pass + next hop resolves → valid → best → installed

The update arrives at Perth's RIB-in for HQ. Loop checks at the door: ORIGINATOR_ID is Melbourne's router-ID, not Perth's — pass. CLUSTER_LIST contains 0.0.0.1, but Perth is not a reflector and has no cluster-ID to match — the check is a reflector's check, so it passes trivially. Gate 1: next hop is HQ (thanks to step 2's next-hop-self), which Perth can resolve — valid. Only path — best. Into the Local RIB, offered to the routing table, installed. Riley can now reach 10.99.60.0/24.

And there is no step 4. Perth learned this route from an IBGP peer, and Perth is a client. Original split horizon binds it completely: the route enters no RIB-out toward any IBGP peer. Clients are mute — which is exactly why, back in step 2, the RR had to reflect to everyone.

The whole trace in one line: Melbourne speaks once (origination is free), HQ relays with a receipt (client → everyone, stamped with ORIGINATOR_ID + CLUSTER_LIST, next hop rewritten), Perth listens and stops (valid → best → installed, then mute). Three routers, three drawer transitions, three different rules.
Section 06 · Redundancy Without Regret

Two Route Reflectors at HQ — why the guards exist

One RR is a single point of failure for the entire AS's route distribution — Penny would not sign off on that for long. So: HQ-FGT-1 and HQ-FGT-2, both route reflectors. Every client (Melbourne, Perth, Brisbane, Adelaide…) peers with both. The two RRs also peer with each other as ordinary non-client IBGP peers. Now every route has two paths through the hub, and losing one HQ firewall loses nothing.

But look at what we just built. Melbourne announces 10.99.60.0/24 to both RRs. RR1 reflects to everyone — including RR2. RR2 reflects to everyone — including its clients, one of whom is Melbourne itself. We deliberately created relay paths that lead straight back to their own sources. Without the guards, this is an update carousel that never stops. With them, every dangerous copy dies exactly one hop before it does damage.

The scenario: one prefix, four dangerous copies

A
RR2 reflects Melbourne's route back down… to Melbourne
Guard: ORIGINATOR_ID — the originator's shield

RR2 learned the route (directly from MEL, or reflected via RR1 — either way). Its relay rules say reflect to clients, and Melbourne is a client. The update genuinely goes out. At Melbourne's door, the loop check runs: ORIGINATOR_ID = 172.16.3.1 = my own router-ID. Silent drop. No log, no error, no entry in RIB-in. Melbourne is never tricked into believing a second-hand copy of its own announcement.

B
RR1 and RR2 hand the route back and forth
Guard: CLUSTER_LIST — the reflectors' shield

RR1 reflects the client-learned route to RR2 (client → everyone includes the non-client peer RR2). If that copy could ever find its way back to RR1 — via RR2's reflection, or through a third reflector in a bigger design — the receiving RR checks CLUSTER_LIST at the door. My own cluster-ID is already in the list → I have handled this copy before → drop. Every reflection prepends another cluster-ID, so the list is a receipt trail: the carousel can never complete a lap.

The design choice: same cluster-ID or different?

DesignBehaviourTrade-off
Same cluster-ID on both RRs
(one logical cluster)		 RR2 rejects RR1's reflected copy at the door — it sees its own cluster-ID in CLUSTER_LIST immediately. Each RR keeps only the copies it heard directly from clients. Smaller RIB-in, less churn. But redundancy now depends entirely on every client peering with both RRs — a client sessioned to only one RR loses its routes if that RR dies.
Different cluster-IDs
(two clusters)		 RR2 accepts RR1's reflected copy as an extra path — CLUSTER_LIST contains only RR1's ID, not its own. Loops are still impossible: any copy arriving back at a reflector it has already visited is dropped. More paths, more memory, more updates — but the reflectors back each other up even for a client that lost one of its two sessions. The common enterprise default.
Notice the division of labour. CLUSTER_LIST is checked by reflectors and stops reflector-to-reflector loops. ORIGINATOR_ID is checked by everyone and protects the one router the CLUSTER_LIST can't: the client that started it all, which has no cluster-ID to match. Two guards, two scopes, zero laps of the carousel — and the price of both is the same price as always: the drops are silent. When routes "vanish" in a two-RR design, check for a client whose router-ID collides with another device, or a cluster-ID mismatch between the RRs, before you blame the tunnel.

🧠
Mental Note
When you relax a structural guarantee, you must replace it with a detection mechanism. Full mesh + split horizon made loops impossible. Reflectors made them possible but detectable — and two reflectors is where that trade finally gets tested. The redundancy design works because the guards exist, not despite them.
"Two mirrors facing each other should reflect forever. These do not. Ask yourself what each mirror writes on the back of the glass before it passes the image along."— Señor, standing between the two HQ racks
Section 07 · Quick Reference & Exam Traps

Keep This In the Top Drawer

TrapThe truth
Where is route-reflector-client configured?On the RR, inside the client's neighbour block. The client's own config is ordinary IBGP. The most commonly inverted RR question.
"Reflection fixed our next-hop problem"Never. Reflection is a Gate 2 (advertisement) mechanism; next-hop resolution is Gate 1 (validity). next-hop-self on the RR is almost always still required. Different drawers, both needed.
Split horizon exact wordingNot re-advertised to any other IBGP peer — not merely "not back to the sender."
RR learns a route from a non-client IBGP peerReflects to clients only. Sideways (non-client → non-client) is still forbidden. Downward: always. Sideways: never.
Client learns a route via the RRClients are mute — ordinary split horizon still binds them. They advertise IBGP-learned routes to nobody.
ORIGINATOR_ID vs CLUSTER_LISTORIGINATOR_ID = the originating router's router-ID, set once, checked by everyone. CLUSTER_LIST = every reflecting RR's cluster-ID, prepended per reflection, checked by reflectors. Scope is the tell.
What does a loop-check drop look like?Nothing. Silent, no log. Absence of errors is not evidence of correctness — if a route never appears in RIB-in, suspect a router-ID or cluster-ID collision.
Valid vs best vs installed* = valid (entered the contest), > = best (won it), routing table = installed (offered and accepted). Three hurdles, three failure modes.
