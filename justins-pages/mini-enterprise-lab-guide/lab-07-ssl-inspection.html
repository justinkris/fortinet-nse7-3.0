Objective

Configure certificate inspection vs full deep inspection, deploy the FortiGate CA certificate to a test client, exempt sensitive categories, and prove that AV now catches the HTTPS threat that escaped in Lab 6.

Why this matters

Without decryption, your expensive NGFW inspects the ~5% of traffic nobody bothers encrypting. But decryption is also the most politically and technically delicate feature you'll deploy: it breaks certificate pinning, raises privacy questions (banking, health), and fails loudly if the client doesn't trust your CA. Engineers who understand the trade-offs get to deploy it; engineers who don't cause outages.

Think first
  1. TLS was explicitly designed to prevent a device in the middle reading traffic. So how can the FortiGate do it at all — what must it forge, and what must the client be persuaded to trust?
  2. What can certificate-inspection see without decrypting anything? (Hint: think about the handshake — SNI and the server certificate.)
  3. Why should banking and health categories be exempted from deep inspection even where it's legal?
  4. A mobile app stops working after you enable deep inspection, while browsers are fine. What is the app doing that browsers aren't?

Stop reading and write your predictions before continuing.

Mental model — the passport office translator

Deep inspection is a trusted translator opening sealed diplomatic mail: the FortiGate terminates the client's TLS session, reads the contents, then re-seals it in a new envelope to the real server. The forged envelope carries the FortiGate's own CA stamp — which only works if every citizen (client) has been told in advance that this stamp is legitimate (CA cert installed). Certificate pinning is a diplomat who memorised the exact original wax seal and refuses any substitute — no amount of trust-store persuasion helps; you must exempt them.

Configuration

config firewall ssl-ssh-profile
    edit "SSL_DEEP_USERS"
        set comment "Deep inspection for HQ users"
        config https
            set ports 443
            set status deep-inspection
        end
        set server-cert-mode re-sign
        set caname Fortinet_CA_SSL
        config ssl-exempt        # exemptions are usually added via GUI categories:
        end                      # Finance & Banking, Health, Personal Privacy
    next
end
# Attach: policy 1 → set ssl-ssh-profile SSL_DEEP_USERS
GUI: Security Profiles → SSL/SSH Inspection → clone deep-inspection, add category exemptions. Download the CA: System → Certificates → Fortinet_CA_SSL → Download

Install the downloaded CA certificate into HQ-PC-1's trusted root store (Windows: certmgr.msc → Trusted Root CAs; Linux: /usr/local/share/ca-certificates/ + update-ca-certificates; Firefox has its own store — import in Settings → Certificates).

Verification

# From HQ-PC-1: browse an HTTPS site, click the padlock — the certificate issuer
# should now read the FortiGate CA, not the site's real CA.

# Re-run the Lab 6 HTTPS test-file download — expect a BLOCK page this time.

# On the FortiGate:
diagnose sys session filter dport 443
diagnose sys session list | grep -i proxy   # deep-inspected flows show proxy/utm state

Reflection

  • Untrusted-certificate warnings on every site is the classic symptom of which single missed deployment step?
  • Describe the failure mode if the FortiGate's clock (Lab 1's NTP!) drifts badly — what happens to every certificate it forges?
  • How would you roll deep inspection into production: which user group first, what exemptions on day one, and what would make you pause the rollout?
Challenge exercise

Create an address group GRP_SSL_EXEMPT_HOSTS with one FQDN in it, exempt it in the profile, and prove with the browser padlock that the exempted site shows its real certificate while others show the forged one. Then answer: why is exemption-by-address weaker than exemption-by-category for compliance purposes?
