Objective

Replace every all in your policies with named address objects, groups, and custom services, following the LoFinance naming standard.

Why this matters

Objects are the vocabulary of your security policy. all → all : ALL tells an auditor nothing and gives an attacker everything. Named objects make policies self-documenting, make change requests reviewable ("add HOST_HQ_WEB2 to GRP_DMZ_SERVERS" is auditable; "edit policy 7" is not), and — critically for Lab 13 — they are the unit FortiManager shares across all devices.

Think first
  1. Ten policies reference the HQ users subnet directly as 10.10.10.0/24. The subnet gets renumbered. Compare the work with and without an address object.
  2. What is the difference in intent between an address group and simply listing multiple addresses in a policy?
  3. A custom service SVC_WEB_8080 vs reusing built-in HTTP with a different port in the policy — why does FortiOS not even offer the second option?
  4. Where's the danger in one giant group GRP_ALL_INTERNAL used everywhere?

Stop reading and write your predictions before continuing.

Mental model — the library catalogue

Raw IPs in policies are books shelved by memory: only the person who shelved them can find them. Address objects are catalogue cards — one card, referenced everywhere, updated once. Groups are reading lists: "First-Year Networking" can gain a book without editing every student's borrowing rules. A library where every librarian invents their own catalogue format (no naming standard) is worse than no catalogue at all.

Configuration (HQ shown; repeat the pattern per site)

config firewall address
    edit NET_HQ_USERS
        set subnet 10.10.10.0/24
    next
    edit NET_HQ_SERVERS
        set subnet 10.10.20.0/24
    next
    edit NET_HQ_DMZ
        set subnet 10.10.40.0/24
    next
    edit NET_HQ_MGMT
        set subnet 10.10.99.0/24
    next
    edit HOST_HQ_WEB1
        set subnet 10.10.40.80/32
    next
    edit HOST_FAZ01
        set subnet 10.10.99.20/32
    next
    edit HOST_FMG01
        set subnet 10.10.99.10/32
    next
    edit NET_BRA_LAN
        set subnet 10.20.10.0/24
    next
    edit NET_BRB_LAN
        set subnet 10.30.10.0/24
    next
end

config firewall addrgrp
    edit GRP_HQ_INTERNAL
        set member NET_HQ_USERS NET_HQ_SERVERS
    next
    edit GRP_ALL_BRANCH_LANS
        set member NET_BRA_LAN NET_BRB_LAN
    next
end

config firewall service custom
    edit SVC_WEB_8080
        set tcp-portrange 8080
    next
end
config firewall service group
    edit GRP_SVC_MGMT
        set member HTTPS SSH PING
    next
end
GUI: Policy & Objects → Addresses / Services

Now edit Lab 3's policies: replace every all source with the correct object, and the mgmt policy's service list with GRP_SVC_MGMT.

Verification

show firewall address
diagnose firewall iprope lookup 10.10.10.100 12345 10.10.99.10 443 tcp VLAN10_USERS

GUI: right-click any object → Where Used. Confirm no policy still says all → all for internal traffic.

Reflection

  • Why did we create objects for the branch LANs on the HQ firewall already, and which two later labs will consume them?
  • What operational review question does Where Used answer that show firewall policy answers only painfully?
  • When is a /32 host object better than a range, even for a pool of servers?
Challenge exercise

LoFinance's compliance officer wants internet access for servers restricted to OS updates only. Build a policy using an FQDN address object (e.g. a Linux mirror hostname) above the general LAN-to-WAN rule, then move the servers out of that general rule. What DNS dependency have you just created, and how would it fail?
