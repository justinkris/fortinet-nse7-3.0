Objective

Master per-device mappings on addresses and interfaces — the mechanism that lets one shared policy package say "the local LAN" and mean a different subnet on every device — and clean up the Lab 13 shortcut properly.

Why this matters

This is the keystone of scalable central management. Without per-device mappings, "shared" packages fill with per-site policy clones and you're back to unmanageable. With them, one policy row serves a hundred branches. It's also the concept that makes ADVPN and SD-WAN templates possible at scale — and the exam loves asking what a mapped object resolves to on a given device.

Think first
  1. A policy row says NET_BRANCH_LAN_LOCAL → internet: allow. Installed to BRA it must mean 10.20.10.0/24; to BRB, 10.30.10.0/24. Where can that translation live — on FMG at install time, or on the device at run time? What actually happens?
  2. If a target device has no mapping for a dynamic object, what are FMG's options at install time? Which would you configure and why?
  3. Contrast dynamic objects with Lab 14's metadata variables. Both are per-device indirection — what does each parameterise, and where does each get consumed?
  4. What's the audit/readability cost of heavy per-device mapping — what question becomes harder to answer by reading the package alone?

Stop reading and write your predictions before continuing.

Mental model — "the stockroom", resolved locally

Back to the franchise manual: "restock from the stockroom" is a dynamic object. Each store's local glossary resolves it to a real room. Head office ships the manual (install); the shipping department consults each store's glossary and prints a localised edition (the rendered config) — so what arrives on site contains the real room number, not the abstraction. The abstraction lives only at head office. If a new store never filled in its glossary, shipping must refuse the order, not guess.

Configuration

FMG GUI (in ADOM_BRANCHES): Policy & Objects → Object Configurations → Firewall Objects → Addresses → edit NET_BRANCH_LAN_LOCAL → enable Per-Device Mapping:
DeviceMapped value
LOF-BRA-FGT0110.20.10.0/24
LOF-BRB-FGT0110.30.10.0/24

Confirm your normalised interfaces from Lab 13 (ni-lan, ni-wan) also carry per-device mappings — interfaces and addresses are the two halves of the same idea. Reinstall the package and read the preview: each device's diff shows its own subnet.

# On BRA after install — the object arrives resolved:
show firewall address NET_BRANCH_LAN_LOCAL
config firewall address
    edit "NET_BRANCH_LAN_LOCAL"
        set subnet 10.20.10.0 255.255.255.0

Verification

  • Install preview per device shows the correct localised subnet.
  • Traffic from both branch PCs still flows; sessions match the package policy ID.
  • Deliberately add a fictional third target with no mapping and run a preview — observe how FMG surfaces the problem, then remove it.

Reflection

  • Your Lab 13 policy now contains zero per-site literals. Enumerate everything Branch C onboarding needs from this ADOM: variables (Lab 14) + mappings (this lab) + target assignment. Nothing else. What does that say about where "site knowledge" should live?
  • When is per-device mapping the wrong tool — e.g. would you map HOST_FAZ01 per device? Why is a genuinely global object better left unmapped?
  • How would you audit "what does policy 1 actually permit on BRB?" — which FMG view or preview answers it fastest?
Challenge exercise

Create a mapped object HOST_BRANCH_PRINTER (BRA→10.20.10.9, BRB→10.30.10.9) and a package policy allowing HQ's helpdesk subnet to reach it on IPP (tcp/631) — noting that this policy can only take effect once branch↔HQ routing exists. Which upcoming lab makes it live, and what does that teach about sequencing policy vs connectivity work?
