Objective

Use provisioning templates (system template + CLI template with metadata variables) so that per-device system settings — hostnames, DNS, NTP, interface IPs — are generated from one definition plus per-device variables.

Why this matters

Policy packages centralised your rules; templates centralise your system config. This is the difference between onboarding Branch C in an afternoon versus a week: define the variables, assign the templates, install. It's also the foundation of zero-touch provisioning, and metadata variables are heavily featured in the 7.6 curriculum.

Think first
  1. Across your two branches, list what config is identical (DNS? NTP? admin timeout?) and what is per-device (LAN IP, hostname). Which belongs in a template body and which in variables?
  2. Why is $(branch_lan_ip) in a template safer than remembering to edit the right number in two places?
  3. What should happen if a template references a variable that has no value for a target device — install anyway, skip the line, or refuse? Which behaviour do you want?
  4. Templates and the device database can both set the same setting. Predict the ordering problem and why FMG needs a defined precedence.

Stop reading and write your predictions before continuing.

Mental model — the mail-merge letter

A CLI template is a mail-merge: one letter, with $(fields) where the recipient's details go, and a spreadsheet of per-recipient values (metadata variables). The genius isn't saving typing — it's that the structure of every letter is provably identical, so a mistake is either in one template (fix once, fixes everywhere) or one row of the spreadsheet (scoped to one device). Compare that to thirty hand-typed letters, each subtly different.

Then a CLI template (assigned to both branches) — note the $(variable) syntax:

# Provisioning Templates → CLI Templates → Create New → "TPL_BRANCH_BASE"
config system global
    set hostname $(branch_hostname)
    set admintimeout 30
    set timezone "Australia/Sydney"
end
config system interface
    edit port2
        set ip $(branch_lan_ip)
        set allowaccess ping
        set role lan
    next
end
config system dhcp server
    edit 1
        set interface port2
        set default-gateway $(branch_lan_ip)
        config ip-range
            edit 1
                set start-ip $(branch_dhcp_start)
                set end-ip $(branch_dhcp_end)
            next
        end
    next
end
Careful

default-gateway $(branch_lan_ip) is a deliberate bug for you to find: the variable holds x.x.x.1/24 but the DHCP gateway wants a bare IP. Either add a second variable (branch_lan_gw) or store IP and mask separately. Finding this in the install preview — not on a dead branch — is the whole lesson.

Assign under Provisioning Templates → Template Group (or assign the CLI template directly to both devices), then install via the wizard and read the preview.

Verification

# FMG: install preview shows the rendered, per-device CLI — variables substituted.
# On each branch after install:
get system status              # hostname correct?
show system dhcp server

Reflection

  • Which failure did the preview catch that would have been service-affecting? What does that teach about where in the pipeline validation belongs?
  • Branch C arrives next quarter. List the complete onboarding steps now — count how many are "create values" vs "create structure".
  • When is a CLI template the wrong tool compared to the GUI system templates — think readability, drift-detection, and who on the team can maintain it?
Challenge exercise

Add a site_id variable (BRA=20, BRB=30) and rewrite your template so every subnet derives from it (10.$(site_id).10.1/24 …). What have you gained, and what new class of mistake have you invented? This tension — abstraction vs transparency — is a design judgement, not a right answer.
