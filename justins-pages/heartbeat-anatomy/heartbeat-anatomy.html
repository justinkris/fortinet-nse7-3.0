What the frame itself looks like

Destination MAC
6 bytes — the partner's physical hbdev MAC (heartbeat ports never get a vMAC)
Source MAC
6 bytes — the sender's physical hbdev MAC
EtherType
2 bytes — 0x8890 / 0x8891 / 0x8893 instead of 0x0800 (IPv4)
FGCP Payload
Proprietary Fortinet data: hello/election state, monitored-interface tallies, sync data. No IP header inside.
FCS
4 bytes — checksum

The key observation: where an ordinary frame says "0x0800 — an IP packet lives inside me," an FGCP frame says "0x8890 — Fortinet cluster business, nothing above layer 2." That's why diagnose sniffer packet needs the "ether proto 0x8890" filter — an IP-based filter would never match it.

Inside the FGCP payload

⚠ Honest caveat: FGCP is proprietary — Fortinet does not publish the byte-level layout of the payload, and neither the study guide nor the exam tests its internal structure. What we can map confidently is what information rides inside it, based on the functions the guide attributes to each ethertype.
0x8890 / 0x8891 — heartbeat payload carries:
  • Cluster identity — group ID, group name hash, password verification (so strangers can't join)
  • Hello / liveness — "I'm still here" at a regular interval; absence = device failover
  • Election parameters — monitored-interface up-count, HA uptime, priority, serial number (the four criteria, carried so both members can run the same election and agree)
  • Failure & redistribution signals — role changes, traffic redistribution coordination
0x8893 — config-sync payload carries:
  • Configuration checksums — how members detect in-sync vs out-of-sync
  • Configuration deltas — the actual settings pushed primary → secondary
  • External files — certificates and similar objects that must match across members
Session-table sync also rides the hbdev by default — unless session-sync-dev moves it to its own lane.

Mental model: the payload is the cluster's ballot paper plus pulse — every frame re-states "who I am, that I'm alive, and how electable I currently am." The election never really ends; it's continuously re-run with fresh facts.

Why Ethernet-typed, not IP

No IP stack needed

FGCP frames don't need routing, ARP resolution, or an IP address on the hbdev to function — they operate one layer below all of that. The cluster can talk before "networking" in the usual sense even exists on that link.

Isolation from data traffic

Because it's a distinct ethertype, ordinary IP traffic and tooling never accidentally intercepts or is confused with heartbeat frames — and you can filter for it precisely with a sniffer.

The two failure signals it produces

🖤

Device Failover

Heartbeat frames stop arriving entirely. Partner is presumed gone. Whole box changes role.

🔌

Link Failover

Heartbeat is fine — but the monitored-interface tally carried inside it changes. Re-election runs anyway.

What rides along separately

Session sync — default path

By default, session-table data shares this same hbdev link. Under heavy load, that bulk data can crowd out the small, latency-critical heartbeat frames — the false-failover trap from Session 46.

Session sync — dedicated path

Configuring session-sync-dev moves the bulk traffic off the heartbeat entirely. If those dedicated links all fail, sync quietly falls back to the hbdev — no alarm raised.

One-line anchor: the heartbeat is a narrow, dedicated nervous system (0x8890/91/93) — thin, fast, and easily drowned out by anything bulky sharing its lane. That's why Fortinet gives you the option to give it a lane of its own.
