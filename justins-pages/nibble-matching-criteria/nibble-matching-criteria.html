What a Rule Can See

One minute. The menu of SD-WAN rule matching criteria — and the two that explain everything else.

Match onCLI
Source address / user / groupset src · set users · set groups
Destination addressset dst
Destination service (ISDB)set internet-service enable + set internet-service-name
Destination applicationset internet-service-app-ctrl
Protocol + port rangeset protocol · set start-port / set end-port
DSCP / ToSset tos · set tos-mask
Incoming interfaceset input-device
Why this beats a static route: a static route matches destination only. This whole table is the gap SD-WAN rules exist to fill — "who is asking" and "what kind of traffic" become steering inputs, not just "where to."
The highlighted row is a bomb. Traffic the FortiGate originates itself — syslog, FortiGuard, DNS, SNMP — never ingressed. No incoming interface, no client source address: nothing on this menu for a rule to read. So local-out traffic bypasses SD-WAN rules by default and takes a plain routing-table lookup. Opt it in per feature: set interface-select-method sdwan (values: auto | sdwan | specify). There is no global switch.
ISDB vs app-ctrl, one line: ISDB is a destination lookup — matches on the first packet. App-ctrl needs to inspect (Client Hello / SNI / cert) — the first session escapes, then the learned destinations are cached. The first video pays for everyone else's.
🧠
Mental note
A rule can only match what the packet presents. Ask "what does this packet carry with it?" and the local-out bypass, the app-ctrl delay, and the ISDB advantage all fall out of the same question.
