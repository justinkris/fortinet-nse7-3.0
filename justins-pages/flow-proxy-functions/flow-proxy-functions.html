NSE7 EF 7.6 · Session 39 · Study Bite

What Runs Where — Flow vs Proxy Functions

Know which features live in which mode — and what it costs you in hardware acceleration.

The mode determines the path. The path determines the performance.
Every security feature you enable is a choice between depth of inspection and hardware offload.
Core Distinction

Two Modes, One Critical Difference

Both flow-based and proxy-based inspection read full packet payloads — including Layer 7 application content. The difference is not depth. It is timing.

⚡ Flow-Based — Real-Time
  • IPS / Intrusion Prevention
  • Application Control
  • Antivirus (flow mode)
  • Web Filtering (flow mode)
  • DNS Filtering
  • Botnet C&C detection
  • SSL Deep Inspection (flow)
  • File Filter (flow mode)
🔍 Proxy-Based — Buffer First
  • Antivirus (proxy mode — full file)
  • Web Filtering (with quota)
  • Web Category Usage Quota
  • Web Profile Override
  • Antispam
  • Data Loss Prevention (DLP)
  • VoIP Inspection
  • Explicit Web Proxy
Hidden trap: Some features appear in both modes but behave differently. Antivirus in flow mode scans as data streams through. Antivirus in proxy mode buffers the complete file before scanning — catching threats that only appear when fully assembled. Same feature name, completely different behaviour and hardware path.
Hardware Acceleration Impact

Which Mode Gets NP7 Offload?

Policy Configuration NP7 Offload CP9 Involved Path
No security profiles ✅ Full offload No NP7 fast path
Flow-based IPS / App Control ✅ NTurbo Yes — IPSA pattern matching NP7 → NTurbo → IPS → CP9 → NP7
Flow-based SSL deep inspection ✅ NTurbo Yes — SSL crypto offload NP7 → NTurbo → IPS → CP9 → NP7
Proxy-based any profile ❌ None Yes — SSL crypto only CPU slow path entirely
Mixed flow + proxy profiles ❌ None Yes — SSL crypto only CPU slow path — proxy wins
np-acceleration disabled ❌ None Possible CPU slow path — explicit override
Exam-critical — All or Nothing: A policy with mixed flow-based and proxy-based profiles does not split traffic between NP7 and CPU. The entire session goes to the CPU. Any proxy profile on a policy kills NP7 offload for every session matching that policy.
🧠 Mental Model
Flow-based = inspect as it flows → NP7 can help via NTurbo. Proxy-based = buffer everything first → CPU only, no exceptions.
Who Does What

The Three Processors and Their Jobs

Processor Primary Role Works With Cannot Do
NP7 Session forwarding, NAT, IPsec crypto, ACL, HPE, IP integrity Flow-based profiles via NTurbo Proxy-based inspection, session helpers, SSL VPN / GRE tunnels
CP9 Pattern matching (IPSA), SSL/TLS crypto for deep inspection, IPsec DH key exchange Both flow and proxy modes Full session forwarding — it is a coprocessor, not a forwarder

CP9 works in both modes: Even when a proxy-based policy forces everything to the CPU, the CP9 can still offload SSL/TLS encryption and decryption if deep inspection is configured. The NP7 is excluded — the CP9 is not.
Slow Path Triggers

Four Reasons NP7 Won't Offload

#TriggerWhy
1Proxy-based security profiles on the policyBuffering requires CPU — hardware cannot hold full streams
2Session helpers required (FTP, SIP, PPTP etc)Dynamic port negotiation requires CPU to read application payload mid-session
3Tunnelled interfaces — SSL VPN, GRE (not IPsec)Software-defined logical interfaces managed entirely by CPU
4np-acceleration explicitly disabled on the policyDirect hardware override — useful for troubleshooting, dangerous in production
IPsec is the exception: IPsec VPN tunnels can be offloaded to the NP7 because the NP7 has dedicated crypto engines that understand IPsec natively. SSL VPN and GRE cannot — they are pure software constructs.
Self-Test

Reconstruct the Mental Model

A policy has flow-based IPS and proxy-based DLP. Which path does traffic take?
CPU slow path entirely. Any proxy-based profile on a policy kills NP7 offloading for every session matching that policy. It is all or nothing — there is no splitting of work between NP7 and CPU.
Can the CP9 ever help with a proxy-based policy?
Yes. Even in proxy mode the CP9 can offload SSL/TLS encryption and decryption if deep SSL inspection is configured. The NP7 is excluded from session forwarding — but the CP9 still assists the CPU with crypto operations.
Why can FTP sessions not be offloaded to the NP7?
FTP uses a session helper. The helper must read inside the application payload of the control connection to extract the dynamically negotiated data port, then open a pinhole for the secondary connection. This requires CPU involvement for the entire session — the NP7 silicon cannot do application-layer protocol tracking.
What is the one tunnel type that CAN be offloaded to the NP7 — and why?
IPsec VPN. The NP7 has dedicated hardware crypto engines that natively understand IPsec encryption and decryption. SSL VPN and GRE are software-defined logical interfaces managed entirely by the CPU and cannot be offloaded.
Name two features that are proxy-only and would silently break NP7 offloading if added to a flow-based policy.
Web category usage quota and VoIP inspection are both inherently proxy-only. Adding either to an otherwise flow-based policy forces the entire policy to proxy mode, killing NP7 offloading for all sessions. The policy inspection mode setting is only the default — individual features can override it silently.
