SESSION 57 · APPLIED SCENARIO · DC · SYDNEY · MELBOURNE

Shift Change or Funeral?

BGP convergence across IPsec tunnels to an HA cluster and a standalone peer — and why they need opposite tools.

At second zero, a failover and a site death are indistinguishable.

You cannot configure two behaviours for two events you cannot tell apart. So you choose where the boundary sits.

THE ENVIRONMENT

Three Sites, Two Tunnels, Two Very Different Peers

One hub, two spokes — and two completely different failure profiles.

The hub is a 2601F HA cluster with NP7 processors and IPsec offloaded into silicon. It builds two IPsec tunnels and runs a BGP peering inside each: one to Sydney, an Active-Passive cluster running as a virtual machine, and one to Melbourne, a single standalone FortiGate.

Everything interesting about this design follows from one asymmetry. The two spokes have identical connectivity and completely different failure profiles. Sydney can experience an event that looks like death but is not. Melbourne cannot.

The FGCP fact that shapes everything below: FGCP synchronises IPsec security associations across the cluster. When Sydney fails over, the tunnel survives. The BGP daemon does not — it runs only on the primary, in a userland process, and processes do not fail over.

MENTAL NOTE
Same hub, same tunnel technology, opposite convergence tools — because the expected failure differs per neighbour.
SCENARIO 01

Sydney Fails Over — The Shift Change

What actually happens, in order:

  1. Sydney's primary fails. FGCP promotes the secondary.
  2. The IPsec SAs are already synchronised, so the tunnel comes back almost immediately — often without the DC noticing at the tunnel layer at all.
  3. The BGP daemon does not survive. Its state machine, its RIB and its negotiated capabilities lived in the memory of a process on the dead unit. The new primary starts a fresh daemon that has never spoken to the DC.
  4. From the DC's perspective, the TCP session on port 179 dies.
  5. Without graceful restart: the DC withdraws every route learned from Sydney. Traffic to Sydney stops — not because forwarding is broken, but because routing announced a death that was really a change of shift.
  6. The peering rebuilds, routes are re-advertised, and service returns after a full BGP convergence that was never necessary.
The correct tool: graceful restart. The cluster tells the DC in advance — as a negotiated capability during the OPEN exchange — that if its session drops it is restarting rather than dying. The DC marks the routes stale, keeps forwarding on them, and waits. The new primary comes up with a usable table and the traffic never stopped.
Sydney side — advertise the capability
config router bgp
    set graceful-restart enable
    config neighbor
        edit <DC-peer-ip>
            set capability-graceful-restart enable
        next
    end
end
DC side — must also be enabled, or the capability is never agreed
config router bgp
    set graceful-restart enable
    config neighbor
        edit <Sydney-peer-ip>
            set capability-graceful-restart enable
        next
    end
end
The trap on this neighbour: enabling aggressive BFD here because it worked on the carrier links. Sydney fails over, the DC declares death at 750 ms, the session is torn down, routes are withdrawn — and graceful restart never gets to speak. You will have configured a feature and then configured a second feature that silently overrules it.
🧠
MENTAL NOTE
A change of shift, announced as a death. Graceful restart is the memo sent in advance.
SCENARIO 02

The Sydney Cluster Falls — Genuine Death

Both members gone. Power event, hypervisor failure, site loss. There is no secondary to promote and nothing is coming back inside a maintenance window.

Now every instinct reverses. Graceful restart, which was the hero of Scenario 1, becomes actively harmful — the DC holds stale routes and keeps forwarding into a site that no longer exists, for the duration of the restart timer. What you want here is exactly what you did not want a moment ago: withdraw the routes, and withdraw them fast.

And here is the design problem in one sentence: at second zero, these two scenarios are indistinguishable from the DC. Both look identical — packets stopped arriving. You cannot configure two different behaviours for two events you cannot tell apart.

Which means the real question is not "which tool for which scenario." It is:

What makes them distinguishable, and how long am I willing to wait to find out?

The answer is a deliberately chosen tolerance window. You set your detection so that a routine Sydney failover falls inside it and a genuine site loss falls outside it. Graceful restart handles everything inside the window without disturbing the routing table. Anything still silent when the window closes is treated as real death and the routes go.

🧠
MENTAL NOTE
Failover and site death are identical at second zero. The design lever is not detection speed — it is where you place the boundary between them.

Sydney: setting the window

If you use BFD on the Sydney peering at all, the intervals must sit comfortably above the cluster's realistic failover-and-reconverge time. Measure it — do not guess. Fail Sydney over deliberately during a maintenance window and time how long the BGP session is genuinely unavailable. Then set your detection well outside that figure. Anything tighter converts every planned patch into an unplanned reroute.

The hardware asymmetry that sets your floor: the 2601F offloads IPsec into NP7 silicon and has control-plane headroom to spare. The Sydney VM performs encryption, tunnel handling, the BGP daemon and any BFD state machine in software on shared vCPU. Because BFD negotiates the actual interval as the greater of the sender's desired TX and the receiver's required RX, the VM's declared capability governs both directions. The slower peer sets the floor — always.

Melbourne: the simple case

No cluster means no failover means graceful restart protects against nothing. BFD is straightforwardly appropriate — subject to the usual test: is there an alternative path to Melbourne, or a second tunnel? If not, BFD there is a monitoring improvement rather than an availability one, and it should be described that way in the design document.

Other convergence levers worth applying here

  • Keep the underlay stable. Flapping tunnels generate BGP updates that consume CPU on the very device least able to spare it.
  • Filter the RIB. Inbound and outbound prefix filters reduce the table each spoke must process — the cheapest convergence improvement available and the one people skip.
  • Do not tune BGP timers as a substitute. On a VM spoke this is the worst option: it multiplies software work on the constrained peer and buys false positives.
  • Verify, do not assume. get router info bfd neighbor and get router info bgp neighbors <ip> confirm that BFD is Up and that graceful restart was actually negotiated rather than merely configured.
🧠
MENTAL NOTE
Measure Sydney's real failover duration before choosing a timer. A guessed threshold turns every planned patch into an incident.
