SESSION 64 · BITE · SESSION TABLE ANATOMY

The Session Table Never Lies

The dashboard shows the rule's opinion. The session table shows what each conversation is actually doing.

A session is a decision that was already made — and cached.

SD-WAN rules decide the path for the first packet. Every packet after that trusts the session table. When the network changes, the gap between those two is where "old calls sound like the ocean" lives.

02 · Field-by-Field

What Each Line Is Evidence Of

FieldMeaningSD-WAN significance
proto / proto_state IP protocol number and its state machine. TCP proto=6 uses two digits (e.g. 01 = ESTABLISHED); UDP proto=17 uses 00 (no reply yet) or 01 (reply seen). Tells you whether the conversation is genuinely two-way. Voice/RTP = UDP, so no retransmission hides a bad path — users hear it.
duration / expire / timeout Age of the session · seconds until it dies if idle · the configured idle timer. A long duration that predates a failover event is your first hint the session was born on the old path. It will keep refreshing expire as long as packets flow — a limping session can live forever.
state= flags Behavioural flags. Key ones: may_dirty, dirty, npu, log, local. The failover story lives here. may_dirty = eligible for re-evaluation. dirty = a change has occurred; re-run route + policy lookup on the next packet. See section 03.
statistic org= / reply= Bytes/packets each direction. Counters incrementing both ways = path exists. Org climbing, reply frozen = one-way traffic — classic broken-return-path or dead-egress signature.
dev=7->3/3->7 Interface index pairs: original direction in→out, reply direction in→out. The physical answer to "which interface is this session actually using?" Map indexes with diagnose netlink interface list. Cross-check against sdwan_mbr_seq.
gwy= Next-hop gateway(s) cached for this session. The session's routing decision, frozen at creation (or last re-evaluation). If the routing table says one thing and gwy= says another — the session hasn't been re-evaluated.
hook / act=snat / dnat NAT actions per direction, with the translated address in parentheses. Critical. act=snat means this session's public identity is welded to one egress link's IP. Moving the session to another member means a new source IP — the far end has never heard of it. This is why SNAT sessions get special treatment on routing changes.
policy_id Firewall policy that matched at session creation. Proof that policy lookup is cached too. First packet: route lookup + policy lookup. Every later packet: session table only. Re-evaluation re-runs both.
sdwan_service_id The SD-WAN rule (service) that steered this session. Which line of Riley's clipboard made the decision. If these two fields are absent, the session matched no SD-WAN rule — it fell through to the implicit rule / plain FIB routing.
sdwan_mbr_seq The SD-WAN member the session is pinned to. The single most direct answer to "which road is Jam on?" Compare it with the member the rule currently prefers (diagnose sys sdwan service). A mismatch is not a bug — it is a cached decision that hasn't been invalidated.
Absence is evidence. No sdwan_mbr_seq / sdwan_service_id in a session that should be steered means the rule never matched it — check for a catch-all above it, wrong matching criteria, or local-out traffic. You learned all three ways this happens in Session 63.
02b · The Other Side of the Comparison

Reading diagnose sys sdwan service

The session table tells you where a session is going. This command tells you where the rule would send a new one. Troubleshooting a pinned session means holding both outputs side by side. Each member line has four things, that is all:

diagnose sys sdwan service 3 — member lines, during the 09:15 brownout
Members(2):
  1: Seq_num(1 port1), alive, sla(0x1), gid(0), cfg_order(1), cost(0), selected
  2: Seq_num(2 port2), alive, sla(0x0), gid(0), cfg_order(0), cost(0)
PieceMeaningTrap it defuses
Seq_num(1 port1) Member number and its interface. Just the name tag. Cross-reference target for the session's sdwan_mbr_seq — same numbering.
alive The health check's probes are coming back. The link is up. A browned-out link is still alive. Up is a fact — not a verdict.
sla(0x1) / sla(0x0) Bitmap of which SLA targets this member currently passes. 0x1 = passes target 1. 0x0 = passes none. The verdict. alive + sla(0x0) together = "up but failing thresholds" — the exact signature of a brownout. Up is a fact; good is a measurement.
selected The rule's current choice for new sessions. Only one member per line carries it (in mode sla). This is the dashboard's "failover complete." It is a promise about future sessions only — it says nothing about sessions already in the table.
The two-output ritual: selected on member 1 (rule's answer) + sdwan_mbr_seq=2 in the session (session's answer) = the failover happened and the old session didn't follow it. Neither output is wrong. They answer different questions: where would a new session go vs where is this session going.
03 · The Flags That Decide Failover

may_dirty, dirty, and the Right to Change Your Mind

FlagWhen setWhat it means
may_dirty At session creation, when the session was matched by a policy and its routing can legitimately change (the normal case). Permission slip: "if the world changes, you may re-evaluate me." Without it, the session will never be reconsidered.
dirty When a relevant change occurs — a routing-table change, an SD-WAN member/rule change, a policy change — FortiGate stamps dirty onto sessions that carry may_dirty. Marching orders: on the next packet, re-run the route lookup and the policy lookup. The session then either re-pins to the new best path or dies at policy if no policy allows the new path.
npu Session offloaded to hardware. Offloaded sessions get pulled back to the CPU for re-evaluation when flagged dirty. Also: an offloaded session's counters may update lazily — don't panic at stale-looking numbers.
local Traffic originated by the FortiGate itself. Local-out — the cobbler's children. Bypasses SD-WAN rules unless a feature opts in with interface-select-method.
The gym membership card
Your first visit, the front desk checks your ID, your payment, and assigns you a locker (route lookup + policy lookup + member). Then they hand you a card. Every visit after that, you swipe the card — nobody re-checks anything. may_dirty means your card can be revoked. dirty means the gym changed its rules overnight and your card is flagged: next swipe, you go back to the front desk for a full re-check. The question Session 64 answers: which events actually flag the card — and why does paying by direct debit (SNAT) change the answer?
04 · The Ritual

Reading a Session with SD-WAN Eyes

When Riley says "old calls are broken, new calls are fine," this is the order a senior engineer reads a stuck session:

  1. Filter first. diagnose sys session filter dst 198.51.100.20 (also src, dport, policy) then diagnose sys session list. Never read the whole table.
  2. Which rule? — sdwan_service_id present? Right rule? Absent = fell through.
  3. Which member? — sdwan_mbr_seq vs the member the rule prefers now (diagnose sys sdwan service). Mismatch = pinned session.
  4. Is it NAT'd? — act=snat means its public identity belongs to that member's link. Note it before predicting behaviour.
  5. Could it move? — may_dirty present? Is dirty ever appearing? If the permission exists but the flag never fires, ask what kind of change occurred — and whether that kind of change triggers re-evaluation at all.
  6. Corroborate. — dev= pair and gwy= should agree with sdwan_mbr_seq. Counters both directions confirm the (bad) path still passes traffic.
The mismatch is the finding. Dashboard says the rule prefers member 1. The session says sdwan_mbr_seq=2. Neither is lying — they are answering different questions. The rule answers "where would a new session go?" The session table answers "where is this session going?" Old calls sound like the ocean precisely because those two answers are allowed to differ.
Open question this bite deliberately does not answer: the SLA breached, the rule swung, may_dirty is set — so why was Jam's session never stamped dirty? What kind of change happened at 09:15, and what kind of change does the dirty machinery actually listen for? That is the heart of Session 64.
