Four Containers, and Why They Decide Everything

RIB-in, Local RIB, RIB-out, routing table — and the bucket that fills one of them.

Four containers. Believing and telling are different drawers.

Almost every BGP fault you will meet is not “BGP is broken.” It is a prefix sitting in the wrong container, perfectly healthy, and silent.

First · a terminology check

Two container metaphors, and they are not the same thing

Both arrived in the same session, so it is worth separating them before anything else.

MetaphorWhat it describesWhere it came from
The drawers (RIBs)The four containers a prefix passes through inside a router. This page is about these.Señor's filing cabinet, Session 59. Study guide pp. 340–341.
The bucket (and the tap)Two methods of originating a prefix — redistribute versus config network. A single point on the journey, not the journey.J.R., Session 60. Study guide pp. 346–347.

The bucket has its own section near the bottom. Everything before that is the drawers — and the reason they matter is simple: they turn “BGP isn't working” into a question with only four possible answers.

🧠
Why this is the highest-value model you own
A prefix problem is always one of: never arrived · arrived but not chosen · chosen but not spoken · spoken but not used. Naming the container first collapses the search space before you have run a single command.

Where Does the Prefix Stop?

Seven faults you have already met this season. Select one and watch which container it dies in — then notice how many of them produce no log, no error and a perfectly healthy session.

🧠
The pattern worth seeing
Faults in containers 1 and 2 hide the prefix from you. Faults at gate 2 hide it from everyone else while it looks perfect on your screen. That single distinction tells you which end of the cabinet to open — and it is the difference between a ten-minute fix and a two-hour one.
Worked examples · name the container before you reveal

Six Scenarios from Your Own Incidents

Each of these actually happened in Sessions 55–60. Say the container out loud, then open the card.

Session 60London traffic transits Singapore at 340ms. Both paths are 2 AS hops. Nothing is broken.Reveal ▾
Container

RIB-in. Both paths are sitting in the waiting room; the question is which one gets through gate 1. This is a best-path question, and best-path is a RIB-in → Local RIB event.

Diagnosis

Paths: (2 available, best #2). Walk the tie-breaker list top-down. They differ nowhere until line 10 — lowest router ID. Not a fault. An absence of policy.

Fix

Raise LOCAL_PREF inbound with a route-map on the EU carrier neighbour, so the argument ends at line 2 and line 4 is never reached. And note where the route-map runs: at gate 1, before the comparison.

Session 60Your own 10.55.10.0/24 is configured under config network. Nobody anywhere sees it. It works perfectly internally.Reveal ▾
Container

The routing table — container four — even though the symptom is about advertisement. This is the one that catches people.

Diagnosis

get router info routing-table all shows 10.55.0.0/16 connected. The network statement asks for a /24. network-import-check demands an exact prefix and mask on an active route. Covered is not matched.

Fix

Add a static route matching exactly. Do not disable the check — that switch is global and silently removes the safety net from every network statement on the box, forever.

Session 60You add a prefix-list-out with one permit rule to advertise the London subnet. Eleven other prefixes vanish from that peer.Reveal ▾
Container

Gate 2, Local RIB → RIB-out. The eleven prefixes are untouched, valid, best and installed. They are simply no longer spoken to this one neighbour.

Diagnosis

Every FortiOS filtering construct — route map, prefix list, access list — ends in an implicit deny. One permit rule does not mean “also allow this.” It means “allow this and nothing else.”

Fix

Add the terminator: set prefix 0.0.0.0 0.0.0.0 / set le 32 / set action permit. And form the habit: ask what a filter silently kills, not what it allows.

Session 59Adelaide is a route-reflector client with exactly one IBGP session, to HQ. Perth advertises a new subnet to HQ over ordinary IBGP. Does Adelaide learn it?Reveal ▾
Container

HQ's RIB-out toward Adelaide. The route is safely in HQ's Local RIB — the only question is whether HQ is willing to say it.

Diagnosis

Yes. If the reflector did not pass non-client routes down to its clients, the cluster would be a sealed box and Adelaide would learn nothing from outside it.

Fix

No fix needed — but hold the compressed rule: a route is relayable if a CLIENT is on either end. Non-client to non-client is still plain IBGP.

Session 58A prefix appears in the BGP database with a * but no >. The session is Established and no errors are logged.Reveal ▾
Container

RIB-in. It arrived and was accepted — but it never made it through gate 1.

Diagnosis

It is not losing a competition; it is ineligible to enter one. Read the NEXT_HOP column, then get router info routing-table details <next-hop>. If the next hop cannot be resolved, the path can never be best.

Fix

Resolve the next hop via IGP or a static route, or rewrite it with set next-hop-self enable. And remember the counter-lesson: absence of errors is not evidence of correctness.

Session 56Two carrier defaults, identical attributes. You enable ebgp-multipath. Is the idle circuit now redundancy?Reveal ▾
Container

Gate 1 into the Local RIB — and then, importantly, container four.

Diagnosis

Equal attributes are the permission that makes multipath legal; the config command is the instruction that makes it happen. Both are required. But ECMP is load sharing, not redundancy.

Fix

None — understand the trade. A route only leaves the routing table when BGP withdraws it, which needs session teardown, which needs the hold timer (180s). For those 180 seconds ECMP hashes roughly half your flows onto a dead path. You have traded a total outage for a partial one, and partial outages are harder to diagnose because the phones only half ring.

The other metaphor

The Bucket and the Tap

This one is not a container in the journey — it is a choice about how a prefix gets injected into the Local RIB in the first place. By default, FortiGate BGP originates nothing at all.

The bucket — redistributeThe tap — config network
What it takesA whole category: connected, static, RIP, IS-IS, BGPOne named prefix at a time
PrecisionNone by itself — filter afterwards with a route-mapExact and auditable
Failure modeYou announce your management network, your guest wireless and three VPN tunnels in one commandSilent non-advertisement when network-import-check finds no exact match
Reach for it whenYou genuinely want everything in a category, or you already have tight filteringYou know precisely what you want to say — which is most of the time
Both, side by side
config router bgp
    config redistribute "connected"     <-- the bucket: a whole category
        set status enable
    end
end

config router bgp
    config network                      <-- the tap: one named prefix
        edit 1
            set prefix 10.55.10.0 255.255.255.0
        next
    end
end

network-import-check (ENABLED by default) reads the ROUTING TABLE,
not the Local RIB, and demands an EXACT prefix and mask on an ACTIVE route.
🧠
Mental note
The bucket and the tap both empty into container two. Origination is an injection into the Local RIB — it never passes through RIB-in, because your own prefixes were never told to you by anybody.
The drill

Name the Container Before You Diagnose

The habit is one sentence, spoken before any hypothesis and before any command:

“The prefix is in the ______, so this is a ______ question.”

It sounds like pedantry. It is not. It is the thing that stops you from retrieving a fault you know well when the state you were given is different — which is the single most expensive habit available in both troubleshooting and the exam.

For the boss battle: the symptom in Session 61 is that one prefix is visible from Melbourne and Brisbane and invisible from Perth, Adelaide, Singapore and the EU carrier — with three firewalls each internally consistent. Look at the table above and notice that no single row explains that. An asymmetry across peers that splits along cluster lines needs at least two containers in the answer. Start there.
