FortiManager — The Architect's Office

One firewall is a single house. FortiManager holds the blueprints for the entire estate. You don't walk to every house with a hammer — you update the blueprint, then deploy it everywhere.

Stop thinking of FortiManager as a bigger firewall. It configures nothing directly.

It is a design office. Changes live in a database until you deliberately Install them. Understanding that gap is the whole game.

The Core Analogy

The Architect, not the Builder

A FortiGate is like a single house — you can walk in and change one thing.

A FortiManager is the architect's office that manages the blueprints for an entire housing estate. Update the blueprint once, then build it out to every house.

This is why the mental model matters: FortiManager doesn't do the firewalling. It designs and distributes what the firewalls will do.

The Central Distinction

Who the firewall is vs how it behaves

Every object in FortiManager falls into one of two buckets. Get this split right and the whole product clicks into place.

Device Manager — Identity

Like HR records. Unique to each firewall.
  • IP addresses
  • Interfaces
  • DNS
  • Routes
  • HA configuration

Policy & Objects — Behaviour

Like the company handbook. Shared by many.
  • Firewall policies
  • IPS profiles
  • Web filter profiles
  • Address / service objects
  • VPN & security rules
Objects are variables. Instead of typing 10.1.1.0/24 in fifty places, you write LAN once. Change the variable, and every policy referencing it updates. Templates are the "building standards" (white walls, smoke alarms, exit signs) that configure the firewall itself; Policy Packages are the "company rulebook" of traffic behaviour — and a data-centre branch can simply be handed a different rulebook.
The Clever Bit

Dynamic Mapping — Mail Merge for firewalls

One template says Dear <<Customer>>. Each letter substitutes a different name. A single policy says LAN → Internet. Dynamic Mapping substitutes a different LAN value per firewall.

policy:
  src = LAN
  dst = Internet
  action = accept
→
Firewall 1 · Branch
LAN = 10.1.1.0/24
Firewall 2 · Branch
LAN = 172.16.0.0/24
Same template. Same policy name. Different values delivered to each firewall.
Nothing Is Live Until You Say So

The Install Pipeline — editing is not printing

FortiManager is the document editor. Typing into a document doesn't print it. The firewall doesn't change until you hit Install — and Install Preview is your Git diff: inspect exactly what will change before touching production.

Step 1
Edit
Change lives only in the FMG database
→
Step 2
Preview
Diff: − removed, + added
→
Step 3
Install
Push to the firewall
→
Step 4
Firewall changes
Running config finally updates
Revisions are your Time Machine / Git history — who changed it, when, and what. Made a mistake? Roll back. Workspace mode is library checkout: without it, five people overwrite each other's pages; with it, one person owns the changes at a time (check out → edit → return). Scripts are macros — rename 500 objects without doing it by hand.

Think Like a Senior Engineer

Troubleshooting — follow the parcel

"My package never arrived." You don't start at the front door — you follow the journey: warehouse → truck → depot → courier → customer. FortiManager troubleshooting is identical. Don't jump to the end. Follow the config's journey and find the first broken link in the chain.
1
Does the change exist in the FortiManager database? (Did the edit actually save?)
2
Was the change queued to install? (Editing ≠ deploying.)
3
Did the install complete? (Or did it fail / partial?)
4
Did the firewall accept it? (Out-of-sync? Rejected object?)
5
Is it in the firewall's running config? (Verify on the FortiGate itself.)
✓
Is traffic actually matching the policy? (The real business symptom.)
