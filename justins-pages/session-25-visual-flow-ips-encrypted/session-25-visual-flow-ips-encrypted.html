The core of your question

Why flow mode doesn't need the whole file

You reasoned it yourself: "IPS is just matching a signature and seeing behaviour — which doesn't require the whole file. AV would." That's exactly right, and here's the mechanism underneath it.

A signature is a pattern — a string like system('dir'), a byte sequence, or a rate-based cadence. You can recognise a pattern the moment it streams past you. You don't need the end of the letter to spot a threatening phrase in the middle of it.

So flow-based inspection keeps only a small sliding buffer — just enough context to catch a signature that spans a few packets — matches as the bytes go by, and lets the traffic keep moving. The complete object is never assembled. That's why it's fast enough to sit in front of a market-data feed that cannot tolerate delay.

Flow-based (IPS)
Match the stream
  • Inspects packets as they stream, in real time
  • Small sliding buffer, not the full object
  • Signature = pattern → visible mid-stream
  • SSL: decrypts in-line, one connection
  • Low latency, low memory — feed-safe
Proxy-based (often AV)
Reassemble, then inspect
  • Buffers the whole file first
  • Needed to hash or unpack an archive
  • Can't judge a file it hasn't fully seen
  • SSL: terminates two TLS sessions
  • More thorough, more latency + memory
Both modes can do full inspection. That was the Session 20 table: full (deep) inspection works in flow and proxy mode alike. Flow decrypts in-stream without splitting the connection; proxy terminates two independent TLS sessions with plaintext in the gap. Same visibility, different plumbing.
SSL inspection · under the hood

How the seal comes off in flow vs proxy

Both inspection modes can perform full (deep) SSL inspection — the IPS engine ends up with the same plaintext to scan either way. What changes is the connection shape, the memory profile, and which security services the mode was built to feed.

Flow-based SSL inspection
Decrypt in-stream · one logical flow
  • FortiGate decrypts packets as they arrive, feeds plaintext to the scanner, re-encrypts on the way out
  • No full-object buffering — only a small sliding window of bytes lives in memory at any moment
  • The forwarding path stays intact: a single logical flow with FortiGate as the invisible middleman
  • Lower CPU and memory — safe in front of latency-sensitive traffic like a market-data feed
  • Best pairing: IPS, application control, web filter — anything pattern-based
Proxy-based SSL inspection
Terminate · reassemble · re-originate
  • FortiGate terminates two independent TLS sessions — one to the client, one to the server
  • Buffers the full object in plaintext between the two sessions before it forwards
  • Required for anything that needs the whole file: AV hashing, archive unpacking, DLP fingerprinting
  • Higher CPU, memory, and latency — trade-off is thoroughness and file-level visibility
  • Best pairing: antivirus, DLP, content-heavy web filtering
Same certificate, same key material. Whichever mode you choose, FortiGate presents its own certificate to the client (signed by a CA the client trusts) and acts as an authorised man-in-the-middle. What differs is how many TLS sessions exist and how much of the object is held in memory when the scanner reads it.
Two knobs, not one. The SSL/SSH inspection profile on the policy sets depth — certificate vs full. The inspection mode (flow vs proxy) is set on the security profiles themselves — the AV, IPS, web-filter, and app-control profiles each carry their own mode flag. Deep inspection can run in either mode; picking flow vs proxy is a separate decision from picking certificate vs full.
The trap · don't conflate the axes

Full inspection ≠ proxy mode

Full (deep) SSL inspection and proxy-based inspection are two independent things. Full inspection is a depth — does the FortiGate decrypt the payload or not. Flow-vs-proxy is an inspection mode — how the engine handles the traffic. Two separate axes, and full inspection works in both modes.

"Full inspection = proxy mode only" is FALSE. Here's the mechanism that makes it false:
Flow + full inspection
One continuous connection
  • FortiGate decrypts in-stream — cracks the seal as bytes go by
  • Hands the plaintext to the IPS engine, re-encrypts on the way out
  • No splitting the session — one logical flow, invisible middleman
Proxy + full inspection
Two sessions, plaintext in the gap
  • FortiGate terminates two separate TLS sessions (client↔FGT, FGT↔server)
  • Fully reassembles the object before inspecting
  • Plaintext lives between the two sessions until the object is complete

Both give the IPS the decrypted payload. Different plumbing, same visibility.

Now watch the trap carefully — it has two halves that pull in opposite directions:

Full inspection does NOT require proxy mode. ✓ Flow mode decrypts happily too.
But proxy mode IS required for some things — specifically the AV case. When the engine needs the whole file reassembled to hash it or unpack an archive, that's proxy territory. Not because of encryption — because of file reassembly.

Don't let those two collapse into each other. Decrypting an encrypted session (a depth question) and reassembling a whole file (a mode question) are different problems. Flow-based full inspection decrypts happily — it just won't buffer the entire object. If your task needs the entire object, that's what pushes you to proxy. The encryption has nothing to do with it.

The exam trap

Certificate vs full — the fork that catches people

There are two depths of SSL inspection, and picking the wrong one is the single most common mistake on this topic:

Certificate inspection reads only the unencrypted handshake — the SNI, the certificate CN/SAN, the issuer. It never decrypts the payload. Envelope only.

Full (deep) inspection is the authorised man-in-the-middle: FortiGate terminates the TLS session, decrypts, lets the IPS read the plaintext payload, then re-encrypts onward. It reads the letter.

The rule to bank: any time you need the IPS to catch something inside an encrypted session — PHP injection over HTTPS, DNS exfiltration hidden in DNS-over-TLS — the depth is always full inspection, never certificate. The DNSExfiltrator.Data.Exfiltration signature requires full inspection for exactly this reason: the hostile DNS is the DNS an attacker has wrapped in encryption to blind you.
The one line that turns the IPS from blind to seeing
config firewall policy
    edit <feed-to-trading-server>
        set ips-sensor "protect-trading-api"
        # swap certificate-inspection OUT for the deep profile:
        set ssl-ssh-profile "deep-inspect-trading"
    next
end

Everything else was already in place — the sensor, the signatures, the policy. It was only ever the inspection depth holding the whole thing back.

Keeper for long-term memory
Certificate inspection reads the envelope. Full inspection reads the letter. The IPS is a letter-reader — so to catch anything inside encryption, the depth is always full. And because IPS matches a pattern, not a whole file, flow mode can read that letter as it streams by — no buffering the entire thing. AV, which must hash or unpack, is the one that reaches for the full object.
