Cheatsheet — FGCP vs FGSP vs VRRP Tap to open
FGCPFGSPVRRP
Identity One shared cluster — single VMAC + IP moves with the primary Independent boxes — each keeps its own identity Floating VIP + virtual MAC (derived from VRID) shared by group members
Config sync Yes — cluster shares one config No — configs drift, admin owns parity No — VRRP only manages identity
Session sync Yes — kernel + hardware sessions Yes — controlled by session-pickup flags No — sessions die on failover
Failover decision Cluster elects primary via heartbeat + waterfall (override, priority, uptime, serial) Upstream load balancer chooses who's active; FGSP just replicates state Master-down timer expires (~3× advertisement + skew) → highest priority takes over
Election tiebreak group-id → group-name → override → priority → sessions → uptime → serial N/A — external LB decides Priority (higher wins) → IP as tiebreaker
Failure detection HA heartbeat (hbdev) + link/interface monitor + GARP on takeover External LB health-check + peer session replication link Missed advertisements; optional vrdst reachability probe
Active-Active? Optional — same cluster identity, load distribution across members Common — both boxes forward, LB steers traffic No — only one master owns the VIP at a time
Ask yourself Who owns identity? Who owns state? Who owns failover? — the answer names the protocol.
