TECHNICAL CONCEPTS COVERED (pp. 84–88)
------------------------------------------------------------------
1. FGCP modes: active-passive (one works, one waits) vs
   active-active (primary load-balances sessions to partner; every
   packet still hits primary FIRST via vMAC, then extra hop to
   secondary and back — both directions, life of session).
2. Cluster REQUIREMENTS — must match or no cluster: same MODEL,
   same FIRMWARE, same HARD-DRIVE CONFIG, same OPERATING MODE
   (NAT/transparent of mgmt VDOM). LICENSING mismatch does NOT
   block formation — cluster drops to LOWEST license level among
   members ("the cluster runs at the level of its poorest member").
3. Setup identity: same group-id, group-name, password, hbdev
   settings. Best practice ≥2 heartbeat interfaces (priority number
   HIGHER = preferred); direct-connect for 2-member; DHCP/PPPoE →
   static during initial formation.
4. Split-brain: silent heartbeat → both members claim primary →
   duplicate vMAC/IP identity collision. Worst failure mode is not
   a dead member but two survivors.
5. PRIMARY ELECTION, override DISABLED (default):
   (1) connected monitored interfaces (most up wins) →
   (2) HA uptime (highest wins, only if gap ≥ 5 MINUTES) →
   (3) priority (highest wins) → (4) serial number (highest).
   Election stops at first deciding criterion. HA uptime resets to
   0 on monitored-iface failure or reboot; manual graceful failover
   = `diagnose sys ha reset-uptime` ON THE PRIMARY (no config
   change). Standings: `diagnose sys ha dump-by vcluster` —
   uptime column 0 = the LOWEST-uptime member (relative marker);
   reset_cnt = number of uptime resets.
6. Override ENABLED (CLI ONLY): priority jumps ABOVE uptime
   (monitored → priority → uptime → serial). Guarantees a preferred
   primary, at the cost of a SECOND failover when the old primary
   recovers (failback). Flapping member + override = perpetual
   failovers (student derived this himself in Q9). Override and
   priority are NOT synchronised — set manually per member.
7. FGCP Ethernet types (not IP): 0x8890/0x8891 heartbeat/election/
   failure-detection; 0x8893 config sync. Sniff:
   `diagnose sniffer packet any "ether proto 0x8890" 4`.
   Hook coined: "88-90 keeps you alive, 88-93 keeps you identical."
8. Virtual MAC concept PRE-DERIVED by student ("one VMAC one IP
   moving as one") — formally taught next session per SG order.
