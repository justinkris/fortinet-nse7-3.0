SESSION 45 · TROUBLESHOOTING GUIDE · FGCP

When Two Boxes Won't Become One Brain

Formation failures, wrong-primary mysteries, and failover surprises — how a senior engineer thinks before typing.

Scenario A

"The cluster won't form"

Symptoms: both units powered and cabled; each still shows standalone; no sync; second unit invisible in the GUI.

Questions before commands: Are these the same model? Same firmware build (not just major version)? Same disk configuration? Same operating mode on the management VDOM? Do group-id, group-name, password and hbdev settings match exactly on both? Are the heartbeat ports actually in the same broadcast domain or directly connected?

get system status <- model + firmware on EACH unit get system ha status <- role, members seen, sync state diagnose sys ha checksum show <- config checksums (mismatch = not synced) diagnose sniffer packet any "ether proto 0x8890" 4 <- is the pulse even on the wire?

Expected outputs: matching models/builds; ha status listing BOTH serials; heartbeat frames visible on the hb port. If the sniffer on the hb interface shows nothing, stop theorising about election — you have a layer-1/2 problem or a wrong port.

Common dead end: obsessing over priority and override settings when the units can't even hear each other. Election settings are irrelevant to a cluster that hasn't formed. Pulse first, politics second.
Trap: "The licenses don't match, that must be why it won't form." No — licensing mismatch never blocks formation; it degrades the cluster to the lowest level. If it won't form, the blocker is model/firmware/disk/op-mode or setup identity.
Scenario B

"The wrong box is primary"

Symptoms: you set priority 200 on FGT-A; FGT-B is primary anyway.

Think in election order, in order:

1. Monitored interfaces — does FGT-A have a monitored port down? (Instant disqualifier — check first, always)
2. HA uptime — is override disabled (default)? Then uptime outranks your priority. Has FGT-A rebooted or had a monitored link flap (uptime reset to 0)? Is FGT-B ≥5 minutes ahead?
3. Priority — only reached if 1 and 2 tie. Is your 200 actually configured on the box you think (not synced!)?
4. Serial — the birth-certificate tiebreak.
get system ha status diagnose sys ha dump-by vcluster FGVMxx92: ...uptime/reset_cnt=7814/0 <- 7814s ahead on HA uptime FGVMxx93: ...uptime/reset_cnt=0/1 <- lowest-uptime member; reset once

Likely root causes: (1) override disabled and the "wrong" box simply has the uptime lead — this is correct behaviour, not a fault; (2) a monitored interface down on your preferred box; (3) priority configured on one member only and never on the other.

Senior move: before changing anything, ask "is this actually broken, or is the election working exactly as designed on facts I forgot about?" Most 'wrong primary' tickets close with zero config changes.
Scenario C

"Failovers keep happening"

Symptoms: repeated role changes; users feel periodic blips; logs full of HA events.

Hypotheses, cheapest evidence first:

  • Override enabled + unstable preferred primary: every recovery retakes the role → one failover per flap. Evidence: get system ha status role history + show system ha for override.
  • Monitored interface flapping: each flap re-runs the election AND resets that member's uptime. Evidence: interface event logs; reset_cnt climbing in dump-by vcluster.
  • Heartbeat starvation: single hb link congested/dying → missed heartbeats → false failovers, worst case split-brain. Evidence: sniff 0x8890 on the hb port; check hb link errors.
Exam trap: "Enabling override reduces failover events" — FALSE. It guarantees an extra failback failover per incident, and turns a flapping member into a failover generator.
Scenario D

"Fail over gracefully — now, without breaking config"

Override stateLeverWhy it works
Disabled (default)diagnose sys ha reset-uptime on the primaryUptime outranks priority; resetting the primary's HA uptime to 0 hands the uptime lead — and the role — to the peer. No config touched.
EnabledRaise a secondary's priority or lower the primary's (per member, CLI)Priority now outranks uptime, so priorities are the steering wheel.

Verify recovery after either: get system ha status shows the intended roles, traffic passes, and — root cause, not just service — you can explain why the election chose as it did.

Method

The senior's sequence for any HA ticket

  1. Restate the business symptom ("trading blipped at 10:20" ≠ "HA is broken").
  2. Formed vs elected vs failed-over — decide which layer the fault lives in. Formation problems make election questions meaningless.
  3. Pulse before politics: confirm heartbeats on the wire before reasoning about roles.
  4. Walk the election criteria in order against evidence — stop at the first criterion that explains reality.
  5. Change nothing until the current behaviour is explained. Then change one thing.
  6. Verify roles, traffic, AND the explanation.
