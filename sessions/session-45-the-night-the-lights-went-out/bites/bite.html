FGCP: Modes, Requirements, Election

One breath

What FGCP is

Two+ FortiGates as one brain: config sync + session sync + one virtual identity (vMAC/IP). Modes: a-p (one works, one waits) or a-a (primary load-balances sessions to its partner — every session still hits the primary first, and partner-assigned sessions pay an inter-chassis hop, both directions, for life).

Two lists — don't blur them

The pulse

Heartbeat

≥2 heartbeat links (higher priority number = preferred; direct-connect a 2-member pair). Why two: silent heartbeat → split-brain → both members claim primary → duplicate vMAC/IP collision. Ethertypes (not IP): 0x8890/0x8891 heartbeat/election/failure-detection, 0x8893 config sync. "88-90 keeps you alive, 88-93 keeps you identical."

diagnose sniffer packet any "ether proto 0x8890" 4
Who leads

Primary election

Override DISABLED (default): monitored interfaces → HA uptime (wins only if gap ≥ 5 min) → priority → serial. Stops at first decider.

Override ENABLED (CLI only): monitored → priority → uptime → serial. Deterministic primary, but the recovered ex-primary takes the role back = guaranteed second failover per incident. Flapping member + override = perpetual failovers. Override & priority are not synchronised — set on each member.

The decision

Why LoFinance chose a-p

A-A's extra capacity costs an inter-chassis hop per balanced session — the toll booth in a redundancy costume. After spending Season 3 removing the ISF hop for 2.1 ms, a-p keeps the NP Direct path untouched. The passive twin is insurance with a sub-second claims process.
