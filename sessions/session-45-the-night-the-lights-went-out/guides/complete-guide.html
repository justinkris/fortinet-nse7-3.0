Concept 1

FGCP: two boxes, one brain

Think first — the question that opened the session

A second identical firewall is racked and powered next to the first. For it to take over mid-trading-day with nobody noticing, what must be TRUE about it?

The answer derived live, in one breath: configuration synchronised, sessions synchronised, and one identity — "one VMAC, one IP, moving as one." That is FGCP's soul. Two or more FortiGate devices operate as an HA cluster in one of two modes:

  • Active-passive (a-p): one member processes all traffic; the other waits, fully synced, invisible until failover.
  • Active-active (a-a): both members process traffic; the primary load-balances sessions to its partner.
Keeper: the passive twin isn't idle silicon — it's insurance with a sub-second claims process.

Concept 3

Heartbeats and split-brain

The shared brain needs a nervous system. FGCP travels between members over designated heartbeat links, using its own Ethernet types — not IP:

EthertypeCarries
0x8890 / 0x8891Heartbeats: member discovery, primary election, data sync signalling, failure detection, traffic redistribution
0x8893Configuration synchronisation
Hook: "88-90 keeps you alive, 88-93 keeps you identical."
Sniffing the pulse
diagnose sniffer packet any "ether proto 0x8890" 4

Why at least two heartbeat links? One word: split-brain. Heartbeat goes silent → each member concludes the other is dead → both promote themselves → two boxes claim the same virtual MAC and IP on the same network. ARP chaos, flapping, shredded sessions. The worst cluster failure isn't a dead member — it's two members who each think they're the survivor. So: ≥2 heartbeat interfaces, each with a priority (higher number = higher priority; the cluster falls to the next link when one dies), directly connected on a two-member cluster.

Concept 4

Primary election — override disabled (default)

Think first

FGT-A: priority 200, six weeks uptime… and a dark WAN port. Should it win? What should the cluster check before anyone's opinion?

The election compares criteria in order and stops at the first one that produces a winner:

  1. Connected monitored interfaces — most up links wins. Facts about now beat preferences from Tuesday.
  2. HA uptime — highest wins, but only if the gap is at least 5 minutes. A photo-finish doesn't count.
  3. Priority — highest wins. The admin's vote, third in line.
  4. Serial number — highest wins. Twins settle it by birth certificate.

HA uptime (not system uptime) resets to 0 when a monitored interface fails or a member reboots. Because uptime outranks priority by default, the senior engineer's graceful failover needs no config change at all:

Graceful failover + standings check
diagnose sys ha reset-uptime          <- run ON the primary
diagnose sys ha dump-by vcluster
...
FGVMxx92: ...uptime/reset_cnt=7814/0
FGVMxx93: ...uptime/reset_cnt=0/1

Reading the output: the member showing 0 in the uptime column is the lowest-uptime member (a relative marker, not literal zero seconds); the other member's number is how many seconds ahead it is (7814s here). reset_cnt counts how many times that member's HA uptime has been reset.

Concept 5

Override: every guarantee has a bill

Enable override — CLI only — and priority jumps above uptime:

monitored interfaces → priority → uptime → serial

Advantage: a deterministic preferred primary. Whenever your chosen box is healthy, it leads. The bill: when the failed primary recovers, it takes the role back — a guaranteed second failover per incident, each one migrating vMACs, re-teaching switches, wobbling in-flight packets. And the derived nightmare case: a flapping member with override enabled retakes primary on every recovery — perpetual failovers. With override off, the same flapping box recovers quietly into standby and bothers nobody.

Two settings that deliberately do NOT synchronise: override and priority. They are per-member opinions — configure them manually on each box. Junior enabling override "in the GUI, on the primary only" is wrong twice: no GUI option exists, and one-sided config leaves the members disagreeing about their own election rules.

Concept 6

Choosing the mode: a-p vs a-a

Junior: "Active-active! We paid for two boxes!" The counter-argument connects two seasons. In a-a, every new session still arrives at the primary first (it owns the virtual MACs). If the primary assigns the session to its partner, every packet of that session takes an extra inter-chassis hop, both directions, for the life of the session. A-A's added capacity is real — most valuable for inspection-heavy (CP/CPU) work — but it is paid for in hops.

Keeper: for a firm that spent Session 40 removing one hop inside one box (the ISF) to reclaim 2.1 ms, active-active is the toll booth in a redundancy costume.

LoFinance's answer: active-passive. The primary carries traffic on the untouched NP Direct path; the twin waits, fully synced.

The signed purchase order, as config
config system ha
    set mode a-p
    set group-id 10
    set group-name "LOFI-EDGE"
    set password <secret>
    set hbdev "ha1" 50 "ha2" 50
end
Learn from the cast

Common mistakes

  • Junior's 600F bargain: same model or no cluster. Bargains that can't cluster are expensive.
  • Licensing one member only: the whole cluster degrades to the poorest license. Verify parity before the PO, not after.
  • "Active-active is obviously better": only if you can afford the inter-chassis hop. Redundancy design must respect prior latency work.
  • Override via GUI / on one member: CLI-only, and override + priority are never synchronised.
  • Answering the neighbouring question: "what must match to form" (model/firmware/HDD/op-mode) vs "what settings must match" (group-id/name/password/hbdev) are different lists. Exams love this seam.
  • Misreading dump-by vcluster: uptime 0 = relative lowest, not zero seconds; reset_cnt is a counter, not a state.
Phase 3

Exam tips & takeaways

Five things worth an exam mark each:
1. Election stops at the FIRST deciding criterion; monitored interfaces always first.
2. Default order: monitored → uptime (≥5 min gap) → priority → serial. Override swaps priority above uptime, CLI-only.
3. Licensing mismatch ≠ formation failure; cluster runs at the lowest license level.
4. reset-uptime = graceful failover without config change (default override). Priority change = the lever when override is enabled.
5. 0x8890/0x8891 heartbeat & election; 0x8893 config sync; FGCP is its own Ethernet type, not IP.

Senior lessons: understand before purchasing; facts about now beat preferences from Tuesday; every guarantee has a bill; and the packet path — not the price tag — decides the HA mode.
