TECHNICAL CONCEPTS COVERED
------------------------------------------------------------------
1. The crack in Junior's claim: it is PROXY-based profiles that
   hard-kill NP offload, not "security profiles" as a category.
   Flow-based inspection is precisely what NTurbo rescues.
2. NTURBO = a PATH feature. Creates a special data path
   redirecting traffic from the ingress interface to the IPS
   engine. Cycle: new session -> NTurbo driver sends packets to
   IPS engine -> IPS engine performs the security action -> IPS
   engine returns packet to NTurbo driver -> NTurbo returns
   packets to the NP. FortiGate runs the NTurbo driver on a CPU
   DEDICATED to processing NP-sent traffic needing flow-mode
   inspection. Result: improved IPS performance; the session
   keeps NP acceleration for everything that isn't inspection.
   Scope rule: "anything handled by the IPS engine can use
   NTurbo."
3. IPSA = an OFFLOAD feature. Offloads flow-based pattern
   matching to the CONTENT PROCESSORS. When enabled, flow-based
   pattern-matching databases are compiled and downloaded to the
   CPs from the IPS engine and the IPS database.
4. CRITICAL SEPARATION taught: NTurbo is NP <-> IPS engine.
   IPSA is IPS engine -> CP. The NP is NOT a party to IPSA.
   Neither feature causes the other; both are beneficiaries of
   the same qualifying condition (flow-based profile).
5. WHERE THE IPS ENGINE LIVES: software on the FortiGate CPU.
   The CP is a called coprocessor, not the engine. Therefore
   NTurbo does NOT get you off the CPU — UTM/NGFW operations are
   still handled by the CPU, with IPSA offloading pattern
   matching to the CPs. Acceleration != free.
6. Life of a packet, NTurbo/flow-based UTM session (established):
   ingress -> NP -> NTurbo driver -> IPS engine (CPU)
   [engine calls CP for pattern matching, result returns]
   -> IPS engine -> NTurbo driver -> NP -> egress. Symmetrical.
   Once the session key is installed, subsequent packets skip
   routing and kernel.
7. Life of a packet, PROXY-based UTM session: NP does NOT offload
   the session. CPU handles processing via IPS engine + FortiOS
   proxy. BUT the NP is not idle — it still conducts some early
   security inspections before handing off, and still does IPsec
   tunnel decryption/encryption. CP still does SSL/TLS crypto for
   deep inspection. SSL deep-inspect sub-flow: IPS engine -> proxy
   decrypts (offloading to CP) -> back to IPS engine for IPS/appctrl
   -> back to proxy for proxy-based inspection -> proxy re-encrypts
   via CP.

KEY CLI DISCUSSED
------------------------------------------------------------------
config ips global
    set np-accel-mode {none | basic}          <- basic = DEFAULT
    set cp-accel-mode {none | basic | advanced}
end                       advanced = default on most models
                          basic = only mode on single-CP8 devices

config firewall policy
    edit <id>
        set np-acceleration {disable | enable}  <- enable = DEFAULT
end

Disable NTurbo globally (np-accel-mode none) or per policy
(np-acceleration disable) FOR TROUBLESHOOTING PURPOSES ONLY.

TROUBLESHOOTING APPROACH TAUGHT
------------------------------------------------------------------
When inspected traffic is unexpectedly slow: check the INSPECTION
MODE before you check the hardware. A single proxy-based profile
is the most common cause of a policy silently dropping to the slow
path. Then confirm NTurbo/IPSA were not left disabled from a
previous troubleshooting session (Junior's S41 sin repeated at a
different layer).
