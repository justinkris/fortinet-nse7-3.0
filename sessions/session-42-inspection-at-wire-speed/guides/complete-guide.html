Inspection at Wire Speed

Cipher demands inspection. Marcus demands physics. Only one of them is quoting a rule that has exceptions.

Junior said security profiles kill the offload. The word he needed was proxy.

💭 THINK FIRST · THE CRACK

Marcus is standing in the network room with a printed memo, arguing that inspection and latency are physically incompatible. Junior agrees with him and is about to talk Cipher out of a design that is actually available. Before you read on, answer these two.

  • Junior says "security profiles kill the offload." Which single word in that sentence is doing damage, and what should it be?
  • If flow-based inspection does not disqualify a session from NP acceleration, what problem still remains? The NP cannot run IPS signatures — so how does an inspected packet ever reach the thing that can?
Q1 — MODEL ANSWER
"Security." The Session 41 rule is that proxy-based profiles disqualify a session from NP offload — and that a single proxy profile poisons the entire policy. Flow-based profiles are not in that category at all. Junior generalised a specific rule into a universal one, which is the exact error that turns an available design into a refused one.
Q2 — MODEL ANSWER
The remaining problem is a path problem, not a permission problem. The NP has no native mechanism to hand a packet to the IPS engine and get it back, so without help the whole session would still collapse onto the slow path. That missing on-ramp and off-ramp is precisely what NTurbo builds.

The Crack in Junior's Claim

Two stakeholders arrived holding absolutes. Cipher's absolute came from a regulator; Marcus's came from a quarter of a million dollars of ultralow-latency fibre and a measured 40 µs path. When two absolutes collide, the engineer's job is not to pick a side. It is to find the cheapest true version of the requirement and then state its cost plainly.

The cheapest true version existed because Junior's premise was wrong. He said security profiles kill the offload. Session 41 taught something narrower and far more useful: proxy-based profiles kill the offload, and one of them poisons the whole policy. Flow-based inspection sits outside that rule entirely — and flow-based is exactly the category NTurbo and IPSA were built to accelerate.

Common misconception: "Turning on IPS drops the session to the CPU." Only if the profile is proxy-based, or if a proxy-based profile shares the policy. A flow-based IPS profile keeps NP acceleration through NTurbo.

💭 THINK FIRST · NTURBO

You know now that flow-based inspection is permitted to keep NP acceleration. But permission is not plumbing. Something still has to physically carry the packet to the inspector and bring it back — and the NP cannot do that on its own.

  • Which component actually decides whether a packet matches an IPS signature — the NP, the CP, or the IPS engine?
  • Given that answer: where does NTurbo need to carry the packet to, and where does it need to bring it back to?
Q1 — MODEL ANSWER
The IPS engine. The NP forwards; it cannot evaluate a signature. The CP is called upon during evaluation but does not own the decision. This matters because it fixes the destination of NTurbo's data path — a packet must physically reach the IPS engine.
Q2 — MODEL ANSWER
NTurbo carries the packet from the ingress interface to the IPS engine, and then from the IPS engine back to the NP for forwarding. It is an on-ramp and an off-ramp. Not NP → CP: NP → IPS engine → NP.
SECTION 02

NTurbo — The On-Ramp

NTurbo does not remove the inspection. It builds the slip roads so the motorway does not have to stop.

Learning purpose: burn in that NTurbo is a path feature with symmetric ramps, and that the CP is a side call rather than a hop in the packet path.

The problem NTurbo solves: ordinary NP offload cannot be used when flow-based security profiles are configured, because the NP has no way to hand a packet to inspection and take it back. Without help, the entire session falls to the slow path — which is precisely the outcome Junior predicted and Marcus feared.

What it does: NTurbo creates a special data path that redirects traffic from the ingress interface to the IPS engine. FortiGate runs the NTurbo driver on a CPU dedicated to processing traffic the NP sends that requires flow-mode inspection.

  • Step 1: For a new session, the NTurbo driver sends packets to the IPS engine.
  • Step 2: The IPS engine processes the required security action.
  • Step 3: The IPS engine sends the packet back to the NTurbo driver.
  • Step 4: NTurbo continues to send processed packets back to the NP.
Remember — the scope rule: anything that is handled by the IPS engine can use NTurbo. Proxy-based inspection is handled by the FortiGate CPU and the FortiOS proxy — not by NTurbo.
Analogy

NTurbo keeps the NP relevant.

  • Without NTurbo, an inspected session has nothing for the NP to contribute — the whole thing runs on the slow path.
  • With NTurbo, the NP keeps the forwarding half of the job and only surrenders the inspection half.
  • The road does not end at the inspection booth. It detours through it and rejoins.
🧠
MENTAL NOTE
NTurbo is a path feature: NP ↔ IPS engine, on-ramp and off-ramp. It does not perform inspection and it does not touch the CP.

💭 THINK FIRST · IPSA

The IPS engine is software running on a CPU, and it must compare every packet against tens of thousands of signature patterns. That is a lot of comparison work for a general-purpose processor.

  • What is the single most expensive operation the IPS engine performs, and what kind of silicon would you rather hand it to?
  • Does IPSA involve the network processor in any way?
Q1 — MODEL ANSWER
Pattern matching. You would rather hand it to a content processor — a coprocessor purpose-built for exactly this. That is IPSA: flow-based pattern matching offloaded to the CP, with the pattern-matching databases compiled and downloaded to the CPs from the IPS engine and the IPS database.
Q2 — MODEL ANSWER
No. IPSA is a relationship between the IPS engine and the CP. The NP is not a party to it. NTurbo is the NP-facing feature; IPSA is the CP-facing feature. Neither causes the other — both are beneficiaries of the same qualifying condition, which is that the profile is flow-based.
SECTION 03

IPSA — The Coprocessor Call

Pattern matching is the expensive part of flow-based inspection, and it is software running on a general-purpose CPU. IPSA allows those pattern-matching operations to be redirected to the content processors, reducing load on the FortiGate CPU and accelerating the match itself.

When IPSA is enabled, flow-based pattern-matching databases are compiled and downloaded to the content processors from the IPS engine and the IPS database. The CP does not become the engine — it becomes the thing the engine calls.

Exam tip: the causal arrow runs one way. Flow-based is the precondition. Because the profile is flow-based, two independent things become possible: the NP can stay in the path via NTurbo, and pattern matching can go to the CP via IPSA. IPSA does not make a session NP-eligible.

💭 THINK FIRST · PACKET PATHS

Two packet paths matter for the exam: the accelerated flow-based path and the proxy-based path. The second one contains a trap that has caught most engineers at least once.

  • Trace an established NTurbo-accelerated flow-based session from ingress to egress. Where does the CP appear in that trace?
  • On a proxy-based UTM session the NP does not offload the session. Does that mean the NP does nothing at all?
Q1 — MODEL ANSWER
Ingress → NP → NTurbo driver → IPS engine on the CPU → NTurbo driver → NP → egress. The path is symmetrical. The CP appears nowhere in the trace — the IPS engine calls it for pattern matching and receives a result. It is a phone call, not a train station. Once the session key is installed in the NP, subsequent packets skip routing and kernel processing.
Q2 — MODEL ANSWER
No — and this is the trap. The NP still conducts some early security inspections before handing packets off to the IPS engine, and it can still be leveraged for IPsec tunnel decryption and encryption. The CP is still used for SSL/TLS encryption and decryption when deep inspection is configured. "Not offloaded" is not the same as "not involved".
SECTION 04

Two Paths, One Trap

The upper path detours. The lower path stops, unpacks, and reassembles — but the NP has not left the building.

Flow-based, NTurbo-accelerated, established session:

Packet path
ingress interface
   -> NP6/NP7
   -> NTurbo driver
   -> IPS engine (CPU)          [calls CP for pattern matching via IPSA]
   -> NTurbo driver
   -> NP6/NP7
   -> egress interface

After the session key is installed in the network processor, subsequent packets for the accelerated session skip routing and kernel processing. UTM/NGFW operations are still handled by the CPU, with IPSA offloading pattern matching to the content processors.

Proxy-based session: the network processor does not offload the session. The FortiGate CPU handles packet processing and inspection through a combination of the IPS engine and the FortiOS proxy. But the NP still conducts some early security inspections before handing off, and can still be leveraged for IPsec tunnel decryption and encryption. Where SSL deep inspection is configured, the CPU leverages the CP for SSL/TLS crypto.

The deep-inspection sub-flow is worth knowing in order: the IPS engine sends the packet to the proxy for decryption; the proxy decrypts by offloading to the CP; the decrypted packet returns to the IPS engine for IPS and application control; it goes back to the proxy for the configured proxy-based inspection; the proxy re-encrypts, again offloading to the CP.

Exam trap: a question asks what the NP can still do on a proxy-based session. The tempting answer is "nothing — proxy kills offload." The correct answer names early security inspections and IPsec encryption/decryption. The IPsec exception is a property of IPsec, not a footnote to the tunnelling rule.

MENTAL NOTE
Not offloaded is not the same as not involved. On a proxy path the NP still does early checks and IPsec crypto; the CP still does SSL/TLS crypto.

💭 THINK FIRST · CONFIG & TRAPS

Two commands govern this behaviour, and they live at different scopes with different value sets. Mixing up which values belong to which command is one of the easiest marks to drop on the exam.

  • Name the global command that controls NTurbo, its two possible values, and its default.
  • How would you disable NTurbo for one firewall policy only, and why would you ever want to?
Q1 — MODEL ANSWER
config ips global → set np-accel-mode {none | basic}. The default is basic, which is the value that provides acceleration. Note this is not enable/disable — those values belong to the per-policy command.
Q2 — MODEL ANSWER
config firewall policy → edit <id> → set np-acceleration disable (default is enable). You would do this for troubleshooting purposes — to determine whether acceleration is implicated in a fault, on one policy, without disturbing the rest of the box. And, per Session 41, you put it back.

The value-bleed trap: np-accel-mode and np-acceleration have similar names, different scopes, and different value sets. If you answer "enable / disable" for the global command you have imported the per-policy values. Learn the three rows above as one table, not three separate facts.

By default NTurbo is enabled to ensure a load-balanced approach for handling IPS signature and pattern-matching tasks. Both disable paths — global np-accel-mode none and per-policy np-acceleration disable — exist for troubleshooting purposes. Neither is a tuning knob for production.

Troubleshooting Workflow

  1. Check the inspection mode before you check the hardware. A single proxy-based profile on the policy is the most common cause of a policy silently dropping to the slow path.
  2. Audit the policy for mixed profiles. Flow plus proxy on one policy offloads nothing at all.
  3. Confirm the acceleration settings were not left disabled from a previous troubleshooting session. Diagnostic settings are temporary by contract.
  4. Remember the birth rule. The offload decision is made once, at session creation, and never re-evaluated. Existing sessions carry the old verdict until they die.
  5. Verify, do not assume. Which is exactly what Session 43 is about.

Senior Engineering Insights

  • "Accelerated" is not "free." Know which component pays for each stage. NTurbo did not get Marcus off the CPU — it stopped the entire session from collapsing onto it. Say that out loud to the business rather than letting them infer something better.
  • When two stakeholders both invoke absolutes, find the cheapest true version of the requirement and state its cost plainly. Compliance said must; trading said can't. Both were partly wrong, and the engineer was the only person in the room who could show why.
  • Precision in category language is not pedantry. "Security profiles" versus "proxy-based profiles" is the difference between telling the business no and telling it yes with a condition.
  • A newly-learned rule arrives absolute. Junior over-generalised the poison rule into "IPS kills offload." The same instinct produces "proxy kills offload, therefore the NP does nothing." Give every new rule a deliberate what are the exceptions pass before you let it harden.

Key Takeaways

  1. Proxy kills offload. Flow does not.
  2. NTurbo = the path, NP ↔ IPS engine.
  3. IPSA = pattern matching offloaded to the CP.
  4. Inspection itself still costs CPU. Acceleration ≠ free.
  5. Not offloaded ≠ not involved — the NP still does early checks and IPsec crypto on a proxy path.
