Troubleshooting Method · Session 17

Thinking Like the Senior Engineer

Symptom Presentation

Backup tunnel's Phase 1 shows up. OSPF neighbor table on the far side is completely empty. No traffic flows. Later, after both faults appear fixed and the primary link is restored, a THIRD symptom appears with everything showing green.

Exam trap shapeThe symptom description ("phase 1 up") is technically true and also nearly useless — it tells you almost nothing about where the fault actually lives. Never let a partially-healthy status stop the investigation.

Questions To Ask First

  • Is this a transport problem (packets can't cross) or a protocol problem (packets cross, protocol won't proceed)?
  • Is the failure symmetric (both ends agree it's broken) or asymmetric (one end thinks it's fine)?
  • Is the table empty or stuck partway through a state machine — these imply different failure classes entirely.
  • Did anything change recently that both ends should agree on but might not (a value set independently on each side)?

Decision Tree

Tunnel shows up but no traffic?
Check tunnel list: sa=0 → Phase 2 never established → bilateral IKE debug → compare proposals.
Traffic flows but OSPF table empty?
Check interface state (up? correct network-type?) → check area on both ends via show router ospf → mismatch = silent discard, not a stuck state.
Adjacency Full, LSDB confirmed correct, but expected route missing?
Don't assume SPF failed. Run routing-table all — look for a lower-AD source (static, connected) claiming the same prefix.
Considering redistributing a static to "fix" a table?
Stop — ask whether redistribution deletes the source route. It never does. If a static is wrongly winning, disable/remove it directly.

Common Dead Ends

Dead endAssuming a mismatched network-type when both ends already show matching POINTOPOINT in ospf interface output — re-read the evidence already on screen before reaching for a different fault category.
Dead endChoosing a non-backbone area for a redundant link based on blast-radius concerns alone, without walking the "what if the primary link is down" scenario first.
Dead endTreating "the LSDB is correct" as proof the routing table must also be correct — they are different data structures, and administrative distance sits between them.

Likely Root Causes (Ranked by This Session)

  1. Hand-set crypto parameter (dhgrp) drifted between two independently-configured ends.
  2. Area assignment drifted or was chosen for the wrong design reason (blast-radius vs backbone-continuity conflict).
  3. A stale or "temporary" static route, forgotten, silently outranking a healthy dynamic route the moment topology changed.

Exam Traps To Remember

Trap 1NO_PROPOSAL_CHOSEN can appear at Phase 1 or Phase 2 — read which exchange type produced it before diagnosing.
Trap 2An empty OSPF neighbor table has at least two distinct causes with identical symptoms: sender-side (wrong network-type) vs receiver-side (area mismatch) hello death.
Trap 3A "select all that apply" or scenario question showing Full adjacency + correct database is designed to make you stop checking — the fault may still be one layer further down, in the routing table itself.
