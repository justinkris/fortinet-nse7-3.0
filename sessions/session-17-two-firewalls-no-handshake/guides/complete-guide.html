Study Guide · NSE7 EF 7.6 · Session 17

Two Firewalls, No Handshake

Cowboy Hat Boss Battle — IPsec Phase 2 mismatch, OSPF backbone partition, and the static route that lied.

Three faults, one lesson: a healthy-looking system can still be wrong at every layer you haven't checked yet.

Session Overview

Señor pre-broke a new DR↔Parramatta backup tunnel (DR2PAR) at both ends, then forced a live failover drill by shutting the primary HQ↔Parramatta link. Three faults surfaced in sequence, each hiding behind the last: a Phase 2 crypto mismatch, an OSPF area mismatch that risked partitioning the backbone, and — once everything else was fixed — a planted static route that silently outranked a perfectly healthy OSPF path the moment the primary link came back.

The throughline: adjacency Full and LSDB-correct does not mean the routing table reflects it. Every fault this session lived one layer below where the previous check said "healthy."

Learning Objectives

  • Read diagnose vpn tunnel list output to distinguish "Phase 1 up, Phase 2 never established" from other failure signatures.
  • Use bilateral IKE debug to find a proposal mismatch from both the initiator's and responder's perspective.
  • Explain why an OSPF area mismatch produces a completely empty neighbor table rather than a stuck or flapping one.
  • Recognise when a redundant link risks partitioning the OSPF backbone, and why area design rules override blast-radius concerns in that case.
  • Diagnose a route that is correct in the LSDB but absent from the OSPF-filtered routing table, using administrative distance reasoning.
  • Understand FortiManager IPsec templates as a shared-structure/per-site-variable split.

Fault 1 — IPsec Phase 2 Never Establishes

Both ends showed Phase 1 as ready, but the tunnel carried nothing. The diagnostic signature:

diagnose vpn tunnel list name DR2PAR
run_state=0 role=primary accept_traffic=0
proxyid_num=1 child_num=0 refcnt=4
stat: rxp=0 txp=0 rxb=0 txb=0
dpd: mode=on-demand on=1 idle=20000ms retry=3 count=0
proxyid=DR2PAR proto=0 sa=0 ref=1

Reading it: sa=0 means no active Security Association — Phase 2 never completed. proxyid_num=1, child_num=0 means one phase2 definition exists but zero children (SAs) have been negotiated from it. rxp=txp=0 confirms zero packets have ever crossed. This is a distinct signature from Phase 1 being down entirely — here, the IKE SA is healthy and ready to negotiate, it just never succeeds.

Diagnostic ladder: tunnel-list (confirm which phase is broken) → IKE debug on both ends (see the actual rejection) → compare the two logged proposals line-by-line.
diagnose vpn ike log filter rem-addr4 <peer> ; diagnose debug application ike -1 ; diagnose debug enable
Initiator (DR):
  ...ESP proposal: AES_CBC/256, SHA256, DH_GROUP=MODP2048
  received notify type NO_PROPOSAL_CHOSEN

Responder (Parramatta):
  incoming proposal: ...DH_GROUP=MODP2048
  my proposal:       ...DH_GROUP=MODP1536
  no proposal chosen

The DH group — the PFS group under phase2-interface — differed. Everything else matched.

SettingWhere configuredMust match both ends?
Encryption / integrityphase2-interface proposalYes
DH group (PFS)phase2-interface set dhgrpYes
Traffic selectorsphase2-interface src/dstMust be mirror images
Fix direction: raise the weaker end (group 5 → 14) rather than downgrade the stronger one — harder math, stronger keys, and no legacy peer forcing a ceiling here. When there's no compatibility constraint, always upgrade toward the stronger setting.
Exam trapNO_PROPOSAL_CHOSEN can appear at Phase 1 or Phase 2 — read which exchange (IKE_SA_INIT vs CREATE_CHILD_SA) the log line belongs to before diagnosing which phase failed.

Fault 2 — OSPF Area Mismatch → Silent Discard

Once the tunnel carried traffic, OSPF still refused to form an adjacency across it — despite both interfaces up, both POINTOPOINT, matching cost, hellos leaving every 10 seconds into a proven-working tunnel.

Think First
  • If hellos are actually arriving, why would the neighbor table stay completely empty instead of stuck partway through the adjacency ladder?
  • What does a router check about an incoming hello before it does anything else with it?

The mechanism: the OSPF header carries an area stamp. A receiving interface checks that stamp before any other OSPF processing happens. A mismatch means the packet is silently discarded at the door — not processed, not disputed, never logged as an attempted neighbor. Neither side has grounds to create even an Init entry, because as far as each router is concerned, no hello from that peer ever arrived.

Same fingerprint, different murderer: this produces the identical symptom as Session 15's broadcast-network-type-on-a-tunnel fault — a completely empty neighbor table. But the failure lives in a different place: broadcast-on-tunnel kills the hello at the sender (no medium to deliver a multicast frame); area mismatch kills it at the receiver (header check fails before processing). Same symptom, opposite cause — the diagnostic lesson is to check both possibilities whenever the table is empty rather than flapping.
show router ospf (config network statement)
LF-DR:   edit 3 / prefix 10.255.0.0/30 / area 0.0.0.0
LF-PAR:  edit 3 / prefix 10.255.0.0/30 / area 0.0.0.1   ← mismatch

Why It Had To Stay Area 0

The first instinct — put the backup link in Area 1, to limit LSDB size / blast radius — is a real area-design principle, but wrong here. Walk the scenario: with the primary link down and the backup in Area 1, Parramatta's Area-0 LAN has no Area-0-only path to DR's Area-0 LAN. OSPF's inter-area rule is strict: all inter-area traffic must transit the backbone, and a non-backbone area can never serve as a relay between two disconnected pieces of Area 0.

The result is called a backbone partition — Area 0 broken into disconnected islands, one of OSPF's genuinely forbidden states. A redundant link connecting two Area-0 sites must itself live in Area 0, precisely so it can keep the backbone whole when the primary path dies.

Exam trapWhen two area-design principles seem to conflict (minimize blast radius vs preserve backbone continuity), backbone continuity wins for any link that is structurally required for connectivity between Area-0 sites.
Fix
LF-PAR # config router ospf
       # config network
       # edit 3
       #   set area 0.0.0.0
       # end

Result: Full/ - forms within seconds on both ends.

Fault 3 — The Static Route That Lied

With the drill "won," Señor restored the primary link at 09:00 as planned — and six minutes later, DR→Parramatta traffic silently died, even though every OSPF indicator remained green: adjacency Full, LSDB correct and verified on both ends via get router info ospf database router <id>, cost and area all matching.

The critical distinction: get router info routing-table ospf is a protocol-filtered view — it only shows routes OSPF actually won. get router info routing-table all shows every protocol's routes and who's actually installed. A route can be completely absent from the first and clearly present in the second under a different, higher-priority source.

get router info routing-table all (LF-DR)
S    10.2.0.0/24 [10/0] via 10.254.0.1, To-HQ
O    10.99.50.0/24 [110/10] is directly connected, Hub-Static (E2)
C    10.255.0.0/30 is directly connected, DR2PAR

An S route — administrative distance 10 — silently outranked the OSPF route (distance 110) to the exact same prefix, even though OSPF's route was cheaper by cost and fully valid. Distance is compared before cost ever enters the decision, and it's compared across protocols, not within one.

SourceDefault ADWins when tied on prefix?
Connected0Always
Static10Beats any dynamic protocol by default
OSPF110Loses to any static covering the same prefix
The Wrong Fix, Reasoned Away Correctly

First instinct: redistribute the static into OSPF. Checked against a Session 15 precedent instead of assumed:

  • Does redistributing a static route into OSPF delete the original static route?

No. Redistribution is advertisement, not deletion — the static route (distance 10) would still sit there afterward, still beating the very OSPF route you'd just told the rest of the world about. This correctly ruled out the fix before it was tried.

Actual fix
LF-DR # config router static
       # edit 4
       #   set status disable
       # end

The moment the static cleared, the correct OSPF route (cost 20, via DR2PAR) appeared immediately in the protocol-filtered view too.

FortiManager IPsec Templates (Concept)

Motivation, stated plainly: the DH-group mismatch this session existed because a value was set by hand on each end, independently, and drifted. That is exactly the failure mode templates exist to prevent.

Config blockShared (template)Per-site (variable)
Phase 1proposal, DH group, PSK method, DPD settingspeer public IP, tunnel name
Phase 2proposal, PFS dhgrp, encapsulationsrc/dst selectors, tunnel-interface IPs
OSPFnetwork-type, cost, hello/dead timersinterface name, area, network statement

A FortiManager template locks the shared structure and exposes only the per-site variables as metafields — pushing consistent config to ten site pairs with zero chance of a hand-typed dhgrp drifting on one end. Hands-on configuration workflow: Session 18.

Key Takeaways

  • A protocol-healthy adjacency (Full, correct LSDB) does not guarantee the routing table reflects it — always check what's actually winning, not just what the protocol calculated.
  • Empty OSPF neighbor tables have (at least) two distinct causes with the same symptom: sender-side hello death (wrong network type) and receiver-side header rejection (area mismatch). Check both.
  • Backbone continuity overrides blast-radius minimization for any link structurally required to keep Area 0 connected.
  • Administrative distance is compared before cost, across protocols — a lower-AD static will silently and permanently bury a cheaper, healthier OSPF route to the same prefix.
  • Redistribution never deletes the source route — it only advertises it elsewhere.
