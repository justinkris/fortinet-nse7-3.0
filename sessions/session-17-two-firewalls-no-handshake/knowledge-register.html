[28] BACKBONE PARTITION / AREA-0 CONTINUITY REQUIREMENT
- Confidence: ★★☆☆☆ (new item, taught not derived)
- Mental model: EMERGING. First instinct when asked "which area for the backup link"
  was Area 1, reasoned from a real and legitimate concern (LSDB size / blast radius if
  Parramatta became an ABR into Area 0) — good instinct in the wrong context, and
  notably drawn from S16 material correctly, just misapplied. Required a full worked
  scenario walk (primary down + backup in Area 1 = what does Area 0 look like as a
  shape) before reversing to the correct answer. Once corrected, could not initially
  articulate what "shape" Area 0 had become ("I don't know what shape... but if
  Parramatta's LAN is in Area 0 the backup link needs to be as well") — got the
  conclusion right via a narrower, more mechanical argument (LAN area must match link
  area) rather than the intended "islands"/partition framing, until Señor supplied the
  term and the reasoning explicitly.
- Common mistakes: applying a correct area-design heuristic (minimize blast radius /
  ABR proliferation) to a scenario where a stronger, overriding rule (backbone
  continuity) actually governs. This is worth naming as a pattern: when two area-design
  principles conflict, backbone continuity/reachability wins over blast-radius
  minimization for links that are structurally required for connectivity.
- Strong areas: correctly reasoned that whatever the answer, both ends of the link
  need to match (consistency instinct is sound).
- Weak areas: has not yet independently produced "backbone partition" as a concept or
  its name; needs one clean unprompted retrieval before this is considered learned
  rather than taught.
- Reinforcement: direct retrieval in S18 — "two Area-0 sites, primary link down, only
  a non-backbone-area path remains between them — what happens, and what's it called?"
- Last reviewed: S17.

[29] ADMINISTRATIVE DISTANCE vs OSPF COST/CORRECTNESS
- Confidence: ★★★☆☆ (new item)
- Mental model: GOOD once evidence was visible, via a genuine self-correction. Initial
  fix instinct (redistribute the static into OSPF) was wrong, but when asked to check
  whether redistribution deletes the source route, correctly retrieved the S15
  precedent unprompted and used it to rule out his own proposed fix — this is a
  stronger form of transfer than simple recall (applying a known fact to invalidate a
  live hypothesis, under pressure, mid-incident). Also correctly distinguished
  `get router info routing-table ospf` (protocol-filtered, showed nothing) from
  `get router info routing-table all` (showed the real winning route) once the second
  command was suggested by Señor — did not derive the need for the `all` view
  independently, but immediately understood and used what it revealed.
- Common mistakes: proposing "redistribute the static" as a fix before checking
  whether it would actually change what's IN the table (same near-miss pattern as
  network-type/Area-1 above — proposing before fully checking against evidence).
- Strong areas: once corrected, produced the RIGHT fix (disable the static) with the
  RIGHT reasoning (AD 10 vs AD 110, a distance problem not a route-existence problem)
  cleanly and did not need it repeated.
- Weak areas: did not independently think to check the "all" routing table view when
  a protocol-filtered view came back empty for a route the LSDB clearly supported —
  this is a genuinely useful diagnostic reflex to build ("if the map is right and the
  protocol view is empty, check what's actually WINNING, not just what OSPF wanted to
  install").
- Reinforcement: plant one scenario in S18 or S19 where a protocol-filtered table is
  empty/wrong and see if the "all" view is reached for without prompting.
- Last reviewed: S17.
