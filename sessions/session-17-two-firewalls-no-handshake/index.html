Concepts Covered 8 concepts
  1. 01
    IPSEC PHASE 2 NEGOTIATION FAILURE FORENSICS
    `diagnose vpn tunnel list name <X>` read cold — sa=0/rxp=txp=0/proxyid_num=1,child_num=0 = "phase 1 up, phase 2 never established," recognized without prompting. Escalated correctly to `diagnose vpn ike log filter rem-addr4 <peer>` + `diagnose debug application ike -1` + `diagnose debug enable` on BOTH ends unprompted — good instinct, since a rejected negotiation only tells the full story from both the initiator's rejection message and the responder's stated reason. Read two side-by-side ESP proposal blocks and pulled the single differing line (DH_GROUP: MODP2048 vs MODP1536) correctly.
  2. 02
    PFS/DH GROUP AS A PHASE-2-INTERFACE SETTING
    confirmed `config vpn ipsec phase2-interface / set dhgrp <N>` must match on both ends; this is what puts a DH_GROUP line inside an ESP proposal in the first place (this is PFS group negotiation, distinct from Phase 1's own DH group).
  3. 03
    DIRECTION-OF-FIX REASONING
    given a mismatch (group 14 vs group 5), student correctly reasoned to raise the weaker end rather than downgrade the stronger one ("higher group, harder math, stronger keys... upgrade the weaker end rather than dragging the stronger end down"), with the correct caveat structure (no legacy peer forcing a ceiling).
  4. 04
    AREA-MISMATCH SILENT-DISCARD MECHANISM (core OSPF teaching moment)
    adjacency staying at zero (not Init, not flapping) when both interfaces are up, POINTOPOINT, correct cost, sending hellos every 10s into a proven-working tunnel — the header's area stamp is checked BEFORE any other OSPF processing; mismatch = silent discard at the door, not "processed and disputed." Student derived roughly two-thirds of this independently, needed the closing mechanism ("checked before OSPF reads anything else") supplied. Explicitly cross-referenced to S15's Chatswood broadcast-on-tunnel fault: SAME symptom (empty table = hellos dying before processing), DIFFERENT mechanism (sender-side death vs receiver-side death) — "same fingerprint, different murderer," a genuine student-adjacent phrase worth reusing.
  5. 05
    BACKBONE PARTITION
    student's first instinct was to place the DR2PAR backup link in Area 1, reasoning from LSDB-size/blast-radius concerns (a real, legitimate area- design instinct, but wrong target for this scenario). Corrected via a scenario walk: with ParraAgg down and DR2PAR in Area 1, Parramatta's Area-0 LAN becomes an isolated island with no Area-0-only path to DR's Area-0 LAN — because inter-area traffic MUST transit the backbone, and a non-backbone area cannot serve as a relay between two backbone fragments. Student needed this named explicitly ("backbone partition") but, once told, correctly stated the resulting fix requirement (link must stay Area 0) even though he could not initially articulate the "shape" question Señor was pushing for. Filed as a newly-taught concept, not yet a derived one — needs a clean unprompted retrieval next time it comes up.
  6. 06
    ADMINISTRATIVE DISTANCE OVERRIDING A HEALTHY OSPF ROUTE
    the session's hardest fault. Full adjacency, correct LSDB (verified via `get router info ospf database router <id>` — DR2PAR link present and correctly described on both ends), yet `get router info routing-table ospf` showed NO route for the neighbor's LAN at all (not deprioritized — absent). Root cause only visible via `get router info routing-table all`: a pre-planted static route (admin distance 10, default) to the same prefix via the OLD path, silently beating OSPF's default 110 without any config-level "wrongness" — the LSDB, the adjacency, and Dijkstra were all correct the whole time. Student's first fix instinct (redistribute the static into OSPF) was WRONG but reasoned to correction cleanly: asked whether redistribution deletes the original static route, and correctly retrieved the S15 precedent (redistributing 10.99.50.0/24 never removed the hub's own static) to conclude redistribution would not help. Landed on the correct fix (disable/remove the static override) once AD-vs- cost was fully separated from AD-vs-route-existence.
  7. 07
    `get router info routing-table ospf` (protocol-filtered) VS `get router info routing-table all` (every protocol, shows the actual winner) — the single most important command-selection lesson of the session
    A protocol-filtered view can look "empty" for a route that fully exists elsewhere in the table under a different, higher-priority source.
  8. 08
    FORTIMANAGER IPSEC TEMPLATES (concept only, hands-on deferred to S18)
    the shared- structure-vs-per-site-variable split, worked out via a table the student was walked through rather than lectured: Phase 1 (shared: proposal/DH group/PSK method/DPD; variable: peer public IP, tunnel name), Phase 2 (shared: proposal/PFS group/ encapsulation; variable: src/dst selectors, tunnel-interface IPs), OSPF (shared: network-type/cost/timers; variable: interface name, area, network statement). Framed explicitly as "the boss battle we just fought is the exact failure mode templates exist to prevent" (the DH-group mismatch specifically — a value that drifted because it was set by hand on each end).
