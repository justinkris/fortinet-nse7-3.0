Learning Objectives

What this finale tested

  • Diagnose, don't configure. Prove what the traffic is doing before touching anything.
  • Separate stacked faults and identify which one owns a given packet — rather than blaming the loudest symptom.
  • Commit precise root-cause verdicts that match your own evidence (the S26 labelling weakness, tested hard).
  • Fix with the business in the room — the right depth of inspection, not the most powerful.
Fault 1 — Over-blocking

Certificate warnings across corp

The evidence ladder: browse to the broken site → inspect the certificate → check the client trust store.

Browser cert viewer — a re-signed certificate
Issued to:  www.asx.com.au
Issued by:  LoFinance_Deep_Inspect_CA      <- not the real chain

⚠ signed by an issuer this device does not trust

The workstation's Trusted Root Certification Authorities store did not contain LoFinance_Deep_Inspect_CA. Then the profile:

NGFW-HQ # show firewall ssl-ssh-profile
edit "deep-custom-outbound"
    config https
        set status deep-inspection
    end
    set server-cert-mode re-sign
    set caname "LoFinance_Deep_Inspect_CA"
🧠
Root cause
The outbound full-inspection profile is re-signing correctly with the FortiGate's own CA — but that CA was never distributed to the client trust stores. The re-signing is correct behaviour; the missing distribution is the fault.

Fix: distribute LoFinance_Deep_Inspect_CA to every corporate device (GPO / MDM), or point the profile's caname at a CA the clients already trust. Don't disable inspection to make the warning go away — that trades a cosmetic problem for a security hole.

Fault 2 — Under-protecting (two faults on one policy)

Trading floor uninspected; gambling block bypassed

The session table was the first tell — trading web egress carried no inspection fingerprint:

NGFW-HQ # diagnose sys session list (trading src)
state=log may_dirty ... av_idx=0
192.168.20.51:51442 -> 203.0.113.88:443

av_idx=0 and a session that is merely logged and forwarded mean no UTM object is bound to this flow. Flow debug named the policy:

NGFW-HQ # diagnose debug flow ...
msg="Allowed by Policy-7: SNAT"
# no do_ssl line, no UTM stage — straight to SNAT + forward

The policy dump revealed two independent faults:

PolicyWhat it doesProblem
Policy 3Corp web out, full UTM stacksrcaddr = Exec/Finance/Support — not Trading
Policy 7Trading-Floor-Internet, service ALL, no profilesWide-open accept, no UTM, and sits above Policy 5
Policy 5Block-Gambling web filter, srcaddr allDead code for trading — never reached (first-match-wins)
🧠
Root cause
Policy 7 is both mis-ordered above the gambling block AND carries no UTM. Fixing the order alone still leaves an uninspected policy — the second fault hides behind the first.
The business-aware fix
Don't drop the trading floor into Finance's full deep-inspection profile. Full re-sign decryption costs microseconds on every trade — and George Kanellis measures microseconds in money. A scoped trading-egress policy carrying the gambling web filter under certificate-inspection reads the SNI and cert CN/SAN to categorise the destination without decrypting a single byte. Category enforced, latency untouched. "Blocks the betting sites without adding a microsecond to a trade."
Fault 3 — The silent signature

The attack that executed

Right box (DCFW), right direction (inbound to the server). Flow debug told a surprising story:

DCFW # diagnose debug flow ...
do_ssl: profile deep-inspect-trading, state=works, matched
IPS receives session, sensor=protect-trading-api
IPS processing, no signature match, action=pass

Deep SSL inspection was working. IPS was attached and receiving decrypted payload. And it still passed the attack — no signature match. That rules out the SSL layer entirely. The fault is inside the sensor:

DCFW # show ips sensor
edit "protect-trading-api"
    config entries
        edit 1
            set severity high critical
            set location server
            set status enable
            set action monitor          <- watch-only (fault behind the fault)
        next
        edit 2
            set rule 51398
            set status disable           <- OWNS this packet
        next
    end
🧠
Root cause — and the discrimination
"no signature match" proves Entry 1 never matched this attack (Monitor still logs on match — there was no log). The owning fault is the explicitly disabled rule 51398. Monitor is the second-order risk: even after re-enabling 51398, the broad filter only watches everything else until it flips to Block. Two fixes.

Fix: set status enable on rule 51398, and flip Entry 1's action from monitor to block.

Common Mistakes

Traps this battle set

  • Blaming SSL inspection for the silent IPS. The state=works, matched line proves decryption ran — the fault is downstream in the sensor.
  • Blaming the loud Monitor action. "no signature match" means Monitor never fired; the disabled rule owns the packet.
  • Re-ordering without adding UTM. Policy 7 has two faults; moving it fixes one.
  • Giving trading Finance's deep inspection. Correct security, wrong depth for the latency path — cert-inspection is enough.
  • Omitting diagnose debug flow trace start N — no trace count, no output.
Senior Engineering Insights

What a senior keeps

Follow the packet, not the dashboard. Every health check was green because the sabotage lived inside the inspection logic, not the path. Only the flow trace and the sensor config told the truth.
Find the fault that owns THIS packet. When faults stack, evidence — not volume — tells you which one is responsible. Sign the verdict the evidence proves.
Right depth, not maximum depth. The most powerful tool is rarely the correct one. Match inspection to the threat surface and price it in the stakeholder's currency.
Exam Tips

NSE7 angles

  • SSL deep inspection running + IPS attached + "no signature match" → fault is inside the sensor, not the SSL layer. Classic "everything upstream looks healthy" trap.
  • Certificate inspection categorises HTTPS via SNI (ClientHello) + cert CN/SAN — no decryption needed for web-filter category blocking.
  • Monitor always logs on match → "no log at all" is proof of no match; look for a disabled/absent entry.
  • Policy evaluation is top-down first-match-wins; a broad accept above a specific block makes the block dead code.
