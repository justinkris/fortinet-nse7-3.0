Inspecting the Inspector

How a senior engineer isolates faults when the security profiles themselves are the fault — and every health check is green.

Follow the packet, not the dashboard.
The emphasis is on thinking — command selection, evidence order, and committing precise verdicts — not memorising syntax.

Questions to ask first

Before any command

Scope the problem
Who is affected and who is not? (Corp yes, trading no — that split IS evidence.)
Which box and which direction owns this flow? (Outbound NGFW vs inbound DCFW are different investigations.)
Did something break, or was something changed? (Cowboy hat = changed. Look for deliberate, not drifted.)
Is the health check green because it's fine, or because the fault is invisible to that check?
Verification commands & expected output

The ladder

1 — is UTM even bound to this flow?
diagnose sys session filter clear
diagnose sys session filter src <ip>
diagnose sys session list
# READ: av_idx=0 and a plain "log may_dirty" session = NO UTM bound.
#       av_idx non-zero = a profile is scanning it.
2 — which policy matched, and did inspection run?
diagnose debug flow filter clear
diagnose debug flow filter saddr <src>
diagnose debug flow filter daddr <dst>
diagnose debug flow filter dport <port>
diagnose debug flow trace start 20      # <-- REQUIRED or you get nothing
diagnose debug enable
# READ: "Allowed by Policy-N" = the matched policy.
#       do_ssl ... state=works, matched  = SSL inspection ran.
#       ips_receive... / no signature match = IPS ran, found nothing.
#       NO do_ssl / NO ips line = traffic bypassed inspection entirely.
3 — read the policy order and the sensor
show firewall policy     # order in config = evaluation order (first match wins)
show ips sensor          # Filter entries vs signature-override entries;
                         # action monitor (log+pass) vs block; set rule N status

Common dead ends

Where engineers waste time

  • Disabling SSL inspection to clear cert warnings. Cosmetic fix, real hole. The warning is a distribution problem, not an inspection problem.
  • Re-ordering policies without checking UTM. Fixes the visible fault, leaves the invisible one.
  • Blaming the Monitor action for a silent attack. Monitor logs on match — silence means it never matched.
  • Assuming a green dashboard means healthy. Inspection-logic faults leave the path intact; only the security logs / flow trace reveal them.
  • Forgetting trace start N. Filters set, debug enabled, and nothing appears — because no trace count was given.
Likely root causes

Ranked by this scenario

  1. Signature disabled by rule ID — silent, deliberate, invisible to health checks. The hardest to spot.
  2. Policy without UTM above a blocking policy — two faults masquerading as one.
  3. CA distribution gap — loud and obvious, easiest to over-react to.
  4. Broad filter on Monitor — the fault behind the fault; matters after you fix the first.
Exam traps

What NSE7 does with this

The healthy-upstream trap. Everything before the failing stage looks fine (route resolves, SSL works, IPS attached). The answer is the one downstream thing that's misconfigured — commit to it.
The plausible-but-not-graded distractor. "No SSL inspection could cause a silent IPS" is TRUE in general but wrong when the trace shows SSL working. Grade the stem's claim, not adjacent truths.
Cert vs deep for category blocking. The distractor offers "deep inspection required" — but SNI + CN/SAN under certificate inspection already categorises the site. Deep is for payload.
The personalised framework

Your seven steps

  1. Restate the business symptom — and note who is / isn't affected (the split is evidence).
  2. Pick the box and direction that own the flow before running anything.
  3. Is UTM even bound? session list → read av_idx.
  4. Which policy, and did inspection run? flow debug → "Allowed by", do_ssl, ips lines.
  5. Gather evidence before changing config — read the policy order and the sensor.
  6. Name the fault that owns THIS packet — evidence over volume; distinguish it from the fault-behind-the-fault.
  7. Verify recovery AND root cause — replay the attack, re-check the session, and price the fix for the stakeholder.
