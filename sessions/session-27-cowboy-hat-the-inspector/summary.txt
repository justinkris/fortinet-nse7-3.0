THREE ROOT CAUSES (all found, all correctly signed by the student)
------------------------------------------------------------------
SYMPTOM 1 — Corp cert warnings.
  Root cause: the NGFW outbound full-inspection profile `deep-custom-outbound` is
  re-signing with CA `LoFinance_Deep_Inspect_CA`, which is ABSENT from the corporate
  client trust stores (never distributed / rolled back). The re-signing is CORRECT
  behaviour; the missing CA distribution is the fault.
  Evidence: browser cert viewer showed issuer = LoFinance_Deep_Inspect_CA; the
  workstation Trusted Root store did not contain it; `show firewall ssl-ssh-profile`
  confirmed re-sign + caname.
  Fix: distribute the CA to all corporate trust stores (GPO/MDM), or correct the
  profile's caname to a CA already trusted.

SYMPTOM 2 — Trading floor uninspected + gambling block bypassed. TWO STACKED FAULTS.
  (a) Ordering: Policy 7 "Trading-Floor-Internet" sits ABOVE Policy 5
      "Block-Gambling"; first-match-wins stops evaluation at 7, so the gambling web
      filter never runs for VLAN20. Policy 5 is dead code for trading.
  (b) No UTM: Policy 7 is `service ALL`, `utm-status` absent, zero profiles — a
      wide-open accept. Even correct ordering wouldn't inspect it.
  Evidence: `diagnose sys session list` (av_idx=0, no UTM binding, session just
  logged/forwarded) → `diagnose debug flow` ("Allowed by Policy-7: SNAT", no
  inspection stage) → `show firewall policy` (revealed the ordering + the bare Policy 7).
  Fix (business-aware — the student reasoned this himself): a SCOPED trading-egress
  policy carrying AT MINIMUM the gambling web filter under CERTIFICATE-INSPECTION
  (reads SNI + cert CN/SAN to categorise, no decryption) — NOT full deep inspection,
  because latency on the trading path costs money on every trade. "Know the traffic
  first, then build." George-facing framing: "blocks the betting sites without adding
  a microsecond to a trade."

SYMPTOM 3 — Trading-API attack executed with no IPS event. TWO FAULTS, one owns the packet.
  Root cause (owns THIS packet): IPS signature rule 51398 was explicitly DISABLED in
  the sensor `protect-trading-api` (config entries / edit 2 / set rule 51398 /
  set status disable). Flow trace read `no signature match` — proving deep SSL
  inspection WAS working (do_ssl profile deep-inspect-trading, state=works, matched;
  IPS receives session; sensor attached) and the failure was the disabled signature,
  NOT the SSL layer and NOT the Monitor action.
  Second-order fault: Entry 1 (broad filter, severity high/critical, server-side)
  is on `action monitor` — so re-enabling 51398 fixes THIS attack, but the sensor
  still only watches everything else until Entry 1 flips to Block. Two fixes required.
  Evidence: correct box (DCFW) + correct direction (inbound) → `diagnose debug flow`
  → `show ips sensor`.
  Fix: `set status enable` on rule 51398; flip Entry 1 action monitor → block.

THE ACTOR: Señor (Cowboy Hat convention). All three faults are deliberate sabotage —
