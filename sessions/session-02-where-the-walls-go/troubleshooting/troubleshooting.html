SESSION 02 · TROUBLESHOOTING GUIDE · THINK LIKE SEÑOR

The Misplaced Wall

Placement problems don't announce themselves as placement problems. They arrive as Riley tickets: "the traders say everything is slow" or "the branch feels weird." This guide is about the thinking.

The symptoms you'll actually receive

  • "Everything got slow after the security change." Classic signature of deep inspection enabled at a high-volume internal point (Junior's mistake #1).
  • "The branch internet is fine but internal apps crawl." Backhaul path or undersized DEFW carrying more than "basic."
  • "The data centre firewall CPU is pinned but the internet is quiet." East-west load — the traffic that never leaves the building.
  • "We bought an NGFW but it's not doing NGFW things." Role confusion: features exist but were never enabled; or enabled on the wrong box.

Questions a senior engineer asks before touching anything

  1. Which direction is the affected traffic? North-south (crosses the perimeter) or east-west (stays inside)? This alone halves the suspect list.
  2. What changed? New profile enabled? New segment? New branch? Placement problems are almost always born in a change window.
  3. What is this box's intended role — and what is it actually doing? Compare the design (ISFW = segmentation, low latency) with the config (someone enabled six UTM profiles on every policy).
  4. Is the load proportional to the placement? A 100-series box between doubled trading floors is doing a 1000-series job.
  5. Where is deep inspection supposed to happen for this flow? If the answer is "twice" (e.g., NGFW gateway AND DCFW running the same engines on the same flow), you may have found redundancy — and your latency.

Verification commands and what to expect

Step 1 — identity: what is this box?
get system status
# Expect: model (tells you the series → intended placement class),
# FortiOS version, hostname, operation mode.
# A FortiGate 60F "temporarily" firewalling a growing campus is a finding.
Step 2 — load: how hard is it working?
get system performance status
# Expect: CPU per-core, memory, average network usage, session setup rate.
# Healthy internal firewall: low CPU even at high throughput (hardware path).
# High CPU + modest throughput = traffic being inspected/processed in software.

diagnose sys session stat
# Expect: total sessions, setup rate. East-west-heavy points show session
# counts far beyond what the internet link alone could explain.
Step 3 — posture: what inspection is this box actually doing?
show firewall policy
# Read the policies on the affected path. Which security profiles are set?
# Does the posture match the placement (segmentation point vs edge)?
Interpretation habit: the commands don't tell you "the wall is misplaced." They give you three facts — identity, load, posture — and you compare them against the reference architecture. That comparison is the skill.

Decision tree

Complaint: "slow" / "weird" after a change │ ├─ Q1: Which direction is the affected traffic? │ ├─ North-south → suspect the EDGE (NGFW duties, profiles, sizing) │ └─ East-west → suspect INTERNAL points (ISFW/DCFW) — continue │ ├─ Q2: Did inspection posture change at an internal point? │ ├─ YES → Is deep inspection needed HERE, for THIS flow? │ │ ├─ NO → move it (edge NGFW / gateway in front of DC) │ │ └─ YES → is it hardware-assisted (HW SSL, NP offload)? │ │ ├─ NO → wrong box for the job → resize/redesign │ │ └─ YES → tune per-segment, not global │ └─ NO → continue │ ├─ Q3: Is the box's series proportional to its placement? │ ├─ NO (e.g., 60F doing campus duty) → step up the ladder │ │ 30–90 → 100–900 → 1000–7000 │ └─ YES → continue │ └─ Q4: Branch involved? ├─ Backhaul design → check the TUNNEL + HQ inspection point, │ not the DEFW's (absent) UTM └─ Local breakout → inspection must exist LOCALLY; DEFW may be undersized by design

Common dead ends

  • Blaming the model number first. "We need a bigger box" is sometimes true — but check posture before budget. Removing one unnecessary profile from an internal path is free.
  • Adding inspection to fix a security worry at the wrong point. The fix for a phished quant is inspection where the threat crosses a boundary, not IPS on every replication stream.
  • Treating the DEFW's missing UTM as the problem. In a backhaul design, that's the design. Look at the tunnel and the HQ inspection point instead.
  • Assuming "NGFW" on the invoice means NGFW in the config. Roles are enabled, not shipped.

Likely root causes, ranked

#Root causeTell-tale
1Deep inspection enabled at a high-volume internal pointSlowness begins at a change window; CPU up, throughput flat
2Box undersized for its placement (series ≠ job)Chronic load, no recent change; growth event in the story
3Duplicate inspection (gateway NGFW + DCFW same engines, same flow)Latency doubles across two hops; both boxes busy
4Branch outgrew its DEFW (local breakout / headcount)Branch-local symptoms; HQ path clean
5Role never enabled ("we bought an NGFW")Security expectation unmet; policies carry no profiles

Exam traps in this territory

Trap 1 — "must be replaced": any option saying a 100–900 box must be replaced to run UTM profiles is wrong. It becomes an NGFW by enabling them.
Trap 2 — "equal protection = equal config": a backhauling DEFW without UTM is not "unprotected" — inspection lives at HQ. Watch for the local-breakout twist that flips the answer.
Trap 3 — the series-number decoy: NGFW spans 100–7000 + VM/cloud. A question hinting "NGFW is the 1000-series tier" is testing whether you know it's a role.
The senior reflex to build: Direction → Change → Role-vs-posture → Proportionality. Four questions, in that order, before any command beyond get system status.
