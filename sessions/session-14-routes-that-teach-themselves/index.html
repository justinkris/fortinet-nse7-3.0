Concepts Covered 9 concepts
  1. 01
    WHY STATIC ROUTES DON'T SCALE (built from the student's own Parramatta experience, not taught cold)
    a static route only knows what a human typed — it has zero awareness of the rest of the network's shape, and doesn't ask "is there a better path now?" The Parramatta outage demonstrated this isn't just "typos happen" — even a PERFECTLY configured static route set still made a dangerous decision (unencrypted fallback to default route) because nothing on the box knew an alternative existed.
  2. 02
    DISTANCE-VECTOR VS LINK-STATE (student coined "routing by rumour" for distance- vector, unprompted — strong, reusable analogy)
    distance-vector routers trust secondhand summaries from neighbors (Chinese-whispers risk — stale/wrong info propagates undetectably). Link-state protocols like OSPF reject this: EVERY router builds its own complete, independent copy of the network topology and calculates routes from raw data, never from a neighbor's opinion.
  3. 03
    LSDB (Link-State Database)
    the identical topology database every router within an area maintains. Since every router runs Dijkstra's algorithm on an IDENTICAL LSDB, answers are always internally consistent — never contradictory, even though each router's shortest-path TREE is rooted at itself (student briefly conflated "different tree root" with "disagreement," self-corrected with one Socratic nudge distinguishing "same map, different starting point" from "different maps").
  4. 04
    OSPF QUIETNESS
    periodic full refresh every 30 minutes; otherwise silent except for triggered updates on actual topology changes — student's Q4 exam answer captured the concept correctly ("don't broadcast when unnecessary, only on demand") but used imprecise terminology; exact term is "triggered/incremental LSA flood," worth one terminology-only rep next session (continuing the S13 naming-recall watch item).
  5. 05
    SCALING PROBLEM WITH ONE FLAT LSDB
    student independently derived BOTH costs when asked — (a) database size growing unbounded with every new site, and (b) blast radius: a single link flap ANYWHERE forces a full SPF recalculation EVERYWHERE, recreating the fragility static routing was meant to fix, just automated.
  6. 06
    AREAS as the fix
    carving the network into zones so a topology change stays inside its area. The ABR's job, described almost completely correctly by the student before being told: holds the FULL/detailed LSDB for each area it touches, but exports only a SUMMARY (prefix + cost) across the boundary — "reach this, go through me." One precision correction supplied: ABR maintains a SEPARATE full LSDB per connected area (not a shared/tracked view) — student had said "keep track of," refined to "maintains one full LSDB PER area."
  7. 07
    TYPE 3 SUMMARY LSA (named for the student after he described its function unprompted)
    generated only by ABRs, carries a prefix+cost summary, deliberately discards the area's internal topology detail — this detail-loss is BOTH the scaling win (small, cheap to send) AND the reason for the backbone rule (see #8).
  8. 08
    BACKBONE AREA (0.0.0.0) MANDATORY HUB RULE
    every non-backbone area MUST connect to Area 0; areas may not exchange summaries directly with each other. Mechanism (student reasoned to correctly after two nudges): once you're at a summary — prefix + cost, detail already discarded — a router has NO WAY to independently verify a route it's about to re-advertise isn't the same one it just received from another direction. Without full topology detail, loop DETECTION is impossible; OSPF's actual solution is structural loop PREVENTION — forbid the topology shape (direct inter-area shortcuts) that could produce an undetectable loop in the first place, rather than trying to detect it after the fact. Student's own words: "they threw away visibility into the topology and would need a map to detect the loop" — this is the correct mental model, arrived at through guided reasoning rather than being told directly.
  9. 09
    ROUTER ROLE TAXONOMY (all four correctly distinguished by session end)
    - Internal router: all interfaces in one area, one LSDB. - Backbone router: at least one interface in Area 0 (an ABR is automatically also a backbone router, since it always touches Area 0 — this compound-role point was tested directly in Q3 and initially MISSED — see below). - ABR: interfaces in multiple (OSPF) areas, one full LSDB per connected area, always touches the backbone. - ASBR: imports non-OSPF routes (static, RIP, foreign AS) into OSPF via redistribution. Student correctly and independently distinguished ASBR's job from ABR's — "different from what an ABR does" — ABR connects two areas that are BOTH already OSPF; ASBR is the doorway where something foreign enters. A single physical router CAN hold multiple roles simultaneously (ABR + backbone router + ASBR, if it touches Area 0, another area, AND redistributes statics) — this compound-role concept was the one gap surfaced in the exam questions (Q3).
