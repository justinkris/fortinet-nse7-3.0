SESSION 14 · TROUBLESHOOTING · OSPF REASONING

Thinking Like a Senior Engineer

This session was conceptual, not incident-driven — so this guide focuses on the REASONING questions a senior engineer asks before touching OSPF config, not commands.

WHY THIS MATTERS

No Config Yet — But Reasoning Starts Now

Session 14 built the mental model OSPF troubleshooting will depend on in later sessions. A senior engineer diagnosing an OSPF adjacency failure, a missing route, or an unexpected summary doesn't start with commands — they start by asking "which area is this happening in, and which router role is responsible for what I'm seeing?"

SYMPTOMS THAT WILL POINT BACK HERE

Symptom Patterns to Recognize Later

  • "A route from another area just disappeared." — First question: did the ABR responsible for that area go down? A single ABR per area is a single point of failure for every summary LSA from that area.
  • "Two areas seem to be exchanging routes directly, and something looks wrong / inconsistent." — This should never be possible in a correctly designed network. If it's happening, the backbone rule has been violated somewhere (a virtual link or misconfiguration bypassing Area 0).
  • "This router isn't doing what I expect for its role." — Before troubleshooting further, check the router against EVERY role definition (internal / backbone / ABR / ASBR) — it may hold more than one role, and the config or behaviour you're troubleshooting might belong to a role you didn't consider.
  • "External routes aren't showing up in OSPF." — Check for an ASBR performing redistribution; by default, protocols are NOT redistributed into OSPF (redistribute settings are disabled by default per protocol).
QUESTIONS TO ASK FIRST

Before Touching Any Configuration

1
Which area is this? Backbone (0.0.0.0) or a numbered area? This changes which rules and which router (ABR vs internal) are relevant.
2
Is the affected router an ABR, ASBR, backbone router, or internal router — or several at once? Don't assume a single role; check each definition independently.
3
Is the missing/wrong route a summary (Type 3) or should it be a full internal route (Type 1)? A Type 3 issue points to the ABR; a Type 1 issue points to internal-area topology.
4
Does this route originate inside OSPF, or is it external (redistributed)? External routes point to an ASBR and redistribution settings, not area/ABR logic.
5
Is Area 0 contiguous and is every non-backbone area actually connected to it? A broken or missing backbone connection is a structural design fault, not a transient issue.
COMMON DEAD ENDS

Where This Reasoning Gets Derailed

Dead end 1: Assuming a router only holds ONE role. A device can be ABR + backbone router + ASBR simultaneously — if you diagnose it as "just an ASBR," you'll miss ABR-specific summary behaviour it's also responsible for.
Dead end 2: Treating a missing route as automatically an ABR/summary problem. If the route should be a Type 1 (internal, same-area) LSA, the fault is inside the area's own topology — checking the ABR first wastes time.
Dead end 3: Assuming OSPF constantly re-announces everything, so "it'll just refresh in a minute." OSPF is quiet by design — a stale route may sit until the 30-minute refresh unless a real topology-change event triggers an update. Don't assume time alone fixes it.
LIKELY ROOT CAUSES

What's Usually Actually Wrong

  • Single ABR serving an area with no redundancy (design gap, not a bug).
  • Redistribution not enabled on the ASBR for the specific protocol needed (default: disabled per protocol).
  • An area configured without a path to the backbone (violates the mandatory hub rule).
  • Confusion between "route summarization" (deliberate, ABR-only, via config range) and "route missing" (a fault).
EXAM TRAPS

Distractor Patterns to Watch For

Trap 1: "OSPF areas can connect directly to each other for efficiency" — FALSE. Always through Area 0.
Trap 2: "An ABR keeps a single shared LSDB across all its areas" — FALSE. One full, separate LSDB per connected area.
Trap 3: Questions listing router roles as single-select when the correct answer is "select all that apply" — always check compound roles.
Trap 4: "OSPF continuously floods LSAs" — FALSE. 30-minute periodic refresh, event-triggered otherwise.
