SESSION 06 · DECISION-MAKING GUIDE

Choosing (and Verifying) a VDOM Type

The Real Skill Here

This session isn't a broken-config troubleshoot — it's a design decision under a business constraint. The senior-engineer skill is asking the right questions BEFORE typing any CLI, so you don't build the wrong VDOM type and have to unwind it later.

Questions To Ask Before Choosing a VDOM Type

  • Does this VDOM need to pass and route real user traffic — or is it purely for administering the box itself? (→ Admin vs Traffic/LAN-Extension)
  • If it passes traffic — can the network's IP addressing and routing topology change, or must it stay invisible to connected devices? (→ NAT vs Transparent)
  • Is the goal to inspect traffic locally, or purely to tunnel a remote site's LAN back to a central controller? (→ Traffic vs LAN Extension)
  • How close is the FortiGate to its VDOM ceiling (default 10)? Check before promising a client anything is "free."
  • Will this VDOM need internet/FortiGuard access independently, or can it rely on the existing management VDOM?

Decision Tree

Does the VDOM need to pass user traffic? │ ├─ NO → ADMIN VDOM (max 1 per box, mgmt/licensing/updates only) │ └─ YES → Is the traffic being routed/inspected locally, or just tunneled to a central controller? │ ├─ Tunneled to controller → LAN EXTENSION VDOM │ (Network + VPN modules only) │ └─ Routed/inspected locally → Can the network topology (IPs, routing) change? │ ├─ YES → TRAFFIC VDOM, NAT MODE │ (full router/firewall — most common) │ └─ NO → TRAFFIC VDOM, TRANSPARENT MODE (invisible L2 inspection — remember: still needs 1 manageip)

Verification Commands

  • get system status | grep "Max number of virtual domains" — confirm the model's VDOM ceiling before promising capacity
  • get sys status — confirm current VDOM mode (multi-vdom enabled?)
  • Global VDOM > System > VDOM — view current VDOM list, types, and which is flagged as the management VDOM (red dot)
  • Global VDOM > System > Global Resources / Edit VDOM (Override Maximum) — check current resource usage before allocating a new VDOM's limits

Common Dead Ends

Dead endAssuming "isolated guest network" automatically means transparent mode. Isolation is about policy/interface separation (any traffic VDOM gives you that); transparent mode is specifically about topology invisibility, which a brand-new guest network doesn't need.
Dead endAssuming a new VDOM is "free" without checking the current VDOM count and resource headroom first — the honest recommendation always includes the ceiling.
Dead endForgetting the manageip requirement on transparent VDOMs and being surprised by a failed node-check error during config.

Exam Traps

Trap 1"Transparent mode needs no IP at all" — false. It needs exactly one manageip for admin access.
Trap 2Confusing "Global VDOM" (device-wide admin view) with "Admin VDOM type" (a traffic-free VDOM instance) — these are two different concepts that share the word "admin."
Trap 3Assuming the only way past the 10-VDOM default is a new physical box — licensing extends the VDOM count first.
