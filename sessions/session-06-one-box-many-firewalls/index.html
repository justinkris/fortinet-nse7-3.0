Concepts Covered 6 concepts
  1. 01
    Core VDOM model
    one physical FortiGate can host multiple independent virtual firewalls (VDOMs), each with its own isolated policy set, interfaces, and security config, all sharing the same underlying hardware. Student self-generated this via direct analogy to VRFs (router virtualization) before being taught — a strong, accurate cross-domain transfer of "carve the control plane, share the physical plane" from a prior networking concept.
  2. 02
    Global VDOM
    NOT a traffic-passing firewall instance. A Super-Admin-only administrative view for settings that affect the entire physical device (firmware, interfaces, some logging, sandboxing). Explicitly excludes Policy & Objects, VPN, and User & Authentication — student correctly reasoned these are excluded because they are inherently per-VDOM concepts; "which VDOM's policy?" is meaningless at the Global layer, directly paralleling "which VRF's routing table?" being meaningless outside any VRF.
  3. 03
    Management VDOM concept
    root VDOM becomes the management VDOM by default (can be changed). Its role — FortiGuard updates, license validation, admin HTTPS/SSH access, FortiManager control — is entirely distinct from passing user traffic. Student correctly inferred, before being told, that a VDOM could exist doing ONLY this and never touching customer traffic — this is the Admin VDOM type.
  4. 04
    VDOM Types (student built the full taxonomy Socratically)
    - Admin VDOM: management/admin only, zero user traffic, missing Policy & Objects, Security Profiles, VPN, User & Authentication, WiFi & Switch Controller. Limited to ONE per FortiGate (unlike traffic/LAN-extension VDOMs, which allow up to 10). - Traffic VDOM, NAT mode: full router/firewall behavior — what the student correctly selected for the guest-network use case, based directly on prior SRX migration experience (Corporate + Cloud VDOMs). - Traffic VDOM, transparent mode: invisible Layer 2 inspection appliance, no routing/NAT, network topology unchanged from the perspective of connected devices. Trade-off student articulated well: gives up full Layer 3 capability (routing, NAT, PBR, WAN load-balancing) in exchange for zero-topology-impact UTM inspection — used where a device/application is highly sensitive to any path change but compliance still requires inspection. GOTCHA (this is the one exam miss): transparent VDOMs still require ONE management IP (manageip) for admin access, even though they pass no routed/NAT'd traffic. Missing this produces a failed node-check error on 'manageip'. Student initially answered "no IP needed at all" — corrected in-session, reasoning re-anchored on "no routing ≠ no IP at all." - LAN Extension VDOM: tunnel-only, connects a remote connector (FortiGate or FortiExtender) to a central FortiGate access controller via IPsec VPN + VXLAN encapsulation. Student needed one hint to land on this (guessed "VPN and Routing" first) — corrected to Network + VPN modules only, reasoning that "routing decisions" ≠ "network module needed to build interfaces/tunnels at all" in a VDOM with no local policy engine.
  5. 05
    Resource governance
    Global Resources (whole-device limits) vs per-VDOM resources (editable per VDOM via Override Maximum, which must stay ≥ current usage). Shared hardware means shared limits unless explicitly governed — student raised this unprompted as the honest caveat to the "free" VDOM pitch.
  6. 06
    Default VDOM ceiling
    most FortiGate models cap at 10 VDOMs by default; extendable via additional licensing (student correctly guessed this by instinct before being confirmed) rather than requiring a second physical appliance as the only option to scale past the default.
