LOFINANCE — SESSION 06 REVISION PACK
"One Box, Many Firewalls" — VDOM Concepts and Types
=====================================================

QUICK SUMMARY
-------------
A FortiGate can be virtualized into multiple VDOMs (Virtual Domains) — up to 10 by
default, extendable via licensing — each with its own isolated firewall policy,
interfaces, and security config, all sharing one physical box. Enabling multi-VDOM
creates two protected VDOMs: Global (a Super-Admin-only, device-wide admin view —
NOT a firewall instance, and missing Policy & Objects / VPN / User & Authentication
because those are inherently per-VDOM concepts) and root (the default management
VDOM: FortiGuard updates, licensing, admin access). There are four VDOM types: Admin
(zero traffic, max 1 per box), Traffic-NAT (full router/firewall — most common),
Traffic-Transparent (invisible Layer 2 inspection, no routing/NAT, but still needs
ONE manageip for admin access), and LAN Extension (Network + VPN modules only,
tunnels a remote LAN to a central controller via IPsec/VXLAN). Resource limits are
governed at Global Resources (whole-device) and per-VDOM (Override Maximum) level.
Business lesson: a new VDOM can replace a $15k second firewall for use cases like an
isolated guest network — but the honest recommendation always names the real
constraint (VDOM ceiling, shared resources), not just "it's free."

CHEAT SHEET
-----------
Enable multi-VDOM:
  config system global
      set vdom-mode multi-vdom
  end

Check VDOM ceiling:
  get system status | grep "Max number of virtual domains"

Create Admin VDOM:
  config vdom
      edit "name"
          config system settings
              set vdom-type admin
          end
  end

Create Traffic VDOM (NAT):
  config vdom
      edit "name"
          config system settings
              set vdom-type traffic
              set opmode nat
          end
  end

Create Traffic VDOM (Transparent):
  config vdom
      edit "name"
          config system settings
              set vdom-type traffic
              set opmode transparent
              set manageip <ip>/<mask>
          end
  end

Create LAN Extension VDOM:
  config vdom
      edit "name"
          config system settings
              set vdom-type lan-extension
          end
  end

GUI paths:
  Enable/view VDOMs .......... Global VDOM > System > VDOM
  Switch management VDOM ..... Global VDOM > System > VDOM > Switch Management
  Global Resources ........... Global VDOM > System > Global Resources
  Per-VDOM resource limits ... Global VDOM > System > VDOM > Edit VDOM (Override Maximum)
  Global security objects .... Global VDOM > Security Profiles (prefix "g-")

DEFAULTS & LIMITS
  Default max VDOMs per FortiGate ........ 10 (traffic + LAN extension types)
  Admin VDOM max per FortiGate ........... 1
  Extend past default ..................... additional VDOM licensing
  Default management VDOM ................ root (changeable)
  Global VDOM access ...................... Super Admin only

KEY DESIGN RULES
  - Global VDOM never passes traffic and never holds Policy/VPN/User&Auth.
  - Admin VDOM never passes traffic; limited to exactly one per box.
  - Transparent VDOM passes traffic at L2 only, but ALWAYS needs 1 manageip.
  - LAN Extension VDOM only has Network + VPN modules — no local policy engine.
  - Global objects (prefix "g-") sync automatically across all VDOMs for
    consistent UTM profiles; reserved naming, don't duplicate it locally.
  - Changing a traffic VDOM to admin type DELETES its Policy & Objects, Security
    Profiles, VPN, User & Authentication config — confirmation required.

FLASHCARDS
----------
Q: What does a VDOM allow a single FortiGate to do?
A: Behave as multiple independent virtual firewalls, each with isolated policy,
   interfaces, and security config, sharing one physical device.

Q: What's the closest non-firewall analogy to a VDOM?
A: A VRF on a router — same principle of virtualizing the control plane on shared
   physical hardware.

Q: Is the Global VDOM a firewall instance?
A: No — it's a Super-Admin-only administrative view for device-wide settings
   (interfaces, firmware, some logging, sandboxing). It never passes traffic.

Q: Why does Global VDOM exclude Policy & Objects, VPN, and User & Authentication?
A: Those constructs are inherently per-VDOM; they're meaningless without a VDOM
   context to belong to.

Q: What is the default management VDOM, and can it change?
A: Root, by default. Yes, it can be changed to another VDOM via Switch Management.

Q: What does the management VDOM actually do?
A: Validates the license, downloads FortiGuard updates, and provides admin
   access/FortiManager connectivity for the whole device.

Q: How many Admin VDOMs can exist on one FortiGate?
A: Exactly 1.

Q: What's the default max number of traffic/LAN-extension VDOMs per FortiGate?
A: 10 — extendable via additional licensing.

Q: What does a Traffic VDOM in NAT mode do?
A: Functions like a regular firewall — full routing, NAT, and policy enforcement.

Q: What does a Traffic VDOM in transparent mode do, and what does it give up?
A: Operates purely at Layer 2, invisible to the network — no routing or NAT. Gives
   up full Layer 3 capability in exchange for zero-topology-impact inspection.

Q: Does a transparent VDOM need any IP address?
A: Yes — exactly one management IP (manageip), even though it passes no
   routed/NAT'd traffic.

Q: What happens if manageip is missing on a transparent VDOM?
A: A failed node-check object error for 'manageip'.

Q: What is a LAN Extension VDOM used for?
A: Connecting a remote connector (FortiGate/FortiExtender) to a central FortiGate
   access controller via IPsec VPN + VXLAN, extending a remote LAN back to the
   controller.

Q: Which two modules does a LAN Extension VDOM have access to?
A: Network and VPN only.

Q: How do you extend a FortiGate past its default VDOM ceiling?
A: Purchase additional VDOM licensing (a second physical appliance is the
   last-resort alternative, not the first option).

Q: What are Global Resources vs per-VDOM resources?
A: Global Resources govern limits for the whole device; per-VDOM resources are
   allocated per individual VDOM via Override Maximum (must be ≥ current usage).
