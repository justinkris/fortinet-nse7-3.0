● SESSION 06 · COMPLETE STUDY GUIDE

One Box, Many Firewalls

VDOM concepts and types — when and why to split a FortiGate · Study Guide 03, pp. 68–74

Story Recap

The Business Hook

Penny gets a vendor quote for $15,000: a dedicated, fully isolated firewall for a client's trading-floor guest network. Junior reaches straight for a rack diagram and new hardware. Señor stops him with one line: "Why buy a box when we already own ten inside this one?" The session is spent working out exactly what that means — and whether it's true.

Learning Objectives
  • Explain what a VDOM is and why it exists (virtualized firewall instances on shared hardware)
  • Describe the role of the Global VDOM and why Policy/VPN/User&Auth are excluded from it
  • Name and distinguish all four VDOM types: Admin, Traffic (NAT), Traffic (transparent), LAN Extension
  • Explain the management VDOM concept and its relationship to root
  • Understand default VDOM limits and how to extend them
  • Understand Global Resources vs per-VDOM resource allocation

1. The Core Idea: Virtualized Firewalls

A FortiGate is one physical appliance — one chassis, one CPU, one set of ports. A VDOM (Virtual Domain) lets that single box behave like up to 10 independent firewalls (default; more via licensing), each with its own isolated policy set, interfaces, and security configuration — all sharing the same underlying hardware.

Analogy — carve the control plane, share the physical plane This is the same trick as a VRF on a router: one physical router, multiple independent routing tables, each instance blind to the others' routes. A VDOM does it one layer up — one physical firewall, multiple independent policy engines, each instance blind to the others' rules.
config system global set vdom-mode multi-vdom end get system status | grep "Max number of virtual domains"

Enabling multi-VDOM mode automatically creates two VDOMs you cannot delete: the Global VDOM and the root VDOM.

2. Global VDOM — Not a Firewall Instance

The Global VDOM is a Super-Admin-only administrative view, not a traffic-passing firewall. It governs settings that affect the entire physical device: interfaces, firmware, some logging, sandboxing configuration.

Deliberately excluded from Global VDOM

  • Policy & Objects
  • VPN
  • User & Authentication
Why excluded: these constructs are inherently per-VDOM. A firewall policy is meaningless without a VDOM context to belong to — "which VDOM's policy?" is a nonsensical question at the Global layer, exactly like "which VRF's routing table?" is nonsensical outside any VRF.

Resource governance

ScopeLocationGoverns
Global ResourcesGlobal VDOM > System > Global ResourcesLimits for the whole physical device
Per-VDOM ResourcesGlobal VDOM > System > VDOM > Edit VDOMLimits per individual VDOM (Override Maximum must be ≥ current usage)

Shared hardware means shared limits (CPU, memory, session table) unless explicitly governed per VDOM — one noisy VDOM can starve the others without guardrails.

3. The Management VDOM

By default, root is the management VDOM. Its job: validate the license, download FortiGuard updates, and provide admin access (HTTPS/SSH) and FortiManager connectivity — for the whole device, regardless of which VDOM you're configuring traffic in.

The management VDOM can be changed to any traffic VDOM (Global VDOM > System > VDOM > Switch Management). If the management VDOM loses internet access or FortiManager/FDS connectivity, FortiGuard databases across all VDOMs can be affected.

4. The Four VDOM Types

Admin VDOM

Management/administration only — zero user traffic. Limited to one per FortiGate (unlike traffic and LAN-extension types, which allow up to 10 by default).

vdom: admin | |-- FortiGuard updates (allowed) |-- port1 (mgmt access only) |-- NO Policy & Objects |-- NO Security Profiles |-- NO VPN |-- NO User & Authentication |-- NO WiFi & Switch Controller
config vdom edit "admin-vdom" config system settings set vdom-type admin end end
GotchaChanging an existing traffic VDOM to admin type deletes its Policy & Objects, Security Profiles, WiFi & Switch Controller, User & Authentication configuration, and dashboards — FortiGate asks for confirmation before proceeding.

Traffic VDOM — NAT mode

Functions like a regular firewall: routes traffic, performs NAT, applies full Layer 3 policy. The most common VDOM type — used for ISFW, DCFW, NGFW, DEFW roles. This is the type selected for LoFinance's new guest-network VDOM.

config vdom edit "guest-wifi" config system settings set vdom-type traffic set opmode nat end end

Traffic VDOM — Transparent mode

Operates purely at Layer 2 — no routing, no NAT. The network looks unchanged to every connected device; the firewall is an invisible "bump in the wire" doing UTM inspection only.

Trade-offYou give up full Layer 3 capability (routing, NAT, policy-based routing, WAN load-balancing) in exchange for zero-topology-impact security inspection. Used where a device or application is highly sensitive to any change in its network path, but compliance still requires inspection.
config vdom edit "transparent-seg" config system settings set vdom-type traffic set opmode transparent set manageip 172.16.16.1/255.255.255.0 end end
Gotcha — exam favoriteTransparent VDOMs still require ONE management IP (manageip) for admin access, even though no routed/NAT'd traffic passes through. Omit it and you get a failed node-check error on 'manageip'. "No routing" does not mean "no IP address at all."

LAN Extension VDOM

Connects a remote connector (FortiGate or FortiExtender) to a central FortiGate access controller via IPsec VPN wrapped in VXLAN encapsulation — extends a remote LAN back to the controller as if it were local.

Only two modules available: Network (to build the tunnel interfaces) and VPN (the tunneling protocol itself). No Policy & Objects, no Security Profiles, no User & Authentication — this VDOM makes no local policy decisions.

config vdom edit "lan-ext-branch" config system settings set vdom-type lan-extension end end

6. Senior Engineering Insight

Before recommending "existing infrastructure" over a purchase, always surface the real constraint underneath the free option. In this session: the default 10-VDOM ceiling and shared CPU/memory/session-table resources across VDOMs. "Free" without a stated caveat is a sales pitch, not an engineering recommendation.

Exam Tips
Q1. Which VDOM type is missing Policy & Objects, Security Profiles, and User & Authentication — and why?
Show answerGlobal VDOM. It governs device-wide settings only and never passes user traffic, so per-VDOM constructs are meaningless there.
Q2. True or false: a transparent-mode traffic VDOM requires no IP address at all, since it operates purely at Layer 2.
Show answerFalse. It requires exactly one management IP (manageip) for admin access, even though it passes no routed/NAT'd traffic.
Q3. A FortiGate is licensed for the default VDOM count and the business needs an 11th. What are the options?
Show answerPurchase additional VDOM licensing to raise the cap (preferred), or as a last resort, deploy a second physical/virtual appliance.
Key Takeaways
  • One FortiGate can host up to 10 (default) independent VDOMs, each with isolated policy/interfaces/config
  • Global VDOM = device-wide admin view, not a firewall instance — excludes Policy/VPN/User&Auth by design
  • Root is the default management VDOM (updates, licensing, admin access) — changeable, but connectivity is essential
  • Four VDOM types: Admin (mgmt only, max 1), Traffic-NAT (full firewall), Traffic-Transparent (invisible L2, still needs manageip), LAN Extension (Network + VPN only, tunnel-only)
  • Resource limits are governed at Global and per-VDOM level — shared hardware needs explicit guardrails
