Seven Firewalls, One Junior

FortiManager in one breath: three layers, two arrows, one reserved letter.

The three layers

🏛️
Top to bottom
Global DB (company-wide) sits above local ADOMs (per-tenant walls) which each hold device databases (one config copy per firewall).

The two arrows

↔️
Direction is everything
Retrieve/Import = firewall → FMG (onboarding, do it first). Install = FMG → firewall (day-2). Assign = global → down into ADOMs.
Retrieve before you rule. FMG can't be master of a box it hasn't learned. Import the real config into the device database first — then all future edits happen on the FMG copy.

The three numbers

Firmware window: version X supports X, X-1, X-2. A 7.6 global ADOM covers 7.6, 7.4, 7.2. Reserved global prefix = g (hard-blocked — you cannot create a local g- object, not "shouldn't").

Where to draw an ADOM

🧱
The axis
Around the administrative + policy unit (who owns/audits/shares a policy package) — not the site. Firmware is a constraint that can force a split, never the reason you choose one. A naming prefix is a hope, not a wall.
Global vs local test: "True for everyone?" → global DB. "Just this tenant?" → their ADOM. Push everything global and you dissolve the wall — you may as well have one ADOM.
Say it to Penny (money first): "When the auditor asks me to prove all seven firewalls enforce the same standard, hand-config means I can't — that's a finding. FortiManager proves it from one source."
