BUSINESS PROBLEM
----------------
LoFinance absorbed Vaultline (HQ + 3 branches + 3 spares = 7 FortiGates). Managing
them by hand does not scale and — the real point — destroys CONSISTENCY, which in a
regulated financial firm under audit is a COMPLIANCE failure, not merely a labour cost.
Constraint from Penny: LoFinance and Vaultline management must be hard-walled from each
other (separate audits) yet share ONE FortiManager, across mixed firmware (7.4 + 7.6).

TECHNICAL CONCEPTS COVERED (all reasoned Socratically, then named)
-----------------------------------------------------------------
1. The real cost of manual management = N devices × M changes (student computed 7×20=140).
   The casualty is CONSISTENCY → in a regulated firm = an audit finding. This is the
   business framing of why central management exists.
2. FortiManager = the central box that collapses N×M back to M (or to 1 for shared items).
   Change the standard once; it installs to the firewalls.
3. DEVICE DATABASE (per firewall): FMG's own authoritative stored COPY of each managed
   firewall's actual configuration. One device DB per managed box (7 firewalls → 7 DBs).
   NOTE the student's slip: called it the "WHO" of the firewall — corrected to the WHAT
   (the config itself); WHO = the ADOM/admin boundary.
4. SOURCE OF TRUTH + SYNC: once managed, FMG's device DB is the master; you stop editing
   the box directly. But FMG can only be master of a firewall it has first LEARNED.
5. DIRECTION VOCAB (exam-critical):
   - RETRIEVE / IMPORT = FortiGate → FMG (onboarding; pull reality into the device DB FIRST).
   - INSTALL = FMG → FortiGate (day-2; push the master down).
   Student sequenced onboarding correctly unprompted: import first, then all changes on
   the FMG copy. "Retrieve before you rule."
6. ADOM (Administrative Domain): a partition inside ONE FortiManager isolating a group of
   devices — own admins, own policies, own objects, OWN FIRMWARE VERSION. Satisfies
   Penny's three constraints (separation + mixed firmware + one box) via two ADOMs:
   LoFinance (7.6) and Vaultline (mixed 7.4/7.6). A naming prefix is NOT an admin boundary.
7. WHERE TO DRAW THE ADOM BOUNDARY: around ADMINISTRATIVE + POLICY units (who owns/audits/
   shares a policy package) — NOT the org chart (site). Student self-generated the deeper
   principle: "if firewalls behave similarly they can share a policy package → same ADOM."
   Firmware is a CONSTRAINT that can FORCE a split, not the primary reason to choose one.
   Site may coincide with a boundary but never causes it.
8. FIRMWARE COMPATIBILITY WINDOW (exam numbers): a global ADOM / source at version X is
   compatible with local ADOMs at X and the TWO versions before it. 7.6 → supports 7.6,
   7.4, 7.2. Reserved global-object prefix = "g" (HARD-BLOCKED; cannot create local g-
   objects — student initially softened this to "would work but confusing," corrected).
9. GLOBAL DATABASE / GLOBAL ADOM: sits ABOVE the local ADOMs. Define a truly universal
   object/policy ONCE, then ASSIGN it DOWN into multiple ADOMs simultaneously. Collapses
   the "two ADOMs × M changes" re-creation of Junior's problem back to 1. The verb is
   ASSIGN (student named the layer but omitted the operation — reinforced).
10. WHEN NOT TO GO GLOBAL: global DB = ONLY genuinely company-wide things (syslog, NTP,
    DNS). Tenant-specific policy/objects stay in the tenant ADOM. Test: "true for
    EVERYONE, or just this tenant?" Everyone→global; just them→their ADOM. Student nailed
    the kill-shot: "push everything global = no admin boundary, may as well have one ADOM."

MENTAL MODELS DEVELOPED
-----------------------
- THREE LAYERS: Global DB (company-wide) > local ADOM (tenant/policy-unit boundary) >
  device database (the individual box's config). Company-wide / tenant-wide / device-specific.
- "The prefix is a hope" — naming ≠ enforcement; an admin boundary must be structural.
- N×M as the true cost metric of manual management; consistency as the regulated-firm casualty.
- "Retrieve before you rule" — onboarding direction runs before FMG can be source of truth.
- The global/local tension: one big ADOM = sharing but no walls; separate ADOMs = walls
  but no sharing; global-over-local = BOTH. That's why the architecture has the shape it does.

EXAM-RELEVANT NOTES
-------------------

- Firmware window rule: X, X-1, X-2 (7.6 → 7.4, 7.2). Common distractor: "one version"
