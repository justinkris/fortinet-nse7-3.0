🔧 SESSION 28 · TROUBLESHOOTING · SENIOR THINKING

Onboarding & ADOM Design — Thinking It Through

Session 28 is a design episode, not a fault hunt — so this guide teaches the reasoning a senior applies when standing up FortiManager for a new estate, and the traps waiting when they get it wrong.

The mindset

Design faults are quiet

A misconfigured ADOM layout doesn't throw an error on day one. It works fine — until an auditor asks a question, or a Vaultline admin sees a LoFinance box, or a syslog change silently applies to five firewalls out of seven. The senior skill here is catching the fault before it becomes an incident, by asking the right questions during design.

Symptom → diagnosis

The four ways this breaks

Symptom 1
"I onboarded a firewall and its whole config vanished."

Likely root cause: the wrong direction was run. Someone did an Install (FMG → FGT) on a device whose device database was empty, overwriting the live box with nothing — instead of a Retrieve/Import (FGT → FMG) to learn it first.

Questions to ask: Was the device database populated before the first push? Did onboarding start with Retrieve, or did someone jump straight to Install?

Rule that prevents it: Retrieve before you rule. FGT → FMG always runs first.
Symptom 2
"A Vaultline admin can see — and edited — a LoFinance firewall."

Likely root cause: separation was implemented as a naming convention (prefixes) rather than an ADOM boundary. Prefixes don't scope visibility; only ADOM membership + admin-to-ADOM assignment does.

Questions to ask: Are the two firms in separate ADOMs, or one ADOM with prefixes? Is each admin account scoped to a single ADOM?

Trap: "It's labelled clearly" is not isolation. The prefix is a hope.
Symptom 3
"I can't add a firewall to this ADOM — version error."

Likely root cause: the device's firmware is outside the ADOM's compatibility window. A source at version X supports X, X-1, X-2 only.

Questions to ask: What version is the ADOM? What version is the device? Is the gap ≤ 2 minor versions? (7.6 ADOM accepts 7.6/7.4/7.2; a 7.0 device does not fit.)

Resolution paths: upgrade the device into the window, or create a separate ADOM at a compatible version. This is firmware forcing a split — the legitimate case.
Symptom 4
"I changed the syslog server once but only some firewalls updated."

Likely root cause A: the object was defined per-ADOM (local) instead of in the global database, so editing one ADOM's copy left the other untouched. Cause B: it is global, but was never assigned down to the ADOM that missed it, or assigned-but-not-installed to the devices.

Questions to ask: Is this object global or local? If global, was it assigned to every relevant ADOM? After assign, was it installed to the devices?

Chain to verify: define (global) → assign (into ADOM) → install (to device). Break any link and the change stops there.
The design decision tree

"Where does this belong?"

Q1 · Is this object/policy true for the WHOLE company?
YES → put it in the global database; assign down to each ADOM.
NO → it's tenant-specific → go to Q2.
Q2 · Which administrative/audit unit owns it?
→ Place it in that tenant's local ADOM (LoFinance or Vaultline). Never push tenant-specific data global — that dissolves isolation.
Q3 · Does every device in that ADOM fit the firmware window (X, X-1, X-2)?
YES → one ADOM is fine.
NO → firmware forces a further split; create a second ADOM at a compatible version.
Q4 · Is a device already live and unmanaged?
→ Retrieve/Import first (FGT→FMG) to populate the device database, THEN manage. Never Install into an empty device DB.
Common dead ends

Where seniors don't waste time

  • Blaming the device for a version-add error before checking the ADOM's own version. The fault is the relationship, not the box.
  • Re-typing a "missing" object on the affected firewall instead of following the define→assign→install chain to find which link broke.
  • Tightening prefixes to fix a visibility leak. No number of naming rules creates an admin boundary.
  • Suspecting the global DB is "buggy" when a change didn't propagate — check whether assign and install actually ran first.
Exam traps

What the question is really testing

The trapThe correct read
"Separate ADOM because different firmware"Firmware is a forcing constraint, not the primary reason. Admin/policy boundary is primary. (False, if stated as primary.)
"Firmware window is one version" / "any version"It is X, X-1, X-2. 7.6 → 7.6/7.4/7.2.
"g- objects just shouldn't be used locally"They cannot be created locally — hard-reserved.
"Install to onboard a new firewall"Onboard with Retrieve/Import (FGT→FMG). Install is the day-2 push.
"Device database holds the policies"Device DB = device settings/config (the WHAT). Policy work belongs in the policy layer (S29).
"Put shared objects in one big ADOM"Shared-across-walls belongs in the global database; one big ADOM sacrifices isolation.
The framework

A senior's onboarding sequence

  1. State the business boundary first. Who must be isolated from whom, and who audits what? That defines your ADOMs.
  2. Check firmware fits. Every device in an ADOM within X, X-1, X-2 — or split.
  3. Retrieve before you rule. Import live devices into their device databases before any push.
  4. Lift the universal up. Company-wide objects → global database, assigned down.
  5. Verify the whole chain. define → assign → install. Confirm the change reached the actual box, not just the database.
The recurring question that catches design faults: "If I change this in one place, does every device that should change actually change — and does nothing that shouldn't?"
