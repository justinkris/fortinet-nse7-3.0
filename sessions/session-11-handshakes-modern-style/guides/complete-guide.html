SESSION 11 · SEASON 2 · BLUEPRINT 5.1 · pp. 200–206

Handshakes, Modern Style

IKEv2 exchanges · IKEv1 comparison · network-id — or: the day a 2014 blog met a shredder.

Story Recap

The Parramatta FortiGate arrived, and so did Junior — clutching a March 2014 blog post recommending IKEv1 aggressive mode "because it's faster." Cipher sat down without a word. Señor folded the printout into a paper aeroplane, binned it, and delivered the line of the season: "Faster — and it shouts your name across the internet while it does it." By the end of the session the blog was shredded, LoFinance had standardised on IKEv2, and Junior had accidentally asked the one question that made network-id necessary.

Learning Objectives

  • State the three jobs every IKE handshake must complete — and the order they must happen in.
  • Explain why IKEv1 aggressive mode exposes peer IDs, and why that is the same fact as why it is faster.
  • Describe the two IKEv2 exchanges (IKE_SA_INIT, IKE_AUTH), what travels in each, and why the total is four messages.
  • Compare IKEv1 main mode, aggressive mode, and IKEv2 across messages, security, and peer-ID exposure.
  • Explain what network-id does, why it is IKEv2-only, and configure it.

1 · The Three Jobs of a Handshake

Two FortiGates that have never met must, over a hostile wire, end up agreeing on and proving:

  1. Agree the language — negotiate encryption/integrity proposals. Both ends must speak the same ciphers.
  2. Grow the shared secret — symmetric ciphers need the same key at both ends, but sending a key over a monitored wire is suicide. Diffie-Hellman solves it: each side keeps a private value, exchanges only public halves, and the maths lets both ends compute the identical secret — while an eavesdropper holding both public halves cannot.
  3. Prove identity — transmit an identity ("I am HQ-FortiGate") plus proof (PSK-derived auth, signature, or EAP in IKEv2).
🔑 "The secret is grown, not posted." The DH key never crosses the internet because it never exists on the internet — it is born simultaneously in two places from private + public halves.

2 · The Ordering Rule (the whole session in one sentence)

Before DH completes, no shared secret exists — everything sent is necessarily plaintext. The moment DH completes, every further message can be encrypted.

Therefore: identities must be sent after Diffie-Hellman. Send them before, and they cross the internet naked.

🤝 "First grow the secret. Then whisper your name inside it." — Señor
This rule is the entire aggressive-mode story. Aggressive mode is fast because it crams identities into the first messages — before DH protection exists. The speed and the leak are the same design choice. Exam questions love testing whether you know these are one fact, not two.

3 · IKEv2: Two Exchanges, Four Messages

IKEv2 packs the maximum into each round trip while obeying the ordering rule:

ExchangeMessagesContentsProtection
IKE_SA_INIT2 (request + response)Proposal menu + DH public halves + noncesPlaintext — but nothing sensitive travels here; public halves are safe by design
IKE_AUTH2 (request + response)Identities + authentication + first child SAEncrypted — the secret already exists

No separate quick mode. The first child SA (phase 2 equivalent) piggybacks inside IKE_AUTH — that's where the second half of the savings comes from. Later child SAs and rekeys use CREATE_CHILD_SA, and housekeeping uses INFORMATIONAL exchanges, but establishment is complete at four messages.

The false dilemma: the blog offered "fast (aggressive) or safe (main mode)." IKEv2 is faster than aggressive mode and safer than main mode. When someone offers you a trade-off — first check whether the trade-off is real. Refuse the question.

The IKEv2 modern toolbox

  • Encryption: AES-GCM (authenticated encryption) vs IKEv1's DES/3DES/AES-CBC.
  • DH groups: 14/15/16 (large MODP) plus 19/20 (elliptic curve) — ECP gives MODP-equivalent security with shorter keys and less CPU. (PFS in phase 2 = running fresh DH per session.)
  • Authentication: PSK, digital signatures, and EAP — plug in external authentication services.
  • Integrity: HMAC-SHA2, AES-XCBC vs IKEv1's MD5/SHA1.
  • Mode config: integrated directly into the exchange — fewer messages, less CPU/memory.

5 · network-id: Two Doors, One Address

LoFinance's design has two tunnels between the same two gateways (dual-ISP aggregate). When a handshake arrives at HQ, source and destination IPs are identical for both tunnels — the addresses cannot disambiguate. The tag must live inside the negotiation.

  • network-id is a Fortinet-proprietary phase 1 attribute.
  • It selects the correct phase 1 between peers, so multiple IKEv2 tunnels can be established between the same local/remote gateway pair.
  • IKEv2 only. IKEv1 has no equivalent (scoreboard row: N/A · N/A · Yes).
  • The value must match at both ends of each tunnel.
🎟️ The nightclub with two identical doors: same club, same street address — the ticket disambiguates because the event is printed on it. network-id is the event on the ticket.
config vpn ipsec phase1-interface
    edit "ParraTunnel-ISP1"
        set ike-version 2
        set network-overlay enable
        set network-id 1
    next
    edit "ParraTunnel-ISP2"
        set ike-version 2
        set network-overlay enable
        set network-id 2
    next
end
Two commands, not one: network-overlay enable switches the feature on; network-id N sets the tag. Forgetting the first is a classic lab error.

Configuration Decisions This Session

  • LoFinance standardises on IKEv2 for all tunnels.
  • HQ↔Parramatta tunnels annotated: network-id 1 (ISP1 path), network-id 2 (ISP2 path) — enabling both phase 1s between the same gateway pair for the S10 aggregate.
  • Tunnels remain designed, not built — the build is Session 12.

Common Mistakes

  • Believing aggressive mode's speed and its ID leak are separate issues — they are one design choice.
  • Reversing before/after in the ordering rule. Identities go after DH. Say it out loud.
  • Counting IKEv2 as "2 messages" (forgetting IKE_AUTH) or "6" (inventing a quick mode it doesn't have).
  • Expecting network-id to work on IKEv1, or on a third-party peer — it's Fortinet-proprietary, IKEv2-only.
  • Setting network-id without network-overlay enable, or mismatching values between ends.

Senior Engineering Insights

  • Judge a protocol by its job list and ordering, not its message count. The count is a consequence of the design.
  • Refuse false trade-offs. "Fast or safe?" was answerable with "both" — the question itself was the error.
  • Date-check sources. A 2014 VPN security blog is archaeology, not research.
  • Derive, then label. If you can rebuild IKEv2 from "DH before identity, pack the envelopes," you can never forget it in an exam hall.

Exam Tips

  • Message counts are pure marks: aggressive 3+3=6, main 6+3=9, IKEv2 2+2=4 (IKE_SA_INIT + IKE_AUTH).
  • Peer-ID exposure discriminator: only aggressive mode sends IDs in clear text. Main mode and IKEv2 both encrypt them.
  • network-id keywords to pattern-match: "multiple tunnels, same gateway pair" → network-id; "Fortinet-proprietary" → network-id; "IKEv1" + network-id → false.
  • Distractor alert: "aggressive mode skips DH" — false; it reorders, it doesn't skip. "PSK sent in clear text" — false; it's the IDs that are exposed.
  • ECP groups (19, 20) = elliptic curve = same strength as big MODP with less CPU. PFS = fresh DH per phase 2 session.

Key Takeaways

① A handshake has three jobs: agree the language, grow the secret, prove identity — in that order.

② Aggressive mode is fast because it breaks the order; the leak is the speed.

③ IKEv2 = four messages, two exchanges, IDs encrypted, child SA piggybacked — faster than aggressive AND safer than main mode.

④ network-id (Fortinet-proprietary, IKEv2-only) lets multiple tunnels coexist between the same gateway pair — the ticket for the two-door nightclub.
