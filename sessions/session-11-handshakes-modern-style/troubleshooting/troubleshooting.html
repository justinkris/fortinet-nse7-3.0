🔧 Troubleshooting: When the Handshake Fails

Session 11 · IKEv2 negotiation, version mismatches, and the two-door problem. Think first, type second.

The Symptom Family

Riley's ticket will say: "VPN is down" or "branch can't reach the trading feed." Your first job is translation: a tunnel that never negotiates is a handshake problem; a tunnel that's up but passes no traffic is a phase 2 / routing / policy problem (Session 12 territory). This guide covers the first family — the negotiation never completes, or the wrong tunnel answers.

Questions Before Commands

  • Has this tunnel ever worked? Never-worked = config mismatch. Worked-then-stopped = something changed (rekey, ISP, certificate, firmware).
  • Which side initiates? The initiator's logs show what was proposed; the responder's show why it was rejected.
  • Same gateway pair, more than one tunnel? If yes — network-id is immediately on the suspect list.
  • Same IKE version both ends? An IKEv2 initiator and an IKEv1-only responder will never agree.
  • Can the peers even reach each other? UDP 500 (and 4500 behind NAT) must flow before any of this matters. Don't debug crypto when the packet never arrived.

Verification Commands

# Where did the negotiation get to?
diagnose vpn ike gateway list

# Filter the noise to one peer, then watch the exchanges live
diagnose vpn ike log-filter dst-addr4 <peer-ip>
diagnose debug application ike -1
diagnose debug enable
#   ...trigger the tunnel, read, then ALWAYS:
diagnose debug disable
What healthy looks like (IKEv2): you should see exactly the structure you derived on the whiteboard — an IKE_SA_INIT request/response pair (proposals + DH), then an IKE_AUTH pair, then the gateway listed as established with a child SA. Four messages. If the debug shows SA_INIT completing but AUTH failing, the language and secret were fine — the introductions failed (authentication/ID mismatch). If SA_INIT itself fails, suspect proposals, DH group, or version.

Decision Tree

1 · Restate — "Parramatta tunnel 1 never establishes" beats "VPN broken."
2 · Scope — one tunnel or both? One = tunnel-specific config. Both = shared cause (ISP, IKE version, PSK, upstream filter).
3 · Expected path — IKE packets: branch WAN → internet → HQ WAN, UDP 500/4500. Verify reachability before crypto.
4 · Find the failing exchange — ike gateway list + debug: did SA_INIT complete? Did AUTH?
5 · One hypothesis at a time — match the failing exchange to a cause (table below). Change one thing. Re-test.
6 · Verify recovery AND root cause — gateway established, child SA present, and you can say why it failed.

Failing Exchange → Likely Cause

ObservationLikely root cause
No response to SA_INIT at allReachability: UDP 500 blocked, wrong peer IP, ISP filtering — not a crypto problem yet
SA_INIT rejected / no proposal chosenProposal mismatch (encryption/integrity/DH group) or IKE version mismatch between ends
SA_INIT completes, IKE_AUTH failsAuthentication: PSK mismatch, wrong/failed certificate, peer-ID mismatch
Handshake completes but the wrong tunnel comes up (two tunnels, same gateway pair)network-id mismatch or missing — responder can't select the correct phase 1; check network-overlay enable + matching network-id on BOTH ends of EACH tunnel
Second tunnel between same peers won't establish while first is upSame as above — without network-id, IKEv2 peers can't distinguish two phase 1s between one gateway pair

Common Dead Ends

  • Debugging crypto before reachability. If UDP 500 never arrives, no proposal on earth will help.
  • Changing the proposal when AUTH is the failing stage. Read which exchange failed first — SA_INIT success means the language and secret were fine.
  • Fixing network-id on one end only. It must match per-tunnel at both ends.
  • Leaving debug running. diagnose debug disable — always.
  • "The blog said switch to aggressive mode." Downgrading the protocol is not a fix; it's a different, worse problem.

Exam Traps

• A scenario with "multiple IKEv2 tunnels between the same gateways failing to establish" is a network-id question wearing a costume.
• "IKEv1 tunnel configured with network-id" — invalid, IKEv2-only, Fortinet-proprietary.
• Given debug output: SA_INIT ok + AUTH fail = authentication/PSK/ID problem, not a DH-group problem.
• Count check: an IKEv2 debug showing quick mode = something is wrong with the question (or it's IKEv1).
