SESSION 39 · TROUBLESHOOTING FRAMEWORK

Rating Day — Thinking Like the Referee

How to triage a Security Rating report the way a senior engineer would — not by re-reading the severity column, louder.

Symptom

"The security rating came back with a C+ and 12 findings. Two stakeholders want opposite things done with it, and a board deadline is 3 days out."

This isn't a technical failure to diagnose — it's a triage and prioritisation exercise, and the discipline transfers directly to real vulnerability-management workflows.

Questions to Ask First

  • What does this artefact actually contain — a score, a grade, findings, or all three stapled together?
  • Who needs which part, and why? (A board doesn't need device names; a security lead doesn't want a bare grade.)
  • Is this score a one-time snapshot or something that re-runs? (It re-runs — on schedule, or on demand.)
  • For each finding: who can exploit it, with what precondition, and what do they get?
  • Which findings are safe to fix blind, and which need evidence first?

Verification Commands

diagnose autoupdate versions # Confirms FortiGuard Security Rating subscription/contract state Security Fabric > Security Rating > [Run Now / Re-evaluate] # On-demand re-run — don't wait on the schedule before a deadline Configuration > Revisions # Confirms scheduled config-backup stitch actually executed diagnose endpoint fctems queue-complete-calls Q-<endpoint_ip> # Manual endpoint quarantine — bypasses the automated FAZ→EMS chain

Expected output: diagnose autoupdate versions should show current contract dates for the rating-relevant service; a lapsed/expired contract will show as an expired date, which correlates directly with a degraded/missing percentile comparison on the rating page.

Decision Tree — Triaging a Findings List

1
Is it internet-facing and precondition-free? (e.g. admin access on WAN) → Fix immediately, regardless of listed severity tier.
2
Is it a zero-disruption config change? (trusted hosts, password policy) → Fix same hour. No user impact, no rollback plan needed.
3
Does the fix require a reboot or service interruption? (firmware upgrade) → Schedule a change window; check business-hours impact for the affected device/site first.
4
Does the finding involve a policy or rule whose real-world usage you don't know? (broad "all/all" policies) → STOP. Pull FortiAnalyzer traffic logs for that policy ID first. Use a window wide enough to catch periodic/rare legitimate flows before tightening.
5
After fixes land → trigger an on-demand rating re-run rather than waiting for the schedule, so verification and reporting aren't blocked on a timer.

Common Dead Ends

Sorting purely by severity column and stopping there. Misses that a "medium" finding might be free to fix now while a "high" finding needs a change window — severity and urgency aren't the same axis.
Assuming the score is a fixed snapshot. Leads to unnecessary waiting for a scheduled re-run when an on-demand trigger exists.
Treating "all/all" policy findings the same as license/config findings. These need evidence (FAZ), not a direct edit — editing blind risks a delayed outage on a rare but legitimate flow.

Likely Root Causes Behind a Poor Rating

  • Admin management interfaces left reachable from WAN (most common critical finding).
  • No trusted-hosts restriction on admin accounts (cheap to fix, often overlooked).
  • Firmware drift on branch/remote devices that don't get routine maintenance windows.
  • Broad policies from early deployment that were never tightened once real traffic patterns were known.

Exam Traps

Trap: "Now" as a Schedule trigger frequency option. It isn't — the options are Once, Hourly, Daily, Weekly, Monthly.
Trap: Assuming the Security Rating checks run FROM the root AGAINST each leaf's live config over the network. They don't — each device self-audits locally; only results roll up.
Trap: Assuming FortiClient on all platforms supports remote quarantine. Linux is explicitly excluded.
Solid ground: the exposure/blast-radius/cost-to-fix triage lens is not a memorised Fortinet term — it's a transferable senior-engineering habit the exam rewards through scenario reasoning, not rote recall.
