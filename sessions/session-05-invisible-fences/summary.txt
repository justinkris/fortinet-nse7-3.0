TECHNICAL CONCEPTS COVERED
----------------------------
1. Policy `dstintf` matching happens AFTER FortiGate's routing lookup resolves the
   real logical egress interface — not the physical wire. A policy pointing at the
   physical trunk (internal1) will never match traffic actually egressing via a
   VLAN sub-interface (VLAN20-Trading), because FortiOS treats these as entirely
   separate interface objects for policy purposes, even though they're the same
   copper. Student derived the full mechanism unprompted, connecting last session's
   VLAN-interface-object concept directly to policy matching for the first time.
2. WHY VLAN sub-interfaces must be separate objects at all: without separate
   objects, a policy on the shared physical trunk couldn't distinguish "traffic to
   VLAN 5" from "traffic to VLAN 20" — segmentation requires separate policy
   surfaces, not just separate tags on the wire. Direct retrieval + extension of
   Session 4 material.
3. Hardware switch (`config system virtual-switch`): ASIC/hardware-offloaded
   packet processing, supports STP, does NOT support wireless SSID members, only
   available on specific FortiGate models.
4. Software switch (`config system switch-interface`): CPU-processed, does NOT
   support STP, DOES support wireless SSID members, available on every model.
   Student initially reversed this pairing (guessed hardware=SSID-capable,
   software=STP-capable) — corrected via reasoning about where each function
   traditionally lives (STP in switch silicon; SSID as a FortiGate software
   construct), then applied it correctly and unprompted on the final exam question.
5. `intra-switch-policy implicit` vs `explicit`: implicit permits traffic between
   members of the same switch bundle without ever routing it through the firewall
   policy engine (same logic as same-VLAN traffic never touching FortiGate at all —
   student correctly reapplied this pattern from earlier in the session). Explicit
   forces same-bundle, same-broadcast-domain traffic through an actual firewall
   policy anyway — FortiGate manufacturing a policy checkpoint inside what would
   otherwise be trusted-by-default Layer 2 space. Student initially conflated this
   with inter-VLAN segmentation (assumed "different Layer 2 domain" was involved)
   before correctly landing on the true use case: segmenting devices that share the
   SAME broadcast domain (e.g. guest laptop vs finance servers on one bundle).
6. Diagnostic split reinforced from Session 4: `diagnose sys session list` absence
   of ANY session for a given traffic flow means the traffic never reached FortiGate
   at all (Layer 2 / same-broadcast-domain / never-crossed-a-router problem) — not a
   policy logic problem. Student correctly reasoned through this for BOTH bugs in
   this ticket independently before being told.
7. `show system switch-interface` / `show system virtual-switch` as the direct way
   to confirm which switch construct is actually configured, rather than guessing
   from context — command names themselves disambiguate hardware vs software.
8. The core meta-lesson of the session: a symptom that sounds contradictory
   ("nobody can reach them, yet they keep printing") is often TWO independent,
   unrelated mechanisms explaining two different halves of one ticket — not one
   confusing bug. Señor's "two questions wearing one ticket" line names this
   directly; student solved both halves separately without being told they were
   separate until after the fact.

TROUBLESHOOTING APPROACH TAUGHT
----------------------------------
When a ticket presents what looks like one contradictory symptom (works AND doesn't
work simultaneously), consider whether it's actually describing two separate traffic
paths with two separate root causes before trying to reconcile them into one
explanation. For each path independently: check `diagnose sys session list` first —
absence means Layer 2 / no-router-crossing, not policy; presence with a policy ID
means check policy logic (dstintf/srcintf resolved AFTER routing, not before).
