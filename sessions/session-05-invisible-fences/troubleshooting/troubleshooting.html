SESSION 5 · TROUBLESHOOTING GUIDE

Diagnosing "Reachable Locally, Unreachable Remotely"

How a senior engineer walks through a symptom that sounds like a contradiction.

SYMPTOMS

What You'll See on the Ticket

  • A device (printer, server, endpoint) is unreachable from other subnets/VLANs.
  • The same device is clearly functioning — it's actively serving local traffic.
  • The reporter frames this as one confusing, contradictory bug.
QUESTIONS TO ASK FIRST

Split the Symptom Before Diagnosing It

  • Is "it works" and "it doesn't work" describing the same traffic path, or two different ones (local vs. remote)?
  • Is the working traffic staying inside one broadcast domain the whole time?
  • Is the failing traffic crossing a VLAN / subnet boundary?
Senior instinct: assume two independent mechanisms until proven otherwise. Don't try to build one unified theory that explains both symptoms — it usually doesn't exist.

DECISION TREE

Working Through the Ticket

  1. Confirm the "working" traffic and "failing" traffic are genuinely two separate paths (different source, different boundary crossed).
  2. For the failing path: run diagnose sys session list. No session → Layer 2 issue (VLAN tag, trunk, ARP). Session with policy_id → check dstintf/srcintf resolution.
  3. For the working path: if it never shows in the session list either, check whether it's staying inside one broadcast domain (same VLAN, or same switch bundle with intra-switch-policy implicit). If so, that's expected behavior, not a bug.
  4. Fix only the actual root cause. Don't touch the working path's config just because it also doesn't appear in the session list — that absence can be entirely normal.
DEAD ENDS & EXAM TRAPS

Where This Goes Wrong

  • Dead end: assuming both symptoms share one cause and hunting for a single unifying bug. Usually wastes the most time.
  • Dead end: "fixing" a software switch's intra-switch-policy when it was never the problem — implicit permit for same-bundle traffic is often working as designed.
  • Exam trap: assuming a policy referencing the physical trunk will also cover VLAN traffic on that trunk. It won't — separate interface objects.
  • Exam trap: reversing hardware/software switch capabilities (STP vs. SSID support). Anchor to where each function lives physically, not rote memorization.
  • Exam trap: assuming `intra-switch-policy explicit` requires a Layer 3 boundary. It doesn't — it operates entirely within one broadcast domain.
LIKELY ROOT CAUSES — RANKED

If You Only Have Time to Check Two Things

  1. Policy dstintf/srcintf pointing at the physical interface instead of the VLAN sub-interface.
  2. Missing or misordered policy rule (wrong match before the correct one, first-match-wins).
