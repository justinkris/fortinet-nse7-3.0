Invisible Fences

Two symptoms, one ticket, zero relationship — how policy interface matching and software-switch behavior can each break a different half of "reachability" without either one being the other's fault.

A ticket that sounds contradictory is often two tickets wearing a trenchcoat.

Riley's printers were unreachable from outside the floor, yet printing fine for traders on it. Both things were true. Neither explained the other.

Interface Matching Hardware vs Software Switch Intra-Switch Policy
💭 THINK FIRST · INTERFACE MATCHING

Junior's policy exists, points at an interface, and still never fires. Before reading on: what do you think FortiGate is actually comparing when it decides whether a policy's dstintf matches a packet — the physical wire the packet arrives on, or something else?

  • If a policy says dstintf internal1 but the packet's real destination is on tagged VLAN 20, does the policy fire?
  • Why can't FortiGate just let policies reference the physical trunk directly and sort out the VLAN some other way?
Q1 — MODEL ANSWER
No. FortiGate resolves the routing decision first — which logical VLAN interface the destination actually lives on — and only then checks policies against that resolved interface. A policy pointing at the physical trunk never matches, because VLAN20-Trading and internal1 are separate interface objects to the policy engine, even on the same copper.
Q2 — MODEL ANSWER
Because one physical trunk can carry several VLANs at once. If policies could only reference the trunk, a single policy would match traffic to every VLAN on that trunk simultaneously — there'd be no way to allow VLAN 5 traffic while denying VLAN 20 traffic on the same wire. Separate interface objects are what give the policy engine separate, independently-controllable surfaces.
SECTION 01

Policy Matching Happens After Routing

Junior configured an allow policy from the print server subnet to the trading printers. On paper it looked correct. In practice, it never fired — the printers stayed unreachable from anywhere off the floor.

The bug: the policy's dstintf was set to internal1, the physical trunk carrying both VLAN 5 and VLAN 20. But FortiGate doesn't check policies against the physical wire a packet is destined for — it first performs its routing lookup, resolves the actual logical egress interface (VLAN20-Trading), and only then searches for a policy whose dstintf matches that resolved interface. Junior's policy, watching internal1, was watching a door the traffic never actually walks through as a named object — it falls through to implicit deny instead.

Common misconception: "The VLAN interface and the physical trunk are basically the same thing, so it shouldn't matter which one the policy references." They share a wire, but they are two entirely separate objects to the policy engine — this is exactly why VLAN sub-interfaces exist as their own construct (see Session 4).
FortiOS CLI — the fix
config firewall policy
    edit 12
        set srcintf "dc-servers"
        set dstintf "VLAN20-Trading"   # was: "internal1"
        set srcaddr "print-server"
        set dstaddr "VLAN20-printers"
        set action accept
        set schedule "always"
        set service "PRINT"
        set logtraffic all
    next
end
🧠
MENTAL NOTE
A policy's dstintf must match the interface FortiGate's routing lookup actually resolves to — never the physical wire underneath it. On a trunk, that means the VLAN sub-interface, every time.

💭 THINK FIRST · HARDWARE VS SOFTWARE SWITCH

FortiGate can merge several physical ports into one shared broadcast domain two different ways. Before reading the comparison: which one do you think would support a wireless SSID as a member, and which would support STP — and why might those two capabilities NOT travel together?

  • STP is a loop-prevention protocol that traditionally lives in dedicated switch silicon. Which switch type does that suggest?
  • An SSID isn't a physical port at all — it's a FortiGate software construct. Which switch type does that suggest?
Q1 — MODEL ANSWER
Hardware switch — STP support is tied to the ASIC-offloaded switching chips, the same silicon that historically implements it on real switches.
Q2 — MODEL ANSWER
Software switch — since it's CPU-processed rather than ASIC-bound, it can absorb a software construct like an SSID interface as a member; the hardware switch, tied to physical ASIC ports, cannot.
SECTION 02

Hardware Switch vs. Software Switch

Riley's floor-3 print stations weren't on a VLAN trunk at all — three plain physical ports, wired years ago into a FortiGate construct that merges multiple ports into one shared broadcast domain with a single gateway IP, emulating an unmanaged switch without buying one.

There are two flavours of this on FortiGate, and the exam loves reversing them:

FeatureHardware SwitchSoftware Switch
ProcessingOffloaded to ASICCPU (system software)

Confirming which type is actually configured is a one-line lookup, not a guess:

FortiOS CLI — confirm switch type
show system switch-interface
show system virtual-switch
Common misconception: pairing STP with software switches or SSIDs with hardware switches. Anchor it instead: STP lives in switching silicon (hardware), SSIDs are a FortiGate software construct (software) — the pairing follows from where each function actually lives.
🧠
MENTAL NOTE
Hardware switch = ASIC, STP, model-limited, no SSIDs. Software switch = CPU, no STP, all models, SSID-capable. Confirm with `show system switch-interface` / `show system virtual-switch` rather than guessing.

💭 THINK FIRST · INTRA-SWITCH POLICY

Three devices sit on the same software switch bundle — same broadcast domain, no VLAN boundary between them. One is a guest laptop; two are finance servers. Before reading on: would `implicit` or `explicit` intra-switch-policy protect the servers, and what does the setting actually force FortiGate to do?

  • Does `explicit` require crossing into a different broadcast domain — or does it apply even within one?
  • If `explicit` is set but no firewall policy exists between the members, what happens to the traffic?
Q1 — MODEL ANSWER
Explicit applies even within a single broadcast domain — it doesn't require a Layer 3 boundary at all. That's the whole point: it manufactures a policy checkpoint inside space that would otherwise be trusted by default.
Q2 — MODEL ANSWER
Dropped. Explicit removes the automatic Layer 2 trust; without a matching policy, traffic between members falls to implicit deny, the same as any other unmatched traffic.
SECTION 03

Implicit vs. Explicit Intra-Switch Traffic

Riley's "PrintGroup" software switch bundles three printers on ports 5-7 with intra-switch-policy implicit — meaning any traffic between those printers (or any device added to that bundle) is permitted automatically, without ever routing through FortiGate's policy engine at all. That's why local printing worked the entire time the ticket was open: it's the same underlying pattern as same-VLAN traffic never crossing FortiGate — no Layer 3 boundary, no policy check, no session log entry.

This was not a bug. implicit is working exactly as designed. It just happened to be an entirely separate mechanism from the actual root cause (the misconfigured policy dstintf covered in Section 1) — two independent explanations for two halves of one ticket.

Exam tip: the existence of `explicit` is the real teaching point here. It exists specifically for cases where devices sharing one broadcast domain still shouldn't implicitly trust each other — e.g. a guest device and a finance server sharing one software switch bundle. Setting `intra-switch-policy explicit` forces same-bundle traffic through a real firewall policy, manufacturing a checkpoint where Layer 2 would otherwise grant free passage.
FortiOS CLI — Riley's config, as found
config system switch-interface
    edit "PrintGroup"
        set vdom "root"
        set member "port5" "port6" "port7"
        set intra-switch-policy implicit
    next
end
🧠
MENTAL NOTE
Implicit = same-bundle traffic never touches a policy. Explicit = FortiGate forces same-bundle traffic through a real policy anyway. Explicit doesn't require crossing a Layer 3 boundary — it exists precisely for devices that share ONE broadcast domain but still shouldn't trust each other by default.
