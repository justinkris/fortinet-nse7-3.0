Three Filters, One Job?

Web filter vs app control vs ISDB — refresh before work or the exam.

The Envelopes

Web filter = the address on the front. App control = the sender's handwriting. ISDB = postal-district lookup — the envelope is never opened.

Five Facts That Win Marks

  • App control is flow-only because it runs on the IPS engine. Proxy buffering (the shoebox) helps AV, which needs the whole file — handwriting is visible in the first few lines.
  • ISDB knows at packet one (IP + port in the header) — that's why it's native to SD-WAN rules. App control needs several packets for a verdict.
  • App-based SD-WAN steering exists but requires Application Detection-Based SD-WAN → System > Feature Visibility, and it learns (first sessions take the default path).
  • diagnose autoupdate versions shows Application Definitions + Internet-service Full Database. The web filter DB is never there — FortiGate queries FortiGuard/FMG-as-FDS live, caching recent ratings.
  • Blocked website + working app = coverage gap, not malfunction. Apps scatter across API/CDN hostnames (graph.facebook.com, fbcdn.net) — no rated URL ever appears.

Policy Design in One Breath

Profiles decide WHAT. Policies decide WHO. "No social media, but marketing needs Facebook" = two policies: Marketing on top (Facebook allowed via app-control override), everyone else below carrying both web filter (Social Networking → block) and app control (Social.Media → block). Never one-tool-per-group; never a hole in a shared profile.

Trap Check

"App control needs proxy mode" — false. "ISDB is a security profile" — false. "Web filter category blocks the app too" — false: front door locked, loading dock open.
