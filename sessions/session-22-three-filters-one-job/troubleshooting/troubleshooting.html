SESSION 22 · TROUBLESHOOTING · SENIOR THINKING

"It's Blocked But Still Works"

Diagnosing filter coverage: which question does the deployed tool actually answer?

The Symptom Family

  • "Facebook is blocked but the Facebook app still works." (Ticket #41)
  • "We block streaming, but users still stream."
  • "SD-WAN isn't steering Office 365 down the right link."
  • "Filtering broke and nobody changed the config." (FortiGuard dependency — S23 preview)
The senior reflex: before touching anything, ask which question does the deployed tool answer, and which question is the symptom asking? A working tool with the wrong coverage looks identical to a broken tool — until you check the wire.

Questions to Ask First

  1. Scope: Who is affected — one user, one group, one policy? Browser, app, or both?
  2. What exactly is blocked: Is the block page appearing (web filter verdict delivered) or is traffic silently passing?
  3. Which profiles are on the matching policy — and is the traffic even matching the policy you think it is?
  4. What does the traffic look like on the wire — which SNIs/destinations is the "blocked" application actually contacting?

Verification Commands & Expected Outputs

# Which policy is the session actually hitting? diagnose sys session list | grep -A 10 <client-ip> # → check policy_id against your intended policy # What is the app really connecting to? (SNI evidence) diagnose sniffer packet any "host <client-ip> and port 443" 4 # → collect ClientHello destinations: graph.facebook.com, *.fbcdn.net ... # Are the local databases present and current? diagnose autoupdate versions # → Application Definitions: version + "Updates Installed" # → Internet-service Full Database: version + contract valid # → NOTE: no web filter database will appear — that is normal # Web filter rating path (on-demand to FortiGuard/FDS) diagnose debug rating # → shows FortiGuard rating servers, weights, round-trip times

Decision Tree

"X is blocked but still works" │ ▼ Is the BROWSER blocked (block page shown)? ├─ NO → web filter not applying at all │ → wrong policy match? profile attached? SSL inspection │ giving visibility (cert vs full)? FortiGuard reachable? │ └─ YES → web filter is WORKING. Coverage question: │ ▼ Is the surviving traffic an APP (non-browser)? ├─ YES → sniff SNIs. Scattered API/CDN hostnames, │ no rated URL → web filter cannot see it. │ FIX: add app control (category or signature) │ to the SAME policy. Verify block in app logs. │ └─ NO (browser survives on some sites) → category rating of that domain? Check the rating cache / FortiGuard lookup. Override? Exemption? Different policy matched?

Common Dead Ends

Dead EndWhy It Wastes Your Morning
"The web filter profile is corrupted"The profile is fine — it answers a different question. Evidence (SNI capture) beats re-creating profiles.
Blocking fbcdn.net wholesaleShared CDN infrastructure — you'll break unrelated services. Identify the application, not its hostnames.
Punching an exception into the shared profileThe hole applies to everyone the profile touches. Exceptions live in policy structure (who), not profile edits (what).
One tool per user groupEither/or thinking recreates the gap. Full coverage = both profiles per policy; only verdicts differ.
Expecting app-based SD-WAN steering by defaultRequires Application Detection-Based SD-WAN (System > Feature Visibility) and it learns — first sessions take the default path. ISDB steers at packet one.

Likely Root Causes (Ranked)

  1. Coverage gap: web filter deployed alone; app traffic invisible to URL matching.
  2. Policy mis-match: traffic hitting a different policy (order, source group, interface) than the one carrying the profiles.
  3. FortiGuard dependency: app-control/ISDB DB stale or contract expired (diagnose autoupdate versions); web filter rating service unreachable and cache cold.
  4. Visibility gap: certificate-only inspection where content matching needed full inspection.

Exam Traps

1. A scenario where "the web filter is malfunctioning" is a distractor — the correct answer identifies the coverage gap and adds app control.
2. diagnose autoupdate versions offered as a way to check the web filter database — invalid; that DB is remote.
3. "Enable app control in proxy mode for deeper detection" — impossible; flow-only (IPS engine).
4. ISDB presented as a security profile in an answer option — it's an object used as a policy destination.
Close the loop like a senior: after adding app control, verify with the application-control logs that the app signature is matching and the verdict is Block — then confirm the marketing policy still permits Facebook. Recovery is not verified until both the block and the exception are proven.
