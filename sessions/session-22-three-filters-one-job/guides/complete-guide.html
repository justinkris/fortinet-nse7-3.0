Concept 1

Web Filter — the address on the front

🤔Think First
A web filter blocking facebook.com works perfectly, yet the Facebook app on the same policy works fine. Before reading on: what is the app connecting to that the browser wasn't?
  • What field does the web filter read in a TLS handshake without decrypting?
  • Does an app's traffic contain a URL a human typed?

The web filter manages website access by inspecting content, URLs, domains, SNI, or SANs, in either flow-based or proxy-based inspection. FortiGuard rates the destination against categories (e.g. Social Networking) and the profile applies the verdict.

Why apps evade it: a browser session is one address on one envelope. An app is machines talking to machines, scattering across API endpoints, messaging brokers, and CDNs:

Packet capture — Facebook app, trading-floor Wi-Fi
TLS ClientHello  SNI: graph.facebook.com        (API calls)
TLS ClientHello  SNI: edge-mqtt.facebook.com    (notifications)
TLS ClientHello  SNI: scontent.xx.fbcdn.net     (CDN — images/video)
TLS ClientHello  SNI: b-api.facebook.com        (telemetry)
# www.facebook.com — never appears
The tool isn't broken. A tool "not working" is often a tool answering a different question. The web filter answers "which website is this?" App traffic answers "no website at all — I'm software."
Concept 2

Application Control — the handwriting

Application control identifies applications through their transmission patterns, using application signatures, protocol decoders, and rate-based IPS signatures. It has its own FortiGuard categories (e.g. Social.Media), so "block social media the software" is an app-control job, regardless of which of fifty hostnames the app hides behind.

Why flow-based only? Because application control runs on the IPS engine, which is flow-based. Proxy mode buffers and reassembles entire content (the "shoebox") — useful when the verdict needs the whole object, as antivirus does. But handwriting is visible in the first few lines: app identification comes from patterns in transit. Buffering the whole session would add latency and memory and buy nothing.

⚙️
Engine Fact — Exam Gold
App control = the IPS machinery pointed at a different question. Flow-only is structural, not a missing feature.

Extra roles: traffic shaping policies, and SD-WAN steering — but application-based SD-WAN requires enabling Application Detection-Based SD-WAN under System > Feature Visibility, and it works by learning: early sessions take the default path until the app is identified, then later sessions are steered.

Concept 3

ISDB — the postal district

The Internet Service Database is a FortiGuard-maintained catalogue of IP addresses (L3) and TCP/UDP ports (L4) per vendor/service — Microsoft-Office365, Netflix, AWS. It performs no inspection whatsoever. It is a Policy & Object, not a security profile: you select an Internet Service as the destination of a policy, in place of an address object.

Lookup vs verdict — the timing argument. An SD-WAN rule must pick an egress link at session start. ISDB knows the answer at packet one — destination IP + port are in the first header. App control needs the handshake and several packets before its verdict exists. That's why ISDB is the native SD-WAN citizen and app-based steering is an opt-in workaround.
Concept 4

FortiGuard — what Penny is renting

All three features malfunction without FortiGuard updates — but they consume FortiGuard differently:

  • App control & ISDB: databases downloaded and stored locally. Verify with diagnose autoupdate versions — look for Application Definitions and Internet-service Full Database.
  • Web filter: the URL universe is too vast to store on the box. FortiGate queries FortiGuard (or FortiManager acting as a local FDS) for rating lookups on-demand, keeping only a cache of recent verdicts.

Policy Design

Profiles decide WHAT. Policies decide WHO.

Penny's requirement — "no social media, but marketing needs Facebook" — is solved in policy structure, never by punching holes in a shared profile (Junior's idea; a profile hole applies to everyone the profile touches).

Policy (top-down)SourceWeb FilterApp Control
1 — MarketingMarketing groupFacebook permittedFacebook signature override → Allow
2 — Trading floor / allEveryone elseSocial Networking → BlockSocial.Media category → Block
Avoid the either/or trap: web filter and app control are not alternatives assigned to different groups. They answer different questions, so a full-coverage policy carries both — only the verdicts change per policy. Web filter alone = front door locked, loading dock open (Ticket #41 by construction).
Exam Preparation

Exam Tips & Traps

  • "App control requires proxy mode" — always false. Flow-only, IPS engine.
  • "ISDB inspects traffic" — false. L3/L4 lookup, no inspection; it's an object, not a profile.
  • SD-WAN steering by application — possible, but only after enabling Application Detection-Based SD-WAN under System > Feature Visibility. Default SD-WAN service matching = ISDB.
  • diagnose autoupdate versions shows app control + ISDB versions. It never shows a web filter database — that one is remote + cached.
  • "Web filter blocks the app too" — no. Domains/URLs vs software. Expect a scenario question shaped exactly like Ticket #41.
  • Read True/False stems twice; decide what YES means before answering.
