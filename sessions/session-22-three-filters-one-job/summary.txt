TECHNICAL CONCEPTS COVERED
--------------------------
1. WEB FILTER: matches domains/URLs (SNI/SAN/CN, URL, content). Flow OR proxy.
   Security profile. CLI: config webfilter profile. Answers "WHERE are you going?"
2. APP CONTROL: L7 signatures, protocol decoders, rate-based IPS signatures. FLOW
   ONLY — because it RUNS ON THE IPS ENGINE (this resolved the student's pre-reading
   question; his hypothesis was inverted but reasoning sound). Security profile.
   CLI: config application list. Answers "WHAT are you?" Has its own FortiGuard
   categories (Social.Media etc.). Also used for traffic shaping.
3. ISDB: database of IP+port per vendor/service. L3/L4 lookup, NO inspection.
   NOT a profile — a Policy & Object; used AS THE DESTINATION in policies and as
   the native match in SD-WAN rules. CLI: config firewall internet-service-name.
   Knows the answer at PACKET ONE (lookup vs verdict).
4. Why apps evade web filters: apps are machine-to-machine, scattering across API
   endpoints, MQTT brokers, CDNs (fbcdn.net) — the "address on the front" is
   fragmented/absent. Web filter isn't broken; it answers a different question.
5. App-based SD-WAN steering exists but requires Application Detection-Based SD-WAN
   under System > Feature Visibility, and works by learning (early sessions take the
   default path until identified). ISDB needs no such trick.
6. Proxy vs flow framing: proxy = shoebox (buffer + reassemble the whole letter →
   AV's home); flow = watch lines being written (app control, IPS). Buffering buys
   app control nothing — handwriting is visible in the first few lines.
7. FortiGuard consumption models: app control + ISDB databases stored LOCALLY
   (diagnose autoupdate versions → "Application Definitions" / "Internet-service
   Full Database"); web filter DB is NOT local — live on-demand rating queries to
   FortiGuard or FortiManager-as-FDS, with only a cache of recent verdicts.
8. hping3 introduced via exam Q9 only (p. 175): -S SYN, -p port, -c count.
