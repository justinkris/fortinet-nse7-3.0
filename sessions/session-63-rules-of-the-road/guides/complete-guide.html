1. The member that never carried a packet

Before touching J.R.'s rule, Señor checked whether Friday was even the crime scene. port3 had been up all weekend and carried nothing — with the rule deleted. The rule was never the reason.

# The evidence
FGT-SYD-EDGE # diagnose sys sdwan health-check
Seq(3 port3): state(dead), packet-loss(100.000%) latency(0.000), jitter(0.000)

FGT-SYD-EDGE # get router info routing-table all
C       192.0.2.0/24 is directly connected, port3   ← connected route ONLY. No member default.

The member configuration was missing one field:

config system sdwan
    config members
        edit 3
            set interface "port3"
            set zone "underlay"          ← optional
            set gateway 192.0.2.1        ← required. Never typed.
        next
    end
end

The gateway is the input FortiGate uses to auto-build the member's static route. Without it, the collapse is total:

No gateway → no member route → the health-check probe has no path out → 100% loss → state(dead) → never eligible for any rule, ever.
Diagnostic signature — read the shape of the failure: a genuinely degraded link returns real numbers, badly. A link with no path out returns zeroes across every metric. Zeroes are a configuration symptom wearing a carrier costume.
Rules do not create reachability. SD-WAN will not steer traffic to a member with no valid route to that destination. A rule only chooses among members that already have one.
"The health check is a thermometer. But a thermometer must be able to touch the patient. J.R. built a thermometer in another room."
— Señor

2. Rule lookup and matching criteria

SD-WAN rules (config service) are evaluated top-down by sequence number; the first match wins; nothing below is ever read. J.R.'s catch-all in seat one made every later rule dead configuration — syntactically valid, permanently unreachable. The fix is structural: specific first, general last, and delete catch-alls rather than demoting them.

Match onCLI
Source address / user / groupset src · set users · set groups
Destination addressset dst
Destination service (ISDB)set internet-service enable + set internet-service-name
Destination applicationset internet-service-app-ctrl
Protocol + port rangeset protocol, set start-port/end-port
DSCP / ToSset tos, set tos-mask
Incoming interfaceset input-device

Contrast: a static route matches destination only. This table is the entire reason SD-WAN rules exist.

The corrected design:

config system sdwan
    config service
        edit 1
            set name "Trading"
            set mode sla
            set src "TRADING-FLOOR"
            set dst "EXCHANGE-FEEDS"
            set health-check "Trading-SLA"
            config sla
                edit "Trading-SLA"
                    set id 1
            end
            set priority-members 1 2      ← port1 preferred, port2 escape hatch
        next
        edit 2
            set name "Streaming"
            set internet-service enable
            set internet-service-name "Google-YouTube"
            set priority-members 2        ← pinned to Baz
        next
    end
end

3. Hashing and the four traders

J.R.'s rule ran mode load-balance with set hash-mode source-ip-based (the default is round-robin). The result: each source IP hashes deterministically to one member — hash the field, modulo the member count, index the list. With long DHCP leases, the same four traders carried the same source IPs for weeks and hashed to Baz every day, all day.

Symptom shape is evidence. "Everyone is a bit slow, sometimes" → per-flow scattering → round-robin. "The same four people, always, everyone else perfect" → per-host pinning → source-ip-based (or source-dest-ip-based). The ticket pattern reveals the algorithm before you read a line of config.
Divisor warning: the modulo runs over the member count. Bringing port3 alive changed the divisor from 2 to 3 — every host rehashed, with no rule touched. Fixing one thing legitimately moved everybody.

Also learned: a probability across users is not a probability across attempts. "50% chance of the bad link" was true of the population and false of any individual — you are Jam, or you are not Jam.

4. The four strategies

GUICLIBehaviourUses SLA?
Manualset mode manualFirst member in priority-members with a route. Full stop.No — steers into a sick link happily
Best Qualityset mode priorityBest measured value of link-cost-factorMeasures, no threshold
Lowest Cost (SLA)set mode slaSLA-passing members eligible; lowest cost; ties by list orderYes — as a gate
Maximize Bandwidth (SLA)set mode load-balanceSpread across SLA-passing members via hash-modeYes — as a gate
The dividing line: sla and load-balance treat the SLA as an eligibility gate. priority treats measurements as a ranking. manual doesn't look.

Best Quality vs Lowest Cost (SLA) — the design argument

Both are dynamic; both consume the same health check. The difference is what they do with the measurement. Best Quality asks "which is best?" — a comparison with no concept of good enough. On a calm day it churns paths for a 1ms prize (NAT identity flapping, re-evaluation, asymmetry risk); on a terrible day it picks the least-bad link and reports success, unable to say both roads are unusable. Mode sla asks "is my preferred road still acceptable?" — stable by default, decisive on breach. Its honesty is exactly as good as its threshold: the threshold is where the business requirement enters the network. Damp the flapping with set hold-down-time 30 — quick to distrust, slow to forgive, one layer up.

"'Best available' and 'acceptable' are different claims."
— Señor
Exam-critical (this session's hardest miss): in mode sla, if every member is out of SLA, the traffic does not fall through to the next rule and is not dropped. Matching happens on src/dst/service — not on health — so the rule still matches, and FortiGate falls back to best effort: the configured priority-members order. A rule is skipped only when it has no usable members at all (all dead, or no route to the destination). Sick ≠ dead.

5. The implicit rule

Delete every SD-WAN rule and traffic still flows, because something un-erasable answers underneath: the implicit rule. It is not a hidden service entry — it is the global setting:

config system sdwan
    set load-balance-mode source-ip-based    ← this line IS the implicit rule's algorithm
end
  • Catches all traffic that matched no configured rule.
  • Load-balances across every member holding a valid route to the destination.
  • Consults no performance SLA — ever. No health check can be bound to it. Out-of-SLA members still have valid routes (S62: out-of-SLA does not remove the route), so the implicit rule steers into brownouts without hesitation.
Design consequence: anything you care about must be caught by an explicit, SLA-aware rule above the fall-through. The implicit rule is fine for traffic that tolerates a bad afternoon — and only for that.

6. Local-out traffic

Cipher's syslog gaps: traffic the FortiGate originates itself (syslog, FortiGuard, DNS, SNMP) never ingressed — no incoming interface, no client source address. Nothing on the matching menu exists for a rule to read, so local-out traffic bypasses SD-WAN rules by default and takes a plain routing-table lookup. Two equal defaults meant ECMP: half the audit trail on Baz.

The fix is per feature — there is no global switch:

config log syslogd setting
    set interface-select-method sdwan     ← auto | sdwan | specify
end
config system fortiguard
    set interface-select-method sdwan
end
config system dns
    set interface-select-method sdwan
end
config system snmp sysinfo
    set interface-select-method sdwan
end
"The cobbler's children have no shoes."
— Señor, on the firewall being the only device ignoring its own policy
Senior lesson: management-plane traffic is the traffic you need most when everything else is broken — and it silently opts out of your design unless you go and fetch it.

7. Application steering — ISDB vs app-ctrl

Matches onFirst packet?
internet-service (ISDB)Known IP / protocol / port for a service — a destination lookupYes
internet-service-app-ctrlDeep inspection — Client Hello / SNI / server certificateNo — first session escapes, then destinations are cached

Riley's "first video after lunch takes the wrong link, then everything behaves": app identification takes several packets, so the first session is steered before the app is known (falling to whatever catches it). Once identified, the app-to-destination mapping is cached — visible via diagnose sys sdwan internet-service-app-ctrl-list — and subsequent sessions match on packet one. The first video pays for everyone else's.

Dependency trap: app-ctrl steering requires the firewall policy to actually run application control. No app-control profile in the policy → no identification → the rule matches nothing, forever, while looking perfectly valid.

Design guidance: use ISDB where a service has stable, published destinations; use app-ctrl only where it doesn't, and accept the first-session leak as the price.

8. The ladder — five rungs

  1. Does a rule MATCH? (and is a catch-all above it eating the match?) — show system sdwan, diagnose sys sdwan service
  2. Is it UP? — get router info routing-table all (member route present ⇒ gateway set)
  3. Is it IN-SLA? — diagnose sys sdwan health-check (real bad numbers vs all-zeroes)
  4. Is it ELIGIBLE? — in this rule's priority-members AND holding a route to this destination
  5. Is it PREFERRED? — diagnose sys sdwan service (mode, selected flags, SLA presence/absence)

Session 64 adds rung 6: is the SESSION pinned? Climb top-down; stop at the first failing rung — everything below a broken rung is unreadable evidence.

Common mistakes

  • Writing a catch-all "to be safe" — it makes every later rule invisible while looking valid. (J.R., Friday 16:40.)
  • Configuring a member with interface + zone and no gateway — up, routed nowhere, dead to probes. (J.R., also Friday.)
  • Believing all-members-out-of-SLA skips the rule. It doesn't; best effort via configured order.
  • Assuming the implicit rule respects SLAs. It can't — no health check binds to it.
  • Expecting the FortiGate's own traffic to follow SD-WAN rules. It bypasses them until opted in per feature.
  • Steering by app-ctrl without application control in the firewall policy — a rule that matches nothing.
  • Choosing Best Quality "because it's dynamic." Both SLA-driven modes are dynamic; the question is gate vs ranking.

Exam tips

  • Member = interface + gateway; zone optional. All-zeroes health check ⇒ no path out ⇒ config, not carrier.
  • First match wins, top-down; unmatched traffic → implicit rule → global load-balance-mode → valid-route members only, no SLA.
  • mode sla, everything out: best effort via configured member order; rule NOT skipped. Skip requires no usable members.
  • Local-out: bypasses by default; fix per feature with interface-select-method {auto|sdwan|specify}.
  • ISDB = first-packet match. App-ctrl = first session escapes, then cached; requires app control in the policy.
  • SD-WAN rules match src / user / app / port / ToS / input-device; static routes match destination only.
  • Identify which rung an exam's command output belongs to before answering — health-check output proves nothing about matching.

Key takeaways

The session in five lines: Interface says where, gateway says how to leave. Rules choose among routes; they never create them. First match wins, and the last line of the clipboard was written before you arrived — and it cannot read a thermometer. The firewall's own traffic isn't on the clipboard at all until you put it there. And the first video pays for everyone else's.
