When the Wrong Traffic Takes the Wrong Road

How a senior engineer troubleshoots SD-WAN steering. The emphasis is thinking, not commands.

Symptoms this guide covers

  • "Trades are slow but streaming is fine" (or the reverse) — traffic classes on the wrong members.
  • "The same N users are always slow; everyone else is perfect."
  • "A new link is up but carries nothing."
  • "A rule I wrote does nothing — traffic ignores it."
  • "Logs / FortiGuard / DNS from the firewall itself behave differently from user traffic."
  • "First session of an application misroutes, then it behaves."

Questions to ask before touching anything

1. Whose traffic, exactly? Everyone-sometimes, some-always, or the firewall itself? The population shape narrows the fault class before any command runs (hash pinning vs per-flow balancing vs local-out).
2. Which rule do you believe caught it? Not which rule you wrote — which rule the box would read first. What sits above it?
3. Is the affected traffic user-originated or self-originated? Self-originated traffic isn't playing this game at all unless someone opted it in.
4. When did it start, and what changed? A new member changes the hash divisor for everyone. A new rule above changes matching for everyone below.

The decision tree (the ladder, operationalised)

START: "Traffic class X is on the wrong member." │ ├─ RUNG 1 — MATCH. show system sdwan → read services top-down as the box would. │ Does an EARLIER rule also match this traffic? │ ├─ YES → found it. Catch-all or overlap above. Fix ordering / delete catch-all. STOP (design fix). │ ├─ RUNG 2 — UP. get router info routing-table all │ Member route (via gateway) present for the intended member? │ └─ Connected route only → no gateway on the member. Fix, re-verify. STOP. │ ├─ RUNG 3 — IN-SLA. diagnose sys sdwan health-check │ ├─ Real bad numbers → carrier/path problem. (Evidence for Baz's renewal meeting.) │ └─ dead + 100% loss + 0.000 everything → nothing ever left → config (rung 2 missed). │ ├─ RUNG 4 — ELIGIBLE. Is the member in THIS rule's priority-members, │ and does it hold a route to THIS destination? │ └─ Alive in health-check, absent from service Members() → not invited. Add it. STOP. │ └─ RUNG 5 — PREFERRED. diagnose sys sdwan service Read Mode, selected flags, and whether an SLA is referenced AT ALL. ├─ mode load-balance + "same users always" → hash-mode pinning (see below). ├─ mode priority churning paths → no threshold; consider mode sla + hold-down-time. └─ mode sla, all members out → traffic uses configured order (NOT skipped, NOT dropped). If it went somewhere unexpected, re-read priority-members order.

Verification commands and what to read in them

diagnose sys sdwan serviceMode(...); which members show selected; src/dst ranges; presence or absence of the health check — absence in a rule that should be SLA-gated is itself the finding.

The "same four users" fingerprint

# Expected: mode load-balance + deterministic hash
Service(1): ... Mode(load-balance) ...
  and in config:  set hash-mode source-ip-based

Stable per-person misery = deterministic function of a stable identity input. Hash(source IP) mod member-count, and long DHCP leases keep the input constant. Confirm by correlating the affected users' IPs; fix by classifying the traffic that matters into an SLA-gated rule rather than fighting the hash.

Before you "fix" it by adding a member: changing the member count changes the divisor — every host rehashes. You may cure four users and sicken four others. Announce it.

Common dead ends

Dead end 1 — blaming the rule for a member problem. "It must be the rule" for port3 was plausible and wrong. Test: delete/ignore the rule mentally — does the symptom survive? If yes, the fault is below rung 1. (Señor's remove-the-explanation move.)
Dead end 2 — staring at the SLA when the implicit rule owns the traffic. If no explicit rule matches, no SLA is consulted anywhere. Tuning thresholds cannot help traffic the thermometer doesn't govern.
Dead end 3 — waiting for fall-through that will never come. All-out-of-SLA does not skip the rule. If you expected rule 2 to catch traffic when rule 1's members degrade, you designed against behaviour that doesn't exist.
Dead end 4 — adding routes for the firewall's own traffic. Static routes "fix" local-out only by accident and break on failover. The real control is interface-select-method per feature.
Dead end 5 — debugging an app-ctrl rule that can never match. No application-control profile in the firewall policy → no identification → the rule is inert while valid. Check the policy before the rule.

Likely root causes, ranked by how often they bite

  1. A broad rule above a specific one (first match wins).
  2. Member missing its gateway (all-zeroes signature).
  3. Important traffic on the implicit rule (unclassified, therefore un-gated).
  4. Hash-mode pinning misread as a link fault.
  5. Local-out traffic never opted in.
  6. App-ctrl steering without app control in the policy / first-session leak misread as flapping.

Exam traps

  • Output-to-rung mismatch: they show health-check output and ask a matching question. Name the rung the evidence belongs to first.
  • "The rule falls through when out of SLA" — distractor. Best effort via configured order.
  • "The implicit rule selects only in-SLA members" — distractor. Valid routes only; no SLA.
  • "Enable SD-WAN for local-out globally" — distractor. Per feature only.
  • Zone offered as the missing member field — gateway is the answer; zone is optional.
The senior habit underneath all of it: before commanding anything, say out loud which rung the symptom lives on and what evidence would prove you wrong. Symptom shape is evidence; the ticket pattern is your first diagnostic output, not noise before the real work.
