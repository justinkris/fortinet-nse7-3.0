==================================================================
TECHNICAL CONCEPTS COVERED
==================================================================
1. MEMBER GATEWAY MECHANISM. interface says where, GATEWAY says how to leave. Gateway is the
   input FortiGate uses to AUTO-BUILD the member's static route. Collapse chain (student built
   it from consequences): no gateway → no member route → probe cannot leave → 100% loss →
   state(dead) → never eligible. DIAGNOSTIC SIGNATURE: dead + 100% loss + latency/jitter both
   0.000 = nothing ever left = CONFIG symptom; a degraded link returns real numbers badly.
2. RULES DO NOT CREATE REACHABILITY. SD-WAN will not steer to a member with no valid route to
   that destination; rules choose AMONG members that already have one.
3. RULE LOOKUP PROCESS. Top-down by sequence, FIRST MATCH WINS, nothing below is read.
   J.R.'s catch-all in seat one = every later rule is dead config that looks valid.
   Specific first, general last. Delete the catch-all, don't demote it.
4. MATCHING CRITERIA (vs static route's destination-only): src/users/groups, dst,
   internet-service (ISDB), internet-service-app-ctrl, protocol + start-port/end-port,
   tos/tos-mask, input-device.
5. FOUR STRATEGIES (GUI/CLI): Manual (`manual`, no SLA, steers into sick links); Best Quality
   (`priority`, best MEASURED link-cost-factor, NO THRESHOLD); Lowest Cost SLA (`sla`, SLA as
   ELIGIBILITY GATE, then lowest cost, then config order); Maximize Bandwidth (`load-balance`,
   SLA gate + hash-mode spread). Dividing line: sla/load-balance use SLA as a GATE; priority
   RANKS; manual doesn't look.
6. A-vs-B DESIGN ARGUMENT (trading feed): Best Quality churns paths for 1ms prizes and cannot
   say "both roads are unusable" (no concept of acceptable); mode sla is stable by default,
   decisive on breach — but only as honest as its threshold. hold-down-time 30 as the
   flap damper ("quick to distrust, slow to forgive" one layer up).
7. HASH MODES. Per-rule `hash-mode` (round-robin default, source-ip-based,
   source-dest-ip-based...). hash mod member-count → deterministic per-host assignment.
   SYMPTOM SHAPE FINGERPRINT: "everyone occasionally slow" = round-robin; "same four always" =
   source-ip pinning. Divisor change warning: fixing port3 changes mod 2 → mod 3, EVERY host
   rehashes with no rule touched.
8. IMPLICIT RULE. Cannot be deleted/edited as a rule; it IS the global `set load-balance-mode`
   under config system sdwan. Balances across all members WITH A VALID ROUTE to the
   destination. DOES NOT CONSULT ANY SLA — out-of-SLA members still receive implicit traffic
   because their routes remain. (Ties to S62's "out-of-SLA does not remove the route.")
9. LOCAL-OUT TRAFFIC. Student derived it in two words: "no interface." Self-originated traffic
   never ingressed → nothing for rules to match → bypasses SD-WAN, plain routing lookup
   (hence Cipher's 50/50 syslog). Fix is PER-FEATURE, no global switch:
   `set interface-select-method {auto|sdwan|specify}` under log syslogd setting,
   system fortiguard, system dns, snmp sysinfo, etc.
10.APPLICATION STEERING. ISDB = destination lookup (IP/proto/port) → matches on FIRST packet.
   app-ctrl = identification via Client Hello / SNI / server cert → several packets in →
   FIRST SESSION ESCAPES to fall-through, then app-to-destination mapping is CACHED
   (diagnose sys sdwan internet-service-app-ctrl-list) so later sessions match on packet one.
   app-ctrl steering REQUIRES the firewall policy to run application control, or the rule
   matches nothing forever while looking valid.
11.MODE SLA ALL-MEMBERS-OUT BEHAVIOUR (exam miss, Q4): rule STILL MATCHES (matching is
   src/dst/service, not health) and falls back to best effort = configured priority-members
   order. Rules are skipped only when NO usable members (all dead / no route). Sick ≠ dead.
