==================================================================
TECHNICAL CONCEPTS COVERED (pp. 44-47)
==================================================================
THREE RUN TARGETS (Device Manager > Scripts):
1) REMOTE FORTIGATE DIRECTLY (via CLI): executes on live box; FMG then AUTO-RETRIEVES the
   result into the device DB (new revision in Configuration Revision History); Config Status =
   Synchronized. EXCEPTION: firewall-policy changes are NOT carried up to the policy layer —
   manual IMPORT required. Use case 1 in guide: pushing a remote CA certificate. Avoid modifying
   objects used in firewall policies (policy-layer conflict risk).
2) DEVICE DATABASE: edits FMG's mirror only; NOTHING pushed to the box; Config Status =
   Modified; deliver with Install Device Settings (or package install if policies rode along).
   Use case 2: custom admin profile.
3) POLICY PACKAGE / ADOM DATABASE: writes to policy layer; package flagged Modified; deliver
   with Install Policy Package (full trip: policy layer → device DB → FortiGate). No
   auto-retrieve role (pump intake is the box; box never touched). Avoid importing the package
   afterwards. Objects attached to the package install; unattached objects stay in ADOM DB.
CLI vs TCL: CLI = fixed sequence of FortiOS commands ("a letter"), familiar syntax, metadata
variables; Tcl = program that GENERATES commands ("a person") — if/else, loops, proc, runtime
variables; costs complexity and 2am auditability. (Full comparison table incl. Jinja/API is
pp. 49-50, S34 territory.)
DEVICE-DB POLICY CHANGES cannot be auto-imported into the policy layer (exam Q7 fact).

==================================================================
MENTAL MODELS / ANALOGIES THAT LANDED
==================================================================
- "The run target is the address on the envelope." Wrong address = staged, not shipped.
- "The change exists only where it landed; downstream needs deliberate delivery, upstream needs
  deliberate import." (Student's consolidated law of the session.)
- "Automate where blast radius is one (mirror sync); gate with a human where blast radius is
  everyone (Import into shared truth)." — student derived the blast-radius asymmetry himself.
- Auto-retrieve = "an upward pump with a fixed intake: box → device DB". Student used this to
  correctly predict no auto-retrieve on package scripts.
- CLI script = letter (cannot decide, cannot invent NEW mistakes); Tcl = person (can decide).
- "Automation removes variance, not mistakes" (Cipher). Wrong change also ships before lunch.
- "A green checkmark means the script ran, never that the change arrived."
- "Reinstall skips the questions, never the confession" (wizard skipped, preview kept) — welded
  anchor for the S30 reinstall fact.
- "Dictionary ≠ Constitution": normalized interface = horizontal translation of port-name
  vocabulary; global ADOM = vertical authority (header/footer rules above local packages).

TROUBLESHOOTING APPROACH TAUGHT
==================================================================
"The script ran but nothing happened": 1) don't debug the script — check the RUN TARGET first;
2) remote → check revision history for auto-retrieve entry; 3) device DB → Modified = staged,
install device settings; 4) package → package Modified, install package; 5) policies never climb
on their own — Import deliberately.
