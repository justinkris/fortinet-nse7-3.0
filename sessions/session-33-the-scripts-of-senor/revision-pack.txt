QUICK SUMMARY (one page)
====================================================================
FortiManager scripts (Device Manager > Scripts) execute in one of THREE contexts, and the
run target — "the address on the envelope" — decides everything:

1) REMOTE FORTIGATE DIRECTLY (via CLI)
   - Executes on the LIVE FortiGate.
   - FMG then AUTO-RETRIEVES the result into the device database (new entry in
     Configuration Revision History). Config Status = Synchronized.
   - EXCEPTION: firewall-policy changes land in the device DB but do NOT climb into the
     policy package — manual IMPORT required.
   - Guide use case: create/push a remote CA certificate. Avoid touching objects used in
     firewall policies (conflict risk).

2) DEVICE DATABASE
   - Edits FMG's mirror ONLY. Nothing reaches the box.
   - Config Status = Modified. Deliver with INSTALL DEVICE SETTINGS (or a package install
     if policy changes are included).
   - Guide use case: custom administrator profile.
   - Device-DB POLICY edits CANNOT be automatically imported into the policy layer. Ever.

3) POLICY PACKAGE / ADOM DATABASE
   - Writes into the policy layer (e.g. address + policy). Package flagged Modified.
   - Deliver with INSTALL POLICY PACKAGE (policy layer -> device DB -> FortiGate).
   - No auto-retrieve role (its intake is the box; the box was never touched).
   - Avoid importing afterwards. Objects attached to the package install; unattached
     objects wait in the ADOM DB.

THE LAW: the change exists only where it landed. Downstream needs deliberate delivery;
upstream needs deliberate import. "Modified" = FMG holding work you haven't shipped.

LANGUAGES: CLI script = a LETTER (fixed FortiOS commands, familiar syntax, metadata
variables, cannot decide, cannot invent NEW mistakes). Tcl script = a PERSON (proc,
if/else, loops, runtime variables — a program that GENERATES commands; can branch per
device, e.g. the Melbourne port2 box). Tcl buys logic; pays in complexity/auditability.

WISDOM: automation removes VARIANCE, not mistakes (Cipher). Automate where blast radius
is one (mirror sync); gate with a human where blast radius is everyone (Import).

CHEAT SHEET
====================================================================
GUI PATHS
  Device Manager > Scripts > Create New          (choose Type: CLI/Tcl; choose run target)
  Device Manager > Scripts > Run Script
  Device Manager > Device & Groups > [FortiGate] (Config Status)
  Dashboard: Summary > Configuration and Installation > Revision  (auto-retrieve receipts)

RUN-TARGET TABLE (memorise cold)
  Target                | Lands on      | Status flip          | Delivery            | Auto-retrieve
  Remote FortiGate      | live box      | Synchronized (after) | none; Import if pol | YES
  Device Database       | FMG mirror    | device Modified      | Install Dev Settings| no
  Policy Pkg / ADOM DB  | policy layer  | package Modified     | Install Policy Pkg  | no

KEY BEHAVIOURS
  - Auto-retrieve = upward pump, fixed intake: box -> device DB. Nothing else. Doesn't
    update the policy package.
  - Remote-script policy changes: retrieved to device DB, Import needed for package.
  - Device-DB policy edits: CANNOT auto-import to policy layer.
  - Package scripts: don't import afterwards; attached objects install, unattached stay
    in ADOM DB.
  - Reinstall Policy: skips the WIZARD, keeps the PREVIEW; also pushes device-DB
    modifications. ("Skips the questions, never the confession.")
  - Green checkmark = ran, not delivered.

DESIGN RULES
  - Choose CLI unless you need decisions; the letter cannot make new mistakes.
  - Stage risky changes against the device DB / package; review; deliver deliberately.
  - Before installing staged work: check revision history for out-of-band changes.

FLASHCARDS
====================================================================
Q: Three script run targets?
A: Remote FortiGate Directly (via CLI); Device Database; Policy Package/ADOM Database.

Q: What does FMG do automatically after a remote-FortiGate script?
A: Auto-retrieve — captures the box's config into the device DB as a new revision;
   status Synchronized.

Q: Remote script created a firewall policy — where does it stop?
A: Device DB. Manual Import required to bring it into the policy package.

Q: Device-DB script ran; what's the status and the fix?
A: Modified; Install Device Settings (or package install if policies included).

Q: Can device-DB policy changes auto-import into the policy layer?
A: No — never. Manual import path only.

Q: Package script created address + policy — how do changes reach the FortiGate?
A: Install the policy package (policy layer -> device DB -> FortiGate).

Q: Which target requires NO install afterwards?
A: Remote FortiGate Directly — the change is already live.

Q: Does auto-retrieve fire after device-DB or package scripts?
A: No. Its intake is the box; the box was never touched.

Q: CLI script advantages?
A: Familiar FortiOS syntax; metadata-variable support; auditable; no new failure modes.

Q: When do you need Tcl?
A: When the script must decide — if/else, loops, per-device branching.

Q: After a package script, why avoid importing?
A: Import drags the device layer's view over your fresh policy-layer work; scripts wrote
   to the source of truth directly.

Q: Reinstall Policy vs Install Policy Package — what's skipped, what's kept?
A: Wizard skipped (device/package already paired); PREVIEW kept. Reinstall also pushes
   device-DB modifications.

Q: Staged device-DB changes Friday, SSH hotfix Saturday, install Monday — risk?
A: The install can overwrite the hotfix. Retrieve/reconcile revision history first; read
   the preview.

Q: Guide use case for a remote script? For a device-DB script?
A: Remote: create/push a remote CA certificate. Device DB: custom administrator profile.

Q: Cipher's law of automation?
A: Automation removes variance, not mistakes — wrong changes also ship before lunch.
