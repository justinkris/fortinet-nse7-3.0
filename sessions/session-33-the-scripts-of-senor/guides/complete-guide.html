Episode Recap

The Disaster Desk Delivers

Tuesday. Junior's Excel thirty-command generator still glowed on the big screen. Señor arrived early, wearing reading glasses nobody knew existed, sat at the worst desk in the southern hemisphere, and opened a window the team had never clicked: Device Manager > Scripts. Ten lines of plain FortiOS CLI — syslog server, admin timeout, DNS fix — run target Remote FortiGate Directly (via CLI). By 09:03 the entire fleet was updated, each run stamped into revision history. Junior's calendar described a week of work that no longer existed.

Then Junior wrote his own script — an admin profile for Riley — aimed it at the Device Database, saw a green checkmark, and told Riley "done." Riley could not log in. The profile existed in FortiManager's mirror, staged and undelivered, its Config Status quietly reading Modified. The envelope had the wrong address.

Cipher's correction, which owns the second half of this course: automation removes variance, not mistakes. A correct change becomes correct everywhere before lunch — and a wrong change becomes wrong everywhere before lunch.
Learning Objectives

Core Concept

Three Targets, One Envelope

Think First
Before reading on: a script runs successfully against each of the three targets. For each one, answer the three beats — where does the change exist, what does FortiManager flag, and what human operation (if any) finishes the job?
  • Remote FortiGate Directly (via CLI) — what does FMG do immediately after the script finishes?
  • Device Database — why does a green checkmark not mean Riley can log in?
  • Policy Package — why does auto-retrieve play no role here at all?
Analogy · The Envelope
The run target is the address on the envelope. The script is the letter inside. A perfectly written letter posted to the wrong address arrives somewhere real — just not where anyone is waiting for it. "The script ran" describes the letter. It says nothing about delivery.

Target 1 — Remote FortiGate Directly (via CLI)

The script executes on the live FortiGate. FortiManager isn't ambushed by this change — it pulled the trigger — so it immediately runs an auto-retrieve, capturing the resulting config into the device database as a new entry in Configuration Revision History. Config Status: Synchronized. Study-guide use case: creating and pushing a remote CA certificate.

The exception: firewall-policy changes made by a remote script are retrieved into the device DB like everything else, but they do not climb into the policy package. That hop requires a manual Import. Also: avoid modifying objects used in firewall policies with remote scripts — you invite policy-layer conflicts.
🧠
Mental Note
Automate where blast radius is one (mirror sync). Gate with a human where blast radius is everyone (Import into the shared policy package).

Target 2 — Device Database

The script edits FortiManager's mirror only. Nothing touches the box. The device's Config Status flips to Modified (exclamation-triangle) — FMG saying "I'm holding work you haven't shipped." Deliver with Install Device Settings (or a policy-package install if policy changes rode along). Study-guide use case: a custom administrator profile — exactly Junior's script.

Junior's law: a green checkmark means the script ran. It never means the change arrived. The Config Status contains the entire diagnosis.

Target 3 — Policy Package / ADOM Database

The script writes into the policy layer — e.g. creating a firewall address and a policy that uses it. The policy package is flagged Modified. Deliver with Install Policy Package — the full trip: policy layer → device DB → FortiGate. Objects the script attaches to the package get installed; objects left unattached remain in the ADOM database.

Guide warning: avoid importing the policy package after these scripts. Scripts write to the source of truth directly; Import is for when truth arrived from below — a careless import can drag the device layer's older view back over your scripted work.

Core Concept

Two Languages: The Letter and the Person

A CLI script is a fixed sequence of FortiOS commands — every device receives identical words. Its strengths: you already speak FortiOS, and it supports metadata variables. A Tcl script is a program that generates FortiOS commands: proc, if/else, loops, runtime variables. It can look at the device it's standing on and decide.

Analogy · Letter vs Person
A CLI script is a letter: written once, identical for every recipient. A Tcl script is a person: it can ask questions and branch. The Melbourne recabled 100F needs port2 where twenty-nine siblings use port1 — a letter gets it wrong; a person checks first. The trade: "The letter cannot make decisions — but the letter also cannot make new mistakes." Tcl buys logic and pays in complexity and 2am auditability.
Señor's ten-liner — plain CLI, remote target (fragment)
config system dns
    set primary 10.10.10.53
end
config log syslogd setting
    set status enable
    set server 10.10.20.14
end
config system global
    set admintimeout 15
end

Readable at a glance — Cipher can audit it in ten seconds. The 2021 Tcl relic below it, with brackets breeding in every line, requires an engineer and coffee. Choose Tcl only when you genuinely need decisions.

Workflow

"The Script Ran But Nothing Happened"

  1. Don't debug the script. Check the run target first — the envelope's address explains most of these tickets.
  2. Remote FortiGate → the change is live. Verify via Dashboard > Summary > Configuration and Installation > Revision — the auto-retrieve entry is your receipt.
  3. Device DB → check Config Status. Modified = staged, not shipped. Install device settings.
  4. Policy package → package flagged Modified. Install the policy package.
  5. Policies involved? They never climb to the policy layer on their own — Import, deliberately, eyes open.
Common Mistakes

Where Engineers (and Junior) Go Wrong

  • Green checkmark = done. No. It means "ran." Only the status/revision history tells you where the work is sleeping.
  • Expecting device-DB policy edits to appear in the package. They cannot auto-import. Ever. (Exam favourite.)
  • Importing the package after a package script. You can drag the device layer's stale view over your fresh policy-layer work.
  • Remote scripts touching policy-referenced objects. Conflict bait for the policy layer.
  • Reaching for Tcl by default. Complexity is a cost. The letter cannot make new mistakes.
  • Staging device-DB changes, then installing days later without checking for out-of-band changes. An SSH hotfix made in between can be overwritten by your Monday install — reconcile revision history first.
Exam Corner

Ten Questions, Answers Hidden

Q1. Remote-FortiGate script changes DNS. Config Status after the automatic follow-up?
C — Synchronized. Auto-retrieve pulls the result into the device DB.
Q2. The remote script also created a firewall policy. True statement?
B. Policy exists on the box and in the device DB; manual Import required for the policy package.
Q3. Device-DB script creates an admin profile. Immediately afterward?
B. Config Status = Modified; an install is required. Nothing on the box yet.
Q4. Policy-package script created an address + policy. To apply to the FortiGate?
C — Install the policy package (policy layer → device DB → FortiGate).
Q5. Which target applies changes to the live FortiGate with no subsequent install?
C — Remote FortiGate Directly (via CLI).
Q6. Script must branch per-device with if/else. Language?
B — Tcl. CLI scripts are fixed sequences; they cannot decide.
Q7. Script changes policy settings in the device database. Correct statement?
B — they cannot be automatically imported into the policy layer. Not flagged upward, not discarded, not auto-imported. Trap note: the correct option is the boring restatement of the design law — don't skip it for options with more drama in them.
Q8. CLI script's chief advantages over Tcl for a FortiGate-fluent team?
B — familiar FortiOS syntax and metadata-variable support.
Q9. Reinstall Policy vs Install Policy Package — the preview?
B — Reinstall skips the wizard but still offers the installation preview. Anchor: "Reinstall skips the questions, never the confession." And reinstall pushes device-DB modifications too — the preview matters more, not less.
Q10. Device-DB changes staged Friday; out-of-band SSH fix Saturday; installing Monday. Check what?
B. Revision history / sync state — retrieve and reconcile the out-of-band change first, or the staged install may overwrite Saturday's fix.
Close

Key Takeaways

  • Run target = the address on the envelope. Three targets, three beats each: lands where / flips what / delivered how.
  • Auto-retrieve is an upward pump with a fixed intake — box → device DB, nothing else. Remote scripts only.
  • Policies never climb on their own. Import is the human gate guarding the shared source of truth.
  • Modified means "staged in FMG, undelivered." Install is the delivery.
  • CLI = letter (FortiOS syntax, metadata variables, no new mistakes). Tcl = person (logic, at a complexity cost).
  • Automation removes variance, not mistakes. Next session, that sentence gets teeth.
Business outcome: a week of fleet-wide work now takes minutes, every run logged in revision history for the auditors, engineering time freed — and staged changes can be reviewed before deliberate delivery.
