Where Is My Change Sleeping?

The script ran. The change didn't arrive. A senior engineer checks the envelope before blaming the letter.

Symptoms

  • "The script ran successfully but the config isn't on the FortiGate." (Junior's ticket, verbatim.)
  • A device or package sits at Modified and nobody remembers why.
  • A policy created by script is visible in the device DB but missing from the policy package.
  • An install pushed changes nobody expected (staged script work riding along, or an out-of-band fix overwritten).

Questions a Senior Asks First

  1. What was the run target? The envelope's address explains 80% of these tickets before any debugging.
  2. What does the Config Status say right now? Synchronized / Modified is the system telling you where the work is sleeping.
  3. Did the change involve firewall policies? If yes, an Import gate sits in the path — was it crossed?
  4. What's in the revision history? Script runs leave receipts via auto-retrieve. No receipt on a "remote" run = the script never touched the box.
  5. Has anything happened out-of-band since staging? An SSH fix between "script Friday" and "install Monday" is a collision waiting for the preview to catch.

Decision Tree

"Script ran but nothing happened" — what was the run target?
REMOTE FORTIGATE → Check revision history for the auto-retrieve receipt.
Receipt present → change IS live. Complaint is about the policy layer? → policies don't auto-import → run Import.
No receipt → script never reached the box → check device connectivity / script errors, not the config.
DEVICE DATABASE → Status Modified? → staged, not shipped → Install Device Settings. Read the preview.
POLICY PACKAGE → Package Modified? → Install Policy Package. Objects unattached to the package will NOT install (they wait in the ADOM DB).
Before ANY install: has the box changed out-of-band since staging?
Yes/unsure → retrieve + reconcile first, or the install may overwrite the fix. The preview is where secrets confess — read it.

Common Dead Ends

  • Debugging the script syntax when the script ran fine — against the wrong target. Check the envelope first.
  • Waiting for the policy to "sync up" into the package. It will wait forever. The Import gate is deliberate.
  • Importing after a package script "to be safe." That drags the device layer's stale view over fresh policy-layer work.
  • Re-running the script on a Modified device. You'll stage the same change twice and learn nothing — the status already told you the answer.

Likely Root Causes, Ranked

  1. Wrong run target selected at script creation (Junior's classic).
  2. Delivery step skipped — Modified status ignored, install never run.
  3. Import step skipped for policy changes after a remote script.
  4. Out-of-band change collided with staged device-DB work.
  5. Package script's objects never attached to the package (stranded in ADOM DB).

Exam Traps

Trap 1: options claiming device-DB policy changes "flag the package Modified" or are "auto-imported." The boring truth — they cannot be automatically imported — is the answer. When one option restates the design law you already know, take it over the dramatic ones.
Trap 2: reinstall vs install. Reinstall skips the wizard, keeps the preview — and pushes device-DB modifications too. "Skips the questions, never the confession."
Trap 3: "which target needs no install?" Only Remote FortiGate Directly — it's already live. Everything else stages.
