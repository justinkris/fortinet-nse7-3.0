The 2.1-Millisecond Heist

Core Concept

The Split: Sort by the Shape of the Problem

🤔Think First
Before you learn any part numbers, answer the question Señor asked before allowing the word "chip":
  • A FortiGate does session lookups, routing, NAT, IPsec crypto, IPS matching, AV, SSL decryption, forwarding. If you split all of that into two fundamentally different kinds of job, where does the line go?
  • What makes the two categories computationally different?

The answer is the whole chapter in one cut. One category is forwarding work: repetitive, per-packet, decided-once. A session was allowed; every subsequent packet with the same 5-tuple deserves the same verdict — so why re-decide? This is a fixed-function job, the same species of problem a switch ASIC solves. The other category is content work: searching inside payloads for patterns — IPS signatures, virus fingerprints, decrypting TLS to look deeper. A completely different computational animal.

Fortinet's umbrella term for its specialist silicon is the SPU — Security Processing Unit (an ASIC). An SPU can be a network processor (NP), a content processor (CP), or both, and on many models both work simultaneously as traffic passes through, offloading resource-intensive work from the main CPU.

🧠
Mental Model
NP is stationed at the ports. CP is chained to the CPU's desk. One moves packets; the other reads them.
Network Processor

NP: The Forwarding Specialist

The NP works at the interface level — it sits in the packet's path at the front door. Its jobs:

  • Accelerates traffic by offloading it from the CPU (established sessions bypass the CPU — Session 41's subject).
  • IPsec VPN encryption and decryption for supported algorithms. A plain IPsec tunnel is a forwarding job wearing a trench coat — repetitive per-packet transform maths, not content searching. This is why "which SPU handles plain IPsec?" answers NP.
  • IP integrity header check and early inspections (ACL, HPE) — door-keeper's chores, done at the doorway to spare the CPU.

The latest generation is the NP7. Key number: it supports up to 12 million sessions, and the limit is processor memory — twelve million session entries physically live in on-chip tables. Once an NP7 hits its limit, overflow sessions are sent to the CPU — the slow path.

The per-chip trap: each physical port is wired to a particular NP7, and the chips do not pool memory. A box with two NP7s has 12M sessions per chip, not 24M wherever you like. You must know which interfaces connect to which NP7 and distribute incoming traffic accordingly. Cable every busy link to one chip and you get CPU overflow on one side and an expensive paperweight on the other — with nothing misconfigured. The failure was committed with a patch cable.

Content Processor

CP: The CPU's Rented Muscle

The CP functions like a coprocessor for the FortiGate CPU. It never sits in the packet path on its own; the CPU stays the decision-maker and slides mountains of work sideways to the CP:

  • Pattern matching for flow-based UTM inspection with the IPS engine — application identification, IPS, antivirus in flow mode.
  • SSL/TLS decryption and encryption for deep SSL inspection.
  • IPsec crypto for supported algorithms (the CP can also do this).

Generations that exist: CP8, CP9, CP10 (CP10 is latest; LoFinance's 900G carries CP9). There is no CP7 — a fact worth stating because a plausible-sounding invented label is exactly what exam pressure produces.

Pre-reading answer: a firewall with only NPs is not sufficient for a heavy SSL-decrypt workload. SSL deep-inspection crypto is CP work; without a CP, that entire load lands on the CPU — precisely the workload that crushes it.
Combined Chips

SoC / SP: Same Organs, Smaller Animal

Entry-level and mid-range FortiGates can't carry two families of dedicated silicon. For those, Fortinet combines light versions of the NP and CP on a single chip — the SoC/SP family. Latest generations: SoC4 and SP5. The SP5's alias is NP7lite — remember it; it appears in CLI output (np7lite_0).

The tradeoff, in one word: throughput. The study guide's phrase — combined NP and CP components on one chip with lower throughput. A branch 90G has both specialists, but apprentice versions sharing one die: no ultralow-latency ports, no 12M-session tables, no NP Direct. Fine for guest Wi-Fi and a couple of tunnels; absurd for a trading floor measured in microseconds.

Architecture

NP Direct: Remove the Toll Booth

Inside a normal FortiGate, between the front ports and the processors sits the ISF — Internal Switch Fabric — a hardware switch chip gluing every port to every processor. It's the any-to-any bridge, and it charges a toll: a small latency cost, paid on ingress and again on egress, on every packet, forever.

NP Direct removes the stop entirely: the FortiASIC network processor is directly connected to the interfaces, eliminating the ISF — the lowest-latency forwarding available. This is the literal answer to Penny's question.

  • NP6 flavour: available on devices with two or more NP6 processors — FortiGate 200E, 2000E, 2500E.
  • NP7 flavour: devices with ultralow-latency interfaces wired directly to the NP7 — FortiGate 400F, 600F, 900G, where ports X5–X8 connect straight to the NP7.
The requirement: offloaded traffic must enter and exit FortiGate through interfaces connected to the same NP6/NP7 processor. The bridge is gone; a port can only reach the chip it's soldered to.
The Rack-Tape Rule (student-derived)
"In on X5, out on X5–X8. Same chip in, same chip out — there is no bridge anymore."

And the cost of the trade: the ISF is what let many front ports share one NP. Remove it on a single-NP box and you strand most of the ports — which is why NP Direct on the NP6 side requires two or more NP6 chips: enough direct-attach silicon to afford giving up flexibility for latency. On this architecture, physical topology is part of the offload design — whoever patches cables is a stakeholder, because a well-meaning "tidy-up" onto a normal ISF port silently reroutes traffic onto the slow path with a clean config diff.

Verification

Proving What's In the Box

Before anyone spends money on a quote, verify the silicon actually in the rack.

Identify the NP — PCI vendor ID 1a29 = Fortinet
# diagnose hardware lspci | grep 1a29
9b:00.0 Class 1000: Device 1a29:4e37

# NPID 4e36 = NP6
# NPID 4e37 = NP7
Identify the CP — or confirm an SoC
# get hardware status
Model name: FortiGate-900G
ASIC version: CP9        <- CP generation on discrete-chip models

# get hardware status
Model name: FortiGate-91G
ASIC version: SOC5       <- confirms the SP5 combined chip
Mystery-box lesson: a 2500E showing 1a29:4e36 + ASIC version: CP9 is NP6 + CP9 — and 4e36 is easy to misread as NP7 in a hurry. The senior follow-up isn't "write it off": the 2500E is an NP-Direct-capable NP6 chassis if it carries 2+ NP6 chips. Count the chips before you count it out.
Common Mistakes

Where This Topic Bites

  • Treating session capacity as per-box. It's per-chip. Cabling is capacity planning.
  • Confusing the two latency diagnoses. ISF-hop latency (architectural, fixed by NP Direct) and CPU-overflow latency (capacity, fixed by traffic distribution) are different faults with different fixes. Distractors bet on you conflating them.
  • Assuming NP handles SSL inspection. SSL deep-inspect crypto is CP work. NP does IPsec; CP does SSL (and IPsec too, for supported algorithms).
  • Inventing labels under pressure. CP generations are 8, 9, 10. SP5's alias is NP7lite, not "SOC5" (SOC5 is what the CLI field displays on a 91G — different fact).
  • Forgetting the same-NP requirement when NP Direct is in play — traffic must enter and exit on the same processor's interfaces.
Senior Insights

How a Senior Engineer Thinks About Silicon

  • Locate the latency before buying anything. Señor ruled out the WAN with arithmetic (40m fibre ≈ 200ns) before opening the firewall. Diagnosis precedes procurement.
  • Sort workloads by problem shape, then pick silicon. Forwarding vs content is the cut; features are just names for the two piles.
  • Verify the rack, not the quote. Two CLI commands tell you what you actually own.
  • Patch-cable failures beat config failures at hide-and-seek. On NP Direct, nothing in the config shows a mis-cabled egress. Document physical constraints where the hands that patch will see them.
  • Every acceleration feature is a trade. NP Direct trades any-to-any flexibility for latency. Know what you gave up, because that's where the next incident lives.
Exam Corner

NSE7 Tips

  • Memorise the mapping: 1a29 = Fortinet vendor ID; 4e36 = NP6, 4e37 = NP7. Expect a raw lspci output to read.
  • 12 million sessions, limited by processor memory, overflow goes to the CPU — all three clauses are askable.
  • NP Direct availability: NP6 with 2+ chips (200E/2000E/2500E); NP7 ultralow-latency interfaces (400F/600F/900G, X5–X8). Requirement: same NP in and out.
  • get hardware status → ASIC version shows CP generation on discrete models, SOC5 on SP5 models.
  • SP5 = NP7lite. SoC = light NP + light CP, lower throughput.
  • Job-to-chip matching questions: IPsec → NP (or CP for supported algos); IP integrity check → NP; SSL deep-inspect crypto → CP; flow-based pattern matching → CP; proxy-based inspection → CPU (no NP offload — Session 42 expands this).
Key Takeaways

If You Remember Five Things

  1. Two specialists: NP moves packets (at the ports); CP reads them (beside the CPU).
  2. NP7 holds 12M sessions per chip, memory-limited; overflow takes the CPU slow path — so cabling is capacity planning.
  3. NP Direct removes the ISF for the lowest-latency forwarding, at the price of flexibility.
  4. Same chip in, same chip out — there is no bridge anymore.
  5. Verify silicon with two commands: diagnose hardware lspci | grep 1a29 and get hardware status.
