SESSION 40 TROUBLESHOOTING GUIDE · SPU ARCHITECTURE

Where Do the Milliseconds Die?

How a senior engineer triages a hardware-acceleration latency complaint — thinking first, commands second.

Symptoms

What Walks In the Door

  • "Traffic through the firewall is slower than the same path used to be" — or slower than an identical box elsewhere.
  • Latency-sensitive traffic (trading, VoIP, storage replication) shows a consistent fixed-cost delay per hop.
  • Latency is fine at low load but a second, taller hump appears on the latency histogram at peak — a bimodal distribution.
  • CPU utilisation climbs with traffic that "should be offloaded."
  • Performance changed after a physical re-cable, with a clean config diff.
First Discipline

Locate Before You Diagnose

Señor's opening move: rule out the WAN with arithmetic (40m of fibre ≈ 200ns) before opening the firewall. Bound the search space with numbers you already know — link lengths, serialisation delays, published port-to-port latencies — so the suspect list shrinks to boxes you own. Diagnosis precedes procurement.

Questions to Ask

Before Touching a Keyboard

  1. Is the delay fixed-cost or load-dependent? Fixed per-packet cost points at path architecture (ISF hops, extra processing stages). Load-dependent points at a capacity limit being crossed (session table full, CPU overflow).
  2. Is it all traffic or some traffic? A bimodal latency profile means two populations — some sessions riding fast silicon, some not. Ask what distinguishes them.
  3. What silicon is actually in this box? Never assume from the model name on the bezel.
  4. Which physical ports does the affected traffic use — and which NP owns them? On multi-NP and NP-Direct hardware, cabling is part of the design.
  5. What changed? Config diffs won't show a patch-cable change. Ask the humans, check the DC access log, look at interface counters moving between ports.
Verification

Commands and Expected Output

# diagnose hardware lspci | grep 1a29 9b:00.0 Class 1000: Device 1a29:4e37 → 1a29 = Fortinet vendor ID → 4e36 = NP6 · 4e37 = NP7 # get hardware status Model name: FortiGate-900G ASIC version: CP9 ← CP generation (ASIC version: SOC5 on an SP5/branch box)

These two commands answer "what am I actually working with" — the mandatory first evidence on any acceleration case. Deeper NPU counters (diagnose npu np7 ...) belong to later sessions; do not guess at their output.

Likely Root Causes

Ranked by Frequency of Embarrassment

  1. Traffic not on the low-latency ports it was designed for — a re-patch, a "tidy-up," a port swap during maintenance.
  2. Ingress and egress on different NPs where NP Direct is in use — breaks the same-chip requirement.
  3. Session concentration on one NP — overflow to CPU at peak while a sibling NP idles.
  4. Wrong-tier hardware for the workload — an SoC/SP5 branch box asked to do a datacentre job, or an NP-only expectation applied to a CP workload (SSL deep inspection).
Exam Traps

How the NSE7 Will Test This

Trap 1: an lspci output showing 1a29:4e36 with an answer option saying "NP7." 4e36 is NP6. Read the hex, not your hope.
Trap 2: "What limits NP7 sessions?" with distractors like licensing or interface count. It's processor memory — and the overflow destination is the CPU.
Trap 3: an NP-Direct scenario where traffic enters X5 and exits a port on a different NP — the requirement is same-NP in and out.
Trap 4: job-to-chip mismatches: SSL deep-inspect crypto assigned to the NP, or IPsec denied to the NP. NP does IPsec; CP does SSL (and IPsec for supported algorithms); flow pattern matching is CP work.
The Point

Thinking, Not Memorising

Every fault in this chapter reduces to one of two questions: "is the path longer than it needs to be?" (architecture) or "did a table fill up?" (capacity). Classify the symptom into one of those before running a single command, and the command list picks itself.
