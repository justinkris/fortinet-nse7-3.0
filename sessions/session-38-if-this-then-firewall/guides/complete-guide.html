Concept 1

Stitch = Trigger + Action(s)

Think First

Junior's smart lights: phone joins Wi-Fi → lights turn purple. What two components must every automation stitch pair together?

A stitch is an administrator-defined automated workflow using if/then logic: an event trigger paired with one or more actions, so FortiOS responds to an event in a preprogrammed way — no human hand between the event and the response. Triggers can come from the FortiGate itself or from other Security Fabric devices via FortiAnalyzer event handlers.

Fabric not required. A standalone FortiGate can run stitches. But when stitches live inside a Security Fabric, they must be configured on the root FortiGate, scoped to all FortiGates or a subset, and pushed to the relevant leaf FortiGates. GUI: Security Fabric > Automation.
Concept 2 — The Exam Favourite

Push at Config Time, Not Trigger Time

Junior's "flaw in the universe": if the root dies, does fabric-wide automation die with it? No. The push happens when the stitch is configured — each leaf then owns a local copy. At trigger time the leaf executes its own copy; the root is not consulted.

The rule: The root FortiGate does not need to be operational for previously configured stitches to function on leaf FortiGates. The root is the author, not the dispatcher. (Rhymes with Season 4's "crown vacant, witnesses keep writing.")
Concept 3 — Use Case 1

High CPU: Photograph the Crime Scene

Stitch creation is a three-step sequence: 1) Trigger → 2) Action(s) → 3) Stitch (pairing them). For CPU spikes, use the preconfigured High CPU trigger, which fires when the CPU exceeds the configured threshold:

CLI — threshold (default 90%)
config system global
    set cpu-use-threshold 90
end

An email alone repeats the original failure — by morning the evidence is gone. So pair two custom actions:

  • Collect_Diagnostics_Action — a CLI script running diagnostic commands at the moment of the spike
  • Email_Diagnostics_Action — notifies staff, with the script's output as the email body via the parameter %%results%%
Sequential + delay. In the stitch, set the actions to Sequential — "use the results of the previous action for the next action" — with a 30-second delay so the diagnostics complete before the email consumes their output. Fire both at once and %%results%% is an empty envelope. The delay is a dependency, not a courtesy: let the kettle boil before pouring.
Concept 4 — Use Case 2

Config Backup: Time Itself Is a Trigger

A recurring backup reacts to nothing — but stitch grammar has no verb-less sentence. The trigger is the Schedule trigger: frequency once, hourly, daily, weekly, or monthly (the GUI menu changes with the frequency you pick), governed by the root FortiGate's time system. The backup action applies to all fabric FortiGates or a defined subset; each run lands as a revision in Configuration > Revisions, ready for rollback when someone fat-fingers a policy at 4pm.

Concept 5 — Use Case 3

IoC Detection & Automated Quarantine

The 03:04 chain. Four components are required: FortiGate, FortiAnalyzer, FortiClient EMS, FortiClient. Prerequisites: FortiClient connects to both EMS and the FortiGate; FortiClient and FortiGate both send logs to FortiAnalyzer; the FortiGate is configured with the EMS IP address and administrator login credentials.

1. FortiClient ──log (malicious site)──▶ FortiAnalyzer 2. FortiAnalyzer discovers IoC ─────────▶ notifies FortiGate 3. FortiGate verifies endpoint is a CONNECTED FortiClient endpoint ──────▶ notifies EMS 4. EMS locates the endpoint ────────────▶ quarantine message 5. Endpoint quarantines ITSELF, blocks all traffic, reports status ─────────────▶ FortiGate + EMS
The scribe is the detective. The FortiGate neither detects the compromise nor executes the quarantine. Detection lives where the memory lives (FAZ reads the logs as a body of evidence); enforcement lives on the endpoint (chain of command: EMS manages FortiClient, so EMS delivers the order).
CLI — trigger quarantine manually
diagnose endpoint fctems queue-complete-calls Q-<endpoint_ip_address>

Not supported on FortiClient (Linux).

Why not just block the IP on a firewall?

Because "you can only block what is inline." A firewall never sees same-L2-domain traffic — inside the broadcast domain, a compromised host reaches its neighbours without crossing any firewall. The endpoint is also mobile (a firewall rule stays in the rack; self-quarantine travels to the café), and a firewall blocks an IP while quarantine blocks the device — DHCP hands out a new address tomorrow. Quarantine at the source is the only containment that follows the compromise around.

Concept 6 — Preview

Security Rating: The Fabric Judges

With a fabric in place, FortiOS can audit it: the Security Rating runs checks across the fabric against Fortinet best practices, producing a score and prioritised, actionable findings — some auto-fixable. The fabric can now tell us when it is sick (High CPU), under attack (quarantine), and badly configured (rating). Watching, acting, judging. Common Mistakes

Traps Observed

  • "The FortiGate detected and blocked it" — Riley's trap. FAZ detects; the endpoint blocks itself.
  • "Root down = automation down" — Junior's trap. Push is at config time; leaves execute local copies.
  • Email-only alerting — an alert without evidence repeats the failure it alerts about.
  • Simultaneous actions — %%results%% arrives empty unless Sequential + delay let the script finish.
  • "A backup needs no trigger" — it needs the Schedule trigger; time is the trigger.
  • Answering the neighbouring question — the delay exists for pipeline dependency, not sampling strategy.
Senior Engineering Insights

What Seniors Take Away

  1. Push at config time, not trigger time. Ask this of every orchestration system you ever meet — it determines whether automation survives the author's death.
  2. Pair every notification with the diagnostics a human would have gathered.
  3. "You can only block what is inline." Enforcement placement determines containment scope; only the endpoint travels.
  4. Sequential + delay = pipeline. When one action consumes another's output, completion is a dependency.
  5. Trace autonomous actions by asking whose job each step is — memory→FAZ, connections→FGT, management→EMS, execution→endpoint.
Exam Tips

NSE7 Watchpoints

  • Root failure question: the answer is previously configured stitches keep running on leaves. Distractors claim otherwise.
  • Four required IoC-quarantine components: FGT + FAZ + EMS + FortiClient. FortiNDR / FortiManager / FortiSwitch are distractors.
  • Know the 5-step order and each actor — especially that FAZ discovers the IoC and the endpoint quarantines itself.
  • Default cpu-use-threshold = 90; know the CLI context.
  • %%results%% = output of the previous (CLI-script) action; requires Sequential.
  • Schedule frequencies (five) and the root's clock.
  • Linux FortiClient exclusion is a classic detail question.
Key Takeaways

If You Remember Five Things

  1. Stitch = trigger + action(s); authored on the root, pushed to leaves.
  2. Leaves execute local copies — root down ≠ automation down.
  3. Sequential + delay make %%results%% arrive full, not empty.
  4. FAZ finds the IoC; EMS carries the order; the endpoint arrests itself.
  5. You can only block what is inline — so containment lives on the endpoint.
