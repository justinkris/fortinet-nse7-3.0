SESSION 38 · TROUBLESHOOTING GUIDE

When Automation Misbehaves

The senior skill for stitches is bidirectional: explain why automation fired (archaeology) and why it didn't (dependency-chain analysis). Commands matter less than knowing whose job each step is.

Symptom Families

SymptomFirst hypothesis family
"Something acted at 3am and nobody knows why"A stitch fired — reconstruct trigger → action from logs; check Security Fabric > Automation on the root
High-CPU email arrived empty / half-filledActions not Sequential, or delay too short for the script to finish before %%results%% was consumed
Stitch never fires on a leaf FGTScope excludes that FGT; push never happened; or trigger threshold not reached (cpu-use-threshold)
Compromised host was detected but NOT quarantinedBreak in the 5-step chain — isolate which link (see decision tree)
Scheduled backup missingFrequency/menu mismatch, or root clock/time-system wrong — the ROOT's clock governs
Root FGT down, panic about automationNon-issue for EXISTING stitches (leaves execute local copies) — real issue: no NEW stitches can be authored/pushed

Questions to Ask Before Touching Anything

  1. What exactly happened, and when? (Timestamp bounds the log search.)
  2. Which device performed the final action? Work backwards from the actor, not forwards from a guess.
  3. Was this an existing stitch or a change? Existing stitches survive root failure; changes require the root.
  4. Whose job was each step? Memory→FAZ · connections→FGT · management→EMS · execution→endpoint.
  5. What is the dependency chain between actions? Any action consuming another's output needs Sequential + adequate delay.

Decision Tree — "Host compromised but not quarantined"

Compromise confirmed, no quarantine │ ├─ Is the endpoint FortiClient (Linux)? ── YES → unsupported. Dead end by design. │ ├─ Step 1: Did FortiClient log to FAZ? │ NO → FC↔FAZ logging broken. Fix logging first. │ ├─ Step 2: Did FAZ raise the IoC? │ NO → FAZ analysis/event-handler side. The scribe never │ turned detective — nothing downstream can fire. │ ├─ Step 3: Did FGT receive the notice AND verify the endpoint? │ NO → endpoint not a CONNECTED FortiClient endpoint, or │ FGT lacks the EMS IP / admin credentials it needs │ to forward the order. │ ├─ Step 4: Did EMS receive and dispatch the quarantine msg? │ NO → FGT→EMS auth/API problem, or EMS can't locate │ the endpoint (does EMS actually manage it?). │ └─ Step 5: Did the endpoint quarantine and report back? NO → FC↔EMS connectivity, or the endpoint went offline before the message landed. Manual override at any point: diagnose endpoint fctems queue-complete-calls Q-<endpoint_ip>

Expected Evidence at Each Link

  • FAZ: the FortiClient log entry for the malicious event, then the IoC event itself.
  • FGT: record of the FAZ notification and the outbound call to EMS.
  • EMS: the quarantine dispatch to the named endpoint.
  • Endpoint: the status change report back to FGT + EMS — the chain closes the loop; a missing status report is itself a symptom.

Common Dead Ends

  • Staring at firewall policies. The quarantine chain doesn't use them; "the FortiGate blocked it" is the intuitive wrong answer.
  • Restarting the root to "fix" leaf automation. Leaves run local copies; the root is only needed to author or change stitches.
  • Lengthening the email action's own settings when the empty-body cause is the missing delay before it.
  • Checking the leaf's clock for a missed schedule. The root's time system governs.

Likely Root Causes, Ranked

  1. Broken prerequisite plumbing (FC↔EMS, FC/FGT↔FAZ logging, FGT missing EMS credentials)
  2. Scope/push mismatch — the stitch was never sent to the FGT expected to run it
  3. Sequential/delay mis-set in multi-action stitches
  4. Threshold never crossed (cpu-use-threshold raised or default misunderstood)
  5. Platform exclusions (Linux FortiClient)

Exam Traps

Trap 1: "Root fails → stitches stop on downstream FGTs." False — config-time push.
Trap 2: "FortiGate detects the IoC." False — FortiAnalyzer discovers it.
Trap 3: "EMS blocks the endpoint's traffic." False — the endpoint quarantines itself.
Trap 4: Component lists swapping EMS for FortiManager/FortiNDR/FortiSwitch.
Trap 5: %%results%% questions that omit Sequential — the parameter is meaningless without it.
The senior habit: when the network acts alone, don't ask "which box did it" — ask "whose job is each step," then walk the chain link by link, verifying evidence at every handoff before changing anything.
