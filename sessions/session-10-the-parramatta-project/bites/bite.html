The Parramatta Project

The one idea

Judge a topology against the actual traffic pattern. LoFinance's trading feed lives at HQ → all critical traffic is branch↔HQ → hub-and-spoke's only weakness (spoke-to-spoke) lands where there is no traffic. That's why it wins — not the comparison table.

Every mesh tunnel = phase 1 + phase 2 + policies both ends + a tunnel slot. Branch models hold fewer tunnels than campus models — the smallest box caps a mesh design.

Hub-and-spoke's two costs (spoke-to-spoke only)

  • Latency — spoke→hub→spoke = double WAN traversal.
  • Hub crypto burden — separate SAs/keys per tunnel, so the hub must decrypt then re-encrypt every spoke-to-spoke packet. Hub = bottleneck + single point of failure.

Redundancy layers

Device (HA) → Link (dual ISP) → Tunnel: bind two tunnels into one interface with ipsec-aggregate (LACP for tunnels): set algorithm redundant.

Trap: per-packet round-robin over unequal-latency links → out-of-order delivery → jitter/glitches for real-time flows. Use redundant, or L3/L4 (hash pins each flow to one tunnel).

Phase 1 / Phase 2 / Selectors

  • Phase 1 = secure authenticated channel between the firewalls.
  • Phase 2 = what's carried; selectors = declared src↔dst subnet pairs; one phase 1, many phase 2s.
  • No matching selector → traffic never enters the tunnel. Declared pipe, not an open hose.
  • Your analogy: a trunk link, but over a tunnel — phase 1 ≈ trunk, selectors ≈ allowed VLANs (matching src/dst pairs).
  • Selectors don't replace policies — selectors gate entry, policies inspect. Two fences.

Exam-trap flash

"Spokes share tunnel keys" = always FALSE. Each tunnel is its own SA with its own keys — exactly why the hub decrypts/re-encrypts.
