TROUBLESHOOTING GUIDE · SESSION 10 · IPSEC TOPOLOGY-LEVEL FAULTS

When the Design Is the Fault

Tunnel config comes later. This guide is about faults whose root cause is the topology decision itself — the kind you diagnose with a map before you diagnose with a CLI.

The mindset

Session 10's core habit: ask where the traffic goes before you evaluate the device. A perfectly configured firewall in the wrong topology still produces slow, glitchy, or fragile networks. These faults show up as performance complaints, not error messages — which is exactly why juniors miss them.

Scenario 1 — "Branch-to-branch is slow, but branch-to-HQ is fine"

Symptom: Users at Spoke A report large transfers to Spoke B crawling; both sites reach HQ at full speed. All tunnels show up.

Questions to ask first

  • What topology is this? (If hub-and-spoke — the path is A→hub→B. Is that path expected?)
  • Was spoke-to-spoke traffic part of the original design assumptions, or has the business changed?
  • What is the hub's CPU doing during the transfer?

Verification

# On the hub — is it sweating crypto?
get system performance status        <- CPU, especially during transfers
diagnose vpn tunnel list             <- SAs up on BOTH spoke tunnels?
# Trace the actual path from Spoke A:
execute traceroute <spoke-B-host>    <- expect the hub as an intermediate hop
Likely root cause: nothing is broken. The topology imposes double WAN traversal plus decrypt/re-encrypt at the hub for every spoke-to-spoke packet. If the business now has real spoke-to-spoke traffic, the design needs revisiting (this is the exact problem ADVPN exists to solve — later in the course), not the config.
Dead end to avoid: tuning phase 2 proposals or MTU on the spoke tunnels. The tunnels are fine — the path is the cost.

Scenario 2 — "Real-time feed glitches, but both tunnels are up"

Symptom: Trading feed to Parramatta stutters/drops ticks. Both aggregate member tunnels are up. Bandwidth graphs look healthy.

Questions to ask first

  • What is the aggregate's load-balancing algorithm?
  • Do the two member tunnels ride different ISPs with different latencies?
  • Is the complaint about loss or about ordering/jitter? (Glitches with zero loss points at reordering.)

Verification

show system ipsec-aggregate          <- check "set algorithm"
diagnose vpn tunnel list             <- both members up, traffic counters on BOTH?
execute ping <peer-underlay-A>       <- latency per ISP path
execute ping <peer-underlay-B>       <- compare: unequal? there's your skew
Likely root cause: algorithm round-robin (per-packet) across unequal-latency links → out-of-order delivery → jitter. Fix: redundant if one link has headroom, or L3/L4 so each flow pins to one tunnel.
Exam trap shape: the distractors will blame phase 2 selectors, IKE version, or DPD. If both tunnels are up, counters increment on both, and the traffic is latency-sensitive — think ordering before you think negotiation.

Scenario 3 — "Some Parramatta machines can't reach an HQ subnet at all"

Symptom: Trading workstations reach VLAN 20 perfectly; office PCs can't reach it at all — not slow, nothing. Tunnel is up.

Questions to ask first

  • Which src↔dst subnet pairs are declared as phase 2 selectors?
  • Is the failing traffic's pair on that list — or was it never declared?
  • Is this a fault… or the design working exactly as intended?

Verification

diagnose vpn tunnel list             <- inspect selectors (src/dst proxy IDs) per phase 2
# Does the failing src/dst pair match ANY listed selector?
Likely root cause: the pair matches no selector, so the traffic never enters the tunnel — dropped pre-encryption. If the office PCs shouldn't reach VLAN 20: this is segmentation succeeding, close the ticket with a smile. If they legitimately need a new destination: add a phase 2 selector (and matching policy) — a design change, made deliberately.
Dead end to avoid: hunting for a deny policy or a routing fault. Selector-excluded traffic fails before both — the tunnel simply never accepts it.

Decision tree

Site-to-site complaint arrives
├─ Traffic works but is SLOW between two spokes → suspect topology path (Scenario 1) — trace the path, check hub CPU
├─ Traffic works but GLITCHES / jitters on real-time flows → suspect aggregation algorithm (Scenario 2) — check algorithm + per-ISP latency
├─ Specific src↔dst pair fails COMPLETELY, others fine, tunnel up → suspect selectors (Scenario 3) — read the phase 2 proxy IDs
└─ Tunnel itself won't come up → negotiation problem → Sessions 11–12 territory (IKE) — don't guess yet

Senior habits reinforced

  • Restate the business symptom, then draw the expected packet path before touching anything.
  • "Both tunnels up" is not "everything healthy" — up-ness says nothing about path choice, ordering, or selectors.
  • Distinguish slow (path/ordering) from dead (selectors/policy/routing) — they have different first questions.
  • Sometimes the correct fix is a design conversation, not a config change.
