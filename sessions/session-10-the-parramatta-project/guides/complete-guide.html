The Parramatta Project

IPsec topology design — hub-and-spoke vs full/partial mesh, and the criteria that actually decide.

Story Recap

Penny fired all three catchphrases in one breath — "How long? How much? Can we do it today?" — because Parramatta opens Monday and needs the HQ trading feed, securely. Junior taped three contradictory connection sketches to the whiteboard (one involving internet-facing RDP); Señor turned them face-down without a word and handed over the marker: "Two sites. One network. Draw it before you cable it." You designed LoFinance's first multi-site network — and set the pattern every future branch will copy.

Learning Objectives

  • Compare site-to-site, hub-and-spoke, dual hub-and-spoke, and full mesh IPsec topologies.
  • Quantify mesh scaling (n(n−1)/2) and explain why branch hardware caps mesh designs.
  • Identify hub-and-spoke's two spoke-to-spoke costs: latency and hub crypto burden.
  • Select a topology by matching its weaknesses against the actual traffic pattern.
  • Design layered redundancy: HA, dual ISP, and IPsec tunnel aggregation with the right algorithm.
  • Explain phase 1 vs phase 2, and how multiple phase 2 selectors extend segmentation across the WAN.

1 · Topology Names Earn Their Existence at Site Three

With exactly two sites, "site-to-site," "hub-and-spoke," and "mesh" are the same thing: one tunnel, one phase 1, one phase 2. The names only diverge when site three arrives — but the design pattern is chosen at site two, because whatever you build first becomes the template everything else copies.

The study guide's four reference topologies (p. 195): Site-to-Site · Hub-and-Spoke · Dual Hub-and-Spoke (redundant hubs) · Full Mesh (shown with ADVPN, covered later in the course).

Full mesh: every site tunnels to every other site → n(n−1)/2 tunnels total. You derived it yourself by summing the handshakes each new site adds:

Every mesh tunnel is a phase 1, a phase 2, and firewall policies on both ends — and it consumes a tunnel slot. The study guide is explicit: a branch model supports fewer tunnels than a campus model, so in a mesh your smallest, cheapest box becomes the ceiling of the whole design. Hub-and-spoke grows linearly: each new branch costs one tunnel, configured in two places.

3 · What Hub-and-Spoke Costs You

Spoke-to-spoke traffic (e.g. future Parramatta → Newcastle) pays twice:

  • Latency — the path is spoke → hub → spoke: two WAN traversals each way, forever.
  • Hub crypto burden — each tunnel is its own security association with its own keys. The hub must decrypt (spoke A's keys), route + policy, then re-encrypt (spoke B's keys) — full crypto twice per packet, per direction, for traffic that isn't even for the hub. As branches multiply, the hub becomes a bottleneck and a single point of failure: hub down = every branch loses every other branch.

4 · The Design Criterion That Decided It

Judge a topology against the actual traffic pattern, not in the abstract. LoFinance's critical traffic — the trading feed — lives at HQ. Every branch's important traffic is branch↔HQ, which takes the direct tunnel in a hub-and-spoke. The topology's weakness (spoke-to-spoke) lands exactly where LoFinance has no traffic. That is why hub-and-spoke wins here — not merely that it has fewer rows shaded red in a comparison table.

The sentence for Penny: "Hub-and-spoke, HQ as hub — everything the branches need lives at HQ anyway, so each branch gets a direct secure path to the feed, and each new branch costs one tunnel and one afternoon, not a redesign." That answers all three of her questions at once.

5 · Layered Redundancy & Tunnel Aggregation

Cipher's challenge: the feed rides one line. Redundancy is layered:

  • Device layer — HA clusters at each site (Season 1 material).
  • Link layer — dual ISPs, so one provider's outage isn't yours.
  • Tunnel layer — with two ISPs each side you can run two tunnels between the same firewall pair, then bind them into one virtual interface with IPsec tunnel aggregation — LACP for tunnels.
config system ipsec-aggregate
    edit "PARRA-AGG"
        set member "Tunnel-ISP-A" "Tunnel-ISP-B"
        set algorithm redundant
    next
end

Aggregation algorithms

AlgorithmBehaviourReordering risk?
redundantAll traffic on first active tunnel; others standbyNone
round-robinBalances per packetYES — avoid for latency-sensitive flows
L3Hash on Layer-3 header — flow sticks to one tunnelNo (within a flow)
L4Hash on Layer-4 header — flow sticks to one tunnelNo (within a flow)
weighted-round-robinRound-robin adjusted by link weightsYes (per-packet)
The reordering trap: per-packet balancing across unequal-latency paths (4ms vs 9ms) delivers packets out of order. TCP buffers and re-sequences (= jitter); UDP applications glitch. For the trading feed: redundant if one link has headroom; L3/L4 if you truly need both links' capacity; never per-packet round-robin.

6 · Phase 1, Phase 2, and Selectors

Phase 1 establishes the secure, authenticated channel between the two firewalls and negotiates keys. Phase 2 uses that channel to negotiate what the tunnel actually carries — including selectors: declared source-subnet ↔ destination-subnet pairs. One phase 1 can carry multiple phase 2 selectors. Traffic matching no selector never enters the tunnel — it is structurally excluded before encryption, not policed after.

Your analogy — keep it"A trunk link, but over a tunnel." One phase 1 ≈ the trunk; each phase 2 selector ≈ an allowed VLAN. Refinement: selectors match src↔dst subnet pairs — closer to a firewall policy's matching than to a VLAN allow-list.

Parramatta's selectors: Trading-subnet ↔ HQ VLAN 20, Office-subnet ↔ HQ file services. The receptionist's PC reaching the trading VLAN isn't forbidden by a policy someone remembered to write — it's impossible, because the pair was never declared. Junior's "garden hose" (tunnel = open pipe) is wrong: a tunnel is a declared pipe, not an open hose. And you still write firewall policies on the tunnel interfaces — selectors gate entry; policies inspect and log. Two fences, deliberately.

Common Mistakes & Exam Traps

  • "Spokes share tunnel keys" — never true. Each tunnel is its own SA with its own keys; that's precisely why the hub must decrypt/re-encrypt spoke-to-spoke traffic.
  • Confusing n−1 (hub-and-spoke) with n(n−1)/2 (mesh) tunnel counts.
  • Ignoring branch-model tunnel-capacity limits when scenario-questions mention entry-level hardware — it's a design criterion, not flavour text.
  • Assuming selectors replace firewall policies (they don't — two fences).
  • Recommending round-robin aggregation for real-time traffic.

Senior Engineering Insights

  • Ask where the traffic goes before you draw the lines.
  • The pattern is chosen at site two; site three just reveals whether you chose well.
  • Redundancy is layered — device, link, tunnel — and each layer answers a different failure.
  • Segmentation doesn't stop at the tunnel mouth; selectors carry it across the WAN.

Exam Questions (from the session)

Q1. 1 HQ + 7 branches, traffic almost exclusively branch↔HQ, entry-level branch FortiGates. Best topology?
B — Hub-and-spoke, HQ as hub. Two embedded criteria (traffic pattern + tunnel-capacity limits) both point the same way.
Q2. T/F: In hub-and-spoke, the hub forwards spoke-to-spoke traffic without decrypting, because spokes share tunnel keys.
False. Separate SAs, separate keys — the hub decrypts and re-encrypts.
Q3. Aggregate set to round-robin over two ISPs with different latencies; real-time feed glitches though both tunnels are up. Cause?
B — Per-packet load balancing causing out-of-order delivery. Fix: redundant or L3/L4.
