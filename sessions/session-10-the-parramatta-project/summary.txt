TECHNICAL CONCEPTS COVERED
--------------------------
1. TOPOLOGY NAMES ONLY DIFFERENTIATE AT 3+ SITES. With two sites, site-to-site =
   hub-and-spoke = mesh in practice. But the pattern is chosen at site two because it
   becomes the template. Student articulated this himself with one guided step.
2. FULL MESH TUNNEL COUNT = n(n-1)/2. Student calculated 3 sites=3, 5=10, 10=45 by hand
   (summed 4+3+2+1 etc. — reasoned, not memorised). Quadratic growth; each tunnel = phase 1
   + phase 2 + policies both ends + a tunnel slot; branch-model FortiGates support fewer
   tunnels than campus models, so the smallest box caps the design.
3. HUB-AND-SPOKE = n-1 tunnels, linear growth, but two costs for spoke-to-spoke traffic:
   (a) LATENCY — double WAN traversal (student got this immediately, framed identically to
   his own Session 9 VDOM chassis-binding reasoning — explicit callback made);
   (b) HUB CRYPTO BURDEN — hub must DECRYPT (spoke A's SA/keys) then RE-ENCRYPT (spoke B's
   SA/keys) every spoke-to-spoke packet both directions, because each tunnel has separate
   keys. Hub becomes bottleneck + single point of failure (hub dies = all inter-branch
   connectivity dies). Student got "resource intensive, scales with sites" himself; the
   named decrypt/re-encrypt mechanic was supplied after his reasoning.
4. DESIGN CRITERION THAT DECIDED IT: judge topology against the ACTUAL traffic pattern.
   LoFinance's critical traffic (trading feed) lives AT the hub — all traffic is
   branch<->HQ, which takes the DIRECT tunnel in hub-and-spoke. The topology's weakness
   (spoke-to-spoke penalty) lands where LoFinance has no traffic. STUDENT REACHED THIS
   HIMSELF from a single hint ("where does the trading feed actually live?") — the
   session's centrepiece reasoning moment.
5. LAYERED REDUNDANCY: student volunteered HA clusters (device layer, Season 1 recall) AND
   dual ISP links (link layer) unprompted when Cipher challenged the SPOF. Taught on top:
   IPSEC TUNNEL AGGREGATION — bind multiple tunnels into one virtual interface (explicit
   LACP analogy, which maps to the student's existing L2 knowledge):
     config system ipsec-aggregate / edit "PARRA-AGG" / set member "Tunnel-ISP-A"
     "Tunnel-ISP-B" / set algorithm redundant / next / end
   Algorithms: redundant (active/backup), round-robin (per-packet), L3, L4,
   weighted-round-robin.
6. PER-PACKET ROUND-ROBIN + UNEQUAL-LATENCY PATHS = OUT-OF-ORDER DELIVERY. Student named
   the reordering consequence himself in one step. Supplied on top: jitter framing, TCP
   re-sequencing cost, UDP glitching; L3/L4 hashing keeps each FLOW on one tunnel (flow
   pinning) = aggregate capacity without intra-flow reordering. Polished answer: redundant
   if one link has headroom; L3/L4 if capacity genuinely needed; NEVER per-packet
   round-robin for latency-sensitive flows.
7. PHASE 1 vs PHASE 2 (introduced at concept level only — deep config is Session 12):
   phase 1 = secure authenticated channel between the firewalls; phase 2 = negotiates what
   is carried, including SELECTORS (declared src-subnet<->dst-subnet pairs); ONE phase 1
   can carry MULTIPLE phase 2 selectors. Traffic matching no selector never enters the
   tunnel — dropped before encryption, structurally excluded, not policy-filtered after.
   Segmentation philosophy extended across the WAN. NOTE: firewall policies on tunnel
   interfaces are STILL required/expected on top of selectors ("two fences" framing —
   selectors gate entry, policies inspect and log).
