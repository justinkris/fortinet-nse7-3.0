The Client That Hears Nothing

How a senior engineer isolates a route-reflector fault when every session is Established and nothing is logged.

Name the drawer before you name the cause.
Most RR faults are misdiagnosed because the engineer investigates the device that is complaining instead of the device that is silent.
Orientation

Why RR Faults Are Hard

Route-reflector faults share a signature that defeats instinct: the broken device shows nothing wrong. The client's BGP session is Established. Its counters increment. Its logs are clean. It simply does not have a route, and nothing on it can explain the absence.

This is because the decision that broke you happened on a different router, in a drawer you have not looked in, and in the case of loop-prevention drops it happened silently by design.

The governing principle: absence of errors is not evidence of correctness. A route that was never advertised and a route that was silently discarded look identical from the receiving end.
Step 01

Symptoms You Will Actually See

Reported symptomWhat it usually means
"The new branch can't reach anything"Client not declared on the RR, or client peering with the wrong device
"Two branches can't reach each other but both reach HQ"Reflection not happening — RR is treating them as ordinary IBGP peers
"It worked before we decommissioned the old sessions"Full mesh was masking a missing route-reflector-client statement
"The route shows on the RR but not the client"RIB-out problem on the RR, or a silent drop at the client
"Routes appear and vanish"Multi-RR CLUSTER_LIST or cluster-ID mismatch
"Route is there but traffic fails"Wrong drawer entirely — next-hop resolution or policy
Step 02

Questions To Ask Before Typing

🧠Scope The Fault First
Every one of these narrows the search before a single command is run. Skipping them is how three-hour outages happen.
  • Which prefixes are affected — all of them, or only those originating at particular sites?
  • Is this one client or every client? One client points at that client's neighbour block on the RR. All clients points at the RR's role itself.
  • Did this ever work? If it worked yesterday, what changed — and was a full mesh quietly masking the fault?
  • Is the affected prefix originated by a client, by a non-client peer, or learned via EBGP? The three cases have different legal outcomes.
  • Is the failure symmetric? Client-to-HQ working but client-to-client failing is a reflection fault, not a peering fault.
  • Am I looking at a route that never arrived, or one that arrived and was not used?
💭
The Habit
Say the drawer out loud before you form a hypothesis: "the prefix is absent from the client's RIB-in, so this is an RR RIB-out question." Naming the drawer makes it impossible to skip reading the state.
Step 03

Commands, And What Good Looks Like

1. Is the session even up? (Rule it out fast, then stop trusting it)
# get router info bgp summary

Neighbor      V  AS  MsgRcvd  MsgSent  Up/Down    State/PfxRcd
10.99.80.1    4  65001   1204     1198  20:14:33          0

# State column shows a NUMBER = Established. PfxRcd 0 with an
# established session is the classic RR misconfiguration signature.
# A hostname in the State column (Idle/Active/Connect) is a
# DIFFERENT fault entirely - that is peering, not reflection.
2. THE decisive command — what is the RR actually saying?
# On the RR:
# get router info bgp neighbors 10.99.80.1 advertised-routes

# EXPECTED (healthy): the client's peers' prefixes appear here.
# FOUND (broken): empty, or only locally-originated prefixes.
#
# This reads the RR's RIB-OUT toward that client. If the RR is not
# saying it, no amount of investigation at the client will explain why.
3. Confirm from the other end
# On the client:
# get router info bgp neighbors 10.99.10.1 received-routes

# This is the client's RIB-IN from the RR. If (2) shows routes being
# advertised and (3) shows nothing received, you have a SILENT DROP -
# almost always ORIGINATOR_ID matching the client's own router-id.
4. Read the database and the status codes
# get router info bgp network

   Network          Next Hop        Metric  LocPrf  Weight  Path
*>i 10.99.70.0/24    10.99.10.1           0     100       0  i
*i  172.28.10.0/24   172.20.0.1           0     100       0  65300 i

# *  valid    >  best    i  internal
# *i without > on a SINGLE path = ineligible, not outvoted.
# Read the NEXT HOP column next.
5. Confirm the RR role is actually configured
# On the RR:
# show router bgp | grep -f route-reflector-client
# or:
# show router bgp

config neighbor
    edit "10.99.80.1"
        set remote-as 65001
        set next-hop-self enable
        set route-reflector-client enable     <-- MUST be present, HERE
    next
end

# Checking the CLIENT for this line is the commonest wasted hour.
# It will never be there. The client does not know it is a client.
Step 04

Decision Tree

Client is missing routes
Is the IBGP session Established?
├─ NO  ──▶ Peering fault, NOT a reflector fault.
│           Check reachability, remote-as, router-id,
│           update-source, ebgp-multihop. STOP HERE.
└─ YES ──▶ Is the prefix in the RR's Local RIB?
            (get router info bgp network, on the RR)
    ├─ NO  ──▶ The RR never had it. Go upstream:
    │           who should have originated it?
    │           Check THAT device's RIB-out.
    └─ YES ──▶ Is the RR advertising it to this client?
                (neighbors <client> advertised-routes)
        ├─ NO  ──▶ RIB-OUT GATE. Three candidates:
        │       1. route-reflector-client not set on the RR
        │          for this neighbour  ◀── MOST LIKELY
        │       2. prefix-list-out / route-map-out filtering it
        │       3. Route learned from a non-client and the
        │          target is also a non-client (legal refusal)
        └─ YES ──▶ Does the client show it in received-routes?
            ├─ NO  ──▶ SILENT DROP at the client.
            │       1. ORIGINATOR_ID = client's own router-id
            │       2. CLUSTER_LIST contains its cluster-id
            │       3. prefix-list-in / route-map-in on the client
            └─ YES ──▶ Present but not installed?
                    Read status codes. * without > on a
                    single path = INELIGIBLE.
                    ──▶ Check NEXT_HOP resolution.
                        get router info routing-table details <nh>
                        Fix: next-hop-self on the RR, or an
                        IGP/static route to the next hop.
Step 05

Common Dead Ends

Dead endWhy it wastes timeDo this instead
Searching the client for RR configurationIt is never there. The client's config is an ordinary neighbour statement by design.Read the RR's neighbour block for that client.
Trusting bgp summary because it looks healthySessions are Established throughout an RR misconfiguration. The command cannot see the fault.Use advertised-routes — it reads the drawer that is actually broken.
Hunting for a log entryORIGINATOR_ID and CLUSTER_LIST drops are silent by design. No log will ever appear.Compare advertised-routes on the RR against received-routes on the client. The gap is the drop.
Restarting the BGP process or clearing the sessionIt temporarily re-advertises, so the fault appears to fix itself and then returns. You have destroyed your evidence.Gather output first. Clear only to test a hypothesis you have already formed.
Adding routes back into a mesh "to see if that fixes it"It will fix it — and hide the real cause permanently.Fix the reflector. The mesh is what you are trying to eliminate.
Blaming next-hop-self for missing routesNext-hop problems produce routes that are present but unusable, never routes that are absent.Absent = advertisement. Present-but-unused = resolution. Different drawers.
Assuming a client will relay to another clientClients are mute. You will wait forever for something that is working as designed.Confirm both clients peer with the RR directly.

Step 07

Exam Traps

  • The "which device" trap. A scenario describes a client with no routes and asks what to configure on the client. The answer is that nothing is configured on the client — the fix is on the RR.
  • The healthy-summary trap. Output shows all sessions Established and the question asks whether BGP is functioning. Established says the conversation is happening, not that anything useful is being said.
  • The split-horizon wording trap. "Not advertised back to the peer that sent it" is offered alongside "not advertised to any other IBGP peer." Only the second is correct.
  • The scope trap. ORIGINATOR_ID and CLUSTER_LIST both appear as options. Read the stem for "originating router" versus "reflecting RR" or "between clusters."
  • The relay-direction trap. A route from a non-client IBGP peer. "Nobody" feels safe and is wrong — clients still receive it.
  • The plane trap. Full-mesh questions offering latency or bandwidth explanations. Peering count is a control-plane cost.
  • The compound trap. A scenario combining a reflection fault with an unresolvable next hop. Fixing one leaves the service broken and the second fault looks like the first fix failing.
Personalised technique: on any multi-select question, evaluate each option as an independent true/false statement. Do not select by shape, and do not stop as soon as you have found the number of options the stem requested — test every option, then count.
The Framework

Carry This Into The Next Incident

  1. Restate the business symptom. Who cannot reach what.
  2. Define the scope. One client or all clients; one prefix or all prefixes.
  3. Name the drawer. Absent from RIB-in? Present but not in the Local RIB? Believed but not in RIB-out? Say it out loud before hypothesising.
  4. Investigate the silent device, not the complaining one. In reflection faults, the answer is nearly always on the router that is not reporting a problem.
  5. Gather evidence before changing anything. Clearing a session destroys the state that would have told you why.
  6. Test one hypothesis at a time.
  7. Verify service recovery and root cause. Working again is not the same as understood.
