SESSION 59 · BITE · ROUTE REFLECTORS & THE RIBs

Only The Centre Is Allowed To Speak

Five minutes on why the IBGP mesh existed, why it collapsed, and what had to be invented to make the collapse safe.

"I ran out of board." — Junior, 08:41
Ninety minutes drawing a correct full mesh for nine sites. Nothing he drew was wrong. The design was.
01 · The Problem

Four People and a Printer

Adelaide needs a firewall. Four staff, one printer. The quoted change window is eleven hours, touching every firewall LoFinance owns.

Why? Because IBGP requires a full mesh: n(n−1)/2 sessions. Eight sites = 28. Nine = 36. Adding the ninth site adds eight new sessions and requires configuring every existing router.

The cause is not a limitation — it is a safety rule. IBGP split horizon: a prefix learned from an IBGP peer is never advertised to any other IBGP peer. Nobody may relay. So everybody must hear everything first-hand.
Analogy
Why the rule is worth its price
Inside an AS, AS_PATH is never prepended — so the loop check is structurally blind. Split horizon replaces it: a route can take exactly one IBGP hop, ever. You cannot build a circle out of segments of length one. Loops become impossible, not merely detected.

Two gates guard the first boundary. The IN filter (prefix-list-in, route-map-in) — do I even want this? And validity plus best-path — does the NEXT_HOP resolve, and if several peers sent the same prefix, which one wins?

One gate guards the second. And it is where every policy tool you have ever met actually lives.

03 · The Fix

Breaking The Rule On Purpose

A route reflector is not a new mechanism. It is a relaxation of split horizon. The RR takes routes from one IBGP peer and advertises them to another — the exact thing the rule forbids.

An RR and its clients form a cluster. Clients peer only with the RR. The RR peers with other RRs and border routers. Junior's grey smear becomes a star: 9 sessions, not 36.

On the RR (HQ) — and nowhere else
config router bgp
    config neighbor
        edit "10.99.60.1"                    <-- the CLIENT's address
            set remote-as 65001
            set next-hop-self enable
            set route-reflector-client enable
        next
    end
end
Read the shape of the command. Everything inside config neighbor / edit <ip> describes the peer at that address, not the box you are typing on. So this line declares "that neighbour is a client of mine" — which only the RR can say. The client's own config is an ordinary IBGP neighbour statement. It has no idea it is a client.
The business sentence: adding Adelaide is now one neighbour block on HQ and one on Adelaide. Two routers touched, not nine. Eleven hours becomes twenty minutes.
04 · The Price

What Replaced Impossibility

Delete the rule that made loops impossible and you must install something that detects them instead.

AttributeSet byTestScope
ORIGINATOR_IDRR, on reflectionRouter-ID of the client that injected it. See your own → drop, silently.Router
CLUSTER_LISTEach reflecting RRCluster-ID prepended. See your own → drop.Cluster
Both are AS_PATH's idea, scaled down. AS_PATH asks "is my AS in it?" — CLUSTER_LIST asks "is my cluster in it?" — ORIGINATOR_ID asks "is that me?" Same question, three scales.
05 · The Relay Rules

Downward Always. Sideways Never.

Route learned fromReflected toWhy
A clientEveryone — other clients, non-client IBGP peers, EBGP peersThe client is mute. If the RR does not relay, nobody hears it at all.
A non-client IBGP peerClients onlyClients peer with nobody else — without this the cluster is a sealed box. But other non-clients are ordinary IBGP peers, so the original rule still binds.
An EBGP peerEveryoneIt was never an IBGP-learned route. Split horizon never applied.

"The reflector speaks freely to its own. To strangers, it only repeats what its own told it."
06 · Five Commands

What To Type When It Breaks

Verification
get router info bgp network                                  # the database — "see"
get router info routing-table bgp                            # installed — "use"
get router info bgp neighbors <client> advertised-routes      # the RR's RIB-out
get router info bgp neighbors <rr> received-routes            # the client's RIB-in
get router info bgp summary                                  # looks PERFECT during RR misconfig
The trap, carried from Session 58: session Established, prefixes present, no logs, no alarms, service broken. Absence of errors is not evidence of correctness. An ORIGINATOR_ID drop is completely silent.
