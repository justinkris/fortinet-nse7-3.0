Troubleshooting Silence

How a senior engineer diagnoses path preference and origination faults that produce no logs.

The fault is rarely a fault. It is usually a silence.

Three of this session's four problems produce no log, no error and no alarm. Troubleshooting them is a discipline of reading state, not of hunting messages.

Before you type anything

Questions to ask first

  • Is anything actually broken? “Slow” and “unreachable” are completely different investigations. A path that is merely suboptimal has no fault in it at all.
  • Which direction? Traffic to a prefix, or advertisement of a prefix? These are opposite ends of the filing cabinet.
  • Who else is affected? One site or the whole AS? That single fact separates a weight/local-pref question from a filter question.
  • What changed, and when? BGP faults are overwhelmingly configuration events, not failures.
  • Name the drawer. Before any hypothesis: is this RIB-in, Local RIB, RIB-out, or the routing table? Say it out loud. A diagnosis that does not begin with a drawer is a guess wearing a lab coat.
The discipline that matters most here: read the state you were given before retrieving the fault you remember. The symptom that looks like next-hop resolution is very often something else that produces the same silence.
Symptom A

“Traffic is going the scenic route”

Presentation

Latency far above expectation. Traceroute shows a geographically absurd path. Everything works — it just works badly. No alarms anywhere.

Step 1 — is BGP even choosing?
get router info bgp network 10.44.0.0/16

BGP routing table entry for 10.44.0.0/16
Paths: (2 available, best #2)          ← THE WHOLE DIAGNOSIS IS IN THIS LINE
  65300 65400
    172.20.0.1 from 172.20.0.1 (172.20.0.1)
      Origin IGP, localpref 100, valid, external
  65500 65400
    172.30.0.1 from 172.30.0.1 (172.30.0.1)
      Origin IGP, localpref 100, valid, external, best

Read the header line before anything else. “2 available” means a competition happened. “1 available” means there was no competition — and that is a completely different investigation (the other path was never received, or was filtered inbound, or is invalid).

Step 2 — verify TWO things after the change
get router info bgp network 10.44.0.0/16

  ✓ the attribute changed        →  "localpref 300"
  ✓ the word "best" MOVED        →  "best #2" → "best #1"

Both. An attribute that changed without moving "best" means you
influenced a line BELOW the line that was actually deciding.
Dead end — checking the session. get router info bgp summary will look perfect throughout this entire fault. The neighbours are Established, prefixes are being received, counters are incrementing. Session health tells you nothing about path preference. Do not spend twenty minutes here.
Dead end — blaming the underlay. Latency of 340ms across the Pacific is not a circuit problem when the circuit is behaving exactly as a Pacific circuit should. The question is not “why is this path slow” but “why is this path chosen.”
Symptom B

“Our prefix isn't reaching them”

Presentation

A partner or peer says they cannot see one of your subnets. Your side shows no errors. The subnet works perfectly internally. Reachability is fine; you are simply not saying it.

COVERED IS NOT MATCHED. Prefix and mask, exactly, and ACTIVE.
FortiGate advertises nothing and logs nothing.
EvidenceRoot causeFix
Prefix absent from bgp network entirelyNever originated, or import check failedAdd matching static route (narrow) — not the global disable

Dead end — adding a second permit rule to the wrong list. If a prefix list is applied outbound and you edit the inbound one, everything will look busier and nothing will change. Confirm which parameter is applied on which neighbour before editing anything.
Dead end — assuming a filter because the symptom is silence. Origination failure and outbound filtering produce identical symptoms from the peer's side. The difference is visible only in your database: is the prefix there at all?
Symptom C

“I fixed one thing and lost everything else”

Presentation

A targeted change — a route-map to set local preference, or a prefix list to add one advertisement — is applied, and a large amount of unrelated routing disappears immediately.

This is almost always the implicit deny, and it is the most predictable self-inflicted outage in FortiOS routing.

The pattern, in both constructs
ROUTE MAP with one match rule       →  everything not matched is DENIED
PREFIX LIST with one permit rule    →  everything not matched is DENIED
ACCESS LIST                         →  same family trait

TERMINATORS:
  route-map:    edit 2 / set action permit
  prefix-list:  edit 2 / set prefix 0.0.0.0 0.0.0.0 / set le 32 / set action permit
The habit that prevents it: before applying any filter, ask not “what does this allow?” but “what does this silently kill?” Then count the things it kills. If the number is larger than one, you have written the wrong filter.
Method

The workflow, compressed

  1.  Restate the business symptom.        "London is 340ms instead of 90ms."
  2.  Slow or broken? To or from?          Direction decides which gate.
  3.  NAME THE DRAWER.                     RIB-in / Local RIB / RIB-out / routing table.
  4.  Gather evidence BEFORE changing.     get router info bgp network <prefix>
  5.  Read the header line.                "N available, best #M"
  6.  Find the FIRST differing line.       That line, and only that line, decided it.
  7.  One hypothesis, one change.          Fix ABOVE the losing line.
  8.  Verify TWO things.                   Attribute changed AND "best" moved.
  9.  Verify the BUSINESS symptom.         Riley refreshes. 94ms.
 10.  Ask what else the change killed.     Every filter denies by default.

Exam traps hidden in these symptoms

Stem saysWhat they want you to doWhat is actually true
“The AS path is shorter via ISP-A”Conclude ISP-A winsOnly if lines 1–3 are equal. Weight and local pref are above AS path.
“All attributes are identical”Reach for MED or IGP metricLowest router ID, unless ECMP is enabled — then up to 10 routes.
“Every router in the AS must prefer…”Set weight (it is first on the list!)LOCAL_PREF. Weight is never advertised, not even internally.
“The subnet is reachable and the session is up”Look for a session faultLook at network-import-check: exact prefix and mask in the routing table.
“A prefix list with a single permit entry”Assume the rest passes unchangedThe rest is denied.
“Which tool can change the attribute?”Pick prefix list (it is mentioned in the stem)Only a route map can set attributes.

Mental note
Absence of errors is not evidence of correctness. Three of the four faults in this session log nothing at all. You will find them by reading state, never by reading messages.
