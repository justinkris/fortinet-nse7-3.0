TECHNICAL CONCEPTS COVERED (pp. 342-348)
==================================================================
1. THE TIE-BREAKER LIST AS AN ARGUMENT CONDUCTED IN ORDER (p.344). Ten lines. BGP compares two
   paths one line at a time, top to bottom, and THE FIRST LINE WHERE THEY DIFFER ENDS THE
   ARGUMENT. Nothing below that line is ever read. Full list taught:
     1 highest weight · 2 highest local pref · 3 originated locally · 4 shortest AS path ·
     5 lowest origin type · 6 lowest MED · 7 lowest IGP metric to next hop ·
     8 prefer EBGP over IBGP · 9 ECMP up to 10 routes · 10 lowest router ID.
2. "ARGUE ABOVE THE LINE YOU ARE LOSING ON." The senior-engineering core of the episode.
   Javier invited a contest on line 4 (AS path length) where we tie. We declined and competed
   on line 2. CHOOSING THE BATTLEGROUND IS THE SKILL; the config is trivial afterwards.
3. FALLING TO LINE 10 IS NOT A FAULT — IT IS AN ABSENCE OF POLICY. Direct extension of S56
   ("a coin flip decided by someone else"). Student produced this himself, unprompted.
4. BEST-PATH SELECTION IS A RIB-in → LOCAL RIB EVENT. Both candidates queue in RIB-in; only the
   winner enters the Local RIB; the loser stays in RIB-in forever waiting for the winner to die.
   "RIB-in = the candidates. Local RIB = the winner. The tie-breaker list is the doorman."
5. ORDER OF OPERATIONS AT THE FIRST GATE: inbound route-map runs BEFORE best-path selection.
   "POLICY RUNS FIRST. THE DOORMAN READS THE RESULT." If the list ran first you would be editing
   the loser's paperwork after the verdict. This is WHY attribute manipulation works at all.
6. WEIGHT vs LOCAL_PREF BY SCOPE — the selection criterion:
     weight     = this FortiGate only. Never advertised anywhere. A PRIVATE OPINION.
     LOCAL_PREF = entire local AS, carried in IBGP updates, STRIPPED at the AS edge.
                  A COMPANY POLICY.
   Because MEL/BNE/PER/ADL must all inherit the preference, LOCAL_PREF is the only viable tool.
   Weight would require touching every firewall — "you run out of board."
7. BGP ATTRIBUTE CATEGORIES (p.343) — TAUGHT LATE, AS REMEDIATION, AFTER THE EXAM. The student
   flagged twice that he did not understand these and missed Q3 and Q9 on them. Taught as TWO
   SEQUENTIAL QUESTIONS:
     Q1: must every BGP router understand it?  well-known : optional
     Q2 (if well-known): must it be on every route?  mandatory : discretionary
     Q2 (if optional): may an ignorant router pass it on?  transitive : non-transitive
   Table: ORIGIN / AS_PATH / NEXT_HOP = well-known mandatory. LOCAL_PREF / ATOMIC_AGGREGATE =
   well-known discretionary. AGGREGATOR / COMMUNITY = optional transitive. MULTI_EXIT_DISC =
   optional NON-transitive.
   THE HOOK THAT LANDED: "ASK HOW FAR IT TRAVELS AND THE CATEGORY ANSWERS ITSELF."
   LOCAL_PREF is stripped at the AS edge ⇒ cannot be mandatory ⇒ discretionary.
   MED is advice for ONE neighbour ⇒ must not be passed on ⇒ non-transitive.
   COMMUNITY exists to survive the boundary ⇒ transitive.
   Memory line: "COMMUNITY travels. MED must not. LOCAL_PREF never leaves home."
8. THE NETWORK COMMAND (p.347) — GENUINE NEW GROUND, as predicted. Student knew only
   redistribution (S57). Framed as BUCKET vs TAP (J.R.'s phrasing, in-story):
     redistribute = pour a whole CATEGORY of routes into BGP, then spend the afternoon
                    taking back what you didn't mean to say
     config network = originate EXACTLY ONE named prefix. Deliberate, auditable.
   It is NOT a filter on redistribution — it is a second, INDEPENDENT origination method.
9. network-import-check (p.347) — THE EXAM QUESTION AND THE EPISODE'S SECOND FAULT.
   Default ENABLED. The prefix is advertised ONLY IF an active route with an EXACT matching
   destination subnet exists IN THE ROUTING TABLE (not the Local RIB — student got this half
   wrong and it was corrected: "network-import-check consults FortiGate's routing table, not
   BGP's head. IMPORT — from outside BGP, into BGP.")
   THE TRAP: routing table has 10.55.0.0/16 connected; network statement says 10.55.10.0/24.
   PREFIX AND MASK MUST MATCH EXACTLY. A /16 does not satisfy a /24. FortiGate says NOTHING —
   no log, no error, no alarm. Reachability is fine; origination is silent.
   Two fixes offered and the student had to choose:
     A) add a matching static route (NARROW, documents intent)
     B) set network-import-check disable (GLOBAL, silent, removes the net for ALL statements)
   Student chose A for the right reason (spotted it is a GLOBAL switch). Chosen for LoFinance.
   Keeper line: "The check is not in your way. The check is the difference between advertising
   a subnet and advertising a rumour."
10. PREFIX LISTS (p.348) — per-neighbour, prefix-list-in / prefix-list-out.
    THE IMPLICIT DENY ALL: anything not matching an entry is DENIED. Taught as a FAMILY

TROUBLESHOOTING APPROACH (as taught)
==================================================================
"TRAFFIC TAKES A STRANGE PATH"
1. Is BGP even choosing? Two paths in the database or one?  → get router info bgp network <p>
2. Read "Paths: (N available, best #M)" — the whole diagnosis is in that line.
3. Compare the two paths TOP-DOWN the tie-breaker list. Find the FIRST line where they differ.
   That line and ONLY that line decided it.
4. If they differ nowhere → you fell to line 10 (router ID). NOT A FAULT. ABSENCE OF POLICY.
5. Fix ABOVE the line you were losing on. Never argue on the opponent's line.
6. Verify: localpref/weight/AS-path changed in the output AND "best" moved.

"PREFIX NOT ADVERTISED AT ALL"
1. Is it originated? config network present? redistribute enabled?
2. If network command → get router info routing-table all → EXACT prefix AND mask, ACTIVE.
3. Originated but silent → prefix-list-out / route-map-out on THAT neighbour → implicit deny.

THE MULTI-SELECT METHOD — RESOLVED AFTER THREE SESSIONS
==================================================================
FINALLY ANSWERED. He reported: "Eliminate, count what feels right."
Diagnosis given in-session: elimination is a RELATIVE test — it ranks options against each
other and answers "which looks most right," which fails precisely when a distractor is TRUE IN
GENERAL but not true OF THE STATE IN THE STEM. That is his documented failure mode (strongest
memories = most dangerous distractors). "Stop when you hit the number" also means the last
option often goes unread — which is the S14 under-select and S15 over-select in both directions.
REPLACEMENT METHOD PRESCRIBED (enforce from S61 onward):
   1. Read the stem twice and WRITE DOWN THE STATE. (This is "name the drawer," exam form.)
   2. Score EVERY option independently TRUE/FALSE against that state. Not against each other.
   3. Count TRUEs LAST, then compare to the number requested.
      Too few → you rejected something for unfamiliarity, not falsity. Re-read.
      Too many → one is true in general but not of this state. Drop that one.
   "The count is a CHECKSUM, not a stopping condition."
   J.R.'s compression: "SCORE FIRST, COUNT LAST."
