SESSION 30 · TROUBLESHOOTING · THINKING NOT MEMORISING

When the preview explodes

How a senior engineer reasons about a FortiManager install that reports far more changes than were authored.

The goal is not to find a command. It is to know which direction the truth is moving.
Symptom

What you're looking at

You (or Junior) authored a small, documented change in the policy layer. You open the Install Wizard, reach the installation preview, and it reports many more changes than you made — policies being deleted, objects renamed, routes you don't recognise. Someone's finger is hovering over the green Install button.

Do not click Install yet. The preview is the last safe moment. Past this point, an unknown change on the box is gone.
Questions to Ask (in order)

Reason before you touch anything

  1. What two things is this diff comparing? Install Policy Package compares the policy package (source) against the device DB + box (target).
  2. How many changes did I actually author? Establish the honest baseline — the delta between "what I did" and "what the preview shows" is the mystery to explain.
  3. Who else can touch this box? Two origins for phantom changes: un-pushed edits in the device DB, and undocumented direct edits on the live box that were never retrieved.
  4. Is this a payment / production / regulated box? Raises the cost of a blind overwrite from "annoying" to "career-ending."
  5. Which direction resolves this safely? Pull (observe), never push (overwrite), until you know what's on the box.

Common Dead Ends

Where people waste hours (or break prod)

  • Clicking Install to "just push it clean." Blind overwrite — destroys the undocumented box config you were about to investigate.
  • Retrieving mid-incident onto a broken box. Captures the fault into the DB and overwrites last-known-good. (S29 lesson.)
  • Importing over the existing policy package. Same-name object conflicts; import into a separate package to audit safely.
  • Trusting revision history to show direct edits. It only records device-DB changes. Direct box edits are invisible.
  • Reverting and walking away. Revert stages to the DB; nothing reaches the box until you install.
Likely Root Causes

What it usually is

  • Undocumented direct-CLI/GUI edits on the managed box (emergency fixes never retrieved) — the classic cause of a huge diff.
  • Un-pushed device-DB changes from another admin — real, documented, but never installed.
  • A stale policy package that never imported the box's actual config after onboarding.
Exam Traps

What the NSE7 will dress up

  • "Install prompts you to resolve the conflict" — false. Install is a blind overwrite; the unknown change is lost.
  • "Retrieve updates the policy package" — false. Retrieve stops at the device DB; import is the bridge.
  • "Reinstall skips the preview" — false. It skips the wizard; the preview stays.
  • "Revert puts the config back on the FortiGate" — false. Revert touches only the device DB.
  • "Global auto-install shows a preview" — false. It bypasses the preview.
