TECHNICAL CONCEPTS COVERED
==================================================================
THE FOUR VERBS (as DIRECTIONS across three boxes: Policy Layer → Device DB → live FortiGate)

  PUSH (down, DANGEROUS — overwrites):
    INSTALL POLICY PACKAGE = policy pkg → Device DB AND → live FortiGate (two-hop).
       Runs the full step-by-step WIZARD + installation preview (can cancel).
    REINSTALL POLICY = device+package already selected, so it BYPASSES THE WIZARD but
       STILL SHOWS the installation preview. Also applies package settings + any
       modifications directly to FortiGate in the device DB.
    INSTALL DEVICE SETTINGS (only) = device layer → box, WITH preview.
    QUICK INSTALL = device layer → box, NO preview, immediate (ZTP / confident).
    REVERT = select an earlier revision from history → rewrites the DEVICE DB ONLY.
       NOT live on the box until an INSTALL. Policy layer NOT synced until an IMPORT.

  PULL (up, SAFE — observes):
    RETRIEVE = live FortiGate → Device DB, saves a new revision, does NOT touch policy pkg.
    AUTO-RETRIEVE = fires when a device is first added; captures direct box changes into
       revision history; still does NOT update the policy package.
    IMPORT = Device DB revision → POLICY LAYER (policy package). The bridge that gets a
       retrieved/box change INTO the policy package. RISK: object NAME CONFLICTS in the
       policy layer (same-name objects).

  KEY RULE: Install is a BLIND OVERWRITE. It does not merge, does not warn about unknown
  box changes; anything on the target not in the source is DESTROYED.

RECONCILE-A-DRIFTED-BOX SEQUENCE (the 214 fix):
  1. RETRIEVE  box → Device DB (DB now matches box; new revision)
  2. IMPORT    Device DB → Policy Layer, into a SEPARATE package (audit copy)
  3. AUDIT     the phantom changes now have names — keep legitimate, remove drift
  4. RE-APPLY  Junior's 3 documented changes on the reconciled package
  5. INSTALL   with preview; diff now = the 3 intended changes
  NOTE: after retrieve+import the diff to the box = 0 (all layers agree on box reality),
  which means Junior's un-pushed policy-layer edits are FLATTENED by the pull → he must
  re-apply them. (Student reasoned this correctly.)

CIPHER'S ROLLBACK (revert is a trap that looks like a finish line):
  1. REVERT   pick old revision → Device DB only
  2. INSTALL  Device DB → box (NOW it's actually live)
  3. IMPORT   Device DB → policy layer (if you want the policy package to agree)
  "Revert stages. Install delivers. Import reconciles."

GLOBAL POLICY PACKAGES (Penny's six rules):
  Global ADOM sits ABOVE local ADOMs (LoFinance, Vaultline). Global objects prefixed 'g'
  (reserved). Author the global policy package ONCE.
  ASSIGN  = global ADOM → local ADOM policy layer; flags the local package MODIFIED (staged).
  Then INSTALL/REINSTALL = local policy pkg → device DB → box.
  SHORTCUT "Automatically install policies to ADOM devices" = global ADOM installs to THREE
  targets in one move (local policy pkg + device DB + box) but BYPASSES the installation
  preview. On a fleet with a known-drifted box, the shortcut = green button with the safety
  removed.
  Business value: consistency becomes a STRUCTURAL guarantee, not a hope. Cipher can CERTIFY
  all firewalls are identical by design (authored once), not by eyeballing 40 packages.

DELIVERY PLAN THAT SATISFIES BOTH PENNY AND CIPHER:
  Assign to both ADOMs (fast) → reconcile the known-drifted box first (retrieve/import/audit)
  → install WITH PREVIEW per package. Clean boxes preview clean and install in seconds; only
  the drifted box is slow, and it SHOULD be. Full speed everywhere drift doesn't live.

TROUBLESHOOTING APPROACH TAUGHT ("install preview shows way more than I changed")
==================================================================
1) A large diff = a DESYNC (three copies of truth drifted), not a disaster.
2) Name the source/target of the diff (Install Policy Package = policy → DB → box).
3) Suspect UNDOCUMENTED DIRECT-BOX EDITS (invisible in revision history — S29 ghost) and/or
   un-pushed device-DB changes.
4) PULL BEFORE YOU PUSH: retrieve → import into a SEPARATE package → audit the phantoms.
5) Only then install. Never click green on a mystery — install is a blind overwrite.

TEASER FOR SESSION 31 — "One Name to Rule Them"
==================================================================
Penny's global rules are pushing beautifully — until they hit reality: the six baseline rules
reference an interface called "port3" on some boxes and "wan1" on others, "LAN" here and
"internal" there. A rule authored ONCE cannot name an interface that is called something
different on every box. Cipher: "You promised me one baseline. Now half the installs are
failing on interface not found." Enter NORMALIZED INTERFACES — abstract interface names that
map, PER-PLATFORM and PER-DEVICE, to the real interfaces underneath. Zones, VLANs, SSIDs as
policy-layer normalized objects (recall from p.34: VPN/AP/FortiSwitch Manager set up
normalized interfaces). Study Guide 02 Central Management, ~p.34 + normalized-interface pages.
Blueprint 2.1. Señor's seeded line: "Yesterday you learned to author a rule once. Today you
learn that a rule cannot point at a name that means something different on every machine —
so we give it a name that means the same thing everywhere, and let the machine translate."
