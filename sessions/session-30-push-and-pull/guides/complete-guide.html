Push and Pull

Retrieve · Import · Install · Revert — the grammar of moving truth around a FortiManager, and global policy packages that let you author once.

A big install preview is not a fault. It is a confession.

214 changes when you authored three means three copies of the truth have drifted apart. The senior move is to pull the truth up and read it — before you push anything down.

Session Overview

What this session is really about

Everything in FortiManager central management is a matter of direction. Configuration lives in three places, and every operation you run is just moving truth between them:

The three copies of truth
[ POLICY LAYER ]  →  [ DEVICE DB ]  →  [ LIVE FORTIGATE ]
   intent             FMG's copy         the actual box
   (policy pkgs)      (per-device)       (running config)

Once you see the pipeline, the four verbs stop being vocabulary to memorise and become directions to reason about. Push moves down (dangerous — it overwrites). Pull moves up (safe — it observes).

🧭
Mental Note
Push overwrites. Pull observes. When you don't know what's on the box — pull first, look, then push.
Story Recap

The 214-line confession

Junior — reformed after the VL-BRANCH-2 haunting — did everything right. Ticket LF-4471: give Vaultline finance access to a new reporting server. He edited the policy layer like a civilised person: one address object, one new policy, one modified policy. Three documented changes. Then he opened the Install Wizard, hit the installation preview, and it reported 214 changes.

Cipher refused to let anyone click the green Install button — the box carries Vaultline's payment traffic. Penny, from her doorway, added a second job: the same six baseline rules on every firewall in both companies by Friday. Author once. Señor arrived, delighted: "The machine is not lying to you. The machine is confessing."

The diagnosis: Junior authored 3 changes; the preview showed 214. So 211 changes exist that Junior never authored in the policy package. They came from two places — un-pushed edits sitting in the device DB, and (the dangerous one) undocumented direct edits made on the live box that were never retrieved. Last week's ghost: direct box edits write zero revisions, so they're invisible until an install trips over them.

Concept 1 — The Push Verbs

Pushing truth down the pipeline

Push operations move configuration down toward the box. They are powerful and dangerous, because a push makes the target match the source — and anything on the target that isn't in the source is destroyed. No merge. No negotiation. No conflict prompt.

VerbDirectionPreview?Notes
Install Policy Packagepolicy pkg → device DB → boxWizard + previewTwo-hop. Full step-by-step wizard; can cancel at preview.
Reinstall Policypolicy pkg → device DB → boxNo wizard, still previewDevice + package already selected, so it skips the wizard — but the preview still appears. Also applies package settings + any modifications to the device DB.
Install Device Settings (only)device layer → boxPreviewPushes only device-layer config (Network/VPN/System/Log).
Quick Installdevice layer → boxNo previewImmediate. For ZTP or when you're confident.
Revertrevision history → device DB only—Selects an old revision. NOT live until an Install. Policy layer needs an Import to sync.
The blind-overwrite rule: If someone made an undocumented direct edit on a managed box and it was never retrieved, an unrelated Install Policy Package will silently destroy it. The install's job is to make the box match the source; it neither knows nor cares that the box change existed.
Analogy
Reinstall vs Install is the difference between a full airport check-in (the wizard walks you through every field) and using the self-service kiosk with your boarding pass already loaded. The kiosk is faster — but it still shows you the boarding pass before you commit. Reinstall skips the wizard, not the preview.
Concept 2 — The Pull Verbs

Pulling truth up the pipeline

Pull operations move configuration up, away from the box. They are safe, because observing the box's reality never destroys anything on it. This is why, when you don't yet know what's on a box, you pull first.

The two-pull mirror
[ POLICY LAYER ]  ◄── IMPORT ──  [ DEVICE DB ]  ◄── RETRIEVE ──  [ BOX ]
                    (2nd hop)                      (1st hop)
VerbDirectionUpdates policy pkg?Notes
Retrievebox → device DBNoSaves a new revision. Captures box reality into the DB — but stops there.
Auto-retrievebox → device DBNoFires when a device is first added; captures direct box changes into revision history. Still needs an import to reach the policy package.
Importdevice DB revision → policy layerYes — this is the bridgeLifts a revision up into a policy package (can rename). Risk: same-name object conflicts in the policy layer.
🌉
Mental Note
Retrieve gets the box into the device DB. Import is the second pull that gets it into the policy package. Retrieve alone never touches the policy layer.
Concept 3 — Reconciling a Drifted Box

How to make 214 honest

You don't fix a mystery diff by pushing. You pull the box's truth up, read it, decide what's legitimate, and only then push a clean, intended package down.

The reconcile-before-push sequence
1. RETRIEVE   box → Device DB          (DB now matches box; new revision)
2. IMPORT     Device DB → Policy Layer  (into a SEPARATE package — an audit copy)
3. AUDIT      the 211 phantom changes now have names.
              Per change: legitimate → keep.   drift/garbage → mark for removal.
4. RE-APPLY   Junior's 3 documented changes on the reconciled package
5. INSTALL    with preview → diff now = the 3 intended changes → click green KNOWING
Watch the number: immediately after step 2, the diff between the policy package and the box is 0 — every layer now agrees on the box's present reality. But that means any policy-layer edits that were never pushed (Junior's three) get flattened by the pull. He must re-apply them in step 4. A pull carrying the box's truth upward buries un-pushed policy work.
Concept 4 — Revert Is Not the Finish Line

Revert stages. Install delivers. Import reconciles.

When Cipher asks for a known-good config from three revisions back, revert is only the first of three verbs. Revert rewrites the device DB only — the live box is untouched, and the policy layer is untouched.

Cipher's clean rollback
1. REVERT    pick old revision → rewrites Device DB only
2. INSTALL   Device DB → live FortiGate   (NOW it is actually on the box)
3. IMPORT    Device DB → policy layer      (if the policy package must agree too)
The trap: engineers who name "revert" correctly still ship a device DB full of good config sitting on a box that never received it — then swear the tool is broken. Revert without a follow-up install changes nothing on the wire.
Concept 5 — Global Policy Packages

Author once, above the ADOMs

Penny wants six identical baseline rules on every firewall across two ADOMs (LoFinance and Vaultline sit in separate ADOMs for audit reasons). Typing them into 40 packages by hand breeds two failures: operational (change rule #4 once = 40 manual edits = 40 chances to fat-finger it) and audit (Cipher can't certify that all 40 are identical without eyeballing each one).

The global ADOM sits above the local ADOMs. Author the global policy package once (global objects carry the reserved g prefix), then distribute it.

Global policy package chain
        GLOBAL ADOM  ── the six baseline rules, authored ONCE
             │
        ASSIGN  →  global ADOM → LOCAL ADOM policy layer (flags MODIFIED, staged)
             │
        INSTALL / REINSTALL  →  local policy pkg → device DB → box

   ── shortcut ──
   "Automatically install policies to ADOM devices"
      = installs to local policy pkg + device DB + box in ONE move
      = BUT bypasses the installation preview  ⚠
OperationDirectionResult
Assignglobal ADOM → local policy layerLocal package flagged Modified. Staged, not yet on any box.
Install / Reinstalllocal policy pkg → device DB → boxRules land on the firewalls (with preview).
Automatically install to ADOM devicesglobal ADOM → policy pkg + device DB + boxOne move, three targets — no preview.
The auto-install shortcut is the green button with the safety removed. On a fleet you know contains a drifted box, skipping the preview means you may erase undocumented config the instant you distribute the baseline.
The Delivery Plan

Satisfying both Penny and Cipher

"Be safe" is not a plan Penny accepts. Here is the senior version — and the insight is that being safe costs almost nothing, because the preview is fast on boxes that aren't drifted.

  1. Assign the global package to both ADOMs. Six rules now staged everywhere. Fast — one move.
  2. Reconcile the known-drifted box first (retrieve → import → audit). The clean boxes have nothing to reconcile — they were never touched.
  3. Install with preview, per package. Clean boxes preview clean and install in seconds. The one drifted box is slow — and it should be.
🎯
Senior Insight
You are not trading speed for safety. You spend caution only where drift actually lives, and move at full speed everywhere else. Penny gets Friday; Cipher gets a preview record for every install.

Troubleshooting Workflow

"The preview shows way more than I changed"

  1. Treat the big diff as a desync report, not a fault.
  2. Name the source and target of the diff (Install Policy Package = policy → DB → box).
  3. Suspect undocumented direct-box edits (invisible in revision history) and/or un-pushed device-DB changes.
  4. Pull before you push: retrieve → import into a separate package → audit the phantoms.
  5. Only then install. Never click green on a mystery.
Common Mistakes

What trips people

  • Softening the blind overwrite. Install does not "prompt about conflicts" — it destroys. Trust the harsh mechanism.
  • Import direction slip. Import is DB → policy (the second hop). Retrieve is box → DB. They are different pulls.
  • "Reinstall skips the preview." It skips the wizard. The preview still appears.
  • Treating revert as the rollback. Revert only stages to the device DB. Without an install, nothing reaches the box.
  • Reaching for the auto-install shortcut on a drifted fleet. No preview = no safety net.
Senior Engineering Insights

The four ideas to keep

  • Push overwrites, pull observes. Direction is the whole risk model.
  • A diff is a confession. A big number is information, not an emergency.
  • Consistency should be structural, not hoped for. Global packages make sameness a property of the architecture.
  • Spend caution only where drift lives. Safety is cheap on boxes nobody touched.
Exam Tips

What the NSE7 will test

  • Retrieve = box→DB, new revision, not the policy package.
  • Import = DB→policy; classic risk is same-name object conflicts.
  • Install Policy Package targets = device DB and box.
  • Reinstall bypasses the wizard but keeps the preview.
  • Quick Install = no preview; global auto-install = no preview.
  • Global chain = Assign (flags Modified) → Install.
  • Undocumented box edit + unrelated install = lost.
