1 · Why the Map Earns Its Memory

The routing table is a list: destination → next hop. It is all the forwarding plane ever consults. So why does OSPF force every router to also carry a complete map of the network — including links it will never use?

LSDB  →  Dijkstra's algorithm  →  SPF tree  →  OSPF routes  →  IP routing table

The routing table is only the last exhaust of the pipe. The map buys independence: when a link dies anywhere, a link-state router needs to hear only the fact of the change — then it recomputes its entire answer set locally, immediately, trusting nobody. A distance-vector router must wait for a chain of retellings, each hop trusting the previous hop's summary — "routing by rumour." This is exactly the property that would have saved the Parramatta outage: nothing on that box knew an alternative existed.

Mental Model

"Rumour spreads answers. Link-state spreads facts. Every router computes its own answers from facts."

2 · LSAs and the LSDB

No router is born knowing the network. The only thing a router genuinely knows firsthand is its own directly connected networks and the cost of each interface. It packages that firsthand truth into a Link State Advertisement (LSA) and floods it to the domain.

  • An LSA advertises links (facts), never routes (answers): "I am R2, I touch Net 1 at cost 2."
  • The LSDB is the collection of all LSAs — your own plus everyone else's. Same testimonies in, identical map out.
  • Routers acknowledge receipt of LSAs — nothing gets lost in the post.
  • On the wire, a link state update = IP header + OSPF header + a string of LSAs, each with its own header.

Quietness: periodic vs triggered

  • Every 30 minutes: routers re-advertise their LSAs even if nothing changed (periodic refresh).
  • Between refreshes: silence — unless a topology change fires a triggered update (triggered/incremental LSA flood), sent immediately by the witness to the change.
Exam Trap

"Every 30 seconds" is RIP's timer. Fortinet plants RIP numbers inside OSPF questions. Read the unit of every number twice — a half-second check worth a full mark. (This one claimed a mark in this very session.)

3 · Cost and the Worked Example

  • Cost is a 16-bit positive number, 1 to 65,535.
  • A route's cost = the sum of exit interface costs along the whole path to the destination.
  • If multiple OSPF routes to the same subnet exist, the lowest total cost wins. Always.

The five-router example (study guide pp. 313–314)

SegmentConnectsCost to enter it
Net 1R1 ↔ R2R1 pays 2 · R2 pays 2
Net 2R2 ↔ R4R2 pays 3 · R4 pays 3
Net 3R2 ↔ R3R2 pays 1 · R3 pays 1
Net 4R3 ↔ R4R3 pays 1 · R4 pays 1
Net 5R3 ↔ R5R3 pays 2 · R5 pays 2

R2's two candidate paths to Net 5:

  • Via R3: R2→Net 3 (1) + R3→Net 5 (2) = 3 ✓ winner
  • Via R4: R2→Net 2 (3) + R4→Net 4 (1) + R3→Net 5 (2) = 6

Dijkstra repeats this comparison for every destination, iteration by iteration, keeping the cheapest known path each round, until every network and router has exactly one winning branch. The finished product is the SPF tree: the local router at the root, lowest-cost paths to everything else. Destinations can be networks or routers.

4 · The Asymmetric Cost Trap (requested scenario)

Cost is directional. An interface cost belongs to the router that owns the interface and is only ever paid on the way out. Take a single link:

DirectionWho paysCost
R1 → R2R1's exit interface1
R2 → R1R2's exit interface2

Neither router ever consults the other side's number. On one link, this is trivia. Across a network of parallel paths, it becomes a trap.

Scenario: the tale of two firewalls

Imagine a client behind FGT-A and a server behind FGT-B, with two parallel paths between the sites — Path North and Path South — each passing through a different firewall pair. An engineer tweaks one interface cost "to prefer the faster circuit"… but only on one end.

DirectionTotal cost NorthTotal cost SouthChosen path
Client → Server10 ✓15North
Server → Client (return)1510 ✓South

Every routing table is technically correct. Every router is loop-free and internally consistent. OSPF is perfectly happy. And yet:

  1. The client's SYN travels Path North and creates session state on the North firewall.
  2. The server's SYN-ACK returns via Path South — arriving at a stateful firewall that has no matching session.
  3. The South firewall drops the reply. Connections time out. Pings may even work (stateless-ish, small) while TCP mysteriously dies — a classic "the network is fine, the app is broken" ticket.
Mental Model

"Cost is paid on exit. Symmetry is your job, not OSPF's." — a careless one-sided cost tweak silently splits forward and return traffic across two stateful devices, with every routing table technically correct.

Senior Practice

When changing an OSPF cost anywhere near parallel paths or firewall pairs: change costs in matched pairs on both directions, then verify with a traceroute in each direction — not just one. On FortiGate specifically, remember HA options like session synchronisation exist precisely because asymmetry happens; but designed-in symmetry beats compensating for asymmetry every time.

5 · Same Map, Different Trees — and Why Loops Mean Broken Sync

  • What is identical: the LSDB (per area). Every router in the same area holds the same map.
  • What is different: each router's SPF tree — R2's tree is rooted at R2, R3's at R3. Five routers, one map, five genuinely different trees.
  • Why there are never contradictions: every tree is computed from the same facts, so each hop's decision dovetails with the next hop's decision.
Mental Model

Routing loops in a link-state network are, almost by definition, a symptom of unsynchronised databases — different maps, not a broken algorithm.

What Full really certifies

The adjacency ladder — Init → 2-Way → ExStart → Exchange → Loading → Full — ends where it ends because Full = identical, synchronised LSDBs between two neighbours. Routing through a neighbour is a bet that its decisions dovetail with yours; that bet is only safe if it computed from the same map. Full is the moment the bet becomes safe. An adjacency stuck below Full is a neighbour whose map you cannot vouch for — and OSPF rightly refuses to route through it.

6 · Areas

One flat LSDB has two scaling costs — both derived, not memorised:

  1. Size: every new site inflates every router's database; memory and SPF computation grow unbounded, punishing the small branch boxes hardest.
  2. Blast radius: a single link flap anywhere triggers a flood and forces a full SPF recalculation everywhere — Chatswood's flaky cable becomes Parramatta's CPU problem.

The fix and its mechanics

  • An area is a logical collection of OSPF networks and routers, identified by a 32-bit number — written as decimal (10) or dotted (0.0.0.10). Same number, two notations: exam sleight-of-hand.
  • Each area has its own separate LSDB. "Identical databases" is true per area. A router with all interfaces in one area holds one LSDB; a router touching multiple areas holds one full LSDB per connected area — separate, never merged.
  • Benefits: smaller LSDBs, contained blast radius (a topology change floods and forces recalculation only within its own area), and route summarisation at area borders.
  • Trade-offs: demands design planning and makes troubleshooting more complex. "A boundary you draw is a boundary you own at 2 AM."
  • Backbone rule: every OSPF network must have a backbone — Area 0.0.0.0 — and all other areas must connect to it (hub-and-spoke), physically or via virtual link. The backbone distributes information between areas.

Router roles — the four-test checklist

RoleDefinition (run as an independent true/false test)LoFinance hub

Discipline

Run all four definitions, every time, out loud — never pattern-match. This checklist has been worth half a mark twice and, this session, defeated a deliberately planted exam trap.

How to read it like the map it is:

  • Each LSA is one row. ADV Router = who testified. Age = seconds since advertised — healthy under 1800s given the 30-minute refresh. Seq# — newer testimony beats older.
  • Per-area sections — two Router Link States blocks on this box because it is an ABR holding two LSDBs. The output's structure IS the architecture made visible.
  • AS External section — this router's ASBR handiwork… if the ADV Router is its own ID. Holding external LSAs proves someone in the domain is an ASBR; advertising them is what makes it you.
  • get router info ospf database router lsa unpacks any row into the full per-link testimony: every link, every cost.
Compliance Win

Exactly one external LSA in the entire domain — 10.99.50.0/24 — verified at the database layer. Penny's rule ("only the management subnet may ever leave that box") is now provable, not just configured.

Troubleshooting Workflow (this session's new layer)

  1. Adjacency shows Full but routes are wrong, missing, or looping? Don't stare at the neighbor table — open the database.
  2. Check per-area structure matches design (right number of LSDB sections?).
  3. Check Age (<1800s) and Seq# progression — stale or stuck LSAs point to flooding/sync problems.
  4. Check the AS External list against policy — anything there that shouldn't be? Anything missing?
  5. Contradictory forwarding in link-state = suspect unsynchronised LSDBs first, never the algorithm.

Common Mistakes

  • Saying routers with identical LSDBs have identical trees — the trees differ (different roots); the map is what's identical.
  • Believing an ABR merges its areas into one shared LSDB — it keeps separate full LSDBs, one per area.
  • Reading "30 seconds" as OSPF's refresh — that's RIP. OSPF: 30 minutes periodic + triggered updates.
  • Assuming a box with external LSAs in its database is the ASBR — check ADV Router.
  • Changing one interface cost near parallel paths without checking the return direction — see the Asymmetric Cost Trap.

Senior Engineering Insights

  • The map buys independence: recompute locally from facts, never wait on a chain of rumours.
  • OSPF cost decisions are also firewall-state decisions — routing and security are one system.
  • Verification beats trust: compliance was proven this session by reading the database, not by re-reading the config.
  • Area boundaries trade troubleshooting complexity for contained blast radius — draw them deliberately, document them, own them.

Exam Tips

  • Pipeline order is directly examinable: LSDB → Dijkstra → tree → routes → RIB.
  • Cost: 16-bit, 1–65,535, lowest total wins.
  • Area ID dual notation: 10 ≡ 0.0.0.10.
  • Backbone must be 0.0.0.0; all areas attach to it.
  • Per-area sections in database brief output = ABR evidence; ADV Router on external LSAs = ASBR evidence.
  • Read the unit of every number twice. Seconds ≠ minutes.

Key Takeaways

  • The LSDB is a crowd-sourced map of firsthand facts; the routing table is its last exhaust.
  • Identical maps, different trees, zero contradictions — and Full is what makes routing through a neighbour a safe bet.
  • Areas contain both scaling costs: database size and blast radius. Backbone 0.0.0.0 is mandatory.
  • The database output is the architecture made visible — and the place compliance gets proven.
