SESSION 16 · TROUBLESHOOTING GUIDE · THINK LIKE A SENIOR

Reading the Map — OSPF Database-Layer Diagnosis

Where this layer sits

Sessions 14–15 gave you the neighbour layer: hellos, network types, the adjacency ladder. This session adds the layer beneath it: the database. Use it when the neighbour layer looks healthy but the routing outcomes are wrong. The core diagnostic principle:

Adjacency Full but routes wrong, missing, or looping? Stop staring at the neighbor table. Open the database — the disagreement lives in the maps, not the handshake.

Symptoms this layer explains

  • Adjacencies all show Full, yet a subnet is missing from the routing table.
  • Traffic loops or takes a nonsensical path while every device claims to be healthy.
  • A route exists but with the wrong cost / wrong next hop.
  • An external route appears in the domain that policy forbids (compliance leak) — or one that should exist doesn't.
  • TCP sessions time out while pings work across parallel paths (→ asymmetric routing suspect).

Questions to ask before touching anything

  1. What exactly is the business symptom, and for whom? One subnet? One site? One direction?
  2. Which routers should carry this destination in their LSDB, and in which area?
  3. Who is supposed to originate this information? A router LSA from a specific router? An external LSA from the ASBR?
  4. Has anything flapped recently? (Fresh Ages everywhere = recent churn.)
  5. Is the return path the same as the forward path? Never assume symmetry.

Decision tree

Symptom: routing outcome wrong
  ├─ Neighbour NOT Full? → S15 layer: network type, hellos, area/timers mismatch. Fix adjacency first.
  └─ Neighbour Full →
      ├─ Destination absent from LSDB? → originator problem: is the source router advertising it? (interface in OSPF? redistribution enabled? filter eating it?)
      ├─ In LSDB on router A but not on area-mate B? → flooding/sync problem between them — compare Seq# and Age both sides.
      ├─ In LSDB but wrong path chosen? → do the cost maths yourself: sum EXIT costs per candidate path; check for a mis-set interface cost. Check BOTH directions.
      ├─ In LSDB but not in RIB? → the RIB chose something else: static/other protocol/administrative distance. Read the full routing table, not just OSPF's slice.
      └─ Unexpected AS External LSA? → read ADV Router → go to THAT box → audit its redistribution + filters. Compliance incident until proven otherwise.

Common dead ends

  • Re-checking the neighbour table repeatedly. Full means the handshake succeeded; it says nothing about whether the right information was ever originated.
  • Blaming Dijkstra. The algorithm doesn't malfunction. Wrong output = wrong input: a wrong cost, a missing LSA, an unsynchronised database.
  • Auditing config to prove compliance. Config shows intent; the database shows reality. Read the AS External section.
  • Testing one direction only. A one-way traceroute proves nothing about the return path — and stateful firewalls only care about the pair.

Likely root causes, ranked

  1. Originator never advertised it (interface not in OSPF / redistribution off / filter too aggressive).
  2. Mis-set interface cost — especially one-sided changes creating asymmetry.
  3. Route present in LSDB but beaten in the RIB by another source (administrative distance).
  4. Flooding/sync fault between area-mates (mismatched LSDBs → loops).
  5. Redistribution leak at an ASBR (filter missing/edited) — find it via ADV Router.

Exam traps at this layer

Trap 1"The output shows external LSAs, therefore this router is an ASBR." No — check ADV Router. Holding testimony ≠ giving it.
Trap 2"Routers in an area have identical trees." Identical LSDBs. Trees differ — each is rooted at its own router.
Trap 3Timer swaps: 30 seconds (RIP) planted in an OSPF question, or Age thresholds quoted in minutes. Read units twice.
Trap 4"The ABR merges its areas into one database." It maintains one separate full LSDB per area.

The senior habit this session adds

When a link-state network misbehaves, a senior engineer's first mental question is not "which router is broken?" but "where do the maps disagree, and who failed to testify?" The database commands are how you ask it.
