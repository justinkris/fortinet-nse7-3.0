The Tripwire

Integrating IPS into enterprise networks — protocol decoders, signature databases, filters vs signatures, rate-based signatures, and the difference between seeing an attack and catching it. Study Guide 06, Security Profiles, pp. 175–186.

The firewall checked the envelope. Today we teach it to read the letter.

Story Recap

The near-miss that wasn't a catch

Tuesday, 8:03 AM. Cipher — quiet all season — drops a threat-intel report on Señor's desk: Sunday 02:14, an exploit attempt hit the trading API, launched from inside the guest VLAN. The lobby Wi-Fi. A visitor's laptop. It reached the API, the API answered the handshake, and then the payload — a size field that lied about its length — collapsed on its own. A parser error, a connection reset. No regulator call.

Señor's blunt verdict: "We did not stop that attack. The attacker stopped it, by being clumsy." The firewall permitted the traffic because it looked like a legitimate connection to a legitimate port. It inspected the envelope and the sender — never the contents. The machinery that reads the contents has been sitting on the box, licensed and idle, all season. Its name is the IPS engine. Today we turn it on and point it at the trading API — carefully, because Junior's instinct to "block everything" would take the trading floor down harder than any attacker.

Concept 1 — The Question IPS Asks

Envelope versus letter

Each security feature you've mastered answers one question. Web filter asks "where are you going?" (domains, URLs, SNI). App Control asks "what are you?" (Layer-7 identity). ISDB knows at packet one "is this IP+port on my list?" All three read the envelope and the sender.

IPS asks a different question entirely: "does the content of this traffic match a known pattern of attack?" It is the first thing on the box that opens the envelope and reads the letter — the payload — comparing it against a catalogue of how exploits look: a size field that lies about its payload, a PHP string smuggled into a URL, a buffer overflow's fingerprint. IPS says "I've seen that shape before, and it was hostile."

The seam
Web filter, App Control and ISDB read the envelope and sender. IPS is the first control that reads the letter — the payload — against a library of known-hostile letters.
Why Sunday matters: the trading API's own parser caught the malformed packet by accident. IPS is built to catch it on purpose, at the firewall, before it ever reaches the API.
Concept 2 — Where the Catalogue Lives

Match locally, update periodically

The catalogue of attack patterns comes from FortiGuard. But unlike web filter — which queries FortiGuard live per-lookup and keeps only a cache — the IPS signature database is stored locally on the FortiGate. The proof is the same rule you built two sessions ago: if a database appears in diagnose autoupdate versions, it lives on the box. IPS Attack Definitions is in that output, with its own version and contract expiry, exactly like the Application Definitions database.

This has to be true for performance. Picture the trading floor: if every packet made a round-trip to FortiGuard before the firewall would forward it, you'd recreate last session's hang — on every session, all the time. IPS inspects at line rate against a local catalogue. FortiGuard's job is not to answer per-packet; it is to keep that local catalogue current via periodic pushed updates.

📌
Mental Note
Match locally, update periodically. A box running last year's IPS signatures is blind to this year's exploits — and it looks perfectly healthy while it's blind. Signature freshness IS a security posture.
Concept 3 — Filter vs Signature

The broad brush and the scalpel

An IPS profile (a "sensor") is a list of entries. Each entry is one of two types:

TypeWhat it doesWhen to reach for it
FilterSelects many signatures at once by broad criteria — severity, protocol, target (client/server), OS. The guide's example (all severities/protocols/OS) resolves to 10,320 signatures in one stroke.Define a policy of coverage: "protect all my servers against all high-severity attacks."
SignatureNames one specific signature and sets its action.Treat one signature differently from the crowd — usually to carve out a false positive.
Pre-reading answer — Filter vs Signature: Filter = the broad brush (coverage by criteria). Signature = the scalpel (one named signature, its own action).
Concept 4 — The Danger of "Block Everything"

Firewall versus wrecking ball

Junior's instinct — Type: Filter, all severities, all protocols, all OS, Action: Block — would block Sunday's attack. It would also block your legitimate traffic that happens to look unusual. That's a false positive, and the guide demonstrates it literally: a legitimate app sending oversized, fragmented packets (hping3 -1 -d 654001 -c 8) trips an "oversized packet" signature. Action is Block. The firewall silently drops them. The far-end app never receives the traffic and stops working.

Here is why the false positive is more dangerous than the attack:

  • It's silent. No error page, no crash — just a hang. Silent failure is worse than loud failure, because loud failure tells you where to look.
  • It looks like correct behaviour. Riley checks the server (healthy), the network (up), the app (running). Nothing is broken — because the firewall is doing exactly what it was told. The team hunts everywhere except the one box that is confidently, correctly wrong.
  • It's self-inflicted. Sunday's attack was someone else's fault; LoFinance was the victim. A false-positive trading halt has no attacker to point at — it's an internal post-mortem with your name on the config change.
The study guide's headline: "Know your network protocols and workflows." An attacker has to be clever and lucky. A bad Filter just has to be switched on.
Concept 5 — The False-Positive Exception

Scalpel on top of the wall

You don't fix a false positive by disabling the Filter — the Filter protects you from everything else. You carve one exception. Three specifics, in this exact order:

  1. Type: Signature — name the one offending signature.
  2. Action: Monitor (packet logging on) — let it pass, but keep a log so you retain visibility.
  3. Placement: ABOVE the broad Filter — IPS entries are evaluated top-down, first match wins.
The gotcha the guide hammers: if the exception sits below the Filter, the Filter matches first, blocks the packet, and your exception is never read — it's dead config. The new signature must have higher priority than the filter signatures; ranked lower, it won't work.
📌
Transferable Rule
In ANY Fortinet ordered list — firewall policy, IPS sensor, SD-WAN rules — the specific exception goes ABOVE the general rule. First match wins.
Concept 6 — Rate-Based Signatures

Content versus count

A normal signature matches the content of a single packet or session. But some attacks have no hostile content in any one packet — the hostility is in the volume, speed, and repetition. A single failed login is Tuesday. Five hundred failed logins in ten seconds is an attack. A rate-based signature doesn't match content; it counts — an event, over a time window, against a threshold.

AttackShapeWhy content-matching is blind
Brute-force / credential stuffingOne source, hundreds of valid login attempts, fast (fits Cipher's guest-VLAN scenario)Every login packet is perfectly valid content. Nothing to match. The attack exists only in the rate.
Resource exhaustion / DoSFlood of legitimate-looking requests to drown the serviceIndividual requests are well-formed; hostility is the cadence, not the payload.
🔫 The keeper analogy
A rate-based attack is legitimate traffic "fired like a machine gun, not a handgun." No single bullet is illegal. The danger is the rate of fire.
Exam discipline: match on the mechanism, not the attack's name. Ask "is the hostility in the content or in the count?" If it's in the count → rate-based signature, whatever label the question uses.
Concept 7 — Placement in the Architecture

The call is coming from inside the house

The classic home for IPS is the edge NGFW — the guide calls it "ideal for applying content filters and IPS" because it's the critical node between internal and external traffic. But Sunday's attack never touched the internet. Guest VLAN → trading segment. Both internal. An IPS profile sitting only at the edge would have watched the attack happen and seen nothing, because the packets never crossed it.

So the tripwire must sit where the two internal zones meet — on the ISFW (Internal Segmentation Firewall), the box between the guest VLAN and the trading segment. The guide notes an ISFW "may serve a dual role as an ISFW and an NGFW." This is that dual role made concrete: IPS on the ISFW is the ISFW acting as an NGFW for internal traffic.

Episode thesis
The perimeter model assumes the threat is outside. Segmentation assumes the threat is already inside. Sunday proved which world LoFinance lives in — the attacker was on the guest Wi-Fi.
FirewallIPS role
NGFW (edge)Standard home for UTM/IPS — inspects internet-facing traffic. Did not see Sunday's internal attack.
ISFWSegments internal zones; may serve NGFW dual-role. Correct place for this IPS profile.
DCFWProtects the data center; may serve as NGFW; can use VDOMs.
DEFWUnder native function (connecting to HQ), doesn't need UTM/IPS in front.
Configuration

Building protect-trading-api

The profile we designed — scoped narrow on purpose (high/critical only, HTTP/HTTPS only, server-side) rather than the guide's cautionary "everything, block":

CLI — IPS sensor with scoped Filter
config ips sensor
    edit "protect-trading-api"
        config entries
            edit 1
                set filter enable
                set severity high critical      # NOT all severities
                set protocol HTTP HTTPS          # scoped to what hits the API
                set location server              # target = protected servers
                set status enable
                set action block
                set log-packet enable
            next
        end
    next
end

The false-positive exception — scalpel on top (lower entry number = evaluated first):

CLI — signature exception, Monitor, placed first
config entries
    edit 0                          # evaluated before the broad Filter
        set rule <offending-signature-id>
        set status enable
        set action monitor           # pass but log
        set log-packet enable
    next
end

Attach to the policy governing guest-VLAN → trading-segment traffic on the ISFW:

CLI — attach to policy
config firewall policy
    edit <guest-to-trading-policy-id>
        set ips-sensor "protect-trading-api"
        set ssl-ssh-profile "certificate-inspection"
    next
end
Note on the SSL profile: certificate-inspection sees only the SNI/handshake. To catch an attack payload inside HTTPS, you need deep/full inspection so FortiGate can decrypt and read the letter. This is Session 25 territory.

Troubleshooting Workflow

The senior judgment sequence

  1. Recognise scope. Attack internal? Then the edge NGFW is the wrong box to inspect.
  2. Locate the boundary. Find the firewall the traffic actually crosses (here: ISFW).
  3. Scope the Filter narrow on purpose. Severity/protocol/target — shrink the false-positive surface before deploying.
  4. Monitor first. Deploy in Monitor, watch the logs, learn "normal," then flip to Block.
The false-positive trap: it doesn't look like a fault — it looks like correct behaviour. Every angle (server/network/app) shows healthy. Check the IPS log.
Common Mistakes

What takes trading floors down

  • All severities / all protocols / all OS / Block. The wrecking ball. Blocks Sunday's attack and Monday's market-data feed alike.
  • Exception placed below the Filter. Dead config. First match wins, top-down.
  • Fixing a false positive by lowering a Filter's severity. Opens the door to a thousand real attacks to fix one misfire. Use a Signature exception instead.
  • Certificate-inspection for an HTTPS payload attack. You'll see the handshake, not the letter. Needs deep inspection.
  • Assuming segmentation-by-address equals inspection. The guest VLAN reached the API's front door. Address separation is the envelope; IPS is the letter.
Senior Engineering Insights

What Señor wanted you to keep

  • We don't want our security to depend on the attacker being bad at their job.
  • Silent failure is worse than loud failure — it hides where to look.
  • Scope before you enforce. Watch before you block.
  • Perimeter thinking assumes trust inside. Segmentation assumes the threat is already in the building.
Exam Tips

How the blueprint tests this

  • Filter vs Signature: "select many at once" = Filter; "one specific, different action" = Signature.
  • Placement in the sensor: any answer implying order doesn't matter is wrong. Exception goes above the Filter.
  • Rate-based: if the stem describes valid packets that are dangerous only in volume/frequency → rate-based. Brute-force and DoS are the flags.
  • Local DB: IPS appears in diagnose autoupdate versions; expired contract = stale DB = blind to new attacks (not "fully protected").
  • Architecture: internal-to-internal attack → ISFW, not edge NGFW.
  • HTTPS payload: requires deep/full SSL inspection, not certificate-inspection.
Key Takeaways

The one-screen summary

Carry these five
1. IPS reads the letter, not the envelope.
2. Filter = broad brush; Signature = scalpel.
3. Exception = Signature + Monitor, above the Filter.
4. Rate-based = count, not content (brute-force, DoS).
5. Internal attack → IPS on the ISFW, not the edge.
