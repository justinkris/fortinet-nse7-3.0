SOCRATIC ARC (student reasoning — strong throughout)
----------------------------------------------------
1. "What question does IPS ask?" Student offered TWO framings; Claude separated
   them: "what is the envelope trying to do" (still App-Control intent territory)
   vs the sharper "does this pattern look familiar / common to an attack" — which
   contains a REFERENCE LIBRARY. Landed on: IPS asks "does the CONTENT of this
   traffic match a known pattern of attack?" First thing on the box that reads the
   LETTER against a catalogue of known-hostile letters. Envelope = web/app/ISDB;
   letter = IPS.
2. "Where does the catalogue live?" Student: FortiGuard source (correct), then
   waffled local-vs-phones-home, offered "combination." Claude used the student's
   OWN S22/S23 rule: if it appears in `diagnose autoupdate versions`, it's stored
   LOCALLY. IPS Attack Definitions IS in that output → local DB, like App Control,
   unlike web filter. Per-packet phone-home would recreate the S23 hang on every
   session. Model: MATCH LOCALLY, UPDATE PERIODICALLY. Student's "combination"
   instinct redirected to the correct mechanism (periodic push, not per-packet).
   Key security consequence student grasped: novel exploit with no signature yet
   sails through; freshness of local DB IS a security posture; expired box looks
   healthy while blind.
3. "How select 10,000 sigs at once?" Student: "category or severity rating."
   → This IS the FILTER (Type: Filter). Guide's example resolves to 10,320 sigs.
   FILTER = broad brush (severity/protocol/OS/target criteria). SIGNATURE = scalpel
   (name one, set its action). This answered pre-reading Q1.
4. "Danger of block-everything?" Student: "legitimate traffic could match a sig."
   = FALSE POSITIVE. Claude pushed to WHO GETS HURT/HOW using the guide's hping3
   oversized-ICMP scenario (hping3 -1 -d 654001 -c 8). Student described it as
   "a hang, no info why, bad UX." Claude sharpened: SILENT failure worse than LOUD;
   the false positive LOOKS LIKE CORRECT BEHAVIOUR, so the team hunts everywhere
   except the one box that's confidently wrong. Business layer: self-inflicted
   trading halt has no attacker to blame — internal post-mortem with your name on
   it. "Junior could take the floor down harder than any attacker."
5. "Build the exception." Student first said "hand-pick the signature" (RIGHT) then
   drifted into Filter-thinking (severity/protocol/OS/target) — Claude flagged this
   as reaching for the broad brush when a scalpel was asked for. Corrected to:
   Type: Signature, Action: Monitor (+ packet logging), placed ABOVE the broad
   Filter. GOTCHA (guide hammers it): top-down, first-match-wins; exception below
   the Filter = dead config. Student's clean retrieval nailed all three (a) Signature
   (b) Monitor (c) Above, otherwise broader filter matches first. Connected to the
   general rule: specific exception goes ABOVE general rule in ANY Fortinet ordered
   list (policy, IPS, SD-WAN).
6. "Rate-based." Student: content-match misses volume attacks; monitors behaviour
   over short window. Coined the KEEPER analogy: "fired like a machine gun, not a
   handgun." Named DDoS (correct family). Claude redirected to Cipher's single-source
   scenario → BRUTE-FORCE / credential stuffing (each login packet valid content;
   attack exists ONLY in the rate). Rate-based sig = count + window + threshold, not
   content match. Full toolkit assembled: signature-match (content), filter (broad),
   signature-exception (scalpel/Monitor on top), rate-based (cadence).
7. Precision lock: "content signature is blind to a ___ attack because ___." Student:
   "resource-exhaustion, because it looks for payload patterns not volume/rate."
   Claude accepted as a MORE GENERAL (valid) category; clarified resource-exhaustion
   (drown the service) vs brute-force (guess credentials) — both rate-based, both
   content-blind. Rule taught: match on MECHANISM (content vs count), not on the
   attack's name.
8. PLACEMENT (pre-reading Q3). Student answered "ISFW to segment the network" in
   five words — the senior answer. Full reasoning drawn out: edge NGFW never saw
   Sunday's traffic (internal, never left building); IPS must sit where the two
   internal zones MEET = ISFW acting in NGFW dual-role. THESIS LINE OF THE EPISODE:
   "Perimeter model assumes threat is outside; segmentation assumes threat is already
   inside." Guest VLAN is inside the walls and untrusted → point IPS inward.

MENTAL MODELS / ANALOGIES (KEEPERS)
-----------------------------------
- "Envelope vs letter": web filter/App Control/ISDB read the envelope+sender; IPS
  reads the LETTER (payload) against a catalogue of known-hostile letters.
- "Match locally, update periodically" (IPS DB is local; FortiGuard keeps it fresh;
  NOT a per-packet phone-home).
- "Scalpel on top of the wall": specific Signature (Monitor) ABOVE broad Filter
  (Block); first match wins.
- "Content vs count": signatures match content; rate-based sigs count cadence.
- "Fired like a machine gun, not a handgun" (STUDENT'S OWN — rate-based attack:
  individually-innocent packets, hostile only in cadence). Keeper.
- "Perimeter assumes threat-outside; segmentation assumes threat-inside." (Episode
  thesis. The call is coming from inside the house — guest VLAN.)
- "The difference between a firewall and a wrecking ball" (Señor, re: scoped Filter
  vs all-severities-block).
- "Checking IDs at the door but never searching the bags" (Riley's root-cause line
  for envelope-not-letter inspection).
- "Watch before block" (Cipher — Monitor mode first, learn normal, then enforce).

TROUBLESHOOTING APPROACH TAUGHT
-------------------------------
The senior move today was JUDGMENT SEQUENCE, not a command:
(1) recognise the attack was INTERNAL → edge NGFW is the wrong place to look;
(2) locate the zone boundary the traffic actually crosses (ISFW);
(3) scope the Filter NARROW ON PURPOSE (severity/protocol/target) BEFORE deploying,
    to shrink false-positive surface;
(4) deploy in MONITOR first, watch, THEN Block. Evidence before enforcement.
Also taught: false positives don't look like faults — they look like correct
behaviour — so they're the hardest outage to diagnose. Check the IPS log, because
every other angle (server/network/app) shows healthy.
