Troubleshooting IPS

How a senior engineer thinks through IPS behaviour — the emphasis is on reasoning, not memorising commands. The hardest IPS problem isn't the attack you missed; it's the legitimate traffic you blocked.

A false positive doesn't look like a fault. It looks like correct behaviour.
That's what makes it the most expensive hour in the incident. Everything shows healthy — except the IPS log.
Symptoms

What you'll see

  • An application hangs — no error, no block page, just a spinner or a "connection reset."
  • Server is healthy, network is up, the app is running — yet traffic isn't getting through.
  • Symptom appeared right after an IPS profile was attached or a signature update landed.
  • Or the reverse: an attack got through and you need to know why IPS didn't catch it.
Questions to Ask First

Define the problem before touching config

  • Scope: one app? one user? one protocol? everything? Surgical scope points at one signature; broad scope points at the engine or the policy.
  • Encryption: is the traffic HTTPS? If so, is the SSL profile deep or certificate-inspection? Cert-inspection can't see the payload.
  • Direction/target: is the affected traffic client-side or server-side? Does the Filter's location match?
  • Content or count: is a single request being blocked (content signature), or bursts (rate-based)?
  • Timing: did it start with a config change or a signature push? A push can introduce a new signature that flags native traffic.

Golden rule: the IPS log is your first stop, not your last. It names the exact signature. Every other check (server, network, app) will say "healthy" during a false positive.
Decision Tree

From symptom to root cause

Is legitimate traffic being blocked, or did an attack get through?
Legitimate blocked → Check the IPS log for the matched signature.
Signature named & it's a real app → carve an exception: Signature type, Monitor action, placed ABOVE the Filter. Do NOT lower the Filter's severity.
No IPS log entry but still blocked → it's not IPS. Look at policy, routing, or another profile.
Attack got through — why?
Was it HTTPS? → certificate-inspection can't read the payload. Need deep inspection.
Was the signature present but stale/expired? → diagnose autoupdate versions. Expired = blind. Renew + update.
Novel attack, no signature yet? → IPS only catches known shapes. Nothing to match. This is a coverage limit, not a fault.
Was it a volume attack (brute-force/DoS)? → content signatures are blind to it. Need a rate-based signature.
Did the sensor even see the traffic? → internal-to-internal attack won't cross the edge NGFW. Must be on the ISFW.
Common Dead Ends

Where engineers waste the hour

  • Rebooting the server / restarting the app. It's healthy. The firewall is dropping the traffic silently.
  • Disabling the whole IPS profile "to test." Fixes the symptom, removes all protection, and teaches you nothing. Read the log instead.
  • Lowering a Filter's severity to fix one false positive. Opens the door to every real attack at that severity. Use a Signature exception.
  • Placing the exception below the Filter. It never runs. First match wins, top-down.
  • Expecting IPS to catch an HTTPS payload under certificate-inspection. It can only see the handshake.
Likely Root Causes

Ranked by frequency

  1. Over-broad Filter (all severities/protocols/OS + Block) catching legitimate unusual traffic.
  2. Exception mis-ordered — placed below the Filter, so it's dead config.
  3. Wrong SSL profile — certificate-inspection where deep inspection is needed for an HTTPS payload.
  4. Stale/expired signature DB — box is blind to current attacks but looks healthy.
  5. Wrong placement — IPS on the edge NGFW when the attack is internal (should be ISFW).
  6. Content signature vs volume attack — needs rate-based coverage.
Exam Traps

What the blueprint hides in the stem

  • "Regardless of placement…" — false. Placement always matters in an ordered list.
  • "Because the DB is local, an expired box is fully protected." — false. Local ≠ current.
  • Scenario names "DDoS" but describes one source hammering one service — that's brute-force; match on mechanism (count), not the label.
  • Internal attack + "put IPS on the perimeter firewall" — the perimeter never sees it. ISFW.
  • "Set the Filter to Monitor to fix the false positive" — that removes blocking for everything. Use a Signature exception on top.
The Senior Habit

Watch before you block

Deploy every new IPS profile on a production-critical policy in Monitor mode first. Watch the logs for a week. Learn what "normal" looks like — which legitimate apps trip which signatures — and carve your exceptions before you flip to Block. The alternative is discovering your false positives live, during market hours, with your name on the change.

📌
Root cause vs restore service
Disabling IPS restores service. Finding the misfiring signature and carving a scoped exception fixes the root cause AND keeps you protected. Only the second one is engineering.
