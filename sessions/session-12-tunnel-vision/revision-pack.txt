QUICK SUMMARY (one page)
----------------------------------------------------------------
ROUTE-BASED IPSEC: creating a phase1-interface makes FortiOS create a
VIRTUAL INTERFACE. The tunnel is a citizen of the routing table and the
policy engine — no special machinery. Packets ENTER the tunnel by winning
a route to it; they PASS by matching a policy on it. Therefore "tunnel
established" and "traffic flows" are two separate achievements.

FIVE-ITEM BUILD CHECKLIST:
  1. Phase 1  (public IPs, IKE version, NAT-T question, PSK/cert)
  2. Phase 2  (SAs + selectors — what may ride)
  3. Tunnel interface IPs (optional now; required for dynamic routing later)
  4. Route    (remote subnet → tunnel interface; static or dynamic)
  5. Policies (per DIRECTION OF INITIATION, both firewalls)

IKE FORMS WITHOUT POLICIES: the handshake is traffic TO the FortiGate
(local-in), not through it. Policies govern only what rides inside.

STATEFUL RULE, REFINED: replies match the session table and are free.
NEW conversations are checked against policy at the door. A two-way site
needs internal→tunnel AND tunnel→internal on BOTH firewalls (unless one
side never initiates).

IKEv2 TIMING: the first child SA (phase 2) piggybacks inside IKE_AUTH —
the ENCRYPTED second exchange — because only after IKE_SA_INIT does the
DH secret exist. That is why IKEv2 has no quick mode.

TROUBLESHOOTING (one-way-silent tunnel): same symptom, two suspects —
missing internal→tunnel at the source site (dies at home) vs missing
tunnel→internal at the far site (dies after decryption). Harvest free
evidence first (working replies prove the far routing table). Run
diag debug flow CLOSEST TO THE SOURCE. "Denied by forward policy check
(policy 0)" = implicit deny = a policy you forgot.

CHEAT SHEET
----------------------------------------------------------------
PHASE 1 (HQ, ISP1 path — S11 decisions carried in):
  config vpn ipsec phase1-interface
    edit "ParraTunnel-ISP1"
      set interface "wan1"
      set ike-version 2
      set peertype any
      set proposal aes256-sha256
      set network-overlay enable
      set network-id 1              ! ISP2 twin: network-id 2; match both ends
      set remote-gw 203.0.113.45
      set psksecret ***

PHASE 2:
  config vpn ipsec phase2-interface
    edit "ParraTunnel-ISP1"
      set phase1name "ParraTunnel-ISP1"
      set proposal aes256-sha256
      set src-subnet 10.1.0.0 255.255.255.0
      set dst-subnet 10.2.0.0 255.255.255.0

STATIC ROUTE INTO TUNNEL (NO gateway — point-to-point):
  config router static
    edit 0
      set dst 10.2.0.0 255.255.255.0
      set device "ParraTunnel-ISP1"

POLICIES (per firewall, two-way site):
  internal → tunnel   ACCEPT      (local users initiate)
  tunnel   → internal ACCEPT      (remote users initiate)

VERIFY:
  diagnose vpn ike gateway list   → version: 2 / network-id / state: established

DEBUG:
  diagnose debug flow filter saddr <ip>
  diagnose debug flow filter daddr <ip>
  diagnose debug flow trace start 10
  diagnose debug enable            (… reproduce …)   diagnose debug disable

KEY OUTPUT LINES:
  "find a route: ... via <tunnel>"                → routing acquitted
  "Denied by forward policy check (policy 0)"     → implicit deny; ADD a policy

GUI PATHS:
  VPN > IPsec Tunnels (phase 1/2) · Network > Interfaces (virtual tunnel
  interface appears here) · Network > Static Routes · Policy & Objects >
  Firewall Policy

DEFAULTS / BEHAVIOURS:
  - Tunnel virtual interface auto-created with phase 1
  - Tunnel static route: device only, no gateway
  - IKE = local-in; tunnel forms with zero forward policies
  - Implicit deny = "policy 0" in debug flow
  - network-overlay/network-id: IKEv2 ONLY, Fortinet-proprietary,
    must match at both ends

DESIGN RULES:
  - Policies count initiation direction, not travel direction
  - Debug closest to the source; follow the packet in packet order
  - Evidence before change: two read-only commands beat ten edits

----------------------------------------------------------------
FLASHCARDS
----------------------------------------------------------------
Q: What does "route-based IPsec" mean mechanically?
A: The phase 1 creates a virtual interface; routing decides tunnel entry
   and normal policies control passage. No special VPN machinery.

Q: Tunnel established, no traffic either direction — two most likely causes?
A: Missing route to the tunnel interface; missing firewall policy on it.

Q: Do replies crossing back through a tunnel need a reverse policy?
A: No — replies match the existing session (stateful). Only NEW
   initiations need a policy in their direction.

Q: A branch user opens a NEW connection to HQ; HQ only has internal→tunnel.
   Result?
A: No session exists, policy lookup finds no match, packet hits implicit
   deny (policy 0), silently dropped.

Q: Which IKEv2 exchange carries the first child SA, and why there?
A: IKE_AUTH — the encrypted second exchange; the DH secret exists by then,
   so subnets/keys never travel in plaintext. Hence no quick mode.

Q: Two IKEv2 tunnels between the same gateway pair — commands?
A: set network-overlay enable + set network-id N (IKEv2 only, must match
   both ends).

Q: What's unusual about a static route pointing into a tunnel?
A: No "set gateway" — the interface name suffices (point-to-point).

Q: What is "policy 0" in debug flow output?
A: The implicit deny — nothing matched. It means a policy is MISSING,
   not misconfigured.

Q: One-way-silent tunnel — the two suspects?
A: Missing internal→tunnel at the initiator's site (dies at home) vs
   missing tunnel→internal at the far site (dies after decryption).
   Identical symptom; debug flow at the source discriminates.

Q: Does the tunnel need a forward policy to FORM?
A: No — IKE is local-in traffic (to the FortiGate, not through it).

Q: Where do you run debug flow first and why?
A: On the firewall closest to the source — the first device to see the
   packet is the first that can kill it; evidence propagates forward.
