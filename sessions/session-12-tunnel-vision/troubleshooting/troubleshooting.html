🔧 TROUBLESHOOTING GUIDE · SESSION 12

The One-Way-Silent Tunnel

How a senior engineer finds why traffic "flows one way and vanishes the other" — without changing a single setting first.

Symptoms

  • Tunnel shows established; one site's users work perfectly, the other site's users get spinning clients — no error messages, no timeouts with reasons, just silence.
  • Often reported as "the VPN feels weird" or "it works from here but not from there."

Questions to ask before touching a keyboard

  • Which direction of initiation fails? Not travel — initiation. "HQ can reach the branch" means HQ-initiated sessions work, replies included.
  • Does anything work from the failing side? If pings work but new TCP sessions don't, that's a different fault family (policy vs MTU/service) — scope it.
  • What free evidence does the working direction give you? Working HQ→branch replies prove: both phase 1/phase 2 SAs are healthy in both directions, selectors cover both subnets, and the branch routing table already routes the HQ subnet into the tunnel. Routing on the far side is acquitted before you log in.

The two suspects

Same symptom, two crime scenes

Suspect A — the initiator's own firewall is missing internal → tunnel. The packet dies at home; it never enters the tunnel at all.

Suspect B — the far firewall is missing tunnel → internal. The packet crosses, decrypts successfully… and dies on the implicit deny at the destination.

The user experience is identical. Only evidence distinguishes them — never assume.

The two lines that decide everything:

msg="find a route: gw-10.1.0.20 via ParraTunnel-ISP1"     routing did its job
msg="Denied by forward policy check (policy 0)"           implicit deny — a policy is MISSING

Decision tree

Tunnel state: established? — if NO, this is a phase 1/phase 2 problem, different guide.
▼ YES — run debug flow on the initiator's firewall
Does the trace show a route via the tunnel interface? — if NO: missing/wrong route at the source. Fix routing.
▼ route OK
"Denied by forward policy check (policy 0)" at the source → Suspect A convicted. Add internal → tunnel here.
▼ source shows "sent to <tunnel>" cleanly — move to the far firewall, repeat
Policy 0 at the destination → Suspect B convicted. Add tunnel → internal there.
▼ neither?
Both firewalls forward — widen scope: server firewall/host, NAT on the policy, selectors, or (if only large packets vanish) MTU territory — Session 13.

Common dead ends

  • Rebuilding the tunnel. The tunnel was never the problem — established plus one working direction proves the crypto end-to-end.
  • Staring at the routing table on the failing side when working replies already acquitted it.
  • Adding a reverse policy "for replies." Replies never needed one; you'll fix nothing and clutter the ruleset. The missing policy is for initiators.
  • Debugging at the destination first. If the source firewall dropped it, the destination never saw it — you'll capture silence and conclude wrongly.

Likely root causes (in order)

  1. Missing initiation-direction policy on one firewall (this session's fault — Suspect A).
  2. Missing/incorrect static route on the initiating side (packet leaks out the default route).
  3. Policy exists but wrong interface pair, wrong address objects, or disabled.
  4. Phase 2 selectors don't cover the failing subnet pair (traffic structurally excluded — recall S10's "declared pipe").

Exam traps

Trap 1: "The tunnel is up, therefore routing/policy must be fine." The exam loves answers like "renegotiate phase 2" for what is a missing-policy scenario. Established SAs prove nothing about forwarding.
Trap 2: Questions that say a new connection fails from site B while site A "works" — the distractor will offer a reply-direction explanation. Initiation direction is what policies count.
Trap 3: "policy 0" in debug output presented as if it were a misconfigured rule. Policy 0 is the implicit deny; the fix is always to add a policy, not edit one.

The method, distilled

  1. Restate the symptom in initiation terms.
  2. Harvest free evidence from whatever works.
  3. Enumerate the suspects the symptom permits.
  4. Choose the one command that discriminates between them.
  5. Run it closest to the source; follow the packet in the order the packet travels.
  6. Convict, fix one thing, verify the user outcome — then confirm root cause, not just recovery.
