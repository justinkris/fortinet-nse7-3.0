TECHNICAL CONCEPTS COVERED
--------------------------
1. ROUTE-BASED IPSEC CORE IDEA: creating a phase1-interface makes FortiOS create a
   VIRTUAL INTERFACE (appears in Network > Interfaces). "The tunnel is an INTERFACE."
   No special diversion machinery — routing decides tunnel entry; policy engine treats
   the tunnel interface like any other interface.
2. THE FIVE-ITEM BUILD CHECKLIST (student dictated most of it himself, unprompted,
   including the NAT-T consideration): (1) phase 1 — public IPs, NAT question;
   (2) phase 2; (3) tunnel interface IPs; (4) route to remote subnet (static or dynamic);
   (5) firewall policies.
3. IKE = LOCAL-IN TRAFFIC: the handshake is traffic TO the FortiGate, so no forward
   policy is needed for the tunnel to FORM. Policies are only for traffic INSIDE it.
   (Señor flagged this instinct as useful later for OSPF control-plane thinking.)
4. STATEFUL RULE REFINED: policies are written per DIRECTION OF INITIATION, not per
   direction of travel. Replies ride sessions for free; NEW conversations pay at the
   door — on BOTH firewalls. Working two-way site = 2 policies per firewall
   (internal→tunnel and tunnel→internal) unless one side never initiates.
5. PHASE 1 CONFIG (built live, HQ side), carrying S11 decisions:
   config vpn ipsec phase1-interface / edit "ParraTunnel-ISP1" / set interface "wan1" /
   set ike-version 2 / set peertype any / set proposal aes256-sha256 /
   set network-overlay enable / set network-id 1 / set remote-gw 203.0.113.45 /
   set psksecret ***. ISP2 twin identical with network-id 2 (Junior typed it).
6. PHASE 2 CONFIG: config vpn ipsec phase2-interface / edit "ParraTunnel-ISP1" /
   set phase1name "ParraTunnel-ISP1" / set proposal aes256-sha256 /
   set src-subnet 10.1.0.0/24 (HQ LAN) / set dst-subnet 10.2.0.0/24 (Parra LAN).
   Selectors NOT re-taught (student owns "trunk over a tunnel" from S10).
7. IKEv2 PHASE 2 TIMING: first child SA piggybacks inside IKE_AUTH (the ENCRYPTED
   second exchange) — safe because the DH secret already exists; hence no quick mode
   and zero extra messages. (This was the S11→S12 queued link check.)
8. STATIC ROUTE INTO TUNNEL: config router static / edit 0 / set dst 10.2.0.0/24 /
   set device "ParraTunnel-ISP1". NO "set gateway" — tunnel interface is point-to-point,
   interface name suffices. Flagged as exam-relevant oddity.
9. VERIFICATION: diagnose vpn ike gateway list → version: 2, network-id: 1,
   state: established.
10. TROUBLESHOOTING: diagnose debug flow (filter saddr/daddr, trace start, debug enable).
    Key output lines taught: "find a route ... via ParraTunnel-ISP1" (routing acquitted)
    and "Denied by forward policy check (policy 0)" (implicit deny = a policy you forgot).
    TWO-SUSPECT DISCIPLINE for one-way-silent tunnels: missing internal→tunnel at the
    initiator's site (dies at home) vs missing tunnel→internal at the far site (dies
    after decryption). Identical user symptom; debug flow names the guilty firewall.
    Rule coined: "follow the packet in the order the packet travels" — debug closest
    to source first; evidence propagates forward.
11. FREE EVIDENCE FROM SYMPTOMS: working HQ→Parra replies PROVE the Parramatta routing
    table already routes 10.1.0.0/24 into the tunnel — routing acquitted before any
    command was run. Student was guided to this; see register.

EXAM-RELEVANT NOTES
-------------------
- Route-based IPsec = virtual interface + route + policies; "tunnel up, no traffic" =
  route or policy missing (classic NSE7 distractor set).
- Static route to tunnel needs device only, no gateway (point-to-point).
- IKEv2 first child SA inside IKE_AUTH (encrypted exchange) — no quick mode.
- "Denied by forward policy check (policy 0)" in debug flow = implicit deny.
- network-overlay enable + network-id N, IKEv2 only, must match both ends.
- IKE is local-in traffic; tunnels form without forward policies.
