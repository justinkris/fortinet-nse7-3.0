Tunnel Vision — the complete guide

Building phase 1 & phase 2 · route-based IPsec · why "tunnel up" is not "traffic flows"

Session Overview

Install day at Parramatta. The branch FortiGate came out of the box, the S10 design and S11 handshake decisions became real configuration, and LoFinance's first production tunnel went live — then immediately taught us its most important lesson when Parramatta-initiated traffic silently vanished. This guide covers the route-based IPsec model, the full build (phase 1, phase 2, tunnel route, policies), verification, and the troubleshooting of a one-way-silent tunnel using diagnose debug flow.

Concept 1 — The Tunnel Is an Interface

When you create a phase1-interface, FortiOS creates a virtual network interface named after it. It appears in Network > Interfaces beside the physical ports. That single fact is the whole meaning of route-based IPsec:

🧠 Mental model — "The tunnel is an INTERFACE"
There is no special VPN traffic-diversion machinery. A packet reaches the tunnel the same way it reaches any interface: it wins a route pointing at it. And it is permitted through the same way as anywhere else: it matches a policy whose source or destination interface is the tunnel. Routing and the policy engine treat the tunnel as an ordinary citizen.

Consequence: a healthy, established tunnel with no route and no policy carries nothing. "Tunnel up" and "traffic flows" are separate achievements.

Concept 2 — The Five-Item Build Checklist

#ItemWhat it doesIf missing…
1Phase 1Secure authenticated channel between gateways (public IPs, IKE version, NAT‑T question, PSK/cert)No tunnel at all
2Phase 2Negotiates SAs for the protected traffic; selectors declare what may rideTunnel channel exists, no data SAs
3Tunnel interface IPsOptional addresses on the virtual interface — needed for dynamic routing, ADVPN, NAT-on-tunnel, managementStatic setups still work; dynamic routing later won't
4RouteStatic or dynamic route sending the remote subnet into the tunnel interfacePackets take the default route to the internet and die
5PoliciesPermit traffic through the tunnel interface — per direction of initiation, both firewallsImplicit deny; silent drops
Key insightThe IKE handshake is traffic to the FortiGate (local-in), not through it — so the tunnel forms with zero forward policies. Policies govern only what travels inside it.

Concept 3 — Replies Are Free; New Conversations Pay at the Door

The FortiGate is stateful. The first packet of a session is checked against the policy table; every subsequent packet — including replies from the far end — matches the session table and never consults policy again.

🧠 Packet A vs Packet B
Packet A — a server's reply to a connection HQ initiated. Matches an existing session → sails through. Needs no reverse policy, ever.

Packet B — a branch workstation opening a brand-new connection to HQ. No session exists. Policy lookup runs at HQ in the direction tunnel → internal. If no policy matches: implicit deny (policy 0), silent drop.

Therefore: write policies per direction of initiation, not per direction of travel. A two-way site needs both directions on both firewalls.

Configuration — The Build (HQ side)

Phase 1 (carrying the Session 11 decisions)

config vpn ipsec phase1-interface
    edit "ParraTunnel-ISP1"
        set interface "wan1"
        set ike-version 2                 ← standardised in S11; Cipher-approved
        set peertype any
        set proposal aes256-sha256
        set network-overlay enable        ← switch the feature on (IKEv2 only)
        set network-id 1                  ← "the event on the ticket"; must match both ends
        set remote-gw 203.0.113.45        ← Parramatta ISP1 public IP
        set psksecret ************
    next
end

The ISP2 twin is identical except the name and set network-id 2 — the network-id is what lets two IKEv2 phase 1s coexist between the same gateway pair.

Phase 2

config vpn ipsec phase2-interface
    edit "ParraTunnel-ISP1"
        set phase1name "ParraTunnel-ISP1"
        set proposal aes256-sha256
        set src-subnet 10.1.0.0 255.255.255.0     ← HQ LAN
        set dst-subnet 10.2.0.0 255.255.255.0     ← Parramatta LAN
    next
end
S11 → S12 linkIn IKEv2 the first child SA (phase 2) piggybacks inside IKE_AUTH — the encrypted second exchange — because only after IKE_SA_INIT does the DH secret exist to protect it. That is why IKEv2 has no quick mode and finishes in four messages.

Static route into the tunnel

config router static
    edit 0
        set dst 10.2.0.0 255.255.255.0
        set device "ParraTunnel-ISP1"     ← NO gateway: tunnel is point-to-point
    next
end

# HQ FortiGate
policy 12:  internal → ParraTunnel-ISP1   ACCEPT   (HQ initiates)
policy 13:  ParraTunnel-ISP1 → internal   ACCEPT   (branch initiates — "Packet B")

# Parramatta FortiGate — mirror image
policy 1:   internal → HQTunnel-ISP1      ACCEPT
policy 2:   HQTunnel-ISP1 → internal      ACCEPT   ← the one Junior forgot… other direction

Troubleshooting Workflow — The One-Way-Silent Tunnel

Symptom: HQ-initiated traffic works, replies included. Branch-initiated traffic spins and vanishes — no error.

  1. Harvest free evidence from the symptom. Working HQ→branch replies prove the branch routing table already sends the HQ subnet into the tunnel. Routing acquitted before any command.
  2. Name both suspects. Same user symptom, two crime scenes:
    • Suspect A: branch missing internal → tunnel — packet dies at home, never enters the tunnel.
    • Suspect B: HQ missing tunnel → internal — packet crosses, decrypts, dies at HQ's implicit deny.
  3. Debug closest to the source — follow the packet in the order the packet travels:
Parra-FGT # diagnose debug flow filter saddr 10.2.0.30
Parra-FGT # diagnose debug flow filter daddr 10.1.0.20
Parra-FGT # diagnose debug flow trace start 10
Parra-FGT # diagnose debug enable

msg="find a route: gw-10.1.0.20 via ParraTunnel-ISP1"   ← routing did its job
msg="Denied by forward policy check (policy 0)"          ← implicit deny: Suspect A convicted
Read it like a verdictPolicy 0 is not a policy anyone wrote. It is the implicit deny — the firewall saying "nothing matched." It means a policy is missing, not that anything is broken.

Common Mistakes

  • Believing an established tunnel implies flowing traffic — items 4 and 5 of the checklist are separate work.
  • Writing one policy per site and trusting statefulness to cover new connections in the other direction (Junior's mistake — the rule covers replies, not initiators).
  • Adding set gateway to a tunnel static route — the interface alone suffices (point-to-point).
  • Changing configuration before gathering evidence — two read-only commands identified the fault and the firewall.
  • Debugging at the destination first — start at the source; evidence propagates forward.

Senior Engineering Insights

  • The tunnel is the easy part. Most of the checklist — and most of the failures — live outside phases 1 and 2.
  • Same symptom, multiple crime scenes. Enumerate the suspects before running commands; then choose the command that discriminates between them.
  • Free evidence first. A working direction is a set of proofs about routes, SAs, and selectors — spend them before spending keystrokes.
  • Silent drops are policy drops. No ICMP error + spinning client is the signature of implicit deny, on one of exactly two boxes.

Exam Tips

  • "Phase 1 and phase 2 are up but no traffic passes" → answer options containing route and firewall policy are the pair to pick.
  • IKEv2's first child SA rides IKE_AUTH (encrypted), never IKE_SA_INIT — a favourite T/F trap.
  • network-overlay enable + network-id N: IKEv2-only, must match at both ends, enables multiple phase 1s between one gateway pair.
  • Static route to a tunnel: set device, no gateway.
  • In debug flow output, "policy 0" = implicit deny.

Session Exam Questions

Q1. Phase 1 and phase 2 establish successfully, but no traffic passes in either direction. Which TWO items are most likely missing?
A — a firewall policy referencing the tunnel interface, and B — a route directing traffic to the tunnel interface. The tunnel forming proves crypto and reachability; it proves nothing about routing or policy.
Q2. True or False: In IKEv2, the first child SA is negotiated inside IKE_SA_INIT, before the DH secret exists.
False. It rides IKE_AUTH, the encrypted exchange — phase 2 describes protected subnets and keys, which must never travel in plaintext.
Q3. A branch user opens a NEW connection to HQ. HQ has only internal → tunnel. What happens to the first packet at HQ?
Route lookup succeeds; the session table has no entry (first packet); policy lookup runs in the direction tunnel → internal; nothing matches; the packet hits the implicit deny (policy 0) and is silently dropped. Fix: add the tunnel → internal policy.

Key Takeaways

  • Route-based IPsec = virtual interface + route + policies. No new machinery.
  • Policies per direction of initiation; both firewalls; replies ride sessions free.
  • IKE forms tunnels via local-in — no forward policy needed for establishment.
  • diag vpn ike gateway list to verify; diag debug flow to convict.
