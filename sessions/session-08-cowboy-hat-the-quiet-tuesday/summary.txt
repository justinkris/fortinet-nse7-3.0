BUSINESS PROBLEM
-----------------
TICKET 1: VLAN 30 (Support) had unintended Layer 2 reachability into VLAN 10 (Finance) —
a support tech reached a finance workstation via RDP that should have been blocked by an
existing, correctly-configured deny policy (policy ID 12, VLAN30-Support -> VLAN10-Finance,
action deny). Root cause: someone (implied cowboy-hat sabotage) created a software switch
object named "OpsBridge" (config system switch-interface) with VLAN10-Finance and
VLAN30-Support both set as members, intra-switch-policy implicit. This merged the two VLAN
interfaces into one Layer 2 broadcast domain at the switch-bridge layer, so traffic between
them never reached FortiGate's routing/policy engine at all — the deny policy was correct
and present but never got the chance to evaluate anything. Resolved: deleted the OpsBridge
switch-interface object directly (delete works without detaching members first — student
tested this assumption rather than assuming a two-step process was required), cleared the
stale live session (treated correctly as an active unauthorized RDP connection needing
remediation, not just housekeeping), then verified via `execute log filter` / `execute log
display` that a retest now shows two distinct srcintf/dstintf VLAN interfaces and a real
policyid=12 action=deny — confirming the correct enforcement mechanism, not just a
different-looking failure.

TICKET 2: Guest-WiFi VDOM total outage. All config-layer checks came back clean — VDOM_Link0/
1 both up, static default routes present and correct on both sides, firewall policies
present/enabled/correctly directional on both root and Guest-WiFi. Root cause (a direct,
deliberate payoff of Session 7's unresolved forward-hook — Señor's unexplained line "Don't
let it inherit whatever root's ceiling was"): when Junior originally built the Guest-WiFi
VDOM, he cloned root's config as a starting template rather than building clean, and
root's explicit `config system vdom-property` session override (session 50, set low
historically to protect shared resources for Trading's latency-sensitive traffic) came
along with the clone. This was invisible at launch under light testing traffic and only
became a hard outage once real guest device volume grew and slammed into the 50-concurrent-
session ceiling — confirmed via `diagnose sys session stat` showing session_count=50
session_limit=50 clash=182 (182 rejected connection attempts piling up against a full
table). Resolved: `config global / config system vdom-property / edit "Guest-WiFi" / set
session 0` (0 = no override, draws from the shared global pool) to restore service
immediately, with an explicit ticket follow-up item to baseline real usage first (student's
own instinct, unprompted) before setting a deliberate, evidence-based cap later — correctly
distinguished from just picking a new round number as a permanent "fix."

TECHNICAL CONCEPTS COVERED
----------------------------
1. Software switch bridging as a VLAN-segmentation bypass mechanism (new synthesis this
   session, building on Session 5's PrintGroup material): `config system switch-interface`
   with `intra-switch-policy implicit` doesn't just bundle physical ports (Session 5) — it
   can bundle any interface objects, including two already-tagged VLAN interfaces, merging
   them into one Layer 2 broadcast domain at the bridge layer regardless of their VLAN IDs.
   This is a stronger and more dangerous mistake than Session 5's version because it
   actively UN-does existing segmentation rather than sitting inside an already-flat domain.
2. The `diagnose sys session list` signature for this bypass class, generalized and
   reapplied confidently: identical ingress/egress interface index (`dev=13->13/13->13`
   this session, vs named different VLAN interfaces for genuinely routed traffic) plus
   `policy_id=0` on a session that DOES show real byte/packet counters (i.e., traffic is
   actually flowing) = bridged/switched, never reached the policy engine. Explicitly
   distinguished from the OTHER thing that also shows `policy_id=0` — a genuine implicit
   deny, which normally produces no persistent session entry at all, since a fully denied
   packet is dropped before a session object is built. Student needed this distinction
   handed to him directly (see STUDENT LEARNING OBSERVATIONS) — track as a real exam trap.
3. `diagnose netlink interface list` as the way to map an opaque interface index number
   (from session-list output) back to a named interface, and to read `master=<index>`
   as the tell that an interface has been enslaved into a bridge/switch construct — new
   command this session, introduced directly (student had no prior exposure).
4. Interface-object deletion behavior: `config system switch-interface / delete "name"`
   succeeds directly without first requiring `unset member` — student guessed a two-step
   process was mandatory, then correctly tested the direct-delete assumption instead of
   padding the command sequence defensively. Good instinct worth reinforcing as a pattern:
   verify FortiOS's actual behavior rather than assuming extra caution steps are required.
5. Session-table hygiene as part of remediation, not separate from it: a stale/live session
   that predates a config fix is NOT automatically re-evaluated against the new config —
   `diagnose sys session clear` (against an active filter set via `diagnose sys session
   filter src/dst`) is required to force re-evaluation on the next connection attempt.
   Student initially treated clearing the session as an optional/risky step ("gut feels
   wrong") before correctly reasoning that leaving a live unauthorized session active IS
   the ongoing security exposure, not neutral inaction.
6. `execute log filter` / `execute log display` as the CLI equivalent of the GUI traffic
   log, and — critically — the correct tool for distinguishing "genuinely denied by policy"
   from "never reached policy at all," since a real deny decision (with policyid, action,
   distinct srcintf/dstintf) shows up in the traffic log even when `diagnose sys session
   list` shows nothing persistent for it. New command, introduced directly.
7. `config system vdom-property` (global context only, NOT inside any individual VDOM) as
   the actual CLI home for the "Global Resources / Override Maximum" concept seeded
   unexplained back in Session 7. Session limit, policy limit, and other per-VDOM ceilings
   live here. `set session 0` = no override (draws from shared global pool), NOT "zero
   sessions allowed" — student briefly floated both readings before landing correctly on
   "no override," an important precision distinction (0 as "unlimited/unset" vs 0 as "hard
   zero cap" is a classic FortiOS reading trap that recurred structurally this session,
   parallel to the policy_id=0 ambiguity in Ticket 1).
8. `diagnose sys session stat` for per-VDOM session count vs. limit, including the `clash`
   counter as direct, countable evidence of rejected connection attempts hitting a full
   table — new command, gave the student the concrete "yes, this is really the bottleneck"
   confirmation he asked for before accepting the resource-limit hypothesis as proven.
9. VDOM config-object isolation, tested directly via a stretch/retrieval question (not
   live troubleshooting): a `config system switch-interface` object built inside one VDOM's
   config context cannot reference an interface belonging to a different VDOM, because each
   VDOM is a fully separate config tree with its own interface namespace — the OpsBridge-
   style bypass mechanism from Ticket 1 is structurally IMPOSSIBLE across a VDOM boundary
   (e.g., root VDOM's OpsBridge could never have reached into the isolated Trading VDOM).
   This directly resolves the specific soft spot flagged at the end of Session 7 (student's
   near-miss reaching for "external L2/physical topology risk" instead of "internal VDOM
   routing/config isolation" when reasoning about Trading VDOM's safety) — see below, this
   took a full restart to land cleanly rather than being self-generated.

TROUBLESHOOTING APPROACH TAUGHT
----------------------------------
Reinforced and extended the Session 4/5 diagnostic split (no session = below the firewall;
session with policy_id = policy logic) into a THREE-way read: no persistent session at all
= likely genuine deny (verify via traffic log, not session list); persistent session with
real traffic AND policy_id=0 AND identical dev in/out = Layer 2 bridge bypass; persistent
session with a real policy_id = normal routed/policy-evaluated traffic, investigate policy
logic/ordering from there. New this session: when EVERY config-layer check comes back
clean (interfaces up, routes present, policies present/enabled/correct) but the symptom
persists, the fault is not in a place `show`/`get` config commands can see at all — pivot
to resource/state diagnostics (`diagnose sys session stat`, `vdom-property`) rather than
re-checking the same config layer more carefully. Also reinforced: an active session filter
left over from a previous ticket will silently narrow the next command's results without
any error — always clear filters between distinct troubleshooting threads, don't assume an
empty result is meaningful until the filter state itself is confirmed clean.
