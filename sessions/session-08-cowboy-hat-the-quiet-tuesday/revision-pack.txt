QUICK SUMMARY (one page)
=========================================================================
Boss battle, two simultaneous tickets, both layered.

TICKET 1: VLAN 30 (Support) could reach VLAN 10 (Finance) via RDP despite a
correct, existing deny policy. Cause: a software switch object ("OpsBridge")
had both VLAN interfaces as members with intra-switch-policy implicit —
this bridged the two VLANs into one Layer 2 broadcast domain, so traffic
between them never reached FortiGate's routing/policy engine at all. The
deny policy was never wrong; it was never consulted. Diagnosed via
`diagnose sys session list` showing identical dev index in/out and
policy_id=0 alongside real traffic counters. Fixed by deleting the
OpsBridge switch-interface object, clearing the stale live session, and
verifying via the traffic log (execute log filter/display) that a retest
now shows a real policyid, action=deny, and two distinct interfaces.

TICKET 2: Guest-WiFi VDOM went completely dark — zero reachability, not
partial. Every config-layer check (interface status, static route, firewall
policy on both root and Guest-WiFi sides) came back clean. Root cause:
Guest-WiFi's `config system vdom-property` session override was 50 —
identical to root's own override, because Guest-WiFi's config was cloned
from root's when it was originally built, and the low session ceiling came
along for the ride (this was the exact thing Señor's Session 7 forward-hook
line — "don't let it inherit whatever root's ceiling was" — was warning
about). Confirmed via `diagnose sys session stat`: session_count=50,
session_limit=50, clash=182. Fixed immediately by setting the override to 0
(no override, shared global pool), with an explicit ticket follow-up to
baseline real usage before setting a deliberate permanent number.

Core lesson tying both together: VDOM isolation and segmentation are not
just about interfaces, routes, and policies existing correctly — they also
depend on config-object namespaces staying separate (switch-interface
bridges) and resource ceilings being deliberately set rather than
accidentally inherited. Both faults were invisible to a surface-level
config read; both required pushing one layer deeper.

=========================================================================
CHEAT SHEET
=========================================================================
COMMANDS
  diagnose sys session filter src <subnet> <mask>
  diagnose sys session filter dst <subnet> <mask>
  diagnose sys session filter clear                — clears an active filter
  diagnose sys session list                         — shows policy_id, dev in/out
  diagnose sys session clear                        — clears sessions matching filter
  diagnose sys session stat                         — per-VDOM session_count/limit/clash
  diagnose netlink interface list                   — maps index → name, shows master=<idx>
  show system switch-interface                      — software switch config + members
  show system virtual-switch                        — hardware switch config
  execute log filter category traffic
  execute log filter field srcip <ip> / dstip <ip>
  execute log display                                — CLI traffic log equivalent of GUI
  config global
    config system vdom-property                      — per-VDOM resource overrides
      edit "<vdom>"
        set session <n>                               — 0 = no override (shared pool)
        set policy <n>
      next
    end

GUI PATHS (equivalent views)
  Network > Interfaces                              — interface status, VDOM assignment
  Policy & Objects > Firewall Policy                — policy table, per VDOM
  Log & Report > Forward Traffic                    — GUI traffic log
  System > VDOM (Global VDOM context)                — VDOM resource overrides

DEFAULTS / BEHAVIOURS
  intra-switch-policy implicit  — members bridged, no policy evaluation between them
  intra-switch-policy explicit  — members still require an explicit policy
  vdom-property value of 0      — NO override; draws from shared global resource pool
  vdom-property value of N (>0) — hard cap of N for that resource in that VDOM
  switch-interface object delete — works directly, no need to unset members first
  session clear                 — required after a fix; stale sessions are NOT
                                   automatically re-evaluated against new config

DESIGN RULES
  - A switch-interface/virtual-switch object can only reference interfaces
    that exist within the SAME VDOM's config tree — never cross-VDOM.
  - policy_id=0 is ambiguous alone: pair with "does a persistent, traffic-
    bearing session exist?" to tell "correctly denied" from "bypassed."
  - Identical ingress/egress interface index = Layer 2 switched, not routed.
  - Fix resource ceilings in two steps: unblock now (remove/raise the cap),
    baseline real usage, THEN set a deliberate permanent value.

=========================================================================
FLASHCARDS
=========================================================================
Q: A session shows dev=13->13 both directions and policy_id=0, with real
   byte/packet counters. What does this indicate?
A: The traffic is being switched at Layer 2 (bridged), not routed — it
   never reached FortiGate's policy engine.

Q: What command maps a raw interface index number back to its name and
   reveals bridge membership?
A: diagnose netlink interface list — look for master=<index>.

Q: What does master=13 on an interface mean in that output?
A: The interface has been enslaved as a member of the bridge/switch
   construct at interface index 13.

Q: Can config system switch-interface bundle two already-tagged VLAN
   interfaces, and what happens if it does?
A: Yes — this merges them into one Layer 2 broadcast domain, defeating
   the VLAN segmentation between them even though the tags are still valid.

Q: True/False — a genuine implicit deny always produces a session entry in
   diagnose sys session list.
A: False, typically — a fully denied packet is usually dropped before a
   session object is built; check the traffic log (execute log display)
   instead to confirm a real deny decision.

Q: Where does config system vdom-property live — inside a specific VDOM's
   context, or somewhere else?
A: Global context only (config global), not inside any individual VDOM.

Q: What does set session 0 mean in vdom-property?
A: No override set — that VDOM draws from the shared global resource pool.
   It does NOT mean zero sessions allowed.

Q: diagnose sys session stat shows session_count=50, session_limit=50,
   clash=182. What does clash represent?
A: The count of connection attempts that were rejected because the
   session table was already full.

Q: Why did Guest-WiFi inherit root's session limit of 50?
A: Its config was cloned from root's when it was originally built, and
   root's explicit session override came along with the clone.

Q: What is the correct two-step remediation philosophy for a resource
   ceiling problem, per this session?
A: Baseline first, cap after — restore service immediately (e.g. remove
   the artificial cap), then measure real usage before setting a
   deliberate, evidence-based permanent limit.

Q: Does deleting a config system switch-interface object require
   detaching its members first?
A: No — the object can be deleted directly.

Q: Why is a stale session left in the table after a config fix a real
   problem, not just clutter?
A: It represents live, unauthorized, ongoing state established under the
   old (broken) config — it must be cleared to force re-evaluation and to
   actually close the exposure.

Q: Could a switch-interface object built inside root VDOM ever bridge an
   interface belonging to the isolated Trading VDOM?
A: No — each VDOM is a completely separate config tree; an object in one
   VDOM's config cannot reference an interface name that exists only in
   another VDOM's namespace.
