01 · OVERVIEW

Session Overview & Learning Objectives

This was a full boss-battle review of Study Guide chapter 03 (VLANs and VDOMs, pp. 54–81), run as two simultaneous, student-driven tickets with minimal hints. Both tickets turned out to be layered — a config check that looked clean on the surface, and a deeper mechanism underneath that required pushing past the obvious answer.

  • Recognize when traffic is being switched at Layer 2 instead of routed through the policy engine, using session-list evidence.
  • Understand how a software switch (config system switch-interface) can bridge two VLAN interfaces together and silently erase segmentation between them.
  • Confirm interface identity from a raw interface index using diagnose netlink interface list.
  • Distinguish a genuine policy deny from a Layer 2 bypass, using the traffic log rather than the session list.
  • Understand per-VDOM resource governance — Global Resources, Override Maximum, and where it actually lives in the CLI (config system vdom-property).
  • Practice the discipline of pivoting from config-layer checks to resource/state diagnostics once every config check comes back clean.

03 · TICKET 1

Ticket 1The Bridge That Ate Segmentation

Symptom: a support tech on VLAN 30 successfully RDP'd into a Finance workstation on VLAN 10 — traffic that should have been blocked by an existing, correctly-configured deny policy.

Think First
  • If a deny policy already exists and is correctly configured, what would have to be true for traffic to still get through?
  • What does it mean if a session shows real traffic flowing but no policy ID at all?

Step 1 — Establish the path

The first move on any reachability-that-shouldn't-exist ticket is the same diagnostic split taught back in Session 4/5: does the traffic even reach the firewall's routing/policy engine at all?

CLI — isolate the flow
diagnose sys session filter src 192.168.30.0 255.255.255.0
diagnose sys session filter dst 192.168.10.0 255.255.255.0
diagnose sys session list
Output — the smoking gun
hook=post dir=org act=noop 192.168.30.44:52210->192.168.10.87:3389(0.0.0.0:0)
hook=pre dir=reply act=noop 192.168.10.87:3389->192.168.30.44:52210(0.0.0.0:0)
misc=0 policy_id=0 pol_uuid_idx=0 auth_info=0 chk_client_info=0 vd=0
orgin->sink: org pre->post, reply pre->post dev=13->13/13->13 gwy=0.0.0.0/0.0.0.0

Two facts matter here, and they point in the same direction: policy_id=0 (never evaluated against a policy), and dev=13→13 both directions — the exact same interface index in and out. Real routed traffic changes egress interface as part of the routing decision; identical index both ways means this traffic never left one Layer 2 domain to begin with.

Step 2 — Identify interface 13

CLI — map index to name
diagnose netlink interface list
Output
if=OpsBridge family=00 type=1 index=13 mtu=1500 link=0 master=13
if=VLAN10-Finance family=00 type=1 index=9 mtu=1500 link=0 master=13
if=VLAN30-Support family=00 type=1 index=10 mtu=1500 link=0 master=13

master=13 on both VLAN interfaces means they've been enslaved into a bridge construct — OpsBridge, interface index 13 itself. Confirm with the config:

CLI — confirm the construct
show system switch-interface
Output
config system switch-interface
    edit "OpsBridge"
        set vdom "root"
        set intra-switch-policy implicit
        set member "VLAN10-Finance" "VLAN30-Support"
    next
end
The mechanism, precisely: a software switch (config system switch-interface) with intra-switch-policy implicit merges every member interface into ONE broadcast domain. It doesn't matter that VLAN10-Finance and VLAN30-Support already had valid, distinct 802.1Q tags — once both are made members of the same bridge, FortiGate switches between them at Layer 2 instead of routing between them at Layer 3. The policy table is never consulted, because nothing ever asked it a question.

Step 3 — Fix, clear, verify

CLI — remove the bridge
config system switch-interface
    delete "OpsBridge"
end
Deleting the object directly works without detaching members first — no need to unset member as a separate step.
CLI — clear the stale session (part of remediation, not housekeeping)
diagnose sys session filter src 192.168.30.0 255.255.255.0
diagnose sys session filter dst 192.168.10.0 255.255.255.0
diagnose sys session clear
CLI — verify via the traffic log, not the session list
execute log filter category traffic
execute log filter field srcip 192.168.30.44
execute log filter field dstip 192.168.10.87
execute log display
Output — correct enforcement confirmed
srcintf="VLAN30-Support" dstintf="VLAN10-Finance"
action="deny" policyid=12 service="RDP" reason="policy violation"

Distinct interfaces on each side, a real policy ID, an explicit deny action. That's the tell that this is now a genuine policy decision — not a different-looking absence of one.

04 · TICKET 2

Ticket 2The VDOM That Inherited a Ceiling

Symptom: Guest-WiFi VDOM totally dark — no internet, no reachability to root, nothing — since sometime the previous day. Worked fine right after the Session 7 build.

Think First
  • If every config-layer check you know how to run comes back correct, what category of problem is left?
  • Back in Session 7, Señor said: "Don't let it inherit whatever root's ceiling was." What was he actually warning about?

Step 1 — Exhaust the config layer

Every check from the Session 7 build came back clean on both sides:

CheckGuest-WiFi sideRoot side
VDOM link interfaceVDOM_Link1 — status upVDOM_Link0 — status up
Default route0.0.0.0/0 via VDOM_Link1 — present0.0.0.0/0 via wan1 — present
Firewall policyGuestVLAN → VDOM_Link1, enabled, NAT onVDOM_Link0 → wan1, enabled, NAT on
Everything you know how to check from the bridge project is correct. That's the actual signal: when the entire config layer is clean and the symptom persists, the fault isn't a misconfiguration — it's somewhere config commands can't see. Pivot to state and resource diagnostics.

Step 2 — Find the ceiling

CLI — per-VDOM resource overrides (global context only)
config global
config system vdom-property
show
Output
config system vdom-property
    edit "root"
        set session 50
        set policy 100
    next
    edit "Trading"
        set session 0
        set policy 0
    next
    edit "Guest-WiFi"
        set session 50
        set policy 100
    next
end
Read this carefully: 0 here means no override set — that VDOM draws from the shared global resource pool. It does NOT mean "zero sessions allowed." Trading's 0 is actually the safe/unlimited reading. Guest-WiFi's 50, on the other hand, is a real, explicit, hard cap — identical to root's own 50. That's not a coincidence: when Guest-WiFi was originally built, its config was cloned from root's, and root's deliberately-tight session override came along with it.
CLI — confirm the ceiling is actually being hit
config vdom
edit "Guest-WiFi"
diagnose sys session stat
Output — confirmed
vd-Guest-WiFi: session_count=50 session_limit=50

Table is exactly full. A guest wifi segment with real device volume will exhaust 50 concurrent sessions almost immediately under normal browsing load — every session beyond the first 50 gets rejected outright, which explains "nothing works at all," not "some things are slow."

Step 3 — Fix, and don't over-fix

CLI — remove the artificial cap
config global
config system vdom-property
edit "Guest-WiFi"
    set session 0
    set policy 100
next
end
Output — verified
vd-Guest-WiFi: session_count=1 session_limit=0
Baseline first, cap after. Setting session 0 restores service immediately without guessing at a new arbitrary number. The correct long-term move is logged as a ticket follow-up: monitor real Guest-WiFi usage over time (FortiAnalyzer/FortiView, or scripted polling of diagnose sys session stat), then set a deliberate override with headroom above the observed peak — not leave it uncapped forever, and not replace one guess with another.

06 · COMMON MISTAKES

Common Mistakes This Session Exposed

  • Trusting an empty session-list result without checking the active filter. A filter set for Ticket 1 was still active when Ticket 2's session list came back empty — the result was technically true but answering the wrong question. Clear filters (diagnose sys session filter clear) between distinct troubleshooting threads.
  • Reading policy_id=0 as one single meaning. It can mean "correctly denied, no session persisted" OR "never evaluated at all, session persisted with real traffic." The differentiator is whether a real, ongoing session exists — not the number itself.
  • Reading set session 0 in vdom-property as a hard zero cap. It means no override — shared global pool. The opposite reading is the trap.
  • Assuming a config object requires its members detached before deletion. Worth testing directly rather than adding an unnecessary step.
  • Treating a stale, live unauthorized session as optional cleanup. It's an active exposure — clearing it is part of the fix, not separate from it.
07 · SENIOR ENGINEERING

Senior Engineering Insights

  • When config is exhausted, the fault has moved categories. Interfaces up, routes present, policies enabled and correctly directional — that's not "keep re-checking," that's "you're now looking for state or resources, not misconfiguration."
  • Never clone a VDOM's config as a shortcut without auditing what came along for the ride. Guest-WiFi's outage traces directly back to a template-clone habit from Junior, months earlier — this is exactly the kind of thing that stays invisible until real load exposes it.
  • Fix now, measure before you commit to a permanent number. "Baseline first, cap after" applies to any resource-governance decision, not just FortiGate session limits.
  • A stale session after a config fix is not automatically re-evaluated. Clearing it is how you force the new config to actually apply to in-flight state.
08 · EXAM TIPS

Exam Tips

  • Any question describing identical ingress/egress interface (or index) with policy_id=0 AND real traffic counters is describing a Layer 2 bridge/switch bypass, not a policy issue.
  • Any question testing whether a genuine deny "shows up" — remember it usually shows in the traffic log with a real policy ID, but often does NOT persist as a session-list entry the way permitted traffic does.
  • vdom-property / Override Maximum values of 0 mean "unset, uses the shared pool" — a very common exam trap read as "zero allowed."
  • Config-object namespace isolation between VDOMs is absolute: an object built in one VDOM's config tree cannot reference an interface belonging to another VDOM, full stop — no exceptions, no cross-VDOM switch-interface members.
09 · TAKEAWAYS

Key Takeaways

1. A switch-interface bridge can un-do VLAN segmentation between two already-tagged VLANs, not just bundle physical ports — same-dev-index + policy_id=0 + real traffic is the tell.

2. When every config check is clean, the fault has moved to resources or state — pivot, don't re-check.

3. Per-VDOM resource overrides live in config system vdom-property, global context, and 0 means "no override," not "none allowed."

4. Fix service first, baseline before you commit to a permanent number.

5. VDOM isolation extends to config-object namespaces, not just interfaces/routes/policies — a switch-interface object simply cannot reach across a VDOM boundary.
