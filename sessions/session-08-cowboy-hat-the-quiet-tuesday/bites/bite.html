SESSION 8 · BITE · 5-MINUTE REVISION

Cowboy Hat: The Quiet Tuesday

The Two Faults

Ticket 1

VLAN 30 ↔ Finance leak. A software switch (OpsBridge) had both VLAN interfaces as members — bridged them at L2, so the correct deny policy never got evaluated at all.

Ticket 2

Guest-WiFi total outage. Session override cloned from root (session 50) — table filled up under real load, every new session rejected.

The Signature to Recognize

Same dev index in/out + policy_id=0 + real traffic = bridged/switched, never reached policy. Different from a genuine deny, which usually leaves no session at all — check the traffic log for that.

diagnose sys session filter src/dst → session list diagnose netlink interface list → find master=<idx> show system switch-interface → confirm the bridge

The Fix (Ticket 1)

config system switch-interface delete "OpsBridge" end diagnose sys session clear (against the active filter — clears the stale, live leak)

Resource Ceilings (Ticket 2)

0 = no override, draws from shared global pool. NOT "zero allowed." A real number (like Guest-WiFi's 50) is a hard cap.
config global config system vdom-property edit "Guest-WiFi" set session 0 next end diagnose sys session stat → session_count vs session_limit, watch "clash"
Baseline first, cap after. Unblock now with 0, monitor real usage, set a deliberate number later — never a permanent guess.

One Rule Worth Memorizing

VDOM isolation isn't just interfaces/routes/policies — it's config-object namespaces too. A switch-interface object built in one VDOM literally cannot see or reference an interface in a different VDOM. No link, no route, no shared object = no path, period.
