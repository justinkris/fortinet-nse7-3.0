SESSION 8 · TROUBLESHOOTING GUIDE

How to Think About Segmentation Gone Wrong

The emphasis is on the thinking, not the command list. Any of these commands can be handed to you in an exam question — the read is what's tested.

Symptom Vocabulary

You hear...Think...
"They can reach something they shouldn't"Either a policy is wrong (allow-by-mistake) OR the policy is never being consulted at all (bypass). Don't assume which until you check.
"Everything just stopped, all at once"Consider resource exhaustion (session/policy limits), not just a broken route or disabled policy — a hard ceiling produces total, sudden failure, not gradual degradation.
"It worked yesterday, nothing changed"Something DID change — either a hidden config edit (cowboy-hat style sabotage in this course, or a real colleague's undocumented change in the field) or a growth-driven threshold was crossed (traffic volume finally exceeded a ceiling that was always there).

Questions to Ask Before Touching Config

  • Does this traffic even reach the firewall's routing/policy engine, or is something intercepting it earlier (switch bundling, bridging)?
  • Is the symptom "wrong decision" (policy allowed/denied incorrectly) or "no decision happened" (never evaluated)?
  • Have I checked every layer of config I know how to check — and if they're all clean, am I actually looking in the wrong category of problem?
  • Is there leftover state (a stale session, an active filter) that could make my next check misleading?

Verification Commands & Expected Output

CommandWhat it tells you
diagnose sys session filter src/dst ... ; session listWhether a session exists at all, its policy_id, and ingress/egress dev index.
diagnose netlink interface listMaps an index number to an interface name; master=<idx> reveals bridge/switch-interface enslavement.
show system switch-interface / show system virtual-switchConfirms which construct (software or hardware) and its exact member list.
execute log filter ... ; execute log displayConfirms a genuine policy decision (policyid, action, distinct interfaces) — the correct tool when the session list is ambiguous or empty.
diagnose sys session statPer-VDOM session_count vs session_limit, plus clash (rejected attempts against a full table).
config global / config system vdom-property / showPer-VDOM resource overrides. 0 = no override (shared pool), a real number = hard cap.

Decision Tree

Reachability that shouldn't exist, or reachability that's suddenly gone │ ▼ diagnose sys session filter (src/dst) → session list │ ├─ No session at all │ │ │ ▼ │ execute log filter/display (traffic log) │ │ │ ├─ Deny entry with real policyid → correctly denied, working as intended │ └─ Nothing at all, not even a deny log → below Layer 3 entirely │ (VLAN tagging / trunk / ARP — Session 4/5 territory) │ ├─ Session present, policy_id=0, real traffic, SAME dev index in/out │ │ │ ▼ │ diagnose netlink interface list → find master= │ │ │ ▼ │ show system switch-interface → confirm bridge + member list │ │ │ ▼ │ Remove offending member(s) or delete the bridge object │ → clear stale session → verify via traffic log │ └─ Session present, real policy_id │ ▼ Investigate policy logic: ordering, srcaddr/dstaddr, service, first-match-wins semantics (earlier session material) If config layer (interface/route/policy) is fully clean and symptom persists: │ ▼ diagnose sys session stat → session_count vs session_limit, clash │ ▼ config global / config system vdom-property / show → look for a suspiciously round or duplicated override value (compare against other VDOMs — a clone artifact often matches another VDOM's number exactly)

Common Dead Ends

  • Re-checking the same config layer a second and third time when it was already confirmed clean — the fault has moved categories, not hidden itself deeper in the same layer.
  • Assuming policy_id=0 always means the same thing — it's ambiguous without also checking whether a real, persistent, traffic-bearing session exists.
  • Trusting an empty session-list result without confirming no leftover filter is still active from a previous check.

Likely Root Causes for This Symptom Class

  1. Interfaces accidentally bundled into a shared switch/bridge construct (physical ports, Session 5; VLAN interfaces, this session).
  2. Policy ordering / first-match-wins issues (earlier course material — always still worth ruling out first if a session DOES show a policy_id).
  3. Resource ceilings — cloned or inherited overrides nobody audited (this session's Ticket 2).
  4. Layer 2 issues below the firewall entirely — tagging, trunk, ARP (Session 4/5 material, ruled out here by the presence of real sessions/traffic).

Exam Traps

Trap 1: assuming policy_id=0 always means "correctly denied." Pair it with whether a real, ongoing session with traffic exists — that's the actual differentiator.
Trap 2: reading set session 0 in vdom-property as "zero sessions allowed" instead of "no override, uses shared pool."
Trap 3: assuming a config object (like a switch-interface bridge) could ever span two VDOMs. It structurally cannot — each VDOM is a fully separate config tree.
Trap 4: forgetting that a session established BEFORE a config fix is not automatically re-evaluated — it must be explicitly cleared.

Personalized Framework for This Student

  1. Restate the symptom precisely: wrong access, or no access — and is it total or partial?
  2. Isolate the flow: diagnose sys session filter + session list, and confirm no leftover filter is skewing the result.
  3. Read policy_id AND dev index together, not policy_id alone.
  4. If evidence points at Layer 2 bypass: map the index, confirm the construct, fix the membership.
  5. If config layer is entirely clean: pivot deliberately to resource/state diagnostics rather than re-checking config.
  6. Fix, clear any stale state, and verify via the correct tool for the mechanism (traffic log for policy decisions, session stat for resource ceilings) — not just the same command you started with.
