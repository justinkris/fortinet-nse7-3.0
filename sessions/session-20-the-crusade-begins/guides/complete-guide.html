Story Recap

Malware over HTTPS is exactly as protected from our security stack as legitimate banking traffic.

1 · From SSL to TLS

SSL 2.0/3.0 and TLS 1.0/1.1 are deprecated. TLS 1.2 remains the broad-compatibility recommendation; TLS 1.3 offers stronger security and faster handshakes (fewer round trips — HTTP/3 uses TLS 1.3 by default). Secure protocol names: HTTP+TLS=HTTPS, SMTP+TLS=SMTPS, FTP+TLS=FTPS.

config system global
    set admin-https-ssl-versions tlsv1-2 tlsv1-3
end

show full | grep -f tls    # find every TLS-related setting in the config
Interop ruleBoth endpoints must share at least one supported TLS version — a version mismatch means no session at all, not a degraded one.

2 · What the Envelope Reveals (no decryption)

Before any application data is encrypted, the TLS handshake exposes two pieces of intelligence in cleartext:

  1. SNI — in the ClientHello: the hostname the client says it wants.
  2. Server certificate — CN/SAN, issuer, validity: who the server proves it is.
The Sealed EnvelopeThe courier cannot read the letter, but the front of the envelope is written in plain ink. The client writes the destination on it (SNI); the server stamps it with a notarised seal (certificate). Certificate inspection reads the envelope. Full inspection opens the letter.

The two nuanced rowsApplication control: officially Yes under certificate inspection — it identifies the application from the SNI. But it cannot see the action: "Google Drive" yes, "file upload to Google Drive" no. HTTPS-attack protection: Limited, not None — invalid/untrusted certificates and known-malicious SNI destinations are still caught; payload-borne attacks behind a clean certificate are not. "Certificate inspection provides NO protection against HTTPS threats" is a classic distractor.

5 · Mode vs Depth — Two Independent Axes

Full inspection is supported in BOTH flow-based and proxy-based inspection modes. A flow-based policy can decrypt in-stream. What is proxy-only:

  • Complete SSL proxying — FortiGate terminates the client session and originates its own session to the server (two independent TCP connections).
  • Protocol port mapping — catching standard services on non-standard ports (e.g., SMTPS on 10587 instead of 587). Requires full inspection + proxy mode + the mapping configured. Flow-based inspection assesses all ports regardless.
Exam trap"Full SSL inspection requires proxy-based inspection mode" = FALSE. Depth (cert/full) and mode (flow/proxy) are independent choices.

6 · How Full Inspection Works — The Authorised Man-in-the-Middle

CLIENT ←——TLS——→ FORTIGATE ←——TLS——→ SERVER
        (FGT's forged cert)      (server's real cert)

TLS was designed to prevent exactly this. The FortiGate does it anyway — with authorisation — via three requirements:

  1. It re-signs (forges) the server's certificate on the fly, copying its identity details, and presents this to the client.
  2. The forgery is only trusted if the signing CA certificate is in every client's trusted root store. Skip this and every HTTPS site shows a warning simultaneously.
  3. The FortiGate must hold the CA private key — the thing that makes the forgery valid rather than merely attempted.
Senior lessonNever enable full inspection before the CA is distributed to clients. The order of operations is the deployment plan. Diagnostic fingerprint: warnings on one site → one bad server certificate. Warnings on all sites → the clients don't trust the forger.

7 · SNI vs SAN — One Problem, Two Solutions

The shared problem: one IP address, many hostnames. The server must present the right certificate — but which one?

Decode keyExam questions hide the answer in quantity words: "a separate certificate for each domain" → SNI (selection). "A single certificate covering multiple domains" → SAN (multi-name validation).

Troubleshooting Workflow (preview — deepened in the troubleshooting guide)

  1. Warnings on all sites at once? → CA missing from client trusted stores.
  2. Web filtering works but AV/IPS sees nothing on HTTPS? → policy is on certificate inspection; payload profiles need full inspection.
  3. Expecting SSL proxying / port-mapping behaviour but not seeing it? → check the policy's inspection mode; those are proxy-only.

Common Mistakes

  • Rounding "Limited" HTTPS protection to "None" (or to "Enough").
  • Conflating full inspection with proxy mode.
  • Inverting SNI/SAN when the question is framed around certificate count.
  • Enabling full inspection before CA distribution.
  • Believing app control under cert inspection sees user actions — it identifies apps only.

Key Takeaways

  • Certificate inspection = read the envelope: SNI + certificate, cheap, no decryption. Enough for web filtering and app identification.
  • Full inspection = open the letter: decryption via authorised MITM. Required for AV, IPS, DLP, and action-level app control.
  • Depth and mode are independent; only complete proxying and port mapping demand proxy mode.
  • The MITM engine: forged certificate + CA in trusted stores + private key on the FortiGate.
  • SNI selects among many certificates; SAN makes one certificate answer for many names.
