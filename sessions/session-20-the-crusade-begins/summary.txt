TECHNICAL CONCEPTS COVERED
--------------------------
1. TLS handshake visibility WITHOUT decryption: TCP handshake → ClientHello (carries
   SNI = client's intended hostname) → ServerHello → server Certificate (CN/SAN,
   issuer, validity). These plaintext elements are the entire raw material of
   certificate inspection. Student produced this sequence cold, correct order,
   unprompted, including "TCP handshake happens first."
2. CERTIFICATE INSPECTION vs FULL INSPECTION (the p.166 table). Student filled the
   entire table from his own derivation: handshake inspection/SNI parsing/cert
   validation/URL extraction/web filtering = both modes; decryption/encrypted-payload
   inspection/SSL proxying = full only. Two refinements supplied by Señor:
   (a) Application control = official table says YES for cert inspection (identifies
   app by SNI); student wrote "NOT FULLY" — engineering-honest but exam answer is Yes.
   App vs ACTION distinction taught: SNI identifies "Google Drive," only decryption
   distinguishes viewing vs uploading. (b) Protection from HTTPS attacks under cert
   inspection = LIMITED, not No (catches bad certs + known-bad SNI destinations;
   blind to clean-envelope payload attacks). "Cert inspection provides NO HTTPS
   protection" flagged as an exam distractor.
3. WHICH SECURITY PROFILES NEED WHICH DEPTH (student's own pre-reading Q1): web
   filtering + app control work under cert inspection (from SNI, cert-CN fallback if
   SNI absent); AV, IPS, DLP need full inspection (payload access).
4. INSPECTION MODE vs INSPECTION DEPTH ARE INDEPENDENT AXES (pre-reading Q2): full
   inspection works in BOTH flow-based and proxy-based modes. Proxy-only specifics:
   complete SSL proxying (two independent TCP sessions) and protocol port mapping
   (e.g., SMTPS on 10587). Flow mode decrypts in-stream without splitting the
   connection. "Full inspection = proxy mode only" = FALSE, and was Q2 of the exam.
5. FULL INSPECTION MECHANISM (MITM): student DERIVED the whole engine himself —
   FortiGate forges/re-signs the server's certificate on the fly, presents it to the
   client; requires the signing CA in every client's trusted root store; FortiGate
   must possess the CA private key. Two TLS sessions (client↔FGT, FGT↔server) with
   plaintext in the gap. "Authorised man in the middle."
6. SNI vs SAN (pre-reading Q3, pp. 167–168 pulled forward at student request): SNI =
   client-side, lives in ClientHello, triggers CERTIFICATE SELECTION when a server
   holds many certs on one IP. SAN = server-side, a list of names INSIDE one
   certificate, one cert validating many domains. SAN supports wildcards
   (*.domain.com); SNI does not. Student's original framing "routing vs validating"
   corrected to "certificate selection vs multi-name validation" — nothing is routed.
