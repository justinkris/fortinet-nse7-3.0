SESSION 20 · TROUBLESHOOTING GUIDE

Troubleshooting SSL Inspection

The Senior Mindset

SSL inspection faults come in exactly three families. Before touching anything, classify which family you're in:

  1. Trust faults — the client rejects the FortiGate's forged certificate (CA distribution problems).
  2. Depth faults — a security profile is blind because the policy's inspection depth is shallower than the profile needs.
  3. Mode faults — a behaviour you expect (proxying, port mapping) belongs to proxy mode and the policy is flow-based.
First question, always"Warnings, or silence?" Users seeing certificate warnings = trust fault. Security profiles silently missing threats over HTTPS = depth fault. Feature configured but not acting = mode fault.

Symptom 1 — Certificate warnings on EVERY site, immediately after enabling full inspection

Questions to ask: All users or some? All sites or one? Did we distribute the CA, and to which platforms/browsers?

Likely root cause: the FortiGate's signing CA certificate is not in the clients' trusted root stores. The clients don't distrust the sites — they distrust the forger.

FingerprintALL sites warn → missing/untrusted CA on the client. ONE site warns → that server's real certificate is genuinely bad (expired, untrusted, name mismatch) and full inspection is faithfully passing the problem through.

Dead end to avoid: disabling inspection to "fix" the warnings. That restores silence, not security — you've retreated from the crusade, not won it.

Symptom 2 — Web filtering blocks correctly, but IPS/AV never fires on HTTPS traffic

Question to ask: what inspection depth is on the policy?

Root cause: certificate inspection. Web filtering works from the SNI (envelope); IPS/AV need the payload (letter). The study guide's own use case: PHP code injection over HTTPS with cert inspection → forward-traffic logs show the HTTPS access, zero IPS security events. Fix: full inspection (or Protecting SSL Server for inbound).

# Verify what the policy is actually applying
show firewall policy <id>        # look at set ssl-ssh-profile
show firewall ssl-ssh-profile <name>   # cert-inspection vs deep-inspection settings
Expected evidenceLog & Report → Forward Traffic: HTTPS sessions present, normal. Log & Report → Security Events → Intrusion Prevention: empty for those flows. That pairing — traffic seen, threats unseen — is the depth-fault signature.

Symptom 3 — Users bypass inspection via a non-standard port

Scenario: SMTPS to port 10587 sails through uninspected while port 587 is controlled.

Root cause chain: protocol port mapping not configured — and it only functions when ALL of: full SSL inspection, proxy-based inspection mode, and the port-mapping setting configured, on an outbound multiple-clients policy. Flow-based inspection assesses all ports regardless of mapping settings.

Senior habit: watch logs for unexpected ports; enable mappings incrementally (each one costs resources).

Symptom 4 — "I enabled full inspection but nothing is being proxied"

Root cause: mode fault. Full inspection in flow mode decrypts in-stream — one TCP connection, no session splitting. Complete SSL proxying (two independent TCP sessions) is proxy-mode behaviour. Decide whether you need decryption (either mode) or proxying (proxy mode).

Decision Tree

HTTPS-related complaint │ ├─ Users see certificate WARNINGS? │ ├─ On ALL sites → CA not trusted by clients → distribute CA, verify store │ └─ On ONE site → that server's real cert is bad → validate upstream cert; │ inspection is working as designed │ ├─ Security profile SILENT on HTTPS threats? │ ├─ Web filter works, AV/IPS/DLP silent → depth fault: │ │ policy is cert-inspection → payload profiles need FULL inspection │ └─ Even web filter blind → is ANY ssl-ssh-profile applied to the policy? │ └─ Feature configured but INACTIVE? ├─ Port mapping ignored → requires full inspection + PROXY mode + mapping set └─ No session splitting → proxying is proxy-mode-only; flow decrypts in-stream

Common Dead Ends

  • Blaming the destination website for warnings that appear on every site (it's your CA).
  • Adding more IPS signatures when the real issue is that IPS never sees plaintext.
  • Switching the policy to proxy mode to "make full inspection work" — it already works in flow mode; identify the actual missing behaviour first.
  • Confusing "we validate certificates" with "we inspect content."

Exam Traps

  • "Certificate inspection provides no protection against HTTPS attacks" → false; it's Limited.
  • "Full inspection requires proxy-based mode" → false; both modes support it.
  • "App control cannot function under certificate inspection" → false; it identifies apps by SNI.
  • Scenario says "separate certificate for each domain" → the mechanism is SNI, not SAN.
  • All-sites warnings scenario → the answer is CA distribution, not TLS versions, not SNI settings.
Señor's ruleClassify the family first — trust, depth, or mode. A fault correctly classified is half fixed; a fault misclassified generates three unnecessary changes before breakfast.
