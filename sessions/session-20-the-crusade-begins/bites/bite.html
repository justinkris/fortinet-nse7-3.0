The Crusade Begins — SSL Inspection

The One Idea

Encryption protects the envelope, not the letter. Certificate inspection reads the envelope (SNI + server certificate, no decryption). Full inspection opens the letter (decryption via authorised man-in-the-middle).

Visible Without Decryption

  • SNI — in the ClientHello: what the client asked for
  • Server certificate — CN/SAN, issuer, validity: what the server proves

The Table (own it, don't memorise it)

FeatureCertFull
Handshake inspection / SNI parsing / cert validation / URL extractionYesYes
Web filtering / Application controlYesYes
Decryption / encrypted-payload inspection / SSL proxyingNoYes
Protection from HTTPS attacksLimitedYes
Two nuances: App control under cert inspection = Yes officially — it names the app (SNI) but never the action (view vs upload). HTTPS protection under cert inspection = Limited (bad certs + bad destinations caught), never "None."

Profiles → Depth

Web filter + app control: work on the envelope. AV, IPS, DLP: need the letter open (full inspection).

Mode ≠ Depth

Full inspection works in both flow and proxy modes. Proxy-only: complete SSL proxying and protocol port mapping (SMTPS on 10587 etc.).

The MITM Engine (full inspection)

CLIENT ←—TLS—→ FGT ←—TLS—→ SERVER
     forged cert      real cert
  • FGT re-signs/forges the server cert on the fly
  • Signing CA must be in every client's trusted root store
  • FGT holds the CA private key
Fingerprint: cert warnings on every site at once = missing CA in clients' stores. One site = one bad server cert. Deploy the CA before enabling full inspection.

SNI vs SAN — decode by counting

  • SNI = client's field in ClientHello → server selects one of many certificates ("separate cert per domain")
  • SAN = list of names inside one certificate → one cert validates many domains; supports wildcards (SNI doesn't)

TLS Versions

SSL 2.0/3.0, TLS 1.0/1.1: deprecated. TLS 1.2 = compatibility. TLS 1.3 = faster + stronger. set admin-https-ssl-versions tlsv1-2 tlsv1-3
