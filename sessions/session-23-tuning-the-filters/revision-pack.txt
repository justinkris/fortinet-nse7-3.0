──────────────────────────────────────────────────────────
  QUICK SUMMARY (one page)
──────────────────────────────────────────────────────────

Web filter is a LIVE CONVERSATION with FortiGuard. App control and ISDB
are STORED KNOWLEDGE. When FortiGuard connectivity or the licence fails,
these three tools fail asymmetrically — and that asymmetry is your
diagnostic key.

FAILURE MODEL
-------------
Web filter DB:   Not local — queries FortiGuard live over HTTPS/443
                 (anycast). Only a rating CACHE lives on the FortiGate.
                 → Fails FAST when FDS is down (cache decay).
App control DB:  Local. Periodic updates.
                 → Frozen signatures during outage; degrades SLOWLY.
ISDB DB:         Local. Periodic updates.
                 → Frozen vendor IPs during outage; degrades SLOWLY.

THE CACHE
---------
FortiGate caches recent web-filter verdicts. Masks outage temporarily
(explains why the LoFinance weekend was quiet). But cache is a DECAYING
asset — entries TTL out, new sites always miss, hit rate slopes down
monotonically. Silent, gradual degradation.

FAIL-OPEN VS FAIL-CLOSED
------------------------
Fail-open (default) = permit AFTER giving up waiting. Fail-closed = deny.
Trap: fail-open ≠ instant permit. FortiGate holds the packet for the
request timeout window; browsers time out first. Fail-open governs the
VERDICT, timeout governs the EXPERIENCE.

RESILIENCE POSTURE
------------------
FortiManager licensed as local FDS. `set include-default-servers disable`
under `config system central-management`. Web filter and app control
rating lookups go inward at LAN speed. Decouples LoFinance from public
FortiGuard availability. Renewal = licence. FMG-as-FDS = insurance.

DIAGNOSTIC ORDER (never change state before reading it)
--------------------------------------------------------
1. `diagnose autoupdate versions`      — paperwork
2. `diagnose debug rating`             — live conversation
3. `diagnose webfilter fortiguard cache dump` — current belief
4. `get webfilter status`              — profile-side view
5. `get system fortiguard`             — configuration view
NEVER as a diagnostic: `execute update-now` (action, not read)

CLI — DIAGNOSTIC PATHS
----------------------

  → Virus Definitions / Application Definitions / Internet-service
    Full Database sections, each with Version/Contract Expiry/Last
    Update Attempt/Result. Web filter absent by design.
diagnose debug rating
  → License / Service / Status (Enable|Disable)
  → Per-server RTT / Flags (F=Functional, DI=Down/Invalid) /
    Curr Lost / Total Lost
diagnose webfilter fortiguard cache dump
  → Local cache entries with category IDs and TTLs
get webfilter status
  → Profile-side view
get system fortiguard
  → Configured servers, ports, override server

DEFAULTS / TIMERS / BEHAVIOURS
-------------------------------
FortiGuard fail policy:     fail-open (default, per model/version)
Cache mode:                 TTL (per-entry TTL from FortiGuard)
Cache location:             local (RAM) on FortiGate
Rating server protocol:     HTTPS/443 (anycast pool)
Web filter DB location:     FortiGuard (never local)
App Definitions location:   Local flash
ISDB Full DB location:      Local flash

DESIGN RULES
------------
· ISDB is a Policy & Object — NEVER a profile. Referenced on the policy.
· Web filter and app control are BOTH security profiles — attach both
  to a policy when both verdicts are needed.
· Exceptions live in POLICY structure (who), not profile edits (what).
· Per-group differences = two profiles + two policies (not one profile
  with an override).
· Enterprise-grade: FortiManager as local FDS + expiry warnings + human.
· Application Detection-Based SD-WAN must be toggled in Feature
  Visibility. ISDB in SD-WAN needs no such toggle.


──────────────────────────────────────────────────────────
  FLASHCARDS (question → answer)
──────────────────────────────────────────────────────────

Q: Where is the web filter database stored?
A: On FortiGuard servers. FortiGate holds only a cache of recent
   verdicts and queries FortiGuard live for uncached URLs.

Q: Where is the Application Definitions database stored?
A: Locally on the FortiGate flash. Updated periodically from FortiGuard.

Q: Where is the ISDB stored?
A: Locally on the FortiGate flash. Updated periodically from FortiGuard.

Q: If FortiGuard is unreachable for 3 days, which features continue to
   work?
A: App control and ISDB continue on their most-recent local databases
   (degrading slowly as signatures/IPs go stale). Web filter fails
   fast as its cache decays with no source to refill it.

Q: What does "License Expired" in `diagnose autoupdate versions` mean
   for app control?
A: The database will not receive further updates. The feature continues
   to work using the last-downloaded signatures. Licence expired ≠
   feature broken.

Q: What is FortiGate's default behaviour when a web-filter rating
   lookup does not receive a response?
A: Fail-open — permit the traffic AFTER the request timeout expires.

Q: Why does fail-open still cause user hangs during a FortiGuard outage?
A: Fail-open kicks in only after the request timeout. The user's
   browser (and patience) run out first. Fail-open governs the verdict;
   the timeout governs the experience.

Q: What command shows the version, contract expiry, and last update
   result for locally-stored FortiGuard databases?
A: `diagnose autoupdate versions`

Q: What command shows the live conversation with FortiGuard rating
   servers?
A: `diagnose debug rating`

Q: In `diagnose debug rating`, what do the flags "DI" mean?
A: Down / Invalid — the server is unreachable.

Q: Why does web filter not appear in `diagnose autoupdate versions`?
A: Because the web filter database is not stored locally. There is no
   version to report. This is proof-by-absence of the on-demand model.

Q: How is ISDB referenced in an SD-WAN rule?
A: As the destination match on the SD-WAN rule (via internet-service-
   name). ISDB is not a security profile.

Q: What must be enabled before application-based SD-WAN steering can
   be used?
A: Application Detection-Based SD-WAN, under System > Feature Visibility.
   Does NOT apply to ISDB — only to app-control-based steering.

Q: How do you decouple LoFinance's web-filter operation from public
   FortiGuard availability?
A: License FortiManager as a local FDS. Under
   `config system central-management`, set
   `include-default-servers disable` and add a rating-type server entry
   pointing at the FortiManager IP.

Q: Facebook is blocked by the trading floor's web-filter profile, but
   the Facebook app on trader phones still works. Why?
A: Web filter matches domains/URLs (HTTPS SNI/SAN/CN). The Facebook app
   scatters traffic across API endpoints, MQTT brokers, and CDNs — the
   "www.facebook.com" hostname is never used. Add app control (Social.
   Media category or the Facebook_App signature) to close the gap.

Q: Marketing needs Facebook, trading floor must not. Same FortiGate.
   What's the design?
A: Two web-filter profiles (one blocks Facebook, one permits it), two
   policies (each with the appropriate profile attached to the correct
   source). Not one shared profile with an exception.

Q: What is Junior's move that a senior would never do during diagnosis?
A: `execute update-now` as a diagnostic step. It's an action, not a
   read. It changes state and destroys evidence.

Q: What is the difference between "the database is expired" and "the
   feature is broken"?
A: Expiry stops updates. The feature continues to run on its last
   downloaded database. App control and ISDB stay functional; web
   filter does not, because it has no local database to fall back on.

Q: How would you explain the FortiGuard subscription value to Penny?
A: "It buys Fortinet's daily research team keeping our firewall's
   picture of the world current. Without it, the firewall would still
   enforce your rules, but against a picture of the internet that's a
   week old, then a month, then a year." (Value framing, not cost-
   avoidance framing.)
