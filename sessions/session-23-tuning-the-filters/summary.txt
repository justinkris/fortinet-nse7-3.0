TECHNICAL CONCEPTS COVERED
--------------------------
1. WEB FILTER FortiGuard model: NO local DB. Live rating lookups over HTTPS/443 to
   the FortiGuard anycast pool. Only a CACHE of recent verdicts lives locally.
2. CACHE DECAY: cache is a decaying asset. Entries have a TTL. FDS down → cache
   entries can never be refilled → hit rate slopes down monotonically → outage
   feels progressively worse over hours/days.
3. Fail-open (default) vs fail-closed: verdict philosophy. Fail-open ≠ instant
   permit — it's "permit AFTER giving up waiting." User experience is governed by
   the REQUEST TIMEOUT, not by the fail policy.
4. `diagnose debug rating` output structure: License, Service, Status, per-server
   RTT/Flags/Curr Lost/Total Lost. Status=Disable is the internal kill switch
   FortiGate flips the moment the license expires.
5. `diagnose autoupdate versions` output: Virus/Application/Internet-service
   sections each have Version, Contract Expiry Date, Last Update Attempt, Result.
   Web filter NOT in that output — proves absence-of-local-DB by absence-from-list.
6. FortiManager as local FDS: `set include-default-servers disable` under
   `config system central-management`, plus rating server entry pointing at FMG IP.
   Decouples the FortiGate's ability to filter from the internet's ability to
   answer. This is the enterprise-grade resilience posture.
7. Web filter profile CLI: `config webfilter profile` → `config ftgd-wf` → `config
   filters` → per-category `set action block/monitor/allow`. Category IDs (26=Social
   Networking, 62=Gambling).
8. Application list CLI: `config application list` → `config entries` → per-signature
   or per-category action. NO FortiGuard live-lookup dependency; runs on the local
   Application Definitions database.
9. ISDB reference on policy: `config firewall internet-service-name` for the object,
   then `set internet-service enable` + `set internet-service-name` on the policy
   itself. NOT a profile attachment.
10. Verification order under fire: `diagnose autoupdate versions` (paperwork) →
    `diagnose debug rating` (live conversation) → `diagnose webfilter fortiguard
    cache dump` (current belief) → `get webfilter status` (profile-side view).
11. Feature Visibility gap CLOSED: student retrieved "System > Feature Visibility"
    unprompted on Q9 (was Q3 gap in S22).

TROUBLESHOOTING APPROACH TAUGHT
-------------------------------
Silent-degradation pattern: nothing changed in config, symptoms appeared gradually.
Diagnostic order: (1) paperwork check first (`diagnose autoupdate versions`) —
proves contract state without changing anything, (2) live-conversation check
(`diagnose debug rating`) — proves reachability without changing anything,
(3) current-belief check (cache dump) — proves what the box thinks right now.
Only THEN consider changes. Junior's move (`execute update-now`) explicitly
called out as a diagnostic anti-pattern: it's an action, not a read, and it
destroys evidence.

Also taught: SYMPTOM SCOPE reading (see Q2 loss below). "No logs for Zoom
specifically" — surgical scope — points at a signature-DB problem, NOT an engine
problem. Engine-wide failure would show broad symptoms, not one-app symptoms.
