Session 23 · Complete Guide

Tuning the Filters

Web filter / app control / ISDB configuration & FortiGuard dependencies · Study Guide 06 pp. 171–176

Session Overview

What this session actually taught

Session 22 answered what each of the three tools sees and where it lives. Session 23 answered what each one depends on to keep seeing. The answer to that second question is a subscription — and when a subscription lapses, the three tools fail in asymmetric ways that tell you exactly which one is which.

The core insight, in one line: web filter is a live conversation; app control and ISDB are stored knowledge. Conversations die instantly when the line goes down. Stored knowledge goes stale slowly. This asymmetry is your diagnostic key when a FortiGuard-related outage strikes.

Session mental model

Web filter dies fast (cache decay). App control and ISDB degrade slowly (frozen signatures). A silent, gradual browsing outage with app control still working = FortiGuard connectivity or licence.

Story Recap

Monday morning at LoFinance

Weekend queue: zero tickets for 48 hours — Riley screenshotted it as historical evidence. At 9:02 AM Monday, every trading-floor browser began hanging. Not blocked. Not denied. Just spinning. Queue went 0 → 31 in eleven minutes.

The strange part: the Facebook app was still being blocked on phones. Slack worked. The trading platform worked. Only browsing hung. Junior loudly confirmed "nothing changed — I diffed the config", and for once was correct.

Señor arrived at 9:14, went directly to his own desk, lifted an unopened envelope postmarked eleven days ago — FortiGuard Services Renewal — and set it on the whiteboard ledge next to the cacti.

Señor: "Junior is correct. The configuration did not change. Ask yourself what else our firewall depends on that is not written in its configuration."

By the end you should be able to explain...

  • How the FortiGate rating cache masks the outage temporarily, and why the cache is a decaying asset
  • The difference between fail-open and fail-closed as security philosophies — and why fail-open still causes user hangs
  • How FortiManager as local FDS decouples LoFinance from public-FortiGuard availability
  • The senior diagnostic order: paperwork → live conversation → current belief → then, and only then, configuration

Concept 1 — Where the knowledge lives

Local database vs live lookup

The three tools have fundamentally different consumption models for the FortiGuard subscription they share.

FeatureWhere the data livesHow it consumes FortiGuardFailure mode when FDS is down
Web filterNot local — FortiGuard serversLive rating lookup per unrated URL, over HTTPS/443 (anycast)Cache-miss hang; degrades fast as cache ages out
Application controlLocal databasePeriodic signature-DB updateFeature keeps working; signatures go stale slowly
ISDBLocal databasePeriodic vendor-IP DB updateFeature keeps working; vendor IPs go stale slowly
Why this asymmetry exists
App-signature and vendor-IP databases are finite — thousands of entries, structured, fits in flash. The web is billions of URLs that change by the minute — no firewall on Earth stores that locally. So web filter ships the question out.
Mental model

Local = survives the outage. On-demand = dies with it.

Concept 2 — Why the weekend looked healthy

The rating cache and its decay curve

If every web-filter decision required a live round-trip to FortiGuard, browsing would feel dial-up-slow on a good day. So FortiGate keeps a cache of recent verdicts — sites already rated in this session or the last few days. Cache hits are instant; cache misses trigger a live lookup.

The cache masks a FortiGuard outage for a while. That's why the LoFinance weekend was quiet — traders were hitting sites they'd already visited that week. Every lookup was a cache hit. No question ever left the box.

But the cache is a decaying asset. Entries have a TTL. New sites are visited every minute. If FortiGuard is unreachable:

  • Aged-out entries can never be replaced — the source that would refill them is down
  • New sites always miss — nothing populates the cache for unfamiliar destinations
  • Hit rate slopes down monotonically — the situation only gets worse
Senior insight
Silent, gradual degradation is the failure mode senior engineers hate most. No alert fires. No config changes. No log says "FortiGuard subscription lapsed." The system just quietly stops being able to answer new questions and rides on its memory until the memory runs out.
Concept 3 — What FortiGate does while waiting

Fail-open vs fail-closed vs the timeout

Fail-open (default)

Availability wins

"If FortiGuard can't answer, let it through and log it."

Business continues; no spinners. But uncategorised destinations pass unfiltered for the whole outage. Sensible for a retail branch, where a broken branch is worse than an hour of relaxed filtering.

Fail-closed

Security wins

"If FortiGuard can't answer, deny it."

Policy holds. Nothing uncategorised gets through. Users see instant failure. Sensible for classified networks or pharma research, where a permitted piece of malware is worse than an angry user.

The trap most engineers hit
Fail-open ≠ instant permit. It means "permit AFTER giving up waiting." FortiGate holds the packet, sends a rating request, waits for the timeout to expire, then applies fail-open. But the browser and the user run out of patience first. To them, "waiting forever" and "blocked" look identical.
The distinction that matters

Fail-open governs the verdict. Timeouts govern the experience. A cache miss with FortiGuard unreachable ≈ a 20-second hang even under fail-open.

Concept 4 — Reading the output

"License Expired" does not mean broken

When you run diagnose autoupdate versions during a subscription lapse, you'll see "License Expired" against multiple databases at once. Junior's read: "Three things are broken!" Senior read: "Three databases have stopped updating; two features are still fully operational, one is not."

Notice what's not in that output — web filter has no line at all. Because it was never a locally-stored database. Its state is proven by absence from this command's output, plus the shared-subscription expiry visible on its siblings.
The distinction

License expired ≠ feature broken. App control / ISDB → degrade slowly (stale signatures). Web filter → degrades fast (cache decay), then hangs.

Concept 5 — Behavioural evidence

Reading diagnose debug rating

Where autoupdate versions shows paperwork, diagnose debug rating shows the live conversation with the FortiGuard rating servers. This is where you catch the actual failure.

diagnose debug rating — the money shot
Locale       : english
License      : Expired
Service      : Web-filter
Status       : Disable                    ← FortiGate has stopped trying

Num. of servers : 3
Protocol        : https
Port            : 443
Anycast         : Enable

  Server IP       Weight   RTT   Flags   TZ   Curr Lost   Total Lost
  173.243.138.194   10     -1    DI      -8      156         156
  208.184.237.68    10     -1    DI      -8       89          89
  96.45.33.68       10     -1    DI      -8       41          41

Read the tells: every server shows RTT = -1 (no round-trip), Flags = DI (Down, Invalid), and Curr Lost = Total Lost — meaning every recent request has been lost with none recovered. At the top: License: Expired and Status: Disable.

The killer line
Status: Disable — FortiGate isn't even trying to talk to FortiGuard for web ratings anymore. It disabled the service internally the moment the licence expired. Every cache miss now walks into a wall.
Configuration

The three profiles, side by side

Web filter profile — the container of category verdicts

config webfilter profile
config webfilter profile
    edit "TF-Web"
        set inspection-mode flow
        set feature-set flow
        config ftgd-wf
            config filters
                edit 1
                    set category 26      # Social Networking
                    set action block
                next
                edit 2
                    set category 62      # Gambling
                    set action block
                next
            end
        end
        set log-all-url enable
    next
end

Notice: the profile is a container of category verdicts. FortiGuard tells the firewall which category the site is in; the profile decides what to do with that category. If FortiGuard can't answer, the whole apparatus above has nothing to act on.

Application list — flow-only, runs on the IPS engine

config application list
config application list
    edit "TF-Apps"
        config entries
            edit 1
                set category 5           # Social.Media
                set action block
            next
            edit 2
                set application 15832    # Facebook_App
                set action block
            next
        end
    next
end

Notice: no FortiGuard live-lookup dependency here. The signatures are already on the box. Freeze the world on July 11th and this profile keeps working with July 11th's signature list — which is why Facebook was still blocked on phones during the outage.

ISDB — not a profile at all

config firewall internet-service-name (used ON the policy)
config firewall internet-service-name
    edit "Facebook-Web"
        set internet-service-id 65646
    next
end

config firewall policy
    edit 42
        set name "Block-FB-For-Traders"
        set srcintf "port3"
        set dstintf "port1"
        set srcaddr "TradingFloor_LAN"
        set dstaddr "all"
        set internet-service enable
        set internet-service-name "Facebook-Web"
        set action deny
        set schedule "always"
    next
end

Notice the structural difference: ISDB is referenced on the policy itself, replacing or narrowing the destination address. No set utm-status enable, no profile attached. The policy IS the filter.

FortiManager as local FDS — the resilience posture

config system central-management
config system central-management
    set type fortimanager
    set fmg "10.0.100.10"
    set include-default-servers disable      ← purist enterprise posture
    config server-list
        edit 1
            set server-type rating
            set server-address "10.0.100.10"
            set addr-type ipv4
        next
    end
end

config webfilter fortiguard
    set cache-mode ttl
end

set include-default-servers disable tells FortiGate do not fall back to the public FortiGuard anycast pool — only ask the local FortiManager-as-FDS. Purist enterprise choice. In hybrid designs, leave it enabled as a safety net. Both are defensible; knowing why you'd pick each is a senior conversation.

Verification

Diagnostic order under fire

The order is deliberate — each command reads a different layer of the failure model, and none of them change state.

OrderCommandWhat it proves
1diagnose autoupdate versionsPaperwork state — versions, contract expiry, last update result for every locally-stored DB
2diagnose debug ratingLive conversation state — is FortiGate reaching FortiGuard, at what RTT, with what result?
3diagnose webfilter fortiguard cache dumpCurrent belief — what verdicts does the box hold right now, and how fresh are they?
4get webfilter statusProfile-side view — is web filter enabled, licensed, configured
5get system fortiguardConfiguration state — which servers, which ports, override configured?
Junior's anti-pattern
execute update-now is an action, not a diagnostic. Running it before you've read state destroys evidence. Never a diagnostic step. Never.
Troubleshooting Workflow

The silent-degradation pattern

  1. Read the symptom carefully. What is the scope of failure? One user? One app? All browsing? Everything? Scope narrows the answer set faster than any other single technique.
  2. Read the paperwork. diagnose autoupdate versions — is anything expired, stale, or failing to update?
  3. Read the live conversation. diagnose debug rating — is FortiGate actually reaching FortiGuard? What's the RTT? What flags?
  4. Read the current belief. Cache dump — what does the box think about the specific site the user reported?
  5. Form a hypothesis and prove it before you change anything. Two independent lines of evidence, minimum.
  6. Fix — and design out the recurrence. Renewal handles today. FortiManager-as-FDS handles the next envelope-on-a-desk.
Common Mistakes

What Junior gets wrong, and why

Mistake 1 — Panicking at three "License Expired" lines
Junior sees three expiry warnings and concludes three broken features. Senior sees three expiry warnings and separates paperwork state from behavioural state. App control and ISDB run on their most recent database. Web filter doesn't have one to run on.
Mistake 2 — execute update-now as a diagnostic
Junior's reflex is to "try refreshing." That's a state change. It destroys the evidence you needed to prove root cause. Senior engineers never refresh, restart, or reboot as a diagnostic step.
Mistake 3 — Assuming fail-open means users won't feel it
Fail-open is the default. Users still felt this morning's outage. Why? Because fail-open kicks in after the request timeout, and the user's browser (and patience) run out well before that. Fail-open governs the verdict; the timeout governs the experience.
Mistake 4 — Trusting FortiGuard's uptime as your resilience posture
Any FortiGate that queries the public FortiGuard anycast pool live is one internet-side incident away from a browsing outage. Enterprise-grade design points web filter and app-control rating lookups at a FortiManager licensed as local FDS instead.
Senior Engineering Insights

What separates the mid-level from the senior

1. Symptom scope first, mechanism second

Before naming the failure mechanism, name its scope. "No logs for Zoom specifically" is surgical — points at signature DB, not engine. "No logs for anything" is engine-wide. Scope kills wrong answers.

2. Paperwork and behaviour

The licence-expiry evidence alone doesn't prove the hang. The RTT=-1 evidence alone doesn't prove the licence. You need both before you touch config. Two independent lines of evidence.

3. Design against unmonitored dependencies

Every subscription is a piece of paper on someone's desk. Design the system so a piece of paper can't take you down. FortiManager-as-FDS + expiry alerts + calendar reminders. Belt, braces, calendar.

4. Explain the invoice, not the technology

Executives pay for value, not for the absence of failure. Frame the subscription as "Fortinet's research team keeping our firewall's picture of the world current" — not as a database update.

Exam Tips

Traps you will see on the NSE 7

  • The absolute-qualifier trap. "Immediately," "always," "cannot," "sufficient." Absolutes are almost always wrong in networking. Reality is timeouts, thresholds, and asymmetries.
  • The familiar-fact distractor. Fortinet loves to offer, as a wrong answer, a true fact you're proud of knowing. If a distractor quotes recently-retained material, be more suspicious, not less.
  • The compound True/False. Two true premises + a "therefore" clause with a subtly false qualifier in the conclusion. Read every qualifier in the conclusion clause before committing. "Correctly" and "sufficient" are common false qualifiers.
  • The command that changes state. If a diagnostic-order question includes execute update-now as an option, it's a trap. Actions are never diagnostics.
  • The scope test. When a symptom is described in the stem, state its scope out loud: one app? one category? everything? one user? Scope kills two wrong answers immediately.
  • ISDB is not a profile. If an answer says "attach the ISDB as a security profile" — it's wrong. ISDB is referenced on the policy as internet-service-name. This is a classic distractor.
Key Takeaways

The five sentences you must own

  1. Web filter is a live conversation; app control and ISDB are stored knowledge.
  2. The web filter cache masks a FortiGuard outage temporarily; it is a decaying asset when the source is unreachable.
  3. License expired ≠ feature broken. Two of the three keep working on frozen data.
  4. Fail-open governs the verdict. Timeouts govern the experience. A cache miss with FDS down still hangs the user.
  5. Enterprise-grade resilience = FortiManager as local FDS, plus fail-open, plus expiry warnings, plus someone reading the mail.
