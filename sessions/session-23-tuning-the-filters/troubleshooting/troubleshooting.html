When the filters stop filtering

Symptoms, questions, evidence, decision tree — for FortiGuard-dependent features

The senior mindset

The failure modes in this chapter are silent and gradual. Nothing alerts. Nothing logs "your subscription lapsed." The system just quietly stops answering new questions and rides on its memory until the memory runs out. Your job is to notice which layer is failing before Junior does the wrong thing.

The one rule: read state before you change state. Every diagnostic command below reads. None of them change anything. That's on purpose.

Symptoms you'll see

Symptom A

Browsing hangs, no block page, apps still work

The classic FortiGuard-connectivity or licence-lapse pattern. Browsers spin; Facebook APP is still being blocked; Slack/trading platform fine. Ticket wording: "Internet broken." The scope is browser-shaped, not machine-shaped.

Symptom B

App control block for one specific app has no effect and produces no logs

Surgical scope — one application. Suggests signature-database problem, not engine problem. If the engine were down, everything would fail, not one app.

Symptom C

Web-filter block for a specific site not firing

Two candidates: (1) FortiGuard has that site categorised differently than you think — cache dump will confirm; (2) SSL inspection isn't set to full and the URL isn't visible to the profile — check the profile.

Symptom D

SD-WAN rule matching Salesforce not steering as expected

If using ISDB — check ISDB DB freshness (diagnose autoupdate versions). If using app control — check Application Detection-Based SD-WAN is enabled under System > Feature Visibility, and remember app-control-based steering is learning-based (first packets take default path).

Questions to ask first

  1. What is the scope of the failure? One user, one app, one category, all browsing, everything? Scope kills wrong answers.
  2. Did the configuration change recently? If Junior says "nothing changed" and you can verify — trust it. That's a signal to look at the environment, not the config.
  3. When did the symptom start? Sudden = network/connectivity/licence flip. Gradual = cache decay, signature staleness, DB drift.
  4. What still works? The intersection of "what fails" and "what works" points directly at the affected layer.
  5. Is there any subscription paperwork nearby? Literally. Check for expiry warnings in the GUI, unopened emails, unread notification banners. The physical envelope-on-the-shelf is a real failure mode.

Verification commands & expected outputs

Paperwork state

diagnose autoupdate versions
Virus Definitions / Application Definitions / Internet-service Full Database
  Version, Contract Expiry Date, Last Update Attempt, Result

Healthy:   Result: OK
Lapsed:    Result: Update failed. (License Expired)
Blocked:   Result: Update failed. (Connection error)  ← firewall/routing/DNS issue

Web filter is not in this output. That's expected — its "database" isn't local. Absence of a webfilter line is proof-by-absence, not a bug.

Decision tree

"Trading floor browsers hang; apps still block correctly."

1. Is the failure browser-shaped, or does it also hit non-HTTP traffic?
Browser-shaped only → suspect web-filter FortiGuard dependency. Continue.
All outbound broken → not this chapter. Check routing, WAN link, policy chain.
2. Run diagnose autoupdate versions. Any "License Expired" or "Update failed"?
Yes → subscription lapse or FortiGuard connectivity. Continue to 3.
No → databases healthy. Skip to 5.
3. Run diagnose debug rating. Status: Disable, RTT: -1, Flags: DI on all servers?
Yes → confirmed. It's a subscription or FDS-reachability problem.
No — servers reachable but License: Expired → licence problem specifically.
No — License: Valid but RTT: -1 → connectivity problem (firewall, DNS, routing).
4. Fix path:
Licence lapsed → renew subscription. Verify diagnose debug rating shows Status: Enable within a few minutes.
Connectivity blocked → check outbound policy to FortiGuard IPs/HTTPS/443, DNS for fortiguard.net domains, upstream firewall rules.
5. If DBs healthy but user still reports a specific site not being blocked:
Check diagnose webfilter fortiguard cache dump for that site — likely miscategorised. Submit a rating request to Fortinet.
Check SSL inspection — if profile is certificate-inspection only, URL path isn't visible; try full inspection.
6. Design step (root cause vs restoration):
Restore = renew. Root cause fix = FortiManager-as-FDS + expiry warnings + calendar reminder.

Common dead ends

Running execute update-now as a diagnostic. It's an action, not a read. It destroys the "Last Update Attempt" evidence and may temporarily change the "Result" field before you've read what it was. If you need to force an update after fixing the root cause, sure. Not before.
Renewing the licence without designing out the recurrence. The next envelope will sit on a desk too. Design a system where a piece of paper cannot take down the trading floor. That's the senior part.
Assuming fail-open covers you. Fail-open is default. It didn't help LoFinance's trading floor because the timeout is measured in seconds, and browsers give up first. Fail-open is a verdict philosophy, not a latency solution.
Confusing "database expired" with "feature broken." App control and ISDB run on their most recently downloaded database. Licence expiry stops the updates, not the feature. Old signatures still fire.

Likely root causes, ranked

  1. FortiGuard subscription lapsed (paperwork). Renew.
  2. Outbound policy blocking access to FortiGuard. Check WAN egress policy — HTTPS/443 to the anycast pool must be allowed. Corporate proxies sometimes intercept.
  3. DNS resolution failure for fortiguard.net domains. Rare but real. Check execute traceroute fortiguard.net.
  4. Certificate issue on the anycast path. Rare; usually shows as certificate errors in the debug rating output.
  5. Misconfigured override server. If a manual override server was set for testing and never removed, FortiGate might be pointing at nowhere. Check get system fortiguard.

Exam traps in this space

  • Which command shows Application Definitions expiry? Answer is diagnose autoupdate versions. Distractors include execute update-now (action, not read) and get system fortiguard (config, not database state).
  • During a FortiGuard outage, which features continue to work? App control and ISDB (local DBs). Web filter fails first (cache decay). Any answer suggesting all three fail together, or that ISDB fails first, is wrong.
  • Application control not matching a specific application — first place to look? Signature database freshness. NOT the IPS engine (surgical scope rules out engine-wide failure).
  • ISDB in SD-WAN rules — how is it referenced? As the destination. Not source. Not a profile. And it does not require Application Detection-Based SD-WAN — that's for app control.
  • Fail-open under FortiGuard outage — what does the user experience? Not "instant permit." A hang until the request timeout expires. Fail-open governs the verdict, timeout governs the experience.
