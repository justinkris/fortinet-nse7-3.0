● TROUBLESHOOTING · SESSION 29

The Haunted Firewall

How a senior engineer diagnoses "the config keeps un-fixing itself" on a managed FortiGate.

SYMPTOMSWhat it looks like

  • A change made on a managed FortiGate reverts, sometimes minutes later.
  • It reverts with nobody at the keyboard — the tell that an automatic FMG reconciliation is running.
  • Help desk logs repeat tickets for the same setting "changing itself back."
  • Revision history looks clean — no entries matching the chaos.

QUESTIONS TO ASK FIRSTBefore you touch anything

  • Is this FortiGate managed by FortiManager? (If yes, the source of truth may not be the box.)
  • Which setting is reverting? Interface/route/VPN/system → device layer. Policy/object → policy layer.
  • Was the change made directly on the box (SSH or box-GUI) or in the FMG layer?
  • Does the box keep losing (FMG pushing) or keep winning (something retrieved it)?
  • Has any install or auto-retrieve run recently?

VERIFICATIONCommands & where to look

On the box — confirm the live value and how it drifts:

# what is the box actually running right now?
show system interface port3
# is this FGT centrally managed, and by whom?
get system central-management

In FortiManager — compare against the stored truth:

  • Device Manager > Device & Groups > Managed FortiGate → check Config Status (Synchronized / Modified / Out-of-sync).
  • … > FortiGate > Dashboard: Summary > Configuration and Installation > Revision → read the timeline of device-DB snapshots.
  • Objects in the Device Layer → confirm what the device DB believes port3 should be.

EXPECTED OUTPUTSReading the evidence

What you seeWhat it means
Box shows .30 minutes after you set .31FMG device DB (.30) is being pushed down. FMG wins.
Config Status = Modified / Out-of-syncDevice DB and live box disagree. A sync/install will resolve it — in FMG's favour.
Revision history has no entries for the reverting changeThe change was made directly on the box — invisible to revision history. Confirms a direct-CLI/box-GUI edit.
A recent install event in the timeline just before a revertThat install pushed the DB down and overwrote the box change.

DECISION TREEWhich fix, and which way

Is the reverting setting a device setting (interface/route/VPN/system)?
No → it's a policy/object → it lives in the policy layer; policy edits only apply on install. (See S30.)
Yes → it's the device layer / device DB. Continue ↓
Is FMG's stored value (device DB) the one you WANT?
Yes → the box is wrong. Install device settings (or restore the good revision) to push the correct value down. Done.
No — the box's current value is correct → decide by situation ↓
Are you in a live incident / do you distrust the box's current state?
No, calm, deliberate box change → Retrieve to adopt the box value into the device DB (then import if it must reach the policy package).
Yes, incident → DO NOT retrieve. Push a known-good down (install / restore revision). Retrieve later, once stable.

COMMON DEAD ENDSWhere people waste an afternoon

  • Re-editing the box again and again. You're arguing with a copy. The box will keep losing until you move the layer.
  • Blaming the box GUI vs SSH. Same mistake — both leave the device DB stale.
  • Trusting a clean revision history to mean nobody touched the box. Direct edits are invisible.
  • Hitting Retrieve to "sync it." Mid-incident this captures the fault and destroys last-known-good.
  • Expecting install to prompt about conflicts. A managed install is a blind overwrite.

LIKELY ROOT CAUSES

  1. Direct-CLI change on a managed box creating a device DB ↔ live-box disagreement, reconciled by an install-direction sync (the classic haunting).
  2. Stale device DB holding an old value that a routine install pushes down.
  3. Undocumented drift that ages for days, then vanishes on the next unrelated install.

EXAM TRAPS

Trap 1 — "install prompts a conflict"
No. Install is a blind DB→box overwrite. An unknown box change is overwritten and lost, not negotiated. Pick the harsh answer.
Trap 2 — "auto-retrieve syncs the policy package"
No. Auto-retrieve/retrieve write to the device DB / revision history. Reaching the policy package needs an import.
Trap 3 — "direct edits appear in revision history"
No. Revision history records the device DB only. Direct-CLI/box-GUI edits write zero revisions.
Trap 4 — "retrieve first to be safe"
Mid-incident, retrieve is the wrong first move — it captures the broken config as truth and overwrites your last-known-good.
Senior takeaway
The fault is almost never the firewall. It's a disagreement between two copies and an operator who didn't decide which copy should win. Read the direction. Choose the truth on purpose. Change the layer, not the box.
