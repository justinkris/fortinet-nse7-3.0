BUSINESS PROBLEM
----------------
A managed FortiGate's config "keeps un-fixing itself." Root cause: device DB (FMG's stored
copy, .30) disagrees with the live box (.31 via SSH); FMG reconciles by pushing DB→box, so
the box always loses. Business risk generalised: direct-CLI changes on managed boxes are
silent, undated, and destroyed by the next unrelated install → drift between the box and the
"source of truth" → cannot prove config matches records at audit → finding → cost. (Financial
firm; ties to the season's audit spine.)

TECHNICAL CONCEPTS COVERED (and the mental models built)
--------------------------------------------------------
1. SOURCE OF TRUTH lives in FMG, not on the box, for a managed FortiGate.
2. TWO LAYERS inside one ADOM per box:
   - DEVICE LAYER → device settings (Network/Interfaces, VPN, System, Log & Report). PER-BOX
     IDENTITY. Student's own framing: "unique to the firewall / what it is."
   - POLICY LAYER → firewall policies + objects, organised as POLICY PACKAGES. SHAREABLE
     INTENT. Student's framing: "how the firewall behaves with traffic."
   Mental model the student SELF-DERIVED: identity is per-box (device), intent is author-once
   -install-many (policy) — that's WHY the split exists.
3. DEVICE DATABASE (device DB) = FMG's own held copy of a box's device settings. Does NOT
   update when the live box is edited — and does NOT care whether the edit was SSH or the
   box's own local GUI (killed the student's "GUI behaves differently" gut instinct: channel
   is irrelevant, LAYER is everything).
4. DIRECTION PAIR (the session's spine):
   - INSTALL DEVICE SETTINGS: device DB → box. FMG's stored truth wins. Has a preview.
   - QUICK INSTALL: same direction, NO preview, applies immediately (ZTP / high-confidence).
   - RETRIEVE: box → device DB. The box's current reality wins; saved as a NEW REVISION;
     does NOT update the policy package (import required for that).
   - AUTO-RETRIEVE: fires when a device is first added to the device layer; captures direct
     box changes into revision history; still does NOT update policy package.
5. REVISION HISTORY = numbered, timestamped snapshots of the DEVICE DB over time. It is a
   history of DEVICE-DB changes, NOT of the live box. THEREFORE direct-CLI/GUI-on-box edits
   write ZERO revisions and are forensically INVISIBLE. Revision history is great for "what
   FMG did," blind for "what someone did to the box." (This is the S30-import bridge: policy
   layer sync is a separate story.)

TROUBLESHOOTING APPROACH TAUGHT ("config keeps reverting on a managed box")
--------------------------------------------------------------------------
1) Symptom→hypothesis: reverts "on its own" on a managed FGT → suspect device DB↔box
   disagreement being reconciled by install-direction sync.
2) Which layer owns the setting? interface/route/VPN/system=device; policy/object=policy.
3) Which direction is winning? box loses→FMG pushing; box wins→a retrieve captured it.
4) Read revision history BUT know its blind spot (device-DB only; direct edits invisible).
5) Resolve DELIBERATELY: decide which copy is truth — push down (install / restore revision)
   or pull up (retrieve). Never retrieve mid-incident (captures fault, kills last-known-good).
