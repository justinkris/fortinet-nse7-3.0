--------------------------------------------------------------------
QUICK SUMMARY (one page)
--------------------------------------------------------------------
A managed FortiGate (VL-BRANCH-2) "kept un-fixing itself." Junior SSHed
into the live box to fix port3 (.30 -> .31); it reverted repeatedly,
including once with nobody at the keyboard. Not haunted: FortiManager's
DEVICE DATABASE held .30 and kept pushing that stale value down over
Junior's direct edit. The box always loses that argument.

For every managed FortiGate, FMG keeps config split in TWO LAYERS inside
the ADOM:
  * DEVICE LAYER  = device settings (Network/Interfaces, Routing, VPN,
    System, Log & Report). PER-BOX IDENTITY. Aimed at the FGT directly
    once installed.
  * POLICY LAYER  = firewall policies + objects, as POLICY PACKAGES.
    SHAREABLE INTENT. Takes effect ONLY on install/deploy.
Unlock: identity is per-box (device); intent is author-once-install-many
(policy). port3 IP is a device setting -> device layer held .30.

DEVICE DATABASE = FMG's own copy of a box's device settings. It does NOT
update when you edit the live box, and does NOT care if you used SSH or
the box's local GUI. Channel irrelevant; LAYER is everything.

DIRECTION PAIR (the spine):
  * INSTALL device settings : device DB -> box. FMG's truth wins. Preview.
  * QUICK INSTALL           : device DB -> box. No preview, immediate (ZTP).
  * RETRIEVE                : box -> device DB. Box reality wins; new
    revision; does NOT update policy package (import needed).
  * AUTO-RETRIEVE           : fires when device first added; captures box
    changes to revision history; still no policy-package update.

REVISION HISTORY = numbered/timestamped snapshots of the DEVICE DB.
Rollback lives here. BUT direct-CLI/box-GUI edits write ZERO revisions ->
forensically invisible. Great for "what FMG did," blind for "what someone
did to the box directly."

SENIOR RULES:
  - Change the LAYER, not the box.
  - Direct-CLI on a managed box = a landmine with an install-shaped
    trigger: the next unrelated install overwrites it, silently, no
    conflict prompt, no record it existed.
  - Never RETRIEVE mid-incident: captures the fault as truth and kills
    last-known-good ("mop while the tap is still running").
  - Name the DIRECTION before you click.

--------------------------------------------------------------------
CHEAT SHEET
--------------------------------------------------------------------
LAYERS (per box, per ADOM)
  Device layer  -> Network/Interfaces, Routing, VPN, System, Log&Report
  Policy layer  -> policies + objects (policy packages); live on install

OPERATIONS / DIRECTION
  Install Device Settings   DB -> box    preview      FMG wins
  Quick Install             DB -> box    NO preview   FMG wins (ZTP/confident)
  Retrieve                  box -> DB    new revision box wins; policy pkg NOT updated
  Auto-retrieve             box -> DB    on add       captures to rev history; import still needed
  (S30 preview) Import      device/box -> policy package  (the bridge)
  (S30 preview) Revert      pick earlier revision -> DB (then import to sync)
  (S30 preview) Install Policy Package  policy layer -> device DB + box (wizard)
  (S30 preview) Reinstall Policy        same, no wizard; also pushes DB modifications

CONFIG STATUS (Device Manager > Device & Groups)
  Synchronized  = device DB matches the box
  Modified      = device DB changed, not yet installed
  Out-of-sync   = box and device DB disagree

GUI PATHS
  Revision history:
    Device Manager > Device & Groups > Managed FortiGate > FortiGate >
    Dashboard: Summary > Configuration and Installation > Revision
  Device layer objects:
    Device Manager > Device & Groups > Managed FortiGate >
    Objects in the Device Layer
  Policy packages:
    Policy & Objects

USEFUL FGT CLI (on the box)
  show system interface port3          # live value
  get system central-management        # is it FMG-managed, and by whom

KEY DEFAULTS / BEHAVIOURS
  - Editing the live box never updates the device DB by itself.
  - Install = blind overwrite (no conflict negotiation for unknown changes).
  - Quick Install has no preview.
  - Retrieve/auto-retrieve do not touch the policy package (import required).
  - Direct edits = 0 revisions.

FLASHCARDS (Q -> A)
--------------------------------------------------------------------
Q: Where is the source of truth for a managed FortiGate?
A: In FortiManager (the device DB for device settings), not on the box.

Q: What does the device layer hold?
A: Device settings: Network/Interfaces, Routing, VPN, System, Log&Report.

Q: What does the policy layer hold, and when does it take effect?
A: Policies + objects (policy packages); only after install/deploy.

Q: Why keep interfaces in device layer and rules in policy layer?
A: Identity is per-box (device); intent is shareable/author-once (policy).

Q: Does editing the live box update the device DB?
A: No. It stays stale until an FMG op (retrieve) runs.

Q: Does it matter whether you used SSH or the box's local GUI?
A: No. Both edit the live box; both leave the device DB stale.

Q: Direction of Install device settings?
A: device DB -> box. FMG's stored truth wins.

Q: Direction of Retrieve?
A: box -> device DB. Box reality wins; saved as a new revision.

Q: Does retrieve update the policy package?
A: No. An import is required for that.

Q: Quick Install vs Install device settings?
A: Same direction; Quick Install has NO preview and applies immediately.

Q: How many revisions did Junior's 4 SSH fixes write?
A: Zero. Revision history records the device DB, not the live box.

Q: A week-old direct-CLI change, then an unrelated Install fires. Result?
A: Overwritten and lost. Not merged, no conflict prompt.

Q: Why is retrieve almost never your first incident move?
A: It captures the broken config into the DB and overwrites last-known-good.

Q: How do you make a correct box change stick durably (two ways)?
A: (1) Edit device layer in FMG then install (truth down); or
   (2) Retrieve so the device DB adopts the box value (truth up).

Q: Where does revision history stop being useful for blame?
A: When the change was made directly on the box - it leaves no revision.

Q8 Quick Install differs in that it...                           -> B (applies immediately, no preview)
Q9 Auto-retrieve saves a revision but does NOT...                -> B (update the policy package; import required)
Q10 (framing) Why ban SSH config on managed boxes? Money/audit first:
    "When we SSH a change onto a firewall, FortiManager doesn't know it
    happened - no record, and the next routine push wipes it out. So if an
    auditor asks me to prove the firewall's config matches our records, I
    can't - the box and our source of truth have silently drifted. That's a
    finding, and findings cost us. Editing through FortiManager means every
    change is recorded and the box always matches what we can prove."

TRACKED-MISS REMINDERS FOR NEXT TIME:
  - Q6 family: when the mechanism is HARSH (overwrite/lost/cannot), answer
    harsh. Don't downgrade to "conflict prompt"/"confusing".
  - Q9 family: RETRIEVE/AUTO-RETRIEVE never touch the policy package -
    IMPORT is the bridge (drilled in S30).
  - Q10 family: lead with the consequence-as-SCENE (the auditor asks and I
    can't answer), and keep to THIS session's topic (drift/silent loss),
    not the S28 ADOM/admin-boundary answer.
