Two Layers of Truth

DEVICE LAYER vs POLICY LAYER · DEVICE DB · REVISION HISTORY · NSE7 EFW 7.6 · 2.1

A managed firewall "kept un-fixing itself." It wasn't haunted — FortiManager's stored copy kept overwriting the box.

The two layers (per box, in one ADOM)

Device layer → device settings: Network / Interfaces, Routing, VPN, System, Log & Report. Per-box identity. Changes aim at the FortiGate directly (once installed).

Policy layer → firewall policies + objects, as policy packages. Shareable intent. Takes effect only on install/deploy.

Unlock: identity is per-box (device); intent is author-once-install-many (policy).

Device database = FMG's own copy

Holds what the box's device settings should be. Does not update when you edit the live box — and doesn't care whether you used SSH or the box's local GUI. Channel is irrelevant; the layer is everything.

The direction pair — know which way truth travels

Install →device DB → box.
FMG's stored truth wins. Quick Install = no preview.
Retrieve ←box → device DB.
Box reality wins; new revision. Does NOT touch policy package (import needed).

Revision history — safety net + blind spot

Numbered snapshots of the device DB. Rollback lives here. But direct-CLI / box-GUI edits write ZERO revisions — forensically invisible. Great for "what FMG did," blind for "what someone did to the box."

Burn these in

  • Change the layer, not the box.
  • Direct-CLI on a managed box = a landmine with an install-shaped trigger (next unrelated install overwrites it, silently, no conflict prompt).
  • Never retrieve mid-incident — it captures the fault and kills last-known-good ("mop while the tap is running").
  • Install = DB→box · Retrieve = box→DB. Name the direction before you click.
