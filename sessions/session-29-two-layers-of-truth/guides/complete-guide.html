Two Layers of Truth

The haunted firewall — device layer vs policy layer, the device database, and revision history.

02 Central Management · pp. 29–31 Blueprint 2.1 NSE7 EFW 7.6

STORY RECAPThe firewall that kept un-fixing itself

Day after the Vaultline pallet. VL-BRANCH-2 has port3 mis-set to 10.20.30.1/24 — it should be 10.20.31.1/24. Junior SSHes into the live box and fixes it. It reverts. He fixes it again. It reverts. At 2:58pm it flips back to .30 with nobody at the keyboard. Riley logs four tickets, the last titled "is the firewall haunted (serious)."

Señor's diagnosis, handed to you rather than Junior: "You edited the firewall, but you did not edit the layer that was about to edit the firewall. You were arguing with a copy."

OBJECTIVESWhat you should be able to explain

  • Where the source of truth lives for a FortiManager-managed FortiGate.
  • The difference between the device layer (device settings) and the policy layer (policies + objects), and why the split exists.
  • What the device database is, and why it stays stale when you edit the live box — SSH or box-GUI alike.
  • The direction of Install (DB→box) vs Retrieve (box→DB), and when each is correct.
  • What revision history records — and the forensic blind spot it leaves.
  • Why retrieve is almost never your first move on a live incident.

CONCEPT 1Two layers inside one ADOM

For every managed FortiGate, FortiManager keeps its configuration split across two databases in the ADOM. They hold different kinds of settings:

Device layerDevice settings:
Network / Interfaces · Routing · VPN · System · Log & Report

Per-box identity
Policy layerFirewall policies + objects,
organised as policy packages

Shareable intent
The mental model that unlocks everything
Identity is per-box → device layer. Intent is shareable → policy layer. An interface IP (port3 = 10.20.31.1) is unique to this box — you can't author it once and stamp it on seven firewalls. A rule ("allow branch LAN → syslog on 514") is intent you'd want on all seven — so it lives in a policy package that can be installed to many.

Where each takes effect: device-layer changes are aimed at the FortiGate directly (once installed). Policy-layer changes take effect only after policy deployment/installation — editing a policy and walking away sends it nowhere.

Best-practice note (exam-relevant)
You can configure policies in the device layer, but you shouldn't. Restrict the device layer to device settings; make policy changes in the policy layer so the device database values update correctly.

CONCEPT 2The device database — a copy that doesn't watch the box

The device database (device DB) is FortiManager's own held copy of a box's device settings. This is the "copy" Señor meant. Two facts do all the work:

  1. The device DB does not update when you change the live FortiGate. It sits there holding its last-known value until an FMG operation runs.
  2. It doesn't care how you edited the box. SSH and the box's own local GUI are the same category of mistake — both are "editing the live box," both leave the device DB stale. The channel is irrelevant; the layer is everything.
# Junior's mistake — direct on the live box, device DB untouched
VL-BRANCH-2 # config system interface
VL-BRANCH-2 (interface) # edit port3
VL-BRANCH-2 (port3) # set ip 10.20.31.1 255.255.255.0
VL-BRANCH-2 (port3) # end
"If FortiManager didn't know Junior's change happened — what does that tell you about where FMG's idea of truth lives?"
It lives in the device DB, up in FortiManager — not on the box. The box only holds the running config. When they disagree, an FMG operation decides who wins.

CONCEPT 3The direction pair — Install vs Retrieve

Everything hinges on which way config travels when the DB and the box disagree. There are two directions, and you must pick one on purpose — or FMG picks for you.

OperationDirectionWho winsPreview?Writes a revision?
Install Device Settingsdevice DB → boxFMG's stored truthYesYes
Quick Installdevice DB → boxFMG's stored truthNo (immediate)Yes
Retrievebox → device DBThe box's reality—Yes (new revision)
Auto-retrievebox → device DBThe box's reality—Yes
The haunting, solved
The box kept losing → FMG was pushing (install-direction). Its device DB said .30. To keep .31 durably you either edit the device layer and install (move the truth down), or Retrieve so the device DB adopts .31 from the box (move the truth up). Both end the disagreement — they just decide which copy is right.
Quick Install vs Install Device Settings
Same direction; Quick Install skips the preview and applies immediately. It's the ZTP / "I'm confident about this change" tool — not for cautious moments.
Auto-retrieve's catch (this is the Q9 trap)
Auto-retrieve captures direct box changes into revision history when a device is first added — but it does not update the policy package. Getting a change into the policy layer still requires an import. (This is next session's bridge.)

CONCEPT 4Revision history — safety net and blind spot

Every time the device DB changes — every install, retrieve, or device-layer edit — FortiManager stamps a new numbered, timestamped snapshot into that box's revision history. It's the box's config timeline: your rollback mechanism and your forensic record of who did what in FMG.

The blind spot that broke the haunting
Revision history is a history of the device DB, not of the live box. Junior hammered port3 on the box four times — but he never touched the device DB, so those four fixes wrote ZERO revisions. Direct-CLI / box-GUI edits are forensically invisible. Revision history is excellent for "what FMG did," and blind for "what someone did to the box directly."

GUI path: Device Manager > Device & Groups > Managed FortiGate > FortiGate > Dashboard: Summary > Configuration and Installation > Revision.

CONCEPT 5Why direct-CLI on a managed box is a landmine

Play the quiet version. An admin makes a direct-CLI change that isn't noticed — no haunting, it just sits for a week. Then someone runs a routine Install Device Settings for an unrelated reason. Install is a blind DB→box overwrite, and the DB never knew about the change, so:

  • The change is overwritten and lost — not merged, no conflict prompt. FMG isn't aware two versions exist; it just stamps its copy down.
  • The trigger is unrelated. The person installing had no idea they destroyed a week of work.
  • It's silent and undated. The change never wrote a revision, so there's no record it ever existed. Afterward, the history looks innocent.
  • The gap ages. For a week the DB was a lie and everyone trusted it. The eventual overwrite is bigger and more surprising.
Senior rule
On a managed box, a direct-CLI change isn't a change — it's a landmine with an install-shaped trigger. Either change the layer (device layer → install), or if you had to touch the box, retrieve immediately so the DB adopts it and the drift never gets to age.

CONCEPT 6Why "Retrieve" is almost never your first incident move

Retrieve doesn't judge — it copies. During an incident the box is, by definition, in a bad state. Hit retrieve and FMG faithfully swallows the broken config into the device DB, stamps it as the new truth, and overwrites the last-known-good version you were holding.

The analogy
Reaching for Retrieve mid-outage is reaching for the mop while the tap is still running. You had a clean copy in the DB you could have installed to recover — retrieve just painted over it with the mess.
The correct instinct
Something broke and I trust FMG's stored truth → push down (install, or restore a good revision). I deliberately changed the box and it's correct → pull up (retrieve). Never blind-retrieve a box you don't trust.

TROUBLESHOOTING"Config keeps reverting on a managed box"

  1. Symptom → hypothesis: reverts "on its own" on a managed FGT → suspect a device DB ↔ live-box disagreement being reconciled by install-direction sync.
  2. Which layer owns the setting? interface / route / VPN / system = device layer. policy / object = policy layer.
  3. Which direction is winning? box keeps losing → FMG pushing. box keeps winning → a retrieve captured it.
  4. Read revision history — but remember the blind spot: direct box edits leave no revision. A clean history ≠ nobody touched the box.
  5. Resolve deliberately: decide which copy is truth. Push down (install / restore revision) or pull up (retrieve) — on purpose, never by reflex. Never retrieve mid-incident.

COMMON MISTAKESWhat trips people (and Junior)

  • "The box GUI behaves differently from SSH." No — both are editing the live box; both leave the device DB stale. Channel is irrelevant.
  • Expecting a "conflict prompt" on install. A managed install is a blind overwrite. An unknown box change is lost, not negotiated.
  • Thinking direct edits show in revision history. They don't. Revision history = device-DB history only.
  • Retrieving during an incident. Captures the fault as truth and kills last-known-good.
  • Editing a policy and expecting it live. Policy layer takes effect only on install/deploy.

EXAM TIPSBlueprint 2.1

  • Match the setting to the layer: Network / VPN / System / Log & Report = device layer.
  • Retrieve = box→DB, new revision, does not touch the policy package.
  • Direct box edit → device DB unchanged until an FMG op runs; writes zero revisions.
  • Install / Quick Install = DB→box; Quick Install has no preview.
  • Auto-retrieve saves a revision but still needs an import for the policy package.
  • When the mechanism is harsh (overwrite / lost / cannot), pick the harsh answer — don't downgrade to "conflict" or "confusing."

Key takeaways

  • Source of truth lives in FortiManager, split into two layers: device (identity) and policy (intent).
  • Install = DB→box. Retrieve = box→DB. Name the direction before you click.
  • The device DB is oblivious to live-box edits — SSH and box-GUI are the same mistake.
  • Revision history records the device DB, so direct edits are invisible.
  • Change the layer, not the box. Direct-CLI on a managed box is a landmine with an install-shaped trigger.
  • Never retrieve mid-incident — you capture the fault and lose your last-known-good.
