The Fast Path

Session offloading — requirements and the life of an offloaded packet · Study Guide ch. 10, pp. 286–291

Two identical firewalls. One is twenty times slower. The difference is which sessions get to ride the NP.
Penny: "Same box. Same config. Same money. Why is one of them slow?"
Story Recap

The Case of the Identical Twins

FW-TRADE-01 and FW-TRADE-02: same FortiGate 900G, same firmware, same FortiManager-pushed policy package. TRADE-01 forwards trading traffic at 40 µs; TRADE-02 spikes to 800 µs with a CPU graph in cardiac arrest. diagnose sys session list confessed immediately: no_ofld_reason: disabled-by-policy. Junior had disabled auto-asic-offload on a policy for Friday packet sniffing and forgot to re-enable it.

Re-enabling it did not fix the live trading sessions — because the offload decision is made once, at session birth. Only after clearing the stale sessions did new handshakes qualify, keys install, and latency fall back to 40 µs.

Senior rule: identical boxes behaving differently means different state, not different hardware.

Concept 1

Slow Path, Fast Path, and the Session Key

The NP6/NP7 is a forwarding replayer: it can execute a decision at wire speed but can never make one. Policy evaluation, routing lookup, and session creation are stateful kernel work — CPU only. So the CPU always witnesses the beginning of every conversation:

  • TCP: the entire three-way handshake (SYN → SYN/ACK → ACK) transits the CPU.
  • UDP: only the first packet takes the slow path.

If the session qualifies, the CPU installs a session key in the NP — a pre-computed answer to "route this, allow this." Every subsequent packet skips routing, skips the kernel, never touches the CPU: the fast path.

FIN and RST packets are always punted back to the CPU, even on a fully offloaded session — teardown means deleting the kernel session entry and uninstalling the NP key, bookkeeping only the state owner can do.

"The CPU witnesses the birth, the NP carries the life, the CPU handles the death."
Concept 2

The Four Disqualifiers

Even with healthy hardware and acceleration enabled, the CPU refuses to offload a session when any of these hold:

#DisqualifierWhy the NP can't do it
1NP acceleration disabled on the policy (set auto-asic-offload disable)Explicit admin choice — usually a forgotten diagnostic setting. Default is enable.
2Proxy-based security profilesProxies buffer and reassemble whole streams on the CPU. A mixed flow+proxy policy never offloads any of its traffic — one proxy profile poisons the whole policy.
3Session requires a FortiOS session helper (e.g. FTP)Helpers read payload to track dynamically negotiated ports — payload reading is CPU work.
4Tunnelling — SSL VPN, GRE, and other tunnelled interfacesWrapped traffic stays on the CPU… except IPsec: NP6/NP7 encrypt/decrypt ESP natively.
Trench coat rule: tunnels DON'T offload — except IPsec DOES. IPsec is "a forwarding job wearing a trench coat": the NP knows how to take the coat off.
Concept 3

Reading the Evidence

Offloaded session — both directions on NP7
FW-TRADE-01 # diagnose sys session list
...
npu info: flag=0x81/0x91, offload=9/9, ips_offload=0/0, epid=128/166, ipid=166/128, vlan=0x0000/0x0000
  • Code 9 = NP7 as the offloading processor.
  • The slash separates originate / reply directions. Each direction is its own forwarding decision and needs its own key.
  • offload=9/9 — both directions accelerated. offload=9/0 — asymmetric: replies grinding through the CPU, latency graph shows it.
Refused session — FortiOS confesses
FW-TRADE-02 # diagnose sys session list
...
npu_state=0x000001 no_offload
no_ofld_reason: disabled-by-policy
no_ofld_reason is FortiOS confessing. Read the confession before forming a hypothesis.
Concept 4

Config Is the Future; the Session Table Is the Present

The offload decision is made once, at session creation. Existing sessions are never re-evaluated against new config. A long-lived session born under disable carries the no_offload verdict for life — fixing the policy changes only future sessions.

Remediation sequence
FW-TRADE-02 # config firewall policy
FW-TRADE-02 (policy) # edit 12
FW-TRADE-02 (12) # set auto-asic-offload enable
FW-TRADE-02 (12) # end

FW-TRADE-02 # diagnose sys session filter dstip 10.20.5.10
FW-TRADE-02 # diagnose sys session clear

Clients reconnect → new handshake transits the CPU under the corrected policy → key installs → offload=9/9. Then verify — never trust the checkbox, read the session table.

Concept 5

Life of an Offloaded Non-UTM Packet

For an established, offloaded session with no security profiles:

  1. Packet arrives at NP ingress interface.
  2. NP performs IP integrity header checking (mandatory).
  3. IPsec VPN decryption, if applicable (optional).
  4. Skips routing. Skips the kernel. Never touches the CPU. The session key is the pre-computed answer, replayed per packet.
  5. Traffic shaping / WAN optimization (optional).
  6. IPsec VPN encryption, if applicable (optional).
  7. Forwarded out the egress interface by the NP.
Troubleshooting Workflow

The Latency Twins Method

  1. Symptom: identical boxes, divergent latency → hypothesise state, not hardware.
  2. diagnose sys session list on affected traffic → read no_ofld_reason.
  3. Map the reason to the four disqualifiers; fix the config cause.
  4. Verify new sessions offload. If long-lived sessions persist as no_offload: filter + session clear.
  5. Re-verify: offload=X/X both directions, latency restored.
Common Mistakes

Where Engineers (and Juniors) Fall

  • Believing "flow-based parts still offload" on a mixed policy. Nothing offloads on a mixed flow+proxy policy.
  • Treating SSL VPN as ordinary SSL traffic — it's a tunnel, disqualifier #4.
  • Forgetting the IPsec exception in both directions (it's the one tunnel that DOES offload).
  • Declaring victory after a config fix without clearing stale sessions.
  • Leaving auto-asic-offload disable behind after packet sniffing — diagnostic settings are temporary by contract.
Senior Engineering Insights

Lessons Beyond the Blueprint

  • Config describes the future; the session table describes the present. Fixing a policy fixes nothing already alive.
  • Read no_ofld_reason before hypothesising — the box tells you why.
  • One proxy profile poisons the whole policy — offload qualification is per-policy, not per-profile.
  • Add diagnostic-setting reverts to the change checklist; the fix that lasts is procedural, not technical.
Exam Tips

How the Exam Asks This

  • Recognise the exact strings: auto-asic-offload (firewall policy scope, default enable), no_ofld_reason: disabled-by-policy, offload=9/9 (9=NP7).
  • TCP = whole 3WH via CPU; UDP = first packet; FIN/RST = always CPU. All three appear as distractor swaps.
  • Eligibility line-ups: FTP (helper) ✗, SSL VPN (tunnel) ✗, GRE ✗, IPsec ✓, plain traffic no profiles ✓, proxy AV ✗, mixed flow+proxy ✗ (all of it).
  • "Fixed the policy but sessions still slow" scenarios → answer is session clear, not more config.
