SESSION 41 · TROUBLESHOOTING GUIDE

Diagnosing the Slow Path

How a senior engineer finds out why sessions aren't riding the NP — thinking first, commands second.

Symptoms

What the Business Sees

  • Latency far above baseline on specific traffic while other traffic is normal.
  • High CPU on a box whose throughput should be trivially within NP capacity.
  • Two "identical" devices with wildly different performance under the same load.
  • Performance degraded after a maintenance window, a policy change, or "someone was troubleshooting on Friday."
First instinct: identical hardware, different behaviour → different state or config, not different silicon. Don't RMA the box.
Questions to Ask

Before You Touch Anything

  1. Which traffic exactly is slow — one policy? one server? one direction?
  2. Are these new sessions or long-lived flows that predate any recent change?
  3. What security profiles are on the matching policy — any proxy-based ones?
  4. Is the traffic tunnelled (SSL VPN, GRE)? Is it a helper protocol (FTP)?
  5. Was anyone sniffing/debugging recently? (Diagnostic settings left behind are a leading cause.)
Verification

Commands and Expected Outputs

Step 1 — scope, then read the session
# diagnose sys session filter dstip 10.20.5.10
# diagnose sys session list

Healthy (offloaded):

Expected — fast path
npu info: flag=0x81/0x91, offload=9/9, ips_offload=0/0, epid=128/166, ipid=166/128

Faulty (refused):

Expected — slow path, with the reason stated
npu_state=0x000001 no_offload
no_ofld_reason: disabled-by-policy
Step 2 — check the policy setting if implicated
# show firewall policy 12
    set auto-asic-offload disable      ← default is enable; this line only appears when changed
Decision Tree

Follow the Confession

  1. Session shows offload=X/X (both directions)? → Offload is healthy. Latency cause is elsewhere (ISF hop? upstream? — different diagnosis, different session).
  2. offload=X/0 (one direction)? → Asymmetric: reply path not qualifying. Check return-path routing/policy for the reply direction.
  3. no_offload present? → Read no_ofld_reason and map it:
    • disabled-by-policy → auto-asic-offload disable on the policy. Re-enable.
    • Proxy profile on the policy (incl. one proxy in a mixed set) → move to flow-based or accept CPU path.
    • Helper protocol → expected behaviour; not a fault.
    • Tunnelled (non-IPsec) → expected behaviour; not a fault.
  4. Config now correct but sessions still no_offload? → Stale sessions: offload was decided at their birth. diagnose sys session filter + diagnose sys session clear, reconnect, re-verify.
  5. Verify recovery: offload=X/X on new sessions AND latency back to baseline. Both, not either.
Common Dead Ends

Where Time Goes to Die

  • Blaming hardware/RMA when the twin box proves the hardware is fine.
  • Fixing the policy and closing the ticket without clearing stale sessions — the "we enabled it, why is it still slow" loop.
  • Hunting for a proxy profile on the wrong policy — confirm which policy the session actually matched first.
  • Treating helper/tunnel non-offload as a fault to fix — it's designed behaviour.
  • Forgetting to clear the session filter afterwards (Session 8's lesson) and misreading the next investigation.
Likely Root Causes

Ranked by Frequency

  1. Diagnostic setting left behind: auto-asic-offload disable after packet capture.
  2. A proxy-based profile added to a previously flow-only policy (poisons the whole policy).
  3. Stale long-lived sessions surviving a correct fix.
  4. Traffic silently re-routed through a tunnel or helper-dependent protocol after a network change.
Exam Traps

Distractors to Expect

  • "The flow-based portion still offloads on a mixed policy" — false; nothing does.
  • SSL VPN offered as offloadable because "it's just SSL" — it's a tunnel.
  • IPsec listed among non-offloadable tunnels — it's the exception.
  • Scenario where config is fixed but sessions stay slow — the correct answer is session clear, not another setting.
  • offload=9/0 misread as "healthy, 9 is present" — the zero is the story.
