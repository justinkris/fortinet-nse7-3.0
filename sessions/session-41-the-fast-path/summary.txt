TECHNICAL CONCEPTS COVERED
------------------------------------------------------------------
1. Slow path vs fast path. CPU always processes: TCP = full
   three-way handshake; UDP = first packet only. CPU performs
   policy/routing/session creation, then installs the session key
   in the NP. Subsequent packets skip routing + kernel + CPU.
2. FIN/RST always punted to CPU — CPU owns session state;
   teardown = delete kernel entry + uninstall NP key.
   Keeper phrase (mine, adopted by student): "CPU witnesses the
   birth, NP carries the life, CPU handles the death."
3. Four offload disqualifiers (exam list):
   (1) NP acceleration disabled on the policy
       (`set auto-asic-offload disable`, per-policy, default
       enable),
   (2) proxy-based security profiles — and a MIXED flow+proxy
       policy NEVER offloads ANY of its traffic (one proxy profile
       poisons the whole policy),
   (3) session requires a FortiOS session helper (FTP classic),
   (4) tunnelling (SSL VPN, GRE, ...) EXCEPT IPsec — NP6/NP7
       handle ESP natively ("forwarding job in a trench coat").
4. Session-list evidence: `npu info: flag=0x81/0x91, offload=9/9,
   ips_offload=0/0, epid/ipid...` — code 9 = NP7; slash =
   originate/reply directions; each direction needs its own key;
   9/0 = asymmetric offload (replies on CPU). Student read the
   9/0 asymmetry correctly unprompted.
5. Offload decision is made ONCE at session creation; existing
   sessions never re-evaluated after config change. Remediation:
   `diagnose sys session filter dstip <x>` then `diagnose sys
   session clear`.
6. Life of an offloaded non-UTM packet: NP ingress → IP integrity
   check → (IPsec decrypt) → SKIP routing/kernel/CPU → optional
   traffic shaping → (IPsec encrypt) → egress. Session key = the
   pre-computed answer to routing+policy, replayed per packet.
7. Parked for S42 (did not leak, not taught): NTurbo, IPSA,
   flow-vs-proxy UTM acceleration.
