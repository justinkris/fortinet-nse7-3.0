SOCRATIC ARC (student reasoning — strong engineering throughout; two cert/direction
substitutions caught and corrected live)
------------------------------------------------------------------------------------
1. WHY "worked on DCFW" ≠ "belongs everywhere": student attacked the INFERENCE not the
   conclusion — control was matched to the DCFW's threat surface (east-west, DC-grade),
   copying the control without the surface copies only the cost. Added the hardware
   truth (DEFW can't run the DCFW's posture) and INDEPENDENTLY rediscovered the guide's
   "double dipping" concern (same flow inspected twice = same security bought twice,
   latency paid twice). "Double dipping" logged as a keeper phrase.
2. NGFW+DCFW redundancy: student split by DIRECTION before judging duplication (senior
   move). North-south (internet↔DC) crosses BOTH → that's the real redundancy the guide
   flags. East-west (server↔server, replication, backup, fan-out) NEVER touches the edge
   NGFW → DCFW is the SOLE guard, not a backup. Kill line: "no first dip, so no double
   dip." Correctly concluded Cipher is half-right (arm the DCFW — right instinct, wrong
   reason: it's not a backup, it's the only inspector of a room the edge can't see).
3. Pick-one on the shared north-south path: student chose full inspection at the NGFW
   (front-door chokepoint, inspect once, everything behind inherits). CORRECTION MADE:
   student called the DCFW "limited security features" — WRONG; DCFW is 1000–7000 with
   HW SSL inspection, the MOST capable box. Reframed: it's ALLOCATION (reserve DCFW
   capacity for east-west only it can see), NOT incapacity. Student accepted.
4. Protect-the-client vs protect-the-server (the session's crux): student initially
   used "inside vs outside the network" as the deciding axis — a DECOY. Corrected to the
   real axis: WHOSE KEY does the FortiGate hold + WHICH ASSET is being protected.
   Protect-the-CLIENT = outbound, FGT impersonates external server with a RE-SIGNED cert
   (its own CA), front door = edge NGFW. Protect-the-SERVER = inbound to YOUR server,
   FGT holds the REAL server key, sits in front of the asset = DCFW (this was yesterday's
   S25 fix — PHP-in-HTTPS came from the INTERNET yet the fix went on the DCFW, proving
   direction-of-user is NOT the decider). Señor: "You don't guard the server from the
   front gate. You guard it from its own doorway."
5. Same-name-≠-same-job: both NGFW and DCFW run "full SSL inspection" but it's NOT
   double-dipping because they're DIFFERENT FLOWS / directions / keys / assets. Student
   landed this in one sentence ("different flows, serving different purposes"). Junior's
   emergent line: "'is this double dipping' is never yes/no — it's WHICH flow, WHICH
   direction, protecting WHAT." Señor SAT DOWN (his "lesson landed" signal): "Cipher
   counted the controls. You counted the flows. That is the whole difference."
6. DEFW: student gave both sides unprompted (S2 pattern) — no UTM today because the
   branch is a TUNNEL ENDPOINT backhauling to HQ (HQ NGFW already inspects; double-dip +
   30–90 box can't carry the posture anyway); FLIP = local internet breakout creates a
   NEW unguarded door → branch traffic never hits HQ → DEFW must inspect. Guide payoff:
   when breakout arrives, DON'T load UTM on the 30–90 — RE-SIZE to a 100–900 midrange
   FIRST, then enable. Cipher's instinct right, sequence wrong (same shape as DCFW arg).
7. ISFW: primary job = segmentation against LATERAL MOVEMENT; its GIFT is low latency,
   which fights heavy UTM (taxes constant east-west user traffic between floors —
   George's microseconds). Student's flip ("IPS scanners or similar") was too vague —
   coached from a CONTROL NAME to a FLOW: the justified case is IPS on the ISFW to catch
   a KNOWN EXPLOIT SIGNATURE MOVING LATERALLY from a compromised endpoint (the S24
   trading-API incident) — the packet the edge/DCFW never see. Second justified case
   (guide): ISFW wearing an NGFW second hat (terminating VPNs / SD-WAN / secure internet
   access). Pay latency for the specific flow, scoped — not blanket-on.
8. SYNTHESIS / register: forced a Penny-facing one-sentence justification. First attempt
   led with technical nouns ("know the flow, scope it"). Coached to lead with the
   CONSEQUENCE; student's second attempt landed "money is lost" FIRST — the register
   reflex fired. Model close: "Inspecting everywhere would cost us money on every trade
   without catching a single extra attack — so we guard only where the attacks actually
   are." George: "first time security spoke my language." REGISTER WEAKNESS: closing but
   not fully retired (still fires second, not first, under exam pressure — see Q10).

CLI / CONFIG DISCUSSED (concepts more than new syntax this session)
------------------------------------------------------------------
Placement is a POLICY-level decision (which ssl-ssh-profile + which UTM sensors ride
which firewall policy on which box). Reused from S25:
  config firewall ssl-ssh-profile — certificate-inspection vs deep-inspection.
  Protect-the-CLIENT (outbound): FGT re-signs with its CA; endpoints must trust that CA.
  Protect-the-SERVER (inbound, "Protecting SSL Server"): FGT loaded with the real server
    cert+key; used on the DCFW inbound trading policy (S25 carryover).
  config firewall policy → set ssl-ssh-profile / set ips-sensor / set av-profile /
    set webfilter-profile / set application-list — the profiles are ATTACHED HERE, so
    "where to inspect" = "which policy on which box carries the profile."
Verification anchors: diagnose sys session list (confirm which box a flow traverses);
get system performance status (CP/CPU headroom before loading UTM on a box).

TROUBLESHOOTING APPROACH TAUGHT (placement disputes)
----------------------------------------------------
Don't argue the control — TRACE THE FLOW. For each disputed link ask: (1) which
DIRECTION? (2) whose ASSET is protected? (3) whose KEY does the FGT hold? (4) does any
OTHER box already inspect this exact traffic? The answers name the box. "Everywhere" is
always wrong because it's the answer you give when you've traced zero flows. East-west =
DCFW is sole guard. North-south shared path = pick ONE primary decrypt point. Branch =
none until breakout. ISFW = scoped IPS for the lateral flow only.

CONFIGURATION DECISIONS
-----------------------
UDIS rejected. Flow-scoped standard adopted: full inspection OUTBOUND at edge NGFW
(protect-client) + primary north-south decrypt; full inspection INBOUND at DCFW
(protect-server) + IPS + sole east-west guard; north-south redundancy on the shared
path TRIMMED to one primary point; ISFW carries SCOPED IPS for lateral-movement only,
latency otherwise preserved; branch DEFWs remain TUNNEL-ONLY, flagged "UTM pending
local breakout → re-size to 100–900 first." No new hardware.
