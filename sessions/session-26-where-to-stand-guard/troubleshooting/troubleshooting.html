SENIOR-ENGINEER THINKING · UTM PLACEMENT

Inspecting the Argument

Placement disputes aren't fixed by turning things on. They're resolved by tracing the flow.

Don't argue the control. Trace the flow.
The person who wins the "inspect everywhere" fight is the one who can name which flow, which direction, which asset.
01 · Symptoms of the Dispute

When You Hear These

  • "If it's good on box A, put it everywhere / one uniform standard / no blind spots."
  • "Belt and braces — inspect at both boxes on the path to the servers."
  • "The branch has no UTM, that's a security gap."
  • "Turn on IPS/AV/web filter on the internal segmentation firewall too."
  • Countered by: "you're stealing my latency / microseconds / throughput."

These are architecture symptoms, not fault symptoms. The cure isn't a config change — it's a flow trace.

02 · The Four Questions

Ask in This Order

#QuestionWhat it settles
1Which direction? North-south (crosses the edge) or east-west (never does)?East-west → the DCFW is the sole guard, full stop. North-south → keep asking.
2Whose asset? Shielding a browser, or a box being connected to?Browser → protect-client. Server → protect-server.
3Whose key? Does the FGT re-sign, or hold the real key?Re-signed CA → edge NGFW. Real server key → the box in front of the server (DCFW).
4Who else already inspects this exact flow?If another box on the path does the same job → redundancy → pick one primary point.
03 · Verification Commands

Prove the Flow Before You Decide

Confirm which box a flow actually crosses
diagnose sys session list          # is this flow even ON this box? which iface in/out?
diagnose sniffer packet any 'host <server> and port 443' 4   # see it live (TAC's rule)
Check headroom before loading UTM
get system performance status      # CPU + session load — can this box carry inspection?
diagnose npu np6 ... / fnsysctl    # offload state — is HW SSL actually engaged?
get firewall ssl-ssh-profile <name>  # cert-inspection vs deep? protecting-server set?
Expected sanity: east-west server-to-server sessions should appear on the DCFW and never on the edge NGFW. If someone claims the edge is "covering" east-west, this list disproves it in one command.
04 · Decision Tree

From Symptom to Placement

Placement decision tree
"Should we inspect this flow here?"
│
├─ Is it EAST-WEST (server↔server, replication, backup)?
│    └─ YES → the edge can't see it. DCFW is the SOLE guard. Inspect here. ✔
│
├─ Is it NORTH-SOUTH (internet ↔ inside)?
│    ├─ Does another box on the path already run the SAME profile on this flow?
│    │    └─ YES → redundancy. Pick ONE primary decrypt point (usually the edge). Trim the other.
│    └─ Are we protecting OUR server (inbound, we hold the key)?
│         └─ YES → inspect in FRONT of that server (DCFW), regardless of client origin. ✔
│
├─ Is it BRANCH traffic?
│    ├─ Backhauled to HQ? → HQ already inspects. No branch UTM. ✔
│    └─ Local internet breakout? → new unguarded door. Re-size branch to 100–900, THEN UTM. ✔
│
└─ Is it INTERNAL segment↔segment (ISFW)?
     └─ Only add IPS for a SPECIFIC lateral-movement threat (known exploit sig).
        Otherwise keep it bare — its job is low-latency segmentation. ✔
05 · Common Dead Ends

Where the Argument Goes Wrong

  • "The DCFW has limited security features, so inspect at the edge." False premise — the DCFW is the most capable inspector you own. You're allocating its capacity to east-west, not admitting it can't. Don't hand this line to Cipher; he'll correctly shoot it down.
  • "Internet traffic, so it's always the edge's job." Origin is a decoy. If the asset being protected is your own server, the inspection belongs in front of that server.
  • "Two boxes run full inspection — that's a double-dip, remove one." Only if it's the same flow. Outbound-client and inbound-server aren't.
  • "Just turn UTM on at the branch." On a 30–90 box that's a physics violation. Re-size first.
  • "More inspection is more security." Only where the threat travels. Everywhere else it's pure latency for zero detection.
06 · Likely Root Causes

What's Really Going On

Nearly every placement dispute traces to one of three misreadings:

  1. Reasoning about the control's name instead of the flow underneath it (Cipher's whole memo).
  2. Confusing redundancy with defence in depth — same control + same flow ≠ depth.
  3. Using packet origin instead of asset ownership to decide client vs server inspection.
07 · Exam Traps

How the Test Bends This

  • "Uniform / everywhere / all firewalls" answers are almost always the distractor.
  • A stem that calls duplicate north-south inspection "defence in depth" — it's the flagged NGFW↔DCFW redundancy.
  • A T/F with "never / any / always / regardless" about the ISFW — the absolute qualifier makes it false (ISFW can wear an NGFW hat).
  • Client/server questions decided by which certificate the FGT presents, not by traffic direction.
  • Commit the verdict your reasoning proves. The graded answer is the label, not your explanation.
