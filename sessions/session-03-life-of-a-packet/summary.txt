TECHNICAL CONCEPTS COVERED
---------------------------
1. Triage principle: cheap checks (ACL, HPE, IP header integrity) run BEFORE
   expensive checks (IPS, AV, SSL decrypt) — don't spend costly engines on traffic
   that hasn't proven it deserves scrutiny. Student derived this unprompted from
   "cost," not from criteria.
2. NP (network processor) vs CP (content processor) split: NP = network-level job,
   purpose-built hardware for wire-speed forwarding/header work at the interface;
   CP = content-level job, pattern matching (IPS), SSL/TLS decrypt, AV signature
   work. Student derived the CP's existence and guessed "CP9" (correct family)
   before being told the name.
3. Session establishment mechanics: for TCP, the ENTIRE three-way handshake (not
   just the first packet) goes through the CPU; for UDP, only the first packet.
   Once the session key is installed in the NP, subsequent packets fast-path and
   skip the CPU. FIN/RST packets ALWAYS return to the CPU regardless of how long
   the session had been offloaded — session state changes (setup AND teardown) are
   CPU-owned; the NP only forwards for sessions it already recognizes.
4. Kernel pipeline order, self-derived by the student with correct reasoning at
   each step: Destination NAT -> Routing decision (+ RPF check + SD-WAN) -> Policy
   lookup/stateful inspection -> Source NAT. Reasoning: DNAT must resolve the real
   destination before routing can pick an egress interface; policy lookup needs the
   resolved destination interface, not the raw packet; SNAT happens last, on the way
   out, after policy has already made its decision.
5. Segmentation-is-real check: internal (east-west) traffic runs the IDENTICAL
   pipeline as north-south traffic. DNAT can be a no-op if there's no VIP configured
   for internal traffic, but POLICY LOOKUP IS NEVER A NO-OP — implicit deny applies
   universally. This is what makes segmentation real rather than decorative
   (direct callback to Session 2's "VLAN with gateway on switch isn't segmentation").
6. NP Direct Architecture (stretch content, correctly reasoned by the student):
   available on FortiGates with 2+ NP6 chips or NP7 ultralow-latency interfaces;
   requires traffic to stay on ONE NP chip's offloaded interfaces for the full
   latency benefit. General NP offload CAN still work across two different NP
   chips via the internal switch fabric (ISF) — it's just slower than NP Direct,
   not impossible. Student initially overstated this as a hard requirement for ALL
   offload; corrected cleanly with the ISF explanation. Directly relevant to
   LoFinance's low-latency trading-floor stakes.
7. Real CLI introduced: `get hardware status` (shows model + ASIC version — NP6/
   NP7/CP9/CP10/SoC-SP for entry/mid-range); `diagnose hardware lspci | grep 1a29`
   (1a29 = Fortinet vendor ID; device ID 4e36 = NP6, 4e37 = NP7 — needed because a
   single ASIC-version summary line doesn't tell you which physical interfaces map
   to which specific chip on multi-NP boxes, which matters for NP Direct topology
   planning). Student reasoned to WHY the second command is needed by connecting it
   back to his own Session 2 point about needing to know interface-to-NP mapping.

TROUBLESHOOTING APPROACH
--------------------------
Pipeline-as-map: implicit deny -> check policy stage; missing/wrong route -> check
routing stage; NP session-limit overflow forcing traffic to CPU -> check offload
stage capacity; CPU load spikes from session churn -> check FIN/RST teardown volume.
`get hardware status` / lspci confirm WHAT hardware exists; the pipeline tells you
WHERE to look for a given symptom.
