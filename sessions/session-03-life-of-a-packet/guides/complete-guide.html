● Study Guide · NSE7 EF 7.6 · Session 03

Life of a Packet

Marcus's order enters through an interface and leaves through another. Between those two points is the entire architecture of FortiOS — and by the end of this guide, you can trace every stage of that journey by name.

The pipeline is identical everywhere — placement only changes which policies and profiles apply to it.
Section 1

Why Cheap Checks Come First

Before any routing decision, before NAT, before deep content inspection — FortiGate runs a small set of mandatory checks right at the network interface: Access Control List (ACL), Host Protection Engine (HPE), and IP integrity header checking.

The engineering logic is triage before you invest: these checks are cheap — a few cycles of "is this header well-formed, is this traffic allowed on this interface at all." Running malformed or clearly-illegitimate traffic through an expensive engine like IPS or antivirus first would waste the FortiGate's most valuable resource (CPU/CP cycles) on traffic that never deserved the scrutiny.

Senior insight: good security architecture spends expensive resources last, on traffic that has already earned the right to be examined closely.
Section 2

Two Specialists: NP and CP

FortiGate hardware acceleration splits into two purpose-built processor families, both called an SPU (security processing unit):

ProcessorJobHandlesLatest model
NP (Network Processor)Network-level — works at the interfaceForwarding/offload, IP integrity checks, IPsec encrypt/decryptNP7 (up to 12M sessions)
CP (Content Processor)Content-level — works on payload/application dataIPS pattern matching, SSL/TLS decrypt for deep inspection, AV signature workCP10

Entry-level and mid-range FortiGates combine both onto a single chip called an SP (e.g. SP5) — a lighter-weight version of both NP and CP on one die, at lower throughput.

Mental model

NP is the specialist who only moves things — fast, wire-speed, no judgment calls. CP is the specialist who reads what's inside — slower, but the one you trust to actually inspect content. Same principle as ISFW-vs-DCFW role specialization from Session 2, just at the chip level instead of the box level.

Section 3

Session Establishment: What Stays on the CPU

The very first packets of any session must pass through the CPU, because there is no session yet for the NP to recognize:

  • TCP: the entire three-way handshake (SYN, SYN/ACK, ACK)
  • UDP: just the first packet

Once the CPU establishes the session and installs a session key into the NP, all subsequent packets for that session take the fast path — routed directly by the NP, bypassing the CPU's routing and kernel processing entirely.

Exception that matters: FIN and RST packets — which tear a session down — always go back through the CPU, no matter how long that session had been fast-pathing through the NP. Session state changes (both setup and teardown) are CPU-owned; the NP only forwards for sessions whose state it already knows.
Section 4

The Kernel Pipeline, in Order

For the first packets of a new session, FortiOS runs through a fixed sequence. Here is the order, with the reasoning for why it can't be any other way:

Interface In
→
NP: ACL / HPE / Header Check
→
Destination NAT
→
Routing / RPF / SD-WAN
→
Policy Lookup / Stateful Inspection
→
CP: UTM/NGFW Content Inspection
→
Source NAT
→
NP: Egress Fast Path
→
Interface Out
  1. Destination NAT — if the packet's addressed to a virtual IP, FortiGate must resolve the real destination before anything downstream can act on it.
  2. Routing decision (+ RPF check + SD-WAN path selection) — needs the real destination resolved first, to pick the correct egress interface.
  3. Policy lookup / stateful inspection — needs the resolved destination interface to match a policy; can't evaluate "is this session allowed" against a raw, unrouted packet.
  4. Source NAT — happens last, on egress, after policy has already made its decision, so hiding the real source address can't interfere with anything upstream.
Section 5

Is Segmentation Real? The Pipeline Says Yes

Does an order that never leaves the building (floor-to-floor, east-west) skip any pipeline stages that an order headed to an exchange (north-south) would go through?

Destination NAT can be a no-op if there's no virtual IP configured for internal traffic — the stage still runs, it just has nothing to translate.

Policy lookup is never a no-op. Every packet — internal or external — is evaluated against policy, or it dies at the implicit deny rule at the end of the policy list. There is no "internal traffic, skip the check" shortcut anywhere in FortiOS.

Why this matters: this is exactly what makes segmentation real instead of decorative (callback to Session 2's "VLAN with gateway on a switch isn't segmentation"). The pipeline is the mechanism that forces inspection, regardless of where traffic originates.
Section 6

NP Direct Architecture (Stretch)

General NP offload can still succeed even when ingress and egress interfaces sit on different NP chips — the chips communicate over the internal switch fabric (ISF).

NP Direct Architecture is a specific, more aggressive feature: on FortiGates with two or more NP6 processors, or NP7 devices with ultralow-latency interfaces, the network processor is wired directly to the interfaces, eliminating the ISF hop entirely. This requires all traffic to pass through the offloaded interfaces of one NP chip — cross-chip traffic loses the direct-path benefit and falls back to ISF-mediated (still offloaded, just not lowest-latency).

Why LoFinance cares: on a trading floor, the difference between an ISF hop and a direct NP path is measured in microseconds — and microseconds are money.
Section 7

Verification Commands

Identify the ASIC / model on the box
# get hardware status
Model name: FortiGate-900G
ASIC version: CP9
Identify NP/CP hardware IDs on the PCI bus
# diagnose hardware lspci | grep 1a29
9b:00.0 Class 1000: Device 1a29:4e37   ← NPID 4e37 = NP7 (4e36 = NP6)

Why the second command matters: get hardware status gives you one ASIC-version summary line for the whole box. On a FortiGate with two or more NP chips, that single line doesn't tell you which physical interfaces map to which specific chip — information you need to plan NP Direct topology correctly and to distribute sessions evenly across NP processors.

Section 8

Exam Tips

  • Watch qualifying words like "established" and "still" — a question about an already-offloaded session is asking about what happens after establishment (FIN/RST), not the handshake that created it.
  • The pipeline order (DNAT → Routing → Policy → SNAT) is a recurring diagram source — memorize the reasoning, not just the sequence, since questions often test edge cases (e.g. "what if there's no VIP?").
  • NP Direct requires same-chip interfaces; general NP offload does not — don't conflate the two.
  • Virtual FortiGates run the identical pipeline logically, but the CPU handles everything — there's no NP/CP to offload to.
Key Takeaways

What to Carry Forward

  • Cheap checks before expensive ones — triage is a design principle, not just a FortiGate quirk.
  • NP = network-level specialist; CP = content-level specialist — same specialization logic as firewall role placement.
  • Session state changes (setup, teardown) are CPU territory; everything else in an established session can fast-path.
  • The pipeline never skips policy lookup — that's what makes segmentation enforceable rather than cosmetic.
