Session 03 · Troubleshooting Guide

Thinking Like the Box: Where Did the Packet Get Stuck?

Symptom-First Reasoning

The pipeline you learned this session isn't just theory — it's your diagnostic map. Every symptom below maps to a specific stage. The goal is to think in stages, not commands.

Symptom: Traffic is unexpectedly blocked

Ask first: is this the FIRST packet of a new session, or a session that was previously working?

Likely stage: policy lookup / implicit deny. Remember — this stage is never a no-op, for any traffic, internal or external.

Dead end to avoid: assuming "it's internal traffic, it must be a routing problem" — check policy match before anything else.

Symptom: Performance degrades under high session count

Ask first: is the NP hitting its session limit? NP7 supports up to 12 million sessions, limited by processor memory.

Likely stage: offload/fast-path stage. Once an NP hits its session limit, overflow sessions get sent to the CPU — the slow path — which degrades performance broadly, not just for the overflow sessions.

Verification:

Check hardware / ASIC in use
# get hardware status

Fix direction: distribute incoming sessions evenly across multiple NP processors — requires knowing which interfaces map to which NP chip.

Symptom: A specific FortiGate model isn't offloading traffic the way you expect

Ask first: what hardware does this box actually have — NP6, NP7, or the combined SP chip on entry/mid-range models?

Confirm NP/CP hardware IDs directly on the PCI bus
# diagnose hardware lspci | grep 1a29
Device 1a29:4e36  → NP6
Device 1a29:4e37  → NP7

Common dead end: assuming every FortiGate model has separate NP and CP chips — entry/mid-range models combine both into one SP chip at lower throughput.

Symptom: CPU usage is high despite offload being configured correctly

Ask first: is this a session-churn problem — lots of sessions opening and closing rapidly? Remember: the ENTIRE TCP handshake, and ALL FIN/RST packets, always hit the CPU.

Likely root cause: high session turnover (e.g. lots of short-lived connections) forces proportionally more traffic through the slow path, even with offload working correctly for long-lived sessions.

Dead end to avoid: disabling NP acceleration to "simplify troubleshooting" — this doesn't fix a churn problem and removes your fast path entirely. Isolate whether it's churn vs. a genuine offload misconfiguration first.

Decision Tree: "Where in the pipeline is this packet stuck?"

Is the session brand new?
→ Yes: expect CPU involvement (handshake/first packet) — this is normal, not a fault.
Is the session established but still slow?
→ Check NP session limits and interface-to-NP mapping.
Is traffic being dropped outright?
→ Check policy lookup / implicit deny — this stage runs for ALL traffic, no exceptions.
Is a security profile (IPS/AV/SSL) involved?
→ Check CP capacity/version — content inspection is CP territory, separate resource from NP.
Is this virtual FortiGate?
→ No NP/CP exist — CPU handles everything. Don't troubleshoot for offload that can't happen.

Exam Traps in This Territory

Trap 1: "For an established session already offloaded, which packets still hit the CPU?" — the answer is FIN/RST, NOT the handshake. The handshake happens BEFORE establishment, so it can't be something that's "still" happening on an already-established session.
Trap 2: "NP7 can't offload across two chips" — false. General offload survives across chips via the internal switch fabric (ISF). Only NP Direct Architecture requires same-chip interfaces.
Trap 3: Assuming internal/east-west traffic skips pipeline stages because it "never leaves the building." DNAT can be a no-op without a VIP — policy lookup never is.
