TECHNICAL CONCEPTS COVERED (EF pp. 147-151)
==================================================================
1. BGP CONVERGENCE has multiple steps (p.148): fast failure detection and propagation;
   processing and finding an alternate path; installing the new path in RIB/FIB. Levers:
   stable network (no port flapping), reduce RIB size with in/out filters, route reflectors
   in IBGP to concentrate updates, faster detection via keepalive tuning or BFD, and
   graceful-restart for HA failover.
2. KEEPALIVE / HOLD TIMERS: FortiGate defaults 60s / 180s. The 3:1 ratio exists so that TWO
   keepalives can be lost to congestion without tearing down a healthy session. BGP is
   DELIBERATELY, STRUCTURALLY PATIENT.
   THE CENTRAL SENTENCE OF THE EPISODE (reuse this): "BGP's only evidence of life is a
   message that arrives on a schedule. Its only evidence of death is the absence of that
   message for a long time."
3. WHY NOT JUST LOWER THE TIMERS (student derived both halves):
   (a) FLAP RISK — aggressive timers detect non-failures as failures. Teardown is expensive:
       every route from that peer withdrawn, withdrawal propagates, best-path reruns, FIB
       rewritten, then full re-advertisement on re-establishment.
   (b) CPU — BGP runs in the routing daemon, in SOFTWARE, on the CPU. Not NP-offloaded. Scales
       badly across many peerings, and load is highest precisely when routes are churning.
4. BFD (p.150): a probe checking peer status. Media-independent. Detects a ONE-WAY DEVICE
   FAILURE in LESS THAN A SECOND. Originally single-hop/same-subnet; FortiGate now supports
   multihop.
   The key reframe: BFD asks "does the DEVICE answer?" not "is the INTERFACE up?" — which is
   precisely the Friday failure mode.
   ARCHITECTURAL POINT (student derived): separate the LIVENESS QUESTION from the ROUTING
   PROTOCOL, and the liveness question becomes cheap enough to ask constantly.
5. BFD CONFIG — TWO PLACES, BOTH REQUIRED:
   - per neighbour: config router bgp > config neighbor > set bfd enable
   - per interface: config system interface > set bfd enable
   Interface timers: bfd-desired-min-tx (default 250ms), bfd-required-min-rx (default 250ms),
   bfd-detect-mult (default 3). 250ms x 3 = 750ms detection.
   NOTE the pedagogical point that landed: BFD uses the SAME 3x logic as BGP. Not a smarter
   idea — the same idea made cheap.
6. BFD IS BIDIRECTIONAL — negotiated session (Down -> Init -> Up), both ends must run it.
   THE QUIET-FAILURE MODE (this landed hard): one-sided BFD means the BFD session never
   reaches Up, BGP DOESN'T CARE, peering stays Established, dashboard stays green, and you
   are still on a 180s hold timer having told Penny the risk is closed.
7. BFD MULTIHOP (p.150): uses config router bfd > config multihop-template, matching src/dst
   SUBNETS rather than binding to an interface, with auth-mode md5 + md5-key. Paired with
   ebgp-enforce-multihop on the neighbour.
   Student derived the risk unaided: latency/jitter across hops you don't own makes a healthy
   path look dead. Therefore MULTIHOP BFD NEEDS LOOSER TIMERS THAN SINGLE-HOP.
8. WHEN BFD IS A BAD IDEA (student-derived, three cases):
   (a) NO ALTERNATIVE PATH — detection without reconvergence is a MONITORING benefit, not an
       AVAILABILITY control. Student got this cleanly and unprompted.
   (b) UNCOOPERATIVE/UNKNOWN PEER — your thresholds are only as good as the worst normal
       behaviour of a path you don't own. Becomes a contract question, not a CLI question.
   (c) MANY PEERINGS — student initially said "more BFD would be good," which was the one
       miss of the session. Corrected: BFD relocates the CPU constraint rather than removing
       it. 250ms = 4 pps each way PER SESSION. 50 spokes = 200 tx + 200 rx per second forever.
       Design lever: TUNE THE INTERVAL TO THE VALUE OF THE ROUTE.
9. GRACEFUL RESTART (p.151): the BGP daemon runs ONLY on the primary. FGCP syncs the session
   table for TRANSIT traffic, but a BGP peering TERMINATES ON the box and lives in a userland
   process's memory. Processes do not fail over. Therefore the peering must be re-established
   from scratch, and the peer would normally withdraw all routes.
   config router bgp > set graceful-restart enable (global) AND per-neighbour
   set capability-graceful-restart enable. Both ends must negotiate it.
   THE PARADOX worth reusing: the new primary can FORWARD perfectly; it is ROUTING that
   announces a death which was actually a change of shift.
10. BFD AND GRACEFUL RESTART ARE OPPOSITE TOOLS:
    BFD shouts "he's dead, reroute NOW." Graceful restart whispers "he's restarting, DON'T
    reroute." On the same neighbour they are in genuine tension. BFD WINS — it hands BGP a
    hard verdict and the session tears down before graceful restart's stale-route logic can
    apply. Student derived the conflict and then named the winner correctly when pushed.
11. THE DESIGN RULE (the takeaway sentence): match detection speed to the failure you actually
    EXPECT on that link, not to the fastest number the platform allows.

  - Sydney: Active-Passive cluster, VM (all software — crypto, tunnel, BGP daemon, BFD state).
  - Melbourne: SINGLE standalone FortiGate.
  - Two IPsec tunnels from DC (one to Sydney, one to Melbourne), BGP peering over each.
Teaching resolved per-neighbour:
  - DC <-> carrier-type link: expected failure is silent far-end death, no HA behind it.
    Aggressive BFD. Buy the 750ms.
  - DC <-> Sydney (HA cluster over IPsec): FGCP DOES sync IPsec SAs, so the TUNNEL survives
    failover — but the BGP daemon does not. Most likely interruption is a FAILOVER, not a
    site death. GRACEFUL RESTART IS THE PRIMARY TOOL. BFD, if used, must be RELAXED — above
    Sydney's realistic failover-and-reconverge time — so a shift change isn't read as a
    funeral.
  - DC <-> Melbourne (standalone): no HA, so graceful restart protects nothing. BFD is
    straightforwardly useful; if that box is gone it is gone.
THE HARDWARE POINT: the 2601F offloads IPsec to NP7 silicon; the Sydney VM does everything in
software on shared vCPU. THE SLOWER PEER SETS THE FLOOR — always. The student was left on this
question at the end of the technical phase and it should be retested cold in S58 or S59.
THE INDISTINGUISHABILITY POINT (the hardest idea of the session, and it landed): HA failover
and total site death are IDENTICAL at second zero from the DC's point of view. You cannot
configure two behaviours for two events you cannot tell apart. The design question is
therefore not "which tool per scenario" but "what makes them distinguishable, and how long am
I willing to wait to find out?"

TROUBLESHOOTING APPROACH (the ladder built this session)
==================================================================
Failover took ~180s              -> is BFD actually Up, or just configured?
                                    get router info bfd neighbor
BFD configured, still slow       -> enabled on BOTH the neighbour AND the interface?
BFD session never comes Up       -> does the FAR END run BFD? (not a CLI problem)
BFD flaps on a healthy path      -> multihop running single-hop timers? check jitter
Routes withdrawn on HA failover  -> was graceful restart NEGOTIATED or only configured?
Detection fast, no recovery      -> is there an alternative path at all?
THE ORDERING PRINCIPLE: verify the detector is ALIVE before blaming the thing it watches.

EXAM-RELEVANT NOTES
==================================================================
- 60s / 180s defaults and the 3:1 RATIONALE (two lost keepalives tolerated) — high-probability.
- "One-way device failure in less than a second" is the study guide's OWN wording (p.150) and
  is near-certain to appear in some form.
- BFD requires enabling in TWO places (neighbour AND interface). Classic partial-config trap.
- 250ms x 3 = 750ms — expect an arithmetic question.
- multihop-template with src/dst SUBNETS + md5 — distinguishes single-hop from multihop config.
- graceful-restart needs BOTH the global command AND the per-neighbour capability command.
  Expect a snippet with one of the two missing.
- "BFD is configured but the session won't come Up while BGP stays Established" -> far end
  isn't running BFD. Excellent scenario-question material.
- THE DISTRACTOR TO WATCH: on any "failover was slow" question, the tempting answer is a BGP
  timer value. The better answer is usually that BFD is not actually Up, or is enabled in only
  one of the two required places.
- BFD on a single-homed peer is NOT an availability control. Trap answer material.
