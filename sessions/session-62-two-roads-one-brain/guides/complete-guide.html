Two Roads, One Brain

The routing table can tell you a path exists. It has never been able to tell you the path is any good.

Nothing was broken. That was the problem.

Baz’s link passed every test anyone had thought to run. Session 62 is about building the test nobody had run yet — and about the difference between a link that is up and a link that is worth using.

💭 THINK FIRST · REACHABILITY VS QUALITY

Baz’s link has been up all month. His dashboard is green, and his dashboard is not lying. Yet every day for roughly forty minutes the trading desk says the world has gone slow, and by the time anyone looks, it has stopped. Nothing is broken. Nothing is misconfigured. Before you reach for a feature, work out what the firewall could possibly have known.

  • A route to 0.0.0.0/0 exists via port2. What information does that route entry actually contain — and what could it never contain?
  • If two default routes are equal-cost and one of them is quietly running at 400ms with 5% loss, what does the FortiGate do with traffic, and why?
Q1 — MODEL ANSWER
A route entry holds a destination prefix, a next-hop gateway, an outgoing interface, an administrative distance and a priority. There is no field for latency, jitter or loss. It is not that the routing table ignores quality — it has no vocabulary for quality at all. It can answer exactly one question: is there a path?
Q2 — MODEL ANSWER
It load-balances across both, because as far as ECMP is concerned they are identical. The degraded link is chosen roughly half the time. This is why the symptom is intermittent and unreproducible: whether a given trade is slow depends on which member the session hashed to.
SECTION 01

Up Is a Fact. Good Is a Measurement.

The route entry has no field for how the journey felt.

Everything in this session grows from one distinction, and it is worth stating in its plainest form. The routing table knows reachability. It does not know quality.

Look at what a route entry contains: a destination, a next-hop, an interface, a distance, a priority. Now try to point at the field that would hold “this path was 400ms and dropping one packet in twenty at 09:41 this morning.” There isn’t one. There has never been one. Routing was designed to answer is there a path, and it answers that question extremely well.

So when Baz says his link is up, Baz is telling the truth. His link is up. The interface has carrier, the gateway responds to ARP, the static route is installed and valid. Every promise Baz has ever made has been kept. He is simply answering a different question from the one Penny is asking — and the reason nobody caught it for a month is that the firewall was answering the same question Baz was.

Common misconception: “If the link were really bad, the route would drop out.” No. A brownout is not an outage. Carrier, ARP and the static route all survive congestion, buffer bloat, and a saturated upstream. The route is perfectly valid the entire time the user experience is unusable.

This is why the tickets could never be closed. Riley had eleven of them, all saying it was slow, it’s fine now. There was no fault to find because there was no fault — there was an unmeasured path.

💭 THINK FIRST · PERFORMANCE SLAS

If the routing table has no vocabulary for quality, something has to go out and get the vocabulary. That something is a performance SLA — an active health check that sends real packets down each member and records what comes back. Before looking at the syntax, decide what it should be measuring and how it should ask.

  • Riley has two complaints on his board: ‘the trading platform freezes then catches up’ and ‘my calls sound like the ocean.’ What can happen to a probe packet, and which complaint does each explain?
  • An administrator sets the health-check server to the ISP’s own next-hop gateway address. What exactly has that proved, and what has it failed to prove?
Q1 — MODEL ANSWER
Three things can happen to a probe: it comes back slowly (latency), it comes back at inconsistent intervals (jitter), or it doesn’t come back (packet loss). The freeze-and-catch-up is TCP meeting loss — TCP recovers, so nothing breaks, but recovery costs time and the user perceives a stall. The ocean is UDP meeting loss and jitter — nothing recovers, the gap is permanent, and the ear fills it with noise.
Q2 — MODEL ANSWER
It has proved that the ISP’s router is alive and will answer. It has proved nothing about the path beyond that router, which is where a brownout almost always lives. Probe something that represents the real destination, past the carrier’s first hop.
SECTION 02

The Thing That Goes and Measures

Three metrics, three different ways for a probe to disappoint you.

A performance SLA (called a health check in the CLI) is the component that converts an opinion into a number. It sends real traffic out of every listed member toward a target you choose, and it records latency, jitter and packet loss for each one, continuously.

FortiOS 7.6 CLI — performance SLA
config system sdwan
    config health-check
        edit "Trading-SLA"
            set server "10.99.1.10"
            set members 1 2
            set protocol http
            set interval 500
            set failtime 5
            set recoverytime 10
            config sla
                edit 1
                    set link-cost-factor latency jitter packet-loss
                    set latency-threshold 150
                    set jitterthreshold 30
                    set packetloss-threshold 2
                next
            end
        next
    end
end

How you ask matters as much as what you ask

The probe protocol is configurable, and the choice is a real engineering decision rather than a formality. FortiOS 7.6 supports ping (ICMP) — the default — plus tcp-echo, udp-echo, http, https, dns, twamp and ftp.

Carriers do not treat all traffic identically. ICMP is frequently de-prioritised in a queue, rate-limited, or handled on a router’s slow path. That means an ICMP probe can report a problem the real traffic never experiences, or miss one it does. Probe the way the traffic travels. For a trading platform served over HTTPS, an HTTPS probe to a target that behaves like the real destination gets you far closer to what the desk actually feels.

SettingWhat it controlsDesign note
serverThe probe targetMust sit past the carrier’s first hop. Probing the gateway proves only that the gateway lives.
protocolHow the probe is sentMatch the real application where you can. ICMP is the default, not the best answer.
intervalMilliseconds between probesFaster detection, more probe traffic. 500ms is a common starting point.
failtimeConsecutive failures before out-of-SLAMultiply by interval to get real detection time.
recoverytimeConsecutive successes before back in-SLAMake this larger than failtime. Recovery is a promise, not an event.
link-cost-factorWhich metrics are judgedAND logic. Every listed factor must pass. One breach fails the whole SLA.
Exam tip: link-cost-factor latency jitter packet-loss with latency and jitter under threshold and loss over threshold means the member is out-of-SLA. All listed factors must be satisfied simultaneously. This is a favourite question shape.
Remember — the threshold is not a technical number. 150ms did not come from Fortinet. It came from the trading desk. Señor’s formulation: “the threshold is a business decision wearing a technical costume.” It is the exact point where Penny’s requirement enters the network as configuration.
Analogy

The SLA is a thermometer. A thermometer does not drive anyone to hospital.

  • The health check measures. It reads 39 degrees and it says “red.”
  • The reading changes nothing on its own. The patient is still sitting in the chair.
  • Something else has to read the thermometer and act. That is the SD-WAN rule, and it is a different object entirely.
  • Confusing the sensor with the actuator is the most common way people mis-describe SD-WAN.
🧠
MENTAL NOTE
The SLA produces a verdict: in-SLA or out-of-SLA. It never moves a packet. Measurement and action are two separate machines.

💭 THINK FIRST · UP, ELIGIBLE, PREFERRED

The SLA has just gone red on Baz’s link. J.R. asks the naive question that turns out to be the whole episode: does the link go down? Junior gets halfway to the answer and stalls on it — ‘it can’t go down, it’s still up, that’s the whole thing.’ Finish his sentence properly.

  • When a member goes out-of-SLA, what changes in the routing table?
  • A member is up, has a valid route, and is in-SLA. Give a reason it might still never carry a single packet of trading traffic.
Q1 — MODEL ANSWER
Nothing. The static route via that member stays exactly where it was, valid and installed. The FortiGate has not forgotten how to reach the internet through Baz. What changed is the member’s standing as a candidate inside SD-WAN rule selection — a status that lives in the SD-WAN engine, not the RIB.
Q2 — MODEL ANSWER
Because it is not listed in that rule’s member list. A member can be up, routable and healthy and still be ineligible for a particular class of traffic, simply because you never offered it as an option. Up, eligible and preferred are three separate states.
SECTION 03

Three States, Not Two

A member must clear three separate gates before it carries your packet.

Most people carry a two-state model of a link: it is up, or it is down. SD-WAN needs three, and keeping them separate is the difference between troubleshooting confidently and guessing.

  • UP — a route exists. This is a routing-table fact. Nothing to do with quality.
  • ELIGIBLE — this member appears in the member list of the rule that matched this traffic. This is a configuration fact, and it is decided by you, in advance.
  • PREFERRED — of the eligible members, this is the one the strategy currently selects, given the SLA verdicts and the ordering. This is a runtime fact and it changes without you touching anything.

Baz’s brownout moves his link from preferred to merely eligible. It never touches up. That is precisely why the route stays in the table — and precisely why the classic exam distractor (“the route is withdrawn”) is wrong.

Watch this trap: asked which link SD-WAN will choose, it is tempting to answer “the one with the lowest latency.” That skips a gate. SD-WAN never surveys every interface on the box — it chooses only from the members the matched rule offered it. Constraint first, then quality.
🧠
MENTAL NOTE
Out-of-SLA does not remove a route. It removes a candidate. Ask ‘up, eligible, or preferred?’ before you ask anything else.

💭 THINK FIRST · MEMBERS AND ZONES

You have two roads and a thermometer. Now build the structure that holds them. A member is the unit SD-WAN selects between; a zone is how you stop that selection from leaking into everybody else’s configuration.

  • FortiOS generates the static route for an SD-WAN member automatically. What must you supply for it to be able to do that?
  • Cipher writes a firewall policy with dstintf set to the zone underlay. Baz sells LoFinance a third link next quarter and you add port3 to that zone. What does Cipher have to change?
Q1 — MODEL ANSWER
An interface and a gateway. The interface says where packets leave; the gateway is the next-hop the auto-generated route points at. Zone, cost and priority are optional refinements — a member is perfectly valid without any of them.
Q2 — MODEL ANSWER
Nothing. That is the entire point of a zone. Membership changes underneath the policy without the policy moving. Without zones, somebody has to remember to add port3 to every WAN-facing policy at every site — and the one they forget becomes a link with no inspection that nobody notices until 2am.

Members, and the Seam Between Teams

Policies plug into the zone. Membership changes inside it without anybody noticing.

FortiOS 7.6 CLI — zone and members
config system sdwan
    set status enable
    config zone
        edit "underlay"
        next
    end
    config members
        edit 1
            set interface "port1"
            set zone "underlay"
            set gateway 203.0.113.1
        next
        edit 2
            set interface "port2"
            set zone "underlay"
            set gateway 198.51.100.1
        next
    end
end

A member is an interface plus the next-hop gateway reachable on it. Those two fields are what FortiOS needs in order to build that member’s static route for you. Everything else — zone, cost, priority, source IP — is optional tuning.

A zone is a named group of members, and it exists to do two jobs.

  • It is a selection target. An SD-WAN rule can name a zone instead of listing individual members, which keeps rules readable when a site has an underlay group and an overlay group.
  • It is the seam between the network team and the security team. Firewall policies reference the zone. You change what is inside it; Cipher’s policies never move.

The second job is the one that pays for itself. Without it, adding a WAN link means editing every policy that faces the WAN, at every site, forever — and the real cost is not the tedium, it is the policy somebody misses. That is a link carrying uninspected traffic, and nothing alerts on it because nothing is broken.

Señor: “Cipher does not want to know how many links Baz has sold us. Cipher wants to know that traffic leaving through any of them was inspected.”
🧠
MENTAL NOTE
Members are what SD-WAN chooses between. Zones are what everyone else refers to, so that your choices stay yours.

💭 THINK FIRST · DIA AND THE RETURN PATH

Brisbane and Perth are being built in six weeks and somebody has to decide whether their internet traffic breaks out locally or is hauled back to Sydney first. Penny will ask. The answer is not ‘DIA is modern’ — it is a trade, and you should be able to name what is on both sides of it.

  • Brisbane goes DIA on Monday. Name what LoFinance gains, and name what Cipher loses.
  • A partner allowlists a single LoFinance public IP for API access. Brisbane starts breaking out locally. What is the first phone call you get on Tuesday, and why?
Q1 — MODEL ANSWER
Gains: no backhaul latency to Sydney; no branch internet traffic on the WAN link; and — the strongest of the three — no branch internet traffic consuming the hub firewall’s inspection capacity. Loses: the single point of enforcement. One chokepoint, one log source, one place a regulator can be shown, becomes eight places that must agree forever. Drift at one of them is invisible.
Q2 — MODEL ANSWER
Brisbane’s API calls now source-NAT to Brisbane’s own public IP, not Sydney’s. The partner rejects them. Every allowlist, geo-restriction and ‘we only accept your Sydney range’ agreement is written against a NAT identity that DIA has just multiplied. This is the DIA consequence people forget, and it fails in production rather than in design review.
SECTION 05

Direct Internet Access, and What It Costs

DIA removes the detour and multiplies your identity on the internet.

Direct Internet Access means a branch reaches the internet from the branch. The alternative — backhaul — tunnels all branch internet traffic to a hub and breaks it out centrally.

DimensionDIABackhaul
LatencyDirect. No detour.Every packet visits the hub first.
WAN loadBranch internet never touches the WAN link.WAN link carries branch internet and corporate traffic.
Hub inspection loadDistributed across branches.Hub inspects the whole company’s internet traffic.
EnforcementOne chokepoint per site. Must be kept identical.Single chokepoint. Easy to prove, easy to audit.
NAT identityOne public IP per site. Allowlists break.One public IP for everyone. Partners stay happy.
Failure domainA site failure affects one site.Hub failure affects every site’s internet.
The consequence people forget: DIA changes who you appear to be. Every partner allowlist, geo-restriction, SaaS tenant restriction and “we only accept connections from your registered range” clause is written against a NAT identity. Break out locally at eight sites and you now have eight identities. Nothing in the design review catches this; a partner’s API does, in production.

The enforcement objection is real but solvable — and solving it is the entire business case for centralised management. Cipher’s condition on the record was: “I’ll accept DIA. I want the same profile set on every one of them, and I want to be told when one differs.” That sentence is a FortiManager requirement written in plain English, and it is why Session 65 exists.

Best practice summary: one SLA per thing you care about, not one per link — the comparison is only fair if every member is judged against the same target and thresholds. Probe past the carrier’s gateway. Probe with the protocol the traffic uses. Set recoverytime higher than failtime. Put members in a zone and let policies reference the zone. Take thresholds from the business, never from the defaults.

MENTAL NOTE
DIA trades a single point of enforcement and a single NAT identity for latency, capacity and blast radius. All four are worth naming out loud before anyone signs.

💭 THINK FIRST · VERIFICATION AND THE FOUR-LAYER LADDER

Riley’s ticket says ‘it’s slow.’ That sentence is consistent with four completely different faults, and the ticket will never tell you which one. A senior engineer’s advantage here is not knowing more commands — it is knowing which question comes first.

  • Trading traffic is going out the wrong link. List the four questions you would ask, in order, and the command that answers each.
  • diagnose sys sdwan health-check shows Baz’s member alive with latency 3ms and 0% loss, but the trading desk still reports slowness. What does that tell you to suspect?
Q1 — MODEL ANSWER
1. Is it up? get router info routing-table all. 2. Is it in-SLA? diagnose sys sdwan health-check. 3. Is it eligible? Inspect the matched rule’s member list. 4. Is it preferred? diagnose sys sdwan service. Four layers, four different faults, one identical user symptom.
Q2 — MODEL ANSWER
That the probe is not experiencing what the user experiences. Suspect the probe target — if it is the ISP gateway, 3ms is measuring the carrier’s front door and nothing beyond it. Or suspect the protocol: an ICMP probe sailing through a queue that is punishing TCP. A green health check is only as honest as the question you configured it to ask.
SECTION 06

Four Layers, One Symptom

The single most useful thing you can carry out of this session is an order. “It’s slow” is compatible with a dead route, a breached threshold, a member you never listed, and a rule preferring something you did not expect. These fail differently and are fixed differently, and the only way to keep them apart is to ask in sequence.

#QuestionCommandIf the answer is no
1Is the member up?get router info routing-table allRouting / interface / gateway problem. Not an SD-WAN problem yet.
2Is it in-SLA?diagnose sys sdwan health-checkEither the link is genuinely degraded, or your probe is asking the wrong question.
3Is it eligible?Inspect the matched rule’s member listConfiguration. You never offered this member for this traffic.
4Is it preferred?diagnose sys sdwan serviceStrategy, ordering or SLA state is selecting something else.
Supporting commands worth knowing: diagnose sys sdwan member for per-member state, diagnose sys sdwan health-check status for the SLA verdict per member, and diagnose sys virtual-wan-link as the pre-7.x alias you will still see in older documentation and in some exam material.
Dead end to avoid: a green health check is not proof of a healthy path. It is proof that the specific probe you configured, to the specific target you chose, using the specific protocol you picked, came back within the specific thresholds you set. Every one of those four choices can make the check agree with Baz instead of with the trading desk.

MENTAL NOTE
Symptom tells you nothing about layer. Ask up → in-SLA → eligible → preferred, in that order, every time.

💭 THINK FIRST · EXAM FOCUS AND SENIOR LESSONS

The blueprint tests this material through scenarios rather than definitions. Most wrong answers are not ignorance — they are the most recently-read term winning over the structurally correct one. Read each option against the question before eliminating any of them.

  • Why is ‘the route is withdrawn from the routing table’ such a durable wrong answer about out-of-SLA members?
  • What are the two required fields of an SD-WAN member, and which commonly-discussed field is optional?
Q1 — MODEL ANSWER
Because it is what a two-state mental model predicts. If a link is either good or gone, then a failed health check must remove it. SD-WAN introduces a third state, and the exam repeatedly checks whether you have it. The route stays; the candidacy changes.
Q2 — MODEL ANSWER
Required: interface and gateway. Optional: zone — along with cost, priority and source IP. Zone gets the most discussion because it matters operationally, which is exactly what makes it a tempting wrong answer on a ‘what is required’ question.
SECTION 07

Exam Focus and Senior Lessons

High-probability exam points

  • link-cost-factor is AND logic. One breached metric means out-of-SLA regardless of the others.
  • Out-of-SLA does not withdraw the route. It changes candidacy inside SD-WAN.
  • A member requires interface + gateway. Zone is optional.
  • failtime × interval = detection time. Expect to be asked in seconds. Do the multiplication.
  • Adding a member to a zone requires no firewall policy change.
  • SD-WAN rules can match source, destination, user/group, application, protocol and port. A static route matches destination only.
  • Probing the ISP next-hop gateway is a classic wrong answer — it proves only that the ISP router responds.
  • Default probe protocol is ping/ICMP; the better answer in a scenario is usually the protocol the application uses.

Senior engineering lessons

Up is a fact. Good is a measurement. Nothing measures quality unless you build the thing that measures it.

The threshold is where the business enters the network. 150ms is a LoFinance number, not a Fortinet number.

Zones are the seam between teams. Design them so the other team never has to hear about your changes.

Be quick to distrust, slow to forgive. Failing costs one bad afternoon; recovering too eagerly costs a flap, and every session moves twice.

“Green across my dashboard” answers the question the dashboard was built to ask — which is rarely the question you are asking. Baz was never lying.
Exam technique for this student: score every option against the question before eliminating any. Familiarity is not evidence. The term that was most recently and most vividly discussed is a distractor more often than it is an answer.
