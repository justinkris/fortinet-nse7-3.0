==================================================================
BUSINESS PROBLEM
==================================================================
SYD-EDGE has two internet links (port1 = Carrier-A, port2 = Baz/Carrier-B), both default routes,
ECMP splitting traffic. Baz's link BROWNS OUT ~40 min/day: interface up, route valid, but ~400ms
latency and ~5% loss. Because the routing table has no field for quality, the FortiGate keeps
using it. Trading traffic lands on it half the time and lags. By the time anyone looks, it has
recovered — so there is never evidence, and Riley's tickets can never be closed.
The problem is NOT a fault. Nothing is broken. The problem is that NOTHING WAS MEASURING.

==================================================================
TECHNICAL CONCEPTS COVERED
==================================================================
1. THE ROUTING TABLE KNOWS REACHABILITY, NOT QUALITY. Central mental model of the episode.
   A route entry = next-hop + interface + distance + priority. There is NO FIELD for latency.
   The routing table is not ignoring quality; it has no vocabulary for it. Student derived this
   unprompted and correctly, by reasoning about the CONTENTS OF THE DATA STRUCTURE.
2. PERFORMANCE SLA / HEALTH CHECK — the thing that goes and measures. `config system sdwan` →
   `config health-check`. Actively probes a target via each member.
3. THE THREE METRICS: LATENCY, JITTER, PACKET LOSS. Derived by the student from Riley's two
   tickets rather than delivered as a list (see Learning Notes — this worked extremely well).
   Also mentioned: mos (7.6, composite voice score), bandwidth / remote-bandwidth.
4. PROBE PROTOCOL IS CONFIGURABLE: ping/ICMP (default), tcp-echo, udp-echo, http, https, dns,
   twamp, ftp. Student volunteered "probe the protocol the traffic actually uses" UNPROMPTED,
   ahead of schedule. Reason it matters: carriers queue ICMP differently; ICMP can lie in
   either direction.
5. THRESHOLDS + link-cost-factor. `set link-cost-factor latency jitter packet-loss` with
   latency-threshold / jitterthreshold / packetloss-threshold. **AND LOGIC** — all listed
   factors must pass; one breach = out-of-SLA. (Exam Q1, student correct.)
6. failtime / recoverytime / interval. failtime × interval = time to declare out-of-SLA.
   RECOVERY SLOWER THAN FAILURE. "Quick to distrust, slow to forgive."
7. OUT-OF-SLA DOES NOT REMOVE THE ROUTE. The route stays. What changes is the member's
   ELIGIBILITY inside SD-WAN rule selection. Student held this line consistently all session
   and got it right on the exam.
8. THREE DISTINCT STATES: UP / ELIGIBLE / PREFERRED. A member can be up and in the routing
   table and still ineligible, simply because it isn't in this rule's member list.
9. SD-WAN RULE vs POLICY ROUTE. Student volunteered the PBR comparison (flagged as inherited:
   "I read somewhere"). Resolved as: PBR is right about the MATCHING half (src, dst, user,
   app, protocol/port, evaluated before the routing table) and WRONG about the output.
   KEEPER FORMULATION: "A policy route picks an interface. An SD-WAN rule picks from a GROUP
   of interfaces, and KEEPS PICKING." Decision made at config time; ANSWER made at packet time.
10.MEMBERS. `config members` — interface + gateway (+ optional zone, cost, priority). The
   gateway is what lets FortiGate build the member's static route automatically.
11.ZONES. Two purposes, both surfaced: (a) grouping for SD-WAN rule selection, (b) THE SEAM
   BETWEEN NETWORK AND SECURITY TEAMS — policies reference the zone, membership changes
   underneath without touching a single policy. Add port3 to the zone, every policy covers it.
   Also: SD-WAN rules can target a zone, not just members (matters for multi-hub, S70+).
12.DIA vs BACKHAUL. FOR DIA: no backhaul latency; no hub link load; **no hub INSPECTION load**
   (the stronger form of the student's point). AGAINST DIA: loses the SINGLE POINT OF
   ENFORCEMENT — one chokepoint becomes eight, eight must agree forever, drift is invisible.
   THE ONE THE STUDENT MISSED AND SHOULD BE RE-TESTED ON: **THE RETURN PATH / NAT IDENTITY** —
   each DIA site NATs to its own local public IP, breaking partner allowlists, geo-restrictions
   and "we only accept your Sydney range" agreements. (He got this on exam Q8 immediately once
   framed as a scenario — so it's a PROMPTING gap, not a knowledge gap. Retest cold in S63/64.)
13.BEST PRACTICES delivered as doctrine: one SLA per thing-you-care-about not per link; probe
   PAST the ISP gateway; probe with the real protocol; recovery slower than failure; members in
   a zone and policies reference the zone; thresholds come from the business.

KEY CLI DISCUSSED
==================================================================
config system sdwan
    set status enable
    config zone
        edit "underlay"
    end
    config members
        edit 1 / set interface "port1" / set zone "underlay" / set gateway 203.0.113.1
        edit 2 / set interface "port2" / set zone "underlay" / set gateway 198.51.100.1
    end
    config health-check
        edit "Trading-SLA"
            set server "10.99.1.10"
            set protocol http
            set interval 500
            set failtime 5
            set recoverytime 10
            config sla
                edit 1
                    set link-cost-factor latency jitter packet-loss
                    set latency-threshold 150
                    set jitterthreshold 30
                    set packetloss-threshold 2
            end
    end
end

Verification (taught as a FOUR-LAYER ORDER, not a command list):
  1. get router info routing-table all      → is it UP? (reachability)
  2. diagnose sys sdwan health-check        → is it IN-SLA? (quality)
  3. (inspect rule member list)             → is it ELIGIBLE?
  4. diagnose sys sdwan service             → is it PREFERRED?

(S59/S60) — the student responds well to a named, ordered set of boxes. It will be extended in
S63 (add: does the rule MATCH?) and S64 (add: is the SESSION pinned?).
