TECHNICAL CONCEPTS COVERED
---------------------------
1. Flat L2 domain risk: any device can reach any other with no inspection point;
   larger domains also mean more broadcast noise (ARP/DHCP flooding) — the student
   named this cost unprompted before being asked.
2. VLAN tag (802.1Q, EtherType 0x8100) as the mechanism that partitions one physical
   medium into separate broadcast domains, forcing inter-VLAN traffic through Layer 3.
3. FortiGate NEVER adds a VLAN tag on ingress — tagging on ingress is always the
   responsibility of the preceding device (switch access port assigning untagged
   traffic to a VLAN before it reaches the trunk). Student reasoned this from first
   principles ("something needs to already be tagging before it reaches the firewall").
4. VLAN tag is a property of the LINK/WIRE being crossed, not a permanent property of
   the packet — FortiGate strips the ingress tag conceptually, routes at L3, and
   re-tags on egress to match the destination VLAN interface. Student derived this
   directly ("needs to add another tag assigned to the interface").
5. Layer 2 devices (switches) can only add/remove tags unchanged; Layer 3 devices
   (FortiGate) can actually CHANGE the tag as part of a routing decision — a classic
   exam trap when reversed.
6. dot1q trunk: single physical link multiplexing multiple VLANs by tag; required
   whenever two VLANs must share one physical connection back to FortiGate.
7. VLAN interface object in FortiOS = logical sub-interface referencing BOTH a
   physical/aggregate interface (which wire) AND a vlanid (which tag on that wire) —
   student correctly explained why both fields are structurally necessary.
8. VLANs are not limited to physical ports — can be configured on aggregate/LACP,
   FortiLink, hardware switch, software switch, and wireless SSID interfaces.
9. Segmentation is only REAL (in Cipher's sense) when the VLAN gateway sits on
   FortiGate, not on a switch — otherwise inter-VLAN traffic never crosses a policy
   decision point at all. Direct callback to Session 1's "filing cabinet with
   labelled but unlocked drawers" analogy, correctly retrieved and reapplied by the
   student without prompting.
10. Troubleshooting split for "policy seems to not be working": `diagnose sys session
    list` first — if the session never appears at all, the fault is BELOW the firewall
    (VLAN tagging/trunk/ARP, Layer 2), not the policy; if it appears with a policy_id,
    the fault is in policy logic/order. Student correctly reasoned this split before
    being told, including naming ARP table inspection as the first Layer 2 check.
