SESSION 04 · TROUBLESHOOTING · VLAN SEGMENTATION

The Floor Wars — Think Like Cipher

Not a command list. A decision process for "why isn't this segmented the way I think it is?"

Symptom Patterns

  • A host on one VLAN unexpectedly appears in a capture from another VLAN.
  • A deny policy between two VLANs "isn't working" — traffic still gets through.
  • A new VLAN interface is configured but hosts on it can't reach anything, including their own gateway.
Common dead end: jumping straight to firewall policy review when the traffic never reached the firewall as a properly tagged frame at all. This wastes time debugging something that isn't broken.

Questions to Ask First

  1. Did this traffic actually cross a trunk link, or is it still flat Layer 2 (no VLAN at all)?
  2. Which device was supposed to apply the VLAN tag — has the switch's access port actually been assigned to the right VLAN?
  3. Is the VLAN interface's gateway hosted on FortiGate, or still on a switch?
  4. Does the traffic even reach FortiGate as a session at all?

Verification Commands & Expected Output

diagnose sys session list session info: proto=6 ... src: VLAN5-Exec 192.168.5.50 dst: VLAN20-Trading 192.168.20.77 ... policy_id=3

Session present, policy_id shown: traffic reached FortiGate and matched a policy. Now inspect policy order and srcintf/dstintf — first-match-wins, so an earlier broad allow rule can shadow your deny rule.

Session absent entirely: the frame never arrived as a routable, correctly-tagged packet. Stop looking at firewall policy. Check:

get system interface | grep -A5 VLAN20-Trading diagnose netlink interface list VLAN20-Trading

Confirm the VLAN interface exists with the right vlanid and parent interface, and that the switch side has the matching access-port VLAN assignment and trunk allowed-VLAN list.

Decision Tree

Is there a session in `diagnose sys session list` at all?
No → go to Layer 2 (tag/trunk/ARP). Yes → go to policy logic.
(No session) Is the switch access port assigned to the correct VLAN?
No → fix the switch config. Yes → check the trunk carries that VLAN ID.
(Session present) Does the policy_id match the rule you expected?
No → an earlier rule is matching first — check policy order. Yes → check action/logging settings on that exact rule.

Likely Root Causes

  • Switch access port never reassigned to the new VLAN (still default/native VLAN).
  • Trunk link not carrying the new VLAN ID (allowed-VLAN list on the switch trunk).
  • VLAN interface gateway accidentally left on a Layer 2 switch instead of FortiGate.
  • Broader allow policy sitting above the intended deny rule (first-match-wins).
Exam trap: "the deny policy is configured but traffic still passes" almost always means policy ORDER, not policy content, once you've confirmed the session is even reaching FortiGate.
