The Floor Wars — Segmenting the Enterprise

Cipher said one word. It turns out one word is enough, once you understand what a VLAN tag actually does.

A VLAN tag is not a label glued to a packet — it's a property of the wire it's currently crossing.

Get that one idea right and half of the FortiGate VLAN exam questions stop being tricks.

💭THINK FIRST · FLAT DOMAIN PROBLEM

Exec floor and trading floor share one switch stack with no VLANs. Before you learn the fix, work out exactly what's broken.

  • What can every device on a flat broadcast domain do to every other device, mechanically?
  • What specifically stops one host from directly framing traffic to another host with nothing checking it?
Q1
Any device can reach any other via direct Layer 2 delivery-by-MAC-address, with no routing and no policy check involved. Larger domains also generate more broadcast/ARP/DHCP noise.
Q2
Nothing. Layer 2 forwarding has no concept of "should this be allowed" — it's pure delivery by MAC address. There is no inspection point unless one is deliberately forced into the path.
SECTION 01

The Flat Domain Problem

A VLAN is a logical subdivision of a network that segregates devices into separate broadcast domains. Without one, exec floor and trading floor devices sit in a single broadcast domain: any device can reach any other directly, with no routing and — critically — no inspection point of any kind. This is exactly the scenario Riley's packet capture exposed: an exec-floor IP chatting normally inside a trading-floor capture.

The cost isn't only security. A larger flat domain means more broadcast traffic (ARP requests, DHCP discovers, unknown-MAC flooding) hitting every device on it — a real operational cost that compounds as the trading floor doubles from 200 to 400 desks.

Common misconception: "Same switch stack, probably fine" (Junior's line) treats physical proximity as if it implies security. It doesn't — Layer 2 adjacency is exactly the problem, not a mitigating factor.
🧠
MENTAL NOTE
If two zones share one broadcast domain, there is no possible inspection point between them — segmentation has to begin at Layer 2 before it can mean anything at the firewall.

💭THINK FIRST · VLAN TAGGING MECHANICS

A VLAN tag is the mechanism, but the exam cares about direction and responsibility just as much as existence.

  • Does FortiGate ever add a VLAN tag to a frame that arrives untagged on ingress?
  • When FortiGate routes a packet from one VLAN to another, does the tag stay the same, disappear, or change?
Q1
No. FortiGate does not add VLAN tags on ingress — that is always the responsibility of the preceding device (typically a switch access port).
Q2
It changes. FortiGate strips the ingress tag, makes an L3 routing decision, and adds the destination VLAN's tag on egress. The tag belongs to the wire being crossed, not to the packet permanently.
SECTION 02

VLAN Tagging Mechanics

802.1Q tagging inserts a 4-byte tag (EtherType 0x8100) into the Ethernet frame, supporting up to 4,094 VLANs. 802.1ad (QinQ/provider bridging, EtherType 0x88A8) stacks an outer provider tag on top, supporting up to 16 million combinations — used mainly by service providers separating customer traffic.

Layer 2 devices (switches) can only add or remove tags unchanged. They cannot alter a tag's value. Layer 3 devices (FortiGate) can change the tag as part of a routing decision — this distinction is a favourite exam trap when the two are swapped.

Ethernet TypeEtherTypeVLAN Capacity
Standard (no VLAN)0x0800 (IPv4)—
802.1Q (single tag)0x81004,094 VLANs
802.1ad (QinQ / double tag)0x88A8 (S-tag) + 0x8100 (C-tag)16 million combinations
Exam tip: "FortiGate removes the VLAN 5 tag on ingress, then adds the VLAN 15 tag after routing" is the exact wording pattern to expect — read it as ingress-strip, then egress-add-per-destination-interface, never as "modifies the existing tag in place."
🧠
MENTAL NOTE
A VLAN tag is a property of the link currently being crossed, not a permanent property of the packet. FortiGate only ever reads a tag that's already there — it never invents one on ingress.

SECTION 03

Configuring VLANs on FortiGate

A VLAN interface in FortiOS is a logical sub-interface: it references a physical (or aggregate) interface to say "which trunk to listen on," plus a VLAN ID to say "which tag on that trunk is mine." Neither field alone is meaningful — this is the design decision behind the Floor Wars fix: VLAN 5 for exec (existing), VLAN 20 for the expanded 400-desk trading floor, both riding one dot1q trunk into the edge FortiGate over interface internal1.

FortiOS CLI — create the trading floor VLAN interface
config system interface
    edit "VLAN20-Trading"
        set vdom "root"
        set interface "internal1"
        set vlanid 20
        set ip 192.168.20.1 255.255.255.0
        set allowaccess ping https
    next
end

VLANs are not limited to physical ports. FortiOS supports VLANs on: aggregate (LACP/802.3ad) interfaces, FortiLink interfaces to a managed FortiSwitch, hardware switches, software switches, and wireless SSID interfaces. You can even reuse the same VLAN ID across different interface names to create entirely separate broadcast domains.

Analogy

A VLAN interface is like a labelled mail slot on a shared mailroom wall.

  • The physical/aggregate interface is the wall the slots are mounted on.
  • The VLAN ID is the label on one specific slot.
  • Mail (frames) only goes into the slot matching its label — the wall alone tells you nothing about which mail belongs where.
🧠
MENTAL NOTE
Two fields, one purpose: `interface` picks the wire, `vlanid` picks the tag on that wire. A VLAN interface object without both is meaningless.

💭THINK FIRST · GATEWAY PLACEMENT

Two VLANs exist. Are exec and trading floor actually "segmented" in Cipher's sense yet?

  • What's still missing before inter-VLAN traffic is forced through a policy decision?
  • If a policy "seems to not be working" on Monday morning, what's the very first command to run, and what does an empty result versus a populated result each tell you?
Q1
The default gateway for each VLAN must sit on FortiGate itself, not on a switch — otherwise inter-VLAN traffic never crosses a policy decision point at all, VLANs or not.
Q2
Run `diagnose sys session list`. No session at all → the traffic never reached FortiGate as a properly tagged, routable frame — check Layer 2 (VLAN tag/trunk/ARP) first. Session present with a policy_id → the fault is in the policy itself (order, srcintf/dstintf, first-match-wins).
SECTION 04

Gateway Placement & Verification

VLANs alone stop Layer 2 flooding between zones — they do not, by themselves, create a security boundary. That only happens once the VLAN's default gateway is hosted on FortiGate, forcing every inter-VLAN packet through a Layer 3 routing decision that a firewall policy can act on.

FortiOS CLI — deny-by-default policy between the two VLANs
config firewall policy
    edit 0
        set name "Exec-to-Trading-Deny"
        set srcintf "VLAN5-Exec"
        set dstintf "VLAN20-Trading"
        set srcaddr "all"
        set dstaddr "all"
        set action deny
        set logtraffic all
    next
end
Verification
diagnose sys session list
session info: proto=1 proto_state=00 ...
  src: VLAN5-Exec  192.168.5.50
  dst: VLAN20-Trading  192.168.20.77
  ...
  policy_id=0
Remember: "Filing cabinet with labelled but unlocked drawers" (Session 1) — VLANs without a firewall-hosted gateway are exactly that: organised, but not locked.
🧠
MENTAL NOTE
No session in the list at all means the fault is below the firewall (Layer 2), not in the policy. Don't debug policy logic before confirming the traffic ever arrived as a routable frame.
