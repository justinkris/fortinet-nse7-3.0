Crying Wolf

The newly-armed IPS drops the market-data feed at market open. TAC wants packets before it says hello. Tuning an intrusion sensor means managing two opposite failures at once: crying wolf about traffic it can read, and going deaf to the traffic it can't.

06 Security Profiles · pp. 176–186 Use cases 2–5 · Lab 6 Blueprint 3.3 — IPS Exam: 10 / 10
The IPS failed in two opposite directions in one morning.
Too loud on what it could see. Deaf to what it couldn't. Both edges of the same blade.
Story recap

Sixty seconds before the bell

Over the weekend, Cipher's week of Monitor logs ended and the Session 24 sensor — protect-trading-api — was flipped from Monitor to Block. At 9:30:00 the market opened. At 9:30:11 the live market-data feed — a licensed, business-critical connection — went dark. Every dashboard was green: the feed server up, the network path clean, the vendor sending normally.

Junior protested that they'd watched it for a week and the logs were empty. Señor's answer became the spine of the session: "Monitor watches. Block acts. You tested a ghost of it." A clean Monitor week proves nothing about Block, because Monitor never drops anything — you never actually tested the thing you deployed.

TAC dialled in and, before greeting anyone, demanded a packet capture off the DCFW: "I don't want your theory. I want the packets." Señor endorsed it as the correct first sentence — evidence before narrative.

Learning objectives

What this session locks in

  • Recognise and diagnose an IPS false positive — the outage that passes every health check.
  • Fix it surgically with a Signature-type exception, Action Monitor, ranked above the Filter.
  • Understand why the IPS is blind inside encrypted traffic and what depth of SSL inspection restores its sight.
  • Separate the two independent axes: depth (certificate vs full) and mode (flow vs proxy).
  • Explain why flow-based full inspection gives the IPS payload visibility without buffering the whole file — and why AV is different.
Concept 1 — false positives

The outage that passes every health check

A false positive is legitimate traffic that trips an attack signature and gets blocked. It is the hardest outage to diagnose precisely because nothing in the path actually broke — the firewall is doing exactly what it was told to do. The server, network, vendor, and application all report healthy, because none of them is the cause.

Think first
If every normal check is green, where is the truth recorded?
Only in the IPS security event log — the single witness that names the signature that fired, the attack name, and the source/destination. This is why TAC wanted the packet capture and the log: every other angle was lying by showing health.
Diagnostic reflex Business-critical traffic dies but server / network / vendor / app all show green → suspect the security profile, not a fault. Go straight to the IPS security event log.
Concept 2 — the surgical fix

Spare the one flow without disarming the sensor

The wrong fix is Junior's instinct: delete the sensor or disable IPS on the policy — which reintroduces the Session 24 blindness that let the original attacker in. The right fix is a scalpel: stay inside the existing sensor and add one exception.

  • Type: Signature (a specific signature ID, read from the log).
  • Action: Monitor — pass the traffic, but keep logging it. You are not blinding yourself.
  • Rank: above the broad Filter. First match wins — rank it below and the Filter drops the packet before your exception is ever evaluated.
IPS false-positive exception — ranked above the Filter
config ips sensor
    edit "protect-trading-api"
        config entries
            edit 1                         # low number → evaluated first
                set rule <signature-id>       # from the IPS security event log
                set status enable
                set action monitor          # pass but log
                set log-packet enable      # FP handling only — not long-term prod
            next
        end
    next
end
Exam trap — packet logging set log-packet enable is for addressing false positives or building signatures. Avoid leaving it on in production unless you sized for it — it is expensive.
Concept 3 — the encrypted blind spot

The IPS reads the letter, not the envelope

Half the inbound trading sessions are now HTTPS, and the IPS log for those is "suspiciously clean" — not because nothing's hostile, but because the IPS cannot see inside them at all. An exploit sent over HTTPS sails through: the signature exists, the sensor is armed, the policy is applied — but the engine is pattern-matching against ciphertext. There is no exploit string to find, only noise.

The keeper analogy
Certificate inspection reads the envelope — the address, the sender, the postmark (SNI, cert CN/SAN, issuer). Full inspection opens the envelope and reads the letter (the decrypted payload). The IPS is a letter-reader. So to catch anything hidden inside encryption, the depth is always full inspection — never certificate.

Depth is decided by encryption

AttackTransportDepth requiredWhy
PHP injectionPlain HTTPCertificate / nonePayload is already cleartext — nothing to decrypt
PHP injectionHTTPSFull (deep)Payload is sealed — IPS needs it decrypted to match
DNS exfiltrationDNS-over-TLS / HTTPSFull (deep)Hostile DNS is hidden inside encryption to evade you
The common thread PHP-in-HTTPS and DNS-exfil-in-DoT are the same disease: a payload an attacker deliberately put inside encryption to blind the IPS. Same cure — full inspection, the only depth that opens the tunnel. The DNSExfiltrator.Data.Exfiltration signature requires it for exactly this reason.
The one line that turns the IPS from blind to seeing on HTTPS
config firewall policy
    edit <feed-to-trading-server>
        set ips-sensor "protect-trading-api"
        # swap certificate-inspection OUT for the deep profile:
        set ssl-ssh-profile "deep-inspect-trading"
    next
end
# deep profile uses "Protecting SSL Server" — FortiGate holds the server cert + key
Concept 4 — two independent axes

Depth and mode never collapse into each other

The confusion that catches almost everyone: assuming that because full inspection decrypts, it must run in proxy mode. It doesn't. Depth and mode are separate axes.

AxisQuestion it answersOptions
DepthCan the IPS read the payload?Certificate (envelope) · Full (letter)
ModeDoes the engine buffer the whole object?Flow (stream) · Proxy (reassemble)

Full inspection works in both flow and proxy mode. In flow mode FortiGate decrypts in-stream over one continuous connection; in proxy mode it terminates two separate TLS sessions and reassembles. Same payload visibility, different plumbing.

Why flow mode doesn't need the whole file

A signature is a pattern — a string, a byte sequence, a cadence. You can recognise a pattern the moment it streams past; you don't need the end of the letter to spot a threatening phrase in the middle. So flow mode keeps only a small sliding buffer, matches as bytes go by, and lets traffic keep moving. The complete object is never assembled — which is why it's fast enough to sit in front of a latency-sensitive market-data feed.

The contrast — AV Antivirus often needs the whole reassembled file to hash it or unpack an archive — you can't hash half a file streaming by. That forces proxy mode. Note the cause: it's reassembly, not encryption, that forces proxy. Encryption decides depth; reassembly decides mode.
The one thing to remember
Encryption decides depth (certificate vs full). Reassembly decides mode (flow vs proxy). Full inspection lives on both modes — flow for latency, proxy only when something (usually AV) must reassemble the whole file. And the IPS fails two ways: it cries wolf about traffic it can read, and goes deaf to traffic it can't.
Verification & troubleshooting

Confirming the fix

  • Read the IPS security event log — the only witness to a false positive. Note the attack name, signature ID, src/dst/service.
  • After adding the exception, confirm the feed flows and that the exception still logs (Monitor, not silent allow) — so a change in the feed's character would still show up.
  • For the encrypted fix, verify with the guide's PHP-injection test: under certificate inspection the attack passes undetected (normal forward-traffic log, no IPS log); after swapping to the deep profile, the same injection is detected and dropped.
  • Keep the exception scoped (specific signature, ideally specific source/destination) so the deliberate hole is exactly feed-shaped — then watch that nothing else starts matching it.
Common mistakes

Where engineers go wrong

  • Trusting a clean Monitor week. Monitor never blocks, so it can't reveal what Block will break. Test the enforced state before you rely on it.
  • Over-powered depth. Reaching for full inspection on plain HTTP wastes a MITM on an envelope that was never sealed. Match the depth to the threat surface.
  • Collapsing depth and mode. "Encrypted, therefore proxy" is wrong — flow-based full inspection decrypts happily. Proxy is forced by reassembly, not encryption.
  • Nuking the sensor to fix one alert. Disabling IPS to stop a false positive reintroduces the blindness that let the original attacker through. Use a scoped exception.
  • Leaving log-packet on in production without sizing for it.
Senior engineering insight

Evidence before narrative

TAC's cold open — "I want the packets" before hello — is the reflex that separates senior troubleshooting from junior troubleshooting. A false positive lies to every dashboard, so a theory built on those dashboards is built on lies. The senior move is to gather the one artifact that can't lie (the capture, the IPS log) before forming a hypothesis. Señor's endorsement of TAC is the lesson: don't reason forward from symptoms that are all green; reason from the one record that actually saw the drop.

The second senior lesson is scope discipline. Both the fix (a scoped exception) and the inspection choice (flow for the latency-sensitive feed, not proxy everywhere) come from the same instinct: apply exactly as much force as the problem requires, at exactly the point it's needed, and no more.

Exam tips

What the blueprint will test

  • False-positive fix = Signature + Monitor + above the Filter. All three parts, plus first-match-wins as the reason for the ranking.
  • Watch absolute qualifiers in true/false stems — "any", "always", "regardless", "only". "Full inspection catches attacks in any encrypted session" is false (exemptions, certificate pinning). "Full inspection runs only in proxy mode" is false (flow does it too).
  • Depth by transport: HTTP → certificate/none; HTTPS → full. Same exploit, same signature — depth decided by encryption.
  • DNS exfiltration needs full inspection because the hostile DNS hides in DoT/DoH. Signature: DNSExfiltrator.Data.Exfiltration.
  • Decryption ≠ detection. Full inspection only exposes the payload; the IPS sensor is what detects. No sensor = no catch, even with full inspection on.

Business outcome & topology

The tripwire, armed and tuned

The market-data feed is restored without disarming the sensor. The protect-trading-api sensor on the ISFW is now in Block with a scoped Signature/Monitor exception protecting the feed. The DCFW's inbound trading policy is upgraded from certificate-inspection to full (deep) inspection in flow mode — giving the IPS eyes inside HTTPS without adding latency to the feed.

Topology update The sensor no longer cries wolf — and it can finally see the wolves that hide. No new hardware. The network grew smarter, not bigger.
