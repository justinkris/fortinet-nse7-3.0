SOCRATIC ARC (student reasoning — strong throughout, two clean self-corrections)
--------------------------------------------------------------------------------
1. False positive named instantly; reasoned WHY it's the hardest outage: nothing in
   the path actually broke, so every normal health check (server/network/vendor/app)
   returns green. Only witness = the IPS/security event log. (Refinement given: it's
   the IPS security-event log specifically, not the traffic log, that names the
   signature.)
2. The fix: Signature-type exception + Monitor + ranked at TOP. Student's reason:
   "priority order matters, granular at the top" = first-match-wins. Refined the
   "new policy OR edit profile" instinct toward the SURGICAL move: stay in the
   existing sensor, add one line, keep the fix's blast radius as small as the
   problem's. Monitor = pass BUT keep logging (not blinding yourself).
3. Encrypted attack (PHP-injection over HTTPS): student answered NO, IPS won't catch
   it — "IPS reads the letter, encrypted = sealed envelope." Reached for the right
   remediation AND its precondition unprompted: SSL inspection with the KEYS on the
   FortiGate to decrypt/re-encrypt. THEN owned the deep-vs-certificate distinction
   (only SEEDED in S24): certificate inspection = envelope/SNI/cert-CN only; FULL
   (deep) inspection = MITM, decrypts payload. PHP-in-HTTPS REQUIRES full inspection.
4. DNS exfiltration: student's first instinct — "DNS isn't encrypted, no certs" — is
   GOOD reasoning that hits the guide's wall (the DNSExfiltrator.Data.Exfiltration
   sig REQUIRES full inspection). Claude handed the thread; student SELF-REPAIRED to
   "it can hide in encrypted traffic" (DoT/DoH). Landed the common thread himself:
   both PHP-in-HTTPS and DNS-exfil-in-DoT are payloads an attacker HID INSIDE
   ENCRYPTION to blind the IPS — same disease, same cure (full inspection).
5. STUDENT-INITIATED deep-dive (his own question): "How does flow-based IPS work — 
   wouldn't we need the full file buffered?" He answered it himself mid-question:
   "IPS is just matching a signature / seeing behaviour, doesn't need the whole file.
   AV would." Claude confirmed + drew a visual. Mechanism locked: flow = sliding
   buffer, match the pattern as bytes stream past, never assemble the whole object;
   SSL full inspection decrypts IN-STREAM in flow mode (one connection); proxy
   terminates two TLS sessions and reassembles. AV forces proxy because hashing/
   unpacking needs the WHOLE reassembled file.
6. DEPTH vs MODE are INDEPENDENT AXES — the session's spine, and the thing the
   student explicitly untangled and then TAUGHT BACK: "encryption decides DEPTH
   (cert vs full); reassembly decides MODE (flow vs proxy); they never collapse."
   Full inspection works in BOTH flow and proxy. Flow chosen for latency; proxy
   forced only when something (usually AV) must reassemble.

TROUBLESHOOTING APPROACH TAUGHT
-------------------------------
The false-positive is "the outage that passes every health check." Method:
(1) Recognise the signature: business-critical traffic dies, but server/network/
    vendor/app all show GREEN → suspect the security profile, not a fault.
(2) Go straight to the IPS SECURITY EVENT LOG — the only witness. Read the attack
    name + signature ID + src/dst/service.
(3) Confirm it's benign (legitimate feed, expected service) → false positive.
(4) SURGICAL fix: Signature exception, Monitor, packet-log, ranked above the Filter.
    Do NOT delete the sensor / disable IPS (that reintroduces the S24 blindness).
(5) Keep the exception SCOPED (specific sig, ideally specific src/dst) so the hole
    is exactly feed-shaped; keep watching the log for anything else matching it.
Evidence-before-narrative endorsed via TAC ("packet capture before hello").
