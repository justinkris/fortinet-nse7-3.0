Crying Wolf

IPS tuning — false positives and encrypted attacks, in five minutes.

The IPS failed in two opposite directions in one morning.
Too loud on what it could read. Deaf to what it couldn't.
Failure 1 · cry wolf
False positive

Legit market-data feed trips a signature and gets blocked. Every health check is green — the truth is only in the IPS security event log. Fix: Signature exception, Monitor, ranked above the Filter (first match wins).

Failure 2 · miss the wolf
False negative

Exploit hides inside HTTPS / encrypted DNS. Signature exists but the IPS reads only ciphertext. Fix: full (deep) SSL inspection so there's a decrypted payload to match.

Depth vs mode — never collapse them

Depth (certificate vs full) = can the IPS read the payload? Certificate reads the envelope (SNI, cert); full decrypts the letter.

Mode (flow vs proxy) = does the engine buffer the whole object? Flow streams; proxy reassembles.

Full inspection runs in both modes. Encryption decides depth; reassembly decides mode.

Why flow doesn't need the whole file

A signature is a pattern — visible mid-stream. Flow keeps a small sliding buffer, matches as bytes pass, never assembles the object. That's why it's fast enough for a latency-sensitive feed.

AV forces proxy — hashing or unpacking needs the whole reassembled file. Cause is reassembly, not encryption.

Depth by transport

PHP injection over HTTP → certificate / none (payload already cleartext).

PHP injection over HTTPS → full. DNS exfiltration in DoT/DoH → full (DNSExfiltrator.Data.Exfiltration). Same exploit, depth decided by encryption.

Bank this
Certificate reads the envelope; full reads the letter. To catch anything inside encryption, depth is always full. Decryption ≠ detection — no IPS sensor, no catch.
