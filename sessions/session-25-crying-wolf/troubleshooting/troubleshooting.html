Session 25 · Troubleshooting Guide

When the IPS cries wolf

How a senior engineer diagnoses a false positive — the outage that passes every health check — and the encrypted blind spot that lets the real wolf through. Emphasis on thinking, not memorising commands.

Symptoms

What it looks like on the bridge

The tell

Business-critical traffic dies suddenly — often right after an IPS change or a Monitor→Block flip.

The lie

Server up. Network path clean. Vendor sending. Application running. Every dashboard green.

The timing

Correlates with enforcement, not with a fault. Nothing "broke" — the firewall is doing exactly what it was told.

The encrypted variant

The opposite: IPS logs are suspiciously clean on HTTPS sessions — because it can't see inside them at all.

Questions to ask first

Before you touch anything

Frame the problem — evidence before narrative
  1. What changed, and when? A Monitor→Block flip is the prime suspect for a sudden false positive.
  2. Get the capture and the IPS security event log — the only witness that isn't lying to you. (TAC's rule: "I want the packets," before hello.)
  3. Is the blocked traffic legitimate and expected? If yes, you're looking at a false positive, not an attack.
  4. For the encrypted variant: is this session HTTPS? What SSL inspection depth is on the policy — certificate or full?
  5. Is an IPS sensor even attached? (Decryption without a sensor detects nothing.)
Why not just reason from the dashboards

A false positive makes every normal health check report health, so a hypothesis built on those checks is built on a lie. Reason from the one record that actually saw the drop — the IPS log — not from the four that didn't.

Verification commands

What to run, and what you expect to see

Confirm the false positive from the log
# GUI: Log & Report > Security Events > Intrusion Prevention
# Expect: action=dropped, a signature name, and src/dst = your legit feed
diagnose ips ... # inspect engine state if needed
# Prove the IPS DB is local & current (not a feed outage):
diagnose autoupdate versions   # IPS Attack Definitions listed = stored locally
Confirm the encrypted blind spot
# Under certificate inspection: attack over HTTPS shows in FORWARD TRAFFIC log
# but generates NO IPS log — FortiGate saw the access, not the attack.
# After swapping to the deep profile, the same injection now DROPS with an IPS log.
get firewall ssl-ssh-profile   # check depth: certificate-inspection vs deep
Expected output = your diagnosis

IPS log shows a drop on legit traffic → false positive. Forward-traffic log but no IPS log on an HTTPS attack → the payload was never decrypted; you're on certificate inspection when you need full.

Decision tree

From symptom to root cause

1
Did traffic die right after an IPS/enforcement change?
Yes → suspect false positive, go to 2.   No → don't assume IPS; check routing/policy/link first.
2
Does the IPS security event log show a drop on this traffic?
Yes, and traffic is legit → false positive, go to 3.   No drop logged → not the IPS; look elsewhere.
3
Fix: scoped Signature exception, Monitor, above the Filter.
Read the signature ID from the log → add exception → verify traffic flows and still logs. Do not disable the sensor.
4
Separate problem: is a known attack getting through on HTTPS?
IPS log clean but attack suspected → check SSL depth. Certificate inspection can't see the payload → switch to full (deep) inspection.
5
Latency-sensitive path (e.g. market feed)?
Use flow mode full inspection — payload visibility without buffering the whole object. Reserve proxy for tasks that need the reassembled file (AV).
Common dead ends

Where time gets wasted

Dead end · chasing the network

Pinging the server, checking interface counters, calling the vendor — all green, all wasted, because nothing in the path broke. The drop is in the security profile.

Dead end · trusting the clean Monitor week

Monitor never blocks, so an empty Monitor log proves nothing about what Block will break. You never tested the enforced state.

Dead end · disabling IPS to stop the alert

Kills the false positive and reintroduces the blindness that let the original attacker in. Fix with a scoped exception instead.

Dead end · adding full inspection on plain HTTP

A MITM to open an envelope that was never sealed — cost and latency for zero benefit. Match depth to the threat surface.

Likely root causes

What it usually is

False positive: a broad Filter (severity/protocol) is catching legitimate traffic whose content resembles an attack — an oversized packet, an unusual but benign application behaviour, a vendor feed that trips a high-severity signature.

False negative on HTTPS: the policy is on certificate inspection (or none), so the IPS never sees the decrypted payload. The exploit — PHP injection, DNS exfiltration over DoT/DoH — rides inside encryption undetected.

Exam traps

What the questions are built to catch

Trap · "any" / "only" / "regardless"

"Full inspection catches attacks in any encrypted session" — false (exemptions, certificate pinning). "Full inspection runs only in proxy mode" — false (flow does it too). Hunt the exception behind the absolute.

Trap · decryption = detection

Full inspection only exposes the payload. Without an IPS sensor attached, nothing is caught. Depth enables; the sensor detects.

Trap · encrypted therefore proxy

Encryption forces depth (full), not mode. Flow-based full inspection decrypts fine. Only reassembly (AV) forces proxy.

Trap · fix the FP by widening/disabling

The correct fix is a narrow Signature/Monitor exception above the Filter — not disabling IPS or loosening the whole Filter.
