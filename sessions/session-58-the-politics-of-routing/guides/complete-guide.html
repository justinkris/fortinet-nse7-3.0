SECTION 01 · THE QUESTION JUNIOR ASKED

Why does BGP exist at all?

Junior arrived late, holding a muffin, and asked the best question of the season: "Why don't we just run OSPF everywhere?" It deserves a real answer, because everything else today falls out of it.

OSPF works by having every router in an area hold an identical copy of the same link-state database, then independently run SPF over it. That shared map is what makes OSPF fast and loop-free — everyone does the same maths on the same data, so everyone agrees.

Now scale that to the internet. Two costs appear. The first is memory: every router holds every link on earth. That is bad, but it is a static problem — you can buy RAM.

The second cost is the one that kills it. In a link-state area, every change must reach every router, and every router must recompute. There is no such thing as a change that concerns only part of the area — that is the design, and it is what buys the guaranteed loop-free result. On a global network, some link is always flapping. A cut in Brazil. A reboot in Frankfurt. Hundreds of events per second, permanently.

The insight that matters: almost all of that computation produces no change in behaviour. A Sydney router burning CPU because a link flapped in Frankfurt will arrive at exactly the forwarding decision it already had. The work isn't just expensive — it is wasted. The architecture forces every router to participate in the details of a problem that isn't theirs.

And that observation defines what a global router actually needs. If Sydney's answer is "the same way out" regardless of Frankfurt's internals, then Sydney does not need Frankfurt's map. It needs to know Frankfurt is reachable, and roughly which direction and through whom.

Three families, three payloads

Protocol familyWhat it carriesLoop prevention
Distance vector (RIP)How far — a metricHop count, split horizon, hold-downs, poison reverse
Link state (OSPF)The whole mapShared database + identical SPF result
Path vector (BGP)Which way it came — AS_PATHAS_PATH self-check

BGP throws the map away. It does not use hop count. It has no hold-down timers. Instead every route carries the ordered list of autonomous systems it has already crossed, and a router rejects any route whose AS_PATH already contains its own AS number.

Why that mechanism is beautiful: the check is local and instant. No timers. No waiting for the network to agree. A router looks at one attribute on one route and knows, by itself, in isolation, whether accepting it would create a loop. No shared state required — which is exactly what you need when you have deliberately thrown away the shared map.
SEÑOR’S FRAMING
"OSPF is a family sharing one map. Everyone sees the same house. If you move the kitchen, everyone's map updates and everyone believes you — because you are family. BGP is nations negotiating treaties. Everyone has a map. Everyone is lying about theirs. Politely."
🧠
MENTAL NOTE
OSPF gets correctness from agreement — everyone holds the same map. BGP gets correctness from provenance — every route carries a record of where it has been, and you refuse anything that has already been through you.

💭THINK FIRST · IBGP VS EBGP

Classifying a peer as internal or external is not a label — it changes real protocol behaviour. Before you memorise the rules, work out what is genuinely different about the two relationships. The answer is hiding in one attribute.

  • A route crosses from AS 65300 into AS 65001 over EBGP. What happens to the AS_PATH? Now it moves from HQ to Melbourne, both in AS 65001. What happens to the AS_PATH this time?
  • Given your second answer — what has happened to BGP’s loop-prevention mechanism inside an AS?
Q1 — MODEL ANSWER
On the EBGP crossing, the advertising router prepends its own AS number. The route now carries a record: it came from 65300. Moving within an AS is not a treaty crossing — nothing is prepended. Melbourne receives the route with the AS_PATH unchanged, still reading 65300, exactly as HQ saw it.
Q2 — MODEL ANSWER
It is completely blind. If nothing writes into the AS_PATH internally, then a route that circles HQ → Melbourne → Brisbane → HQ arrives back looking entirely innocent. HQ has no way to recognise it. Inside an AS, BGP has no map, no hop count, no hold-down timers, and no working AS_PATH check — every loop-prevention tool it owns is inoperative.
SECTION 02 · THE INTERNAL PROBLEM

IBGP split horizon

The AS_PATH check cannot fire inside an AS, because nothing is prepended internally. BGP cannot invent a new attribute to fix this — carrying internal state is exactly the shared-map problem it was built to escape. So it takes the only remaining option, and it is brutally simple.

A prefix learned from an IBGP peer is never advertised to another IBGP peer.

Note carefully how this differs from the version you know from RIP:

RIP split horizonIBGP split horizon
ScopeInterfacePeer type
RuleDon’t advertise back out the interface you learned it onDon’t advertise to any other IBGP peer
Directional?Yes — only back toward the senderNo — it goes nowhere internally
Exam trap. The plausible wrong answer is always the RIP-flavoured one: "not advertised back to the peer it was learned from." That is not the BGP rule. It doesn’t matter that Melbourne and Brisbane are different routers on different links — if the route came in via IBGP, it dies there.

The consequence is worth stating precisely, because it explains why BGP needs no detection machinery internally: a route can only ever take one IBGP hop. You cannot build a circle out of segments of length one. The loop is not detected — it is made structurally impossible.

SEÑOR, TAPPING THE WHITEBOARD TWICE
"EBGP says: I will tell you where it has been. IBGP says: I will not tell anyone at all."

The exception that saves the protocol

If a route can only take one IBGP hop, how does anything ever propagate? Because the rule says nothing about where the route entered the AS. Routes learned via EBGP are freely advertised to IBGP peers — that is the entire job of a border router. The rule only bites on the second internal hop.

WHAT IS AND ISN’T ALLOWED — AS 65001
Javier (AS 65300) --EBGP--> HQ          legal, AS_PATH gets 65300 prepended
HQ --IBGP--> MEL                        LEGAL   (HQ learned it via EBGP)
HQ --IBGP--> BNE                        LEGAL   (same reason)
MEL --IBGP--> BNE                       BLOCKED (MEL learned it via IBGP)
BNE --IBGP--> MEL                       BLOCKED (same)

So every IBGP speaker must hear every prefix first-hand, from the router that learned it externally. Nobody internal may relay. Which forces a shape.

MENTAL NOTE
Loops detected is a policing problem — it needs timers and machinery. Loops made impossible needs nothing. IBGP chose impossibility, and paid for it with the full-mesh requirement.

SECTION 03 · THE PRICE OF THE RULE

Full mesh, and why 28 matters

If nobody may relay, then every IBGP speaker must peer directly with every other IBGP speaker. That shape is called a full mesh. It is not a design choice — it is forced on you by split horizon.

But the number is the least interesting part. Feel the operational shape instead: add one site to your eight and you do not add one session — you add eight, and you must touch every existing router to do it.

This is the real cost. Turning up a branch office stops being a branch change and becomes a fleet-wide change window. That is an operational failure mode, not a bandwidth one — and it is why the study guide states flatly that full-mesh peering in large networks is difficult to administer and not scalable.

💭THINK FIRST · NEXT-HOP RESOLUTION

Melbourne shows Singapore’s prefixes in the BGP database, marked valid. The routing table has nothing. Both sessions are Established. Nothing is logged. Riley’s phrasing is the whole diagnosis if you listen to it: "I can SEE the route. Why isn’t it a route I can USE?"

  • Which two commands separate "arrived and was rejected" from "never arrived"? What difference in output distinguishes them?
  • The prefix shows *i but not >, and there is only one path to it. What does that tell you — and what should you read next?
Q1 — MODEL ANSWER
get router info bgp network is the database — everything BGP has learned and considers valid, plural paths allowed. get router info routing-table bgp is what BGP has installed — singular per prefix, and the only thing the box can actually use. Present in the first but not the second means it arrived fine and was rejected during selection. Present in neither means it never arrived.
Q2 — MODEL ANSWER
* is valid, > is best, and only the best path is offered to the routing table. With a single candidate, "not best" cannot mean "lost a competition" — it means ineligible. Read the NEXT_HOP column next: a BGP path cannot be installed unless the router has a route to its next-hop address.
SECTION 04 · PENNY’S ACTUAL FAULT

Valid, but not best

Melbourne’s output, in full. Everything you need to solve it is on screen.

MEL-FGT — THE DATABASE
MEL-FGT # get router info bgp network
BGP table version is 12, local router ID is 10.99.60.1
Status codes: s suppressed, d damped, h history, * valid, > best, i - internal
Origin codes: i - IGP, e - EGP, ? - incomplete

   Network          Next Hop         Metric  LocPrf  Weight  Path
*i 172.28.10.0/24   172.20.0.1            0     100       0  65300 i
*i 172.28.20.0/24   172.20.0.1            0     100       0  65300 i

Total number of prefixes 2
MEL-FGT — WHAT IS ACTUALLY INSTALLED
MEL-FGT # get router info routing-table bgp

MEL-FGT #

The prefixes arrived. They are marked * — valid. The AS_PATH reads 65300, so provenance is intact and split horizon is not involved. The i confirms IBGP. But the > is missing, and only the best path is offered to the routing table.

Read the legend, not your memory. That status-codes line is printed on every one of these outputs — including the screenshot in an exam question. Every code in it is there for a reason. The entire diagnosis this session lived in one missing character.

One candidate that isn’t chosen isn’t losing a competition — it is ineligible. So read the Next Hop column: 172.20.0.1. That is Javier’s loopback, in Singapore, in AS 65300.

IBGP preserves the NEXT_HOP unchanged. HQ did not insert itself. Melbourne is being told: “to reach 172.28.10.0/24, send to a loopback in another country.”

Now strip the protocol away. Forget that it is BGP. A router has been handed an instruction of the form "send it to address X." What must be true before it can act? It must be able to reach X.

CAN MELBOURNE REACH THE NEXT HOP?
MEL-FGT # get router info routing-table details 172.20.0.1
Routing table for VRF=0
Routing entry for 0.0.0.0/0
  Known via "static", distance 10, metric 0, best
  * 10.99.60.254, via port1

Only the default route — and a default route does not satisfy next-hop resolution by default on FortiGate. Melbourne genuinely has no path to Singapore’s loopback.

BGP next-hop resolution. Before a path can be marked best and installed, the router must have a route to that path’s NEXT_HOP. If the next hop is unresolvable, the path stays valid but not best — sitting in the database forever, never installed, never withdrawn, never logged as an error.
Why this fault is nasty in production: nothing fails. The session is Established. The prefix is present. get router info bgp summary looks perfect. There is no alarm, no log, no down state — and the service is broken.
🧠
MENTAL NOTE
Seeing is the database. Using is resolution. Riley said it before anyone diagnosed it. When a BGP route is present but unusable, the next-hop column is almost always the answer.

SECTION 05 · TWO FIXES, NOT EQUIVALENT

next-hop-self is a promise

Option A — make the next hop resolvable. Give Melbourne an IGP or static route to 172.20.0.1. Correct when you genuinely want Melbourne routing directly toward that next hop.

Option B — rewrite the next hop at the border. HQ tells its internal peers “send it to me” instead of “send it to Singapore.”

HQ-FGT — APPLIED TOWARD THE IBGP PEER
config router bgp
    config neighbor
        edit "10.99.60.1"
            set remote-as 65001
            set next-hop-self enable
            set description "MEL-DR (IBGP)"
        next
    end
end
Direction matters and is commonly inverted on the exam. next-hop-self is applied on the border router, toward its IBGP peers, for routes learned via EBGP. It says: I am the door to the outside world — aim at me and I’ll take it from there.

The trade-off you are accepting

OptionCost
A — static/IGP routesForeign addressing leaks into your internal routing. Every time Javier renumbers or adds a peering address, you edit statics at every site. Your IGP now depends on a stranger’s addressing decisions.
B — next-hop-selfTraffic engineering is gone — all Singapore-bound traffic must physically transit HQ, even if a better path existed. HQ becomes a failure domain — if HQ’s forwarding breaks, Melbourne loses Singapore even with healthy circuits.

For LoFinance, B is right — but only because there is no direct Melbourne–Singapore circuit. Traffic was transiting HQ regardless. You are not losing a path you had. In a real multi-exit topology the same command would be a mistake.

SEÑOR
"Next-hop-self is a promise. Do not make it unless you can keep it."
🧠
MENTAL NOTE
The next hop you advertise is a claim about the shape of your network. next-hop-self makes that claim deliberately false in order to make it simple — which is fine right up until the moment it isn’t.

SECTION 06 · THE EXPOSURE NOBODY CONFIGURED

Becoming a transit AS by accident

Two rules you now hold interact in a way people miss. Rule 1: a route from an IBGP peer goes no further internally. Rule 2: EBGP-learned routes are freely advertised — to IBGP peers, and to other EBGP peers.

LoFinance will soon peer with Singapore, London, and an ISP. All EBGP. The moment London’s session comes up, with a perfectly correct configuration and no mistake anywhere: HQ tells London how to reach Singapore, and tells Singapore how to reach London.

LoFinance is now carrying traffic between two parties on behalf of neither. Sydney’s circuits have become the backhaul for a conversation that has nothing to do with Sydney.
Do not confuse the two mechanisms — this is a high-value exam trap.
AS_PATH loop prevention is automatic, per-route, needs no configuration. It stops routes circling. It does nothing about transit.
Transit prevention is deliberate, configured, and outbound. Nothing automatic will save you.
THE TREATY, WRITTEN DOWN — ADVERTISE ONLY WHAT WE ORIGINATE
config router prefix-list
    edit "LOFI-OWN-PREFIXES"
        config rule
            edit 1
                set prefix 10.99.0.0 255.255.0.0
                set action permit
                set ge 16
                set le 24
            next
            edit 2
                set prefix 0.0.0.0 0.0.0.0
                set action deny
                set le 32
            next
        end
    next
end

config router bgp
    config neighbor
        edit "172.20.0.1"
            set remote-as 65300
            set prefix-list-out "LOFI-OWN-PREFIXES"
            set description "Javier - SG DC (EBGP)"
        next
    end
end
SEÑOR, TO J.R., WHO WAS WRITING FAST
"Nobody attacked you. Nobody misconfigured anything. You simply did not say no, and BGP took that as yes."

MENTAL NOTE
Defaults have a direction. BGP’s default is to tell. Not telling is the thing you configure. Every outbound filter you don’t write is a treaty you signed by silence.

Senior engineering lessons

  • A route in the database is not a route in the routing table. Second time you have met this. Third time it should be reflex.
  • Silence is not health. No log, no alarm, no down state — and the service was broken. Absence of errors is not evidence of correctness.
  • Strip the protocol when the protocol stops making sense. The next-hop fault dissolved the instant we stopped thinking “BGP” and thought “a router being told to send something to an address.” That reframe transfers to every protocol you will ever debug.
  • Defaults have a direction. Ask of every protocol: what does it do if I say nothing?
🧠
MENTAL NOTE
Structural prevention beats policing. AS_PATH and split horizon both make bad states impossible rather than detectable — which is why BGP needs none of RIP’s timers. Look for that pattern in every protocol you meet.

SECTION 08 · EXAM PRACTICE

Ten questions — NSE7 EF 7.6

Answer before revealing. On select-all questions, evaluate every option independently.

Question 1
Which statement about IBGP split horizon is correct?
  • A. A route learned from an IBGP peer is not advertised back to the peer it was learned from
  • B. A route learned from an IBGP peer is not advertised to any other IBGP peer
  • C. A route learned from an EBGP peer is not advertised to IBGP peers
  • D. Split horizon applies only when route reflectors are configured
▾ Reveal answer

Answer: B

Option A is the RIP-flavoured distractor — interface-scoped and directional. The BGP rule is peer-type-scoped: it goes nowhere internally, full stop.

Question 2
An AS contains 6 IBGP speakers with no route reflectors. How many IBGP sessions are required?
  • A. 6
  • B. 12
  • C. 15
  • D. 30
▾ Reveal answer

Answer: C

n(n−1)/2 = 6×5/2 = 15. Deriving it as 5+4+3+2+1 is more reliable under pressure than recalling the formula.

Question 3
A prefix appears in get router info bgp network marked *i but not in get router info routing-table bgp. There is only one path. Most likely cause?
  • A. The BGP session is in Active state
  • B. The AS_PATH contains the local AS number
  • C. The NEXT_HOP is not resolvable in the routing table
  • D. ebgp-multipath is not enabled
▾ Reveal answer

Answer: C

If the session were Active the prefix wouldn’t be there at all. If AS_PATH contained the local AS it would have been rejected, not stored as valid. Multipath is irrelevant with one path.

Question 4
BGP is classified as which type of routing protocol?
  • A. Link-state
  • B. Distance-vector
  • C. Path-vector
  • D. Hybrid
▾ Reveal answer

Answer: C

It carries the AS_PATH — the sequence of autonomous systems traversed. Distance-vector is the most tempting distractor and some documentation loosely calls it that, but path-vector is the precise classification.

Question 5
When a FortiGate advertises a route to an EBGP peer, what happens to AS_PATH?
  • A. It is cleared
  • B. The local AS number is prepended
  • C. The local AS number is appended to the end
  • D. It is left unchanged
▾ Reveal answer

Answer: B

Prepended — added to the front. Order encodes recency, which is what makes the path readable. Option D describes IBGP behaviour.

Question 6
set next-hop-self enable is configured on HQ. To which neighbours and for which routes does it take effect?
  • A. EBGP neighbours, for routes learned via IBGP
  • B. IBGP neighbours, for routes learned via EBGP
  • C. All neighbours, for all routes
  • D. IBGP neighbours, for locally originated routes only
▾ Reveal answer

Answer: B

Option A is the inversion, and it is the single most common wrong answer on this topic. The border router tells its internal peers to aim at it for external routes.

Question 7
Which mechanism prevents an AS from unintentionally becoming a transit AS?
  • A. AS_PATH loop prevention
  • B. IBGP split horizon
  • C. Outbound route filtering (prefix lists / route maps)
  • D. ebgp-enforce-multihop
▾ Reveal answer

Answer: C

AS_PATH is the seductive wrong answer: it is automatic and it does concern AS relationships — but it prevents loops, not transit. Transit prevention is always something you configure.

Question 8
An organisation runs BGP, has its own AS number, connects to three different ISPs, and carries only its own traffic. What type of AS is it?
  • A. Stub
  • B. Multihomed
  • C. Transit
  • D. Confederated
▾ Reveal answer

Answer: B

Multihoming is about the number of external connections. Transit is about whether traffic that isn’t yours flows through you. Three ISPs alone never makes you transit.

Question 9
Which BGP attributes are classified as well-known mandatory? (Select all that apply.)
  • A. ORIGIN
  • B. AS_PATH
  • C. LOCAL_PREF
  • D. NEXT_HOP
  • E. MULTI_EXIT_DISC
▾ Reveal answer

Answer: A, B, D

LOCAL_PREF is well-known discretionary; MED is optional non-transitive. Technique: run each option as an independent true/false rather than assessing the set as a whole — that is where select-all questions are won and lost.

Question 10
Why can AS_PATH not be used for loop prevention between IBGP peers?
  • A. IBGP peers strip the AS_PATH attribute
  • B. The AS number is not prepended when a route is passed within the same AS
  • C. AS_PATH is an optional non-transitive attribute
  • D. IBGP peers use LOCAL_PREF instead
▾ Reveal answer

Answer: B

Nothing is stripped and nothing is added — the attribute simply never changes internally, so the self-check has nothing to detect. This is precisely why split horizon must exist.

🧠
MENTAL NOTE
Notice that in almost every question above, the wrong answer is a real fact placed in the wrong slot — RIP’s split horizon, IBGP’s next-hop behaviour, AS_PATH’s genuine function. The exam rarely invents nonsense. It relocates truth.
