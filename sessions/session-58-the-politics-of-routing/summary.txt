TECHNICAL CONCEPTS COVERED
==================================================================
1. WHY BGP EXISTS (answering Junior). Link-state does not scale globally because every change
   must reach every router and every router must run SPF — and on a global network some link
   is always flapping. STUDENT'S OWN INSIGHT: most of that computation produces NO CHANGE in
   the local forwarding decision. The work isn't merely expensive, it's wasted. From there:
   if the answer is the same regardless of Frankfurt's internals, you don't need Frankfurt's
   map — only reachability and direction.
2. PATH VECTOR, formally positioned against the other two families:
   RIP = how far (distance). OSPF = the whole map (link state). BGP = which way it came (path).
3. AS_PATH LOOP PREVENTION — retrieved COLD and COMPLETE with the check condition attached
   ("if an AS sees its own AS in the AS_PATH it won't accept it"). This had been deferred four
   sessions; the S56/S57 registers recommended formal retirement as a retrieval target and
   fresh reteaching. It did NOT need reteaching. RETIRE THIS AS A CONCERN — it is owned.
   Emphasised the property that matters: the check is LOCAL and INSTANT, needs no shared
   state, no timers, no hold-downs. That's precisely what you need when you have no map.
4. THE IBGP/EBGP BEHAVIOURAL SPLIT — not a label, a behaviour change.
5. AS_PATH IS NOT PREPENDED WITHIN AN AS. Therefore the loop check is structurally BLIND
   inside an AS. Inside AS 65001 BGP has no map, no hop count, no hold-down, and no working
   AS_PATH check — every loop-prevention tool is inoperative.
6. IBGP SPLIT HORIZON (p.337) — the answer to (5). Formulated precisely and contrasted with
   the RIP version: RIP = don't advertise back out the interface you learned it on
   (interface-scoped, directional). IBGP = a prefix learned from an IBGP peer is NEVER
   advertised to ANY other IBGP peer (peer-type-scoped). Key framing: this makes a loop
   STRUCTURALLY IMPOSSIBLE, not detected — a route can only take ONE IBGP hop ever, and you
   cannot build a circle from segments of length one.
7. THE CRITICAL EXCEPTION the student needed handed to him: split horizon says nothing about
   where the route ENTERED the AS. EBGP-learned routes are freely advertised to IBGP peers.
   The rule only bites on the SECOND internal hop. HQ→MEL is legal; MEL→BNE is not.
8. FULL MESH REQUIREMENT as the forced consequence. n(n-1)/2. Student computed 28 for eight
   sites by deriving the summation 7+6+5+4+3+2+1 rather than reciting the formula.
   Operational framing (matters more than the number): adding the 9th site adds EIGHT sessions
   and requires touching EVERY existing router — a branch turnup becomes a fleet-wide change
   window. Study guide wording: "difficult to administer and not scalable."
9. BGP NEXT-HOP RESOLUTION — the actual fault. IBGP PRESERVES the NEXT_HOP unchanged; HQ does
   not insert itself. Melbourne was told to send to 172.20.0.1 (Javier's loopback, AS 65300)
   and had no route to it. A path whose next hop is unresolvable is VALID BUT NOT BEST —
   sits in the database forever, never installed, never withdrawn, NEVER LOGGED.
10. STATUS CODE FORENSICS: `*` = valid, `>` = best, `i` = internal. `*i` without `>` on a
    SINGLE-path prefix means it is not losing a competition — it is INELIGIBLE. The whole
    diagnosis lived in one missing character.
11. next-hop-self: applied on the BORDER router, toward IBGP peers, for EBGP-learned routes.
    Trade-off taught explicitly: kills traffic engineering and concentrates the failure domain
    on HQ. Correct for LoFinance ONLY because there is no direct MEL–SG circuit — "you are not
    losing a path you had." Would be wrong in a real multi-exit topology.
12. ACCIDENTAL TRANSIT AS, now with live topology. EBGP-learned routes are advertised to other
    EBGP peers by default. The moment London comes up, with zero misconfiguration, LoFinance
    backhauls SG↔LON. Defence is OUTBOUND route filtering (prefix-list/route-map), NOT
    AS_PATH. Student retrieved this correctly and cold from S54.
13. Attribute categories touched lightly for Q9 only (well-known mandatory: ORIGIN, AS_PATH,
    NEXT_HOP; LOCAL_PREF = well-known discretionary; MED = optional non-transitive).
    NOT taught in depth — S59/S60 material.

==================================================================
TROUBLESHOOTING APPROACH (as taught this session)
==================================================================
1. Prefix present or absent?           → get router info bgp network
2. Present but unusable?               → read the STATUS CODES. `*` without `>`.
3. Single path, not best?              → not losing a competition; INELIGIBLE.
4. Ineligible how?                     → read the NEXT_HOP column.
5. Can I reach that next hop?          → get router info routing-table details <nh>
6. No?                                 → resolve it (IGP/static) or rewrite it (next-hop-self)

Counter-lesson explicitly taught: session Established + prefix present + no logs + no alarms
+ service broken. ABSENCE OF ERRORS IS NOT EVIDENCE OF CORRECTNESS.
