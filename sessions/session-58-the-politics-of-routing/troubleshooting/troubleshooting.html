SESSION 58 · TROUBLESHOOTING GUIDE

How a senior engineer isolates a route that arrived but will not install

The hardest faults are the ones that never raise an alarm.

STEP 01 · RECOGNITION

Symptoms that mean this fault

The defining characteristic of the Session 58 fault class is that nothing appears wrong. There is no down state, no log entry, no alarm, no counter incrementing. That absence is itself the fingerprint.

Symptom as reportedWhat it usually means
“I can see the route but can’t use it”Database vs routing table. Next-hop resolution, or a filter.
One site reaches the remote AS, another doesn’tNext-hop resolution at the site that fails, or IBGP split horizon (route needed two internal hops).
Everything Established, nothing logged, service brokenAlmost always valid-but-not-best. Sessions are healthy; selection is failing.
A third site went live and traffic appeared on links that shouldn’t carry itAccidental transit. Missing outbound filter.
Prefix absent entirely on an internal routerSplit horizon (two IBGP hops), or an inbound filter, or never advertised.
The trap that catches everyone: get router info bgp summary looks perfect during this fault. Both peers Established, prefixes received, uptime healthy. If you use summary as your health check, you will conclude BGP is fine and go looking at firewall policies for an hour.
🧠
MENTAL NOTE
Absence of errors is not evidence of correctness. A protocol can be working exactly as designed and still not be delivering the service you wanted.
STEP 02 · INTERROGATION

Questions to ask before touching anything

  1. What is the business symptom, restated as a packet? “Melbourne can’t reach Singapore” means: a host in 10.99.60.0/24 sends to 172.28.10.0/24 and something in the path has no route.
  2. Which router is actually failing? The one that lacks the route — not the one that has it. Penny’s complaint named HQ; the fault was at Melbourne.
  3. Did the prefix arrive at all? Everything downstream depends on this answer, and one command settles it.
  4. How many IBGP hops would this route need to take? If the answer is two, stop — it’s split horizon, and no amount of next-hop investigation will help.
  5. Where did the route enter the AS? EBGP-learned routes propagate to IBGP peers freely. IBGP-learned ones don’t propagate at all.
  6. What is the next hop, and can this specific box reach it?
  7. What does the evidence I already have rule out? If the route is in the database, advertisement worked — every advertisement-side hypothesis is already dead.
Question 7 is the discipline that saves the most time. Before offering a hypothesis, check whether the output already on your screen has excluded it. A route sitting in the database excludes every theory about telling — reflectors, filters outbound at the peer, split horizon, session state.
🧠
MENTAL NOTE
Restate the symptom as a packet before you restate it as a protocol. “Melbourne can’t reach Singapore” is a business sentence; “this box has no route to that next hop” is an engineering one.
STEP 03 · EVIDENCE

Commands, and what to look for

1 — DID IT ARRIVE? (THE DATABASE)
get router info bgp network

Status codes: s suppressed, d damped, h history, * valid, > best, i - internal

*>i 172.28.10.0/24  10.99.50.1   0  100  0  65300 i    HEALTHY  (valid + best)
*i  172.28.20.0/24  172.20.0.1   0  100  0  65300 i    FAULT    (valid, NOT best)
(absent)                                                NEVER ARRIVED

Read the legend every time. It is printed on the output and it will be printed on the exam screenshot. The presence or absence of a single > is the entire diagnosis.

2 — IS IT INSTALLED? (THE ROUTING TABLE)
get router info routing-table bgp

Empty, while the database has entries  ->  selection is failing, not transport
3 — CAN I REACH THE NEXT HOP? (THE DECIDING COMMAND)
get router info routing-table details <next-hop-from-column-2>

Routing entry for 172.20.0.0/24 ... via port2      ->  RESOLVABLE, look elsewhere
Routing entry for 0.0.0.0/0      ... static        ->  DEFAULT ONLY = NOT RESOLVED
%  Network not in table                            ->  UNRESOLVABLE, root cause found
Dead end worth knowing: a default route does not satisfy next-hop resolution by default on FortiGate. Engineers routinely see 0.0.0.0/0 in the output, think “there’s a route, so it resolves,” and move on. It does not.
4 — CONFIRM THE PEERING SHAPE
get router info bgp neighbors <ip>
    remote AS == local AS   ->  IBGP  (split horizon applies, next hop preserved)
    remote AS != local AS   ->  EBGP  (AS_PATH prepended, next hop set to self)

get router info bgp summary
    Established + prefixes received, WHILE SERVICE IS BROKEN  ->  expect selection fault
🧠
MENTAL NOTE
Command 3 is the one that ends the investigation. Commands 1 and 2 only tell you which fault class you are in; command 3 names the root cause.
STEP 04 · DECISION TREE

From symptom to root cause

SYMPTOM: an internal router cannot reach a remote-AS prefix | +-- get router info bgp network -- is the prefix there? | +-- ABSENT | | | +-- How many IBGP hops from the router that learned it externally? | | TWO or more -> ROOT CAUSE: IBGP SPLIT HORIZON | | Fix: peer directly (full mesh) or route reflector (S59) | | ONE -> continue | | | +-- Inbound filter on this router? (prefix-list-in / route-map-in) | +-- Outbound filter on the advertiser? (prefix-list-out / route-map-out) | +-- Is the advertiser actually originating it? (network / redistribute) | BGP originates NOTHING by default. | +-- PRESENT, marked *> -> BGP is fine. Look at policy, NAT, return path. | +-- PRESENT, marked * but NOT > -> VALID BUT NOT BEST | +-- More than one path to this prefix? | YES -> it lost selection. Walk the tie-breakers: | weight > local-pref > local origin > AS_PATH length > | origin > MED > IGP metric > EBGP over IBGP > router-id | NO -> it is INELIGIBLE, not defeated. Read NEXT_HOP. | +-- get router info routing-table details <next-hop> | +-- Specific route exists -> resolvable; look for filters/attributes +-- Only a default route -> ROOT CAUSE: NEXT HOP UNRESOLVED +-- No route at all -> ROOT CAUSE: NEXT HOP UNRESOLVED | +-- FIX A: give this router a route to the next hop (IGP/static) | Cost: foreign addressing enters your internal routing +-- FIX B: set next-hop-self on the BORDER router toward IBGP peers Cost: no traffic engineering; border becomes failure domain
🧠
MENTAL NOTE
Notice the shape: the tree splits first on arrived or not, then on valid or best, and only then on why. Never jump to the fix layer before you know which of those three you are in.
STEP 05 · DEAD ENDS

Where engineers lose the hour

Dead endWhy it’s seductiveWhat kills it
Checking session state repeatedly“It must be the peering” is the reflexEstablished sessions are compatible with total service failure. State was never the problem.
Reaching for route reflectorsSymptom shape — “the other router has it and this one can’t use it”If the route is in the database, advertisement already worked. Reflectors change who may tell, not what a router can resolve.
Assuming a default route resolves the next hopThere is a route in the outputFortiGate does not use a default route for next-hop resolution by default.
Hunting firewall policiesIt’s a firewall, and traffic isn’t flowingPolicy failures don’t remove routes from the routing table. An empty routing-table bgp is a routing fault.
Blaming the remote ASIt’s their prefix and their next hopThe prefix arrived intact with correct AS_PATH. They did their job; you can’t reach the address they named.
Enabling multipath“Only one route installed” pattern-matches to ECMPWith zero routes installed and one candidate, multipath is irrelevant. (S56 material misapplied.)
The redirect that works. When stuck, don’t ask “what could it be?” Ask “what has my evidence already ruled out?” A prefix in the database eliminates every advertisement-side theory in one move — and that is usually most of the list.
🧠
MENTAL NOTE
The route-reflector dead end is the instructive one: it is a correct fact applied to the wrong evidence. Most bad diagnoses are, which is why checking evidence beats recalling facts.
STEP 06 · EXAM TRAPS

How this appears on the NSE7

  • The RIP-flavoured split horizon. “Not advertised back to the peer it was learned from” is offered as an option in almost every version of this question. The BGP rule is not to any other IBGP peer.
  • next-hop-self inverted. “EBGP neighbours, for routes learned via IBGP” is the mirror image of the truth and reads plausibly. It is: IBGP neighbours, EBGP-learned routes, on the border router.
  • AS_PATH offered as transit prevention. Automatic, AS-related, and completely wrong for this question. Transit is prevented by outbound filtering, always.
  • Screenshot questions. When you’re shown BGP output, read the status codes legend first and check for >. That single character is frequently the entire question.
  • “Distance vector” for BGP. Some documentation uses the phrase loosely, so it feels defensible. The classification asked for is path vector.
  • Session count questions. n(n−1)/2. Distractors are usually n, 2n, and n². Derive by summation if the formula slips.
  • Select-all attribute questions. Well-known mandatory = ORIGIN, AS_PATH, NEXT_HOP. LOCAL_PREF is discretionary; MED is optional non-transitive. Evaluate each option as an independent true/false.
🧠
MENTAL NOTE
Most distractors here are real facts in the wrong slot. The exam relocates truth rather than inventing nonsense — so ask “is this true of this thing?” not “is this true?”
STEP 07 · THE METHOD

Your personalised sequence

1. Restate the symptom as a packet  →  2. Name the failing router  →  3. Did it arrive?  →  4. What does that rule out?  →  5. Valid or best?  →  6. Read the next hop  →  7. Resolve or rewrite  →  8. Verify both service and root cause

Step 4 is the one to build into a habit. It is cheap, it is fast, and it is what separates a diagnosis from a guess.
