SESSION 55 · TROUBLESHOOTING · BGP PEERING

Diagnosing a Session That Will Not Come Up

How a senior engineer reads a BGP state — and why the state chooses the command, not the other way round.

Junior's diagnostics were all correct. None of them could ever have found the fault.

The expensive troubleshooting error is not a wrong answer. It is a correct answer to a question that could never have mattered.

01

Symptoms — what you are told

What the business saysWhat it usually means technically
"The new circuit is billing but not earning."Link up at layer 1/2; peering not Established.
"It says Active, which sounds good?"TCP connection attempt is failing. Transport layer.
"It keeps coming up and dropping."TCP forms; OPEN or KEEPALIVE negotiation fails. Parameter layer.
"The peering is up but they can't see our network."Established with nothing advertised. Policy layer.
"The route disappeared when ISP2 came online."Almost certainly best-path selection, not a fault at all.
"Some of their routes work and some don't."Next-hop reachability, or an inbound prefix-list.
Translate before you touch anything. Every one of those business sentences already tells you which of four layers to investigate. Most of the diagnostic work is done before you open a CLI session.
🧠
MENTAL NOTE
"Up and dropping" and "up and empty" are completely different faults. If you take only one thing from the symptom table, take the habit of asking which kind of "up" the reporter means.
02

Questions to ask before the first command

  • Is this device the endpoint or the transit? BGP peering terminates on the FortiGate. That is local-in policy territory, not firewall policy. Getting this wrong costs a morning.
  • Is the peer directly attached? EBGP defaults to TTL 1. If there is any device between you and the peer, multihop is mandatory and its absence looks exactly like a routing failure.
  • What address does the peer expect us to source from? FortiGate sources from the egress interface IP by default. If the ISP configured your loopback as their neighbour, they will ignore your SYN entirely.
  • Has this ever worked? A session that never came up is a configuration problem. A session that used to work is a change problem — find the change.
  • What changed on their side? BGP is the one protocol where half the configuration is owned by someone else.
  • Is anything actually declared for advertisement? BGP originates nothing by default. "No routes advertised" is frequently the correct behaviour of an empty configuration.
The discipline: before running any command, ask what output would change your mind. If no possible result could, you are querying the wrong table.
🧠
MENTAL NOTE
Half of a BGP configuration lives on a device you do not administer. "What did they change?" is a legitimate first-line diagnostic, not an excuse.
03

Decision tree — the state selects the layer

Start here, always
get router info bgp summary        → read the State/PfxRcd column

├─ Idle
│   → Is the neighbour configured at all? Administratively down?
│   → Is there ANY route toward the peer address?
│
├─ Connect / Active  (stuck, counters at zero)      [TRANSPORT LAYER]
│   → get router info routing-table details <peer-ip>
│   → Is TCP 179 permitted INBOUND to this FortiGate?
│      · interface allowaccess
│      · local-in policy    <— NOT firewall policy
│   → Upstream ACL or ISP-side filter on 179?
│   → Neighbour IP correct? Source address correct?
│   → Is the peer directly attached?
│      · No  → set ebgp-enforce-multihop enable
│
├─ OpenSent / OpenConfirm  (cycling, counters climbing)  [PARAMETER LAYER]
│   → get router info bgp neighbors <ip>   — read the LAST ERROR
│   → remote-as matches what they actually are?
│   → router-id collision?
│   → authentication / password mismatch?
│   → capability mismatch?
│
├─ Established, PfxRcd = 0                              [POLICY LAYER]
│   → Outbound: anything under config network / redistribute?
│   → network-import-check — EXACT-match active route present?
│   → prefix-list-out filtering everything?
│   → Inbound: prefix-list-in? Are they advertising at all?
│
└─ Established, prefixes received, traffic still wrong   [SELECTION LAYER]
    → get router info bgp network        — is it in the database?
    → get router info routing-table bgp  — did it win?
    → If in database but not table: lost best-path, or next-hop unreachable
🧠
MENTAL NOTE
Working in this order makes the wasted morning structurally impossible. You cannot end up interrogating the transit path for a transport fault, because the state told you which layer you were on before you chose a command.
04

Expected outputs and how to read them

Broken — stuck in Active
Neighbor        V   AS   MsgRcvd MsgSent TblVer InQ OutQ Up/Down State/PfxRcd
203.0.113.1     4 65100       0       0      0   0    0 never   Active

Read it: zero messages in both directions and "never" for uptime. Nothing has ever been exchanged. This cannot be a policy problem — policy operates on a session that exists. Transport layer.

Broken — cycling on parameters
Neighbor        V   AS   MsgRcvd MsgSent TblVer InQ OutQ Up/Down State/PfxRcd
203.0.113.1     4 65100      14      31      0   0    0 00:00:04 OpenConfirm

Read it: counters are climbing, so TCP is forming fine, but uptime keeps resetting to a few seconds. Messages sent far exceed received — we keep trying, they keep rejecting. Go straight to get router info bgp neighbors and read the last error.

Broken — up and empty
Neighbor        V   AS   MsgRcvd MsgSent TblVer InQ OutQ Up/Down State/PfxRcd
203.0.113.1     4 65100     412     409      3   0    0 00:41:12          0

Read it: the last column is now a count, not a state name. Forty-one minutes stable. Everything below policy is healthy. Check origination and filters — and check network-import-check before anything else.

Healthy
Neighbor        V   AS   MsgRcvd MsgSent TblVer InQ OutQ Up/Down State/PfxRcd
203.0.113.1     4 65100     418     415      4   0    0 00:47:03          1
The column that does two jobs. State/PfxRcd shows a state name when the session is not Established and a prefix count when it is. A "0" there is not a state — it is Established with nothing received, which is a completely different problem from Active.
🧠
MENTAL NOTE
Read the counters alongside the state. Zero-and-zero means nothing ever happened. Climbing-but-resetting means it keeps happening and keeps failing. Those two shapes tell you the layer before you read a single other field.
05

Common dead ends

Dead endWhy it feels rightWhy it cannot work
Auditing firewall policy for a failed peeringIt is the table that solves most FortiGate problemsFirewall policy governs traffic through the device. BGP terminates on it. The output will be clean and meaningless.
Pinging the peer and declaring the path healthyICMP succeeds, so "the network is fine"ICMP and TCP 179 are filtered independently. allowaccess permitting ping proves nothing about 179.
Rebuilding the neighbour statement from scratchIt feels decisive and takes two minutesRecreates the same configuration and destroys the evidence. If the fault is external, you have learned nothing and changed state.
Disabling network-import-checkThe advertisement appears immediatelyYou are now advertising reachability you do not have. Symptom gone, black hole created.
Adding ebgp-enforce-multihop speculativelyIt is one line and sometimes fixes thingsOn a genuinely adjacent peer it disables a security control for no benefit. Confirm the hop count first.
Escalating to the ISP on a stuck Active"It must be their end"Roughly half of stuck-Active faults are local-in policy or a wrong source address. Prove your own side first.
🧠
MENTAL NOTE
Notice that four of these six dead ends involve changing something. Under pressure the instinct is to act; the discipline is to gather evidence first, because every change you make destroys the state that would have told you the answer.

MENTAL NOTE
The top two causes account for most stuck-Active sessions and both are entirely on your own side. Prove your own side before you write the ISP ticket — not for politeness, but because it is where the fault statistically lives.
07

Exam traps

  • Router-id offered as a cause of Active. It is not. Router-id travels in the OPEN, which requires TCP to already be up. Wrong layer entirely.
  • A covering supernet next to a config network declaration. The supernet is the distractor. Import check demands an exact match.
  • "The route is missing" when a second peer was just added. Check whether it is in the BGP database first. A route that lost best-path selection is not missing.
  • Firewall policy shown in a peering-failure scenario. If the FortiGate is the peering endpoint, the firewall policy is scenery.
  • TTL values. EBGP 1, IBGP 255. Questions test this as a scenario ("the peer is two hops away") rather than as recall.
  • An answer that would be correct for OSPF. Multicast, automatic discovery, periodic refresh, protocol 89 — all plausible-looking and all wrong for BGP.
Technique: for any BGP scenario question, identify the layer first and eliminate every option belonging to another layer. That usually removes two of four before you have thought about the content.
🧠
MENTAL NOTE
Layer-first elimination is not a shortcut, it is the same reasoning the decision tree encodes. If it works in the exam it is because it works in the rack.
