SESSION 55 · BITE · BGP NEIGHBORS & BASE CONFIG

Five Minutes on Making Peers

Why BGP rides TCP, what Active really means, and why a healthy session can advertise nothing.

The link was up. The interface was up. The routing table was correct. And nothing worked.

Every layer of a BGP peering can be healthy while the layer above it is silently broken. Knowing which layer you are on is the whole skill.

SECTION 01

Why BGP rides TCP 179

OSPF runs directly on IP as protocol 89, multicasts hellos to 224.0.0.5, and maintains its own adjacencies — its own hellos, dead timers, sequence numbers, acknowledgements and retransmission lists. It finds its own friends and guarantees its own delivery.

BGP opens a TCP session to a unicast address you configured by hand. That single decision drives everything else about the protocol.

BGP gives upBGP gains
Because TCP is unicast Peer discovery. Every neighbour declared by hand, on both sides. —
Because TCP needs a route Self-bootstrapping. Something else must already know how to reach the peer. —
Because TCP is reliable — Sequencing, acknowledgement, retransmission, flow control — none of it in BGP's own code.
Consequence worth carrying: because TCP guarantees delivery, BGP never re-floods. It sends an update once and considers it delivered forever, until it explicitly withdraws it. OSPF refreshes every LSA every 30 minutes whether anything changed or not. That is why BGP carries a million routes and OSPF cannot.
🧠
MENTAL NOTE
BGP is a routing protocol that requires routing to already work. "No route to the peer address" belongs at the top of every stuck-session checklist — it is the protocol's oldest and most boring failure.
SECTION 02

The state machine, and the two lies in it

BGP FINITE STATE MACHINE
Idle → Connect → Active → OpenSent → OpenConfirm → Established
  • Idle — not trying. Refusing to try.
  • Connect — SYN sent to the peer on 179. Waiting. (Sounds finished. Isn't.)
  • Active — the TCP attempt failed. Retrying, and listening in case the peer initiates. (Sounds healthy. Is the worst state in the protocol.)
  • OpenSent — TCP up. OPEN sent, awaiting theirs.
  • OpenConfirm — OPENs exchanged and accepted. Awaiting the first KEEPALIVE.
  • Established — peering complete. Only now do UPDATE messages carrying prefixes flow.
The ladder: stuck early = transport · cycling late = parameters · Established but empty = policy.
Why Active vs OpenConfirm matters diagnostically

Two faults that feel identical to a user produce completely different states — because they fail at different layers.

  • Local-in policy blocking 179 → the SYN dies. Stuck in Active, zero messages either way.
  • Duplicate router-id → TCP forms fine, the router-id travels inside the OPEN message, the peer objects, NOTIFICATION, teardown. Session cycles through OpenSent/OpenConfirm forever.
  • Peer three hops away, EBGP TTL 1 → SYN expires in transit. Stuck in Active, indistinguishable from a routing failure until you check the topology.
🧠
MENTAL NOTE
EBGP defaults to TTL 1 on purpose — only a directly-attached device can ever peer with you, so a spoofed SYN from three hops away dies in transit. IBGP defaults to 255 because IBGP peers sit inside your own AS and routinely peer loopback-to-loopback across several internal hops. Adjacency was never the assumption.
SECTION 03

Base configuration — four things

FortiOS CLI — LoFinance HQ, EBGP to ISP-1
config router bgp
    set as 65001
    set router-id 10.99.50.1
    config neighbor
        edit "203.0.113.1"
            set remote-as 65100
            set description "ISP-1 Parramatta"
        next
    end
    config network
        edit 1
            set prefix 192.168.10.0 255.255.255.0
        next
    end
end

Who I am (as). What I'm called (router-id). Who I talk to (neighbor + remote-as). What I'm willing to say (network).

Set the router-id statically. Unset, FortiGate derives it from your addressing — highest interface IP, preferring a loopback. That makes BGP stability a side effect of interface changes: an engineer adding a management VLAN in six months can drop your ISP peering without ever opening the BGP config. A changed router-id tears down and rebuilds every session on the box.

Note the address chosen: 10.99.50.1 is internal, stable, and deliberately not on a transit link, so no circuit change can move it.

🧠
MENTAL NOTE
The reason to set router-id statically is not tidiness. It is decoupling — removing a dependency between your BGP sessions and unrelated interface work done by people who never touch BGP.
SECTION 04

Established, healthy, and advertising nothing

get router info bgp summary
Neighbor    V   AS   MsgRcvd MsgSent TblVer InQ OutQ Up/Down  State/PfxRcd
203.0.113.1 4 65100     412     409      3   0    0 00:41:12          0

Session up for 41 minutes, counters climbing, both sides healthy — and zero prefixes moving. The culprit is network-import-check, enabled by default.

FortiGate will only advertise a config network prefix if an active route with an exactly matching destination subnet exists in the routing table. Not a covering route. A match.

Declared in config networkRouting table holdsAdvertised
192.168.10.0/24192.168.10.0/24 (active)192.168.10.0/24
172.16.5.0/24172.16.0.0/16 (active)Nothing
Why this behaviour exists: advertising a prefix is a commitment to carry traffic for it. Import check is BGP declining to write a cheque the routing table can't cash.
Trap: set network-import-check disable makes the advertisement appear. It does not make it true. Disabling it as a fix for a missing advertisement is how you black-hole traffic for a network you don't actually have. Legitimate uses are narrow — aggregation, deliberate origination.
🧠
MENTAL NOTE
An Established session with zero prefixes is not a transport problem and never was. Stop checking cables and check policy: import check, prefix-lists, or simply nothing worth advertising.
SECTION 05

Four commands, four questions

CommandThe question it answers
get router info bgp summaryAre we peering, and with whom? One line per neighbour: AS, counters, uptime, state or prefix count. First command, always.
get router info bgp neighbors <ip>Why is this one neighbour behaving this way? Peer IP and router-id, remote AS, state, negotiated capabilities, timers, last error.
get router info bgp networkWhat is in the BGP database — everything learned and originated, before best-path selection.
get router info routing-table bgpWhat actually won and got installed.

The gap between the last two is the point. A prefix in the database but not the routing table means it lost best-path selection (the loser stays in the database forever, valid and unused) or its next-hop is unreachable. Inbound-filtered prefixes usually appear in neither.

Operationally: during a multi-ISP incident, "we lost the route" and "we stopped preferring it" look identical in the routing table and completely different in the database. Only one of them is a fault.
🧠
MENTAL NOTE
Up is not peered. Peered is not exchanging. Exchanging is not installed. Four different truths — four different commands.
