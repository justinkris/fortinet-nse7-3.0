TECHNICAL CONCEPTS COVERED (EF pp. 137-141)
==================================================================
1. BGP transport: TCP port 179, unicast, manually configured neighbours. Contrast with OSPF
   (IP protocol 89, multicast 224.0.0.5, self-discovering).
2. THE TRADE-OFF (student derived both halves): BGP LOSES peer discovery (TCP is unicast, so
   neighbours must be declared by hand on both sides) and acquires a bootstrap dependency
   (something else — static route, connected interface, IGP — must already know how to reach
   the peer address). BGP GAINS sequencing, acknowledgement, retransmission and flow control
   for free from TCP, none of which it has to implement. Consequence: BGP sends updates
   incrementally and never re-floods; OSPF refreshes every LSA every 30 minutes. This is why
   BGP scales to a million routes and OSPF does not.
3. THE STATE MACHINE as a diagnostic ladder:
   Idle → Connect → Active → OpenSent → OpenConfirm → Established
   - Idle: not trying / refusing to try
   - Connect: SYN sent to peer on 179, waiting
   - Active: TCP connection attempt FAILED, retrying and also listening
   - OpenSent: TCP up, OPEN sent, awaiting theirs
   - OpenConfirm: OPENs exchanged and accepted, awaiting first KEEPALIVE
   - Established: peering complete, UPDATE messages now flow
   Señor's framing: "Two of these are lies" (Active sounds healthy, Connect sounds finished).
4. CAUSES OF STUCK-IN-ACTIVE (built as a list with the student): no route to peer address;
   TCP 179 blocked upstream (ISP ACL) or locally (local-in policy / allowaccess); wrong
   neighbour IP; AS number mismatch; source address mismatch (FortiGate sources from egress
   interface IP by default — if peer expects a loopback, they ignore the SYN); TTL expiry on
   a non-adjacent peer.
5. EBGP TTL = 1 by default (GTSM security control — only a directly-attached device can peer
   with you; spoofed SYNs from multiple hops away die in transit). IBGP TTL = 255 by default,
   because IBGP peers are inside your own AS and frequently peer loopback-to-loopback across
   internal hops — adjacency was never the assumption. Fix for non-adjacent EBGP:
   `set ebgp-enforce-multihop enable`. PARKED as the centre of S56.
6. ROUTER-ID: carried inside the OPEN message. A duplicate is therefore detected during OPEN
   exchange and tears the session down with a NOTIFICATION — it does NOT hang in Active, it
   cycles through OpenSent/OpenConfirm repeatedly. Set it statically because an unset
   router-id is derived from interface addressing (highest IP, prefers loopback), making BGP
   stability dependent on unrelated interface changes — an engineer adding a management VLAN
   can drop your ISP peering without touching BGP config.
7. `config network` and NETWORK-IMPORT-CHECK (enabled by default): FortiGate only advertises
   a declared prefix if an EXACT-match active route is in the routing table. A covering route
   (172.16.0.0/16 vs declared 172.16.5.0/24) advertises NOTHING, silently, with the session
   showing healthy. `set network-import-check disable` overrides this. Justified only rarely
   (aggregation, deliberate origination); using it to "fix" a missing advertisement
   black-holes traffic.
8. THE FOUR COMMANDS and the distinct question each answers:
   - get router info bgp summary          → are we peering, and with whom?
   - get router info bgp neighbors <ip>   → why is THIS peer behaving this way? (peer IP and
                                             router-id, remote AS, state, negotiated
                                             capabilities, timers, last error)
   - get router info bgp network          → what's in the BGP database (Local RIB), before
                                             best-path selection
   - get router info routing-table bgp    → what actually won and got installed
9. THE GAP BETWEEN DATABASE AND ROUTING TABLE: a prefix present in `bgp network` but absent
   from `routing-table bgp` means either (a) it lost best-path selection — the loser stays in
   the database forever, fully valid, never used — or (b) its next-hop is unreachable and the
   path is marked invalid. Inbound-filtered prefixes typically appear in NEITHER. The
   operational point: if you only look at the routing table you cannot distinguish "we lost
   the route" from "we stopped preferring it," which look identical there and completely
   different in the database.
10. BGP flow diagram (p.140): BGP RX → RIB-in → in filter → Local RIB (+ redistributed routes)
    → out filter → RIB-out → BGP TX.

==================================================================
TROUBLESHOOTING APPROACH
==================================================================
Read the STATE first, and let the state select the layer:
1. Idle              → BGP isn't trying: neighbour misconfigured, admin-down, no route at all
2. Active (stuck)    → TCP failing: routing to peer, local-in policy / allowaccess, upstream
                        ACL, source address mismatch, TTL/multihop
3. OpenSent/OpenConfirm (cycling) → TCP works, parameters disagree: AS mismatch, router-id
                        collision, authentication, capability mismatch
4. Established, 0 prefixes → peering works, policy doesn't: network-import-check,
                        prefix-lists, nothing to advertise
Then: for traffic destined TO the FortiGate, check local-in policy and interface allowaccess,
NOT firewall policy. This was the actual root cause and Junior's mistake.

JUNIOR'S MISTAKE AND WHAT IT TAUGHT
==================================================================
Junior spent ten minutes verifying firewall policy (port1→port3, permit any) and repeatedly
insisted it was fine. It WAS fine. Firewall policy governs traffic passing THROUGH the
FortiGate. BGP peering traffic is destined TO the FortiGate itself, and is governed by
local-in policy and interface `allowaccess` — a completely different table. Every one of his
diagnostics was technically correct and structurally incapable of finding the fault.
LESSON: when the FortiGate is the endpoint, not the transit device, you are in a different
policy table. Also closes his S54-foreshadowed "BGP is OSPF but bigger" instinct in the same
episode.
