Session 55 — Making Peers

BGP neighbors, networks and base configuration. The morning five people spent watching a state machine.

The link was up. The interface was up. The routing table was correct. And nothing worked.

Every layer of a BGP peering can be independently healthy while the layer above it is silently broken. Knowing which layer you are standing on is the entire skill this session teaches.

💭 THINK FIRST · TRANSPORT

OSPF runs directly on IP as protocol 89 and finds its own neighbours with multicast hellos. BGP opens a TCP session to an address you typed in by hand. That is not a cosmetic difference — it determines what BGP has to implement, what it can never do, and what it depends on before it can even start.

  • What does BGP lose by sitting on top of TCP instead of running directly on IP?
  • What work has BGP been able to delete from its own specification because TCP already does it?
Q1 — MODEL ANSWER
Two things. First, peer discovery: TCP is a unicast, point-to-point transport — there is no such thing as a multicast TCP session — so peers can never find each other and every neighbour must be declared by hand on both sides. OSPF finds its friends; BGP has to be introduced. Second, and more subtly, BGP acquires a bootstrap dependency: something else (a connected interface, a static route, or an IGP) must already know how to reach the peer address before the TCP session can be attempted. BGP is a routing protocol that requires routing to already work.
Q2 — MODEL ANSWER
Sequencing, acknowledgement and retransmission. OSPF had to build all three into its own specification — LSA sequence numbers, LSAcks, a per-neighbour retransmission list — along with hello and dead timers for liveness. BGP implements none of it and inherits the lot from TCP. The consequence that matters operationally is that BGP has no need for periodic re-flooding: it sends an update once and considers it delivered forever until it explicitly withdraws it. OSPF refreshes every LSA every thirty minutes whether anything changed or not. That difference is why BGP scales to a million routes and OSPF does not.
SECTION 02

Why BGP rides TCP 179

Señor's point about Javier's three-character email is a real one. Naming the port names the transport, and naming the transport determines the protocol's entire character.

OSPFBGP
TransportIP protocol 89, directTCP port 179
AddressingMulticast 224.0.0.5 / 224.0.0.6Unicast, manually configured
Neighbour discoveryAutomatic, via hellosNone — declared by hand, both sides
ReliabilityBuilt into the protocol (seq, LSAck, retransmit list)Inherited from TCP
LivenessHello 10s / dead 40sKEEPALIVE 60s / hold 180s (defaults)
Refresh modelPeriodic — every LSA re-flooded every 30 minIncremental — send once, withdraw explicitly
PrerequisiteA working L2 segmentWorking routing to the peer address
The scaling consequence. Because TCP guarantees delivery, BGP never has to say anything twice. That is what makes a table of nearly a million internet routes tractable. A link-state protocol that re-floods its entire database on a timer could not do it — the refresh traffic alone would collapse the network.
Analogy — the bootstrap dependency

BGP is a courier who can only deliver to an address that is already on the map — and BGP is the thing that draws the map.

  • OSPF shouts down the corridor and whoever answers is a neighbour. It needs no prior knowledge of anything.
  • BGP is handed a street address and told to go. If the street is not already in the A–Z, it never leaves the building.
  • Which is why "no route to the peer address" is the oldest, most boring, and most frequently correct answer to "why won't this session come up."
🧠
MENTAL NOTE
Put "is the peer address actually reachable, and from the right source address?" at the top of every stuck-session checklist. Verify with get router info routing-table details <peer-ip> — and check the source too, because FortiGate sources the session from the egress interface IP by default, and a peer expecting a different address will simply ignore your SYN.

💭 THINK FIRST · THE STATE MACHINE

The FSM is usually taught as a list of six words to memorise. That is the least useful thing about it. Read properly, the state you are sitting in tells you which layer has failed — and therefore which commands are worth running and which are a waste of a morning.

  • A session sits in Active. List everything that could prevent a TCP SYN on port 179 from completing, roughly ordered by where in the path it would break.
  • A duplicate router-id exists between two peers. Would that produce Active? If not, what state would you see, and why?
Q1 — MODEL ANSWER
Active means the TCP connection attempt failed. Causes, from the inside out: (1) no route to the peer address; (2) local-in policy or interface allowaccess on the FortiGate itself dropping inbound 179 — the FortiGate is the destination here, not a transit device, so firewall policy is the wrong table; (3) an upstream ACL or transit device filtering 179; (4) wrong neighbour IP configured; (5) AS number mismatch; (6) source address mismatch, where the peer only accepts 179 from an address you are not sourcing from; (7) TTL expiry, if the peer is not directly attached and EBGP multihop has not been enabled.
Q2 — MODEL ANSWER
It would not produce Active. The router-id travels inside the OPEN message, which is only sent once TCP is already established. So a collision is detected during OPEN exchange and the session is torn down with a NOTIFICATION — producing a session that repeatedly cycles through OpenSent and OpenConfirm rather than stalling early. That distinction is the diagnostic value of the state machine: stuck early means transport, cycling late means parameters.

TTL: a security control disguised as a limitation

EBGP sends its packets with a TTL of 1 by default. This is deliberate. If a packet can only survive one hop, then only a directly-attached device can ever peer with you — a spoofed BGP SYN from three hops away dies in transit. Free protection, at the cost of one rule: the peer must be adjacent.

The moment the peer is not directly attached, the SYN expires en route and the session sits in Active looking identical to a routing failure. Same symptom, entirely different cause. The fix is set ebgp-enforce-multihop enable.

IBGP defaults to TTL 255 — and the asymmetry explains itself. IBGP peers sit inside your own AS and routinely peer loopback-to-loopback across several internal hops. Adjacency was never the assumption, so there is no adjacency to enforce.
🧠
MENTAL NOTE
Two faults that feel identical to a user produce different states because they fail at different layers. Local-in policy blocking 179 and a peer three hops away both give you Active. A duplicate router-id gives you a session that cycles. Learn to read the state before you form a hypothesis.

💭 THINK FIRST · BASE CONFIGURATION

Four commands build a peering: who I am, what I am called, who I talk to, and what I am willing to say. The third of those — the router-id — is routinely dismissed as cosmetic, which is exactly why it produces confusing outages.

  • What specifically breaks if two BGP routers in the same peering end up with the same router-id, and where in the session lifecycle does it break?
  • What specifically breaks if you leave the router-id unset and let FortiGate choose one?
Q1 — MODEL ANSWER
The router-id is carried inside the OPEN message, so a collision is detected during OPEN exchange — not quietly in the background. The peer rejects the OPEN and tears the session down with a NOTIFICATION. Practically, you see a session cycling repeatedly through OpenSent and OpenConfirm, never reaching Established, with the counters incrementing the whole time. It never stalls in Active, which is the diagnostic signature that separates a router-id collision from a transport fault.
Q2 — MODEL ANSWER
FortiGate derives the router-id from your addressing — typically the highest IP on an interface, preferring a loopback if one exists. Your router-id therefore becomes a side effect of interface configuration. Add an interface with a higher address, or remove the one it chose, and the router-id changes; when the router-id changes, BGP must tear down and rebuild every session on the box. The real risk is organisational rather than technical: an engineer adding a management VLAN six months from now can drop your ISP peering without ever opening the BGP configuration. Setting it statically removes that coupling.

Why 10.99.50.1 specifically? It is a stable internal management address that belongs to no transit link. No circuit change, ISP migration, or interface renumbering can ever move it. That is the whole design argument in one address.
Analogy — the router-id as a passport number

Your router-id is your identity in the negotiation, and it is checked at the door before anyone lets you in the room.

  • It is presented in the OPEN message — the equivalent of handing over your passport at the border, not something checked later.
  • Two travellers with the same passport number do not get quietly flagged; both get turned around at the desk. Hence cycling, not stalling.
  • Letting the system generate it from whatever address happens to be highest is like letting your passport number change every time you move house.
🧠
MENTAL NOTE
"Set it statically because it's good practice" is a slogan, not an argument. The argument is decoupling: an unset router-id makes your BGP session stability dependent on unrelated interface work performed by people who will never open your routing configuration.

💭 THINK FIRST · ADVERTISING

The session reaches Established. Counters climb. Both sides report a healthy peer. And the prefix count is zero in both directions. Nothing is broken at the transport layer, nothing is broken at the parameter layer — so the fault must be somewhere neither of those layers can see.

  • You declared 192.168.10.0/24 under config network. What would the FortiGate need to believe about that prefix before it is willing to tell a stranger it can reach it?
  • The routing table holds an active 172.16.0.0/16. You declare 172.16.5.0/24 under config network. What is advertised, and why?
Q1 — MODEL ANSWER
That it can actually reach it. Specifically, network-import-check — enabled by default — requires an active route in the routing table whose destination subnet exactly matches the declared prefix. Advertising a prefix is a commitment to carry traffic for it, and import check is BGP declining to write a cheque the routing table cannot cash.
Q2 — MODEL ANSWER
Nothing at all, silently, while the session continues to look perfectly healthy. A covering /16 is not an exact match for a declared /24, so import check suppresses the advertisement. This is the classic form of the trap: the operator sees a route in the table that obviously covers the prefix and concludes the configuration must be wrong somewhere else. You can override it with set network-import-check disable, but doing so as a fix for a missing advertisement means advertising reachability you do not have — which black-holes traffic. Legitimate uses are narrow: aggregation, and deliberate origination of a prefix that genuinely is not installed locally.

The senior distinction: disabling import check makes the advertisement appear. It does not make it true. If you reach for this setting because a prefix is missing rather than because you are deliberately originating an aggregate, you have converted a visible configuration problem into an invisible black hole.
Alternative origination path. config network is not the only way to advertise. config redistribute pulls connected, static, RIP, OSPF or IS-IS routes into BGP. By default BGP advertises nothing — origination is always an explicit act.

An Established session with zero prefixes was never a transport problem. Stop checking cables. Check origination (is anything declared?), check import check (does an exact-match active route exist?), then check filters (prefix-lists in and out).

💭 THINK FIRST · THE FOUR COMMANDS

Four show commands, and the temptation is to treat them as four ways of looking at the same thing. They are not. Each answers a question the others cannot, and the gaps between them carry diagnostic information that no single command contains.

  • Describe a scenario where a prefix appears in get router info bgp network but not in get router info routing-table bgp. What is that gap telling you?
  • Why does it matter operationally that you can see the losers as well as the winner?
Q1 — MODEL ANSWER
The commonest cause is best-path selection. BGP receives two paths to the same prefix — say one from each of two ISPs — evaluates their attributes, installs exactly one, and the loser remains in the database indefinitely: fully valid, never used. A second cause is an unreachable next-hop: the path was received and stored, but BGP has no route to the advertised next-hop address, so the path is marked invalid and never considered. A prefix dropped by an inbound prefix-list generally appears in neither, since the filter acts before the local RIB.
Q2 — MODEL ANSWER
Because during an incident, "we lost the route" and "we stopped preferring it" look identical in the routing table and completely different in the database. Only one of them is a fault. If the only place you ever look is the routing table, you cannot distinguish a peer that has withdrawn a prefix from a peer whose path simply lost on attributes — and you will spend the outage troubleshooting a healthy neighbour.

Four commands, four questions

CommandQuestion it answersKey fields

Read the State/PfxRcd column carefully. It is one column doing two jobs. If the session is not Established it shows a state name. If it is Established it shows a prefix count. A "0" there means Established with nothing received — which is a completely different problem from "Active".

Where filtering happens

BGP route flow (EF Study Guide p.140)
  BGP RX → RIB-in → [in filter] → Local RIB → [out filter] → RIB-out → BGP TX
                                            ↑
                                     redistribute
                             (connected, static, RIP, OSPF, IS-IS)

Two filter points, in and out, sitting either side of the local RIB. Routes redistributed from the routing table join at the local RIB, which is why they are subject to outbound filtering but never inbound.

The dead end that cost a morning: checking firewall policy for a session that terminates on the FortiGate. Firewall policy governs traffic passing through the device. BGP peering is traffic destined to the device, governed by local-in policy and interface allowaccess. Every diagnostic run against the transit path will come back clean, correctly, and tell you nothing.
🧠
MENTAL NOTE
Read the state, then choose the layer, then choose the command. The reason to work in that order is that it makes the wasted morning structurally impossible — you cannot end up querying the transit path for a transport fault if the state has already told you which layer you are on.

Exam tip — the state is the hint. Fortinet scenario questions rarely say "the TCP session failed." They show you a state and expect you to infer the layer. Any question that quotes "Active" is a transport question. Any question that describes a session that "comes up and drops repeatedly" is a parameter question. Any question with an Established session and a complaint about missing routes is a policy question.
Exam tip — the covering-route trap. When a question gives you both a config network prefix and a routing table, check for an exact match before anything else. The presence of an obviously-covering supernet is the distractor, not the answer.

SECTION 10

Key takeaways

  • TCP 179. Unicast transport means manual neighbours; reliable transport means no re-flooding; and needing a route means BGP has a bootstrap dependency on routing that already works.
  • The state selects the layer. Stuck early = transport. Cycling late = parameters. Established but empty = policy.
  • Active is the deceptive state. It means the TCP attempt failed.
  • EBGP TTL 1, IBGP TTL 255. One is a security control that assumes adjacency; the other assumes the opposite.
  • Router-id travels in the OPEN. Collisions kill sessions late, not early. Set it statically to decouple BGP from interface work.
  • network-import-check wants an exact match. A covering route advertises nothing, silently, on a session that looks healthy.
  • Four commands, four questions. The gap between the database and the routing table is best-path selection, not a fault.
  • To the box, not through it. BGP peering is local-in policy territory.
🧠
MENTAL NOTE
Up is not peered. Peered is not exchanging. Exchanging is not installed. Four different truths — four different commands.
