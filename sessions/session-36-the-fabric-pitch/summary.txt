TECHNICAL CONCEPTS COVERED (pp. 255-261)
==================================================================
- PURPOSE: Security Fabric = cybersecurity mesh platform on FortiOS; convergence+consolidation;
  three pillars BROAD / INTEGRATED / AUTOMATED (anchor taught post-exam: Broad = Vasse's
  question/see everywhere; Integrated = the islands problem/boxes talk; Automated = respond
  without humans). "BIA — the fabric pitch in three letters." Open ecosystem via connectors.
- TREE TOPOLOGY: one ROOT FortiGate (most upstream — LoFinance = HQ NGFW), downstream
  FortiGates point to upstream; root holds topology views (Physical/Logical), connector config,
  authorization. Root chosen by POSITION IN FLOW, not traffic volume.
- CORE COMPONENTS: FortiGate + FortiAnalyzer = minimum fabric. FAZ = "the fabric's memory";
  topology answers "what exists now," FAZ answers "what happened last Tuesday" (auditors live
  in last Tuesday). EMS/FortiClient extend visibility ONTO the endpoint (student invented this
  before being taught).
- FABRIC vs EXTERNAL CONNECTORS: fabric connectors = Fortinet products (FortiGate, FAZ, EMS,
  Central Management, Sandbox, + FortiNAC/FortiNDR family), configured on the ROOT under
  Security Fabric > Fabric Connectors. External connectors = non-Fortinet/multivendor: public
  SDN (AWS/GCP), private SDN (Kubernetes/VMware), endpoint/identity (FSSO/RADIUS), threat
  feeds. Student's one-liner: "depends on if it is a Forti product or not" — accepted verbatim.
- EXTERNAL FEEDS: plain-text list at a URL; fetched periodically via HTTP/HTTPS/STIX (or Push
  API to FortiGate REST API); creates a RESOURCE OBJECT usable directly in firewall policies;
  list updates without touching policy. Story: replaces Junior's manual blocklist copying.
- FABRIC LOGGING (Use Case 1): the fabric AS A WHOLE logs each session ONCE — the FIRST
  FortiGate to handle it. Members know peers' MAC addresses; packet from a fabric peer's MAC =
  no duplicate traffic log. EXCEPTIONS: (1) NAT on an upstream device forces an ADDITIONAL
  traffic log (new src IP/port = evidence existing nowhere else — "the Rosetta Stone" joining
  10.x identity to public identity); (2) UTM events ALWAYS log wherever inspection happens
  (suppression applies to duplicate traffic logs, never UTM). FAZ auto-correlates, no config.
- ROOT FAILURE: each member logs to FAZ INDEPENDENTLY/directly; root down ≠ logging dark.
  Root coordinates; it is not a logging relay.
- JOINING: downstream sets "Join Existing Fabric" + upstream IP; admin must AUTHORIZE on root
  (FortiGates: System > Firmware & Registration; other products e.g. FortiNDR: alert icon /
  notification on root GUI). Knowing the root IP is never sufficient — two-sided handshake.

TROUBLESHOOTING APPROACH TAUGHT
==================================================================
"Why is this session logged twice / not at all?" reasoning chain: 1) Who was the FIRST fabric
FortiGate in path? That one owns the traffic log. 2) Did any upstream member NAT? → expect a
second, legitimate traffic log. 3) UTM logs appear wherever inspection ran — not duplicates.
4) Missing logs with root down? Root is not a relay — check the member↔FAZ path itself.
5) New device not appearing in fabric? Check authorization queue on root before debugging IP.
