Security Fabric — When the Fabric Frays

The commands matter less than the order of your questions. Fabric problems come in two families: membership ("device isn't in the fabric") and evidence ("logs are duplicated / missing / unexplainable"). Identify the family before typing anything.

Symptom Family 1 — "The device isn't showing in the fabric"

Questions to ask first:

Has anyone authorized it on the root? (This resolves the majority of cases — check before touching any config.)
Is its role set to Join Existing Fabric, and does the upstream IP actually point at a fabric member's join-enabled interface?
Can it reach the upstream device at all — routing, intermediate policy?

Verification path:

  • Root GUI → System > Firmware & Registration → look for the pending request (refresh the page — requests aren't always instant). Non-FortiGate products (FortiNDR etc.): check the notification/alert icon on the root's top bar.
  • Downstream → Security Fabric > Fabric Connectors > Security Fabric Setup → confirm role and Upstream FortiGate IP/FQDN.
  • Root → Security Fabric > Physical/Logical Topology → authorized members appear here.
Dead end to avoid: re-typing IPs, rebooting, or re-entering the fabric name while a perfectly healthy request sits unauthorized in the root's queue. Dave's firewall was never broken. It was unloved.
Likely root causes: unauthorized request (most common) · wrong upstream IP · join not allowed on the upstream interface · plain reachability.

Symptom Family 2 — "FortiAnalyzer shows duplicate / missing / weird logs"

Before any command, rebuild the expected packet path and apply the logging laws:

Session path: Client → ISFW → NGFW(NAT+WebFilter) → Internet Expected evidence: ISFW → 1 traffic log (first FortiGate in path — owns the session) NGFW → 1 traffic log (ONLY because it NATs: new src IP/port) → UTM log(s) (web filter — logs where inspection runs) DCFW → nothing (not in this path) FAZ → correlates all of the above automatically

Decision tree:

  • More traffic logs than expected? → Is one of the "extra" devices NATing? Then it's correct behaviour, not a fault. If no NAT → is the upstream device actually a fabric member? Peer-MAC recognition only suppresses duplicates between fabric members; a non-member upstream logs normally.
  • Fewer logs than expected? → Which device was first in the path? Only that one owes a traffic log. If even the first device's log is missing → check that device's connection to FAZ, not the fabric.
  • UTM event "missing"? → You're looking on the wrong box. UTM logs live on the device that ran the inspection, regardless of who logged the traffic.
  • "Root is down, are we blind?" → No. Members log to FAZ directly and independently. If leaf logs stopped when the root died, the real fault is coincidental (shared path/power), not fabric design — investigate the member↔FAZ path.

Symptom Family 3 — "The external feed isn't blocking anything"

Is the resource object actually referenced in a firewall policy — or was it created and never used?
Is the source list plain text in a supported format, reachable over HTTP/HTTPS/STIX from FortiGate?
Has a fetch happened since the list changed? (Feeds update periodically — "the list changed 30 seconds ago" isn't a fault.)

Exam Traps

TrapTruth
"Every fabric FortiGate in the path logs the session."Only the first — plus any member that NATs.
"NAT log is a duplicate and FAZ discards it."It's required evidence (translated IP/port) and FAZ correlates it.
"UTM logs are suppressed like duplicate traffic logs."Never — UTM logs at the inspection point, always.
"Root failure stops downstream logging."Members log to FAZ independently.
"Join Existing Fabric + correct IP = member."Inert until authorized on the root.
"FSSO/RADIUS are fabric connectors."External connectors — not Fortinet fabric products.

The Senior Habit

Every fabric-evidence question is answered by three restatements, in order: What was the packet's path? Who was first? Who changed the packet or inspected it? The log layout falls out of those answers before you open a single GUI.
