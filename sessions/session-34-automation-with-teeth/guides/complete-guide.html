Story Recap

The Thirty Amber Triangles

Junior built his first CLI template — a fleet-wide DMZ addressing standard driven by a metadata variable — and, proud of himself, assigned it to all thirty managed FortiGates before going home. Wednesday morning: thirty rows of amber. Device-layer status Modified, provisioning-template status Modified, on every device. In line four he had hand-typed the variable reference instead of using the $ picker. One character wrong.

Señor peeled a mandarin and said: "Good. It is Wednesday, and the network still works." Because the network still working was the whole lesson. Assign stages. Install ships. The typo had touched zero live firewalls. Amber is the honest colour — the machine confessing before it acts. The dangerous state would have been thirty green triangles all quietly carrying the wrong DMZ name.

Cipher's law, paid off: "Automation removes variance, not mistakes." A continuous template would have shipped the wrong change perfectly, identically, to all thirty — which is exactly what would have made it a catastrophe.

Concept 1 — The Stencil

What a CLI Template Actually Is

Think First
A template has been assigned to thirty devices. Nobody clicked Install. What has changed, and where? Is the typo live on any FortiGate?

A plain device-DB script edits one device's mirror — one box, one snapshot. But snapshots rot: hand-edits and imports slowly pull thirty devices out of alignment, with nothing holding them the same. A CLI template solves this. It is a provisioning template — a stencil that sits above the device database and, at install time, presses down and overwrites whatever config it covers.

That "overwrite" is the key word from the guide: CLI templates overwrite the device database. The template wins. That is why assigning Junior's template flipped two statuses — the device layer (about to be overwritten) and the provisioning template (the stencil itself changed/assigned). The policy package stayed quiet, because nothing touched the policy layer.

Mental Model
A CLI template is a stencil laid over the device database. You paint the config through it once, and every device under the stencil gets the same pattern — and gets it again every time you press down. You cannot out-paint the stencil from underneath.
📌
Anchor
Assign stages. Install ships. A template assigned but not installed has touched zero live firewalls — amber is the machine confessing before it acts.
Concept 2 — Assign vs the Eight Verbs

Where Assign Fits in the Machine

Your eight operations — retrieve, import, install, reinstall, quick install, revert, install device settings, refresh — are all delivery/sync verbs. They move config between layers. Assign is not one of them. Assign is a binding verb: it declares "this stencil now applies to these devices." It creates the relationship and flips the device-layer status to Modified — but it ships nothing.

Assign edits the recipe; Install cooks it. Assign changes what the device database will resolve to at the next install. The live FortiGate hears nothing until a real install verb runs.
OperationKindTouches live FortiGate?
Assign (template)BindingNo — stages device-layer Modified
Install / ReinstallDeliveryYes
Install Device SettingsDelivery (device layer only)Yes
RetrieveSync (box → device DB)Reads from it
ImportSync (device DB → policy layer)No
Concept 3 — Two Flavours

Pre-Run vs Continuous

"Are templates just for onboarding, or pushed at every install?" — the fork the whole episode turns on. There are two kinds, and they behave oppositely:

Pre-Run CLI TemplateContinuous CLI Template
PurposeOne-time onboarding (ZTP/LTP)Ongoing enforcement
After installAuto-unassigns; does not remain in device DBStays bound
Re-applicationFire once, goneRe-overwrites on every install
DriftNot corrected after detachAutomatically stomped

Junior used the continuous flavour. That is why his typo is a recurring threat, not a one-shot: every future install re-presses the stencil. This is the "teeth" — a continuous template's whole job is to keep winning. The feature and the danger are the same sentence.

🧊
Exam Trap
"Enforce a standard fleet-wide and auto-correct drift" → continuous template. "Set it once" → script. The keywords permanent / standard / drift point to the stencil, never the letter.
Concept 4 — The Bent Pointer

Metadata Variables: Value vs Reference

A metadata variable lives in two separate places, and knowing which is which decides your blast radius:

  • The value lives in Policy & Objects > Advanced > Metadata Variables — the dictionary. It supports per-device mapping (a default plus custom values per firewall).
  • The reference — $(dmz_name) — is used inside the template. It reaches into the dictionary and pastes whatever it finds.

Junior's typo was in the reference (template line four), not the value. The dictionary entry dmz_name was sitting there correct; the pointer to it was bent. He hand-typed instead of using the $ picker — and the picker exists precisely so you cannot misspell a reference, because you choose from real entries.

The dictionary is right; the pointer is bent. And note the horrible symmetry: either a wrong value or a wrong reference hits all thirty — because both the dictionary entry and the template are single shared objects. That is "one character, thirty firewalls."
Mental Model
The $ picker is a seatbelt. Hand-typing the reference is declining it. Junior didn't just make a typo — he opted out of the guardrail the tool was holding out to him.
Concept 5 — The Single Door

Policy-Package Scripts & the Two-Hop Install

Think First
A script creates an address object and drops it in a policy, aimed at the policy package. Green "script ran." Which status flips, what verb ships it, and — the trap — is the object live yet?

Green means the script ran — it wrote the object and policy into the policy layer. The policy package flips Modified. But the FortiGate has heard nothing. Staged, not shipped — the identical law as templates, worn by a different surface. The object is not live until Install Policy Package runs.

And that install is a two-hop path:

Install Policy Package — delivery path
policy layer  →  device database  →  FortiGate
              (render)          (push)

Nothing skips the device database. Policy-layer changes don't teleport to the box; they render into the mirror first, then the mirror ships. The device DB is the single door every change walks through to reach the FortiGate — which is exactly why templates (device layer) and policy scripts (policy layer) converge there.

🚪
Anchor
The device database is the single door. This one insight explains the two-hop install and why reinstall can drag staged device-DB changes down with it.
Concept 6 — Attached vs Unattached

What an Install Carries

Junior's script created two address objects: one placed in a policy, one made "for later" and never referenced. After Install Policy Package, only the referenced object reaches the FortiGate. The orphan stays parked in the ADOM database, uninstalled.

Install carries what the package USES, not everything it CONTAINS. An object no policy references is unused config — the install has no reason to render it down. It isn't lost; it's just not carried, because nothing points at it. (A tool left in the drawer.)
Concept 7 — Reinstall, Mechanically

Skips the Wizard, Keeps the Confession

Reinstall bypasses the installation wizard (device and package are already paired — no "which device? which package?"). It keeps the installation preview. Skips the questions, never the confession.

Why does keeping the preview matter more, not less? Because reinstall also drags staged device-DB modifications down through the same door — straight to the FortiGate. So a "routine" reinstall of a policy package can quietly ship a device-layer change you forgot was staged. The preview is where you catch it.

🔍
Anchor
"Reinstall skips the questions, never the confession." The preview earns its keep because reinstall carries device-DB drift down alongside the package.

Troubleshooting Workflow

"The Fix Won't Stick"

  1. Don't fix the FortiGate. A continuous template re-overwrites the box on every install. Fix the level that wins — the template.
  2. Metadata typo? Decide: is it the value (dictionary) or the reference (template line)? A wrong reference resolves to nothing across every assigned device.
  3. Canary first. Install to a staging/prototype device that mirrors prod but doesn't matter if it breaks — verify where failure is free.
  4. Read the preview on the canary.
  5. Widen to the fleet only after the canary is green.
The consolidated framework: Fix at the level that wins → verify where failure is free → widen only after evidence.
Common Mistakes

Where Engineers Slip

  • Fixing the FortiGate directly under a continuous template — stomped on next install.
  • Reading a green checkmark as "delivered." Green = ran. Amber/Modified = staged. Install = shipped.
  • Hand-typing a metadata reference instead of using the $ picker — declining the seatbelt.
  • Installing the whole policy package to deliver a device-layer tweak — match the install to the layer.
  • Assuming an unreferenced object installs. Only what the package uses is carried.
  • Treating reinstall as low-risk — it can drag staged device-DB changes down; read the preview.
Senior Engineering Insights

Beyond the Exam

  • Blast radius is a design axis. Automation that removes variance also removes the variance in your mistakes. Gate fleet-wide changes with a human and a canary.
  • Fix where the config wins. Any fix applied below the enforcing layer is temporary by construction.
  • Verify where failure is free. A staging box that mirrors prod turns "courage" into "a lab."
  • The device database is the single door. Reason about every change as: where does it live, and what door must it walk through to become real?
Exam Tips

NSE7 Targeting

The "still true in six months" test. When two options both achieve the immediate result, pick the one that holds the state (continuous template) over the one that merely acts (one-time script). Requirement words like permanent, standard, fleet-wide, drift are the tell.
  • Assign flips device-layer + provisioning-template status, live box untouched.
  • Pre-run auto-unassigns; continuous re-applies on every install.
  • Policy-package script → package Modified → Install Policy Package (policy → device DB → box).
  • Only referenced objects install; orphans stay in the ADOM DB.
  • Reinstall = skip wizard, keep preview; it also carries device-DB mods down.
  • Value in Metadata Variables; reference ($(...)) inside the template.
