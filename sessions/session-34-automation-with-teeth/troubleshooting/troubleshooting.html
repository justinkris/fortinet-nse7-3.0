Troubleshooting Guide · Session 34

When Automation Bites Back

How a senior engineer thinks through a template or script that "ran" but didn't land — or landed everywhere.

The emphasis is thinking, not memorising commands.
Every fault below is a question about which layer and which door, not about syntax.
Symptoms You'll See

What the Screen Tells You

Thirty amber triangles after assigning a template — device-layer and provisioning-template status both Modified.
"The script ran (green) but nothing changed on the box."
"I fixed it on the FortiGate but it reverted after the next install."
"An object I made never showed up on the firewall."
"A routine reinstall changed something I didn't expect."
Questions to Ask First

Before You Touch Anything

  • Which layer did the change land in? Device layer (template / device-DB script) or policy layer (policy-package script)?
  • Was anything installed, or only assigned/run? Amber/Modified = staged. Green checkmark = ran, not shipped.
  • Is a continuous template assigned to this device? If so, any fix below it is temporary.
  • Is the object referenced by a policy? Unreferenced objects don't install.
  • For a metadata fault: value or reference? Dictionary entry vs template line.

Decision Tree

"It Ran, But Nothing Happened"

Is the Config Status Modified (amber)?
Yes → it's staged, not shipped. You didn't install. → Install the right layer:
  • Device-layer change (template / device-DB script) → Install Device Settings.
  • Policy-layer change (policy-package script) → Install Policy Package (two-hop).
No — it says Synchronized but the change still isn't right?
Object missing? → Is it referenced by a policy? If not, it stayed in the ADOM DB (not carried).
Value wrong on all devices? → Metadata value (dictionary) or reference (template). Fix the right one.
Fix keeps reverting? → A continuous template is overwriting you. Fix the template, not the box.
Recovery Sequence

The Safe Fix

  1. Fix at the level that wins. If a template enforces it, edit the template. If a shared value is wrong, edit the dictionary entry.
  2. Re-insert references with the $ picker — restore the seatbelt.
  3. Canary install to a staging/prototype device that mirrors prod but doesn't matter if it breaks — verify where failure is free.
  4. Read the install preview on the canary. Confirm exactly what ships (watch for device-DB drift riding along).
  5. Widen to the fleet only after the canary is green.
Framework: Fix at the level that wins → verify where failure is free → widen only after evidence.
Common Dead Ends

Where Time Gets Wasted

  • Debugging the script's syntax when the real fault is the run target / layer. Check where it landed first.
  • Fixing the FortiGate by hand under a continuous template — you're painting under a stencil.
  • Re-running Install Policy Package to deliver a device-layer change — heavier than needed, bigger surface for something else to be stale.
  • Assuming a made object installed because the script succeeded — unreferenced = uninstalled.
  • Skipping the reinstall preview because "it's just a reinstall" — that's exactly when device-DB drift ships silently.

Exam Traps

Where the Question Bites

  • "Ran successfully" in the stem does not mean "on the FortiGate." Look for whether an install happened.
  • "Enforce / auto-correct drift" → continuous template, never a one-time script — even if the script would set it correctly today.
  • Reinstall keeps the preview and can carry device-DB mods — not the "low-risk quick push" it looks like.
  • Only referenced objects install — distractors will offer "both objects" or "neither."
  • Match install to layer — Install Device Settings for device-layer, Install Policy Package for policy-layer.
