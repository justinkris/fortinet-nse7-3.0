==================================================================
BUSINESS PROBLEM
==================================================================
A continuous CLI template assigned fleet-wide can enforce a standard across 30 firewalls with one
edit — but a single typo in the template (or in a shared metadata value) hits ALL THIRTY
identically, because the stencil and the dictionary entry are single shared objects. "Automation
removes variance, not mistakes": the wrong change would ship perfectly, everywhere, at once. The
save was that ASSIGN STAGES, INSTALL SHIPS — nothing reached a live FortiGate.

==================================================================
TECHNICAL CONCEPTS COVERED (pp. 48-53)
==================================================================
CLI TEMPLATES (device layer):
- A CLI template is a provisioning template. It "overwrites the device database" — a STENCIL that
  sits ABOVE the device DB and presses down at install, winning over device-layer config.
- ASSIGN is a BINDING verb, NOT a delivery verb (distinct from the eight sync verbs: retrieve,
  import, install, reinstall, quick install, revert, install device settings, refresh). Assign
  binds the stencil to devices and flips device-layer status to Modified — but ships nothing.
  Assign edits the PENDING device-DB state, never the live box.
- Assigning a CLI template flips TWO statuses: device-layer Modified AND provisioning-template
  Modified. (Contrast: a device-DB script flips only device-layer.) The policy package stays
  quiet — nothing touched the policy layer.
- You CANNOT out-edit a template from below: a manual fix on the FortiGate (or its mirror) is
  re-overwritten on the next install. Fix must happen IN THE TEMPLATE (the level that wins).
- PRE-RUN CLI TEMPLATE: one-time, ZTP/LTP onboarding; AUTO-UNASSIGNS after quick-install; does
  NOT remain in the device DB. Fire once, gone.
- CONTINUOUS CLI TEMPLATE: stays bound; NOT auto-unassigned; RE-OVERWRITES on EVERY install =
  drift enforcement. This is the "teeth." Junior used this flavour → recurring typo threat.
- Delivery of a device-layer template change to one box: INSTALL DEVICE SETTINGS (not Install
  Policy Package — don't drag the whole policy layer to fix a device-layer setting). "Match the
  install to the layer."
METADATA VARIABLES (S32 payoff):
- Two separate places: VALUE lives in Policy & Objects > Advanced > Metadata Variables (the
  dictionary; per-device mapping supported → default + custom). REFERENCE (e.g. $(dmz_name)) is
  used INSIDE the template. "The dictionary is right; the pointer to it is bent."
- The $ pop-up picker prevents misspelling the reference (you choose from real entries).
  Hand-typing = declining the seatbelt. Junior's typo was in the REFERENCE (template line 4), not
  the value.
- EITHER a wrong value OR a wrong reference hits all 30 — both are single shared objects. That is
  the "one character, thirty firewalls."
POLICY-PACKAGE SCRIPTS (policy layer):
- Run target = policy package. Green "script ran" flags POLICY PACKAGE Modified; NOTHING is live
  on the FortiGate yet (staged, not shipped — the SAME law as templates, different surface).
- Delivery: INSTALL POLICY PACKAGE — a TWO-HOP path: policy layer → DEVICE DATABASE → FortiGate.
  Nothing skips the device DB; it is the single door every change walks through to reach the box.
- ATTACHED vs UNATTACHED: install carries objects the package USES (referenced by a policy) plus
  the rules; an unreferenced object stays PARKED in the ADOM database, uninstalled. "Install
  carries what the package uses, not everything it contains."
REINSTALL (S30/S33 carry, now anchored WITH mechanism):
- Reinstall SKIPS THE WIZARD (device/package already paired), KEEPS THE PREVIEW. "Skips the
  questions, never the confession."
- WHY the preview matters MORE: reinstall drags STAGED DEVICE-DB MODIFICATIONS down through the
  same door (two-hop). The preview is where you catch the device-layer change you forgot was
  staged, riding down on a "routine" reinstall.

TROUBLESHOOTING APPROACH TAUGHT
==================================================================
Continuous-template misfire / "the fix won't stick":
1) Don't fix the FortiGate — the template re-overwrites it. Fix the LEVEL THAT WINS (template).
2) For a metadata typo: is it the VALUE (dictionary) or the REFERENCE (template line)? Story clue
   decides. Reference typo → resolves to nothing → all assigned devices.
3) Canary install to a staging/prototype box that mirrors prod (failure = free).
4) Read the install PREVIEW on the canary.
5) Widen to the fleet only after the canary is green.
Framework consolidated: FIX AT THE LEVEL THAT WINS → VERIFY WHERE FAILURE IS FREE → WIDEN AFTER
EVIDENCE.
