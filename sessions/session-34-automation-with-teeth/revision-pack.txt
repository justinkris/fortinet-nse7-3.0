## 1. QUICK SUMMARY (one page)
######################################################################

Junior built his first CONTINUOUS CLI TEMPLATE (fleet DMZ standard, driven by
a metadata variable) and assigned it to all 30 FortiGates — with a one-character
typo in the variable REFERENCE (hand-typed instead of using the $ picker). Thirty
amber triangles. But: ASSIGN STAGES, INSTALL SHIPS. Nothing reached a live box.
Amber = the machine confessing before it acts.

Key machine facts:
- A CLI template is a device-layer STENCIL that OVERWRITES the device database at
  install. It WINS — you cannot out-edit it from below (a manual FortiGate fix is
  re-overwritten next install). Fix the STENCIL, not the glass.
- ASSIGN is a BINDING verb, not a delivery verb. It flips TWO statuses (device
  layer + provisioning template) and ships nothing. (Assign edits the recipe;
  install cooks it.)
- PRE-RUN template = one-time onboarding (ZTP/LTP), AUTO-UNASSIGNS after install.
  CONTINUOUS template = stays bound, RE-OVERWRITES on every install = enforcement.
  That recurring re-application is the "teeth."
- METADATA VARIABLE has two homes: VALUE in Policy & Objects > Advanced > Metadata
  Variables (dictionary, per-device mapping); REFERENCE $(name) INSIDE the template.
  The $ picker prevents misspelling the reference. "Dictionary right, pointer bent."
- POLICY-PACKAGE SCRIPT: green = ran, not shipped; flips PACKAGE Modified; deliver
  with INSTALL POLICY PACKAGE (two-hop: policy layer -> device database -> FortiGate).
  NOTHING skips the device DB — the single door.
- ATTACHED vs UNATTACHED: install carries objects the package USES (referenced by a
  policy) + the rules. Unreferenced objects stay parked in the ADOM DB.
- REINSTALL skips the WIZARD, keeps the PREVIEW — and matters MORE because it drags
  staged device-DB modifications down the same door; the preview catches them.

Discipline consolidated (effectively wrote George's change-control policy):
  FIX AT THE LEVEL THAT WINS -> VERIFY WHERE FAILURE IS FREE -> WIDEN AFTER EVIDENCE.

## 2. CHEAT SHEET (commands · GUI paths · defaults · behaviours)
######################################################################

GUI PATHS
  Create / assign CLI template ...... Device Manager > Provisioning Templates > CLI Template
  Metadata variable VALUE ........... Policy & Objects > Advanced > Metadata Variables
  Run a script (pick target) ........ Device Manager > Scripts
  Config Status (Sync/Modified) ..... Device Manager > Device & Groups > Managed FortiGate
  Revision history .................. Dashboard > Summary > Configuration and Installation > Revision

REFERENCE SYNTAX
  In a template:  set subnet $(dmz_subnet)    <- insert via the $ picker, never by hand

INSTALL VERBS — MATCH TO LAYER
  Device-layer change (template / device-DB script) .... Install Device Settings
  Policy-layer change (policy-package script) .......... Install Policy Package (two-hop)
  Already-paired re-push .............................. Reinstall (skips wizard, keeps preview)

STATUS BEHAVIOURS
  Assign template ......... device-layer Modified + provisioning-template Modified (box untouched)
  Device-DB script ........ device-layer Modified only
  Policy-package script ... policy package Modified (object NOT live until install)
  Remote-FortiGate script . runs on box, FMG auto-retrieves to device DB (except policy changes)

TEMPLATE FLAVOURS
  Pre-run ..... one-time, ZTP/LTP, AUTO-UNASSIGNS, not retained in device DB
  Continuous .. persists, OVERWRITES device DB on EVERY install, enforces/corrects drift

TWO-HOP INSTALL PATH
  Install Policy Package:  policy layer  ->  device database  ->  FortiGate
  (Nothing skips the device DB. It is the single door.)

DESIGN RULES
  - Fix at the level that wins (template > device config).
  - Match the install to the layer (don't ship the constitution to fix a street sign).
  - Verify where failure is free (canary in staging that mirrors prod).
  - Read the preview — especially on reinstall (device-DB drift rides along).
  - Use the $ picker for references (seatbelt).

## 3. FLASHCARDS (Q -> A)
######################################################################

Q: A CLI template is assigned to 30 devices, no install run. What's true?
A: Device-layer + provisioning-template status = Modified on all 30; nothing is
   on any live FortiGate. Assign stages, install ships.

Q: Is "Assign" one of the eight install/sync verbs?
A: No. Assign is a BINDING verb (creates the template->device relationship). It
   flips status but delivers nothing.

Q: Pre-run vs continuous CLI template — the defining difference?
A: Pre-run AUTO-UNASSIGNS after install (one-time onboarding). Continuous STAYS
   BOUND and re-overwrites on every install (enforcement).

Q: A CLI template "overwrites the device database." You fix the setting directly on
   the FortiGate. After the next install?
A: The template re-overwrites it; the manual fix is lost. Fix the template.

Q: Where does a metadata variable's VALUE live? Its REFERENCE?
A: Value in Policy & Objects > Advanced > Metadata Variables (dictionary). Reference
   $(name) inside the template.

Q: What does the $ pop-up picker protect against?
A: Misspelling the reference — you choose from real dictionary entries. Hand-typing
   declines that guardrail.

Q: A policy-package script shows a green "ran" result. Is the object live on the box?
A: No. The policy package is flagged Modified. It's live only after Install Policy Package.

Q: What path does Install Policy Package take?
A: policy layer -> device database -> FortiGate (two-hop). Nothing skips the device DB.

Q: A policy-package script makes two objects; one is in a policy, one isn't. After
   install, which reaches the FortiGate?
A: Only the referenced one. The orphan stays in the ADOM database (install carries
   what the package USES, not everything it contains).

Q: How does Reinstall differ from a standard Install?
A: Skips the WIZARD (device + package already paired), KEEPS the PREVIEW.

Q: Why does the reinstall preview matter MORE, not less?
A: Reinstall also drags staged device-DB modifications down the same door; the
   preview is where you catch a device-layer change you forgot was staged.

Q: Enforce identical SNMP fleet-wide with drift auto-corrected — script or template?
A: Continuous CLI template. A one-time script sets it today but corrects no future drift.

Q: Correct install verb to deliver a device-layer template change to one box?
A: Install Device Settings (not Install Policy Package — don't drag the whole policy layer).

Q: Why does a single-character typo hit all 30 firewalls?
A: The template and the metadata dictionary entry are single shared objects; every
   assigned device inherits the same fault.

Q: MCQ heuristic when two options both work?
A: Ask "which is still true in six months?" permanent/standard/drift -> template;
   set-once -> script.
