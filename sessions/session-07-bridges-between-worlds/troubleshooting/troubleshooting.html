SESSION 07 · TROUBLESHOOTING GUIDE

Diagnosing Broken Inter-VDOM Connectivity

How a senior engineer thinks through "Guest-WiFi can't reach the internet," not just which commands to run.

Symptom Intake — Ask These First

  • Is this a total outage (nothing at all reaches anywhere) or partial (reaches root but not the internet, or vice versa)?
  • Did this VDOM ever work, or is this a brand-new build with no prior known-good state?
  • Was anything changed recently — new link, new route, new policy, VDOM resource limits touched?
  • Is the failure isolated to one VDOM, or does it affect traffic between multiple VDOM pairs?
A "new build, never worked" report points you straight at missing config (link/route/policy). A "used to work, now broken" report points you at something that changed — check Global Resources / Override Maximum and recent commits first.

The Three-Layer Model — Where Could It Be Failing?

Every inter-VDOM path has exactly three things that must all be true, in this order. Check them in order — don't jump to policy first just because that's usually where people look.

  1. Does the link interface pair exist and is each half correctly assigned to the VDOM you expect?
  2. Does each VDOM's own routing table have a route pointing traffic toward its half of the link (or onward, on root's side)?
  3. Does each VDOM's own policy table have a matching policy (srcintf/dstintf) allowing that specific traffic?

Verification Commands — What to Actually Run

Confirm the link interface exists and check VDOM assignment
diagnose sys vd list
config system interface
    show | grep -A 5 "VDOM_Link"
Confirm the route exists, per VDOM
config vdom
edit Guest-WiFi
get router info routing-table static
get router info routing-table all
Confirm the policy exists and is actually matching traffic
config vdom
edit Guest-WiFi
show firewall policy
diagnose firewall iprope list 100017
Live packet trace — the single most useful command here
diagnose debug flow filter addr <guest-device-ip>
diagnose debug flow show console enable
diagnose debug enable
Expected output shape: a healthy flow trace shows the packet being routed (egress interface chosen) as a distinct line BEFORE the policy match line. If the trace stops right after routing with no policy match logged at all, that's your stage-of-failure answer, not a guess.

Decision Tree

Does the link interface pair show up in both VDOMs' interface lists?
NO → Link was never created, or was created but assigned to the wrong VDOM. Rebuild/reassign at Global VDOM > Network > Interfaces.
YES → continue
Does get router info routing-table all in the affected VDOM show a route pointing at its half of the link?
NO → Missing route. This is a routing-table problem, not a policy problem — flow trace will confirm the packet never gets an egress interface assigned.
YES → continue
Does the flow trace show a policy match on that VDOM for this traffic?
NO → Route is fine; policy is missing or the srcintf/dstintf pair doesn't match what's actually happening. Check policy order and interface names carefully — typos in interface names are the #1 cause here.
YES → continue to the OTHER VDOM and repeat all three checks — remember, every hop needs its own full pass.
Both VDOMs pass all three checks and it's still not working?
Check Global Resources / Override Maximum — a VDOM silently hitting a session-table or policy-count ceiling can behave exactly like a missing policy. Also check NAT configuration on root's egress policy — routing + policy can both succeed and traffic can still fail post-NAT.

Common Dead Ends

"The policy is definitely there, so it must be routing." — Check policy order and exact srcintf/dstintf spelling before assuming routing. A policy sitting below a broader deny rule, or referencing the wrong interface name, looks identical to "no policy" in symptoms.
"I added the physical cable as a workaround and now it 'sort of' works." — This masks the real gap (missing route or policy on the virtual link) and creates a second, undocumented path that will confuse the next engineer. Fix the actual link/route/policy chain instead.
"Root has internet, Guest-WiFi has a route to root, so it should work." — Having a route toward root doesn't mean root has a policy or route allowing THAT traffic onward to the internet. Every hop needs its own full three-layer check — don't assume the far side is fine just because it "already had internet."

Exam Traps

  • Assuming link creation alone enables traffic flow — it never does.
  • Assuming "no policy = deny logged" — in inter-VDOM contexts, a missing route means the packet may never even reach policy evaluation to be logged as a deny.
  • Confusing internal VDOM isolation (no link = no path) with external L2/physical isolation (a completely separate, non-VDOM concern).
  • Forgetting that BOTH VDOMs need independent verification — a fix on one side that "should" work is meaningless if the far side was never checked.
