SESSION 07 · BITE · INTER-VDOM LINKS

Bridges Between Worlds

The Problem

No Cable Needed

Two VDOMs on the same physical FortiGate never need a physical cable between them. Use an inter-VDOM link — a paired virtual interface — instead of burning two physical ports.

Building It

One Object, Two Auto-Named Interfaces

Built once from Global VDOM > Network > Interfaces > Create New > VDOM Link — both VDOM sides chosen in that same step. Base name X becomes X0 and X1.

Gotcha: which side is 0 vs 1 isn't a fixed rule — verify each interface's actual VDOM assignment, don't assume.
Three Requirements

Link + Route + Policy — All Three, Both Sides

  • Route in each VDOM's own routing table, pointing at its half of the link.
  • Policy in each VDOM's own policy table (srcintf/dstintf) — no policy spans two VDOMs.
  • Routing decides the egress interface first; policy lookup happens after, using that dstintf.
The Core Model

Isolation by Default, Connection by Exception

VDOMs default to "physically nonexistent to each other" — not "connected unless blocked." No link = no interface = no route possible = no path. Stronger than a deny rule, because there's nothing to evaluate a policy against.

Trading VDOM stays safe because nobody ever built a bridge there — not because of a rule that could be loosened.
Watch For

Internal vs. External Isolation

VDOM isolation (internal, routing-table level) is a different boundary from physical/L2 cabling isolation (external, outside the FortiGate entirely). Don't conflate the two on the exam.
