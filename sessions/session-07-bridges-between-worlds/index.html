Concepts Covered 7 concepts
  1. 01
    Inter-VDOM link purpose
    connects two VDOMs on the SAME physical FortiGate via a paired virtual interface instead of a physical cable — eliminates wasted physical ports for VDOM-to-VDOM traffic. Student independently reasoned this out from first principles (physical waste argument) before being told, extending the "carve the control plane, share the physical plane" VRF analogy from Session 6.
  2. 02
    VDOM link creation
    built once, from Global VDOM context (Global VDOM > Network > Interfaces > Create New > VDOM Link) — NOT created twice, once per VDOM. It is a single object spanning two VDOMs, with both VDOM assignments chosen in the same creation step. Student initially assumed (incorrectly, briefly) that it needed to be built on both sides independently — self-corrected almost immediately after the naming-convention discussion, then explicitly asked the clarifying question, which was answered directly (good instinct to check rather than assume).
  3. 03
    Auto-generated interface naming
    FortiGate appends 0 and 1 to the base link name (e.g. VDOM_Link → VDOM_Link0 / VDOM_Link1) because two interfaces cannot share an identical name inside the same device's interface list — student reasoned this out himself when prompted ("how would the CLI tell them apart?"). Refinement taught: which side gets 0 vs 1 is NOT a meaningful choice to memorize — it's assigned based on which VDOM you configure first in the creation dialog, and you verify placement afterward via each interface's actual VDOM assignment (config system interface / GUI interface list), not by assuming a fixed rule.
  4. 04
    Two separate config layers required per side, since each VDOM maintains a fully independent routing table AND policy table
    - Routing: static (or dynamic) route in each VDOM pointing traffic toward its own end of the link (device = VDOM_Link0 or VDOM_Link1, dst = far-side network or 0.0.0.0/0.0.0.0 for internet-bound default in Guest-WiFi's case). - Policy: firewall policy in each VDOM's OWN policy table (srcintf/dstintf pair), since Session 6 established Policy & Objects doesn't exist at the Global VDOM level — there's no such thing as a policy spanning two VDOMs. Student correctly derived the full two-hop packet path unprompted: Guest-WiFi route → Guest-WiFi policy → cross link → root route (root's PRE-EXISTING default route, correctly identified as already present, not newly needed) → root policy → WAN/NAT.
  5. 05
    Routing-before-policy retrieval (Session 4/5 callback)
    reconfirmed cold, with minor phrasing looseness on first pass ("no routing and the dst int needs to be chosen first") — restated cleanly by Claude and student's subsequent answers (Q2 exam question) demonstrated the model landed correctly on retry.
  6. 06
    VDOM isolation-by-default model (the session's central mental model, self-built through Socratic pressure, not handed over)
    VDOMs do NOT default to "connected unless blocked by policy" — they default to "physically nonexistent to each other unless an inter-VDOM link explicitly exists." No link = no interface = no route possible = no path at all, which is STRONGER isolation than a firewall deny rule, because there's nothing present to even evaluate a policy against. Student's first attempt at explaining Trading's safety incorrectly reached for "accidentally through root, physically out of the box and back" — a live, useful near-miss, since it describes a genuine but DIFFERENT failure mode (external L2/switching misconfig, completely outside VDOM routing/policy control) rather than anything internal to VDOM link/route/policy state. This distinction (internal VDOM isolation vs. external physical/L2 topology risk) was explicitly separated out and should be considered "taught" but worth one light retrieval check in a future session, since it required correction rather than being self-generated cleanly.
  7. 07
    Practical implication / exam-relevant takeaway
    extending an inter-VDOM boundary later (e.g. Trading ↔ Guest-WiFi for "one file transfer") requires building a brand new inter-VDOM link plus its own routes and policies — the ABSENCE of a link is itself the entire security control, not a rule that could accidentally be loosened. Student named this correctly and immediately when asked directly.
