Section 1 — The Business Problem

Why Not Just Run a Cable?

Session 6 ended with a new Guest-WiFi VDOM built and correctly isolated from Corporate, Cloud, and Trading VDOMs — but that isolation was total. It also couldn't reach the internet, because nothing pointed it toward root, the VDOM with the actual WAN connection.

Junior's instinct was to physically wire the guest switch into root's LAN port — a real, working fix, but it wastes two physical interfaces to connect two virtual firewalls that already live inside the same chassis. That's the whole point of VDOMs undone: you virtualized ten firewalls onto one box specifically to avoid burning ports and buying hardware per zone.

🧠
Mental Note
"He needed a route, not a router." — Two VDOMs on the same physical FortiGate should never need a physical cable to talk to each other.
🪢 Analogy
An inter-VDOM link is like two offices on the same floor of one building deciding to install an internal door between them, instead of one office mailing letters out to the street and the other office receiving them from the street. Same building, same floor — the "internet route" was never necessary for VDOM-to-VDOM traffic in the first place.
Section 2 — Building the Link

One Object, Two Interfaces

An inter-VDOM link is built once, from the Global VDOM context: Global VDOM > Network > Interfaces > Create New > VDOM Link. You are never creating "two links, one per VDOM" — you assign both VDOM sides in the same creation step, and FortiGate generates a paired virtual interface: one half lives in each VDOM.

FortiGate auto-names the two halves by appending 0 and 1 to your chosen base name, because two interfaces cannot share an identical name inside the same device's interface list.

Base NameInterface 0Interface 1
VDOM_LinkVDOM_Link0VDOM_Link1
Gotcha: which side gets "0" vs "1" is not a rule to memorize — it's assigned based on which VDOM you configure first in the creation dialog. Verify placement afterward via each interface's actual VDOM assignment (config system interface or the GUI interface list), never assume.

Section 3 — Routing

Every VDOM Has Its Own Routing Table

Creating the link interface pair is necessary but not sufficient. Each VDOM maintains a fully independent routing table, so each side needs its own route pointing traffic toward its half of the link.

CLI — root's example route (Señor's demo)
config vdom
edit root
config router static
edit 0
    set dst 172.16.30.0 255.255.255.0
    set device "VDOM_Link0"
next
end
CLI — Guest-WiFi's default route (student's build, correct)
config vdom
edit Guest-WiFi
config router static
edit 0
    set dst 0.0.0.0 0.0.0.0
    set device "VDOM_Link1"
next
end

Important nuance caught correctly in-session: root already has a pre-existing default route to the internet via its WAN interface. That route is NOT new — only Guest-WiFi's route toward the link is new. Don't re-write routes that already exist.

Section 4 — Policy

No Such Thing as a Policy That Spans Two VDOMs

Firewall policies live inside a specific VDOM's own policy table — Session 6 already established Policy & Objects doesn't even exist at the Global VDOM level. That means every inter-VDOM data path needs two separate policies, one in each VDOM's own table.

VDOMPolicy srcintfPolicy dstintf
Guest-WiFiGuest VLAN interfaceVDOM_Link1
rootVDOM_Link0WAN / internet-facing interface
Full packet trace (guest device → 8.8.8.8): Guest-WiFi routing table (default route → VDOM_Link1) → Guest-WiFi policy table (Guest VLAN → VDOM_Link1, allowed) → crosses the link → root routing table (existing default route → WAN) → root policy table (VDOM_Link0 → WAN, allowed) → NAT applied on egress at root.
Section 5 — The Core Model

Isolation by Default, Connection by Exception

This is the session's central takeaway. VDOMs do not default to "connected unless a policy blocks it." They default to "physically nonexistent to each other unless an inter-VDOM link explicitly exists."

No link = no interface = no route possible = no path at all. That's stronger isolation than a firewall deny rule, because there's nothing present to even evaluate a policy against.

🧠
Mental Note
Trading VDOM stays safe not because someone remembered to write a deny rule, but because nobody ever built a bridge there. Absence of a link IS the security control.
Watch for this trap: "packets might leak through root and back around externally" describes a completely different risk — physical/L2 switching topology outside the FortiGate — not anything inter-VDOM routing or policy controls. Keep "internal VDOM plane" and "external cabling plane" as two separate isolation boundaries in your head.

Section 6 — Recap

Exam Tips & Senior Takeaways

  • Link creation ≠ traffic flow. Always pair the link with routes AND policies on both sides — three separate requirements, not one.
  • Routing succeeds before policy is even checked. A route with no matching policy silently drops traffic and looks like a policy bug, not a routing bug, until you check which stage actually failed.
  • Verify VDOM assignment, don't assume it. The 0/1 naming split tells you interfaces are unique — it doesn't tell you which VDOM owns which without checking.
  • Extending an inter-VDOM boundary later always means building something new — a fresh link, fresh routes, fresh policies — never "loosening" an existing rule. The absence of a link is not a rule that can accidentally weaken.
Senior engineering habit: when recommending "use what we already have" over a purchase, always name the real constraint underneath the free option (VDOM count ceiling, shared CPU/session-table resources) — don't just call it free.
