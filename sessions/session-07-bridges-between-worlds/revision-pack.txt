LOFINANCE — SESSION 07 REVISION PACK
"Bridges Between Worlds" — Inter-VDOM Links and Routing
=========================================================================

QUICK SUMMARY (one-page recap)
=========================================================================
The new Guest-WiFi VDOM was correctly isolated from Trading, but also totally
isolated from root — no internet access at all. Junior wanted to fix this with
a physical cable (two wasted ports). The real fix: an inter-VDOM link — a
paired virtual interface built once from Global VDOM context, auto-named
base0/base1, one half assigned to each VDOM. The link alone does nothing:
each VDOM still needs its OWN route (independent routing tables) and its OWN
policy (independent policy tables, no such thing as a policy spanning two
VDOMs) before traffic actually flows. Full path for guest→internet: Guest-WiFi
route → Guest-WiFi policy → cross the link → root's pre-existing route → root
policy → NAT → internet. Trading VDOM stays completely unreachable because no
link/route/interface toward it exists anywhere — VDOMs default to "physically
nonexistent to each other" unless deliberately linked, which is stronger
isolation than any firewall deny rule could provide.

=========================================================================
CHEAT SHEET
=========================================================================
GUI PATH TO CREATE A LINK:
  Global VDOM > Network > Interfaces > Create New > VDOM Link
  (created ONCE, both VDOM assignments chosen in the same step)

AUTO-NAMING CONVENTION:
  Base name "X" → X0 (first-assigned VDOM) and X1 (second-assigned VDOM)
  Verify which VDOM owns which via actual interface assignment — don't assume.

THREE REQUIREMENTS FOR TRAFFIC TO FLOW (all required, both sides):
  1. The link interface pair (virtual, no physical cable)
  2. A route in EACH VDOM's own routing table
  3. A policy in EACH VDOM's own policy table (srcintf/dstintf)

CLI — STATIC ROUTE INSIDE A SPECIFIC VDOM:
  config vdom
  edit <vdom-name>
  config router static
  edit 0
      set dst <network> <mask>       (or 0.0.0.0 0.0.0.0 for default)
      set device "<link-interface-name>"
  next
  end

KEY DEFAULT BEHAVIOR:
  VDOMs are isolated by default — "physically nonexistent to each other"
  unless an inter-VDOM link explicitly exists. Not "connected unless
  blocked." This is stronger than a firewall deny rule.

ORDER OF OPERATIONS (every hop):
  Routing decision (pick egress interface) → THEN policy lookup (allow/deny)
  A route with no matching policy still drops traffic — looks like a policy
  bug but the routing succeeded.

TWO DIFFERENT ISOLATION BOUNDARIES — DON'T CONFUSE THEM:
  Internal VDOM isolation  = no link/route/interface inside the FortiGate
  External L2/physical isolation = separate switch/cabling topology outside
                                     the FortiGate entirely, unrelated to VDOMs

=========================================================================
FLASHCARDS
=========================================================================
Q: Why use an inter-VDOM link instead of a physical cable between two VDOMs
   on the same FortiGate?
A: Avoids wasting two physical ports; the two VDOMs already share the same
   physical hardware.

Q: How many times do you create an inter-VDOM link — once, or once per VDOM?
A: Once, from Global VDOM context — both VDOM assignments chosen in the same
   creation step.

Q: A link is named "Finance_Link." What are the two auto-generated interface
   names?
A: Finance_Link0 and Finance_Link1.

Q: How do you determine which VDOM owns Finance_Link0 vs Finance_Link1?
A: Check each interface's actual VDOM assignment (config system interface or
   GUI interface list) — it's not a fixed/memorizable rule.

Q: True or False — creating the inter-VDOM link interface pair is enough for
   traffic to flow between the two VDOMs.
A: False. You also need a route AND a policy on both sides.

Q: Can one firewall policy govern traffic across two different VDOMs?
A: No — every VDOM maintains its own independent policy table; no policy
   spans two VDOMs.

Q: In the packet path Guest-WiFi → root → internet, does root need a NEW
   route added?
A: No — root already has a pre-existing default route to the internet via
   its WAN interface; only Guest-WiFi needs a new route toward the link.

Q: A route exists pointing at the link interface, but no matching policy
   exists. What happens to the traffic, and at which stage does it fail?
A: Traffic is dropped. Routing succeeds (egress interface is chosen); it
   fails at the policy lookup stage.

Q: Does routing or policy lookup happen first on a FortiGate?
A: Routing decides the egress interface first; policy lookup happens after,
   using that determined dstintf.

Q: Trading VDOM has no inter-VDOM link, no route, and no interface pointing
   toward it from any other VDOM. Can Guest-WiFi traffic reach Trading VDOM,
   even accidentally, purely through VDOM configuration?
A: No — there is no interface to route toward and no path exists at all.
   This is stronger isolation than a policy deny rule.

Q: Six months from now, someone wants Trading-to-Guest connectivity for one
   file transfer. What's the single config artifact that must be newly built?
A: A new inter-VDOM link, plus its own dedicated routes and policies on both
   sides — the absence of a link isn't a rule that can be "loosened."

Q: What's the difference between internal VDOM isolation and external L2/
   physical topology isolation?
A: Internal VDOM isolation is about link/route/interface existence inside the
   FortiGate's virtual firewalls. External L2/physical isolation is about
   actual cabling/switch topology outside the FortiGate — a completely
   separate risk that VDOM configuration doesn't control.
