Concept 1 · The Idle Road

BGP Learns Plurally, Installs Singularly

A routing table entry maps one prefix to one best next hop. That isn't a BGP rule — it's how forwarding works. So when two ISPs each advertise 0.0.0.0/0, BGP holds both in its database and installs exactly one.

The loser doesn't vanish. It sits in get router info bgp network indefinitely — perfectly valid, fully formed, never forwarding a packet. This is the same database-versus-routing-table gap from Session 55, now with a bill attached to it.

◆
Mental Note
Junior's error had a precise shape: he assumed learning two paths means using two paths. BGP learns plurally and installs singularly.

The tie-breaker tail. Both carriers are one AS-hop away. No MED, no local-pref, no weight, same origin. Every attribute ties — so selection runs all the way down the list and settles on lowest router-id. Which circuit sits idle is effectively decided by someone else's addressing scheme.

Analogy
A coin flip decided by a stranger.
Penny is paying two invoices. One road is idle because two ISPs happened to number their routers in a particular order. Nothing about the design chose it.
Concept 2 · ECMP

Equal Is Permission. Enable Is Instruction

Two conditions must both hold before FortiGate installs multiple BGP routes for one prefix:

  1. The precondition — every BGP attribute must tie. Equality makes multipath legal.
  2. The instruction — multipath must be explicitly enabled. Without it, identical paths still produce a single installed route.
The most common error on this material is treating ECMP as an automatic consequence of equality. It is not. It is a configured capability. If you remember one thing: equal is permission, enable is instruction.
Enabling multipath
config router bgp
    set ebgp-multipath enable        # peers in a DIFFERENT AS
    set ibgp-multipath enable        # peers in the SAME AS
end

FortiGate will share traffic across up to 10 equal BGP routes. Without ECMP, the route to the router with the lowest BGP router-id wins.

◆
Choosing the knob
You do not choose IBGP or EBGP. Compare your local AS to the neighbour's remote AS — if they match, it's internal and the knob is ibgp-multipath. They tell you which you already have.
Concept 3 · The Honest Part

ECMP Is Capacity, Not Resilience

Penny's follow-up was the right one: "So if Parramatta goes down, we're covered, yes?"

Traffic does survive — the remaining member carries everything once the dead path leaves the routing table. But when does it leave? Only when BGP withdraws the route, which requires the session to tear down, which requires the hold timer to expire. Default: 180 seconds.

DesignDuring the 180s windowDiagnosability
Single path via ISP-1100% of internet flows failObvious. Everything is down.
ECMP across both≈50% of flows fail — hashed onto a holeHarder. The phones only half ring.

Read that table carefully. ECMP did not improve the failure behaviour — it changed who gets hurt. You traded a total outage for a partial one, and the partial outage includes half the traffic that would otherwise have been fine.

Concept 4 · Loopback as BGP Source

It Cannot Go Down Because It Was Never Physically Up

Singapore has two physical circuits to HQ. Junior's proposal — one BGP session per link — is not absurd. It superficially delivers both redundancy and ECMP. It fails for two reasons:

  • Session state is bound to a cable. Fail link A and session A dies with it: routes withdraw, hold timer runs, reconvergence follows. A loopback-sourced session simply reroutes underneath BGP via link B. The TCP connection survives. BGP never notices.
  • Everything doubles. Two neighbours, two sets of routes, two policies — on both sides.
◆
The whole concept in one line
BGP peering is a TCP connection. Bind it to an address that has no cable, and losing a cable becomes a routing event instead of a session event.

Why multihop, when they share a cable? Because the loopbacks are not on the cable. Reaching the peer's loopback requires a routing decision on the peer itself — that is more than one routed hop, and default EBGP TTL is 1. The study guide states it plainly: multihop is required because the loopback interface is not the next hop.

Señor
"The cable is one hop. The loopbacks are not on the cable."
Configuration

The Three Pieces

1 · BGP neighbour — HQ to Singapore
config router bgp
    set as 65001
    set router-id 10.99.50.1
    config neighbor
        edit "172.20.0.1"
            set remote-as 65300
            set update-source "Loopback0"
            set ebgp-enforce-multihop enable
            set description "Javier - SG DC"
        next
    end
end

update-source forces the TCP session to originate from the loopback. Without it FortiGate sources from the egress interface IP, the peer sees a SYN from an address it isn't expecting, and ignores it — straight back to Active. This was on your own stuck-in-Active list in Session 55 as "source address mismatch." It is no longer hypothetical.

2 · Reachability — a route to the peer's loopback
config router static
    edit 0
        set dst 172.20.0.1 255.255.255.255
        set gateway <link-A-next-hop>
        set device "port2"
    next
    edit 0
        set dst 172.20.0.1 255.255.255.255
        set gateway <link-B-next-hop>
        set device "port3"
    next
end

Equal-cost, over both underlays. This is what lets the TCP session reroute beneath BGP when a circuit dies.

3 · Firewall policy — the piece people miss
config firewall policy
    edit 0
        set name "BGP-Loopback-to-SG"
        set srcintf "Loopback0"
        set dstintf "port2"
        set srcaddr "HQ-Loopback0"
        set dstaddr "SG-Loopback"
        set service "BGP"
        set action accept
        set schedule "always"
    next
end
Why this is the nastiest fault in the material. The session is now born on Loopback0 and leaves via port2 — two interfaces. That makes it a firewall policy matter, not a local-in matter. And it fails identically to Session 55's fault: stuck in Active. Same symptom, opposite table. Check local-in here and you have made Junior's mistake in a mirror.

Concept 5 · Scaling

neighbor-group and neighbor-range

When a hub has many spokes peering over two underlays, declaring every neighbour by hand doesn't scale. A neighbour group holds the common settings; a neighbour range says which peer addresses inherit them.

Hub configuration — EF p.146
config router bgp
    set as 65100
    set router-id 172.16.1.254
    set ibgp-multipath enable
    config neighbor-group
        edit "SpokeISP1"
            set interface "ISP1"
            set remote-as 65100
        next
    end
    config neighbor-range
        edit 1
            set prefix 10.1.0.0 255.255.255.0
            set neighbor-group "SpokeISP1"
        next
    end
end

Any peer whose address falls inside 10.1.0.0/24 inherits the group's settings. Note the multipath knob: local AS 65100, remote AS 65100 — same number, therefore internal, therefore ibgp-multipath.

Senior Engineering Insights

What a Senior Engineer Takes From This

  • Equal is permission. Enable is instruction. Never assume a capability is a behaviour.
  • ECMP changes the shape of an outage, not its existence. Partial outages are harder to diagnose than total ones — a design that halves the blast radius also halves the clarity of the alarm.
  • Bind sessions to things that cannot break. A loopback has no cable, no transceiver, no patch panel, and no one to unplug it.
  • Same symptom, opposite table. Two consecutive sessions produced identical symptoms with opposite root causes. Memorise the packet path, not the fix.
  • Tell the customer the residual risk in their own language. Penny was told about the 180 seconds, and her response was to ask what it costs to fix. That is the correct outcome of a good explanation.
Exam Tips

What Fortinet Will Actually Ask

  • "Two equal paths, defaults — how many install?" One. The single most likely place to drop a mark by assuming ECMP is automatic.
  • "Why multihop when they share a cable?" Because the loopback is not the next hop. This is the guide's own wording and near-certain to appear.
  • Config-snippet scenarios will test ebgp- vs ibgp-multipath. Compare the two AS numbers before anything else.
  • The stuck-loopback ladder (policy → update-source → multihop → reachability) is prime scenario material. The distractor to watch for is local-in policy — right answer in Session 55, wrong answer here.
  • neighbor-range defines the IP range whose peers inherit a group's settings — straight recall.
  • ECMP shares up to 10 routes; without it, lowest router-id wins.
