TECHNICAL CONCEPTS COVERED (EF pp. 142-146)
==================================================================
1. WHY BGP INSTALLS ONE PATH BY DEFAULT: a routing table entry maps one prefix to one best
   next hop. BGP learns plurally (RIB / database) and installs singularly (routing table).
   The loser stays in `get router info bgp network` forever, valid, unused. Direct extension
   of the S55 database-vs-routing-table gap.
2. THE TIE-BREAKER TAIL: with two carrier defaults of equal AS-path length, no MED, no
   local-pref, no weight, selection falls all the way to LOWEST ROUTER-ID — i.e. the idle
   circuit is chosen by someone else's addressing. Framed to the student as "a coin flip
   decided by someone else."
3. ECMP: `set ebgp-multipath enable` / `set ibgp-multipath enable` under config router bgp.
   Up to 10 BGP routes shared (per supplement p.344).
   KEY SEQUENCING CORRECTION MADE THIS SESSION: the student said equal attributes mean both
   install "and ECMP kicks in." Corrected to: equal attributes are the PRECONDITION that makes
   multipath legal; the config command is what makes it happen. Both required.
   Señor's phrasing: "EQUAL IS PERMISSION. ENABLE IS INSTRUCTION."
4. ECMP IS NOT REDUNDANCY — the senior-engineering core of the episode. The route only leaves
   the routing table when BGP withdraws it, which requires session teardown, which requires
   hold-timer expiry (default 180s). During those 180s, ECMP hashes ~50% of flows onto a dead
   path. Single-path: 100% broken for 180s. ECMP: ~50% broken for 180s, INCLUDING half of the
   traffic that would otherwise have been fine. Trade a total outage for a partial one, and
   partial outages are harder to diagnose "because the phones only half ring."
   THE STUDENT DERIVED THE ~50% AND THE 180s HIMSELF. This is the strongest reasoning of the
   session and directly sets up S57.
5. LOOPBACK AS BGP SOURCE: loopbacks never go down because they were never physically up.
   Two-per-link sessions bind session state to a cable; a loopback session reroutes underneath
   BGP when a cable fails and BGP never notices — zero withdrawal, zero reconvergence. Also
   halves config/policy/neighbour management versus two sessions.
6. MULTIHOP FOR ADJACENT ROUTERS: even with a single direct cable, loopback-to-loopback is
   >1 routed hop (arriving at the remote loopback is a separate routing decision on the peer),
   so default EBGP TTL 1 kills it. `set ebgp-enforce-multihop enable` required.
   NOTE: the student counted THREE hops; the strict count is TWO transits. He was corrected
   gently — his conclusion and reasoning method were sound, only the arithmetic was off, and
   the only thing TTL cares about is ">1".
7. UPDATE-SOURCE: `set update-source "Loopback0"`. Without it FortiGate sources from the egress
   interface IP; the peer sees an unexpected source address and ignores the SYN → stuck in
   Active. This was ON THE STUDENT'S OWN S55 stuck-in-Active list ("source address mismatch")
   and was explicitly handed back to him as no longer hypothetical.
8. THE FIREWALL POLICY REQUIREMENT (the episode's stretch question, p.144): because the TCP 179
   session is now sourced on Loopback0 and exits a physical port, it crosses TWO INTERFACES and
   therefore requires a FIREWALL POLICY (loopback ↔ physical), not local-in policy. This is the
   MIRROR of Junior's S55 mistake — same symptom (Active), opposite table.
9. REACHABILITY PRECONDITION: a static route or IGP must resolve the remote loopback. The
   student named this first, before the policy answer — correct and necessary, just answering
   the routing question when a policy question was asked.
10. neighbor-group / neighbor-range (pp.145-146): hub-and-spoke overlay scaling. `neighbor-range`
    defines the prefix whose peers inherit the group's settings. `ibgp-multipath` used because
    local AS == remote AS. Taught lightly, flagged as exam-visible, marked for depth when the
    Singapore spokes multiply.

TROUBLESHOOTING APPROACH (as taught this session)
==================================================================
Both peers Established, one path installed  → are attributes truly equal? then: multipath
                                               not enabled, or a tie-breaker fired above
                                               router-id
Loopback peering Active                     → WHERE DOES THE PACKET START, WHERE DOES IT
                                               LEAVE? Two interfaces = firewall policy
Loopback peering Active, policy verified    → what source IP does the peer SEE? (update-source)
Loopback peering Active, source verified    → how many ROUTED HOPS between the loopbacks?
                                               (ebgp-enforce-multihop)
Peer loopback unreachable                   → is there a static/IGP route to it at all?

The ordering question Señor used, and which worked: "Where does the packet start. Where does
it leave. Say both out loud." Saying the two interfaces aloud produced the answer immediately.

JUNIOR'S MISTAKES AND WHAT THEY TAUGHT
==================================================================
1. RIGHT ANSWER, WRONG MECHANISM: "BGP already load balances, it's got two paths, it'll use two
   paths." Correct conclusion (use both links), wrong model. Lesson: BGP LEARNS PLURALLY AND
   INSTALLS SINGULARLY. Being right about the outcome while wrong about the mechanism means you
   cannot troubleshoot it when it doesn't work.
2. "WHY NOT JUST PEER TWICE, ONE SESSION PER LINK?" — the productive wrong answer of the
   episode. It is not absurd; it superficially delivers redundancy AND ECMP. It fails because
   session state is bound to a cable, and because it doubles config on both sides. This is the
   ideal setup for loopback sourcing and should be reused as a teaching device.
