🔧 TROUBLESHOOTING GUIDE · SESSION 31

"Interface Not Found" on Install

How a senior engineer isolates a policy-install interface-resolution failure

The Symptom

Installation aborted: Interface(s) not found on device: port3 — on one or more boxes, while other boxes in the same push install cleanly. The interface "obviously" exists (the internet works on that box).

The junior reflex is to go stare at the box's cabling. Don't. The box working means nothing here — the failure is upstream of the wire.

Questions to Ask First

Before touching anything, answer these three in order:

  • Which boxes fail and which succeed? A model-shaped pattern (all 100Fs fail, the 200F passes) points straight at name-per-model.
  • Does the rule name a literal or a normalized interface? This decides whether you're missing a rule change or a mapping.
  • What does the failing box's device DB actually call its interfaces? That's the ground truth the install validated against.

The Workflow

1
Reframe: it's name resolution, not hardware

Install validates every interface name in the package against the target's device database. No match → abort. The physical link state is irrelevant.

Ask: is the name in the rule present in this box's device-DB interface list?
2
Literal or normalized?

If the rule names a literal (port3), it will only ever install on boxes that happen to use that exact name. Fix = replace with a normalized interface.

If it's already normalized, the abstraction is fine — the mapping is where the fault lives. Go to step 3.

3
Is there a mapping for THIS box?

Check per-device first (it wins), then per-platform. A normalized name with no applicable mapping resolves to nothing → same error.

Where to look
Policy & Objects > Normalized Interfaces > WAN
  Per-Device   → is THIS box listed?  (wins if present)
  Per-Platform → is THIS model listed?
4
Case-check the string — the silent killer

WAN and wan are two different objects. If the rule says WAN and the mapping was built against wan, you get the exact same "not found" while everything looks correct.

Ask: does the string in the rule match the mapping string byte-for-byte, capitals included?
5
Confirm the mapped target exists

The mapping points WAN → port1. Does port1 actually exist in that box's device DB? A mapping to a non-existent target fails just as hard.

Decision Tree

install fails: "interface not found" │ ├─ do OTHER boxes install clean? │ ├─ yes, and failures are model-shaped ──▶ name-per-model. Normalize it. │ └─ no, ALL boxes fail ────────────────▶ suspect the rule OR a global mapping typo │ ├─ rule names a LITERAL? ───────────────────▶ replace with normalized interface │ ├─ rule names a NORMALIZED interface? │ ├─ mapping for this box exists? │ │ ├─ no ──▶ add per-device (this box) or per-platform (its model) │ │ └─ yes ──▶ CASE-CHECK: WAN vs wan (two different objects!) │ │ └─ still failing ──▶ does the mapped target (port1) exist in device DB? │ └─ both per-device & per-platform set? │ └─ per-DEVICE wins. Verify you edited the one that governs.

Common Dead Ends

Staring at the cabling. The link is up; that's not the question. You're debugging a name, not a wire.
Re-pushing unchanged. The install is deterministic — it will fail identically until the name or mapping changes. Re-clicking green is not a diagnostic.
Cloning the package per model. "Works," but destroys single-baseline consistency and drifts the moment someone edits only some copies. Treat as an anti-fix.

Likely Root Causes (ranked)

  • Literal interface name in the rule that doesn't exist on all target models. (Most common.)
  • Missing mapping for a specific box or model.
  • Case mismatch between rule string and mapping string.
  • Mapping target points at an interface not present in that box's device DB.

Exam Traps

"The interface is physically down / missing." Tempting hardware-framed distractor. The correct framing is name resolution against the device DB.
"Per-platform overrides per-device." Backwards. Per-device wins — specific beats general.
"WAN and wan resolve to the same interface." False — case-sensitive, two different objects.
"Defining a normalized interface renames the FortiGate port." False — substitution happens only at install; the box keeps its own name.
"Global ADOM handles the interface translation." No — global ADOM distributes which rules exist; normalized interfaces handle how each rule resolves. Different machinery.
