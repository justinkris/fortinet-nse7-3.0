⏱ 5-MINUTE BITE · SESSION 31

One Name to Rule Them

Normalized interfaces · per-platform & per-device · pp. 35–39

The rule stored a literal string. Give it an abstraction instead, and let the machine translate per box.

The problem in one line

Six baseline rules named the literal port3. The WAN is port3 on the 200F but port1 on the 100Fs and wan1 on the 60E — so install aborted with interface not found. It's name resolution, not hardware. The cable is plugged in; the name isn't in the box's device DB.

The fix: normalized interfaces

Two pieces

1. Abstraction — the rule references WAN (a policy-layer abstract name), not a literal. The rule is now cabling-agnostic.

2. Mapping — a per-box translation table: WAN→port1 here, WAN→port3 there. FMG substitutes at install.

Defining a normalized interface never renames the FortiGate interface. Map lan→port4 and the box still calls it port4.

Two mappings

Per-platform

Binds to a model. "Consistency." Define once, whole class inherits. New identical box = auto-covered. Case-sensitive.

Per-device

Binds to one box. "Individual needs." For the box that breaks ranks with its own model — not merely an odd box.

Precedence

Per-device wins. Specific beats general — same shape as longest-prefix routing. If both exist for a box, the narrower statement governs.

The silent killer

WAN ≠ wan. Case-sensitive. A rule naming WAN with a mapping defined against wan gives the same "interface not found" — except everything looks correct. One shift key.

The payoff

Onboard a new differently-cabled company? Rules stay frozen — add one mapping per new layout. The audited baseline is provably unchanged; only the translation table grows. Consistency with less human error = money made, an afternoon not a rewrite.

Whole loop: literal → normalized abstraction → per-platform/per-device mapping → case-exact → resolves per box.
