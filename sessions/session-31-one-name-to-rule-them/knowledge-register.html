NORMALIZED INTERFACES (abstraction + mapping)   <- THIS SESSION, CORE
  Confidence: ★★★★☆ (new item; high because self-derived, not just taught)
  Mental model: STRONG and self-built. Owns: rule references an ABSTRACT name, not a literal;
    the intelligence is the PER-BOX TRANSLATION, not the name; defining a normalized interface
    does NOT rename the FortiGate (substitution at install only); normalized name lives in POLICY
    layer, real port in DEVICE layer. Grasped that install validates package names against the
    target DEVICE DB and aborts on no-match ("interface not found") — a name-resolution failure.
  Strong areas: the two-piece structure (abstraction + translation); cabling-agnostic rules;
    layer placement; onboarding payoff (rules frozen, add mappings only).
  Common mistakes THIS session:
    (a) Q1 framing slip: "interface may not exist" (hardware) vs "literal name not in device DB"
        (resolution). Reframe drill: an install failure with the cable plugged in is ALWAYS
        name resolution first.
    (b) "wan refers to an actual interface" on the case trap — conflated a normalized name with a
        device name because they can SHARE a string. Corrected: the string doesn't tell you the
        layer; the name can be an abstraction.
  Weak areas: precision of the failure-mode sentence under first pass.
  Recommended reinforcement: one cold "what layer / what gets renamed / what does install check"
    triple in S33+; retire to spot-check if clean.
  Last reviewed: S31.

PER-PLATFORM vs PER-DEVICE MAPPING   <- THIS SESSION, CORE
  Confidence: ★★★★☆
  Mental model: STRONG on identity and use-case (per-platform = whole model / "consistency";
    per-device = one box / "individual needs"). Volunteered the per-device escape hatch unprompted
    ("if one needs to change you could use per-device"). PRECEDENCE mastered by derivation:
    per-device wins, specific-beats-general, self-linked to longest-prefix routing when offered.
  Common mistakes THIS session:
    (a) MINIMUM-SET arithmetic: answered "1 per-platform, 2 per-device" for the 8-box / 3-model
        fleet. Missed that the 200F and 60E are each a DIFFERENT PLATFORM and can ALSO be covered
        per-platform (fleet-of-one). True minimum = 3 per-platform, 0 per-device. He conflated
        "odd box" with "needs per-device." KEY CORRECTION LANDED: per-device is for a box that
        CONTRADICTS its own platform, NOT merely a box that's the only one of its kind.
  Weak areas: the "one representative of its model still uses per-platform" nuance — verify once
    more with a fresh fleet-count scenario in S32/S33.
  Recommended reinforcement: a mixed-fleet counting scenario where TWO boxes share a platform but
    disagree on cabling (forces genuine per-device) vs a solo-model box (per-platform suffices).
    That contrast is the exact discrimination he needs to solidify.
  Last reviewed: S31.
