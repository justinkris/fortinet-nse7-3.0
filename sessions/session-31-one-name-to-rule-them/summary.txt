==================================================================
BUSINESS PROBLEM
==================================================================
One policy package (Penny's six audited baseline rules) must install across a heterogeneous
fleet whose WAN interface has a DIFFERENT physical name on every model (port1 / port3 / wan1),
WITHOUT cloning the package (which would re-introduce drift and destroy Cipher's certifiable
single-baseline guarantee). Deadline: Friday.

==================================================================
TECHNICAL CONCEPTS COVERED (pp. 35-39)
==================================================================
- NORMALIZED INTERFACES: abstract interface objects that live in the POLICY layer and are
  referenced by rules INSTEAD of literal device-interface names. A rule that names a normalized
  interface is cabling-agnostic.
- FMG validates every interface name in a package against each TARGET's DEVICE DB before
  install; a name with no match aborts the install ("interface not found"). This is a NAME-
  RESOLUTION failure, not a hardware failure.
- Normalized interface != device interface even when they share a name (e.g. policy-layer
  "port1" is an abstraction; device-DB "port1" is the metal). Defining a normalized interface
  does NOT rename anything on the FortiGate — FMG only substitutes the mapped name at install.
- TWO MAPPING TYPES:
    * PER-PLATFORM ("consistency"): binds normalized name -> real interface for a whole DEVICE
      MODEL. Define once, every box of that model inherits; new identical boxes are auto-covered.
      CASE-SENSITIVE.
    * PER-DEVICE ("individual needs"): binds normalized name -> real interface for one SPECIFIC
      box. The exception handler.
- PRECEDENCE: PER-DEVICE WINS over per-platform when both exist for the same box. General
  principle: SPECIFIC OVERRIDES GENERAL (same shape as longest-prefix-match in routing;
  per-device is the /32 of interface mapping).
- CASE SENSITIVITY: WAN != wan. A rule referencing "WAN" with mappings defined against "wan"
  yields the SAME "interface not found" error even though everything LOOKS correct. The guide
  flags this specifically for per-platform (case-sensitive matching).
- Guide guidance: keep interface names CONSISTENT between device layer and policy layer to avoid
  confusion at scale (a normalized name colliding with a literal device name confuses the human,
  not the machine).
- Onboarding payoff: a new differently-cabled company/box requires ZERO rule changes — you only
  ADD one mapping per new interface layout. The audited rules stay frozen; only the translation
  table grows. Distinguish: GLOBAL ADOM distributes WHICH rules exist everywhere (assign/install
  from S30); NORMALIZED INTERFACES resolve HOW each rule finds its interface on each box. Both
  work together but are different machinery — an exam distractor swaps them.

==================================================================
KEY CLI / OPERATIONS DISCUSSED
==================================================================
GUI-driven (Central Management):
  Policy & Objects > Normalized Interfaces > (create abstract iface, e.g. WAN)
    -> per-platform mapping (bind to a device model)
    -> per-device mapping (bind to one specific managed FortiGate)
  Device Manager > Device & Groups > Managed FortiGate > FortiGate > Network > Interfaces
    (the DEVICE-layer real interfaces the normalized names map to)
  Install preview shows the SUBSTITUTED (real) interface name; approve at preview.
Error to recognise on install: "Interface(s) not found on device: <name>" = normalized name has
  no mapping for this box, OR a case mismatch (WAN vs wan), OR a literal name that doesn't exist
  on this model.

==================================================================
TROUBLESHOOTING APPROACH TAUGHT ("interface not found on install")
==================================================================
1) It's NAME RESOLUTION, not hardware. Cable-plugged-in is irrelevant.
2) Literal or normalized? Literal -> only installs where that exact name exists.
3) Normalized -> is there a mapping for THIS box? Check PER-DEVICE first (it wins), then
   per-platform.
4) CASE-CHECK the string. WAN != wan. Per-platform matching is case-sensitive — the silent
   killer that makes everything "look" right.
5) Confirm the mapped target interface actually exists in that box's device DB.

==================================================================
CONFIGURATION DECISIONS
==================================================================
- Replace the literal "port3" in the six baseline rules with a normalized interface named WAN.
- Minimum mapping set for the 8-box fleet: THREE per-platform mappings — 100F->port1,
  200F->port3, 60E->wan1 — covers all eight (six 100Fs inherit from one). Per-device reserved
  ONLY for a box that breaks ranks with its own model (e.g. an emergency-recabled 100F).
- Case discipline enforced: rule string and mapping string must match exactly, capitals included.

EXAM-RELEVANT NOTES
==================================================================
- Install validates package interface names against each target's DEVICE DB; no match = abort
  ("interface not found"). It's name resolution, not hardware. (Q1)
- Defining a normalized interface does NOT rename the FortiGate interface; FMG substitutes the
  mapped name at install only. (Q2)
- Per-platform = consistency across a device MODEL; per-device = individual box override. (Q3, Q8)
- PER-DEVICE OVERRIDES PER-PLATFORM. Specific beats general. (Q4)
- Normalized interfaces are CASE-SENSITIVE; WAN != wan; per-platform matching case-sensitive. (Q5, Q9)
- Normalized interface lives in POLICY layer; the real port lives in the DEVICE layer. (Q6)
- Onboarding differently-cabled boxes = rules unchanged, ADD mappings only. (Q7)
- Not all objects support per-device mapping (web filter profiles do NOT — foreshadows S32/
  dynamic objects). Keep this handy: if a config that doesn't support per-device is imported
  with per-box differences, FMG is forced to choose between values.
