One Name to Rule Them

Normalized interfaces · per-platform & per-device mapping · Study Guide 02 Central Management, pp. 35–39

A rule cannot point at a name that means something different on every machine — so give it a name that means the same thing everywhere, and let the machine translate.
Section 1 · Story Recap

The Monolingual Package

Friday, Penny's deadline. Junior — much improved — rolled Penny's six audited baseline rules out to the fleet. Three installs went green. Then VL-BRANCH-1, VL-BRANCH-2 and the Parramatta 60E all failed with the same error:

Installation log
Interface(s) not found on device: port3
Policy package installation aborted.

Every box has a WAN interface — the internet cable is plugged in everywhere. But the models name it differently: the Vaultline 100Fs call it port1, LoFinance HQ's 200F calls it port3, and the Parramatta 60E calls it wan1. The six rules had been authored naming the literal string port3, so they installed only on the one box that happened to call its WAN port3.

Junior's fix: clone the package into three per-model copies. Cipher's fix: absolutely not — that fractures the certified single baseline she and Penny promised the auditors. Señor's diagnosis: "The package is not broken. The package is monolingual."

Section 2 · Learning Objectives

What You Should Own After This

  • Explain why an install fails with interface not found even when the interface physically exists — and why it's a name-resolution failure, not hardware.
  • Describe a normalized interface: an abstract, policy-layer interface name a rule references instead of a literal port.
  • Distinguish per-platform mapping (whole model / consistency) from per-device mapping (one box / individual needs).
  • State the precedence rule — per-device wins — and the general principle behind it.
  • Avoid the case-sensitivity trap: WAN ≠ wan.
  • Know that defining a normalized interface never renames the FortiGate interface.
Section 3 · The Core Idea

Store a Role, Not a String

When you type port3 into a rule's incoming-interface field, the policy package stores a literal string. It has no idea that port3 means "the WAN port" or "the one with the internet cable." Before installing, FortiManager validates every interface name in the package against each target's device database. Match → push. No match → abort.

Think First
If the rule must stop containing a literal string, what should it reference instead — and what extra information must FMG then hold to turn that reference back into port1 here and port3 there?

The answer has two halves, and both are essential.

Half one — the abstraction. The rule references a normalized interface: an abstract name (say WAN) that lives in the policy layer. The rule commits to a role, not to the metal. Every box can host "the WAN port" regardless of what the copper is called underneath.

Half two — the mapping. An abstract name with no translation matches nothing. The intelligence isn't the name; it's the per-box translation table FMG consults at install time: "on this box, WAN = port1; on that box, WAN = port3."

🧠
Mental Note
The name is not the intelligence. The per-box translation is. A normalized interface = one abstract reference + a mapping that resolves it differently on each machine.
Critical distinction: a normalized interface is not the same object as a device interface, even when they share a name. Policy-layer port1 is an abstraction; device-DB port1 is the physical port. Defining or mapping a normalized interface never renames the FortiGate interface — FMG just substitutes the mapped name when the rule installs. Map lan to port4 and the box still calls it port4.
Section 4 · The Two Mappings

Per-Platform vs Per-Device

Per-platform lets you say once: "On the FortiGate-100F platform, WAN = port1" — and all six identical 100Fs inherit it. Add a seventh 100F next month and it's already covered. That's the consistency-at-scale win.

Per-device handles the exception: "On this specific 60E, WAN = wan1." You reach for it when a box contradicts its own platform — for example, an emergency-recabled 100F whose WAN is now physically on port5 while every other 100F still uses port1.

🧠
Mental Note
Per-platform can cover even a single box (a fleet-of-one). Per-device isn't "for the odd box" — it's specifically for the box that breaks ranks with its own model.
Section 5 · Precedence

Specific Beats General

If a box has both a per-platform answer (WAN→port1) and a per-device answer (WAN→port5) for the same normalized interface, FortiManager installs the per-device value — port5.

The principle: the more specific mapping wins. A per-device mapping is a statement about this one box; per-platform is a statement about the whole class. The whole reason to write a per-device override is to say "this box is an exception — ignore the class rule here." If generality won, per-device could never override anything, which would make the feature pointless. So it must win.

🔗 Cross-feature pattern
This is the same shape as longest-prefix-match in routing: the most specific route wins over the general one. Per-device is the /32 of interface mapping. When you meet a Fortinet precedence question, ask "which statement is narrower?" — the narrower one almost always governs.
Section 6 · The Silent Killer

Case Sensitivity

Normalized interface names are case-sensitive. WAN and wan are two different objects. If your rules reference WAN but the mappings were defined against wan, the rules point at an object with no mapping at all, and install produces:

Same error, nastier cause
Interface(s) not found on device

This is the exact same error you started with — except now everything looks correct. The name is there, the mapping is there, the cable is plugged in. The only difference is one shift key. The study guide flags this specifically for per-platform matching, which is case-sensitive.

Instruction discipline: the string in the rule and the string in the mapping must match exactly, capitals included. WAN is not wan, and the machine will not forgive the difference.
Naming hygiene: the guide also recommends keeping interface names consistent between the device layer and the policy layer. A normalized name that collides with a literal device name confuses you (which one does this rule mean?), not the machine — but at fleet scale that confusion causes real outages.
Section 7 · Configuration Walkthrough

Fixing Penny's Friday

Section 8 · Troubleshooting Workflow

"Interface Not Found" on Install

  1. It's name resolution, not hardware. The cable being plugged in tells you nothing. Stop looking at the port.
  2. Literal or normalized? If the rule names a literal, it only installs where that exact name exists — replace it with a normalized interface.
  3. Is there a mapping for THIS box? Check per-device first (it wins), then per-platform.
  4. Case-check the string. WAN ≠ wan. This is the failure that looks correct.
  5. Does the mapped target exist? Confirm the real interface (e.g. port1) actually appears in that box's device DB.
Section 9 · Common Mistakes

Where People Faceplant

  • Cloning the package per model (Junior's plan). "Solves" the symptom by destroying the single-baseline consistency guarantee; the copies drift the moment someone edits only two.
  • Assuming per-device is "for odd boxes." It's for boxes that contradict their own platform. A solo-model box is fine on per-platform.
  • Case mismatch between rule and mapping — the invisible outage.
  • Believing a normalized interface renames the FortiGate. It never does.
  • Blurring global ADOM with normalized interfaces. Global ADOM = which rules exist everywhere. Normalized interfaces = how each rule resolves its interface on each box. Different machinery; an exam distractor swaps them.
Section 10 · Senior Insight

The Onboarding Payoff

Penny asks: "Next quarter we bring on a third company, cabled God-knows-how. How much of my six-rule baseline changes?"

Answer: none. The rules reference an abstraction, so a new box cabled any way doesn't contradict anything the rule said. The rules stay frozen. You add one mapping per new interface layout — an afternoon, not a rewrite. The audited baseline is provably unchanged because the only thing that changed is a translation table.

💰
Business framing
"Consistency across the fleet lets us make money with less human error — and onboarding a new company costs an afternoon, not a rewrite, with the certified baseline untouched." Lead with the money; the mechanism follows.
Section 11 · Key Takeaways

The Whole Loop

Literal name → normalized abstraction → per-platform / per-device mapping → case-exact → install resolves it per box. Miss any link and you get "interface not found."
  • A rule stores a literal string; install validates it against each target's device DB.
  • Normalized interface = abstract policy-layer name + per-box translation.
  • Per-platform = consistency (whole model); per-device = individual needs (one box).
  • Per-device wins — specific beats general (longest-prefix shape).
  • WAN ≠ wan; matching is case-sensitive.
  • Defining a normalized interface never renames the FortiGate.
  • New differently-cabled boxes: rules frozen, add mappings only.
