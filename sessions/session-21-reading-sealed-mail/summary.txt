TECHNICAL CONCEPTS COVERED
--------------------------
1. CA-distribution failure fingerprint: ALL sites warning = missing CA in client
   trust stores; forging engine itself working perfectly (issuer field = evidence).
2. Fall back vs roll back: retreat to last known-good LAYER (certificate
   inspection), never to zero.
3. EXEMPTIONS: exempted traffic gets NO inspection of any depth — FortiGate steps
   out entirely; client negotiates directly with the real server. Built-in profiles
   read-only → clone. Fortinet ships Finance/Health/Privacy categories pre-exempt.
   THREE reasons banking stays exempt permanently: (a) compliance/data custody
   (plaintext financial creds must not exist mid-path — student got this),
   (b) protecting administrators from the CAPABILITY itself ("the most senior thing
   an engineer can do is refuse a capability" — student got the human version),
   (c) CERTIFICATE PINNING — student produced the NAME himself ("something I've
   heard before") but placed the pin on the server; corrected: the pin lives IN THE
   APP, compiled-in; app asks "is this literally the cert/key I was born knowing,"
   not "do I trust the issuer." Trusted ≠ pinned. Junior's line: "Browser asks who
   signed it. App asks is it literally you."
4. INBOUND vs OUTBOUND full inspection — the symmetry: full inspection always
   requires possessing a private key; the only question is WHOSE. Outbound: can't
   own the far end → forge with re-signing CA → all client pain. Inbound: we OWN
   the server → load real cert + private key → zero client impact, no trust store,
   no pinning problem (the pinned cert IS the real one). Student derived the
   direction logic himself ("we don't control their devices... we need the keys of
   the server"). Profile scope dropdown = this table: Multiple Clients Connecting
   to Multiple Servers (outbound/forging) vs Protecting SSL Server (inbound/real
   key). This answered the student's own pre-reading Q1.
5. PROTOCOL PORT MAPPING: proxy engine dispatches sessions to protocol inspectors
   BY DESTINATION PORT before any decryption ("pigeonholes"/"costumes"). SMTPS
   default = 587 only; 10587 is never claimed by any inspector even with full
   inspection enabled. Fix: config <protocol> / set ports 587 10587. THREE
   preconditions (exam multi-select): full SSL inspection + Multiple Clients
   Connecting to Multiple Servers + PROXY-BASED mode. Flow-based inspects ALL ports
   regardless — port mapping is irrelevant there and the 10587 evasion FAILS
   against flow.
6. TLS VERSION CONTROL: config firewall ssl-ssh-profile / config https /
   set unsupported-ssl-version block / set min-allowed-ssl-version tls-1.2.
   CLI-ONLY (not in GUI — exam bait). Applies per-protocol: also SMTPS, POP3S,
   IMAPS, FTPS. Blocks at HANDSHAKE (before cert exchange — why the cert-inspection
   fallback didn't fix the scanners). Negotiation logic: server offering 1.1+1.2 →
   ALLOWED (negotiates 1.2, meets minimum). Legacy-device judgement answer: never
   lower the global minimum; scoped exception policy pinned at BOTH ends (source =
   scanner subnet, dest = freight site only) + logged + expiry/replacement date.
   "An exception without an expiry is not an exception. It is the new normal."

MENTAL MODELS / ANALOGIES (new)
-------------------------------
- BLAST RADIUS IS DIAGNOSIS (three radii: all sites / one site / one device class).
- FALL BACK ≠ ROLL BACK (retreat one layer, not to zero).
- THE WAX SEAL (pinning): "some recipients check the wax seal" — app born knowing
  the exact seal; trust store irrelevant.
- WHOSE PRIVATE KEY? — the single question that splits outbound (forge) from
  inbound (real key). The 2x4 ownership table (own clients? own server?).
- PIGEONHOLES / COSTUMES: proxy must choose a protocol costume before inspecting
  (dispatch by port); flow watches through the window, needs no costume, sees all
  ports. "The costume room is proxy. The window is flow." — WELD LABEL TO MECHANISM.
- "Just for" = the two most expensive words in network security (Junior).
- Exception shaping: "shaped exactly like the hole it fills, and no bigger."
- New SNI/SAN decode key (after Q2 miss): SAN = Single cert, All the Names ('Single'
  hides in SAN's letters); SNI = the Name Indicates which cert → MANY certs.
