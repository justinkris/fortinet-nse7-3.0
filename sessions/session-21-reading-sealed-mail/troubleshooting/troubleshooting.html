When Full Inspection Breaks Things

The Master Principle: Blast Radius Is Diagnosis

Certificate and TLS symptoms carry their diagnosis in their scope. Before touching any config, classify the blast radius:

Who is affected?Most likely layerFirst evidence to check
Every site, every user, every browserClient trust store (inspection CA missing)Cert viewer: issuer field + issue time
One website, all usersThat site's certificate or its chainSame site from outside the network
One device class onlyDevice capability (TLS version, platform trust store, pinning)Same site from a modern device
Browser works, native app failsCertificate pinning in the appConfirm CA is OS-trusted; app still fails → pin
One protocol on a weird port evades inspectionProxy dispatch gap — no port mappingTraffic logs by destination port

Questions a Senior Asks First

  1. What changed, and when — does the failure timestamp match a change window?
  2. Who exactly is affected, and who is not? (The unaffected population defines the boundary.)
  3. What certificate is actually being presented? (Look — don't guess.)
  4. Does the failure happen before or after certificate exchange? (Handshake-level version blocks fail before certs — a cert fix cannot help.)
  5. Is the business impact time-critical enough to need a surgical bridge (exemption / fallback) before the root-cause fix?

Decision Tree — Certificate Warnings After Go-Live

Warnings appear →
├─ All sites? → CA not in client trust stores → fall back to certificate-inspection → push CA (GPO/MDM) → flip forward.
├─ One site? → inspect that site's real cert (expired / name mismatch / bad chain) → not your inspection deployment.
├─ One device class? → check TLS version support → if legacy-only: scoped exception policy (both ends pinned + expiry), never lower the global minimum.
└─ Browser OK, app broken? → certificate pinning → permanent exemption for that destination. No trust-store fix exists.

Common Dead Ends

  • Rebuilding the profile ("maybe it's corrupted") — the profile is doing exactly what it was told; the fault is order-of-operations.
  • Full rollback — throws away certificate inspection, which was never broken.
  • Pushing the CA to fix a pinned app — the app doesn't consult the trust store.
  • Pushing the CA to fix version-blocked devices — the handshake dies before certificates exist.
  • Enabling full inspection to catch traffic on non-standard ports — necessary but not sufficient in proxy mode; the dispatcher needs the port mapped.
  • Hunting the GUI for min-allowed-ssl-version — CLI-only.

Likely Root Causes, Ranked

  1. CA generated but not distributed before enabling full inspection (deployment order).
  2. Pinned applications inside non-exempt categories.
  3. Version minimum blocking legacy clients at the handshake.
  4. Proxy-mode dispatch gaps on non-standard ports (deliberate evasion or misconfigured apps).
  5. Missing/wrong server certificate + key in a Protecting SSL Server profile (inbound).

Exam Traps

  • "Full inspection is enabled, so all ports are inspected" — false in proxy mode without port mapping; true in flow mode.
  • Port-mapping preconditions multi-select: full + Multiple-Clients-to-Multiple-Servers + proxy — the "flow-based" option is the planted wrong answer.
  • Version stems: server offering 1.1 and 1.2 against a 1.2 minimum is allowed — the exam wants you to see only the low number.
  • Cert-inspection HTTPS protection = Limited — reject both "No protection" and "only with IPS."
  • Read the stem's polarity twice: is it asking whether the attack succeeds or the defence succeeds?
The recovery pattern to memorise: classify blast radius → verify with the presented certificate → bridge the business (fall back + exempt) → fix root cause (CA push) → flip forward in the correct order → confirm both recovery and root cause.
