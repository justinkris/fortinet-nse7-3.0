==================================================================
TECHNICAL CONCEPTS COVERED (pp. 40-43)
==================================================================
- DYNAMIC OBJECTS (per-device object mapping): one policy-layer object name (e.g. Branch_LAN)
  carries a DIFFERENT value per managed device. Generic object in the package; per-device values
  stored IN FMG attached to each device entry; substitution AT INSTALL. The FortiGate receives
  only the resolved value — it never sees the generic name or knows the mapping exists ("the box
  gets the answer, not the question").
- METADATA VARIABLES: created/edited at Policy & Objects > Advanced > Metadata Variables (policy
  layer). GUI icon = MAGNIFYING GLASS with $ symbol. In supported fields, TYPE $ to trigger the
  variable popup (+ to create new). Each variable = DEFAULT value + optional PER-DEVICE overrides.
  Per-device value WINS over default (specific beats general — same law as S31 precedence).
- Variables' extra power over dynamic objects: usable in PROVISIONING TEMPLATES and CLI TEMPLATES
  as well as the policy layer — can make ANY templated config line vary per box, not just an
  address value. (CLI template + $ variable can overwrite both layers; assigning a template flags
  device-layer AND provisioning-template status Modified — triangle/exclamation.)
- UNRESOLVED VARIABLE AT INSTALL: no default + no per-device value = install FAILS for that
  device, nothing pushed. Consolidated law: "install is a validation gate, not a delivery van" —
  missing interface mapping (S31), case-mismatched mapping, and unfilled variable are three
  costumes of ONE machine: unresolvable reference at install = abort, fail loudly.
- Empty default as a DESIGN FEATURE: a tripwire forcing conscious per-branch fill-in at
  onboarding.
- WHEN NOT TO ABSTRACT: a value identical everywhere (HQ DNS) belongs in a plain STATIC object —
  the package is already the single source of truth for invariant values; editing it once +
  reinstall propagates everywhere. A variable there adds zero capability and real cost
  (indirection at 2am, unfilled-value risk, invitation for a rogue per-device override that
  silently breaks the invariant). "Variables are for values that vary. Indirection is a loan;
  variance pays the interest."
- NOT ALL OBJECTS SUPPORT PER-DEVICE MAPPING. Supported (memorise-worthy): addresses, VIPs & VIP
  groups, IPv4/IPv6 pools, virtual servers, ZTNA servers, user groups, LDAP/RADIUS/TACACS+
  servers, FortiNAC/FSSO agent, metadata variables, dynamic local certificate, dynamic VPN
  tunnel. NOT supported: ALL security profiles (web filter, AV, IPS, app control, DNS filter,
  etc.), services & groups, schedules, traffic shapers, internet service, and more. When
  per-device is unavailable and per-box differences are needed → UNIQUE NAMES per variation
  (back to copies — the disease, accepted only where the cure is locked out).
- Import conflict: importing a per-box-different object of an UNSUPPORTED type forces FMG to
  choose between FortiGate value and its own → conflict; fix = uniquely named object, re-import.

  lines, install = opening night when placeholder becomes performance.

KEY GUI / OPERATIONS DISCUSSED
==================================================================
- Policy & Objects > Advanced > Metadata Variables (create/edit; per-device mapping supported).
- Policy & Objects > Firewall Objects > Address → per-device mapping on an address object.
- Type $ in a supported field → variable popup; + to create inline.
- Magnifying-glass-$ icon identifies variables in the GUI.
- Install preview shows the SUBSTITUTED per-device truth — "the preview is where secrets confess."
- Error family: unresolvable reference at install aborts that device's install.

==================================================================
TROUBLESHOOTING APPROACH TAUGHT
==================================================================
1) Resolution first, not hardware/policy logic.
2) Identify reference type: literal / normalized / dynamic object / metadata variable.
3) Does a PER-DEVICE value exist for THIS box? Check specific entry before default.
4) Case-check strings (WAN != wan).
5) Read the install preview — it shows the substituted truth.
