Objects with Secrets

Dynamic objects and metadata variables — one object, many truths. Thirty branches, one certified package.

Learning Objectives

Core Concept 1

Dynamic objects — one object, many truths

Friday's machine had three parts: a generic name in the rule, a per-box translation stored in FortiManager, and a substitution at install. Dynamic objects are the same machine pointed at objects instead of interfaces.

  • The generic object (e.g. Branch_LAN) lives in the policy package and is what every rule references.
  • Each device's real value lives in FMG, attached to that device's entry — enable per-device mapping on the object and give Branch1 → 172.16.1.0/24, Branch2 → 192.168.0.0/24, and so on.
  • Substitution happens at install. The install preview shows the resolved value.
The box gets the answer, not the question. The FortiGate receives only 172.16.1.0/24 in its config. It never sees the generic name, never stores a mapping, never knows the secret exists. All the intelligence lives in FortiManager.
Analogy — The Theatre Production

The policy package is the script — one copy, certified, frozen. Each device's per-device value is that actor's own lines, filled in privately. Install is opening night: the moment the placeholder becomes a performance. Nobody rewrites the script per actor; they only fill in the lines.

Core Concept 2

Metadata variables — $(the general weapon)

  • Created and edited at Policy & Objects > Advanced > Metadata Variables (policy layer).
  • Marked in the GUI by a magnifying-glass icon with a $ symbol.
  • In a supported field, type $ to open the variable popup; click + to create one inline.
  • Each variable holds a default value plus optional per-device overrides. The per-device value wins — specific beats general, same law as per-device interface mapping.
  • Reach: variables work in the policy layer and in provisioning / CLI templates. A dynamic object can only make an address vary; a variable can make any templated line of configuration vary per box. A CLI template using a $ variable can overwrite settings in both layers — assigning it flags the device layer and provisioning-template status as Modified (exclamation-in-triangle), prompting a push.
🧠Dynamic object vs variable: same soul, different reach. If the varying thing is an address in a policy, either works. If the varying thing is a line of config in a template, only the variable can do it.
Core Concept 3

The unresolved variable — fail loudly

Onboard Branch 17 in a hurry, forget to fill in its value, and install… fails for that device. FMG will not push an empty or unresolvable value. Nothing lands.

Install is a validation gate, not a delivery van. Fail loudly beats succeed wrongly. A certified rule silently matching nothing would be far worse than a red install.

Design consequence: a variable with no default is a feature for per-branch values — the empty default is a tripwire forcing every onboarding to consciously supply the value.

One law, three costumes

  • Missing interface mapping (S31) → abort: Interface(s) not found on device.
  • Case-mismatched mapping (WAN ≠ wan) → same abort — everything looks configured.
  • Unfilled metadata variable → install fails for that device.

One machine: install validates every reference against what FMG knows about the target box, and aborts rather than pushing a guess.

Core Concept 4

When not to abstract

Penny wanted HQ DNS (10.10.10.53, identical on all thirty branches, forever) made into $(hq_dns). Wrong tool. A plain static object in the package already gives "edit once, everywhere gets it on reinstall" — the package is the single source of truth for invariant values.

The variable adds zero capability and real cost: one more indirection to chase at 2 a.m., one more unfilled-value install risk, and an open invitation for someone to add a per-device override "just for one branch" — silently breaking the invariant the design was supposed to guarantee.

Variables are for values that vary. Indirection is a loan; variance pays the interest. If nothing varies, the abstraction is pure cost.
Core Concept 5

Not everything can carry a secret

Per-device mapping is only available for certain object types. When it isn't, and per-box differences are required, the name itself must carry the difference — unique names per variation. You are back to copies, accepted only where the cure is locked out.

Addresses; VIPs & VIP groupsALL security profiles — web filter, antivirus, IPS, app control, DNS filter, email filter…
IPv4 / IPv6 pools; virtual servers; ZTNA serversServices & service groups; schedules
User groups; LDAP / RADIUS / TACACS+ serversTraffic shapers; internet service; IP pool groups

Import conflict trap: importing a per-box-different configuration of an unsupported type forces FMG to choose between the FortiGate's value and its own. Resolution: create a uniquely named object on the FortiGate and import it again.

Troubleshooting Workflow

When an install fails or a branch misbehaves

  1. Resolution first — not hardware, not policy logic.
  2. Identify the reference type: literal / normalized interface / dynamic object / metadata variable. Each resolves differently.
  3. Does a per-device value exist for THIS box? Check the specific entry before trusting the default.
  4. Case-check every string. WAN ≠ wan — two different objects.
  5. Read the install preview — it shows the substituted truth. The preview is where secrets confess.
Common Mistakes

Where people (and Junior) fall

  • Thirty copies. Works day one; drifts every day after. Every future change ×30; first missed edit = silent drift; the auditable single baseline becomes thirty separate claims.
  • Putting the real value in the shared object. One object, one value, thirty boxes — the copies problem rebuilt inside a single object. The value goes in the per-device mapping.
  • Abstracting the invariant. $(hq_dns) with identical entries everywhere — the new-hammer mistake. Static object.
  • Per-device mapping a security profile. Not supported. Unique names are the only move — a favourite exam distractor.
  • Thinking mappings get pushed. Mappings and variables are FMG-side translation consulted at install. They never appear on the FortiGate — only resolved values do.
Senior Insights

The judgement layer

  • Day-one cost vs every-day-after cost. Junior optimises the first day; you price the next eighteen months.
  • Abstraction is bought, not free. Add indirection only where variance pays for it.
  • Empty defaults are tripwires — a deliberate onboarding forcing-function, not sloppiness.
  • Specific beats general — third appearance of the master pattern: longest-prefix routing → per-device interface mapping → per-device variable value.
  • Compress features into laws. Three error messages became one validation-gate principle. Principles predict; lists just recall.
Exam Tips

How the exam probes this

  • Same-named address, different value per box → per-device mapping (not address groups, not global objects).
  • Metadata variables: Policy & Objects > Advanced; magnifying-glass-$ icon; type $ to invoke; per-device beats default.
  • Variables reach into provisioning/CLI templates — their edge over dynamic objects. Expect a question on which tool covers templates.
  • Distractor: "per-device map the web filter profile." Locked. Unique names.
  • Unresolved variable at install → install fails for that device, nothing pushed.
  • Substitution timing: at install, inside FMG; the FortiGate stores only resolved values.
