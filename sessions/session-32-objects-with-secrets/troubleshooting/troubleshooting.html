Secrets That Don't Resolve

How a senior engineer hunts a per-device value that never arrived — thinking first, clicking second.

Symptoms you'll actually see

  • Policy package install fails for one or more devices — others succeed with the identical package.
  • Error family: interface/object not found, unresolved variable, or a device-specific validation failure at install.
  • A branch installs fine but a baseline rule matches no traffic — the address resolved to the wrong subnet.
  • A value that "should be the same everywhere" differs on one box — an invariant silently broken.

Questions to ask before touching anything

  1. Which devices fail, which succeed? Same package, mixed results = per-device resolution problem, not a package problem.
  2. What kind of reference is failing? Literal string / normalized interface / dynamic object / metadata variable. Each has its own resolution path.
  3. Was this device recently onboarded? New boxes are the classic unfilled-value victims.
  4. Did anyone "help" with a per-device override recently? Rogue overrides are how invariants break silently.
  5. What does the install preview say? The preview shows the substituted truth — read it before forming a hypothesis.
Core stance: mixed install results across identical packages are a resolution problem. The package is innocent until the per-device table proves otherwise.

Common dead ends

  • SSH-ing to the FortiGate hunting for the mapping. It isn't there. Mappings and variables are FMG-side translation; the box only holds resolved values.
  • Editing the shared object's "value" to fix one branch. You just changed it for everyone — the copies problem rebuilt inside one object.
  • Re-installing repeatedly hoping validation relents. The gate isn't flaky; the reference is unresolvable. Fix the table, not the retry count.
  • Blaming hardware/cabling for what is a name/value resolution failure. If the cable question even comes up, you've skipped step one.
  • Adding a per-device override to a value that must be invariant "just to make it work" — you've silently broken the fleet-wide guarantee.

Likely root causes, ranked

  1. Unfilled per-device value on a newly onboarded device (variable with no default).
  2. Missing per-device mapping row on a dynamic object for that box.
  3. Case mismatch between rule reference and mapping (WAN/wan).
  4. Wrong (typo'd) per-device value — installs fine, matches nothing. Only the preview or traffic tests reveal it.
  5. Attempted per-device mapping on an unsupported object type — the "fix" was never possible; unique names required.
  6. Rogue per-device override on an invariant value — the drift nobody notices until audit.

Exam traps

Trap 1: "Per-device map the web filter profile." Security profiles don't support it — unique names is the correct answer.
Trap 2: "The FortiGate falls back to the default when the variable is empty." No — with no resolvable value, install fails for that device. Nothing is pushed.
Trap 3: "The mapping is pushed to the device DB / FortiGate." Mappings are consulted at install inside FMG; only resolved values land on the box.
Trap 4: Importing per-box-different configs of an unsupported type → conflict; the fix is a uniquely named object re-imported — not forcing a mapping.
The senior habit in one line: when identical packages produce different install results, open the per-device table and the install preview before forming any other hypothesis. The preview is where secrets confess.
