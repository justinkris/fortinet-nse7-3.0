SESSION 15 · TROUBLESHOOTING GUIDE

OSPF Won't Form on a Dial-Up Hub

How a senior engineer thinks through silent adjacencies — the thinking, not just the commands.

Symptoms

  • IPsec tunnel is UP; pings between hub and spoke succeed.
  • OSPF neighbor table on the hub is completely empty — not stuck, not flapping, empty.
  • No routes learned in either direction; static reachability still works.
  • Variant symptom set (later stage): adjacencies at Full, but a specific subnet known only via static route on the hub never appears on spokes.

Questions to Ask Before Touching Anything

  • Is the transport healthy? (Tunnel state, ping.) If yes, stop suspecting the tunnel — suspect the protocol's assumptions.
  • Is the table empty or broken? Empty means hellos were never delivered. Flapping/Init/ExStart means they arrived but the conversation failed. These have different suspect lists.
  • What network type is configured on each end — and can the medium underneath honour that type's assumptions?
  • Is this a dial-up VPN? Is net-device disabled (one shared interface for all spokes)?
  • For missing routes: was the route ever born inside OSPF, or is it foreign (static/BGP/connected) and awaiting redistribution?

Verification Commands & Expected Outputs

CommandWhat you're checkingHealthy P2MP hub

Decision Tree

1 · Separate transport from protocolTunnel down or pings failing → this is an IPsec problem, not OSPF. Stop here and fix transport first. Tunnel up + pings OK → proceed; the protocol's assumptions are the suspect.
2 · Classify the neighbor tableEmpty → hellos never delivered → go to step 3 (delivery/network-type path). Neighbors stuck below Full or flapping → hellos delivered but parameters disagree (area ID, timers, subnet, MTU, authentication) → different investigation.
3 · Check the network type against the mediumget router info ospf interface. Broadcast on a tunnel/dial-up interface = configured assumption ("shared medium delivers my multicast") that reality can't honour. The multicast hello dies at the interface — this alone explains total silence.
4 · Match the topology to the correct typeDedicated tunnel, exactly one far-side neighbor → point-to-point (the default). One shared interface, many dial-up spokes (net-device disable) → point-to-multipoint. Fix on the hub, confirm hello intervals pass.
5 · Verify with the fingerprintFull/ - per spoke. Full = identical LSDBs. Dash = no election ran — exactly what P2MP should produce. Seeing DR/BDR/DROther here means something is still configured broadcast.
6 · Missing foreign routes → redistribution pathRoute exists on the hub as a static but never reaches spokes: OSPF only advertises what it was told about. Check config redistribute static (default disabled). Enabling it makes the box an ASBR — and it's all-or-nothing, so apply a filter (access list for one known prefix) before opening the valve.
7 · Verify recovery AND root causeRoutes present on spokes, nothing extra leaked past the filter, and you can state why the original config failed — not just that changing it helped.

Common Dead Ends

Dead end · Rebuilding the tunnelThe tunnel was never broken — pings proved it. Restarting IPsec burns time and adds change-noise while the neighbor table stays empty for the same reason.
Dead end · Hunting for a firewall policy blocking OSPFPlausible-sounding, but a policy drop wouldn't be the first hypothesis when the sending side never successfully emits deliverable hellos. Check what the hub is trying to send before hunting for who's blocking it.
Dead end · "The DR election must be broken — let's tune priorities"On a hub-and-spoke, there is no election to fix. Tuning priority on a star treats a symptom of the wrong network type as a tunable.
Dead end · Enabling redistribution to "make routes appear"If adjacencies aren't at Full yet, redistribution changes nothing — and opening it unfiltered plants a future incident. Fix adjacency first, then advertise, then filter.

Likely Root Causes (ranked)

  1. Network type broadcast configured on a dial-up hub tunnel interface (no shared medium to deliver multicast hellos).
  2. Network type mismatch between hub (P2MP) and spoke expectations after a partial fix.
  3. Foreign route never redistributed (default disabled) — routes "missing" that OSPF was never told about.
  4. Redistribution enabled but the filter denies the intended prefix (or exact-match excludes a subnet variant).

Exam Traps

Trap 1: Output shows Full/DROther — the question asks the network type. DROther = an election ran = broadcast. Full/ - = P2P or P2MP.
Trap 2: "Dial-up spokes, net-device disabled, adjacencies won't form" — the answer is the hub's network type, not IPsec settings, not policies.
Trap 3: "Why doesn't R2 advertise its connected network Net 1 into OSPF?" — redistribution is disabled by default for every source, including connected.
Trap 4: Filtering question mentioning ge/le → prefix list (and it applies only to distribute-list-in). Mentioning attribute modification → route map.

The Thinking, Distilled

Every step above is one move: state the assumption, then test whether reality honours it. Broadcast assumes a shared medium — test: is there one? P2P assumes one neighbor — test: how many spokes? Redistribution assumes you told OSPF about the route — test: did you? Engineers who memorise commands find the fault eventually; engineers who interrogate assumptions find it first.
