Speaking OSPF on FortiGate

Network types · Redistribution · Route filtering · OSPF commands — Chatswood comes online as Area 1

Story Recap

Junior arrived at 6:30 AM to "surprise everyone": Chatswood's dial-up tunnel to the hub's shared To-Spokes interface, OSPF enabled, set network-type broadcast — because a blog post said the internet is "basically a big Ethernet." Tunnel up, pings fine, neighbor table empty for twenty minutes. Señor taped a sticky note to the monitor: "EXHIBIT A. DO NOT FIX." By session's end: hub reconfigured point-to-multipoint, three adjacencies at Full, one legacy management subnet redistributed through a one-line access-list filter, and the hub wearing three role hats at once.

Learning Objectives

  • Explain the three OSPF network types as assumptions about the medium, and match each to a topology.
  • Diagnose why a broadcast network type produces total silence on a dial-up hub interface.
  • Read get router info ospf neighbor output as forensic evidence of network type.
  • Explain redistribution, the ASBR role, and why FortiGate ships all redistribution disabled.
  • Choose between access lists, prefix lists, and route maps by requirement, not sophistication.
  • Run all four router-role definitions as independent checks against one device.

1. Network Types Are Assumptions About the Medium

OSPF's network type isn't cosmetic — it encodes what the protocol may assume about the wire underneath it. Configure a type whose assumptions the medium can't honour, and the CLI accepts the lie cheerfully while the protocol goes silent.

TypeCore assumptionDR/BDR?Where it fits
Broadcast (multiaccess)Shared medium — one multicast frame reaches every attached router for free (Ethernet)Yes — election heldLAN segments with multiple routers
Point-to-pointExactly one neighbor on the far side of the linkNoDedicated site-to-site tunnels. FortiGate default on tunnel interfaces
Point-to-multipointMany neighbors reachable through one interface, but no shared medium and no multicast delivery — treats the interface as a bundle of individual P2P conversationsNo — election never runDial-up VPN hub with net-device disable: one interface, many spokes
The anchor rule (p.129)
Multiple dial-up spokes terminating on a single hub interface (net-device disabled) → the hub's tunnel interface must be point-to-multipoint.

Broadcast machinery (for contrast)

On a real multiaccess segment, routers elect a DR and BDR (highest priority wins, ties broken by highest Router ID, priority 0 = never DR). Full adjacencies form only with the DR/BDR to cut resource use. Two multicast addresses do the work:

  • 224.0.0.5 — AllSPFRouters: hello packets; the DR/BDR send LSA updates and acks here.
  • 224.0.0.6 — AllDRouters: all other routers send their LSA updates and acks here (i.e., to the DR/BDR).

Why Junior's config produced total silence

The hub, in broadcast mode, addressed its hello to 224.0.0.5 and expected the medium to deliver it to everyone at once. But under To-Spokes there is no shared wire — three separate encrypted tunnels to three different public IPs. Broadcast mode does not replicate the packet per tunnel; that job belonged to the medium that doesn't exist. The hello dies at the interface. No hellos delivered → no neighbors discovered → the table is empty, not broken. (A wrong-but-compatible setting tends to produce flapping or stuck-in-Init neighbors; total emptiness points at delivery failure.)

Broadcast mode would also attempt a DR election among spokes who cannot hear each other's votes — wrong at the delivery layer and the topology layer.

Señor's whiteboardOne word written above To-Spokes: "Deaf." The multicast hello is a shout into an interface with no medium that can hear it.

What point-to-multipoint actually does

  • Treats the shared interface as a bundle of individual point-to-point conversations.
  • Replicates and unicasts hellos and updates per spoke — the protocol takes over the delivery job the medium can't do.
  • Each spoke forms exactly one adjacency — with the hub. Spokes never hear each other's hellos; they learn each other's networks because the hub floods LSAs onward. Hellos stay per-link; topology information is what crosses the hub.
  • No DR/BDR election, ever. With one adjacency per spoke and no shared segment, there is nothing for a DR to optimise.

2. The Fix and Its Fingerprint

config router ospf
    config ospf-interface
        edit "Hub"
            set interface "To-Spokes"
            set network-type point-to-multipoint
        next
    end
end

Verification:

get router info ospf neighbor

Neighbor ID     Pri   State      Dead Time   Address        Interface
10.0.1.2        1     Full/ -    00:00:37    10.10.10.2     To-Spokes
10.0.1.3        1     Full/ -    00:00:34    10.10.10.3     To-Spokes
10.0.1.4        1     Full/ -    00:00:39    10.10.10.4     To-Spokes
  • Full — top of the adjacency ladder (Init → 2-Way → ExStart → Exchange → Loading → Full). Full means both routers hold identical LSDBs; a healthy adjacency parks here.
  • The dash — no role, because no election ran. DROther is a campaign result ("I stood in an election and lost"), not a neutral word for "ordinary neighbor." It can only appear where an election happened.
Forensic rule
The role column is a fingerprint of the network type. Full/DR, Full/BDR, Full/DROther → broadcast segment. Full/ - → point-to-point or point-to-multipoint. You can identify the network type without ever opening the config.

3. Redistribution — Teaching OSPF About Foreign Routes

The hub held a static route to the DR site's legacy management subnet (10.99.50.0/24). OSPF never advertises what it wasn't told about. Redistribution injects routes born outside OSPF (static, connected, BGP, RIP, IS-IS) into the OSPF domain — and the moment a router does this, it becomes an ASBR by definition.

config router ospf
    config redistribute static
        set status enable      <- default is disable, for ALL sources
    end
end
Why default-closed matters (p.130)
Every redistribution source ships disabled. OSPF's superpower — automatic, trusted, domain-wide propagation — is exactly its blast radius. During the Parramatta outage, the hub carried emergency statics and an unencrypted default route. Open-by-default redistribution would have flooded those fleet-wide, automatically, in seconds. One careless static on one box becomes everyone's route.

And note: redistribute static is all-or-nothing — every static on the box goes in unless you filter.

4. Route Filtering — The Fine-Grained Valve

ToolTypeActionExtras / limits
Access listSimple — filter on prefixesPrevent route injection via distribute-list-in; also filters redistributed protocols in distribute-list configSimplest, most auditable
Prefix listSimple + logic — prefixes with operatorsApplies only to distribute-list-inge/le operators match a prefix and a range of mask lengths
Route mapAdvancedFilter incoming external routes via distribute-route-map-in; filter redistribution per protocol in config redistribute <...>Can embed access/prefix lists as match objects; can modify attributes, not just permit/deny

Choosing: the least-powerful-tool discipline

Penny's requirement: "only the management subnet may ever leave that box, nothing else, ever." One exact prefix, permit/deny only. Eliminate by mechanism: no attributes to modify → route map out; no mask-length ranges → ge/le unneeded → prefix list out. Access list wins.

The 2 AM tax
Every layer of machinery you deploy is a layer the on-call engineer must read and understand while the pager is screaming. A route map referencing a prefix list with operators is three indirections to audit; a one-line access list is self-evidently correct at a glance. Complexity you don't need is a standing tax on every future incident.
config router access-list
    edit "MGMT-ONLY"
        config rule
            edit 1
                set prefix 10.99.50.0 255.255.255.0
                set exact-match enable
            next
        end
    next
end

6. Compound Router Roles — Live on the Hub

The hub now holds interfaces in Area 0 and Area 1, and redistributes a static. Run all four definitions as independent true/false checks:

RoleTestHub result
Internal routerAll interfaces in a single area?✗ — straddles Area 0 and Area 1
ABRInterfaces in multiple areas?✓
Backbone routerAny interface in Area 0?✓
ASBRAny non-OSPF routing source imported?✓ — redistribute static
Exam trap — both directions
Roles are overlapping labels, not exclusive categories. Stopping early and selecting everything are the same error: skipping the per-definition check. "Select all that apply" punishes both. The discipline is in the checking, not the count.

Troubleshooting Workflow (as run this session)

  1. Separate the symptom: tunnel up + pings OK + OSPF silent → transport is fine; the protocol's assumptions are not.
  2. Follow the packet: broadcast-mode multicast hello has no shared medium → dies at the interface.
  3. Read the evidence type: an empty neighbor table means hellos were never delivered; a broken/flapping table means they were delivered but disagreed.
  4. Eliminate by assumption: P2P assumes exactly one far-side neighbor; To-Spokes has three.
  5. Select by mechanism: P2MP replicates-and-unicasts per spoke, no election.
  6. Verify the fingerprint: Full/ - per spoke confirms both adjacency health and network type.

Common Mistakes

  • Configuring broadcast on a dial-up hub interface ("the internet is basically Ethernet").
  • Expecting P2MP to elect a DR — no shared segment, no election, ever.
  • Reading DROther as "ordinary neighbor" — it means lost an election, so it implies broadcast.
  • Assuming redistribution is on by default, or forgetting it's all-or-nothing without a filter.
  • Reaching for a route map when an access list satisfies the requirement.
  • Treating router roles as a single-answer question — under- or over-selecting instead of checking each definition.

Senior Engineering Insights

  • A network type is a contract with the medium. Verify the medium can honour it before blaming the protocol.
  • Command output is forensic evidence — the neighbor role column identifies the network type without the config.
  • Defaults encode blast-radius judgments: OSPF's trust model is why redistribution ships closed.
  • Choose the least powerful tool that meets the requirement; complexity is a tax paid at 2 AM.
  • Compliance requirements stated in one sentence should be implementable in roughly one line — if they aren't, question the design.

Exam Tips

  • "Dial-up spokes, net-device disable, single hub interface" → answer is point-to-multipoint, every time.
  • Default tunnel-interface network type: point-to-point.
  • Full/ - vs Full/DROther is a favourite distractor pair — the dash means no election ran.
  • Redistribution: default disabled for bgp / connected / isis / rip / static alike.
  • Prefix lists' ge/le and their restriction to distribute-list-in; route maps modify attributes and can embed the other two.
  • Select-all role questions: run Internal / ABR / Backbone / ASBR as four independent tests.

Key Takeaways

  1. Network type = medium assumptions. Broadcast needs a shared wire; a dial-up hub has none → point-to-multipoint.
  2. P2MP: per-spoke unicast replication, one adjacency per spoke, no DR — and Full/ - is its signature.
  3. Redistribution makes an ASBR; it ships disabled because OSPF propagates trust automatically and domain-wide.
  4. Filter with the simplest tool the requirement allows: access list → prefix list (ge/le) → route map (attributes).
  5. One router can be ABR + backbone + ASBR simultaneously. Check every definition, every time.
