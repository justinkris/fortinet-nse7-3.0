SESSION 43 · BITE · ACCELERATION DIAGNOSTICS

Five Minutes: Proving the Speed

The evidence, the flags, and the one distinction that decides the fix.

A checkbox proves intent. Output proves behaviour.

Read this before work. Five minutes, six ideas, nothing decorative.

One

The Evidence Line

diagnose sys session list
npu info: flag=0x81/0x91, offload=9/9, ips_offload=9/9, epid=128/166 ...
  • offload=9/9 → forwarding accelerated. 9 = NP7, 6 = NP6. Slash = originate/reply.
  • ips_offload=9/9 → inspection riding NTurbo.
  • 9/0 → asymmetric. Half a fast path is a fault.
Two

Which Table Owns the Field

The question that settles every one of these: what can this table physically record? A session table holds one row per session, so it records paths — not the thousands of calls made inside a session's inspection.

FieldCommandFeature
offload=X/Ysession listNP forwarding
ips_offload=X/Ysession listNTurbo (a path)
— nothing ——IPSA (a phone call)
npu_flag=NNvpn tunnel listIPsec SA in NPU
The trap: session-list fields use X/Y. npu_flag is a single value. 9/9 is never an npu_flag.

Why CP8 can't do advanced
The extra pattern types were never etched into that silicon. A software mode that varies by hardware generation is a thin wrapper over a fixed instruction set.
Four

npu_flag — the whole table

ValueMeaning
00Kernel only — nothing in the NPU
01Outbound SA copied
02Inbound SA copied
03Both — healthy steady state (01 + 02)
20Unsupported cipher/HMAC — never

Why the copy is lazy: NPU memory is finite. SAs load into the kernel at phase 2 up and are copied to the NPU on first traffic, per direction.
