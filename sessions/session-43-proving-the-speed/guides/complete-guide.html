Proving the Speed: Evidence, Not Adjectives

Hardware acceleration configuration and diagnostics — ch.10, pp. 294–302. The session where the tunnel finally got looked at.

A checkbox proves intent. Command output proves behaviour.

Everything in Session 43 hangs off that distinction. Cipher would not accept a dashboard screenshot, and she was right not to — the box had been quietly doing software crypto for twenty-six sessions while every configuration page said it was fine.

Phase One

Story Recap

Tuesday, 07:41. Taped to the inside of the network-room door, at eye height so you cannot enter without reading it, three questions in Señor's handwriting and a doodle of a cactus wearing sunglasses. "The door does not give hints."

Inside, Cipher at the whiteboard with a printed copy of Directive 2026-14 and a red pen. Her demand was not for a configuration change. It was for evidence: the specific line of command output proving that a trading session on VLAN 40 is both inspected and accelerated. Junior offered a dashboard screenshot. Cipher, without turning around: "A checkbox proves intent. I asked for evidence."

And on the bench — a patch lead draped over a monitor like a scarf, with a paper label:

EL TÚNEL. HQ ↔ DR. Age: 26 sessions. Grievance: nobody has ever checked on me.

Riley's 06:20 ticket said DR replication crawled all night. The tunnel to Parramatta. Same box, same config, nothing changed. Señor had said the words "IPsec is a forwarding job wearing a trench coat" four times across four sessions, and four times we had nodded and moved on. This was the session the coat came open.

By the end: the evidence pack was accepted, a twenty-six-session-old latent defect was found and correctly diagnosed, and Señor peeled the label off the patch lead and put it in his pocket. "El Túnel has been seen. He can rest."

Phase Two

Learning Objectives

  • Produce auditable evidence of hardware acceleration from CLI output rather than configuration state.
  • Know exactly which table owns which field — and what each table is physically capable of recording.
  • Configure and reason about np-accel-mode, cp-accel-mode, and why one of them varies by hardware generation.
  • Explain what the NP does to an ESP packet that it never does to a TCP packet, and where it gets what it needs.
  • Read npu_flag and distinguish can't offload from won't offload.
  • Configure accelerated inter-VDOM links and know the one command that deliberately breaks acceleration for troubleshooting.
Concept One

The Evidence Pack: Reading the Session Table

◆Think First
You have configured IPS on VLAN 40. The GUI shows the profile applied. An auditor asks you to prove the traffic is actually being inspected and still riding the NP7. What do you type, and — before you press enter — what line do you expect to see?
  • Where does behaviour live, if configuration only describes intent?
  • Why would a per-session record be more convincing to an auditor than a per-policy setting?

The proof lives in one line of diagnose sys session list. Not in the policy, not in the dashboard — in the session table, which records what is actually happening to a live flow right now.

The evidence line — healthy, inspected, accelerated
HQ-NGFW # diagnose sys session list
...
npu info: flag=0x81/0x91, offload=9/9, ips_offload=9/9, epid=128/166, ipid=166/128, vlan=0x0000/0x0000
...
FieldWhat it provesReading
offload=9/9Forwarding is accelerated on the NP9 = NP7. 6 = NP6. Slash separates originate / reply direction — each needs its own key.
ips_offload=9/9Inspection is riding the NTurbo pathNon-zero in both directions = the session is on the NTurbo data path, not stranded on the CPU.

Concept Two — the session's central lesson

Which Table Owns Which Field

This is where three separate errors were made in one session, all of them the same error. The fix was never more information — it was a single question:

◆
The Tool
"What can this table physically record?"

A session table holds one row per session, with per-direction values. So it can record facts about the path a session takes. It cannot record something that happens thousands of times inside a single session's inspection — there is nowhere to put it.

That single structural fact settles the NTurbo / IPSA confusion permanently:

FeatureWhat it isFootprint
NTurboA path — a session either rides it or doesn't, decided at session birthips_offload in the session list
IPSAA phone call — the IPS engine offloading pattern matching to the CP, per matchNone. Invisible in the session table.
Analogy
The road and the phone call.
A delivery van's logbook records which route it took — that's NTurbo, one entry per journey. It does not record every phone call the driver made from the cab. IPSA is the phone calls. They happen, they matter, they speed things up enormously — and the logbook has no column for them.
Do not over-correct. "IPSA leaves no footprint in the session table" is true. "IPSA is unmeasurable" is false — there are counters elsewhere on the box. Know the difference between this evidence source can't show it and no evidence exists.
FieldLives inTells you about
offload=X/Ydiagnose sys session listNP forwarding acceleration
ips_offload=X/Ydiagnose sys session listNTurbo path
no_ofld_reasondiagnose sys session listWhy this session wasn't offloaded
npu_flag=NNdiagnose vpn tunnel listIPsec SAs copied into the NPU
The trap that caught us three times: session-list fields use X/Y slash notation. npu_flag is a single two-digit value in a different command. 9/9 is never a valid npu_flag.

Concept Three

Acceleration Modes and the Fixed Instruction Set

◆Think First
cp-accel-mode advanced is the default on a 900G. On a device with a single CP8, basic is not merely the default — it is the only option. Software modes don't usually depend on which chip is fitted. Why does this one?
  • What would have to be true about CP8 silicon for FortiOS to refuse to offer advanced?
  • Where else have you seen a capability list baked into hardware rather than software?
IPS acceleration — global
config ips global
    set np-accel-mode {none | basic}            # NTurbo. Default: basic
    set cp-accel-mode {none | basic | advanced} # IPSA. Default: advanced
end

config firewall policy
    edit <policy_id>
        set np-acceleration {enable | disable}  # NTurbo, per policy. Default: enable
    next
end
SettingScopeValuesDefault
np-accel-modeGlobal (ips global)none | basicbasic
cp-accel-modeGlobal (ips global)none | basic | advancedadvanced — but basic only on single-CP8 devices
np-accelerationPer firewall policyenable | disableenable

Why CP8 can't do advanced. A content processor is not a software library — it is a pattern-matching engine with a fixed instruction set etched into silicon. advanced mode offloads more types of pattern matching than basic. The circuits for those extra types were never fabricated on the CP8. FortiOS therefore doesn't offer the mode, because it would be offering to hand the chip work it has no instructions to perform. basic is the subset every CP generation can actually execute.

◆
Senior Generalisation — remember this one, it pays out below
A software mode that varies by hardware generation is always a clue that the mode is a thin wrapper over a fixed instruction set.

Concept Four — the four-session debt

IPsec on the NP: Opening the Trench Coat

◆Think First
You have said for four sessions that the NP handles IPsec. Name the work. What does the NP7 do to an ESP packet that it never does to an ordinary TCP packet — and where does it get what it needs to do that?
  • What does a TCP packet need? Session-key match, header rewrite, forward. What is missing from that list for ESP?
  • If the NP needs the keys, where must they physically be?

The work is crypto. Encrypt on the way out, decrypt on the way in. That is the only reason ESP is special to the NP — everything else about it is ordinary forwarding, which is why it earns the exception in the offload disqualifier list ("tunnelling breaks offload, except IPsec").

To do that crypto, the NP needs the Security Association — the algorithm and the keys. And it can't fetch that from the kernel on every packet; that would defeat the entire point. The SA must be copied into NPU memory.

Why the copy is lazy

NPU memory is finite. It is the same scarcity that caps an NP7 at roughly 12 million sessions and spills the overflow to the CPU. FortiOS therefore applies the obvious economy: don't spend NPU memory on a tunnel that isn't carrying anything.

Sequence: Phase 2 comes up → IKE negotiates → SAs created and loaded into the kernel → nothing copied to the NPU yet → first packet arrives → the SA for that direction is copied into the NPU → first packet in the other direction → second SA copied.
Per-tunnel and global offload controls
config vpn ipsec phase1-interface
    edit "HQ-to-Parramatta"
        set npu-offload {enable | disable}   # PER TUNNEL. Default: enable
    next
end

config system global
    set ipsec-asic-offload {enable | disable} # GLOBAL. Diffie-Hellman key
end                                           # exchange for ESP. Default: enable
Two different scopes, easily confused. npu-offload is per phase-1 tunnel and governs the ESP data path. ipsec-asic-offload is global and governs CP offload of the Diffie-Hellman key exchange. Exam questions like to swap them.
Why doesn't FortiOS copy SAs at phase 2 up?

Concept Five — exam gold

npu_flag: Can't versus Won't

Because the SAs are copied independently, per direction, on first traffic, the tunnel needs a way to report how far it has got. That is npu_flag, in diagnose vpn tunnel list — a single value encoding both directions as a bitfield.

ValueMeaningVerdict
00SAs in the kernel only — nothing copied to the NPUAmbiguous: no traffic yet, or offload disabled
01Outbound SA copied to NPUIn progress
02Inbound SA copied to NPUIn progress
03Both SAs copied — fully offloadedHealthy steady state
20Unsupported cipher or HMAC — cannot be offloadedPermanent fault

The counting is the giveaway: outbound is bit 1, inbound is bit 2, and 01 + 02 = 03. That is the logic. 20 deliberately breaks the pattern, because it isn't a progress value at all — it's a refusal.

Señor's Whiteboard
"00 means not yet. 03 means yes. But 20 means never. And never has a reason."
When you see 20, you have not finished. You have been told the category of the fault. The specific fault is on a different line, two lines up.

The live case: HQ ↔ Parramatta

diagnose vpn tunnel list name HQ-to-Parramatta
name=HQ-to-Parramatta ver=2 serial=3 10.10.1.1:0->10.10.2.2:0
bound_if=5 lgwy=static/1 tun=intf mode=auto/1 encap=none/544
proxyid_num=1 child_num=0 refcnt=9 ilast=1 olast=1
stat: rxp=88214 txp=91003 rxb=12884901 txb=13221376
proxyid=Parramatta-P2 proto=0 sa=1 ref=2 serial=1
  SA: ref=3 options=18 type=00 soft=0 mtu=1438 expire=1823
  enc: aes-256-cbc  auth: sha512
  npu_flag=20 npu_rgwy=10.10.2.2 npu_lgwy=10.10.1.1 npu_selid=1

npu_flag=20 is the diagnosis: this tunnel has never been offloaded and never will be as configured. Every ESP packet is being encrypted and decrypted by the CPU in software. That is why replication crawled.

The cause is on the enc: / auth: line. AES-256-CBC is fine — it has been in NP silicon since the NP4 era. SHA-512 is the problem: a 512-bit hash that many hardware crypto engines never implemented, because almost nobody asks for it.

◆
The generalisation, cashed
The NP's crypto is not a library — it is etched circuits for a fixed list of algorithms. Same principle as the CP8. Ask for something off the list and the chip cannot decline politely; FortiOS stamps 20 and hands the whole tunnel to the CPU.
One unsupported algorithm poisons the whole tunnel. Not just the hashing — the entire ESP data path drops to software. Compare: one proxy profile poisons the whole policy (Session 41). Same shape, different layer.

Concept Six — the under-taught corner

Accelerated Inter-VDOM Links

NP processors provide VDOM links named npu_vlink0 and npu_vlink1. By default both are assigned to the root VDOM, which accelerates nothing.

◆
The Rule
Assign one link to each VDOM. One end in VDOM A, the other in VDOM B. Both ends in root = no acceleration.

The number of VDOM links per NP is limited. To create more accelerated inter-VDOM paths, add VLAN interfaces on top of the NPU vlink interfaces — using the same VLAN ID on both.

VLAN over NPU VDOM link
config system interface
    edit "VLANvlink1Sales"
        set vdom "Sales"
        set ip 10.10.10.10 255.255.255.252
        set interface "npu0_vlink1"
        set vlanid 10                 # must match on BOTH vlinks
    next
end

Deliberately breaking acceleration to troubleshoot

You cannot sniff traffic that never touches the CPU. To packet-capture on an npu_vlink, temporarily disable offload on the firewall policy:

The sniffing workaround
config firewall policy
    edit 1
        set auto-asic-offload disable   # Default: enable
    next
end

# Confirm it took effect:
# diagnose sys session list
npu_state=0x000001 no_offload
no_ofld_reason: disabled-by-policy
Operational warning. auto-asic-offload disable affects all traffic on that policy, not just the flow you're chasing. Set it back to enable as soon as the capture is done. This is the exact class of mistake that leaves a tunnel on npu-offload disable for twenty-six sessions.
Don't confuse the three "turn it off" commands: auto-asic-offload (policy, NP forwarding) · np-acceleration (policy, NTurbo) · npu-offload (phase-1 tunnel, IPsec).

Troubleshooting Workflow

SLOW TUNNEL │ └─ diagnose vpn tunnel list name <tunnel> │ ├─ npu_flag=03 ───────────► Offload is healthy. The fault is elsewhere. │ Check MTU, routing, the far end, the link. │ ├─ npu_flag=20 ───────────► CAN'T offload. │ └─ Read the enc: / auth: line. │ Which algorithm is off the silicon's list? │ └─ Fix requires a crypto change │ └─ requires SIGN-OFF. │ ├─ npu_flag=00 ───────────► Is traffic actually flowing? (check rxp/txp) │ │ │ ├─ No traffic ───────► Normal. Lazy copy. Not a fault. │ │ │ └─ Traffic flowing ──► WON'T offload. │ └─ show vpn ipsec phase1-interface <name> │ Look for: set npu-offload disable │ └─ Someone left it off. Put it back. │ └─ npu_flag=01 or 02 ─────► One direction only. Transient during ramp-up; persistent = investigate the quiet direction.
The clause most engineers skip. This tunnel had SHA-512 from the day it was built. It was never offloaded. So why did it only crawl last night? Fixing the algorithm restores service; it does not explain the change. Something else moved — data volume, CPU contention, a new workload on the box. Find it, or you will be back.
◆
Root cause vs service restoration
A latent defect that becomes visible has two causes: the defect, and whatever removed the headroom that was hiding it. Fix both.
Phase Three

Common Mistakes

Junior's two

  • "Just screenshot the dashboard — it says IPS enabled." A checkbox proves intent. Configuration is a claim; the session table is a witness. An auditor is not asking what you meant to do.
  • "We just change it to SHA-256. I'll do it now." The fix was technically correct and would have worked — which is exactly what made it dangerous. Cipher wrote the crypto standard. A correct fix applied without the sign-off it requires is still an incident.

Ours

  • Importing slash notation across tables. Predicting npu_flag as 0/0, 9/0, 0/9, 9/9. Good pattern-matching, wrong table. Fixed by asking what the table can record.
  • Attaching IPSA's meaning to NTurbo's field. ips_offload proves the NTurbo path. It says nothing about the CP.
  • Naming the fix before stating the fault. "set npu-offload enable" is a remedy, not a diagnosis. On an exam that asks for the cause, it scores zero.
  • Guessing with a question mark. "NP7, 8?" costs exactly what a wrong answer costs — and destroys the signal that "not sure" would have preserved.
  • Answering clause one and stopping. See below.
The habit that cost the most this session. Six of ten gauntlet answers were half-marks, and in every single one the first clause was right and the second was absent — four of them on material produced correctly within the hour. This is not a knowledge gap. It is a stopping habit. Count the question marks in the question. Two questions, two sentences.

Key Takeaways
1. A checkbox proves intent; output proves behaviour.
2. Ask what the table can physically record — it tells you which feature owns the field.
3. NTurbo is a path (visible). IPSA is a phone call (invisible here).
4. 00 = not yet · 03 = yes · 20 = never, and never has a reason.
5. Can't (20, crypto) vs Won't (00 with traffic, disabled) — one needs a signature.
6. A mode that varies by hardware generation is a wrapper over a fixed instruction set.
