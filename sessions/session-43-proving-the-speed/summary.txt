TECHNICAL CONCEPTS COVERED
------------------------------------------------------------------
1. EVIDENCE vs INTENT. A checkbox proves intent; session-table
   output proves behaviour. Cipher's framing, and the spine of
   the whole session.
2. Reading `diagnose sys session list` as an evidence artefact:
   `offload=9/9` (forwarding accelerated, 9=NP7, both directions)
   and `ips_offload=9/9` (inspection riding NTurbo).
3. WHICH TABLE OWNS WHICH FIELD — the session's central lesson.
   `ips_offload` = NTurbo's footprint (a PATH, per-session,
   per-direction). IPSA leaves NO session-table footprint (it is
   a per-pattern-match CALL, thousands per session, nowhere to
   record it in a one-row-per-session table).
   THE TOOL TAUGHT: "what can this table physically record?" —
   the student fixed three separate wrong answers with it.
4. cp-accel-mode {none|basic|advanced}; default ADVANCED on
   CP9/CP10 boxes; single-CP8 devices limited to BASIC ONLY.
   MECHANISM DERIVED: the extra pattern types aren't etched in
   CP8 silicon, so FortiOS won't offer a mode the chip has no
   instructions for.
   GENERALISATION BANKED: "a software mode that varies by
   hardware generation is a thin wrapper over a fixed instruction
   set." Deliberately planted, then CASHED IN 20 minutes later on
   the SHA-512 problem. This pay-off structure worked extremely
   well — reuse it.
5. IPsec on the NP: the work is ENCRYPT/DECRYPT (crypto), the
   only reason ESP is special. The NP needs the SA (algorithm +
   keys) COPIED INTO NPU MEMORY. Student intuited "stores the
   keys in silicon" unprompted — good.
6. WHY THE COPY IS LAZY: NPU memory is finite (same scarcity that
   caps NP7 at ~12M sessions). Don't spend it on an idle tunnel.
   SAs go to the KERNEL at phase 2 up; copied to NPU on FIRST
   TRAFFIC, per direction.
7. npu_flag TABLE (in `diagnose vpn tunnel list`):
     00 = kernel only, nothing in NPU
     01 = OUTBOUND SA copied
     02 = INBOUND SA copied
     03 = BOTH (healthy steady state) — it's a BITFIELD, 01+02=03
     20 = unsupported cipher/HMAC, CANNOT offload (doesn't fit
          the bitfield — FortiOS's way of saying "refusal, not
          progress")
8. THE CAN'T-vs-WON'T PAIR (exam gold):
     npu_flag=20 -> CAN'T. Look at the `enc:`/`auth:` line.
     npu_flag=00 WITH TRAFFIC FLOWING -> WON'T. Someone set
       `npu-offload disable` on phase 1. Confirm with
       `show vpn ipsec phase1-interface <name>`.
     00 is AMBIGUOUS on its own (no traffic yet OR disabled).
9. ONE UNSUPPORTED ALGORITHM POISONS THE WHOLE TUNNEL — explicit
   rhyme with S41's "one proxy profile poisons the whole policy."
   The student was shown the shape-match; it landed.
10. Also covered (lightly, from pp. 295-298): npu_vlink0/vlink1
    inter-VDOM acceleration and the assign-one-link-to-each-VDOM
    rule; `auto-asic-offload disable` as the sniffing workaround
    with the put-it-back-immediately warning; DVLAN/VXLAN offload
    on NP7/SP5; `ipsec-asic-offload` (GLOBAL scope, DH key
    exchange, distinct from per-tunnel `npu-offload`).

KEY CLI DISCUSSED
------------------------------------------------------------------
diagnose sys session list
  -> npu info: flag=0x81/0x91, offload=9/9, ips_offload=9/9
diagnose vpn tunnel list name <tunnel>
  -> enc:/auth: line; npu_flag=00|01|02|03|20; npu_rgwy/npu_lgwy
config vpn ipsec phase1-interface / set npu-offload enable|disable
  (PER TUNNEL, default enable)
config system global / set ipsec-asic-offload disable
  (GLOBAL, DH key exchange, default enable)
config ips global / set np-accel-mode {none|basic}   (def basic)
                    set cp-accel-mode {none|basic|advanced}
                    (def advanced; basic-only on single-CP8)
config firewall policy / set np-acceleration enable|disable
config firewall policy / set auto-asic-offload enable|disable
  (the npu_vlink sniffing workaround)
diagnose npu np7 dvlan-mode-list / dvlan-mode
show vpn ipsec phase1-interface <name>

TROUBLESHOOTING APPROACH TAUGHT
------------------------------------------------------------------
Slow tunnel workflow: `diagnose vpn tunnel list name <x>` ->
read npu_flag -> if 20, go straight to the enc:/auth: line and
identify WHICH algorithm -> if 00 and traffic IS flowing, go to
phase 1 config and look for npu-offload disable. Symptom is
identical; the flag tells you which branch.
