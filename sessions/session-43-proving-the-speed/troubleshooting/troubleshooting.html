TROUBLESHOOTING GUIDE · SESSION 43

The Tunnel That Was Never Checked

How a senior engineer diagnoses a slow IPsec tunnel and an unaccelerated policy — thinking first, commands second.

The command is the easy part. Knowing which output line answers which question is the skill.

Everything below assumes the fault is real and the config "looks fine". That is the normal case, not the exception.

Section One

Symptoms that lead here

  • Replication, backup or file transfer over a site-to-site tunnel is slow — but not broken.
  • Latency is milliseconds where the design predicted microseconds.
  • CPU on the firewall is higher than the traffic volume justifies.
  • Two identical boxes, same config, different performance.
  • An auditor asks you to prove something the dashboard already claims.
  • Performance degraded "overnight" with no change record.
The tell for this class of fault: nothing is down. Everything works. It just costs more than it should. Availability monitoring will never find it — only someone who goes looking.
Section Two

Questions to ask before you type

  1. Is this new, or newly visible? The single most important question. A defect present since build behaves very differently from a change last night.
  2. Has anyone been troubleshooting on this box recently? Most disabled acceleration is left over from someone's packet capture.
  3. What is the expected path? Which chip, which ports, which direction. If you can't state it, you can't tell whether the output is wrong.
  4. Am I being asked for a fix, or for evidence? Cipher's distinction. They require different outputs.
  5. If I fix this, whose standard am I changing? Crypto, inspection mode and policy structure all belong to somebody.
Section Three

Verification commands and expected output

Is forwarding accelerated?

diagnose sys session list
# HEALTHY
npu info: flag=0x81/0x91, offload=9/9, ips_offload=9/9, epid=128/166 ...

# NOT OFFLOADED — and FortiOS tells you why
npu_state=0x000001 no_offload
no_ofld_reason: disabled-by-policy

Read it as: 9 = NP7, 6 = NP6. Slash = originate / reply. Both directions need their own key — 9/0 means replies are on the CPU.

Is the tunnel offloaded?

diagnose vpn tunnel list name <tunnel>
stat: rxp=88214 txp=91003 rxb=12884901 txb=13221376   # ← is traffic FLOWING?
  enc: aes-256-cbc  auth: sha512                      # ← the WHY, if flag=20
  npu_flag=20 npu_rgwy=10.10.2.2 npu_lgwy=10.10.1.1
Always read rxp/txp alongside npu_flag. They are what makes 00 interpretable. Without traffic counts, 00 tells you nothing.

Confirming the human factor

Was it turned off deliberately?
# show vpn ipsec phase1-interface "HQ-to-Parramatta"
    ...
    set npu-offload disable          # default is enable — a human did this

# show ips global                     # np-accel-mode / cp-accel-mode
# show firewall policy 12             # np-acceleration / auto-asic-offload

Section Six

Likely root causes, ranked

  1. Leftover troubleshooting state. npu-offload disable, auto-asic-offload disable, np-accel-mode none — someone disabled it for a capture and never restored it. By far the most common.
  2. Unsupported hash chosen at build time. SHA-512 selected once by someone being careful. Never offloaded, never noticed.
  3. A proxy-based profile on the policy. One poisons the whole policy (Session 41).
  4. Stale sessions holding a decision made before the fix.
  5. Asymmetric path — one direction not qualifying.
  6. Chip resource exhaustion — NP session-table limit reached, overflow to CPU (Session 40).
◆
The pattern behind #1 and #2
Both are one-off human decisions that never expire. Acceleration faults are usually archaeology, not events.
Section Seven

Exam traps

  • Slash notation across tables. 9/9 is a session-list value. npu_flag is a single two-digit value. A distractor offering 03/03 is bait.
  • The ambiguous 00. Any stem giving npu_flag=00 without traffic information cannot be diagnosed. If a question does supply traffic counters, they are the point.
  • Two IPsec offload scopes. npu-offload = per phase-1 tunnel, ESP data path. ipsec-asic-offload = config system global, Diffie-Hellman.
  • Three "disable acceleration" commands. auto-asic-offload (policy, NP forwarding, the sniffing workaround) · np-acceleration (policy, NTurbo) · npu-offload (tunnel).
  • The CP8 exception. "What is the default cp-accel-mode?" — advanced, unless single CP8, where basic is the only mode.
  • IPSA visibility. Any answer claiming a session-table field proves CP pattern-match offload is wrong.
  • Cause vs remedy. If the stem asks "what is causing this", set npu-offload enable is not an answer.
Section Eight

The workflow, compressed

1. State the business symptom in one sentence.
2. Ask: new, or newly visible?
3. State the expected path before you look at any output.
4. Pull the table that records behaviour — session list or tunnel list.
5. Read the confession (no_ofld_reason / npu_flag) before hypothesising.
6. Separate can't from won't — they have different owners.
7. Fix, clear stale sessions, re-verify from the same table.
8. Explain what you could not prove, and what else must have changed.
◆
Step 8 is the senior step
Cipher explicitly credited the caveat about what the session table cannot show — "the part most engineers leave out." Stating the boundary of your evidence is not weakness. It is what makes the rest of it trustworthy.
