Fabric Deployment & Device Detection

Deployment order

Root first — it's a dependency. Downstream devices need three things only the root provides: a fabric that exists (Serve as Fabric Root), a fabric name, and a listening interface ("Allow other Security Fabric devices to join"). The root also requires FortiAnalyzer before the fabric can exist. Downstream: Join Existing Fabric + Upstream FortiGate IP/FQDN.

Two gates

1) Listening interface — internal-facing only; the join interface is attack surface, never the ISP port. 2) Authorization — a human on the root accepts each request. Membership = trust.
Menu trap: authorize on the root at System > Firmware & Registration (or via the alarm-icon badge) — not the Security Fabric menu. Hook: joining is registration → lives with Registration.

Device detection (passive, per-interface, access layer)

Enable on ISFW access ports — devices are born at the access layer; the edge sees shadows. Three passive leaks:
  • MAC OUI — vendor from the first 3 bytes (hardware DNA)
  • DHCP — hostname + option fingerprint (the introduction)
  • Traffic — DNS / SNI / User-Agent (the accent)
set device-identification enable      # per interface
diagnose user device list             # MAC, vendor, type, os, host, first-seen
"Devices cannot shut up about what they are."

Fabric logging (the Minerva rules)

First fabric FortiGate in the path owns the traffic log. Every NAT point upstream mints one more legitimate traffic log (NAT = new identity = new log — the Rosetta Stone). UTM logs live where inspection ran. FAZ correlates them all — and its history makes visibility retroactive (97 days back on day one).

FortiNDR & SAML SSO

Downstream files to FortiNDR transit through the root — FAZ = memory, root = spine. Inline blocking: AV first → hold session → ML verdict → release or kill.

SAML SSO: root = the only permitted IdP; downstream FGTs, FAZ, FMG = SPs; hop devices via top-banner list, one browser session. One fabric, one anchor, one voucher.

Business line

Day one of fabric visibility exposed Minerva — a rogue server that had been copying financial data to a personal cloud nightly for 97 days. State facts, not "could be": fact + duration + money category.
