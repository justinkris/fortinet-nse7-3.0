QUICK SUMMARY (one page)
------------------------------------------------------------------
Deploy ROOT FIRST — downstream devices join by pointing at a fabric
name, listening interface, and root IP that must already exist. The
root requires a FortiAnalyzer before the fabric can be enabled.
Two gates protect membership: (1) the join-listening interface —
internal-facing only, because the join door is attack surface —
and (2) human AUTHORIZATION on the root at System > Firmware &
Registration (alarm-icon badge shortcut). Membership is trust:
topology visibility, telemetry, SSO, automation.

Non-Fortinet devices can't join; the fabric learns of them via
PASSIVE device detection, enabled PER-INTERFACE at the ACCESS layer
(devices are born at the access layer; the edge sees shadows).
Identity leaks: MAC OUI (vendor/DNA), DHCP hostname + option
fingerprint (introduction), traffic chatter — DNS/SNI/User-Agent
(accent). Detected devices populate Physical/Logical Topology and
the asset inventory — which is how a 97-day rogue server (Minerva)
surfaced on day one of visibility, investigated backwards through
FAZ's log memory.

Logging rules (applied live): first fabric FGT in path owns the
traffic log; each upstream NAT point mints one more legitimate
traffic log (NAT = new identity = new log); UTM logs live where
inspection ran; FAZ correlates.

FortiNDR: downstream files TRANSIT THROUGH THE ROOT (spine), not
FAZ (memory). Inline blocking = AV first → hold session → ML
verdict → release/terminate.

SAML SSO: root = the ONLY permitted IdP; downstream FGTs, FAZ, FMG
register as SPs; hop devices from the top-banner list without
re-auth in the same browser session. One fabric, one anchor, one
voucher.

FLASHCARDS
------------------------------------------------------------------
Q: Which device must be configured first in a fabric, and why?
A: The root — downstream devices need an existing fabric name,
   listening interface, and root IP to point at.

Q: What must exist on the root before the fabric can be enabled?
A: A FortiAnalyzer (IP configured in Security Fabric Setup).

Q: Exact menu to authorize a downstream FortiGate?
A: On the ROOT: System > Firmware & Registration.

Q: Two root-side reasons a join request never appears?
A: (1) Awaiting authorization; (2) join-listening interface not
   enabled / wrong interface.

Q: Why restrict the join-listening interface?
A: Fabric membership is trust; listening on the ISP interface would
   let anyone on the internet knock. Attack-surface decision.

Q: Where does device detection belong, and why?
A: Access layer (ISFW), per-interface — closest to the source;
   the edge sees forwarded/NAT'd shadows.

Q: Three passive identity leaks used by device detection?
A: MAC OUI (vendor), DHCP hostname/option fingerprint, application
   chatter (DNS, TLS SNI, HTTP User-Agent).

Q: Is device detection active scanning?
A: No — entirely passive observation of traffic the device sends.

Q: Client → ISFW → NGFW(SNAT+webfilter) → Internet: traffic logs?
A: Two — ISFW (first fabric FGT owns the session log) and NGFW
   (SNAT = new identity = new log). UTM log(s) on NGFW.

Q: Downstream FGT sends a file to FortiNDR — via what?
A: Transits through the ROOT FortiGate (not FortiAnalyzer).

Q: FortiNDR inline blocking flow?
A: FGT AV first; if unknown, hold session → FortiNDR ML verdict →
   release if clean, terminate if malicious.

Q: SAML SSO fabric roles?
A: Root = only permitted IdP; downstream FGTs, FAZ, FMG = SPs;
   no re-auth within the same browser session.

Q: Where do pending fabric requests show besides the menu?
A: Number badge next to the alarm icon in the root's top banner.

Q: CLI to list detected devices?
A: diagnose user device list.

Q: (FMG retrieval) Does reinstalling a policy package keep the
   approved normalized-interface mapping?
A: YES — reinstall keeps the previously approved mapping.
