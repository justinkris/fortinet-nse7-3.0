TECHNICAL CONCEPTS COVERED
------------------------------------------------------------------
1. Root-first deployment order + WHY (downstream needs an existing
   fabric name + listening interface + reachable root to point at).
2. Root setup fields as SECURITY BOUNDARIES: join-listening
   interface = attack surface (never ISP-facing — anyone on the
   internet could knock); authorization = human trust gate.
   Two gates: where you can knock + who lets you in.
3. Downstream join config (role, upstream IP/FQDN, SAML option).
4. Authorization workflow: root, System > Firmware & Registration
   (+ alarm-icon badge). MENU-NAME TRAP: NOT under Security Fabric.
   Hook taught: "joining is registration, so it lives with
   registration."
5. Device detection: PASSIVE, per-interface, belongs at the ACCESS
   LAYER (ISFW). Artifacts: MAC OUI (hardware DNA), DHCP hostname +
   option fingerprint (the introduction), app-layer chatter —
   DNS/SNI/User-Agent (the accent). CLI: config system interface /
   set device-identification enable; diagnose user device list.
6. Topology views (Physical/Logical) on root; detected endpoints
   populate views + asset inventory.
7. Fabric logging correlation (pp. 262 drill, applied to Minerva):
   first fabric FGT in path owns the traffic log; NAT upstream =
   second legitimate traffic log (Rosetta Stone); UTM logs where
   inspection ran; FAZ auto-correlates.
8. FortiNDR use case (exam Q): downstream files to FortiNDR TRANSIT
   THROUGH THE ROOT (not FAZ). Inline blocking: FGT AV first; if
   clean-unknown, session held, FortiNDR ML verdict, release/kill.
9. SAML SSO in fabric: root = only IdP; downstream/FAZ/FMG = SP.
