Fabric Deployment & Rogue Devices

Scenario A — "I clicked Join and nothing happened"

Symptom

A downstream FortiGate is configured to Join Existing Fabric but never appears in the root's topology.

Questions before commands

  • Does the root exist yet? (Deployment order is a dependency — was it configured first?)
  • Which interface did the join request arrive on — and which interface is the root listening on?
  • Has anyone checked the authorization queue? Pending ≠ broken.

Decision path

1
Root's authorization queue first. System > Firmware & Registration (or the alarm-icon badge). If the device is listed as Pending — nothing is broken; a human simply hasn't said yes. This is the cheapest check, so it goes first.
2
Not in the queue? Check the root's listening interface. "Allow other Security Fabric devices to join" must include the interface the downstream request arrives on. Right door / nobody listening is silent.
3
Then downstream config. Upstream FortiGate IP/FQDN correct? Reaches the root's join interface (not some other interface of the root)?
4
Only then basic reachability. Ping/route from downstream to the upstream IP. Most people start here; the fabric-aware engineer ends here.
Dead ends: debugging routing for an hour when the request is sitting in the authorization queue; hunting "licensing" issues (not a taught join-failure cause); looking for the authorize button under the Security Fabric menu.

Scenario B — the unknown device (Minerva pattern)

Symptom

The topology map shows a device or switch nobody recognises.

Evidence before action — in this order

1
Where is it? Physical Topology → which FortiGate, which port. (Minerva: DCFW port4, behind an unmanaged switch.)
2
What is behind it? 
diagnose user device list
  → MAC (OUI = vendor), type, os, hostname, port, first-seen age
One MAC = one box. First-seen age is your incident timeline.
3
Who does it talk to? FortiAnalyzer history filtered on its IP — visibility is retroactive when you have memory. Read the logs with the ownership rules: first fabric FGT = traffic log; NAT upstream = second log (the Rosetta Stone to public-IP identity); UTM where inspected.
4
Correlate with the business. Asset register, HR/contractor records, change tickets. Technology tells you what; the business tells you who and why.
5
Then act — contain, preserve evidence, escalate. Never unplug first: you destroy live state and tip off an active adversary.
Senior behaviours: scope before touching; one hypothesis at a time; state confirmed facts as facts ("it HAS been exfiltrating for 97 days"), not hedges.

Scenario C — "The topology map is missing my endpoints"

  • Device detection is per-interface — is set device-identification enable on the actual access interface?
  • Is detection enabled at the access layer? Enabling it only at the edge shows shadows, not devices.
  • Detection is passive — a silent device won't appear until it speaks (DHCP renew, ARP, any traffic).

Exam traps

  • Authorization menu: System > Firmware & Registration — the Security Fabric menu is the distractor.
  • "Which device is required for the fabric?" → FortiAnalyzer (not FortiManager, not EMS).
  • FortiNDR file path from a downstream FGT → through the root (FAZ is the distractor).
  • Log-count questions → count the first FGT + every NAT point; UTM separately.
  • SAML: only the root can be IdP; FortiAuthenticator is not required.
