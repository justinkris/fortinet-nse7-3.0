Weaving It Together

Security Fabric deployment · authorization · device detection · the Minerva incident · SAML SSO

The fabric went live root-first; Dave's 14-day-pending branch FortiGate was finally verified and authorized. Then the topology map bloomed and revealed what nobody wanted: an unmanaged switch nobody bought on DCFW port4, and behind it Minerva — a contractor's Dell server that had been exfiltrating LoFinance financial data to a personal cloud account nightly for 97 days. Learning Objectives

  • Deploy a Security Fabric root-first and explain why the order is a dependency, not a ritual
  • Configure root and downstream Security Fabric Setup fields and treat each as a security boundary
  • Authorize downstream devices in the correct (counterintuitive) menu
  • Explain passive device detection: what leaks identity, and where detection belongs
  • Apply the fabric logging rules to a real investigation
  • Place FortiNDR file transit and SAML SSO roles correctly in the fabric trust model

1 · Root-First Deployment

A downstream FortiGate joins by pointing its Upstream FortiGate IP/FQDN at an existing root. Before the root exists, there is no fabric name to join and no interface listening for requests — a join attempt simply dies. The root defines three things everyone else depends on:

  1. That a fabric exists — role: Serve as Fabric Root
  2. The fabric's name — the identity downstream devices join
  3. The listening interface — Allow other Security Fabric devices to join

Root config — HQ-NGFW

Security Fabric > Fabric Connectors > Security Fabric Setup
  Security Fabric role:                        Serve as Fabric Root
  Allow other Security Fabric devices to join: port5   (internal-facing)
  Fabric name:                                 LoFinance-Fabric
  FortiAnalyzer IP:                            10.0.99.10

The root requires a FortiAnalyzer before the fabric can be enabled. The fabric refuses to exist without its memory. (Exam MCQ favourite.)

Downstream config — HQ-ISFW / HQ-DCFW / branch

Security Fabric > Fabric Connectors > Security Fabric Setup
  Security Fabric role:        Join Existing Fabric
  Upstream FortiGate IP/FQDN:  10.0.12.254
  SAML Single Sign-On:         (optional at join)

2 · Setup Fields Are Security Boundaries

"Every setup field is a security boundary wearing a text box costume."

Why must you pick a specific join interface instead of listening everywhere? Because the NGFW has an ISP-facing interface. If the fabric listened on every port, anyone on the internet with a FortiGate — or software pretending to be one — could send join requests to your trust perimeter. Fabric membership grants topology visibility, telemetry sharing, and eventually SSO and automation: it is a trust relationship, so the listening interface is an attack-surface decision.

The Two GatesGate 1 — the listening interface: where may you even knock? Gate 2 — authorization: even at the right door, a human must let you in. Defense in depth, applied to the fabric itself. Dave stood at the right door for two weeks.

3 · Authorization

Authorization always happens on the root. The pending request appears as a badge next to the alarm icon in the root's top banner, and the full queue lives at:

Menu trapSystem > Firmware & Registration — not under the Security Fabric menu, where every reasonable human first looks. Hook: joining the fabric is a form of device registration, so it lives with Registration. This exact path was missed twice in one session; drill it.

Before authorizing, verify — Cipher's one word. Confirm the serial against the asset register. Authorization is a trust decision, not a click.

4 · Device Detection

FortiGates join the fabric; TVs, printers, and rogue servers cannot. The fabric learns they exist through passive device detection — eavesdropping on traffic the device was going to send anyway.

LeakWhat it revealsMetaphor
MAC OUI — first 3 bytes of every frame's source MACManufacturer, before any IP existsHardware DNA
DHCP — hostname + option fingerprintSelf-declared name, OS/type patternThe introduction
Traffic chatter — DNS, TLS SNI, HTTP User-AgentWhat it is by what it talks toThe accent
"Devices cannot shut up about what they are."

Placement: detection belongs at the access layer (ISFW) — closest to the source. By the edge, traffic is forwarded, aggregated, possibly NAT'd: devices are born at the access layer; the edge sees their shadows. It is per-interface, not global:

config system interface
    edit "port3"
        set device-identification enable
    next
end

HQ-DCFW # diagnose user device list
vd 0  d8:9e:f3:aa:41:07  gen 42
    created 97d  seen 44s  port4
    hardware vendor 'Dell'  type 'Server'
    os 'Linux'  host 'minerva'

Detected devices populate the root's Physical / Logical Topology views and the asset inventory.

5 · The Minerva Investigation — Logging Rules Applied

Path: minerva → DCFW → NGFW (SNAT + web filter) → Internet. For one nightly session:

  1. DCFW writes the traffic log — the first fabric FortiGate in the path owns the session's traffic log; upstream members do not duplicate it.
  2. NGFW writes a second, legitimate traffic log — because SNAT gave the session a new identity. NAT = new identity = new log. That log is the Rosetta Stone joining private IP to public IP.
  3. NGFW writes any UTM logs — UTM logs live where inspection ran (web filtering here; DCFW's IPS faces inbound and stays silent).
  4. FortiAnalyzer auto-correlates them into one linked view — and because it held 97 days of history, the investigation could look backwards from day one of visibility.
Senior lessonEvidence before action: where is it connected, what MACs sit behind it, who does it talk to — then touch cables. Visibility is retroactive when you have memory.

6 · FortiNDR in the Fabric

Any fabric FortiGate can submit suspicious files to FortiNDR — but for downstream devices, files transit through the root FortiGate, not FortiAnalyzer. Inline blocking flow: FortiGate AV runs first; a signature match blocks locally. Unknown-but-suspicious files cause FortiGate to hold the user session, send the file to FortiNDR for ML/ANN analysis, then release (clean) or terminate (malicious) on the verdict. FortiNDR's own join request is accepted on the root like any member.

"FAZ is the fabric's memory. The root is the fabric's spine.
Logs flow to memory; fabric transit flows through the spine."

7 · SAML SSO in the Fabric

One login, then hop between fabric devices via the top-banner device list — no re-authentication within the same browser session.

RoleWho may hold it
Identity Provider (IdP)Root FortiGate only
Service Provider (SP)Downstream FortiGates, FortiAnalyzer, FortiManager

Why root-only? The fabric has exactly one trust anchor: the root defines the fabric, holds the listening interface, and authorizes every member. The device that vouches for identities must be the same device that decides membership. One fabric, one anchor, one voucher.

Common Mistakes (including today's)

  • Configuring downstream "Join" before the root exists — the request has nothing to point at.
  • Looking for authorization under the Security Fabric menu — it's System > Firmware & Registration.
  • Assuming the NGFW owns the traffic log because it's the edge — the first fabric FortiGate owns it.
  • Sending FortiNDR files to FAZ in your head — transit goes through the root.
  • Believing detection is active scanning — it is passive, and per-interface.
  • Hedging a confirmed incident ("could be exfiltrating") — Minerva did, for 97 days. State fact + duration + money category.

Exam Tips

  • Root requirements MCQ: FortiAnalyzer is required; FortiManager/EMS are not.
  • "Downstream joined but doesn't appear": check root's authorization queue and the join-listening interface before debugging IP reachability.
  • SAML distractors: "any FortiGate can be IdP" and "requires FortiAuthenticator" are both false.
  • Log-count scenarios: count NAT points +1 traffic log each; UTM logs sit where the profile ran.

Key Takeaways

Order of operations is a security property. Every setup field is a boundary. Detection is passive and lives at the access layer. First fabric FortiGate owns the traffic log; NAT mints a second. FAZ is memory, root is spine, and only the root vouches for identity. And on day one of visibility, LoFinance found a 97-day breach — which is the entire business case in one sentence.
