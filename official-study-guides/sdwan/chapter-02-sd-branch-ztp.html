Chapter 02 · NSE7 SD-WAN 7.6

Turning Up a Branch Without Touching It

How FortiManager, device blueprints, CSV imports, and FortiAnalyzer combine to onboard, provision, and monitor SD-WAN branches at scale.

A hundred branches, one shipping box at a time.

Zero-touch provisioning is what turns an SD-WAN rollout from a road trip into a spreadsheet. FortiManager holds the plan; the FortiGate arrives, phones home, and configures itself.

ZTP Overview Blueprints & CSV SD-WAN Monitoring
💭 THINK FIRST · ZTP OVERVIEW

An SD-WAN project rarely involves one FortiGate — it involves dozens or hundreds. Before we look at any GUI, ask yourself what the administrator actually does when a branch box is powered on for the first time, and how much of that work has to be done by hand.

  • What is the difference between zero-touch provisioning (ZTP) and low-touch provisioning (LTP), and where does each place the burden of initial configuration?
  • Which four onboarding methods does the study guide list, and what does each one rely on to tell the FortiGate where FortiManager lives?
Q1 — MODEL ANSWER
With ZTP, the administrator performs no configuration on the FortiGate itself. At power on, the device pulls its IP address, netmask, and gateway from a DHCP server, and DHCP option 240/241 tells it the FortiManager IP address and credentials. With LTP, a technician performs a small amount of manual configuration — typically the management interface IP, default route, and FortiManager IP with connection credentials — so that the FortiGate can reach FortiManager. In both flavors, FortiManager then installs the detailed configuration (routing, SD-WAN rules, firewall policies, and so on) on the device.
Q2 — MODEL ANSWER
The four onboarding methods are: ZTP using FortiCloud and FortiDeploy, where the FortiGate at factory default authenticates to FortiCloud with its serial number and is directed to FortiManager based on device registration data; ZTP using DHCP option, which requires no FortiCloud and relies on DHCP option 240/241 to hand over the FortiManager location; ZTP using USB boot, where the FortiGate reads its initial configuration from a USB stick; and LTP, where a site technician manually applies enough configuration to reach FortiManager and then triggers the connection using a serial number or pre-shared key.
SECTION 01

What Zero-Touch Provisioning Actually Promises

Zero-touch provisioning is best understood as a promise about the human, not the machine. A large SD-WAN project can involve hundreds of FortiGate branch devices, and the traditional model — an engineer opening a serial console, typing in a management IP, uploading a config, verifying the tunnels — simply does not scale. ZTP groups together a family of techniques and tools that let FortiManager push a detailed and personalized configuration to each FortiGate without anyone touching the device beyond unpacking it and plugging it in.

The study guide is careful to name two flavors. Zero-touch provisioning (ZTP) requires no preconfiguration of the FortiGate at all — the device boots on factory defaults, obtains its network settings automatically over DHCP, and is told where FortiManager lives either by FortiCloud or by DHCP options 240/241. Low-touch provisioning (LTP) requires a small amount of manual preparation: an administrator or site technician sets the management interface IP address, the default route, and the FortiManager IP and credentials, and only then does FortiManager take over and install the rest.

Underneath both flavors sits the same foundation. Three ingredients make ZTP and LTP possible: a device blueprint that describes what a given model of FortiGate should look like when it comes online; a set of provisioning templates — CLI, IPsec, BGP, SD-WAN, policy packages — that carry the actual configuration payload; and a CSV file whose rows list every device to be deployed together with the values of the metadata variables that personalize the templates for each site. The templates are written once; the CSV multiplies them across the fleet.

Choosing the onboarding method is the planning step that comes before any of this. ZTP using FortiCloud and FortiDeploy is the option for a FortiGate that has factory defaults and internet reachability through an external DHCP server — it authenticates to FortiCloud with its serial number and is redirected to the correct FortiManager. ZTP using DHCP option 240/241 does the same job without requiring FortiCloud connectivity, provided the DHCP server on the WAN interface has been configured with the FortiManager location. ZTP using USB boot pulls the initial configuration off a USB stick. LTP is the escape hatch for sites where none of those automated paths are practical.

MethodPreconfigurationHow FortiGate Finds FortiManager
ZTP · FortiCloud + FortiDeployNone (factory default)Authenticates to FortiCloud by serial number, receives FortiManager location from device registration data
ZTP · DHCP optionNone (factory default)DHCP server provides FortiManager location via option 240/241
ZTP · USB bootNone on FortiGate; config on USB stickInitial configuration read from USB at boot
LTPManual management IP, default route, FortiManager IP and credentialsAdministrator or technician configures FortiManager reachability by hand
Two ways to think of the trade-off: ZTP maximizes automation at the cost of requiring some infrastructure (FortiCloud reachability or a DHCP server that speaks option 240/241). LTP trades a small amount of on-site typing for independence from that infrastructure. Both end at the same place — FortiManager installing the personalized configuration.
🧠
MENTAL NOTE
"Zero touch" does not mean zero preparation. It means the preparation was done once, on FortiManager, before any hardware was shipped. The FortiGate stays untouched because the plan is already waiting for it.

💭 THINK FIRST · BLUEPRINTS & CSV

The ZTP workflow has three phases — preparation done once, staging where each site is declared, and deploying where technicians simply connect the boxes. Two objects do most of the work: the device blueprint that describes a model, and the CSV file that describes the fleet.

  • What does a device blueprint let you predefine, and why should you not assign both provisioning templates and a device group inside the same blueprint?
  • Which columns must a CSV file contain to import model devices into FortiManager, and what happens if the import contains a serial number that is already managed?
Q1 — MODEL ANSWER
A device blueprint predefines parameters per device model: Automatically Link to Real Device (auto-link, required for ZTP/LTP), Enforce Firmware Version, Enforce Device Configuration, Pre-Run CLI Template, Assign Policy Package, and Provisioning Templates. It can also add the device to a group. However, if you add a device to a group, it will inherit all templates associated with that group — so assigning provisioning templates and a group on the same blueprint risks conflicting configuration. You should choose one path or the other for a given blueprint.
Q2 — MODEL ANSWER
The required CSV columns are serial number, device blueprint, and name. For VM models, vm_interface_number is also required. For SD-WAN branches the file typically carries branch_id for the SD-WAN template, and any metadata variable names as additional columns. If a serial number in the CSV corresponds to a device already managed by FortiManager, the import triggers a warning and that line is ignored — which is how FortiManager prevents an accidental override of production configuration.
SECTION 02

Device Blueprints and CSV Imports Do the Heavy Lifting

The ZTP workflow diagram in the study guide splits the effort into three vertical bands. In the Preparation band, the administrator defines metadata variables, CLI templates, IPsec templates, BGP templates, firewall objects, SD-WAN templates, and policy packages — the entire configuration payload — bundled together as the SD-WAN overlay templates. This work is done once per SD-WAN deployment. In the Staging band, each site is provisioned on FortiManager by creating a device entry with its serial number and device name, either one at a time or, more realistically, by importing a CSV that lists every branch at once. In the Deploying band, technicians simply connect each site — FortiManager recognizes the FortiGate and loads the intended configuration on it automatically.

A device blueprint is the object that makes the staging step compact. A blueprint is a model-specific description — one for a FortiGate-40F branch, another for a FortiGate-VM64-KVM, another for a FortiGate-1800F hub — that stores the parameters FortiManager should apply to a device of that model as soon as it comes online. Per blueprint, the administrator can enforce a firmware version, add the device to a group, run a pre-run CLI template, assign a policy package, and assign one or more provisioning templates. The special toggle Automatically Link to Real Device (auto-link) must be enabled for ZTP or LTP to function at all — it is what lets FortiManager attach the blueprint to the physical device the moment it registers.

When a FortiGate powers on and reaches FortiManager, FortiManager checks whether a corresponding device model exists. If a blueprint matches, FortiManager authorizes the device, applies the firmware enforcement, and starts the configuration process. If more than one blueprint exists for the same device model, the CSV file's device blueprint column tells FortiManager which one to use per serial number. If no blueprint matches, the device is added to the unauthorized device list and waits for the administrator.

The parameter Enforce Device Configuration deserves its own paragraph. When enabled, it allows the auto-link process to change the configuration of the interface that the FortiGate uses to reach FortiManager — useful for setting an alias or interface speed on port1 during ZTP/LTP. When disabled, FortiManager will not touch that interface. The warning in the study guide is explicit: misconfiguring the interface used to connect to FortiManager — a wrong IP address or gateway — can break connectivity between the two devices and cause ZTP to fail outright.

Do not do both. Assigning provisioning templates and adding the device to a group inside the same blueprint invites conflicting configuration, because the device inherits all templates associated with that group in addition to the ones you named directly.

The CSV file is the multiplier. Building the file with columns for serial number, device blueprint, name, and any metadata variables lets an administrator bulk-import an entire fleet in a few clicks. The first columns must be serial number, device blueprint, name and — for VMs — vm_interface_number. Additional columns whose headers match metadata variable names populate the per-device variable values automatically. For SD-WAN branches, the file typically also carries branch_id, either supplied explicitly or generated by the auto-assign branch ID feature. For HA clusters the file carries cluster ID, cluster name, HA priority, and HA mode.

CSV FieldRequired?Purpose
serial numberRequiredIdentifies the FortiGate uniquely
device blueprintRequiredSelects which blueprint FortiManager applies to this device
nameRequiredHuman-readable device name in FortiManager
vm_interface_numberRequired for VM modelsNumber of ports to provision on the virtual FortiGate
branch_idRequired for SD-WAN templatePer-branch identifier consumed by the SD-WAN template

On file import, FortiManager sets metadata variable values for every predefined column and ignores columns that already correspond to previously defined variables. If the import detects a serial number that matches a device already in production, it warns the administrator and skips that line — a deliberate guardrail against overriding a live configuration. Devices imported by CSV are placed directly in the managed device list; when the FortiGate makes its first connection, it does not pass through the unauthorized list at all.

Sequence to memorize: Physically connect and power on the FortiGate → DHCP hands over the WAN IP, gateway, and FortiManager address → FortiGate connects to FortiManager → FortiManager auto-accepts, optionally enforces firmware, and pushes the configuration elements → the administrator confirms success in the FortiManager monitor panels.
🧠
MENTAL NOTE
A blueprint describes a model. A CSV row describes a device. Metadata variables bridge the two — the same template becomes a hundred different personalized configurations because a hundred different rows filled in a hundred different variable values.

💭 THINK FIRST · SD-WAN MONITORING

Once the fleet is deployed, monitoring is what turns a static configuration into an operational network. FortiManager offers Map, Table, and Template views of the SD-WAN estate; FortiAnalyzer aggregates the logs. Ask yourself what each view is best at surfacing before you go looking for a specific fault.

  • Which three SD-WAN monitor views does FortiManager provide, and what is each one designed to show?
  • What must be enabled to store SD-WAN member statistics beyond the default 10-minute window, and how do you cap that storage?
Q1 — MODEL ANSWER
FortiManager provides three views under SD-WAN Manager > Network > Monitor. The Map View plots devices geographically and color-codes interface status — hover for details, click for historical graphs. The Table View lists every SD-WAN device with the status of each member, upload and download volumes, and application traffic. The Template View is available only for topologies defined with the overlay orchestrator — it shows hubs as star icons and spokes as device icons, with green/red tunnel lines and green/orange/red device icons reflecting health-check outcomes.
Q2 — MODEL ANSWER
By default, FortiManager displays interface statistics for the past 10 minutes only. To store longer history you must enable the SD-WAN history monitor with config system admin setting / set sdwan-monitor-history enable. Once enabled, FortiManager stores data on disk for up to 180 days per managed SD-WAN device. To cap disk use, set rtm-max-monitor-by-days to a smaller number of days under the same config system admin setting stanza.
SECTION 03

Monitoring the Fabric from Manager to Analyzer

After ZTP delivers the branches, monitoring makes them useful. FortiManager offers a family of SD-WAN monitor views tailored to different questions. The Map View answers "where is the pain?" — it draws each device on a world map based on the longitude and latitude configured at device registration and paints its interfaces with a color code. Hovering over a member shows utilization details calculated from the interface's estimated-upstream-bandwidth and estimated-downstream-bandwidth; clicking a device opens historical graphs. The Table View answers "which member is doing what?" — a flat list of SD-WAN interfaces per device with link mode, upload and download volumes, and application traffic. The Template View answers "does the overlay match the design?" — available only for topologies built with the overlay orchestrator, it shows hubs as star icons, spokes as device icons, and tunnels as lines that go green when up and red when down. Device icons themselves turn green when all health checks pass, orange when some pass, and red when all fail.

By default, the historical graphs cover the past ten minutes. For real forensic work — noise complaints from a specific site last Tuesday, capacity planning across a quarter — the administrator enables the SD-WAN history monitor and, if disk space is a concern, caps the retention window. The two CLI knobs are simple and both live under config system admin setting.

FORTIMANAGER · ENABLE HISTORICAL SD-WAN MONITORING
config system admin setting
    set sdwan-monitor-history enable
    set rtm-max-monitor-by-days <nb of days>
end

Beyond passive monitoring, FortiManager exposes two active measurement tools. Cloud-assisted speed tests run from the SD-WAN monitor Map View against Fortinet-maintained cloud servers, but only for interfaces with the WAN role, on devices carrying a valid SD-WAN service bundle subscription — specifically the SD-WAN Underlay & Application Monitoring Service. Each interface is capped at ten speed tests per day; between tests, FortiGate serves the cached result. Because QoS policies and traffic shapers are generic, this course does not cover them directly — but the measured bandwidth column is where a designer looks first when tuning them. Hub-and-spoke speed tests are a CLI-only, license-free alternative that measures throughput between two FortiGates in a hub-and-spoke topology by configuring one as a speed-test server and the other as a client. The relevant snippets:

HUB-AND-SPOKE SPEED TEST · SERVER AND CLIENT
config system global
    set speedtest-server enable
    set speedtestd-server-port <integer> [default 5201]
    set speedtestd-ctrl-port <integer> [default 5200]
end

config system speed-test-schedule
    edit <interface>
        set dynamic-server {enable | disable}
        set ctrl-port <integer> [default 5200]
        set server-port <integer> [default 5201]
        set update-shaper {disable|local|remote|both}
    next
end

For hub-and-spoke tests to work at all, set allowaccess speed-test must be enabled on both the underlay and the IPsec overlay tunnel interfaces on each side. If the IPsec tunnel is configured with exchange-ip, speed-test access must also be enabled on the associated interface — including loopback interfaces. Optionally, set speedtest-bypass-routing enable tells SD-WAN to stop steering traffic through a member while it is participating in a speed test, so that the measurement is not polluted by unrelated flows.

Tunnel state is not just a read-only concept. On the VPN Monitor page under Device Manager > Monitors > VPN Monitor, enabling Show Table reveals every IPsec VPN tunnel created by IPsec templates and the SD-WAN overlay wizard. Right-clicking a link on the map, or selecting a row in the table, exposes Bring Tunnel Up and Bring Tunnel Down — a deliberate operational lever for forcing a tunnel state without touching the underlying configuration.

FortiAnalyzer is the piece that turns short-lived logs into a lasting record. Operating in Analyzer mode, it aggregates log data from every registered SD-WAN FortiGate and stores SD-WAN traffic logs, SD-WAN event logs, and periodic health-check SLA status logs. In the Log View > Traffic table, enabling the SD-WAN columns (SD-WAN Quality, SD-WAN Rule ID, SD-WAN Rule Name) reveals which SD-WAN rule matched each session and which member it was steered to — with rule ID 0 meaning the session used the default SD-WAN rule. In Log View > Event > SD-WAN, event logs record member state transitions such as a health-check port flipping from dead to alive or a service being disabled because no outgoing path remained.

The health-check SLA status logs are a separate category worth explicit attention. They are disabled by default and must be turned on via sla-fail-log-period and sla-pass-log-period on the performance SLA — or in the FortiManager GUI under SD-WAN Manager > Rules > Performance SLA > Advanced Options. Both values default to zero (no logs generated); setting them to a positive integer generates a log every N seconds while the member is failing or passing, respectively. Some of the FortiView graphs on FortiAnalyzer depend on these logs.

FortiView, finally, is the reporting surface. The Secure SD-WAN Monitor page delivers a per-device, per-time-range view with widgets for SD-WAN Bandwidth Overview, SD-WAN Performance Status, SD-WAN Rules Utilization (a Sankey graph), and SD-WAN Utilization by Application. The SD-WAN Summary page aggregates across the fleet — SD-WAN Health Overview, Top SD-WAN SLA Issues, Top SD-WAN Applications, Top SD-WAN Talkers. All widgets refresh automatically every five minutes; the manual refresh button in the upper right handles the impatient.

Where to look first: A single site behaving oddly — start in FortiManager's Map View. A rule not steering the way you expected — enable the SD-WAN columns in FortiAnalyzer's traffic logs. A member flapping — check the SD-WAN event logs and, if the underlying performance SLA has sla-fail-log-period/sla-pass-log-period set, the periodic health-check status logs.
