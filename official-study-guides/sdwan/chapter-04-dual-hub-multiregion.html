Chapter 04 · NSE7 SD-WAN 7.6

Scaling the Overlay Across Regions

How dual-hub redundancy, multiregion segmentation, and BGP self-healing turn a single-site SD-WAN into a resilient, geographically distributed fabric.

A single hub is a single story; a dual-hub, multiregion fabric is a network that keeps talking when a story ends.

This chapter maps the shapes of large SD-WAN deployments, the FortiManager knobs that make them practical, and the BGP tricks that let the overlay heal itself when a link degrades.

Dual-Hub Multiregion Region Routing
💭 THINK FIRST · DUAL-HUB

A single hub is a single point of failure — geographical, physical, and administrative all at once. Before you accept a second hub as the answer, ask yourself what kinds of redundancy it actually delivers, and what it costs in overlay count, IPsec state, and BGP peerings on every spoke.

  • What is the difference between a Primary & Secondary dual-hub deployment and a Primary & Primary deployment, and which one load-shares versus fails over?
  • Why does the study guide recommend IBGP inside a region and grouping overlays per device and location?
Q1 — MODEL ANSWER
In a Primary & Secondary dual-hub, each spoke establishes IPsec tunnels and BGP peerings with both hubs at all times, but the SD-WAN rules on the spoke assign a higher member cost to the secondary hub's interfaces. Under normal conditions the spoke steers to the primary; if the performance of the primary links degrades, the spoke falls back to the secondary. In a Primary & Primary deployment, both hubs handle traffic simultaneously — the spoke picks either hub according to SD-WAN rules and per-link performance, which produces load-sharing rather than fail-over.
Q2 — MODEL ANSWER
IBGP inside a region is recommended because it is more scalable than static routing and it lets FortiGate learn multiple equal paths through the overlay, which SD-WAN then treats as candidate members. Grouping overlays per device and per location keeps rule definitions manageable — the administrator writes one rule that references the "HUB1" zone rather than enumerating every underlay-specific tunnel — and it aligns with how the SD-WAN Manager overlay orchestrator names things so member cost can be applied consistently to every interface reaching a given hub.
SECTION 01

Two Hubs, One Region

The single-hub design that carries a small SD-WAN cleanly falls apart the moment the hub becomes a single point of failure for hundreds of branches. The Fortinet answer is to extend the base design with a second SD-WAN gateway, usually housed in a different building or a different city so a fibre cut, a power outage, or a maintenance window on one hub does not silence the whole overlay. Each gateway may be a standalone FortiGate or an HA cluster, and the two forms of redundancy compose — you can put an HA cluster at the primary hub and a standalone at the secondary, or run HA clusters on both, or add HA to spokes only at the most critical sites.

Operationally, dual-hub is a dial-up IPsec story. Each hub acts as a dial-up IPsec server on every available underlay, and every spoke builds a static IPsec tunnel toward each hub over each underlay. A spoke with two WAN uplinks and two hubs therefore carries four overlays — two to the primary and two to the secondary — and monitors the health of all four continuously. According to the SD-WAN rule definitions, the spoke steers traffic to the primary hub under normal conditions and falls back to the secondary based on link quality or hub availability. The hubs themselves communicate through the underlay — through the internet and the WAN links — and Fortinet does not require a direct hub-to-hub link, though the administrator is free to build one.

Dynamic routing is strongly preferred over static routes for these designs, and IBGP is the recommended protocol inside the region because it scales, propagates SD-WAN's health-influenced route selection, and pairs naturally with ADVPN when spoke-to-spoke shortcuts become desirable. ADVPN itself is a separate lesson, but the study guide is explicit that a geo-redundant dual-hub topology can be combined with ADVPN.

Two forms of redundancy, one topology. Geo-redundancy comes from placing the second hub in a different location; intra-site redundancy comes from an HA cluster — FGCP or FGSP. They are independent choices and can be mixed per site to match how critical each hub really is.

💭 THINK FIRST · MULTIREGION

One region can hold a lot of spokes, but the moment you draw a line across a map — East vs West, Americas vs EMEA, retail vs manufacturing — the design changes shape. Multiregion is where SD-WAN stops being a hub-and-spoke and starts being a fabric of hub-and-spoke blocks stitched together by EBGP and static hub-to-hub tunnels.

  • When should you consider a multiregion design over a very large single-region design, and what does each region actually contain?
  • What is the difference between the IPsec overlays inside a region and the IPsec overlays between regions?
Q1 — MODEL ANSWER
Consider a multiregion design when the deployment reaches thousands of sites, when traffic inside a region is significantly higher than traffic crossing regions, or when the sites correspond to geographical areas that need independent administration. Each region is a full hub-and-spoke block: it has its own SD-WAN topology, which can be single-hub or dual-hub, and each hub can be standalone or HA. Fortinet recommends dual-hub inside every region for redundancy. Regions are then interconnected in a full mesh between the regional hubs.
Q2 — MODEL ANSWER
At the region level the IPsec configuration mirrors a normal single-region deployment — hubs are dial-up IPsec endpoints for spokes, and every spoke builds a static IPsec tunnel to each hub over every available underlay. Between regions the picture flips: the hubs themselves build a full-mesh topology of static IPsec tunnels over each available underlay, with every hub establishing a tunnel over each transport to every other hub. There are no dial-up endpoints between regions — the hub-to-hub mesh is deterministic and static.
SECTION 02

A Region Is a Hub-and-Spoke Block

Multiregion is what the SD-WAN turns into once geography stops being a footnote. As the solution expands and the number of spokes climbs into the thousands, the administrator segments the fabric into regions — one per geographical area, or per business unit, or per any logical grouping that matches the expected traffic pattern. Inside each region, branches connect to the regional SD-WAN gateway; between regions, the regional gateways interconnect in a full mesh. When a user in one region needs to reach a workload in another, the traffic flows through the regional gateways rather than through a spoke-to-spoke shortcut.

The design test is straightforward: if traffic within a region is significantly higher than traffic between regions, multiregion is a fit. If the number of sites is modest and traffic patterns are flat, a single region — possibly with a very generous dual-hub — is usually simpler. Multiregion adds real complexity in BGP, in IPsec overlay counts on the hubs, and in FortiManager templates.

The IPsec design has two layers. At the region level, hubs act as dial-up IPsec endpoints for the spokes they serve. Each hub exposes a separate dial-up endpoint on each available underlay, and every spoke builds a static IPsec tunnel toward each hub over each underlay. Between regions, all hubs build a full-mesh topology of static IPsec tunnels — every hub establishes a tunnel to every other hub over every available underlay transport. Optionally, ADVPN can be enabled at the region level to allow intra-region shortcuts; if the design also needs inter-region ADVPN shortcuts, the forwarder mode must be enabled on each hub so discovery requests are forwarded across the region boundary.

Fortinet's example topology in the study guide names the pieces cleanly: a West Region with WEST-DC1 and WEST-DC2 as dual hubs (ASN 65001), an East Region with EAST-DC1 as its hub (ASN 65002), and a Multi-Regional SD-WAN/ADVPN fabric that carries EBGP between the two ASNs. It is common for the entire SD-WAN network to consist of a single region — but at thousands of sites, the multiregion design is the honest choice.

ADVPN caveat. By default, ADVPN operates within a region. If you want spoke-to-spoke shortcuts to cross a region boundary, you either use IBGP as the only routing protocol across the whole fabric, or you deploy an ADVPN 2.0 topology. Otherwise, inter-region traffic flows through the regional hubs.
🧠
MENTAL NOTE
Inside a region: dial-up IPsec, hub-and-spoke. Between regions: static IPsec, full mesh. Two topologies, one fabric.

Beyond the pure enterprise design, the study guide also frames these topologies as MSSP blueprints. An MSSP delivering SD-WAN as a managed service has three architectural choices for the hub: install a multi-tenant hub on MSSP premises with one VDOM per customer and one SD-WAN instance per VDOM; install a dedicated hub per customer on MSSP premises (typically a FortiGate VM in a cloud infrastructure); or place the hub on the customer's own premises — a central office or data centre — with only the management plane remaining on the MSSP side. In multi-tenant designs, ADOMs on FortiManager keep each customer setup isolated, and most customer traffic is either spoke-to-spoke through ADVPN shortcuts or direct internet access (DIA) on the spokes, which means the hubs themselves see very little data-plane traffic.

The related SD-branch concept fits naturally here: an SD-branch is an SD-WAN spoke site enriched with extensions — FortiSwitch and FortiAP on the LAN side, FortiExtender on the WAN side for 4G/5G. All of them run in managed mode, controlled by the local FortiGate through FortiLink, so from the management plane's point of view the switch ports, wireless SSIDs, and cellular WAN are simply logical interfaces on the spoke FortiGate. That lets microsegmentation, 802.1X, NAC, and security profiles reach the LAN with the same policy language used on the overlay.

💭 THINK FIRST · REGION ROUTING

Routing across regions is where BGP earns its keep. Inside a region you want IBGP for tight coupling to SD-WAN's health checks; between regions you want EBGP so you can summarize, apply policy, and keep each region's control plane loosely coupled. And once you have BGP, you can use it to let the overlay heal itself: spokes tell hubs which links are meeting SLA, and the hubs use that signal to steer return traffic.

  • Why IBGP within a region and EBGP between regions, and where does route summarization belong?
  • How does SD-WAN self-healing with BGP work — what do route-map-out-preferable and route-map-out actually advertise, and why does the hub not need SD-WAN configured?
Q1 — MODEL ANSWER
Assign a single BGP autonomous system to each region. Inside the region, every spoke advertises its LAN prefixes to the hubs using IBGP, and the hubs readvertise those routes and their own LAN prefixes back to the spokes. Between regions, configure EBGP peering between the hubs to exchange inter-region routes; each region's hub then readvertises the learned routes to its own spokes. Route summarization is possible between regions but not within a region — SD-WAN needs to know all available paths to a destination to select the best member for a rule, so intra-region routes are usually left unsummarized. If ADVPN is planned inside a region, use dynamic BGP; alternatively, configure every hub as a BGP route reflector for intra-region ADVPN so each spoke learns all paths to every other spoke in the region.
Q2 — MODEL ANSWER
SD-WAN self-healing uses BGP communities to inform BGP neighbors of link SLA status. On the spoke, route-map-out-preferable is applied when the SLA is met and route-map-out is applied when the SLA is missed — each map sets a different community, so the same prefix is advertised with a "good" community over a healthy overlay and an "avoid" community over an unhealthy one. The BGP peer at the hub does not need to be a FortiGate at all; it simply matches the received community in a route-map-in and adjusts route priority, metric, or tag accordingly. That means the hub steers return traffic through the healthier overlay without ever running its own SD-WAN health check, and because every spoke advertises with the same community strings, one set of service rules on the hub covers every branch — the solution scales natively.
SECTION 03

BGP Between the Blocks, and Self-Healing Across Them

Once the overlays are up, the devices must exchange reachability so every site can reach every other site. Fortinet's recommended protocol for these topologies is BGP — an industry-standard, flexible, feature-rich dynamic protocol that composes cleanly with SD-WAN's per-member health checks. Inside a region the choice is IBGP (with BGP on loopback or BGP per overlay); between regions the choice is EBGP, one autonomous system per region.

The design guidelines the study guide lists are worth memorizing as a checklist:

  • One AS per region. Assign a single BGP autonomous system to each region so the intra-region control plane stays clean.
  • IBGP from every spoke to its hubs to advertise LAN prefixes.
  • EBGP peering between hubs to exchange inter-region reachability.
  • Hubs readvertise everything they learn to their own spokes — inter-region routes plus their own LAN prefixes and destinations learned from outside the SD-WAN network.
  • Dynamic BGP for ADVPN inside a region, or route reflection on each hub as an alternative that lets every spoke learn all paths to every other spoke in the region.
  • Summarize between regions, not inside them. SD-WAN needs to see all available paths to a destination to select the best member for its rules, so intra-region summarization defeats the point; summarize the routes exchanged between the regions.
Scalability of the orchestrator. When FortiManager's overlay orchestrator builds BGP router IDs for the branches, it uses a metadata variable — good for up to 255 branches. Beyond 255, the orchestrator switches to a Jinja formula (ipmath(branch_id|int)) that increments the network part of a base router ID as the branch count grows. It also subdivides the overlay network for BGP per-overlay neighbor ranges: a class C overlay with two hubs and three overlays yields eight /27 subnets with 30 addresses each — capping the design at 30 spokes. A class B overlay accommodates more than 8,190 spokes, and the orchestrator defaults to 10.10.0.0/16 for exactly that reason.

On top of BGP, the study guide layers three SD-WAN-specific BGP options that matter most for dual-hub and multiregion designs:

  1. BGP route tags. SD-WAN rules can steer traffic to routes carrying a specific route tag. In FortiOS 7.4 and later, a firewall address object of type route-tag is used as an SD-WAN service destination — the community learned from a BGP neighbor is mapped to a route tag via a community-list and a route-map, and the resulting address object is what the SD-WAN service rule targets.
  2. SD-WAN self-healing with BGP. On the spoke, route-map-out-preferable is applied to BGP advertisements when the SLA is met, and route-map-out is applied when it is not. Each maps to a different community — a "good" community for healthy overlays and an "avoid" community for unhealthy ones — so BGP neighbors, whether FortiGates or not, learn which overlay to prefer for return traffic without ever running their own health check.
  3. Route mapping according to SLA status with route-map-out-preferable and minimum-sla-meet-members. The latter refines behavior when multiple SD-WAN members feed the same BGP neighbor: by default, route-map-out-preferable is applied when at least one member meets the SLA; setting minimum-sla-meet-members 2 requires two members within SLA before the preferable route map is used.

The mechanics of self-healing follow directly. The spoke associates each overlay with a community — say 65000:11 for tunnel T1 preferred, 65000:21 for tunnel T2 preferred, 65000:50 when no SLAs are met — and advertises the same prefix with the community that matches the current SLA state. On the hub, a community-list matches each incoming community and a route-map sets a corresponding priority (lower is preferred): 11 for the primary healthy path, 21 for the alternate healthy path, 50 for the "avoid" community. Standard BGP path selection then routes hub-initiated traffic through the overlay with the lowest priority — and the whole thing works even if the hub is a third-party router with no SD-WAN awareness.

SPOKE — BGP self-healing route maps
config router route-map
    edit "comm1"
        config rule
            edit 1
                set match-ip-address "net192"
                set set-community "65000:11"
            next
        end
    next
end

config router bgp
    config neighbor
        edit "172.31.0.1"
            set route-map-out "comm5"
            set route-map-out-preferable "comm1"
        next
        edit "172.31.0.2"
            set route-map-out "comm5"
            set route-map-out-preferable "comm2"
        next
    end
end

For multi-VRF deployments — the segmentation over single overlay use case — FortiGate supports VRF-aware overlays. On the control plane, MP-BGP VPNv4 advertises VRF information alongside each LAN prefix; on the data plane, traffic is tagged using vpn-id-ipip encapsulation (a variant of IPIP with the VRF ID encoded into the extra IP header inside the ESP payload) so a single IPsec tunnel can carry multiple segmented flows without collapsing their separation. Health checks can also be made VRF-aware: set vrf and set source under config health-check let the spoke tag probes with the correct VRF so the hub forwards them to the right health-check server.

Finally, the hub-to-edge SD-WAN problem — how the hub steers sessions initiated behind it toward the healthier overlay to a given spoke — has two supported answers. Remote health probing lets the spoke embed SLA measurements (latency, jitter, packet loss) inside the ICMP probes it sends over each overlay; the hub reads those measurements and applies priorities to IKE routes (set priority-in-sla and set priority-out-sla) so lower priorities go to overlays inside SLA. Remote health probing requires active-mode ping health checks on the spokes, works with only one SLA criterion per hub, and pairs naturally with BGP-on-loopback or static routing (with static IP addressing for tunnels, since it is not compatible with mode-cfg). The alternative is BGP self-healing, described above, which distributes the same signal through communities rather than through embedded probes.

Neighbor health monitoring. An SD-WAN neighbor can be monitored by SLA link quality (set mode sla, one or more members per neighbor) or by speed-test status (set mode speedtest, exactly one member per neighbor, using hub-and-spoke or cloud-assisted speed tests). Only SLA mode supports multiple members per neighbor.
🧠
MENTAL NOTE
The spoke speaks the truth about link health; BGP communities carry that truth to the hub; the hub — SD-WAN or not — obeys standard BGP path selection. That is the whole self-healing story.
