Chapter 05 · NSE7 SD-WAN 7.6

Shortcut Tunnels on Demand

How ADVPN quietly turns a hub-and-spoke overlay into a full mesh whenever two branches actually need to talk.

Every efficient overlay is a compromise between mesh and hub.

A pure mesh is fast but unmanageable; a hub is manageable but slow. ADVPN chooses both — hub for control, mesh for the packets that need to move.

ADVPN Concepts SD-WAN + ADVPN Design ADVPN Configuration
💭 THINK FIRST · ADVPN CONCEPTS

A hub-and-spoke IPsec overlay is cheap to build and easy to reason about, but every spoke-to-spoke conversation has to walk through the hub. In a global network with geographically distant branches, that detour becomes latency, jitter, and load on a single device. ADVPN keeps the hub in charge of who-can-talk-to-whom while letting the packets themselves take a straight line between spokes.

  • What role does the hub actually play in a shortcut negotiation, and why is that role indispensable even though the shortcut bypasses the hub for data?
  • Why did Fortinet introduce ADVPN 2.0 rather than continuing to enhance ADVPN 1.0, and what does ADVPN 2.0 move off the hub and onto the spokes?
Q1 — MODEL ANSWER
The hub is the matchmaker. When spoke1 sends the first packet toward spoke2, that packet still traverses the hub via the parent tunnels. The hub recognises that a more direct path is possible and sends a shortcut-offer to spoke1. Spoke1 replies with a shortcut-query, which the hub forwards to spoke2; spoke2 replies with a shortcut-reply containing its public IP and an auto-generated PSK (or digital certificate authentication), which the hub forwards back to spoke1. Only then do the two spokes initiate their own IKE negotiation directly. Without the hub facilitating that IKE information exchange, neither spoke would know the other's public IP or the shared key needed to build the on-demand tunnel.
Q2 — MODEL ANSWER
ADVPN 1.0 was tied to specific BGP designs — route reflection with next-hop convergence, BGP per overlay, BGP on loopback — and the hub carried heavy processing to reflect routes to thousands of spokes. It also gave the initiator no visibility into the remote spoke's link performance, so shortcut endpoints reused the same interfaces as the initial flow. ADVPN 2.0, introduced in FortiOS 7.4.2, is natively integrated with SD-WAN and pushes edge discovery and path management down to the spokes. The initiator now learns the remote spoke's WAN link IP address, transport group, link quality, cost, and member order via an expanded shortcut-reply, and calculates the best shortcut path itself, per SD-WAN service or rule. It removes the dependency on a specific BGP design and adds resiliency against underlay outages and degraded performance.
SECTION 01

ADVPN in Concept — Shortcuts Without a Full Mesh

ADVPN — Auto-Discovery VPN — is a Fortinet-proprietary framework built on top of standard IKE and IPsec. Its purpose is stated simply on the overview slide: enable direct spoke-to-spoke communication in a hub-and-spoke topology, without the cost or the configuration overhead of a full mesh. A pure hub-and-spoke topology forces every spoke-to-spoke flow through the hub, which adds latency and can hurt user experience when hub and spokes are geographically distant. A full mesh removes the detour but multiplies configuration and cost. ADVPN gives you the benefits of a full-mesh topology in a hub-and-spoke network.

The mechanism is a well-choreographed IKE exchange. Overlays between a spoke and the hub are called parent tunnels; direct tunnels negotiated between spokes over those parents are called shortcuts. Shortcuts are created on-demand from the IPsec phase-1 auto-discovery settings on each peer: auto-discovery-receiver enabled on the spoke overlay tells the hubs that this spoke can negotiate shortcuts, and auto-discovery-sender enabled on the hub overlay that connects the spoke allows the hub to broker shortcut negotiation between spokes. In a dual-region topology with two hubs, auto-discovery-forwarder must be enabled on the overlay between the two hubs so ADVPN information can flow across regions.

The IKE message sequence that establishes a shortcut is important to memorise. The initial packets flow through the hub in the ordinary way. The hub notices that both peers have ADVPN enabled and sends a shortcut-offer to spoke1. Spoke1 acknowledges the offer with a shortcut-query, which the hub forwards to spoke2. Spoke2 acknowledges the query with a shortcut-reply containing its public IP address, its local subnet, the desired destination subnet, and an auto-generated PSK (digital certificate authentication is also supported). The hub forwards the reply back to spoke1, and only then do the two spokes initiate a direct IKE negotiation. After routing convergence, spoke-to-spoke traffic no longer transits the hub.

Requirements at a glance: ADVPN needs a routing protocol on the overlay — Fortinet recommends IBGP, but OSPF, RIPv2/RIPng, and static routing are supported. PIM and multicast work. NAT is supported for on-demand tunnels. Both IPv4 and IPv6 are supported for ADVPN 1.0; ADVPN 2.0 supports IPv4 only.

Fortinet introduced ADVPN 2.0 in FortiOS 7.4.2 to address ADVPN 1.0's scale limits. ADVPN 1.0 was heavily dependent on BGP mechanisms (route reflection, BGP next-hop recursive resolution, IPsec phase-2 selectors, BGP per overlay, BGP on loopback) and it did not scale gracefully into networks with thousands of branches. It also gave the initiator no visibility into the remote spoke's WAN links — the shortcut always reused the same interfaces as the original flow, forcing administrators to configure overlay-stickiness policy routes to avoid impossible cross-overlay shortcuts. ADVPN 2.0 is natively designed for SD-WAN, supports very large topologies, limits hub involvement, improves underlay selection for tunnel establishment (using overlays available on both spokes with matching link performance), and improves resiliency against underlay outages and degraded performance.

The heart of ADVPN 2.0 is edge discovery and path management. Edge discovery expands the IKE shortcut-reply so the initiator spoke learns the remote spoke's IP address, transport group, link quality, link cost, and member configuration order for each remote WAN link. Path management, performed on the initiator, calculates the best shortcut path per SD-WAN service or rule and advises IKE to establish the shortcut using the selected interface pair — which may differ from the interface used for the initial data. After the shortcut is up, the spokes exchange link status every 5 seconds over the shortcut using UDP port 4684, for best-quality or lowest-cost (SLA) rules. This exchange is not configurable; it happens automatically for those rule strategies.

Phase-1 SettingWherePurpose
auto-discovery-receiverSpoke overlayTells the hub this spoke can negotiate shortcuts
auto-discovery-senderHub overlay to spokeAllows the hub to broker shortcut negotiation between spokes
auto-discovery-forwarderHub-to-hub overlayForwards ADVPN sender/receiver info between hubs (dual region)
auto-discovery-crossoverHub (block)Prevents shortcuts between overlays with different network IDs

Shortcut endpoint selection differs sharply between the two generations. With ADVPN 1.0, the shortcut endpoints simply mirror the interfaces used by the initial flow through the hub, and the administrator must add overlay-stickiness policy routes to prevent shortcuts that would try to bridge incompatible transports (for example, an ISP overlay to an MPLS overlay). With ADVPN 2.0, the spoke picks the interface based on link performance information received from the remote spoke, different SD-WAN rules can trigger different shortcuts, and the spoke can reevaluate performance and negotiate a new shortcut. Overlay stickiness is no longer relevant; instead, administrators assign a transport group ID to each link to differentiate compatible from incompatible interfaces.

When a spoke detects that the SLA on the member carrying the current shortcut is no longer met, it does not simply give up. It sends ADVPN control packets — UDP trigger packets — to the hub on UDP port 4684, sourced from its loopback IP toward the hub's loopback IP. The hub responds by sending a fresh shortcut-offer to trigger the negotiation of a new shortcut on a better link. That interaction is what turns ADVPN 2.0 into a resilient runtime system rather than a one-shot tunnel builder.

MENTAL NOTE
The hub matchmakes; the spokes marry. In ADVPN 1.0 the hub also chose the venue. In ADVPN 2.0 the spokes pick the venue themselves — using SLA information exchanged every 5 seconds over UDP/4684.

💭 THINK FIRST · SD-WAN + ADVPN DESIGN

The original ADVPN design used one BGP peer per overlay, route reflection on the hub, and prefix redistribution. It was easy to explain but it degraded as branches multiplied. Fortinet's later releases added several routing overlay designs — BGP on loopback, IPsec selector with static routing, dynamic BGP — each of which trades off scalability, simplicity, and the reliance on route reflection differently.

  • Why does the traditional BGP-per-overlay design scale poorly, and what does BGP on loopback change to make it scale better?
  • What problem do transport groups and overlay placeholders solve in an ADVPN 2.0 SD-WAN design?
Q1 — MODEL ANSWER
In BGP-per-overlay each branch establishes a BGP peer for every overlay it has with the hub, and the hub must reflect routes to every peer. Route reflection is a "square problem" — O(n²) — because the number of routes to reflect equals routes multiplied by peers. The hub's CPU and the spokes' unnecessary route-learning both explode as branches increase. BGP on loopback collapses each branch into a single IBGP session with the hub, terminated on loopback interfaces at both ends, and lets FortiGate resolve routes recursively through all available overlays. Only one BGP route is generated per LAN prefix, so BGP add-path is no longer required, and the hub reflects a much smaller number of routes between a much smaller number of peers.
Q2 — MODEL ANSWER
Transport groups differentiate WAN interfaces that are compatible for shortcut establishment from those that are not — for example, ISP interfaces versus MPLS. Members inside the same transport group can build shortcuts between each other; members in different groups cannot. By default all members sit in transport group zero, so shortcuts are allowed everywhere; the administrator adds distinct group IDs to prevent inter-group shortcuts. Overlay placeholders solve the mirror problem: two spokes may share a direct MPLS connection to each other but have no MPLS connection to the hub. A placeholder phase-1 interface with auto-discovery-dialup-placeholder enable and no static remote gateway lets ADVPN 2.0 build a shortcut over the MPLS transport between the two spokes while the regular parent tunnels continue to run over broadband to the hub.
SECTION 02

Designing the Overlay Around ADVPN and SD-WAN

The original SD-WAN + ADVPN design used a single routing protocol — BGP — on the overlay, one BGP peer per overlay, route reflection on the hub, and prefix redistribution. That design remains easy to understand and easy to configure, and it still fits small to mid-size topologies. Its drawbacks appear at scale: it is not suitable for topologies with thousands of branches, the hub sees a high processing load reflecting routes to spokes, and the spoke routing tables grow exponentially as branches are added. Later FortiOS versions introduced alternatives — BGP on loopback, IPsec selector with static routing, BGP without route reflection and IPsec selector, and dynamic BGP — that improve scalability and remove or limit the reliance on route reflection.

BGP on loopback is the most consequential alternative. Each branch establishes a single IBGP session with the hub, terminated on loopback interfaces at both ends, regardless of how many overlays exist between them. FortiGate resolves the routes recursively through all available overlays, and generates and propagates a single BGP route per LAN prefix. Route reflection remains a "square problem" — O(n²), where the number of routes to reflect equals the routes multiplied by the peers — so reducing both peers and routes significantly improves the scalability of the route reflector. Because only a single BGP route is generated per LAN prefix, BGP add-path is no longer required and the hubs do not inflate the route count when reflecting between spokes or advertising to other regions.

Edge discovery and path management, introduced with ADVPN 2.0, are the design's most visible new features. Edge discovery expands the IKE shortcut-reply so the local spoke — the initiator — learns the remote spoke's WAN link information: IP address, transport group, link quality, link cost, and member configuration order. Path management runs in the background on the local spoke, monitors WAN link information available for local and remote overlays, calculates the best shortcut path per SD-WAN service or rule, and advises IKE to establish (or reestablish) a shortcut using the selected path. After the shortcut is established, both spokes exchange members' transport group, link quality, and link status every 5 seconds over the shortcut using UDP traffic on port 4684 — automatically, for shortcuts associated with best-quality or lowest-cost (SLA) rules.

Shortcut endpoint selection — ADVPN 2.0: SD-WAN controls the shortcut negotiation. The spoke initiates. The interface used for the shortcut can differ from the interface of the overlay used for the initial data packets sent through the hub. Different SD-WAN rules can trigger and use different shortcuts. The spoke can reevaluate performance and negotiate a new shortcut if the link degrades. Overlay stickiness — the workaround required in ADVPN 1.0 — is no longer relevant, but administrators can assign a transport group ID per link to keep incompatible interfaces (for example ISP vs MPLS) from being paired.

Once a shortcut is up, it must be maintained. The spokes exchange health-check information — UDP on port 4684 — every 5 seconds, containing link status, link quality, and transport groups for all members. When the SLA on the current shortcut member degrades, the spoke sends UDP trigger packets to the hub on the same port, sourced from its loopback IP toward the hub's loopback IP; the hub then sends a shortcut-offer to trigger the negotiation of a new shortcut. Both settings need to be permitted end-to-end and on any intermediate devices.

Three configuration levers govern the shortcut lifecycle. Idle timeout (set idle-timeout enable with set idle-timeoutinterval N) brings down a shortcut after N minutes of no user traffic — health-check traffic does not count as user traffic. Dependent lifetime (set auto-discovery-shortcuts dependent) tears the shortcut down immediately when its parent goes down, which saves resources and speeds up routing convergence; the default is independent. Hold-down time on the SD-WAN service (set hold-down-time 20) delays failback to a recovered member — including recovered shortcuts — so the SLA is verified before traffic is steered back. Fortinet strongly recommends configuring at least 20 seconds on SD-WAN + ADVPN deployments using lowest-cost (SLA) rules.

For dual-region designs, network IDs let you either restrict or multiply shortcuts between the same pair of endpoints. Assigning the same network ID to two overlays that share a physical transport (both on ISP) allows shortcuts between them, and different IDs (say, MPLS at ID 2) prevent physically impossible shortcuts when set auto-discovery-crossover block is enabled on the hub. Setting a different network ID per overlay also allows multiple shortcuts between the same pair of local and remote gateway IPs — one via hub1, one via hub2, both up at the same time. The feature requires IKEv2. When you use FortiManager's SD-WAN overlay template, network-overlay is enabled and a different network-id is defined per overlay automatically.

Two more design tools deserve attention. Overlay placeholders allow ADVPN 2.0 to build shortcuts over transports that are not connected to the hub — the classic case being two spokes with a direct MPLS link between them where neither has MPLS to the hub. A placeholder phase-1 interface with set type dynamic, set net-device enable, and set auto-discovery-dialup-placeholder enable is defined with no static remote-gw; SD-WAN treats it as if it were a normal overlay, and shortcuts can be established between the spokes over that placeholder while the regular parent tunnels continue to run over broadband to the hub. Health checks run on both the regular shortcut and the placeholder shortcut, so SD-WAN can pick the interface that best meets the SLA.

Dynamic BGP is the other alternative: BGP neighbor connections between spokes are formed only after an ADVPN shortcut is established between them. The hub learns only the summary route from each spoke, so it no longer needs to reflect routes at all; the spokes learn routes solely from the BGP neighbours they actually talk to. Dynamic BGP is compatible with BGP per overlay and BGP on loopback, and works for both IPv4 and IPv6. It removes the requirement for route reflection on the hub entirely.

💭 THINK FIRST · ADVPN CONFIGURATION

Configuration for ADVPN falls neatly into two halves. The IPsec half sets the auto-discovery options and, on hub, disables net-device while on spokes it must be enabled with allowaccess ping. The SD-WAN half — new with ADVPN 2.0 — enables advpn-select on the zone and defines transport groups per member. FortiManager can do all of this via the SD-WAN overlay template.

  • Why must net-device be disabled on the hub but enabled on the spokes, and why does the spoke overlay need set allowaccess ping?
  • What does set advpn-select enable at the SD-WAN zone level actually turn on, and how do transport groups plug into it?
Q1 — MODEL ANSWER
On the hub, net-device is disabled so that the hub does not create a network device per dial-up child tunnel — the hub aggregates many spokes on one parent interface, and net-device would multiply resources unnecessarily. On the spoke, net-device must be enabled (it is disabled by default) so that the spoke has a proper interface for each overlay; SD-WAN needs that interface identity to run health checks and steer traffic. The set allowaccess ping line is required on the spoke because SD-WAN sends ping probes to the overlay IP address of the remote spoke to monitor the shortcut's performance and health — if ping is not allowed on the interface, the probes fail and the shortcut is marked dead.
Q2 — MODEL ANSWER
set advpn-select enable at the SD-WAN zone level turns on ADVPN 2.0 for that zone — it tells SD-WAN that this zone should participate in edge discovery, path management, and the enhanced shortcut-reply exchange. Once enabled, you can optionally define a transport-group for each SD-WAN member (default is zero, which means shortcuts are unrestricted). Different group IDs prevent ADVPN 2.0 from negotiating shortcuts between members in different groups — the standard use is to segregate ISP-connected members (group 1) from MPLS-connected members (group 2) so no impossible cross-transport shortcut is attempted.
SECTION 03

Configuring, Monitoring, and Debugging ADVPN in Practice

ADVPN configuration divides cleanly into two parts — an IPsec part that is similar to the original ADVPN and an SD-WAN part that is new with ADVPN 2.0. On the IPsec side, the phase-1 interface holds the auto-discovery commands: auto-discovery-receiver enable on the spoke overlay, auto-discovery-sender enable on the hub overlay that faces spokes, and auto-discovery-forwarder enable on the hub overlay that faces the other hub (dual-region). On the SD-WAN side, configuration is applied on every spoke: set advpn-select enable at the SD-WAN zone level activates ADVPN 2.0 for that zone, and (optionally) a transport-group is defined per member. By default every member is in transport group zero, so shortcuts are allowed between all of them. Backward compatibility is preserved — an ADVPN 2.0 branch that receives a shortcut-reply without WAN link details falls back to the ADVPN 1.0 method for that peer only.

Hub — Phase-1 (SD-WAN overlay design)
config vpn ipsec phase1-interface
    edit "VPN1"
        set net-device disable
        set auto-discovery-sender enable
    next
end
Spoke — Phase-1 and interface
config vpn ipsec phase1-interface
    edit "HUB1-VPN1"
        set net-device enable
        set auto-discovery-receiver enable
    next
end
config system interface
    edit "HUB1-VPN1"
        set allowaccess ping
    next
end

Note that net-device is disabled on the hub and enabled on the spoke — it is disabled by default, so the spoke line must be added explicitly. Ping must be allowed on the spoke overlay because SD-WAN sends ping probes to the overlay IP address of the remote spoke to monitor the shortcut. Without allowaccess ping, the health checks fail and the shortcut is marked dead.

FortiManager SD-WAN overlay template — ADVPN 1.0 defaults: Selecting Auto-Discovery VPN on the overlay template makes FortiManager set add-route disable, net-device disable, and auto-discovery-sender enable on the hub IPsec template; additional-path enable on the hub BGP template; add-route disable, net-device enable, and auto-discovery-receiver enable on the spoke IPsec template; and recursive-next-hop enable, additional-path enable on the spoke BGP template. FortiManager 7.4 and earlier build ADVPN 1.0 via this template; ADVPN 2.0 must be enabled explicitly.

Transport groups are defined under config system sdwan in the members list: each member gets a transport-group number, and shortcuts are negotiated only between members that share the same group. A typical pattern is ISP interfaces in group 1 and MPLS in group 2, so ADVPN 2.0 never tries to bridge the two. Overlay placeholders extend this by allowing shortcuts on transports that don't reach the hub: a placeholder phase-1 interface (set type dynamic, set net-device enable, set auto-discovery-dialup-placeholder enable) is defined with no static remote-gw, and the placeholder tunnel is added as an SD-WAN member with the appropriate transport group.

SD-WAN — advpn-select and transport groups
config system sdwan
    set status enable
    config zone
        edit "overlay"
            set advpn-select enable
            set advpn-health-check "HUB1_HC"
        next
    end
    config members
        edit 4
            set interface "H1_T11"
            set zone "overlay"
            set transport-group 1
        next
        edit 13
            set interface "Placeholder_MPLS_1"
            set zone "overlay"
            set transport-group 2
        next
    end
end

Monitoring is done with a small set of diagnostic commands. On a local spoke, diagnose sys sdwan advpn-session displays the WAN link information received from the remote spoke — remote spoke device name and zone, the selected path for the shortcut, and the remote link performance details refreshed every 5 seconds. The command produces no output for shortcuts that do not require SLA monitoring; it only shows shortcuts associated with best-quality or lowest-cost (SLA) rules. The command get router info routing-table all and get vpn ipsec tunnel summary reveal how the routing table changes as the shortcut is established: the recursive lookup for the remote spoke's LAN prefix shifts from the parent tunnel (for example HUB1-VPN1) to the shortcut child (HUB1-VPN1_0), and a new static route to the remote spoke's overlay IP (say 192.168.1.2/32) appears as directly connected on the shortcut interface.

Logs identify shortcut tunnels with the advpnsc field — set to 1 for shortcuts and 0 otherwise. To filter for ADVPN-only messages on the FortiGate CLI: execute log filter category event then execute log filter field advpnsc 1. On FortiAnalyzer, activate the AD-VPN Shortcut column in Log View > Event > VPN to view the flag. FortiManager's VPN Monitor page shows shortcut tunnels graphically on a map and in a table with bandwidth statistics. FortiAnalyzer's Secure SD-WAN Monitor displays an expand button next to any interface with one or more shortcuts; expanding it reveals shortcut tunnel details.

Debugging shortcut negotiation uses the IKE real-time debug filtered on multiple IP addresses. diagnose vpn ike log filter mrem-addr4 <hub-ip> <spoke-ip> lets you print IKE debug for several IPv4 destinations at once (mrem-addr6 for IPv6). The debug output on the hub shows SHORTCUT_OFFER, then SHORTCUT_QUERY received from spoke1 and forwarded to spoke2, then SHORTCUT_REPLY received from spoke2 and forwarded to spoke1. On spoke1 you will see the received SHORTCUT_REPLY, followed by "adding new dynamic tunnel" and finally the SNMP tunnel-UP trap. The child dial-up tunnel on the hub and the shortcut tunnel on the spoke use the same name convention — HUB1-VPN1_0 — but represent different things: on the hub, it is the child dial-up tunnel connecting hub to spoke1; on spoke1, it is the shortcut to spoke2.

Common gotcha: Idle shortcuts inherit the parent tunnel's lifetime by default, so a shortcut with no user traffic can linger for hours consuming resources. Set idle-timeout enable and idle-timeoutinterval N on the spoke phase-1 to bring down idle shortcuts. Health-check traffic does not count as user traffic. Separately, set auto-discovery-shortcuts dependent tears down shortcuts immediately when the parent goes down — the default (independent) leaves them up, which can slow routing convergence and waste resources.

ADVPN can also be layered with alternative BGP designs. For ADVPN with BGP on loopback, the hub's phase-1 uses set ike-version 2, set net-device disable, set exchange-ip-addr4 <hub-loopback>, set add-route disable, and set auto-discovery-sender enable; the hub's BGP enables ibgp-multipath and recursive-next-hop, and each neighbor-group sets next-hop-self enable, interface loopback, and update-source loopback. A firewall policy from the IPsec zones to the loopback zone must be defined so BGP traffic can reach the loopback IP. On the spoke, set tag-resolve-mode merge and set recursive-next-hop enable are required so the next-hop resolves both through the best-match route and through same-tag routes at the same time; each hub is marked with a different tag via a route-map set-tag in the incoming direction.

For ADVPN with dynamic BGP, the hub enables the usual ADVPN sender settings plus BGP neighbor-group and neighbor-range, and route reflection is disabled globally (client-to-client-reflection disable) or per neighbor-group (route-reflector-client disable). The hub announces the spoke LAN summary through a network statement (with network importcheck disable) or a black-hole route. Each spoke enables auto-discovery-receiver, defines its own neighbor-group and neighbor-range, and — critically — configures set passive disable so the spoke can initiate the dynamic BGP connection to another spoke once the shortcut is up. For ADVPN with phase-2 selector on the hub, IPsec is similar to ADVPN with route-reflector, but BGP additional-path advertisement is enabled and route-reflector-client is left disabled; on the spoke, mode-cfg enable, mode-cfg-allow-client-selector enable, add-route enable, and phase-2 src-addr-type name/dst-addr-type name allow the injection of custom selectors on the shortcut.

MENTAL NOTE
Two parts to remember, one flag to test. IPsec sets the auto-discovery role, SD-WAN sets advpn-select and the transport groups, and advpnsc=1 is the log field that tells you a tunnel is a shortcut.
