Appendix · NSE7 SD-WAN 7.6

Reading the Overlay When Something Goes Wrong

A field guide to the CLI commands, SLA maps, and redundancy features that turn an unhealthy SD-WAN into a debuggable one.

A tunnel is only as trustworthy as the health check watching it.

This appendix summarises the diagnose commands that expose what SD-WAN is actually doing on the box, and the FEC and packet-duplication features that let a lossy underlay still carry business-critical traffic.

General Diag SLA & Health Rules & ADVPN FEC & Duplication
💭 THINK FIRST · GENERAL DIAG

Before reaching for an SD-WAN-specific command, ask yourself the older question: is the packet even arriving, and if it is, what does FortiGate think it should do with it? A sniffer and a debug flow answer both — and they show SD-WAN behaviour in the outgoing interface they pick.

  • Why must trace level 4 be used with diagnose sniffer packet when investigating SD-WAN outgoing-member selection?
  • What does the sdwan_mbr_seq field in a session entry tell you, and when is it deliberately absent?
Q1 — MODEL ANSWER
Trace level 4 makes the sniffer print the interface name alongside the packet header, which is exactly what you need to see when SD-WAN steers the flow — for example, whether the ICMP echo request that left port5 came back through HUB1-VPN1, HUB1-VPN2, or HUB1-VPN3. Combine that with interface any so every possible outgoing member is captured, and a host filter to limit the noise. Remember: for FortiGates with NP2/NP4/NP6 offloading, disable ASIC offload at the policy level first, or the PCAP process cannot see the traffic at all.
Q2 — MODEL ANSWER
In the diagnose sys session list output, sdwan_mbr_seq is the SD-WAN member ID currently steering the session, and sdwan_service_id is the SD-WAN rule ID that produced that decision. Both fields are absent when the session matched the SD-WAN implicit rule and was therefore handled by ordinary FIB routing, or when the traffic never touched SD-WAN at all — that absence is itself a diagnostic clue that the flow is not being steered by any user-defined rule.
SECTION 01

General-Purpose Commands in an SD-WAN Context

Troubleshooting SD-WAN does not throw away the classical FortiGate toolbox — it extends it. The workflow still starts with a sniffer to prove packets are actually arriving, a debug flow to see the forwarding decision, a session list to inspect what state was built, and route-lookup commands to explain why that state was built. What changes in an SD-WAN deployment is that each of those commands exposes new SD-WAN-specific fields you must know how to read.

Start with diagnose sniffer packet. Use interface any so every possible outgoing member is captured, apply a filter tight enough to isolate the flow you care about, and set trace level 4 so the interface name appears next to each packet — that is how you confirm which overlay tunnel SD-WAN actually selected.

💭 THINK FIRST · SLA & HEALTH

Every SD-WAN steering decision ultimately rests on one number: does this member meet its SLA targets right now? The sla_map bitmask compresses that answer into a single hex digit — and if you cannot read it, you cannot explain why a rule flipped members.

  • How do you decode an sla_map of 0x6 when a member has three configured SLA targets — and why is 0x0 ambiguous?
  • When would you rely on diagnose sys link-monitor-passive admin list instead of diagnose sys sdwan health-check status?
Q1 — MODEL ANSWER
The sla_map is the hexadecimal value of a binary bitmask. Bit 0 is the first configured SLA target, bit 1 the second, and so on; each bit is 1 if that target is met and 0 if it is failing. So with three targets, 0x6 is binary 110, meaning targets 3 and 2 pass (bits 2 and 1) but target 1 fails (bit 0). 0x0 has two possible meanings that you must disambiguate against the config: it can mean "all configured SLA targets are currently failing," or it can mean "no SLA targets are configured for this member at all." Always cross-check the Performance SLA configuration to know which one applies.
Q2 — MODEL ANSWER
When the health check is in passive or prefer-passive mode. The active lnkmt process measures via probes and is reflected in diagnose sys sdwan health-check status. In passive/prefer-passive mode, a separate lnkmt_passive process gathers packet-loss, latency, and jitter from real user traffic at the kernel level, and its data is exposed by diagnose sys link-monitor-passive admin list (with variants by-interface, by-application, and remote list for peer-reported SLA). You also enable it at the policy with set passive-wan-health-measurement enable, which forces FortiOS to disable auto-asic-offload on that policy because NPU cannot support passive measurement.

💭 THINK FIRST · RULES & ADVPN

SD-WAN rules are not static: their outgoing interface list is recomputed as members go up, down, and in and out of SLA. When a rule seems to have "stopped working," ask what its live status is — and when an ADVPN shortcut refuses to form, listen to what IKE has to say.

  • What is the difference between "service disabled caused by no destination" and "service disabled caused by no outgoing path" — and which one implicates routing?
  • Why is mrem-addr4 critical when you are debugging IKE for an ADVPN spoke-to-spoke shortcut?
Q1 — MODEL ANSWER
"Service disabled caused by no destination" means there is no route to the destination in the routing table — a routing problem, not a member problem. "Service disabled caused by no outgoing path" means every outgoing interface for the rule is down, so even though the destination is known, no member can reach it. Both surface in the warning line above the member list in diagnose sys sdwan service4, and both switch a rule to disabled — but the fixes are different: fix the FIB for "no destination," fix the underlays or overlays for "no outgoing path."
Q2 — MODEL ANSWER
Ordinary IKE log filters accept a single peer address, but an ADVPN shortcut is a three-way conversation — the initiating spoke, the hub, and the target spoke — with messages like shortcut-query, shortcut-reply, and shortcut-offer. mrem-addr4 (or mrem-addr6 for IPv6) lets you specify multiple IP addresses — up to six — so a single debug session captures all sides of the exchange. Without it you would see only one endpoint's chatter and miss the actual negotiation.

💭 THINK FIRST · FEC & DUPLICATION

When an underlay is lossy but you still need voice, video, or a critical session to survive, Fortinet gives you two very different tools: FEC ships parity packets so lost ones can be rebuilt, and packet duplication ships the same packet across multiple overlays so at least one arrives. Both trade bandwidth for reliability — and one supports NPU offload while the other does not.

  • Why does enabling FEC on a policy force FortiOS to disable hardware offload for that traffic, and what does adaptive FEC do about it?
  • What is the operational difference between forced packet duplication and on-demand packet duplication, and how does the receiver dedupe?
Q1 — MODEL ANSWER
FEC computes parity packets from the payload of original packets — that computation is not implemented in NPU silicon, so FortiOS automatically disables hardware offload for any session subject to FEC and increases CPU usage instead. Adaptive FEC mitigates this by applying FEC only when link conditions warrant it: an SLA health check drives the FEC mapping profile, so redundant packets rise as packet loss rises and fall (or stop) when the link is clean. In practice you apply adaptive FEC selectively — for example on UDP-based VoIP policies where FEC matters — and leave TCP-based flows alone so their offload survives.
Q2 — MODEL ANSWER
Forced (set packet-duplication force) always duplicates the packet across up to duplication-max-num members regardless of link quality. On-demand (set packet-duplication on-demand) inspects the outgoing member selected by the SD-WAN rule lookup and duplicates only when that member does not meet any of its SLA targets — or has none configured — so bandwidth is saved when the link is healthy. On the receiver, set packet-de-duplication enable tells FortiGate to accept only the first copy that arrives and drop the rest. Unlike FEC, packet duplication ships verbatim copies (useful for OOB inspection or packet capture) and supports hardware offloading.
SECTION 04

Forward Error Correction and Packet Duplication

Forward Error Correction (FEC) is a Fortinet-proprietary error-correction technique for IPsec overlays. The sending FortiGate transmits additional parity packets through the tunnel; the receiving FortiGate uses them to reconstruct any lost or corrupted originals. FEC is useful for delivering a better user experience for business-critical applications like voice and video, at the cost of extra bandwidth on the tunnel. It does not interoperate with third-party IPsec gateways.

FEC works by sending fec-redundant parity packets for every fec-base packets sent within the fec-send-timeout window. If the timeout is reached first, FortiGate sends the parity packets anyway. Direction matters: fec-egress makes this side generate parity; fec-ingress makes it process parity received from the peer.

FEC — BASE CONFIGURATION
config vpn ipsec phase1-interface
  edit "HUB1-VPN1"
    set fec-egress enable
    set fec-ingress enable
    set fec-base 10
    set fec-redundant 1
    set fec-send-timeout 5
  next
end

config firewall policy
  edit <id>
    set fec enable
  next
end

When FEC is enabled, the tunnel carries a specific SPI of 0x00000001 for both original and parity ESP packets. A sniffer will show the original ICMP echo request egressing the underlay followed by two additional ESP packets with that SPI — those are the parity packets.

Offload trade-off: FortiOS automatically disables hardware offload for sessions subject to FEC and CPU usage rises. You do not need to disable offload manually on the policy.

FEC pays off. In a test using 100 ICMP echo request packets across a noisy overlay:

ConfigurationPacket Loss Observed
Before enabling FEC27%
fec-redundant 1, fec-base 20, timeout 812%
fec-redundant increased to 31%

Adaptive FEC dynamically adjusts the base and redundant packet ratio based on the SLA health check. When packet loss is low, FEC uses a low redundancy ratio; as loss rises, redundancy rises with it. Apply it per policy and reserve it for streams most sensitive to packet loss — for example UDP-based VoIP.

ADAPTIVE FEC — MAPPING PROFILE
config vpn ipsec fec
  edit "m1"
    config mappings
      edit 1
        set base 8
        set redundant 2
        set packet-loss-threshold 10
      next
    end
  next
end

config vpn ipsec phase1-interface
  edit "HUB1-VPN1"
    set fec-health-check "1"
    set fec-mapping-profile "m1"
    set fec-base 10
    set fec-redundant 1
  next
end

Packet duplication takes a different approach. Instead of computing parity, it sends verbatim copies of the original packet through up to three additional SD-WAN members. Because the copies are exact, they are useful for data-loss protection and out-of-band inspection or packet capture. Unlike FEC, packet duplication supports hardware offload. The requirements are that the best route to the destination must resolve to an SD-WAN member, and the members used for duplication must have a route to the destination.

On the receiving side, enable packet deduplication so FortiGate accepts only the first copy of a packet and drops any additional copies — this saves resources at the receiver and prevents the next hop from having to discard the extras.

FORCED PACKET DUPLICATION
# Spoke — always duplicate
config system sdwan
  set duplication-max-num 3
  config duplication
    edit 1
      set srcaddr "10.0.1.0/24"
      set dstaddr "10.1.0.0/24"

  end

On-demand packet duplication is the smart cousin. FortiGate first selects the outgoing interface using the SD-WAN rule lookup, then inspects that interface's SLA state. If the interface meets at least one SLA target, FortiGate does not duplicate. If the interface meets no SLA targets — or has none configured — FortiGate duplicates. This conserves bandwidth on healthy links and only pays the duplication cost when the picked path looks poor.

Finally, you can restrict packet duplication to traffic matching one or more SD-WAN rules by setting service-id in the duplication rule. Once a service-id is set, the 5-tuple options on the duplication rule are hidden — matching is delegated entirely to the SD-WAN rule, which lets you leverage application steering and ISDB targets without redeclaring them.

RESTRICT DUPLICATION TO SD-WAN RULE
config system sdwan
  set duplication-max-num 3
  config duplication
    edit 1
      set service-id 1
      set packet-duplication force
    next
  end
end
