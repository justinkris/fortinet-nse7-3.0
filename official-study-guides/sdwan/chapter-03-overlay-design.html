Chapter 03 · NSE7 SD-WAN 7.6

Designing the Overlay the Way Fortinet Recommends

A working tour of IPsec overlays, IBGP routing choices, and the OaaS shortcut for building SD-WAN hub-and-spoke fabrics.

Every SD-WAN fabric begins with a choice about the overlay.

Underlays are what you rent from carriers; overlays are what you engineer on top. This chapter is about the engineering.

💭 THINK FIRST · OVERLAY FUNDAMENTALS

Underlay links are what carriers sell you — MPLS circuits, broadband, LTE. Overlays are the virtual links you build on top of those underlays so that routing, security, and identity can live in a place that the carrier cannot see or change. Before you touch any config, ask why Fortinet insists on IPsec for the overlay when GRE and IP-in-IP would also technically work.

  • Why does Fortinet recommend IPsec as the tunneling protocol for SD-WAN overlays instead of GRE or IP-in-IP?
  • What are the two IKE phases, what does each of them negotiate, and why is IKEv2 recommended for SD-WAN?
Q1 — MODEL ANSWER
IPsec is a standard protocol (RFCs 4301, 4302, 4303) that authenticates and encrypts traffic between peers, so the overlay carries both privacy and integrity out of the box. It is also compatible with FortiManager central management and the SD-WAN overlay orchestrator, which know how to build IPsec templates automatically. GRE and IP-in-IP are supported but come with trade-offs — GRE is mainly useful for Cisco-style VPN interoperation (usually GRE over IPsec), and IP-in-IP (RFC 1853) has limited overlay routing options. For those reasons IPsec is Fortinet's recommended overlay tunneling protocol.
Q2 — MODEL ANSWER
Phase 1 negotiates the IKE SA — the secure channel that both peers will use to talk to each other. In IKEv1 that is the main/aggressive mode; in IKEv2 it is IKE_SA_INIT and IKE_AUTH. Phase 2 negotiates the IPsec SA — the actual keys used to encrypt and decrypt data traffic. In IKEv1 that is quick mode; in IKEv2 it is CREATE_CHILD_SA. IKEv2 is recommended for SD-WAN because it has a simpler negotiation process, more features (network ID, EAP authentication, IKE SA integrity algorithm), and exclusively supports the network-id feature that lets you establish multiple tunnels between the same local and remote gateways — a requirement in SD-WAN failover scenarios and with ADVPN.
SECTION 01

Underlays, Overlays, and Why IPsec Wins

In the vocabulary of SD-WAN, underlays are the physical links — the MPLS circuit, the broadband service, the LTE modem — that the carrier provides. Overlays are virtual links built on top of those underlays using tunneling. That distinction matters because the overlay is where you get to make design decisions: it is where routing lives, where encryption lives, and where the fabric's identity is expressed. The underlay is rented; the overlay is engineered.

FortiGate supports three tunneling protocols for building overlays. IPsec is a standard protocol (RFCs 4301, 4302, 4303), compatible with FortiManager central management and the SD-WAN overlay orchestrator, and it is the Fortinet-recommended overlay tunneling protocol. GRE — usually deployed as GRE over IPsec — is offered mainly for Cisco-style VPN interoperability. IP-in-IP (RFC 1853) is a standard protocol with limited overlay routing options. In practice, and for the rest of this chapter, overlays are IPsec.

IPsec is a suite of protocols for securing IP communications. Two of its members carry most of the weight. IKE (Internet Key Exchange) handles the handshake, tunnel maintenance, and disconnection; it uses UDP 500 by default, and UDP 4500 when NAT traversal (NAT-T) is required. ESP (Encapsulating Security Payload) provides data integrity and encryption; it uses IP protocol 50 natively, or is encapsulated in UDP 4500 when NAT-T is negotiated. ESP is the workhorse; ESP has no port numbers of its own, which is precisely why NAT-T exists — to let ESP survive translation on the path.

IKE has two phases. Phase 1 negotiates the IKE SA, the secure channel used to negotiate everything else. Phase 2 negotiates the IPsec SA, the actual keys used to encrypt and decrypt data traffic through the tunnel. In normal two-way traffic, the exchange is secured by a pair of IKE SAs — one for each direction. SAs expire and must be renegotiated after their lifetime. IKEv2 is recommended for SD-WAN because it simplifies negotiation, adds features (network ID, EAP authentication, IKE SA integrity algorithm), and exclusively supports network-id — a requirement for establishing multiple tunnels between the same gateways in SD-WAN failover and ADVPN scenarios.

Overlay topology at a glance: Fortinet's standard SD-WAN overlay is a hub-and-spoke built with dial-up IPsec tunnels — the hub is the dial-up server, each spoke is a client. For higher performance, group overlays per ISP (same-ISP overlays generally beat cross-ISP overlays on latency). SD-WAN is usually deployed on the spokes; it is also required on the hub when hub-to-spoke traffic exists or ADVPN is in play.

🧠
MENTAL NOTE
Phase 1 negotiates the IKE SA; Phase 2 negotiates the IPsec SA. IKEv2 for SD-WAN — the network-id feature is why.

Overlay topology choices in this space are narrow but real. A dynamic hub-and-spoke topology with dial-up tunnels is the default; each spoke dynamically brings up a tunnel over each underlay to each hub. Hub-and-spoke with static tunnels trades scale for stability and readability — every tunnel is explicitly named, so logs are easier to interpret. Full mesh with static tunnels gives direct connectivity to every site but suffers scalability problems (the number of tunnels grows as N × (N−1) / 2 per ISP), which is why Fortinet suggests dynamic hub-and-spoke with ADVPN instead of a full mesh for most SD-WAN designs.

Common misconception: "Full mesh is faster because everything is direct." True in a lab. In practice, the configuration and tunnel count explode with the number of sites — dynamic hub-and-spoke plus ADVPN shortcuts gives you on-demand direct paths without the static-mesh overhead.

💭 THINK FIRST · OVERLAY DESIGN

Once you have committed to IPsec and to a hub-and-spoke, the interesting design questions begin. How do the tunnel interfaces get their IP addresses — from IKE mode-cfg, or manually via exchange-interface-ip? Should net-device be enabled on the spoke, disabled on the hub, or the other way round? And what routing protocol carries prefixes across the fabric?

  • What does the net-device parameter do, and why does Fortinet recommend it disabled on the hub and enabled on the spokes?
  • Given a topology with three regions, 100–200 spokes per region, and overlays over each underlay, which routing protocol should the administrator use — and why?
Q1 — MODEL ANSWER
The net-device parameter is a Fortinet-proprietary attribute configured on the IPsec phase 1 interface that determines whether FortiGate creates a dedicated kernel interface for each dial-up tunnel. With net-device enable, a named interface is created per phase1name_index for each tunnel — great for monitoring and troubleshooting, but it consumes CPU resources and slows down tunnel setup and tear-down, so it is a poor fit for devices with many dial-up tunnels. With net-device disable (the default), a single network interface is shared per phase1 and a tunnel ID is generated for each dynamic tunnel; setup and tear-down rates improve. Fortinet therefore recommends disabling net-device on the hub (which handles many tunnels) and enabling it on the spokes (which typically have few). Additionally, if SD-WAN is configured on the hub, FortiOS does not allow dynamic IPsec tunnels to be SD-WAN members.
Q2 — MODEL ANSWER
BGP — specifically IBGP. Static routing is supported but does not scale well and is not compatible with ADVPN. Of the dynamic protocols FortiGate supports (RIP, OSPF, IBGP, EBGP), IBGP is recommended because it scales, preserves the next hop (which is convenient for ADVPN), and is compatible with ADVPN. In this scenario BGP on loopback is the recommended sub-design — a single IBGP session per hub, a loopback interface uniquely identifying each SD-WAN node, and spoke summary routes advertised over that single session per hub. It extends well to multi-region deployments, which the three-region scenario requires.
SECTION 02

Tunnel IPs, net-device, and the Routing Choice

Designing the overlay begins with the tunnel-establishment mode. Static tunnels are stable and common with full-mesh topologies but require configuration on both ends for every new tunnel — heavy maintenance overhead. Dynamic tunnels are recommended for hub-and-spoke because adding a spoke does not require any change on the hub, and dynamic tunnels are ADVPN-compatible. In FortiGate terms this is expressed as the phase1-interface type (either dynamic for a dial-up server or static for a peer with a known IP).

Next comes the question of how the tunnel interface gets its IP address. Two options exist. IKE mode config (mode-cfg enable) is the IKE standard: the administrator defines a range of IP addresses on the hub, and each spoke receives its tunnel interface IP address from the hub during negotiation. Exchange interface IP (exchange-interface-ip) is a Fortinet-proprietary IKE attribute; the administrator manually assigns the tunnel IP on both the spoke and hub sides, and IKE exchanges those addresses so each gateway recognizes the remote overlay IP. Mode config is simple; exchange-interface-ip lets you pick fixed addresses, which is useful when the tunnel IP is also used for monitoring or, in the BGP-on-loopback design, when the loopback IP itself is the tunnel address.

SD-WAN overlay orchestrator note: When you drive the overlay with FortiManager's SD-WAN overlay orchestrator, it configures the hub as a dial-up server, defines mode-cfg with the administrator's IP range for BGP-per-overlay designs, and switches to exchange-interface-ip for BGP-on-loopback designs — matching Fortinet's recommended settings automatically.

The net-device parameter is Fortinet-specific and lives on the phase 1 interface. Enabled, it creates a dedicated kernel interface for each dial-up tunnel (name is phase1name_index); disabled — the default — it shares a network interface and generates a tunnel ID per dynamic tunnel in the form of an IP address, used as the gateway in the route entry. Recommended settings are hub: disabled (many tunnels, performance matters, and SD-WAN configured on the hub disqualifies dynamic tunnels from SD-WAN membership anyway) and spokes: enabled (few tunnels, and the named interfaces help monitoring and troubleshooting).

🧠
MENTAL NOTE
net-device on the hub = disabled (throughput). net-device on the spokes = enabled (readability). Never both the same way.

Routing across the overlay is the last big choice. Static routing is supported but does not scale and is not compatible with ADVPN. Of the dynamic protocols FortiGate supports (RIP, OSPF, IBGP, EBGP), Fortinet recommends IBGP: it scales, preserves the next-hop, and integrates cleanly with ADVPN. A common multi-region routing design uses EBGP between regions, IBGP within regions, and static routes between spokes and their local LAN.

Inside IBGP itself, Fortinet offers two implementations. BGP per overlay is the historical design — every spoke establishes a separate IBGP session over each overlay towards each hub, and each spoke advertises its LAN prefixes over every established session. It works with ADVPN but scales poorly as the number of overlays grows. BGP on loopback is the newer, recommended design — every spoke establishes a single IBGP session per hub, terminated on the loopback interface which uniquely identifies each SD-WAN node. Spokes advertise LAN prefixes over that single session per hub. Configuration is simpler, the number of advertised routes drops sharply, and the design extends well to multi-region deployments.

If ADVPN shortcuts are in the picture, spokes still need to learn each other's routes. Two options exist. Route reflection — the historical solution — has the hub learn all routes from each spoke and reflect them to every other spoke; simple, but the hub burns CPU proportional to the spoke count and routing tables get large. Dynamic BGP — the newer alternative — has the hub learn only spoke summary routes and lets spokes establish BGP peering with each other only after an ADVPN shortcut is up. That produces on-demand peering, drastically fewer routes, and better hub-CPU behavior in large ADVPN fabrics. Dynamic BGP is compatible with both BGP-per-overlay and BGP-on-loopback.

Multi-region rule of thumb: EBGP between regions, IBGP within a region, static routes between spokes and their local LAN. Prefer BGP on loopback when the topology exceeds a small single region.

💭 THINK FIRST · OVERLAY DEPLOYMENT

The deployment tools are a ladder. Small topologies can be built with the FortiGate GUI's VPN wizard or the CLI; large ones live inside FortiManager, where the SD-WAN overlay orchestrator, the IPsec provisioning templates, and the (legacy) VPN Manager each play a role. Recommended IPsec templates encode Fortinet's best practices so administrators do not have to remember them one setting at a time.

  • What are the three main tools for configuring SD-WAN overlays, and when do you choose each?
  • In BGP-per-overlay, why must the spoke bind BGP packets to a specific tunnel interface using interface and update-source?
Q1 — MODEL ANSWER
Three main tools exist. The FortiGate GUI or CLI is available for small topologies with a limited number of devices, each spoke configured individually and no central management — the GUI's VPN Wizard supports Site to Site, Hub and Spoke with ADVPN, and Remote Access. The SD-WAN overlay orchestrator in FortiManager is recommended for SD-WAN — it guides the administrator through tunnel and BGP configuration for single-hub, dual-hub, or multi-hub (up to four hubs) topologies and applies Fortinet-recommended settings automatically. IPsec Provisioning Templates in FortiManager offer predefined templates (IPsec Fortinet Recommended, BRANCH IPsec Recommended, HUB IPsec Recommended) that add the same recommended settings. The VPN Manager is legacy central VPN management and is not recommended for new SD-WAN setups.
Q2 — MODEL ANSWER
With BGP per overlay you want each IBGP session to travel over the tunnel it belongs to, not over some other overlay. Binding BGP packets with set interface "HUB1-VPN1" forces the outgoing interface, and set update-source "HUB1-VPN1" ensures the source IP address is that of the selected interface. Together they prevent cross-overlay BGP peerings — a subtle but real risk on SD-WAN because FortiOS would otherwise perform a route lookup and could pick a different outgoing interface. Note that update-source is technically redundant when interface is set (the interface forces the source), but it is still good practice for consistency, and it matters when the source IP is not the interface IP — for example when the source IP is a loopback address.
SECTION 03

Deploying the Overlay at Scale

The three-tool ladder starts with the FortiGate GUI or CLI for small topologies. The GUI's VPN Wizard offers three flavors — Site to Site, Hub and Spoke with ADVPN, and Remote Access — and, when you pick the hub-and-spoke path, produces a configuration key that simplifies matching the spoke to the hub. The CLI exposes every parameter and is where you refine what the wizard drafted. Beyond a handful of sites, this approach becomes untenable and central management takes over.

At scale you use FortiManager, and inside FortiManager you have three options in descending order of preference. The SD-WAN overlay orchestrator is recommended for SD-WAN: it guides overlay-tunnel and BGP configuration for single-hub, dual-hub, or multi-hub (up to four hubs) topologies and applies Fortinet-recommended settings automatically. The IPsec Provisioning Templates — HUB_IPsec_Recommended, BRANCH_IPsec_Recommended, and IPsec_Fortinet_Recommended — apply the same recommended parameters when you would rather work at the template level. The legacy VPN Manager is central VPN management and should not be used to deploy new SD-WAN setups.

Orchestrator choices in one line: topology (single/dual/multi-hub), IKEv2 with hub as dial-up server, authentication (pre-shared key or Signature), ADVPN (Legacy or 2.0), IBGP as the routing protocol, and either BGP-per-overlay (default) or BGP-on-loopback. The orchestrator generates matching IPsec and BGP templates for both hub and spoke, following best practices.

When the orchestrator prepares IPsec templates for a BGP per overlay design, the hub settings include type dynamic, net-device disable, mode-cfg enable, add-route disable, IP address assignment, network-overlay enable, a distinct network-id per overlay, and the tunnel IP address plus remote IP subnet. Branch templates use type static, net-device enable, mode-cfg enable (with assign-ip enable), network-overlay enable, matching network-id, a local-id, and phase2 auto-negotiate enable.

For BGP on loopback, the hub template drops mode-cfg and enables exchange-interface-ip with exchange-ip-addr4 set to the hub's loopback IP. Branches mirror that with exchange-ip-addr4 set to the spoke's loopback IP. The other core parameters — dynamic type on the hub, static type on the branch, net-device disabled on the hub and enabled on the branch, network-overlay enabled with a network-id — remain the same.

SPOKE BGP CONFIG — BGP PER OVERLAY
config router bgp
    set as 65000
    set router-id 10.200.99.1
    set ibgp-multipath enable
    config neighbor
        edit "192.168.1.61"
            set interface "HUB1-VPN1"
            set remote-as 65000
            set update-source "HUB1-VPN1"
        next
        edit "192.168.1.125"
            set interface "HUB1-VPN2"
            set remote-as 65000
            set update-source "HUB1-VPN2"
        next
    end
    config redistribute "connected"
        set status enable
        set route-map "port5_only"
    end

In BGP-per-overlay, IBGP is used because it preserves the next-hop for prefixes (convenient for ADVPN); ECMP is expected due to SD-WAN's multipath nature, so ibgp-multipath enable is required. Each neighbor entry sets interface and update-source to bind BGP packets to a specific overlay and prevent cross-overlay peerings; while update-source is technically redundant when the interface already forces the source, it stays for consistency and for future scenarios where the source IP differs from the interface. The redistribute "connected" section, filtered by a route-map, injects the LAN prefix (for example 10.0.1.0/24 on port5) into BGP for advertisement.

On the hub side of BGP-per-overlay, one neighbor-group per overlay defines standard BGP settings; route-reflector-client enable is toggled on only if ADVPN is planned, because reflection is what lets spokes learn each other's prefixes. A neighbor-range associates an IP address prefix with each group so the correct settings apply to peers from each overlay's IP range.

The BGP on loopback design relies heavily on the loopback IP address, which uniquely identifies each SD-WAN node. The spoke sets router-id and update-source to the branch loopback, enables ibgp-multipath, and — critically — enables recursive-next-hop and tag-resolve-mode merge. That last piece introduces a Fortinet-proprietary BGP extension called tag-based recursive resolution (tag-match). Routes received from each hub are tagged via a route-map (for example, tag 1 for hub HUB1), and recursive resolution then attempts to resolve those routes using only routes with the same tag. In merge mode it combines the tag-match result with the standard best-match. The tag has only local significance — FortiGate does not advertise it — and does not break any BGP standard.

On the hub with BGP-on-loopback, the config enables both ibgp-multipath and ebgp-multipath, recursive-next-hop, recursive-inherit-priority, and defines a single neighbor-group for all branches (usually named EDGE) with a neighbor-range matching the loopback IP prefix.

Two more deployment moves matter. BGP route reflection (via route-reflect-client enable) is required only with ADVPN; it lets the hub reflect learned routes from one spoke to other spokes in the group. Advertise additional paths (also required with ADVPN) is how FortiGate propagates more than one path per prefix so that SD-WAN can see and steer over all available overlays. It uses additional-path enable, additional-path-select N (where N is the number of overlays), and per-neighbor-group additional-path send plus adv-additional-path N; receivers use additional-path receive. The result is duplicate BGP routes on the FIB — consolidated for cosmetic readability on the routing table but still present and usable.

🧠
MENTAL NOTE
Route reflection + additional paths + ADVPN → spokes learn all paths to all destinations. Without additional paths, SD-WAN skips overlays it never learned about.

Dynamic BGP is the alternative to route reflection and is compatible with both BGP-per-overlay and BGP-on-loopback. Instead of the hub reflecting every possible route to every spoke, the hub learns only spoke summary routes; spokes establish BGP peering with each other only after an ADVPN shortcut is up, then exchange routes over that shortcut. The result: the hub avoids the CPU cost of reflecting, and spokes learn only the routes they actually need. On the spoke you keep set passive disable (so it can initiate) and define a neighbor-group and neighbor-range for the other spokes; on the hub you disable route-reflector-client and define a neighbor-group and neighbor-range that covers all spoke prefixes. Peerings between spokes are called on-demand peering — they exist only where user traffic requires them.

Exam trap: Dynamic BGP is not exclusive to BGP-on-loopback — it works with BGP-per-overlay too. And if spokes fail to establish BGP peering after an ADVPN shortcut is up, look for set passive disable on the spoke's neighbor-group.

💭 THINK FIRST · SD-WAN OaaS

SD-WAN Overlay-as-a-Service (OaaS) shifts the hub off the customer's plate. Fortinet hosts and manages the hubs in FortiCloud; the customer owns the spokes. Configuration happens in a web portal, not on a per-device CLI. The exam question is: what exactly does OaaS create on the branch when you click Deploy?

  • What does Fortinet provide in an OaaS deployment, and what does the customer still own?
  • What specific objects does OaaS create on each branch FortiGate, and why does it prefix every one of them with oaas?
Q1 — MODEL ANSWER
Fortinet provides the managed hubs with a geo-redundant dual-hub high-availability architecture, and the OaaS orchestrator that creates and installs policies across multiple devices. The customer owns and manages the spoke devices — FortiGate hardware or in-house VMs at branch and data centre locations — and manages the overlay connections through the OaaS web interface. Fortinet does not host the customer's LAN; the customer runs it. The benefits are cost saving on device hosting and management, simplicity, and fast delivery, with SD-WAN intelligent path steering and BGP dynamic routing running on top of the managed overlay.
Q2 — MODEL ANSWER
OaaS creates one IPsec tunnel per ISP to the primary hub only, with ADVPN parameters so spokes can build shortcut tunnels between themselves. It sets up BGP on loopback using private AS 65001 (which must remain exclusive to OaaS) over the reserved subnet 10.200.0.0/16. It creates one SD-WAN zone (oaas_overlay) grouping all OaaS overlay interfaces, a performance SLA with a default 100ms latency threshold, and a lowest-cost SD-WAN rule (oaas_default). It adds two firewall policies (oaas_default for spoke LAN-to-spoke LAN traffic and oaas_bgp for health checks to loopbacks) and a system zone (oaas_lan_zone) plus a firewall address group (oaas_corp_network). Every object is prefixed with oaas so administrators can immediately identify what OaaS created — and can safely add their own policies alongside without confusion.
SECTION 04

Overlay as a Service and What It Configures for You

SD-WAN Overlay as a Service (OaaS) is a Fortinet SaaS offering that provides on-demand secure connectivity between locations with point-and-click setup and monitoring. Fortinet provides the managed hubs — geo-redundant, dual-hub, high-availability — and the OaaS orchestrator that installs configuration across multiple devices. The customer provides ownership and management of the spoke devices and manages the overlay via the OaaS web interface. It is a division of labor: Fortinet owns the overlay infrastructure; you own the endpoints.

OaaS is a licensed service included in the FortiGate Cloud Standard Subscription, itself part of the SD-WAN service bundle; a subscription is required for each remote device. Supported topologies are SD-WAN with dual hub (primary and secondary) and SD-WAN with ADVPN. Spokes must run FortiOS 7.4.4 or later. Before adding a site, the spoke must be internet-connected and registered with FortiCare; LAN and WAN interfaces defined; basic routing configured so branches can reach the hub; and the interfaces used for OaaS must not already be associated with an SD-WAN zone, a network zone, or a firewall policy — OaaS creates its own.

Reserved subnet: OaaS reserves 10.200.0.0/16 for overlay IP addressing of all spokes; do not use this network in either the LAN subnets or the WAN network. It can be changed from the OaaS portal Settings if it conflicts with the existing network.

In the OaaS portal you select two hub locations (for redundancy), add sites (each with a name, type — Branch or Data Center — and a registered FortiGate serial number), add ISP links per site (name, interface, optional cost), define the LAN subnet per site, and click Apply to deploy. OaaS then relies on the management tunnel each FortiGate has with FortiCloud to retrieve interface information and install its configuration. The Home page shows a graphical view of the topology with color codes for device status: green (All Pass), orange (Operational), red (Disconnected), gray (Unmanaged).

What OaaS creates on each branch is worth memorizing because it forms the whole exam surface for this section:

  • IPsec tunnels: one tunnel per ISP interface, to the primary hub only (secondary is reached via the dual-hub HA architecture). Each tunnel carries the ADVPN parameters that let spokes build shortcut tunnels between themselves.
  • BGP routing: BGP on loopback using the private AS 65001, which must remain exclusive to the OaaS configuration.
  • SD-WAN zone oaas_overlay: groups all OaaS overlay interfaces into one zone.
  • Performance SLA: monitors performance through each overlay and any established ADVPN shortcuts, with a default latency threshold of 100 ms.
  • SD-WAN rule oaas_default: a lowest-cost (SLA) rule that steers traffic through the best-performing overlay link by latency.
  • Firewall policies: oaas_default allows all traffic between spoke LAN subnets over the overlays; oaas_bgp allows health-check traffic to hub and spoke loopback interfaces.
  • System zone oaas_lan_zone: a system zone including all branches' LAN subnets.
  • Firewall address group oaas_corp_network: the address group referenced by the default policy.

Every object OaaS creates is prefixed with oaas so administrators can immediately identify what belongs to the orchestrator and what belongs to them. This matters because OaaS does not lock the FortiGate down; the spoke administrator remains free to add additional firewall policies, tighten wildcard rules, and refine configuration as needed. The oaas_default policy uses wildcard address objects and services — you almost always want to add narrower policies alongside it.

🧠
MENTAL NOTE
OaaS = Fortinet owns the hubs and the AS (65001); the customer owns the spokes and the LAN. The oaas prefix marks the boundary.
Common misconception: "OaaS lets me mix customer-managed hubs with Fortinet hubs for extra redundancy." False — an OaaS topology is dual-hub within FortiCloud, not a hybrid with in-house hubs. And OaaS SD-WAN rules are not limited to lowest cost / SLA only despite the default rule; you can define additional rule types on the branch after deployment.
