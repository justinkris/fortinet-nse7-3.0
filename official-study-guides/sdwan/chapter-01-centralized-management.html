Chapter 01 · NSE7 SD-WAN 7.6

Bringing SD-WAN to the Branch From One Console

How FortiManager plans, provisions, and orchestrates an SD-WAN fabric that would be intractable to build one CLI at a time.

Scale is not a feature — it is a design constraint.

A single branch is a configuration problem. A hundred branches is an operational problem. FortiManager is the console where the second one becomes tractable.

Define & Plan Configure with FMG Overlay Template
💭 THINK FIRST · DEFINE & PLAN

A hundred branches, three ISPs each, a dual data-centre core, and one administrator to install the whole thing on a Tuesday. Before you touch a single FortiGate CLI, ask what FortiManager is actually doing for you — and what still has to be decided on paper first.

  • What does FortiManager give a large SD-WAN network that a room full of individual FortiGate CLIs cannot?
  • Which decisions about addressing, routing, and topology have to be made before the SD-WAN overlay template will let you proceed?
Q1 — MODEL ANSWER
FortiManager provides single-pane-of-glass management for the whole SD-WAN fabric. It provisions SD-WAN templates, CLI templates, and firewall policies across many devices at once; it supports per-device SD-WAN configuration where required; it monitors SD-WAN status; it acts as a central repository for configuration revision control and security audits; it deploys and monitors complex IPsec overlays; it offers guided configuration of overlay tunnels and BGP route deployment through the SD-WAN overlay orchestrator; it enables zero-touch provisioning for new branch sites; and it exposes scripts and JSON APIs to automate device provisioning and policy changes. The upshot is that consistent policy across the enterprise stops being an aspiration and becomes an install job.
Q2 — MODEL ANSWER
Before defining an SD-WAN overlay template you must (1) have FortiManager already managing the FortiGate devices that will form the hub or hubs, (2) create a device group for your branch devices, and (3) configure the WAN, ISP links, and other interfaces on your imported devices. Then, on the network-planning side, you must decide the topology type — single hub, dual hub (primary/secondary or primary/primary), or multi hub — allocate the overlay network address space (default template value 10.10.0.0/16), allocate the loopback IP address space (default 172.16.0.0/16), and select a BGP AS number for the new SD-WAN overlay region (default 65000). Skipping the planning phase does not gain you time; it hands the same decisions back to you halfway through the wizard.
SECTION 01

Planning an SD-WAN Before You Provision One

FortiManager is the key component for deploying SD-WAN across a large network. Its centralised, single-pane-of-glass approach is not a marketing phrase — it is the reason that a hundred-branch rollout stops being a hundred separate configuration exercises and becomes one templated install. FortiManager provisions SD-WAN templates, CLI templates, and firewall policies across many devices at once, monitors SD-WAN status, acts as a central repository for revision control and security audits, deploys and monitors complex IPsec overlays, guides you through overlay orchestration with BGP, performs zero-touch provisioning for new sites, and exposes scripts and JSON APIs for automation.

Underneath the console, FortiManager organises everything it knows in management layers. The Global Database holds global objects and header/footer policy packages — the outermost wrapping around every ADOM's traffic. The ADOM layer is where policy packages are created, managed, and installed on devices; each ADOM has a common object database of addresses, services, and security profiles, and ADOMs let administrators group devices for monitoring and management. The Device Manager layer records each managed device's name, model, IP address, current firmware, revision history, and real-time status, and keeps a device database for every managed FortiGate. Configuration changes can be installed on a single FortiGate or pushed to many devices simultaneously.

SD-WAN itself can be managed two different ways from this console, and the choice matters. In per-device management, you edit SD-WAN settings on individual devices — useful when branches have genuinely different SD-WAN requirements and must be maintained separately. In central management via SD-WAN Manager, you configure SD-WAN templates once from the Templates section and assign them to one or more FortiGates. For each template you define members, zones, performance SLAs, rules, and so on; instead of applying a change to every device, you apply it to the shared template and let FortiManager push it out. Central management is the natural choice when many devices share the same shape.

Before you can define an SD-WAN overlay template, the FortiGate hubs must already be under FortiManager's control — either as model devices or discovered physical devices — and their WAN, ISP, and other links must be configured. Branch devices can join now or later, but a branch device group must exist before you proceed. Then network planning fixes the shape of the region: the topology type, the overlay network address space (default 10.10.0.0/16), the loopback IP address space (default 172.16.0.0/16), and the BGP AS number (default 65000). None of these decisions are optional — the wizard will ask for them either at the top or in the middle.

Two management modes, one console: per-device SD-WAN objects live in the device settings database; central-management SD-WAN objects live in the ADOM database. The GUI pages look almost identical, but the storage layer tells you which mode you are in.

MENTAL NOTE
Per-device is maintain each; central is maintain one, deploy many. The pane looks the same — the storage layer, and the operational cost of a change, does not.

FortiManager also ships FortiAI as a licensed assistant that helps define configurations such as FortiManager scripts, VPNs, and BGP settings, with the FortiAI dialogue reachable from the upper-right corner or the left-side menu. And the Device Manager wizards — Add Device, Install Wizard, Import Configuration, and Re-install Policy — cut most repetitive tasks down to a few clicks: Add Device brings devices under central management and imports their configuration; the Install Wizard previews and pushes changes from the Device Manager or Policy & Objects pane; Import Configuration pulls interface mappings, policy databases, and objects into a policy package; Re-install Policy performs a quick reinstall with preview. It is worth noticing that Import Configuration and Re-install Policy are hidden inside the right-click menu on a managed device.

🧠
MENTAL NOTE
You cannot start the SD-WAN overlay template until the hubs are managed, the branch device group exists, and you have decided your overlay subnet, loopback subnet, and BGP AS. Everything else the wizard can give you defaults for.

💭 THINK FIRST · CONFIGURE WITH FMG

A template is only as reusable as its variables. If every branch's port1 gateway were baked into the SD-WAN template, the template would only ever fit one branch. The trick that makes central management scale is that FortiManager lets you push the same policy shape while letting each device carry its own values.

  • What are metadata variables, and why are they the key to reusing one SD-WAN template across dozens of branches with different gateways?
  • How do template groups, system templates, and CLI templates fit together — and when would you choose a Jinja CLI template over a plain CLI script?
Q1 — MODEL ANSWER
Metadata variables are ADOM-level, user-defined parameters (also available at the Global ADOM level for per-ADOM mapping) that let one variable name carry a different value on each device. In the SD-WAN template you can use them in Interface Member, Gateway, and Health-Check Server fields — for example, a $(sdwan_port1_gw) metadata variable holds 192.2.0.2 on branch1_fgt and 203.0.113.2 on branch2_fgt, while $(inet3) tells FortiManager to install port3 as the member on branch1_fgt and port8 on branch2_fgt. You reference a metadata variable by typing $ in a supported field (a magnifying glass with a $ sign marks such fields), pick from the list or create a new one with +. Each variable should have a default value plus per-device mappings; FortiManager will refuse to install a configuration that uses a variable it has no value for on that device. CSV/JSON import and export exist so provisioning hundreds of values does not become manual data-entry work.
Q2 — MODEL ANSWER
A template group bundles a System Template, an IPsec Tunnel Template, a BGP Template, a Static Route Template, an SD-WAN Template, and one or more CLI Templates into a single unit that gets assigned to a device or device group in one step. System templates are subsets of a model device configuration — DNS, alert email, admin settings, log settings, SNMP — that facilitate identical device-level settings across many devices. CLI templates come in two flavours: plain CLI scripts (FortiGate CLI commands with metadata variables prefixed by $) and Jinja scripts, a Python-like template engine that supports loops, arithmetic (e.g. ipmath), and per-device logic, with metadata variables referenced using {{ }}. Reach for a Jinja template when you need programmatic behaviour — computing a loopback IP from a branch_id, for example — and stick with a plain CLI script when you are simply substituting values. A template group can mix both types and applies its scripts top-down. Only one CLI template or CLI template group per device.
SECTION 02

Templates, Metadata, and the Craft of Reusable Configuration

Whether you configure SD-WAN in per-device mode or with central management, the FortiManager pages look almost identical to the FortiGate GUI: the same three sections — SD-WAN Zones, Performance SLA, and SD-WAN Rules — and the same conceptual language of members, zones, SLAs, and rules. The critical difference in central mode is that fields like Interface Member, Gateway, and Health-Check Server accept metadata variables: user-defined parameters, defined at the ADOM level, that let a single template carry per-device values.

A metadata variable is created under Policy & Objects > Advanced > Metadata Variables. Each variable has a name (letters, numbers, and underscores only), an optional default value, and a per-device mapping table. When you reach a field that supports variables, a magnifying glass with a $ icon marks it — type $ and FortiManager shows you a menu of existing variables, with a + to create a new one and a pen icon to edit an existing one. You can use metadata variables in firewall address objects, IP pools, virtual IPs, and SD-WAN template fields. For IP/Netmask fields both parts must be supplied separately — $(LAN_Subnet)/$(LAN_Mask), $(LAN_Subnet)/255.255.255.0, or two literal values are all acceptable. FortiManager will not install a configuration whose variables lack values on a given device, so setting a default (or an explicit per-device mapping) is a best practice, not a nicety.

For large fleets, populating every mapping by hand is untenable — the import and export features accept CSV and JSON, letting you provision hundreds of values from a spreadsheet or an automation pipeline, and letting you export the current state as a starting point for the next round. As a native alternative for Jinja templates only, device database variables ({{DVMDB.name}}, {{DVMDB.hostname}}, {{intf.ip}}, and so on) expose device attributes without you having to duplicate them as metadata; they are not compatible with plain CLI scripts, and when typing {{ in the GUI FortiManager only lists metadata variables — device DB variables must be typed by hand.

Why variables exist: a template with no variables fits exactly one device. A template with well-chosen metadata and device DB variables fits an ADOM. Design your variables the way you would design an API — name them for meaning, give them defaults, and export/import their mappings.

Templates come in families and are best assembled into template groups that get assigned as a unit. A group typically contains a System Template (device-level settings like DNS, alert email, admin, logging, SNMP), an IPsec Tunnel Template, a BGP Template, a Static Route Template, an SD-WAN Template, and one or more CLI Templates. System templates can be created new, cloned, or built by importing the settings of an existing device — associate them with a profile and use Preview CLI Configuration to see the commands they will emit before install.

CLI templates come in two forms with different powers. A plain CLI script is exactly what it sounds like — FortiGate CLI commands, with metadata variables referenced by a $ prefix — and is the right choice when you need substitution but not logic. A Jinja script uses Jinja2 syntax, Python-like, supporting loops, conditionals, and arithmetic such as ipmath; metadata variables are wrapped in {{ }}, and device DB attributes are available for free. A template group can mix CLI and Jinja types, applied top-down, but a device can only receive one CLI template or one CLI template group at a time.

CLI script — plain
config system global
    set hostname $(hostname)
end

config system interface
    edit "HUB-Lo"
    set vdom "root"
    set ip $loopback_hub 255.255.255.255
    set allowaccess ping
    set type loopback
    next
end
Jinja script — with arithmetic
config system global
    set hostname {{hostname}}
end

config system interface
    edit "Branch-Lo"
    set vdom "root"
    set ip {{'10.200.99.0'|ipmath(branch_id|int)}} 255.255.255.255
    set allowaccess ping
    set type loopback
    next
end

Firewall policies for SD-WAN traffic live in a policy package. The critical rule: policies must reference SD-WAN zones, not individual members. The other interfaces in the policy are usually normalized interfaces, an abstraction that lets one policy reference "LAN" on branch1_fgt (mapped to port5) and "LAN" on branch2_fgt (mapped to port1). When FortiManager installs an object that references a normalized interface, it reads the mapping rules and assigns the correct physical port to each target device. If you plan to use an IPsec interface as an SD-WAN member you do not need to normalise it; if you use the IPsec interface directly in a firewall policy (outside SD-WAN), you do.

Configuring an SD-WAN zone is a matter of naming the zone, selecting at least one interface member, and — optionally — setting service-sla-tie-break. The tiebreaker applies to all SD-WAN rule strategies except load-balancing, and is most commonly used with the lowest-cost strategy. cfg-order uses member configuration order (default). fib-best-match selects the member with the most specific route to the destination. input-device enforces overlay stickiness — traffic received on one overlay egresses on the same overlay when possible; the CLI equivalents are set input-zone and set tie-break input-device. When you create a zone in an SD-WAN template, FortiManager automatically creates a corresponding normalized interface.

Performance SLA configuration in FortiManager mirrors FortiGate's own. The header defines the probe method — Active, Agent Based, Passive, Prefer Passive, or Remote — and the protocol (Ping, DNS, FTP, HTTP, HTTPS, TCP CONNECT, TCP ECHO, TWAMP, UDP ECHO), a server or two (best practice: two, to guard against a single-server false positive), and the members. Optional SLA Targets define latency, jitter, and packet-loss thresholds for the lowest-cost strategy. Link Status configures Check Interval (default 500 ms), Failure Before Inactive (default 5), Restore Link After (default 5), and Probe Timeout (default 500 ms). Action When Inactive lets a state change update static routes matching the member gateway, or bring alert interfaces up or down. The FortiManager map view highlights members that miss an SLA target in red and members that are alive but SLA-violating in orange, with fully healthy in green.

Common misconception: firewall policies for SD-WAN traffic must reference SD-WAN zones, never individual members. Referencing a member directly hides the rule from the SD-WAN service logic.
🧠
MENTAL NOTE
Metadata variables carry per-device values across a shared template. Template groups carry a full device shape. Normalized interfaces carry per-device port names. Together they turn one template into a whole fleet.

Finally, if you already have an SD-WAN configuration on a live device and want to templatise it, FortiManager can import the SD-WAN settings of a managed device directly into a new SD-WAN template. From SD-WAN Manager > Rules > SD-WAN, choose Import from the More menu, pick the source device, and the resulting template can then be assigned to other branches with the same shape.

💭 THINK FIRST · OVERLAY TEMPLATE

Configuring one IPsec tunnel is tedious. Configuring a dual-hub overlay with ADVPN, IBGP, and a hundred branches is a project. The overlay template is Fortinet's answer to that project — a wizard that emits an entire family of coordinated templates from a handful of design choices.

  • What does the SD-WAN overlay orchestrator actually produce when you complete the wizard, and why does that matter for how you review and install the result?
  • What are the hard restrictions on SD-WAN overlay templates — which mistake is easiest to make when combining them with VPN Manager or multiple ADOMs?
Q1 — MODEL ANSWER
The overlay template generates a coordinated set of templates — BGP, IPsec Tunnel, Static Route, and CLI — and groups them per role (HUB1, HUB2, BRANCH). For the branch devices it also adds members and zones to the SD-WAN template. Device assignment is done for you. That matters because the wizard is not the endpoint: you land in SD-WAN Manager > Templates > Template Groups where you can review each individual template, fine-tune settings — a BGP timer, an IPsec proposal, an SD-WAN rule — and only then push to devices via the Install Wizard. Reviewing before installing is not optional in practice; it is the whole point of having a template layer in the first place.
Q2 — MODEL ANSWER
All devices participating in an SD-WAN overlay template must live in the same ADOM — you cannot span an overlay across ADOMs. Each FortiGate can be configured by only one SD-WAN overlay template at a time. And VPN Manager and an SD-WAN overlay template cannot be used together on the same network. For a new deployment Fortinet recommends the overlay template because it takes a comprehensive approach that walks you through IPsec, BGP, and SD-WAN in one guided flow. Trying to mix VPN Manager with the overlay template on the same fabric is the most common structural mistake because they look complementary but overlap in what they configure.
SECTION 03

The Overlay Wizard Where Complexity Becomes Templated

Most SD-WAN deployments require complex overlay configurations for data-centre or cloud connectivity. FortiManager's SD-WAN overlay template is a wizard that automates and simplifies the process using the IPsec and BGP settings recommended by Fortinet. It walks you through the design decisions and emits a coordinated set of configuration templates — SD-WAN, overlay, IPsec, BGP, static route, CLI — for hubs and branches, so you configure the region once rather than device by device.

The workflow is disciplined. First, define the network topology and subnets you plan to use. Second, preconfigure your SD-WAN devices with administrator settings, interface IPs, and the WAN links. Third, import the devices into FortiManager. Fourth, run the SD-WAN overlay orchestration wizard. Fifth, review the generated templates. Sixth, install the configurations on devices. Branch devices can be added later to the branch device group — the region will accept newcomers as long as the shape holds.

Region settings come first in the wizard. You set the template name, description, and topology — Single HUB, Dual HUB (Primary & Secondary), Dual HUB (Primary & Primary), or Multi HUB. The Primary & Secondary mode is asymmetric — one hub is preferred by BGP; Primary & Primary weights both hubs equally. The overlay authentication method is Pre-shared Key (auto-generated by FortiManager; you can replace it) or Signature (certificate-based, more secure). Under Advanced you set the Loopback IP address (default 10.200.99.252/255.255.255.0), the BGP-AS number (default 65000), BGP on Loopback, Dynamic BGP, Route Reflection, Auto-Discovery VPN (disable, Legacy, or ADVPN 2.0 — default disabled), and Segmentation Over Single Overlay. Enabling ADVPN in the overlay template automatically enables the necessary IPsec and SD-WAN settings.

TopologyShapeTypical Use
Single HUBOne data-centre, one overlay per underlaySmall footprint or lab
Dual HUB (Primary & Secondary)Two hubs, one preferred by routingDC1 primary, DC2 failover
Dual HUB (Primary & Primary)Two hubs, equally weightedActive/active hub load-sharing
Multi HUBThree or more hubs across regionsGlobal fabrics; regional segmentation

When the wizard finishes, its output arrives in SD-WAN Manager > Templates > Template Groups. Typically you will see three groups — one for HUB1, one for HUB2, and one for BRANCH — each containing a System Template, an IPsec Tunnel Template, a BGP Template, a CLI Template, and, for branches, the SD-WAN Template and a Static Route Template with the branch underlay routes. The templates are already assigned to the correct devices or device group. At this stage the administrator can review each template, fine-tune settings, and only then use the Install Wizard to push changes.

SD-WAN member installation targets deserve a call-out. When a template defines an SD-WAN member, you can restrict which devices actually receive it — by device group, by subgroup, or by individual device. Leave the target empty and the member installs on every device the template is assigned to. This matters when the region is not perfectly uniform: some branches have three WAN links, some only two; some sites have MPLS, some do not. The example in the study guide shows a template with three members — port1 (ISP1), port2 (ISP2), and port4 (MPLS) — where port1 and port2 apply to every branch but port4 installs only on devices in the main group.

The convert single-hub to dual-hub flow is a small but telling capability: existing network settings, address objects, and routes are reused for the second hub — you simply add another device to the appropriate branch device group and pick a topology upgrade. It is the same underlying template family, promoted to a richer shape.

Restrictions to memorise: all devices participating in an SD-WAN overlay template must live in the same ADOM. Each FortiGate can be configured by only one SD-WAN overlay template. VPN Manager and an SD-WAN overlay template cannot be used together on the same network — for a new deployment, pick the overlay template.
🧠
MENTAL NOTE
The overlay template is a generator, not an install button. It emits template groups; you review them, tune them, and then install. Skipping the review step wastes the leverage the wizard just gave you.

Two knowledge-check facts worth pinning to the wall: a FortiGate you added to FortiManager but that is not showing up in SD-WAN Manager is almost always missing the Enable SD-WAN Management flag from the More menu on the Device Manager pane. And the overlay orchestration wizard supports four topology types, not two — Single HUB, Dual HUB (Primary & Secondary), Dual HUB (Primary & Primary), and Multi HUB.
