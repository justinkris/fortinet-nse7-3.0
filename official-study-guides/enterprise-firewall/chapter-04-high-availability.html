Two Boxes, One Shared Heartbeat

How FortiGate turns a pair of firewalls into a single fault-tolerant identity using FGCP, FGSP, and VRRP.

High availability is not a product — it is a promise the network keeps to itself.

Cluster election, virtual MAC addresses, session sync — every mechanism exists so a failure never becomes an outage.

💭 THINK FIRST · FGCP FUNDAMENTALS

Two identical FortiGates sit side by side. Before they can behave like one logical firewall, they must agree on identity, verify they are eligible to cluster, and continually reassure each other that they are alive. Every setting in this section — group ID, heartbeat, licensing — exists to make that agreement possible.

  • Why must every FGCP member share the same model, firmware, and licensing — what breaks if they do not?
  • What is the practical difference between active-passive and active-active mode, and when does either actually help you?
Q1 — MODEL ANSWER
Cluster members must be identical because FGCP synchronizes configuration, sessions, and hardware-dependent state between them. Different models have different ASICs, interface counts, and capacities. Different firmware means different feature schemas. Mismatched licensing forces the cluster to fall back to the lowest common denominator — if only one member has FortiGuard Web Filtering, none of them will use it. Same hard drive layout and same operating mode (NAT vs. transparent) round out the requirement so no member has a hidden capability the others cannot match.
Q2 — MODEL ANSWER
In active-passive mode one FortiGate carries all traffic while the other stands by, ready to take over on failure. It is simple and predictable. Active-active leverages the CPU and memory of the secondary by having the primary distribute proxy-inspected sessions to it. The goal is not really load balancing — the primary still receives all inbound traffic — it is to make use of otherwise-idle hardware for CPU-heavy inspection work.
SECTION 01

FGCP — the shared identity of two peers

The FortiGate Clustering Protocol (FGCP) is Fortinet's native HA solution. It lets two to four FortiGate devices operate as a single logical unit, sharing configuration, sessions, and — most visibly to the outside network — MAC addresses. FGCP supports two operating modes: active-passive, where one member forwards traffic while the others stand by, and active-active, where the primary distributes proxy-inspected sessions to secondary members to spread CPU load.

Every member of the cluster must be functionally identical. That means the same hardware model (or VM model), the same firmware version, the same licensing, the same drive layout, and the same operating mode for the management VDOM. If licensing differs, the cluster gracefully degrades to the lowest license level present — buy FortiGuard Web Filtering for only one member and none of them will use it. On the configuration side, every member must share the same HA group ID, group name, password, and heartbeat interface settings before FGCP will even form the cluster.

Two best practices are worth internalizing early. First, always define at least two heartbeat interfaces so that a single link failure cannot fracture the cluster — FGCP will use the next-highest-priority heartbeat when the first fails. Second, if you are cabling DHCP or PPPoE interfaces into the cluster, temporarily switch them to static addresses during initial setup to prevent address-assignment races, then revert once the cluster is formed.

  • HA group ID: A discriminator, 0–255 (up to 1023 with logic 4). Two clusters in the same broadcast domain must use different IDs to avoid vMAC collisions.
  • Heartbeat Ethernet types: FGCP uses 0x8890 and 0x8891 for elections, sync, and failover detection; 0x8893 is reserved for configuration synchronization.
  • Session-sync interfaces: Configured with set session-sync-dev, they offload session-sync traffic from the heartbeat link so heartbeats are never starved during high session churn.
  • Virtual clustering: Extends FGCP to multi-VDOM active-passive setups (up to 30 virtual clusters), and via VDOM partitioning lets each physical unit be the primary for a different VDOM.
FortiOS CLI — Minimal FGCP configuration
config system ha
    set mode a-p
    set group-id 10
    set group-name "Training"
    set password <password>
    set hbdev "port3" 10 "port4" 20
    set session-sync-dev port5
end
Exam tip: The heartbeat priority number after each hbdev entry is a preference — higher wins. Make sure the number matches your physical cabling; the intuitive "50" you might expect for the primary link is fine as long as the other link's value is lower.
🧠
MENTAL NOTE
FGCP is a strict club — same model, same firmware, same license, same everything. Different anywhere and the cluster either refuses to form or silently drops to the lowest common level.

💭 THINK FIRST · ELECTION & VMAC

Two peers boot at nearly the same time. Someone must be the primary — and the network downstream has to keep talking to the same MAC address after failover, or every switch will have to relearn its forwarding tables. FortiGate solves both problems with a strict election order and a set of virtual MAC address logics.

  • With HA override disabled (default), which criterion actually decides the primary — and why is uptime prioritized over priority?
  • After a failover, why does the new primary adopt the old primary's virtual MAC addresses instead of using its own hardware MACs?
Q1 — MODEL ANSWER
With override disabled, FortiGate evaluates connected monitored ports first, then HA uptime, then priority, then serial number — stopping at the first criterion that produces a winner. Uptime is prioritized over priority so a healthy running primary keeps its role even if a peer boots with a higher priority; this prevents needless failovers during maintenance windows. To force a manual failover you run diagnose sys ha reset-uptime on the current primary to zero its uptime.
Q2 — MODEL ANSWER
Downstream switches learn MACs, not identities. If the new primary used its own hardware MAC after failover, every switch would keep sending traffic to the dead primary's port until its MAC table aged out. By adopting the old primary's virtual MAC and broadcasting a gratuitous ARP, the newly elected primary essentially says "I'm still here, just on a different port" — and switches update their forwarding tables in one packet.
SECTION 02

Election, override, and the virtual MAC illusion

When FGCP boots or a failover event fires, members negotiate to choose a single primary. With the default of override disabled, the criteria — evaluated in order, stopping at the first winner — are: (1) most connected monitored ports, (2) highest HA uptime, (3) highest priority, and (4) highest serial number. Uptime coming before priority is deliberate: it prevents a peer that just rebooted with a slightly higher priority from stealing the primary role from a healthy device.

Enable override (CLI only, via config system ha / set override enable) and priority moves ahead of uptime. Now the device you configure with the highest priority will always take the primary role whenever it is available — including yanking the role back after a reboot. The trade-off is more failover events; the win is deterministic role assignment. Note that override and priority are not synchronized between members — you must enable them per device.

To force a failover you have two clean options. If override is disabled, run diagnose sys ha reset-uptime on the current primary; that command zeroes HA uptime (not system uptime) and the other member's uptime advantage takes over. If override is enabled, change the priority. The command diagnose sys ha dump-by vcluster shows the uptime difference in seconds and the reset counter for each member.

Every interface on the primary — except heartbeat and reserved-management interfaces — is assigned a virtual MAC address. When a failover occurs, the new primary adopts the same virtual MACs so downstream switches do not have to relearn anything. A gratuitous ARP is broadcast so switches update their forwarding tables. On stubborn high-end switches, you can also enable set link-failed-signal enable to briefly shut down the former primary's interfaces during failover, forcing switches to see a link event and flush their MAC tables.

💭 THINK FIRST · ACTIVE-ACTIVE FLOW

In active-active mode the primary receives every incoming packet — traffic always arrives at the primary virtual MAC. But some sessions are distributed to secondaries for inspection. Which ones, how, and why does the return path get routed the way it does?

  • Which traffic is distributed to secondaries by default, and what does the load-balance-all setting change?
  • Why does the secondary source its outbound traffic from its own physical MAC address instead of the primary's virtual MAC?
Q1 — MODEL ANSWER
By default only sessions subject to proxy inspection are distributed to secondaries — because proxy inspection is CPU-heavy and benefits most from spreading load. Enabling load-balance-all extends the distribution to flow-inspected and uninspected sessions as well. Together with the schedule setting (round-robin, ip, ipport, weight-round-robin, etc.), that determines how sessions are handed out.
Q2 — MODEL ANSWER
Return traffic from a session distributed to a secondary is sourced from that secondary's egress port — but that port is not the vMAC owner, so it uses its physical MAC. If it tried to use the primary's vMAC it would create a MAC address conflict on the LAN. The primary's vMAC is only for traffic that egresses the primary itself.
SECTION 03

Active-active — inspection load spread across otherwise-idle CPUs

The goal of FGCP active-active is not to double throughput — it is to make use of secondary devices' CPU and memory for inspection-heavy sessions. All external traffic arrives at the primary's virtual MAC. The primary then decides whether to keep a session or distribute it to a secondary. By default, only proxy-inspected sessions are distributed. To distribute flow-inspected or uninspected sessions as well, enable load-balance-all.

For a distributed proxy session the primary encapsulates the first packet inside an 0x8891 Ethernet frame so the secondary gets both the packet and the session metadata it needs to process the flow. Subsequent packets are simply MAC-rewritten: source becomes the primary's physical MAC, destination becomes the secondary's physical MAC. On the return path from server to client, the secondary sources the reply from its own physical MAC — because it does not own the primary's vMAC and using it would create conflicts.

A hard limitation to remember: FGCP active-active cannot load balance traffic that traverses inter-VDOM links. If your architecture uses inter-VDOM routing, that traffic will pin to the primary regardless.

Reality check: Because all inbound traffic still arrives at the primary, you almost always see more load on the primary than on any secondary. Active-active spreads inspection cost, not ingress cost.

MENTAL NOTE
Active-active in FGCP is a CPU-sharing arrangement, not a doubling of throughput. The primary is always the receptionist; secondaries are the inspection room.

💭 THINK FIRST · FGSP

FGCP requires a primary and load balances internally. But what if your network already has external load balancers distributing traffic to firewalls — and you just want the firewalls to share sessions so any of them can handle a return packet?

  • How is FGSP fundamentally different from FGCP in terms of who does the load balancing and how sessions are shared?
  • What session types does FGSP synchronize by default, and what configuration is needed for the rest?
Q1 — MODEL ANSWER
FGCP is a full HA solution with a primary that owns virtual MACs and distributes traffic internally. FGSP is only about sharing sessions between peer devices — there is no primary, no vMAC. External load balancers are responsible for directing traffic. If those balancers send all traffic to one peer, FGSP feels active-passive; if they spread traffic across peers, it feels active-active. The peers exchange session state so failover is seamless.
Q2 — MODEL ANSWER
By default FGSP synchronizes IPv4 and IPv6 TCP sessions plus IPsec tunnels. To sync UDP, ICMP, expectation (SIP/FTP pinholes), NAT, or SCTP sessions, you must enable the corresponding session-pickup-* flag under config system ha. You can also filter which sessions to synchronize using config session-syn-filter — matching by source or destination interface, subnet, or IPv6 range.
SECTION 04

FGSP — session sharing among independent peers

FortiGate Session Life Support Protocol (FGSP) is fundamentally different from FGCP: it is a session-sharing HA solution, not a clustering one. FGSP peers can be standalone FortiGates or existing FGCP clusters. There is no primary election; there are no virtual MACs. External load balancers distribute traffic, and FGSP simply ensures every peer knows about every session so failover is transparent.

An FGSP deployment can include 2–16 standalone FortiGates or 2–16 FGCP clusters of two members each. More peers means more synchronization traffic and more memory to keep tables aligned. If the upstream balancer directs all sessions to one peer, the deployment behaves like active-passive; if it spreads them, it behaves like active-active. The rule is that any given session's packets — including return packets — should ideally hit the same peer.

By default FGSP synchronizes IPv4 and IPv6 TCP sessions and IPsec tunnels. To include other session types you enable them explicitly:

FortiOS CLI — Extending FGSP session sync
config system ha
    set session-pickup enable
    set session-pickup-connectionless enable   # UDP, ICMP
    set session-pickup-expectation enable      # SIP/FTP pinholes
    set session-pickup-nat enable              # NAT sessions
end

FGSP peers can also do standalone configuration synchronization, effectively giving you the FGCP config-sync feature between devices that are not in an FGCP cluster. You enable this by setting mode standalone and standalone-config-sync enable. Layer-2 adjacency is required for config sync but not for session sync alone.

Three use cases worth remembering:

  • Encrypted session sync: Enable encryption and set a psksecret under config system standalone-cluster; FGSP auto-creates an IPsec tunnel plus supporting policies. Layer-3 only.
  • Layer-2 asymmetric inspection: Set layer2-connection available when peers share a subnet on internal and outgoing interfaces. Return traffic that lands on the wrong peer is bounced back at Layer 2 to the session owner.
  • Layer-3 asymmetric inspection (cloud): When no Layer 2 exists (public cloud), leave layer2-connection unavailable; peers tunnel misrouted return traffic to each other over UDP.
Remember: FGSP does not do load balancing itself. The upstream balancer decides; FGSP just makes sure any peer can process any packet in a session.
🧠
MENTAL NOTE
FGCP is one firewall pretending to be two. FGSP is many firewalls agreeing to share what they know, so an external load balancer can send any packet to any of them.

💭 THINK FIRST · VRRP & COMPARISON

Not every environment uses Fortinet-only devices. VRRP is the open-standard alternative — the one you reach for when the peer might be a Cisco router or another vendor. Knowing when to pick FGCP, FGSP, or VRRP is a strategic call that shapes the rest of your architecture.

  • How does a VRRP backup know when the primary has failed, and how can you tie that decision to a remote reachability check?
  • You need session synchronization AND you want to load balance between peers. Which HA protocol is the right fit?
Q1 — MODEL ANSWER
VRRP routers periodically send VRRP advertisements. The primary has the highest priority. If backups stop receiving advertisements — either because the primary died or because it stopped sending — the highest-priority backup takes over. You can tie the primary's advertisement to remote reachability by configuring up to two destination IPs (e.g. 8.8.8.8). If the primary cannot reach the monitored destination, it stops advertising, triggering failover even though it's still alive on the LAN.
Q2 — MODEL ANSWER
FGSP is the answer. FGCP in active-passive mode has no load balancing (VDOM partitioning gives you load sharing but not load balancing). FGCP active-active is not really about balancing traffic either. VRRP has neither session sync nor load balancing. FGSP paired with external load balancers gives you both: the balancers spread traffic and FGSP shares sessions so any peer can pick up any packet.
SECTION 05

VRRP and the right-tool-for-the-job comparison

The Virtual Router Redundancy Protocol (VRRP) is an open standard, which is what makes it interesting: FortiGate can join a VRRP domain that includes non-Fortinet routers. FortiOS supports both VRRP v2 and v3, IPv4 and IPv6, and you can add multiple virtual routers to the same physical or VLAN interface.

VRRP works through periodic advertisements. The device with the highest priority becomes primary and sends VRRP advertisements to peers. If the backup with the next-highest priority stops receiving those advertisements for the timeout period, it becomes the new primary. FortiOS extends the standard by letting you monitor up to two remote destinations (e.g. 8.8.8.8) — if the primary loses connectivity to the monitored destination, it stops advertising, deliberately triggering failover even though the LAN interface is up.

FortiOS CLI — VRRP with remote destination monitoring
config system interface
  edit port10
    config vrrp
      edit 200
        set vrip 10.10.101.250
        set priority 200
        set adv-interval 250
        set vrdst 8.8.8.8
        set vrdst-priority 30
      next
    end
  next
end

Choosing between FGCP, FGSP, and VRRP is a deliberate architectural decision. FGCP is native, gives you redundancy, load sharing, session sync, and configuration sync — but only among 2–4 identical FortiGates. FGSP is redundancy plus session and config sync between standalone units or between clusters, requiring an external load balancer for actual distribution. VRRP is the cross-vendor open-standard fallback: unlimited scale in principle, but no session sync, no load balancing, no config sync.
