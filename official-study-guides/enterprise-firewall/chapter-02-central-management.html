Chapter 02 · NSE7 EF 7.6

One Console for the Whole Fabric

How FortiManager separates device settings from policy, tames per-device variance, and turns manual firewall administration into templated automation.

FortiManager is not just a UI — it is a discipline of layers.

Device layer vs policy layer, per-platform vs per-device, dynamic vs static: every operation in this chapter is really a question about which layer you are touching.

💭 THINK FIRST · ADOMS & LAYERS

FortiManager stores every FortiGate's configuration twice — once at the device layer and once at the policy layer — and wraps both inside an Administrative Domain (ADOM). Before you can predict what an "install" will do, you must know which layer you are editing and which ADOM you are inside.

  • Why does FortiManager split each ADOM into a separate device layer and policy layer?
  • What is the point of a Global ADOM, and how does its version compatibility affect local ADOMs?
Q1 — MODEL ANSWER
The device layer holds settings that are inherently per-device — network interfaces, VPN endpoints, system parameters, log/report destinations — and each FortiGate has its own device database and revision history. The policy layer holds the objects and rules that are meant to be reusable across many devices — firewall policies, security profiles, address objects, users. Splitting them lets one policy package be deployed to many devices while each device keeps its own physical reality.
Q2 — MODEL ANSWER
The Global ADOM stores objects and policies you want shared across many ADOMs, so you can push a single policy set to multiple sites. Global objects are prefixed with a reserved g and the Global ADOM is compatible with local ADOMs at the same version or up to two versions earlier — a Global ADOM at 7.6 supports local ADOMs at 7.6, 7.4, and 7.2. Mismatched versions mean sync problems.
SECTION 01

ADOMs, Device Layer, Policy Layer

FortiManager is a centralized management solution featuring ADOMs (Administrative Domains), device databases, policy layers, and provisioning templates. It supports global-ADOM capabilities and streamlines configuration across every managed Fortinet device from one console. When ADOMs are not enabled, everything lives in a default ADOM named root, which itself contains a device layer, a policy layer, and its own set of ADOM functions.

When you enable multiple ADOMs, each becomes a virtual instance of FortiManager with its own scripts, provisioning templates, SD-WAN Manager, VPN Manager, AP Manager, FortiSwitch Manager, Extender Manager, Fabric View, and FortiAnalyzer capability. FortiGuard serves all ADOMs — the root, the 7.6, the 7.4, the 7.2 — and the ADOM version must remain compatible with the FortiOS on the FortiGate devices it manages. Mixing FortiOS versions inside an ADOM is allowed (especially for migration) but you should keep firmware aligned for stable synchronization.

Objects in the device layer — Network, VPN, System, Log & Report — are per-firewall by nature and produce a revision history for each managed FortiGate. Objects in the policy layer — Policy, Firewall Objects, Security Profiles, User & Authentication, Security Fabric, Advanced — are meant to be portable across many devices via a policy package. The rule of thumb is straightforward: even though you can edit policies in the device layer, don't. Make policy changes in the policy layer so the policy package stays the source of truth.

Remember: Device layer changes hit FortiGate directly after install. Policy layer changes only reach FortiGate after you install (or reinstall) the policy package.

Global objects are reserved with the prefix g to prevent collision with custom objects. That reservation matters — if you name a custom object gsomething in a local ADOM, you invite conflicts the moment you push a global policy that includes an identically named global object.

🧠
MENTAL NOTE
Device layer = physical reality per box. Policy layer = shared intent across many boxes. ADOM = the walls around one workspace. Global ADOM = the roof over many workspaces.

💭 THINK FIRST · INSTALL OPERATIONS

"Install" and "retrieve" are not the only verbs. FortiManager has eight distinct sync operations, each with a different direction of travel and a different scope. Get them wrong and you can silently overwrite a working configuration.

  • What is the difference between Install device settings and Quick install, and when would you pick each?
  • Why does Auto-retrieve capture drift into revision history but NOT update the policy package?
Q1 — MODEL ANSWER
Both push the device layer configuration onto FortiGate. Install device settings offers a preview before install so you can review the delta — the safer default when the changes are non-trivial or unfamiliar. Quick install skips the preview and applies immediately — ideal for zero-touch provisioning (ZTP) or when you already know exactly what is being pushed.
Q2 — MODEL ANSWER
Auto-retrieve is scoped to the device layer's revision history. When someone edits FortiGate directly, auto-retrieve records the change as a new revision so the device layer catches up. The policy package is a separate database, and syncing changes into it requires an explicit Import — which is deliberate, because importing can conflict with existing objects in the policy layer.
SECTION 02

Retrieve, Install, Revert, Reinstall, Import — the Eight Verbs

The device-and-policy split leads directly to a family of install and sync operations. Each verb has a specific direction and a specific target — memorize them and most FortiManager confusion evaporates.

  • Retrieve: Captures the current FortiGate configuration and saves it as a new revision. Does not touch the policy package.
  • Auto-retrieve: Runs automatically when a FortiGate is added to the device layer, and captures any drift the moment it happens. Records into the device revision history; the policy package still needs a separate Import to catch up.
  • Install device settings: Applies the device layer configuration to FortiGate, with a preview to review changes first.
  • Quick install: Same target as Install device settings, but no preview. Best for ZTP or when you are confident.
  • Revert: Chooses an earlier device revision from the history for potential import or installation. To synchronize device and policy layers after a revert, you still need to import.
  • Import: Moves policies from a selected revision in the device layer into the policy package. Beware of naming collisions with policy-layer objects.
  • Install policy package: Transfers the policy package from the policy layer to both the device layer and FortiGate, with a step-by-step wizard and installation preview.
  • Reinstall policy: Bypasses the wizard because the FortiGate and package are already selected. Offers an installation preview and applies policy-package settings plus any modifications directly to FortiGate in the device database.
Common misconception: Editing FortiGate directly is not caught by the policy layer. Auto-retrieve updates the device revision history only. Sync the policy package by running Import explicitly — and expect object-name conflicts if any of those imported names already exist.

Global policy packages have their own installation model. Automatically install policies to ADOM devices lets a global ADOM push a policy package into three targets at once — the local ADOM policy layer, the device layer, and FortiGate — bypassing the install preview. Assign is the calmer version: the global ADOM updates the policy layer and marks it modified, and you complete the push with a manual install or reinstall.

🧠
MENTAL NOTE
If the change originated on FortiGate, the sync verb is Retrieve or Import. If the change originated on FortiManager, the verb is Install or Reinstall. Direction is everything.

💭 THINK FIRST · DYNAMIC OBJECTS

If every branch has an interface called LAN but each branch's LAN sits on a different subnet, you either write one policy per branch — which does not scale — or you teach FortiManager how to substitute per-device values behind a single shared name.

  • How does per-device mapping let one policy-layer LAN address object mean different subnets on different FortiGates?
  • Why do some object types (like web filter profiles) not support per-device mapping, and how do you work around it?
Q1 — MODEL ANSWER
Per-device mapping on a firewall address named LAN maps Branch1 to 172.16.0.0/24 and Branch2 to 192.168.0.0/24 under the same object name. When the policy is installed to Branch1, FortiManager substitutes the Branch1 value; when it's installed to Branch2, it substitutes the Branch2 value. One shared policy, N per-device realities.
Q2 — MODEL ANSWER
Certain object types — including web filter profiles, application signatures, and protocol options — don't support per-device mapping. If different settings are needed per FortiGate, you create uniquely named profiles for each variant (for example web-filter-branch1, web-filter-branch2) rather than reusing one name.
SECTION 03

Normalized Interfaces, Per-Device Mapping, Metadata Variables

Dynamic mapping is the mechanism that keeps one shared policy usable across many FortiGates. Its three main forms are normalized interfaces, object configurations with per-device mapping, and metadata variables.

Normalized Interfaces standardize interface references. Two configuration modes exist: per-platform mapping applies uniform settings across similar device models, and per-device mapping allows customized configurations for individual devices. For example, port3 on a FortiGate VM can be standardized across similar models via per-platform, while port4 — mapped to the LAN interface — takes a per-device mapping so it can vary per firewall. Normalized interfaces are case sensitive (LAN and lan are different objects), and can be assigned in both the policy layer and the device layer.

Object configurations extend the same idea to firewall objects. Under Policy & Objects > Firewall Objects > Addresses, the LAN address object can be given per-device mappings — Branch1 to 172.16.0.0/24, Branch2 to 192.168.0.0/24 — so the same policy substitutes the right value per firewall. Not every object type supports this: the study guide splits objects into Available and Unavailable lists for per-device mapping. Available includes Addresses, VIPs, IP Pools, User Groups, LDAP servers, and metadata variables; Unavailable includes AntiVirus, Web Filter, DNS Filter, IPS Signatures, and Threat Feeds.

Watch: If you import a device-layer configuration containing an object type that the policy layer does not support for per-device mapping, FortiManager is forced to choose between the FortiGate value and its own. The safe practice is to identify which object types are eligible and give the un-eligible ones unique names before importing.

Metadata variables — displayed with a magnifying-glass and dollar-sign icon — are the third piece. Type $ in a supported field to invoke a variable, and FortiManager will let you assign default values and per-device overrides under Policy & Objects > Advanced > Metadata Variables. Metadata variables can appear in policy-layer object fields and in provisioning templates and CLI templates, which is what makes them the workhorse of dynamic per-firewall configuration.

💭 THINK FIRST · ZTP, LTP & TEMPLATES

You cannot fly to every branch to configure a new FortiGate. Zero-touch and low-touch provisioning let a fresh box out of the shipping crate discover its manager, adopt its policy, and go live — as long as you designed the model device and templates in advance.

  • What is the difference between ZTP and LTP in terms of preconfiguration on the FortiGate?
  • Why do pre-run CLI templates auto-detach after installation, while regular CLI templates do not?
Q1 — MODEL ANSWER
In ZTP, the new FortiGate connects to the WAN/internet automatically and links to FortiManager with no preconfiguration — genuinely zero-touch. In LTP, you must do minimal initial setup (network settings, FortiManager location) before FortiGate can discover and connect. LTP is what you use when the branch environment cannot deliver ZTP's automatic connectivity.
Q2 — MODEL ANSWER
Pre-run CLI templates exist for one-time, provisioning-time tasks. Once quick install pushes them and the real FortiGate joins, FortiManager auto-unassigns the pre-run template from the device database — it does not linger in the ongoing configuration. Regular CLI templates are meant for continuous use and stay assigned; they overwrite the device database on install so ongoing changes need to be reflected there.
SECTION 04

Zero-Touch, Low-Touch, and Templates

FortiManager provisions new FortiGates through model devices. You configure a model device before the physical FortiGate arrives — identified by serial number or a pre-shared key — and the model carries the intended settings and policies. When the real device connects, FortiManager's auto-link process installs the settings from the model onto it, and then replaces the model device with the newly linked real device.

How that new FortiGate discovers FortiManager decides whether the flow is ZTP or LTP. In Zero-Touch Provisioning (ZTP) the FortiGate autonomously reaches the WAN and links to FortiManager with no preconfiguration — as close to unbox-and-walk-away as the industry gets. In Low-Touch Provisioning (LTP) you do a minimal initial setup on the FortiGate (network config, FortiManager location) before it can find its manager. Beyond FortiManager, other ZTP/LTP methods include FortiSOAR with FortiManager, Ansible, Terraform, custom API calls, FortiProvision, FortiZTP, and FortiDeploy.

Pre-run CLI templates are the provisioning-time cousins of regular CLI templates. They exist for one-time tasks and do not remain in the FortiGate device database after installation. The flow: the administrator creates a pre-run CLI template and assigns it to a model FortiGate; quick install pushes the template's configurations to the device database; FortiManager auto-unassigns the pre-run template after the quick install; when the new FortiGate connects, if Automatically Link to Real Device is enabled, the configurations are auto-applied; otherwise the admin manually pushes them after the FGFM (FortiGate-FortiManager Communication Protocol) tunnel is up.

CLI templates, in contrast, are designed for repeated use. They overwrite the device database and stay assigned. Both pre-run and standard CLI templates support metadata variables and both CLI and Jinja scripting.

Practical distinction: Pre-run = one-shot bootstrap; regular CLI template = ongoing configuration source of truth. If a CLI template is going to be re-applied whenever you install device settings, treat it as the primary place to make changes — don't work around it.
🧠
MENTAL NOTE
Provisioning templates override the device database. Set them once, name them well, and treat them as the authoritative version of your device layer.

💭 THINK FIRST · SCRIPTS

A FortiManager script can run in three completely different places — directly on FortiGate, on the device database, or on a policy package — and each destination has its own consequences for synchronization. Choose the wrong destination and you will spend the afternoon reconciling drift.

  • What happens to FortiGate, the device database, and the policy layer when a script runs "Remote FortiGate Directly (via CLI)" versus "Device Database" versus "Policy Package or ADOM Database"?
  • Which language/format would you pick for automation that needs dynamic configuration schemas — Tcl, Jinja, or plain CLI?
Q1 — MODEL ANSWER
Remote FortiGate Directly: script runs on FortiGate itself; auto-retrieve syncs the device database provided the changes are not firewall policies. Device Database: script edits the FortiGate's device database on FortiManager — Config Status shows Modified, and you must install device settings to push. Policy Package or ADOM Database: script edits the policy layer — Policy Package Status shows Modified, and you must install the policy package to push objects and policies to the device layer and FortiGate.
Q2 — MODEL ANSWER
Jinja templates (written in Python) provide dynamic configuration schemas with metadata variables — the right fit when you need programmatic templating. Tcl handles automation tasks and also supports dynamic variables. Plain CLI is simplest and integrates natively with FortiOS, but lacks dynamic variables — pick CLI templates instead when you need variables in an otherwise CLI-shaped script.
SECTION 05

CLI, Tcl, Jinja — Three Places, Three Languages

You can execute scripts on FortiManager in three contexts: directly on FortiGate, in the device database, and on the policy package (or ADOM database). Each context has different sync behavior.

  • Remote FortiGate Directly (via CLI): The script runs on the FortiGate. Auto-retrieve then updates the device database automatically — provided the changes do not involve firewall policies. Policy changes still require manual intervention to sync with the policy layer. Useful for tasks like installing a CA certificate directly.
  • Device Database: The script modifies the FortiGate's device database on FortiManager. Config Status becomes Modified. To finish, run Install device settings (or a policy install if policy changes are also required). Ideal for creating administrator profiles and other device-layer objects.
  • Policy Package or ADOM Database: The script creates or modifies objects inside a policy package — for example, a firewall address plus a policy referencing it. Policy Package Status becomes Modified; install the policy package to push both objects and rules to the device layer and FortiGate. Objects added to the package are installed; objects not referenced by the package remain in the ADOM database.

Scripts in the global database also require manual action to apply changes to global policies — either through Assign or through automatic installation. Scripts can be written in CLI or Tcl, and CLI templates can additionally use Jinja for Python-style templating.

Use-case shorthand: Solid FortiOS admins should default to CLI scripts and CLI templates. Programmers reach for Tcl, Jinja, or JSON API — Ansible, Terraform, Postman, and Swagger UI are the surrounding ecosystem.
Example: Create a firewall address from a policy-package script
config firewall address
    edit "DMZ_Address"
        set subnet $DMZ_LAN_ADDRESS 255.255.255.0
    next
end
config firewall policy
    edit 1
        set srcaddr "DMZ_Address"
        ...
    next
end

The $DMZ_LAN_ADDRESS reference above is a metadata variable — the same value can be defaulted at the ADOM level and overridden per FortiGate via per-device mapping. That is the payoff of the dynamic-object machinery from Section 3: it plugs cleanly into templates and scripts.

MENTAL NOTE
Every script is a bet on where the change should live. Pick the destination before you write a single line — and remember that Modified is the flag that says "you still owe FortiGate an install."
