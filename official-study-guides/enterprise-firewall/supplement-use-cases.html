Putting the Pieces Together — Enterprise Firewall Use Cases

How the same FortiGate stack — VDOMs, OSPF, ADVPN, BGP, Security Fabric, FortiManager automation — combines to solve five archetypal enterprise deployments.

The features are individually memorable. The combinations are what pass the exam and ship the network.

This is where isolated skills — VDOMs, OSPF areas, IPsec templates, BGP route reflectors, Security Fabric automation — collide into working reference architectures.

💭 THINK FIRST · ISFW SEGMENTATION

An HR subnet inside HQ needs to be walled off from the rest of the LAN, still reach the corporate web server, still browse the internet — but nothing else. The tool is an internal segmentation firewall (ISFW). The design question is: where does the segment live logically, how does it get routed, and how do you police what traffic crosses the border?

  • Why put the HR segment in its own VDOM and use an inter-VDOM link instead of just adding another interface to the root VDOM?
  • Why is OSPF (rather than static routes) the right tool for connecting the new HR VDOM to the rest of the HQ network?
Q1 — MODEL ANSWER
A separate VDOM gives HR its own routing table, its own policy set, and its own admin scope. Traffic between HR and root is forced through a routed VDOM link where firewall policies can be applied — you get true segmentation, not just VLAN isolation, and blast-radius is capped if HR gets compromised.
Q2 — MODEL ANSWER
OSPF lets both sides of the VDOM link learn each other's networks automatically without manual per-subnet static routes on both VDOMs. Adding a new HR subnet or a new HQ subnet later just means adding it to the correct area — routing takes care of itself. In a stub area configuration, HR gets a default route to the backbone without receiving external LSAs it doesn't care about.
USE CASE 01

ISFW Segmentation with Dynamic Routing — walling off HR from the corporate LAN

An internal segmentation firewall (ISFW) sits inside the corporate LAN and enforces policy at trust boundaries between departments, environments, or sensitivity tiers. The canonical use case is separating a sensitive segment — HR, PCI cardholder, OT/ICS — from general employee traffic, without exposing that segment to the internet edge.

In the reference topology, HQ-ISFW is a FortiGate managed by FortiManager with a root VDOM handling general traffic and a new Zone2 VDOM that will hold the HR segment. HQ-PC-3 lives on VLAN102, tagged on port5, and the design goals are three: (1) implement segmentation on the HR network, (2) add dynamic routing so both VDOMs learn each other's subnets, and (3) allow the HR segment to reach the corporate web server and the internet.

Segmentation. On HQ-ISFW, create VLAN102 on port5 with VLAN ID 102, bind it to the Zone2 VDOM, assign IP 10.0.3.254/24, and permit PING + HTTPS on the interface. Then create an inter-VDOM link pair (e.g. C0 in root at 172.16.1.249/30 and C1 in Zone2 at 172.16.1.250/30) so the two VDOMs can route to each other. FortiManager's Install Wizard pushes the device settings.

Dynamic routing. Configure OSPF on both VDOMs so they exchange the HR subnet automatically. On HQ-ISFW's Zone2: Router ID 0.0.0.4, Area 0.0.0.1 (type STUB), Networks 172.16.1.248/30 and 10.0.3.0/24. On the root VDOM: Router ID 0.0.0.3, Area 0.0.0.1 (type STUB), Network 172.16.1.248/30. Both sides of the VDOM link now participate in the same OSPF area; the HR subnet is injected into the backbone area upstream from the root VDOM.

Allow the HR segment to reach the corporate web server and internet. The final piece is firewall policy. On the HQ-DCFW policy package, create a new firewall policy that lets port5 traffic (source: all HR subnet) reach the HQ-Web-1 server via the appropriate outgoing interface. On the HQ-ISFW policy package (both root and Zone2), scripts pushed from FortiManager add the required policies so traffic can egress toward the internet via the DCFW. Test:

  • Basic connectivity: from HQ-PC-3, ping 10.0.3.254 (the VLAN102 IP).
  • Dynamic routing: on HQ-ISFW root, run get router info ospf status; on HQ-DCFW, get router info routing-table all — you should see the 10.0.3.0/24 subnet appear as an OSPF inter-area route.
  • Segmentation: from HQ-PC-3, ping HQ-Web-1 (10.0.5.11) — should succeed. Browse internet — should succeed. From HQ-DCFW, ping HQ-PC-3 (10.0.3.52) — should fail, confirming that reverse-direction traffic is denied.
Exam tip: a STUB area drops all Type 5 external LSAs, so it is ideal for a leaf segment like HR that only needs a default route to reach the outside world. Don't advertise a full internet routing table into a segment that doesn't need one.
🧠
MENTAL NOTE
ISFW segmentation is a VDOM + inter-VDOM link + OSPF stub area + one-way policy. The pattern repeats across HR, PCI, guest, OT, and dev/prod separation — same recipe, different labels.

💭 THINK FIRST · DCFW HUB & ADVPN

Two branches need to reach the HQ data center, and occasionally need to reach each other. A pure hub-and-spoke IPsec design routes branch-to-branch traffic through HQ — inefficient and slow. ADVPN (auto-discovery VPN) lets the spokes negotiate direct tunnels on demand while still registering with the hub. The design question is: how do you enable that without turning the hub into a bottleneck?

  • What is the difference between the hub's auto-discovery-sender role and the spokes' auto-discovery-receiver, and what actually triggers a shortcut tunnel between spokes?
  • Why is IBGP (rather than OSPF) usually the routing protocol of choice over ADVPN?
Q1 — MODEL ANSWER
The hub, as sender, watches for spoke-to-spoke traffic and sends a shortcut-offer notification to the two spokes involved. The spokes, as receivers, use that offer to negotiate an on-demand IKE session directly between themselves. Once the shortcut tunnel is up, traffic flows spoke-to-spoke without transiting the hub — but the hub still owns route learning and policy distribution.
Q2 — MODEL ANSWER
IBGP scales better across large spoke counts, supports route-reflector designs that eliminate full-mesh IBGP, and gives per-neighbor policy control (prefix lists, MED, community). OSPF over ADVPN would flood LSAs to every spoke through the hub, and adjacency management on dynamically formed shortcuts is fragile. IBGP with the hub as RR is the recommended enterprise pattern.
USE CASE 02

DCFW Hub with ADVPN and IBGP — hub-and-spoke that speaks directly when it needs to

A data center firewall (DCFW) role suits FortiGates that anchor site-to-site VPN topologies from behind the internet edge. The task here: HQ-NGFW acts as the hub, BR2-FGT-1 and BR3-FGT-1 as spokes; auto-discovery VPN (ADVPN) enables direct spoke-to-spoke tunnels on demand; and IBGP distributes internal routing across the fabric using the hub as a route reflector.

Hub-side VPN. On the Core1 VDOM of HQ-NGFW, activate the FortiManager provisioning template HUB_IPsec_Recommended with Template Name HUB_overlay1, ADVPN enabled, Outgoing Interface port2, IPv4 Start 172.16.1.2, End 172.16.1.254, Netmask 255.255.255.0, Pre-shared Key 123456789. Then edit the resulting VPN tunnel and configure Mode Config disabled, Transport UDP, Local ID fortinet, and in Tunnel Interface Setup, IP 172.16.1.1/32, Remote IP 172.16.1.254/24. Enable auto-discovery-sender.

Spoke-side VPN. Activate the BRANCH_IPsec_Recommended template with Template Name Branches_overlay1, ADVPN enabled, Outgoing Interface port2, Local ID fortinet, Remote Gateway 100.65.0.101 (the hub public IP), PSK 123456789. Edit Branches_overlay1 > HUB1-VPN1, disable Mode Config, set Transport UDP, and in Tunnel Interface Setup, IP $(Tunnel_IP), Remote IP 172.16.1.1/24. Enable auto-discovery-receiver on each spoke.

Per-device metadata variables. To keep one template but different values per spoke, define FortiManager metadata variables under Policy & Objects > Advanced:

VariableBR2-FGT-1BR3-FGT-1
Tunnel_IP172.16.1.2/32172.16.1.3/32
Router_ID172.16.1.2172.16.1.3
LAN_BR172.20.2.0/24172.20.3.0/24

Traffic policies. On the hub, allow traffic (a) from spokes into the Core1 VDOM, (b) from Core1 out to the spokes, and (c) between the spokes (this is what triggers the ADVPN shortcut). On each spoke, allow traffic to and from the hub. Scripts pushed from FortiManager (Device Manager > Scripts > Hub / Branch) drop those policies in place.

IBGP. Enable BGP in FortiManager's provisioning templates. On the hub, create a BGP template with Name Hub_overlay1, Local AS 65100, Router ID 172.16.1.1. Under Neighbor Group, define advpn with Remote AS 65100, Route Reflector Client enabled, Next Hop Self enabled. Under Neighbor Range: Prefix 172.16.1.0/255.255.255.0, Neighbor Group advpn, Max Neighbor Number 2. Networks: 10.0.12.0/24.

On the spokes, create BGP template Branches_overlay: Local AS 65100, Router ID $(Router_ID), Neighbor IP 172.16.1.1, Remote AS 65100, Connect Timer 120. Networks: $(LAN_BR). Assign Branches_overlay to BR2-FGT-1 [root] and BR3-FGT-1 [root], and Hub_overlay1 to HQ-NGFW [Core1]. Install device settings.

FortiOS CLI · Verify tunnels and ADVPN routing
# On BR2-FGT-1
diagnose vpn tunnel list
# Look for HUB1-VPN1 (to hub) and dynamically-created VPN1_0 (spoke shortcut)

# Confirm the routing table
get router info routing-table all
# 172.20.3.0/24 [200/0] via 172.16.1.3 (recursive via HUB1-VPN1_0)
# — dynamically discovered spoke, route via the shortcut tunnel

# Test the shortcut path
BR2-FGT-1 # execute ping-options source 172.20.2.254
BR2-FGT-1 # execute ping 172.20.3.254

On the hub's Core1 routing table you'll see OSPF inter-area, static, and BGP routes coexisting — 172.16.1.0/24 connected via VPN1, and 172.20.2.0/24 and 172.20.3.0/24 as BGP routes originated by the spokes.

🧠
MENTAL NOTE
ADVPN turns hub-and-spoke into "dynamic full mesh when the traffic demands it." Combine it with IBGP + the hub as route reflector and you have a scalable, self-healing WAN topology.

💭 THINK FIRST · DISTRIBUTED NGFW

Building the two branches — BR2-FGT-1 and BR3-FGT-1 — as spokes in the earlier exercise wasn't just about VPN. Each branch is itself a next-generation firewall protecting a local LAN, so the "distributed NGFW" use case is really the story of how one central template and a handful of per-device variables let you run tens or hundreds of identical branches.

  • Why are FortiManager metadata variables ($(Tunnel_IP), $(Router_ID), $(LAN_BR)) preferable to per-branch device settings edited by hand?
  • What is the difference between a FortiManager script and a provisioning template, and when would you use each?
Q1 — MODEL ANSWER
Metadata variables let a single template describe every branch. Adding a new branch means creating a Per-Device mapping row, not editing the template. Configuration drift is impossible: every device's overlay tunnel IP, router ID, and LAN subnet come from the same source of truth. Scaling from 5 to 500 branches is a spreadsheet exercise, not a re-implementation.
Q2 — MODEL ANSWER
A provisioning template holds structured configuration (IPsec tunnels, BGP, SD-WAN, system settings) that FortiManager renders into device config on install — think of it as a declarative reusable block. A script is a raw CLI blob (or Tcl snippet) run against one or more devices at install time; scripts are ideal for one-shot per-device commands (custom firewall policies, one-off tweaks) that don't fit a template.
USE CASE 03

Distributed NGFW — one template, many branches

The distributed NGFW pattern is when identical branch offices each run their own NGFW with local inspection, local internet egress (or backhaul through the hub), and a VPN back to HQ. FortiManager makes this scalable through provisioning templates parameterized by metadata variables — the same idea as environment variables in a CI pipeline, applied to firewall configuration.

The pattern. Define the template once (IPsec tunnel, BGP, LAN interface, firewall policy shell), reference variables everywhere a value would differ between branches ($(Tunnel_IP), $(Router_ID), $(LAN_BR)), and set the per-device values in Policy & Objects > Advanced > Metadata Variables. When you add a new branch, you fill in three cells in a spreadsheet — not a page of CLI.

Templates vs. scripts. FortiManager provides two mechanisms for pushing configuration:

  • Provisioning Templates — declarative, structured, reusable. IPsec tunnel templates, BGP templates, SD-WAN templates, System templates. Assign to a device (or device group) and re-render on install.
  • Scripts — CLI or Tcl. Run against a device (or group) at install time. Ideal for policy additions, one-off tweaks, or anything the templates don't cover — like Hub and Branch scripts that added the firewall policies for our ADVPN traffic in the previous use case.

Per-branch install workflow. Once templates and variables are in place, the operational loop is: (1) add the new branch device to FortiManager, (2) assign the appropriate templates (IPsec, BGP, SD-WAN, System), (3) create Per-Device Mappings for the metadata variables, (4) assign the scripts, (5) click Install Wizard. FortiManager renders the templates, executes scripts, and installs the resulting configuration. Verify with diagnose vpn tunnel list on the new spoke and get router info bgp neighbors to confirm the IBGP session comes up with the hub.

Local inspection. The distributed NGFW model implies each branch does its own security inspection — flow-based UTM at the branch is preferred (see Chapter 10 · Best Practices): NP/NTurbo offload keeps CPU utilization sane, and the hub doesn't become a bottleneck for every UTM decision. Proxy-based inspection can still live on specific policies where deep content examination is needed (guest wifi, high-risk applications), but as a rule the branch is a fast-path device.

Remember: the same reference architecture works for 5 branches or 500. What changes is the number of rows in the metadata-variables spreadsheet, not the number of templates.
🧠
MENTAL NOTE
Distributed NGFW = one template + one script + N metadata rows. Templates express the pattern; variables express the differences; scripts fill the gaps templates can't reach.

💭 THINK FIRST · SECURITY FABRIC

Individually, HQ-NGFW, HQ-DCFW, and HQ-ISFW are excellent firewalls. Collectively — if they don't share telemetry, topology awareness, and automation triggers — they are three silos. The Security Fabric is Fortinet's answer: a root-and-leaf trust relationship that lets FortiGates, FortiAnalyzers, and FortiManagers act as one platform.

  • What is the difference between a "Fabric Root" FortiGate and one that joins an existing fabric, and why can only one FortiGate be root?
  • How does an admin authorize a downstream device once it requests to join, and where is that seen from the root's GUI?
Q1 — MODEL ANSWER
The Fabric Root anchors the trust hierarchy — it's the FortiGate every other fabric device reports up to, and the one that aggregates topology, logs, and alerts. Downstream FortiGates set their role to "Join Existing Fabric" and point to the root's upstream IP/FQDN. Only one root because the fabric is a tree; multiple roots would fragment topology view and duplicate telemetry.
Q2 — MODEL ANSWER
On the root's GUI, System > Firmware & Registration shows pending device requests. Refresh the page after downstream setup to see HQ-ISFW and HQ-DCFW appear with status "Online" and registration "Pending" — approve, and their status flips to "Registered." Only registered devices contribute to Physical Topology and other fabric views.
USE CASE 04

Security Fabric Integration — from three firewalls to one platform

The Security Fabric lets multiple FortiGates plus FortiAnalyzer, FortiManager, and other Fortinet products share telemetry, topology awareness, and automation. Enabling it takes three moves: designate a root, join downstream devices, and authorize the join requests.

Root configuration. On HQ-NGFW-1 (Global VDOM), navigate to Security Fabric > Fabric Connectors > Security Fabric Setup and configure Security Fabric role Serve as Fabric Root, Allow other Security Fabric devices to join on port5, and Fabric name fortinet. Optionally enable FortiCloud account enforcement, allow downstream device REST-API access, and register the fabric as a global object.

Downstream setup. On HQ-ISFW and HQ-DCFW, configure Security Fabric > Fabric Connectors > Security Fabric Setup with role Join Existing Fabric, Allow other Security Fabric devices to join on port3 for HQ-ISFW and port2 for HQ-DCFW, Upstream FortiGate IP/FQDN 10.0.12.254, and SAML Single Sign-On Manual. These downstream FortiGates now advertise themselves upstream.

Authorization. Back on HQ-NGFW-1 Global, navigate to System > Firmware & Registration. Refresh — you should see HQ-ISFW and HQ-DCFW appear with status Online. Their registration status will initially be pending; authorize each, and they become Registered. Only registered devices contribute to fabric views.

Common misconception: if HQ-ISFW and HQ-DCFW don't appear on the root, the two most common causes are (1) the downstream device's upstream IP isn't reachable from the port you designated to receive fabric traffic, or (2) the fabric name mismatches. Both must match exactly.

Fabric view verification. On the FortiManager GUI, navigate to EFW_Use_Case > Fabric View > Physical Topology. Select Device Count to see all devices correctly. You may need to log out and log back in to see the Physical Topology menu after enabling fabric on the FortiGates. The diagram will show the fabric tree: FortiManager as management, HQ-NGFW-1 (Fabric Root), HQ-DCFW and HQ-ISFW below.

🧠
MENTAL NOTE
The Security Fabric is a tree of trust: one root, N downstream, all sharing topology and telemetry. Without it, your FortiGates are excellent — but blind to each other.

💭 THINK FIRST · HYBRID MESH & AUTOMATION

A hybrid-mesh scenario runs cloud-hosted workloads alongside on-prem branches and a private data center — the fabric described so far is exactly that shape. The final piece is automation: any configuration change should automatically snapshot the affected device's config to disk, so drift and human error are recoverable.

  • What are the three parts of an Automation Stitch (Trigger, Action, Stitch), and why does Fortinet separate them?
  • Which log entries confirm that a stitch actually fired, and where do you look?
Q1 — MODEL ANSWER
A Trigger is an event (Configuration Change, High CPU, IOC detection, HA failover). An Action is the response (backup config to disk, email admin, run a webhook, quarantine an endpoint). A Stitch binds one or more triggers to one or more actions. Separation lets you reuse: one "Backup Config Disk" action can wire to a configuration-change trigger on one device and to a scheduled trigger on another.
Q2 — MODEL ANSWER
On the affected device's GUI, Log & Report > System Events > Logs shows entries like "stitch:Config_Change is triggered" and "Automation stitch triggered." The corresponding Revision entry under admin > Configuration > Revisions shows a comment like "Autod backup config by stitch: Config_Change" — the config snapshot itself is stored, viewable, and restorable.
USE CASE 05

Hybrid Mesh & Automation — self-recording change control

The hybrid mesh scenario ties everything together: an HQ campus with ISFW segmentation, a data center firewall as ADVPN hub, two (or two hundred) branch NGFWs, and cloud workloads reachable via VPN. All of it is orchestrated by FortiManager, monitored by FortiAnalyzer, and observed as a Security Fabric. What's left is closing the loop: making the fabric self-record every change so audit, rollback, and drift detection are automatic.

Automation Stitches. On each FortiGate, enable the automation-related Feature Visibility toggles: Automation Stitch, Automation Trigger, Automation Action. On the FortiManager side, this is done under Device Manager > Managed FortiGate > <device> > Feature Visibility > Customize.

Build the backup stitch. On HQ-NGFW, navigate to Security Fabric > Automation Stitch > Create New:

  • Name: Config_Change
  • Trigger: Configuration Change — fires when any admin makes a config change
  • Action: Backup Config Disk — writes the running config to disk with an autogenerated revision label

Install via Install Wizard. Repeat as needed for HQ-DCFW and HQ-ISFW to close the same loop on those devices.

Retrieve revisions from FortiManager. On FortiManager, navigate to Device Manager > Managed FortiGate > HQ-NGFW > Summary > Configuration and Installation > Revision > Total Revision, click the revision history icon, and Retrieve Config. The revision list will show every automatic snapshot, timestamped and labeled with the stitch that produced it.

Test the automation. On HQ-DCFW SSH, run a trivial config change:

FortiOS CLI · Trigger the automation stitch
config system interface
    edit port2
        set description modification
    next
end
exit

On the HQ-DCFW GUI, navigate to admin > Configuration > Revisions and expand the current firmware build (e.g. 7.6.3 build 3510). You should see a new entry with username daemon_admin, timestamp of the change, and comment "Autod backup config by stitch: Config_Change."

Cross-check on Log & Report > System Events > Logs: entries include "stitch:Config_Change is triggered" (Notice level), "Configuration is changed" (Alert Notification), the actual interface edit ("Edit system:interface port2"), and the surrounding admin session events. Together they form an unbroken audit trail — who made the change, what they changed, and a frozen snapshot to roll back to if the change turns out to be wrong.

Remember: automation stitches turn every FortiGate into its own change-management tool. Combined with FortiManager revision retrieval, you get versioned config on every device with no additional effort from operators.
🧠
MENTAL NOTE
A hybrid mesh isn't just topology — it's a self-observing, self-recording fabric. Trigger + Action + Stitch is the sentence that turns "we should back up before changes" into "the fabric already did."
