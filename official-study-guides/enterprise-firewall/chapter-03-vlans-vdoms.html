Chapter 03 · NSE7 EF 7.6

Slicing the Wire, Partitioning the Firewall

VLANs carve one link into many broadcast domains; VDOMs carve one FortiGate into many virtual firewalls — and together they are how a single box serves many tenants.

Segmentation is a language spoken at every layer of the stack.

VLANs speak it at layer 2, VDOMs speak it inside FortiGate itself, and switches — hardware, software, and virtual — translate between them.

💭 THINK FIRST · VLAN BASICS

Two hosts on the same physical wire but in different VLANs cannot talk to each other without a router. Two hosts on different physical wires but the same VLAN can. The wire is not the boundary — the tag is.

  • Why does FortiGate rewrite the VLAN tag when it routes a frame from VLAN 5 to VLAN 15, instead of just forwarding it?
  • What is a dot1q trunk, and why does it matter when a switch connects to a FortiGate?
Q1 — MODEL ANSWER
VLANs are separate broadcast domains. A frame arriving at FortiGate with a VLAN 5 tag belongs to that broadcast domain; it cannot appear in VLAN 15's domain without the tag being rewritten. FortiGate, as a layer 3 device, strips the VLAN 5 tag, performs its routing decision, and adds the VLAN 15 tag on egress so the destination switch delivers the frame into the correct broadcast domain.
Q2 — MODEL ANSWER
A dot1q trunk is a single link carrying multiple VLANs, each frame tagged with an 802.1Q header identifying its VLAN. Trunks matter because a FortiGate typically has fewer physical ports than an enterprise has VLANs — the trunk lets one physical port terminate many VLAN sub-interfaces, one per VLAN ID.
SECTION 01

VLANs Are Tagged Broadcast Domains

A VLAN is a logical subdivision of a network that segregates devices into separate broadcast domains. Devices in the same broadcast domain can communicate directly; devices in different VLANs require inter-VLAN routing or policies to talk. When more than one VLAN shares a single interface, that link is called a dot1q trunk.

The canonical inter-VLAN flow: a host on VLAN 5 sends a frame toward a host on VLAN 15. Switch A receives the frame on its untagged VLAN 5 access port and adds the VLAN 5 tag before sending it up the trunk to FortiGate. FortiGate receives the tagged frame, routes it from VLAN 5 to VLAN 15 — rewriting the VLAN ID in the process — and sends it back down the trunk. Switch A receives the tagged VLAN 15 frame, removes the tag, and delivers it out the untagged VLAN 15 access port to the destination host.

The tag rewrite is the crucial mental model: layer 2 devices (switches) add or remove VLAN tags but do not modify them. Layer 3 devices (routers, FortiGate) can modify the tag as part of a routing decision. FortiGate itself does not add tags on ingress — that is the previous device's job — but it does add tags on egress after routing. In the FortiGate GUI, VLAN interfaces support IEEE 802.1Q (the standard VLAN tagging protocol) or IEEE 802.1ad (a provider-bridging variant).

Tip: On the FortiGate GUI, add a VLAN ID column to Network > Interfaces to see every configured VLAN at a glance. It saves you from clicking through each sub-interface.
🧠
MENTAL NOTE
Switches translate tag-vs-no-tag; routers translate tag-vs-different-tag. That single line captures 80% of inter-VLAN routing questions.

💭 THINK FIRST · FRAMES & VXLAN

802.1Q gets you 4,094 VLANs. That was plenty in 1998 and nowhere near enough for a modern hyperscale data centre. The industry's answer is either to stack tags (802.1ad, QinQ) or to encapsulate the whole frame in UDP (VXLAN).

  • Why do 802.1ad and QinQ each support 16 million VLANs when they still use 12-bit tags?
  • What problem does VXLAN solve that 802.1Q cannot, and why does it use UDP port 4789?
Q1 — MODEL ANSWER
Both stack two 12-bit VLAN tags. 802.1ad uses an outer S-tag (EtherType 0x88A8) for the provider VLAN and an inner C-tag (0x8100) for the customer VLAN — 4,094 × 4,094 combinations gives roughly 16 million. QinQ works similarly with two 0x8100 tags stacked. The scalability comes from combinations, not from wider tags.
Q2 — MODEL ANSWER
VXLAN scales past the 4,094-VLAN ceiling to 16 million virtual networks using a 24-bit VXLAN Network Identifier (VNI), which matters most in data centre and cloud environments where thousands of tenants need isolation. It encapsulates the original Ethernet frame inside UDP so it can tunnel over any IP network; UDP port 4789 is the IANA-assigned default for VXLAN encapsulation.
SECTION 02

Ethernet Frames, Stacked Tags, and VXLAN

The layer 2 domain has several Ethernet frame types identified by EtherType. A standard IPv4 frame uses 0x0800 and has no VLAN tag. Tagged variants add fields between the source MAC and the EtherType:

  • 802.1Q: Single VLAN tag, EtherType 0x8100, 4 bytes. Supports up to 4,094 VLANs.
  • 802.1ad (provider bridging / double tagging): Outer S-tag 0x88A8 (provider VLAN) plus inner C-tag 0x8100 (customer VLAN). Combines 4,094 × 4,094 for roughly 16 million VLANs, commonly used by service providers to segregate customer traffic.
  • 802.1Q-in-802.1Q (QinQ): Stacks two 0x8100 tags — an outer and an inner. Also yields 16 million VLANs. Common in service-provider and large-enterprise networks that need extra scalability and segregation.

For data centres and cloud environments, tag stacking is not enough. VXLAN (Virtual Extensible LAN) is a network virtualization protocol that encapsulates the original Ethernet frame inside a UDP packet, tunneling it over an IP network. VXLAN uses UDP port 4789 by default. The encapsulated frame carries outer headers (Ethernet, IP, UDP, VXLAN) plus the original inner Ethernet frame and payload — a total of about 1,550 bytes for a 1,500-byte original.

VXLAN's 24-bit VNI (VXLAN Network Identifier) supports up to 16 million virtual networks, dwarfing 802.1Q's 4,094. It is the practical answer to multi-tenant scale in data centres and cloud environments where VLAN limits would otherwise force painful redesigns.

💭 THINK FIRST · SWITCH OPTIONS

FortiGate can carry VLANs on aggregate interfaces, hardware switches, software switches, virtual VLAN switches, FortiLink to FortiSwitch, and wireless SSID interfaces. Each option is a trade-off between throughput, wireless support, STP, and CPU load.

  • What is the essential difference between an LACP aggregate interface and a FortiLink to FortiSwitch?
  • When would you pick a hardware switch over a software switch, and vice versa?
Q1 — MODEL ANSWER
An LACP aggregate (802.3ad) bundles multiple physical ports on FortiGate into one logical interface for bandwidth and redundancy — every member must connect to the same switch. FortiLink extends FortiGate management down to a FortiSwitch, so FortiGate becomes the switch's controller and can manage VLANs, MCLAG, and switch ports from the FortiGate GUI itself.
Q2 — MODEL ANSWER
Pick a hardware switch when you need throughput — packets are offloaded to hardware, STP is supported, and there is no CPU cost. Pick a software switch when you need to include wireless SSID interfaces in the same broadcast domain, since only software switches support SSIDs; the cost is that the CPU handles the packets.
SECTION 03

Where VLANs Can Live on FortiGate

VLANs are not restricted to physical interfaces. FortiGate supports VLAN sub-interfaces on many interface types, and the same VLAN ID on two different interfaces yields two completely different broadcast domains (assuming the interfaces are separate physical/logical entities).

  • Aggregate interfaces (LACP, 802.3ad): Bundle multiple physical ports for bandwidth and redundancy. All member ports must connect to the same switch (or MCLAG pair). Use two, four, or eight members.
  • FortiLink interfaces: A special aggregate that turns FortiGate into the controller for FortiSwitch devices, letting you manage VLANs directly from the FortiGate GUI. Best path: WiFi & Switch Controller > FortiSwitch VLANs.
  • Hardware switches: Layer 2 switches implemented in silicon; share the same broadcast domain across multiple ports with intraswitch traffic allowed. Configured with config system virtual-switch. Supports STP and offloads packets to hardware, but not all models support them and they do not include wireless SSIDs.
  • Software switches: Same broadcast domain shared across multiple interfaces plus wireless SSIDs, processed in software by the CPU. Configured with config system switch-interface. Intraswitch traffic can be blocked with set intra-switch-policy explicit. Do not support STP.
  • Virtual VLAN switches: Similar to hardware switches but virtualized in software; useful when hardware switch support is absent.
  • Wireless SSID interfaces: A VLAN ID can be tied to a wireless SSID so the same broadcast domain spans wired and wireless clients.

MCLAG (Multichassis Link Aggregation) further enhances redundancy: two FortiSwitch devices act as one unit, so a FortiGate aggregate can survive the loss of an entire switch. Both scenarios typically use FortiLink with 802.3ad to connect the firewall and switches, and VLANs live under the FortiLink interface once created via WiFi & Switch Controller > FortiSwitch VLANs.

💭 THINK FIRST · VDOMS OVERVIEW

A single FortiGate can be sliced into multiple virtual firewalls called VDOMs, each with its own security configuration, its own network segmentation, its own admin scope. The pattern turns one appliance into a shared platform — carefully.

  • Why does enabling VDOMs create both a Global VDOM and a root VDOM, and why can neither be deleted?
  • How do global resources differ from VDOM resources, and why does the Override Maximum have to be at least Current Usage?
Q1 — MODEL ANSWER
The Global VDOM holds device-wide settings — interfaces, firmware, DNS, some logging, sandboxing — that must exist regardless of which tenant is deployed. The root VDOM holds the default traffic-forwarding and management state that the FortiGate needs to remain operational and manageable. Neither can be deleted because the FortiGate would have nowhere to store its base configuration or its always-on management path.
Q2 — MODEL ANSWER
Global resources apply to the entire FortiGate device (all VDOMs together); VDOM resources are allocated per VDOM. Setting an Override Maximum below Current Usage would immediately violate the running configuration, so FortiGate enforces Override Maximum ≥ Current Usage. Global caps are edited under System > Global Resources; VDOM caps under System > VDOM > Edit VDOM.
SECTION 04

VDOMs Split One FortiGate into Many

A VDOM (Virtual Domain) partitions a single FortiGate into multiple virtualized security instances, each with its own security configuration and network segmentation. Enable the feature via GUI or the CLI:

Enable multi-VDOM
config system global
    set vdom-mode multi-vdom
end
get sys status | grep "Max number of virtual domains"

Enabling multi-VDOM automatically creates the Global VDOM and the root VDOM. Neither of these VDOMs can be deleted. Most FortiGate models support up to 10 VDOMs; higher counts require additional licensing. Use get system status and look for Max number of virtual domains to confirm the limit on your hardware.

You switch between VDOMs using the icon at the top right of the GUI. Only Super Admin users have access to the Global VDOM. Deleting a VDOM is not a single click — you must first delete its policies, remove its firewall objects that reference its interfaces, and reassign any interfaces linked to that VDOM back to the root VDOM.

Global VDOM settings affect the entire FortiGate: interfaces, firmware, DNS, some logging, sandboxing options. Because of its scope, only top-level administrators should modify it. Three modules are unavailable in the Global VDOM by design: Policy & Objects, VPN, and User & Authentication — those are inherently per-VDOM.

Resources come in two flavours. Global resources — such as total active sessions across the whole device — are managed at VDOM Global > System > Global Resources. VDOM resources — such as firewall policies and firewall addresses allocated to a specific VDOM — are managed at VDOM Global > System > VDOM > Edit VDOM. The Override Maximum value must always be higher than or equal to Current Usage.

Global objects (prefix g-) let you author UTM profiles once and apply them consistently across every VDOM: g-default antivirus, g-default web filter, g-default application control, and so on. Modifying a global profile updates it across every VDOM at once. The reserved g- prefix should not be reused for custom objects. When importing configurations from FortiGate into FortiManager, prefer FortiManager's global objects over FortiGate's for better synchronization.

🧠
MENTAL NOTE
Global VDOM = the landlord. Root and other VDOMs = the tenants. Global objects = the house rules that apply to every apartment.

💭 THINK FIRST · VDOM TYPES

Not every VDOM is meant to pass traffic. Admin VDOMs handle management only. Traffic VDOMs run in NAT or transparent mode. LAN extension VDOMs stretch a LAN via IPsec and VXLAN. Choosing the wrong type shapes the wrong architecture.

  • What is the difference between an Admin VDOM and a Traffic VDOM in transparent mode?
  • Why does an inter-VDOM link need static or dynamic routing plus firewall policies to actually pass traffic?
Q1 — MODEL ANSWER
An Admin VDOM handles no user traffic — it exists only for FortiGate administration, dashboarding, log/report review, and downloading FortiGuard updates. It contains only the Dashboard, Network, System, Security Fabric, and Log & Report modules. A Traffic VDOM in transparent mode passes user traffic invisibly at layer 2, no NAT or routing — it uses UTM profiles to inspect traffic while sharing a broadcast domain, and requires a management IP for admin access.
Q2 — MODEL ANSWER
An inter-VDOM link is a virtual point-to-point interface — two virtual interfaces named VDOM_Link0 and VDOM_Link1, one in each VDOM. Just like any router-to-router link, packets need routes to know where to go, and firewall policies to be permitted between VDOMs. Without both, the link is present but useless.
SECTION 05

Admin, Traffic (NAT/Transparent), and LAN Extension

Four VDOM types cover FortiGate's virtualization needs:

  • Admin VDOM: Dedicated to management of FortiGate; handles no user traffic. Contains only Dashboard, Network, System, Security Fabric, and Log & Report modules. Allows FortiGate to reach the internet to download FortiGuard updates, which affect all VDOMs.
  • Traffic VDOM (NAT mode): The most common type. Passes user traffic like a regular firewall, performs NAT, and can serve as the Management VDOM — the VDOM used to download FortiGuard updates and validate the firewall license. By default the root VDOM is the management VDOM; you can change it via Global VDOM > System > VDOM > Switch Management.
  • Traffic VDOM (Transparent mode): Operates invisibly at the IP layer, no interface IPs. Enables UTM profile inspection without changing layer 3 settings like gateways. All interfaces share the same broadcast domain; use the forward-domain command to subdivide a VDOM into multiple broadcast domains using VLAN IDs. A management IP address is mandatory — miss it and you get a failed node check object for 'manageip' error.
  • LAN Extension VDOM: Extends a LAN using VPN IPsec and VXLAN, connecting a FortiGate connector (FortiGate or FortiExtender) to a FortiGate access controller. Contains only Network and VPN modules.
VDOM types via CLI
config vdom
    edit "name of the vdom"
    config system settings
        set vdom-type admin           # Admin VDOM
        # -- or --
        set vdom-type traffic
        set opmode nat                # Traffic NAT
        # -- or --
        set vdom-type traffic
        set opmode transparent        # Traffic transparent
        set manageip 172.16.16.1/24
        # -- or --
        set vdom-type lan-extension   # LAN extension
    end

When VDOMs need to talk to each other — for example, when a tenant VDOM lacks internet and needs to reach the root VDOM to get out — you use an inter-VDOM link. Create the link at VDOM Global > Network > Interfaces > Create New > VDOM Link. FortiGate automatically appends 0 and 1 to interface names: an inter-VDOM link named VDOM_Link yields VDOM_Link0 in one VDOM and VDOM_Link1 in the other. To make the link useful, configure static or dynamic routing plus firewall policies for both directions.
