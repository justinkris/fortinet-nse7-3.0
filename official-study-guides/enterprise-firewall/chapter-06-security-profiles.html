Chapter 06 · NSE7 EF 7.6

Inspecting What the Firewall Cannot See By Default

SSL/SSH inspection, web filtering, application control, ISDB, and IPS — the tools that turn allow rules into informed decisions.

A packet passing through a firewall is not the same as a packet the firewall understands.

Security profiles are how FortiGate reads the inside of an allowed session — and why deciding what to decrypt is as important as deciding what to block.

SSL/SSH Inspection TLS Hardening & SNI/SAN Web · App · ISDB IPS Integration UTM Placement
💭 THINK FIRST · SSL/SSH INSPECTION

Most enterprise traffic is encrypted. A firewall that only sees the outside of a TLS session can enforce very little beyond who is talking to whom. FortiGate offers two levels of TLS visibility — and choosing between them is a trade-off between CPU cost and inspection depth.

  • What can SSL certificate inspection actually see, and where does it fall short compared to full SSL inspection?
  • Why would enabling "protocol port mapping" in an SSL/SSH profile matter for a service running on a nonstandard port?
Q1 — MODEL ANSWER
Certificate inspection sees the SSL handshake — SNI, server certificate, URL from the certificate — which is enough to feed web filtering, application control, and server-name validation. What it cannot see is the encrypted payload, so it cannot detect malware in the tunnel, cannot inspect POST bodies, cannot catch payload-based exploits, and cannot proxy the session. Full SSL inspection decrypts, inspects, and re-encrypts — costing significantly more CPU and RAM but giving true content visibility.
Q2 — MODEL ANSWER
Attackers and misconfigured services often run standard protocols on nonstandard ports — SMTPS on 10587 instead of 587, HTTPS on 8443, SSH on 2222. Without protocol port mapping, FortiGate only inspects the well-known port and lets the nonstandard-port traffic bypass inspection entirely. Enabling port mapping tells FortiGate "any traffic to this port should be inspected as this protocol," closing the bypass. It works only with full SSL inspection and proxy-based flows; flow-based already inspects all ports regardless.
SECTION 01

SSL/SSH inspection — the difference between seeing metadata and seeing content

Modern application traffic is overwhelmingly encrypted. HTTP became HTTPS, SMTP became SMTPS, FTP became FTPS — the layer that changed in each case was TLS. Without any inspection, FortiGate can enforce policies based on IP, port, and time, but every payload is opaque. Two inspection modes change that.

SSL certificate inspection lets FortiGate examine the TLS handshake itself: the Server Name Indication, the server certificate, and any URL fields in the certificate. That is enough to categorize the destination for web filtering, feed application control, and validate the server name — all without decrypting the payload. It is comparatively cheap on CPU and RAM.

Full SSL inspection takes it further: FortiGate acts as a middleman, terminating the client's TLS session, opening its own session to the server, decrypting and inspecting the payload, then re-encrypting. That enables IPS, antivirus, DLP, and every content-based feature to see inside the tunnel. The cost is significantly more CPU and RAM, plus the operational overhead of distributing FortiGate's CA certificate to clients so they trust the substitute cert.

Protocol port mapping is the setting to know when standard protocols run on nonstandard ports. In the SSL/SSH inspection profile you can map SMTPS to port 10587, HTTPS to 8443, and so on — telling FortiGate to inspect that traffic as its declared protocol regardless of the port. It only takes effect under Multiple Clients Connecting to Multiple Servers with full SSL inspection and proxy-based inspection; flow-based already inspects all ports.

FortiOS CLI — SSL/SSH inspection profile with TLS floor
config firewall ssl-ssh-profile
    edit "ssl_ssh_profile"
        config https
            set unsupported-ssl-version block
            set min-allowed-ssl-version tls-1.2
        end
    next
end
Trade-off: Full inspection is not free. Enable it selectively — on the outbound policies that carry your riskiest categories — rather than blanket-applying to every rule. Watch your CPU graphs after turning it on.
🧠
MENTAL NOTE
Certificate inspection reads the envelope. Full SSL inspection reads the letter. Protocol port mapping keeps a service from hiding on the wrong port.

💭 THINK FIRST · TLS HARDENING & SNI/SAN

TLS versions come with expiry dates. SSL 3.0 died in 2020; TLS 1.0 and 1.1 followed. Modern deployments want to speak TLS 1.2 or 1.3 — but only if you tell the firewall to insist on it. Meanwhile, SNI and SAN quietly determine which certificate a server presents in the first place.

  • Why do min-allowed-ssl-version and unsupported-ssl-version both matter, and what specifically breaks if only one is set?
  • What real problem does SNI solve, and how is SAN a related-but-different tool for multi-domain certificates?
Q1 — MODEL ANSWER
min-allowed-ssl-version defines the floor — everything below it is rejected. unsupported-ssl-version tells FortiGate what to do when the server offers something below the floor: block, allow, or inspect. If you set only the min-version, an old server that advertises TLS 1.0 might still complete the handshake because the "unsupported" default action was not aligned. Both together give you deterministic behavior: TLS 1.0 and 1.1 are blocked; TLS 1.2 and 1.3 pass through.
Q2 — MODEL ANSWER
SNI was introduced in 2003 as a TLS extension so that one IP address could host many HTTPS sites with different certificates. Without SNI, a server behind one IP could only present one cert — SNI lets the client say "I want example.com" during the ClientHello, and the server picks the matching cert. SAN, from 1999, is a certificate extension listing every alternate name the certificate is valid for; one cert with SAN entries for fortinet.com, www.fortinet.com, and pub.kb.fortinet.com covers all three domains without SNI-based selection. SNI negotiates which cert to present; SAN determines what a single cert covers.
SECTION 02

TLS hardening plus SNI and SAN in practice

Deprecated TLS versions are still out there. SSL 2.0, 3.0, TLS 1.0, and 1.1 are all deprecated — TLS 1.2 (2008) is the modern baseline and TLS 1.3 (2018) is the default in HTTP/3. You should be actively enforcing this on outbound policies. In the SSL/SSH inspection profile, set min-allowed-ssl-version tls-1.2 and unsupported-ssl-version block. The first defines your floor; the second tells FortiGate to actually enforce the floor rather than let the connection proceed. The setting also applies to SMTPS, POP3S, IMAPS, and FTPS.

SNI (Server Name Indication) is a TLS extension that lets a client indicate which hostname it wants during the ClientHello. Before SNI, one IP could only host one cert; SNI is what makes shared HTTPS hosting practical. The stages:

  1. Client sends ClientHello with SNI = example.com.
  2. Server picks the cert for example.com and returns ServerHello + cert + hello done.
  3. The TLS handshake completes; the encrypted session flows on top.

SAN (Subject Alternative Name) solves a related problem from the certificate side: one cert can be valid for multiple names by listing them all in the SAN field. A single cert with SANs for fortinet.com, www.fortinet.com, and pub.kb.fortinet.com covers three sites. SAN supports wildcards, so *.fortinet.com covers every subdomain without enumerating them. SAN reduces the number of certificates you have to manage; SNI decides which cert to hand out when many exist.

Finally, HTTP/3 changes the transport: it runs over QUIC (UDP) with TLS 1.3 baked in. If your inspection strategy assumes TCP-only, HTTP/3 traffic will slip past. Modern SSL/SSH inspection profiles cover this, but it is worth verifying that flow-based inspection is enabled where UDP-based application traffic matters.

Cipher suite anatomy: A suite like TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256 encodes protocol (TLS), key exchange (ECDHE), authentication (RSA), encryption (AES-128-GCM), and hash (SHA-256). If client and server share no common suite, the handshake fails — that is why matching TLS profiles between the two devices matters.
🧠
MENTAL NOTE
Floor plus fallback: min-version says how low you allow, unsupported-version says what to do when the server offers less. SNI picks the cert; SAN says what one cert is worth.

💭 THINK FIRST · WEB · APP · ISDB

Three features that get grouped together but do very different things. Web filter cares about URLs. Application control cares about how the traffic behaves at Layer 7. ISDB cares about which vendor owns the destination IP. Picking the right one for the job is what separates a working policy from a fragile one.

  • Where do each of these features sit in the OSI model, and how does that determine what they can and cannot decide?
  • Why is ISDB not a security profile, and what real-world jobs is it good at?
Q1 — MODEL ANSWER
Web filter inspects domains and URLs — it needs at least the SNI (certificate inspection) or the plaintext HTTP host (full inspection). Application control runs at Layer 7 and identifies applications by their transmission patterns, not their ports. ISDB works at Layers 3 and 4 — it is essentially a large database of IP addresses and TCP/UDP ports maintained by FortiGuard, mapping ranges to known vendors and services. Web filter says "what site is this?" App control says "what app is this?" ISDB says "who owns this IP?"
Q2 — MODEL ANSWER
ISDB is a policy and object tool, not a security profile — it lives in the Addresses section of a firewall policy or SD-WAN rule. It does no inspection at all. What it's good at: allowing "Microsoft 365" as a destination without hand-maintaining a list of IPs; steering all "Salesforce" traffic through a specific SD-WAN member; blocking traffic to a geographical zone. Because it operates on IPs and ports, it is CPU-cheap compared to inspection-based approaches.
SECTION 03

Web filter, application control, and ISDB — three tools, three OSI layers

These three features get treated as siblings but they operate at different OSI layers and produce different kinds of decisions.

  • Web filter is a security profile that manages website access by inspecting domains, URLs, SNI, or SAN. It runs in either flow-based or proxy-based inspection. Its job is "what site is this and should the user get there?" Requires FortiGuard for on-demand category lookups; results are cached.
  • Application control is a security profile that operates purely at Layer 7 in flow-based inspection. It identifies applications by transmission patterns, protocol decoders, and rate-based IPS signatures — regardless of which port the app is using. Its job is "what app is this, and should this app be allowed / rate-shaped / prioritized?"
  • ISDB (Internet Service Database) is a policy-and-object tool, not a security profile. It is a FortiGuard-supplied database of IP-and-port ranges keyed by vendor/service. It performs no inspection at all. Its job is "who owns this IP, and how should I route or allow traffic to it?"

Where each fits: Restrict Netflix by category → web filter. Rate-shape peer-to-peer regardless of port → application control. Send all Microsoft 365 traffic through your dedicated SaaS SD-WAN link → ISDB.

💭 THINK FIRST · IPS INTEGRATION

IPS is the difference between "allow this traffic" and "allow this traffic if it does not look like an attack." But every IPS rule that blocks too eagerly generates a false positive that breaks something a user cares about. The design work is not adding rules — it is calibrating them.

  • When a legitimate application gets blocked by an IPS filter, what is the recommended pattern for un-blocking it without disabling protection for everyone else?
  • Why is a PHP-injection attack over HTTPS invisible to an IPS with only certificate inspection enabled, and what specific SSL profile setting fixes it?
Q1 — MODEL ANSWER
Add a Signature-type entry at the top of the rule set with Action = Monitor for the specific offending signature — because signature rules evaluate top-down and the first match wins. That lets you exempt the false positive precisely without turning off the broader Filter that catches real attacks. Enable packet logging on the exemption temporarily so you have evidence of what's being allowed while you investigate.
Q2 — MODEL ANSWER
Certificate inspection sees the handshake but not the payload — a PHP injection embedded in an HTTPS URL is inside the encrypted body, so IPS never sees the malicious pattern. To detect it, the SSL/SSH profile must use full SSL inspection with Protecting SSL Server enabled, plus a certificate the client will trust (internal CA, FortiGate self-signed, or a public CA-signed cert). Once decryption is on, the existing IPS profile with the PHP.URI.Code.Injection signature drops the packet.
SECTION 04

IPS — catching attacks without breaking legitimate traffic

FortiGate's Intrusion Prevention System (IPS) profile is built from Filters (broad, criteria-based) and Signatures (specific). Actions include Block, Monitor, Reset, or Default. Fortinet's guidance is to know your network before you turn on aggressive Filters — the wider the Filter (all severities, all protocols, all OSs, both client and server targets), the more likely something legitimate looks suspicious.

Use case 1 — protect both server and client targets. Configure a Filter with Type=Filter, Action=Block, Filter Target=Client and Server, all severities, all protocols, all OSs. In one example this results in over 10,000 signatures being enabled. Very protective, very likely to false-positive.

Use case 2 — resolve false positives. Suppose a Linux tool like hping3 generates large ICMP packets that get flagged as ICMP.Oversized.Packet. To exempt that signature without disabling the whole Filter, add a Signature-type rule at the top with Action=Monitor for the specific signature, enable packet logging. Rules evaluate top-down; the first match wins, so the exemption fires before the block.

Use case 3 — monitor before you block. When you first roll out a Filter, set Action=Monitor and enable packet logging so you see what would have been blocked. Review for false positives, adjust, then switch Action=Block. On a production allow-list policy this is the safe pattern.

Use case 4 — inspect HTTPS attacks. A PHP code injection in an HTTPS URL is invisible to IPS with only certificate inspection — the payload is encrypted. To catch it, switch the SSL/SSH profile to full SSL inspection with Protecting SSL Server enabled and provide a certificate the client will trust: an internal CA-signed cert for lab/internal use, a FortiGate self-signed cert for lab work, or a public CA-signed cert for production. Once decryption is active, the existing PHP.URI.Code.Injection signature can see and drop the malicious URL.

Use case 5 — DNS exfiltration. DNS runs over UDP and is normally unencrypted. Attackers encode data into subdomain queries and exfiltrate to an attacker-controlled authoritative name server. The DNSExfiltrator.Data.Exfiltration signature detects this pattern. If the environment uses DNS over TLS, you must enable SSL/SSH inspection (multiple clients, multiple servers, full inspection) to see the queries at all.

Discipline reminder: Packet logging is a diagnostic tool, not a production one. It generates a lot of data — use it while you tune a signature or investigate a false positive, then disable it.
🧠
MENTAL NOTE
Broad Filters catch a lot including things that are not attacks. To exempt safely, add a Signature-type rule to the top with Monitor. To see encrypted attacks, decrypt them.

💭 THINK FIRST · UTM PLACEMENT

Not every FortiGate should carry every UTM profile. NGFW at the perimeter is the obvious answer, but ISFWs, DCFWs, and DEFWs each play different roles in the segmented architecture — and turning on full UTM everywhere costs CPU you may need elsewhere.

  • Why is a DEFW usually a poor candidate for heavy UTM profiles, and what remedy does the study guide suggest?
  • How does an ISFW differ from an NGFW in the enterprise topology, and when might one act as the other?
Q1 — MODEL ANSWER
A DEFW (distributed enterprise firewall) is typically a smaller model whose purpose is to connect a remote site to headquarters over an encrypted tunnel — it is sized for that job, not for full UTM. If you need UTM at the branch, the guide's suggestion is to upgrade to a midrange model (100–900 series) that has the horsepower for content inspection, rather than force the DEFW to do work it was not built for.
Q2 — MODEL ANSWER
An ISFW (internal segmentation firewall) sits between internal zones — primarily for segmentation, low latency, and often integrated wireless LAN. It does not always run heavy UTM because latency matters most on internal links. An NGFW sits at a gateway boundary (usually to the internet or between security zones) and is where you concentrate content inspection. When an ISFW carries VPN traffic for sensitive segments or needs traffic filtering to reach the internet or shape SD-WAN, it takes on NGFW-like responsibilities.
SECTION 05

UTM placement — right profile, right firewall

Where you attach UTM profiles depends on what job each firewall does in the topology. Enterprise designs typically distinguish four roles:

  • NGFW (Next-Generation Firewall): the perimeter or inter-zone gateway. It is where content filters, IPS, and application control are most useful — the internet edge sees the most diverse and highest-risk traffic.
  • ISFW (Internal Segmentation Firewall): segments internal zones from each other. Latency is often more important than deep inspection here; ISFWs may also carry integrated wireless LAN. An ISFW can play the role of an NGFW when it terminates VPNs, reaches the internet, or drives SD-WAN policy.
  • DCFW (Data Center Firewall): protects data-center workloads. Usually sits behind another firewall that acts as a gateway. Often benefits from segmentation with VDOMs and may share content-filtering duties with the upstream NGFW.
  • DEFW (Distributed Enterprise Firewall): the branch-office device, purpose-built to connect a remote site back to HQ. Sized for tunnel termination, not for full UTM. If a branch needs meaningful content inspection, the recommendation is to move to a midrange model (100–900 series) rather than overwhelm the DEFW.

The recurring lesson: UTM is not free. Every profile you attach to a policy adds inspection cost. Concentrate them where they buy the most protection per CPU cycle, and where the traffic actually justifies deep inspection.

Placement heuristic: Perimeter NGFW carries the heavy load. ISFWs prioritize low latency. DCFWs share duties with the NGFW upstream. DEFWs stay lean — if they need UTM, get a bigger box.
🧠
MENTAL NOTE
Every UTM profile is CPU you spend. Concentrate them where the risk is highest and the box is sized for the work — not blanket-applied to every firewall in the topology.
