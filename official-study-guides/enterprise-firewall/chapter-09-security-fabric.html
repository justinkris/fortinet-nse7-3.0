Many products, one coordinated response

💭 THINK FIRST · FABRIC VS. EXTERNAL CONNECTORS

The Security Fabric distinguishes two categories of connector — fabric and external — and the distinction is not cosmetic. One glues Fortinet products together with shared identity and configuration; the other imports data or telemetry from the outside world. Confuse the two and you either miss integrations you already own or spend effort on integrations that don't exist.

  • Where in the GUI do fabric connectors and external connectors live, and what's the operational difference?
  • Why is an SDN connector for AWS classed as external rather than fabric?
Q1 — MODEL ANSWER
Fabric connectors are configured under Security Fabric > Fabric Connectors and target Fortinet products (FortiGate, FortiAnalyzer, FortiClient EMS, plus additional Fortinet devices like FortiNAC and FortiMail). External connectors live under Security Fabric > External Connectors and target non-Fortinet ecosystems (SDN, endpoint/identity, threat feeds).
Q2 — MODEL ANSWER
AWS is not a Fortinet product. The SDN connector reads attribute changes — like an address object's IP range — from a public cloud environment using AWS APIs. That makes it a bridge to a multi-vendor ecosystem, which is the external connector's role.
SECTION 01

Fabric connectors, external connectors, and where each belongs

The Fortinet Security Fabric is a cybersecurity mesh built on three ideas: broad — detect threats and enforce security everywhere; integrated — close security gaps and reduce complexity; automated — enable faster time-to-prevention and efficient operations. It is powered by FortiOS and it is deliberately open, so the same Fabric can talk to third-party devices through connectors.

FortiOS splits those connectors into two families, each with a different job. Fabric connectors — Security Fabric > Fabric Connectors on the root FortiGate — link Fortinet products together. Core network security connectors cover FortiGate, FortiAnalyzer, and FortiClient EMS. Security fabric connectors extend to central management, sandbox, and additional supported Fortinet devices such as FortiDeceptor, FortiMail, FortiMonitor, FortiNAC, FortiNDR, FortiPolicy, FortiTester, FortiVoice, and FortiWeb. EMS connectors, for example, enable automatic actions like endpoint quarantine.

External connectors — Security Fabric > External Connectors — reach out to the rest of the world. Public SDN (AWS, GCP) and private SDN (Kubernetes, VMware, SAP) connectors update FortiGate about attribute changes so address objects follow the cloud environment. Endpoint/identity connectors (FSSO, RADIUS) push credentials once. Threat feed connectors import FortiGuard categories, IP addresses, domain names, MAC addresses, or malware hashes as dynamic lists that firewall policies can reference by name.

External feeds come in two update flavours: external feed pulls from a URL over HTTP, HTTPS, or STIX; push API receives webhook updates directly into the FortiGate REST API. Both create a dynamic resource object you can use as source or destination in a policy — no policy edit required when the list itself changes.

Design cue: if you're integrating a Fortinet product, look under Fabric Connectors first. If you're integrating a cloud, an IdP, or a threat feed, external connectors is the place.
🧠
MENTAL NOTE
Fabric connectors bind Fortinet products; external connectors bring in everyone else — and both feed automation stitches downstream.

💭 THINK FIRST · CORRELATED LOGGING

Chain three FortiGate devices in a Security Fabric and every packet potentially traverses all of them. If each device logs the flow, FortiAnalyzer receives three copies of the same session — a nightmare for reporting and a waste of storage. The Fabric solves that with an unusual rule: the first FortiGate that handles the session is the only one that logs the traffic.

  • When does an upstream FortiGate break the "only the first one logs it" rule?
  • What happens to logging if the root FortiGate goes offline?
Q1 — MODEL ANSWER
The upstream FortiGate generates another traffic log if it performs NAT — because the translated port and address need to be captured. Additionally, upstream FortiGates still generate UTM logs (web filtering, IPS) whenever their configured profiles apply, regardless of whether the traffic log itself was suppressed.
Q2 — MODEL ANSWER
Each FortiGate in the Fabric logs directly to FortiAnalyzer, independent of the root. If the root goes down, leaf devices keep sending logs to FortiAnalyzer without interruption — the root is a coordination point, not a logging proxy.
SECTION 02

Correlated logging on FortiAnalyzer

The FortiAnalyzer Fabric connector consolidates traffic logs across every FortiGate in the Security Fabric. Its cornerstone rule is that a session is logged only by the first FortiGate that handles it. Every downstream (upstream in Fabric terms) member of the Fabric knows the MAC addresses of its Fabric peers. When it receives a packet whose source MAC belongs to another Fabric member, it does not duplicate the traffic log — that log already exists.

There are two important exceptions. First, if an upstream FortiGate performs NAT, it emits a second traffic log so the translated addresses and ports are captured. Second, upstream FortiGates still produce UTM logs — web filtering, IPS, DLP — whenever their profiles inspect the traffic. Traffic logs are deduplicated; UTM logs are additive because each UTM engine's verdict is independent.

FortiAnalyzer correlates all these logs automatically. For a three-device Fabric with an ISFW (internal segmentation firewall) at the access layer, an NGFW at the edge, and a DCFW protecting the data centre, a single client's web session generates: one traffic log from the ISFW (first to see the flow), one NAT log from the NGFW, and any web-filter or IPS logs from the NGFW/DCFW that apply. FortiAnalyzer stitches them into one coherent view of the client's activity — no configuration required.

Importantly, each FortiGate logs to FortiAnalyzer independently of the root or the other leaves. If the root FortiGate fails, log flow from every remaining device to FortiAnalyzer continues without interruption. The root is a coordination point for policy and topology, not a logging proxy.

  • Traffic logs: emitted once, by the first Fabric FortiGate that sees the session.
  • NAT logs: additional log written by any Fabric FortiGate that performs NAT.
  • UTM logs: emitted independently by every Fabric FortiGate whose security profiles inspect the session.
🧠
MENTAL NOTE
One traffic log per session, extra logs only for NAT or UTM — the Fabric fights log noise so FortiAnalyzer can correlate real events.

💭 THINK FIRST · FORTINDR, FORTINAC, SAML SSO

Adding a Fortinet product to the Fabric is not just a matter of pointing it at the root. Each integration exchanges a specific artefact: FortiNDR sends malware verdicts about held files; FortiNAC pushes dynamic firewall addresses tied to authenticated users; SAML SSO lets admins hop between Fabric UIs without re-authenticating. Understanding what each exchanges tells you what breaks if the link is misconfigured.

  • How does FortiNDR block malware that FortiGate's antivirus signatures miss?
  • Why does FortiNAC need two user login events before its dynamic firewall address is usable?
Q1 — MODEL ANSWER
In inline blocking mode, FortiGate first runs the file against its antivirus database. On a miss, it holds the user session, forwards the file to FortiNDR, and waits for a verdict. FortiNDR applies machine learning and artificial neural network analysis and returns clean or malicious within seconds. Malicious files are dropped; clean ones are released to the user.
Q2 — MODEL ANSWER
The first login teaches FortiNAC the user–device–IP mapping; the second login triggers the REST API push of a dynamic firewall address to FortiGate. The FortiGate then treats that address as an SSO source in policies. The second event is what closes the loop and lets the user's IP appear in the FortiNAC Tag (IP Address) section of Policy & Objects > Addresses.
SECTION 03

FortiNDR, FortiNAC, and SAML SSO in the Fabric

FortiNDR extends antivirus with machine-learning and artificial-neural-network analysis of files that don't match a signature. When you enable Security Fabric on FortiNDR, it submits a connection request that appears as an authorisation notification on the root FortiGate. Once accepted, FortiNDR joins the Fabric and appears in both the Logical Topology and Physical Topology views. FortiGates in the Fabric can then submit files to FortiNDR for verdict.

In inline blocking mode, when a file matches an antivirus signature FortiGate blocks it outright and FortiNDR is not involved. If it does not match, FortiGate temporarily holds the user session, sends the file to FortiNDR, and awaits a verdict of clean or malicious. FortiGate then either releases the file to the user or terminates the session. On the root device, the Detection Statistics (24 Hours) and Network Anomalies (24 Hours) widgets show FortiNDR activity.

FortiNAC collects information from switches, autonomous APs, wireless controllers, and other infrastructure to know which devices are connecting. When a user logs in to FortiNAC, the REST API pushes the device IP, FortiNAC firewall tags, and FortiNAC group names to the root FortiGate. FortiGate creates a dynamic firewall address object that follows the user, enabling SSO-style access control. Two user login events are required before this dynamic address becomes usable — it appears in Policy & Objects > Addresses under the FortiNAC Tag (IP Address) section, ready to be used as source or destination in a policy.

Finally, SAML SSO lets an administrator authenticate once to the root FortiGate (which acts as identity provider) and then jump to downstream FortiGates, FortiAnalyzer, or FortiManager (each acting as service provider) without logging in again — as long as it's inside the same browser session. Only the root FortiGate can be the IdP; every other Fabric device is registered as an SP.

Design tip: reserve SAML SSO for administrator workflows across the Fabric. It's not designed to replace SSL VPN or FSSO for end users — those still go through their own connectors.
🧠
MENTAL NOTE
FortiNDR catches what signatures miss, FortiNAC turns identity into dynamic firewall addresses, and SAML SSO lets one admin login unlock the whole Fabric.

💭 THINK FIRST · AUTOMATION STITCHES

Manual response is slow and forgetful. Automation stitches let you say "if X happens, do Y" once and let FortiOS remember. But a stitch has three parts — trigger, action, and the stitch itself — and getting the sequencing right is what turns a stitch from a nice idea into a working defence.

  • What is the difference between a stitch trigger, an action, and the stitch itself?
  • When would you choose sequential action execution over parallel?
Q1 — MODEL ANSWER
A trigger defines the event to watch — high CPU, compromised host, schedule, HA failover, security-rating summary, etc. An action defines what to do — run a CLI script, send an email, quarantine an endpoint. A stitch pairs one trigger with one or more actions, sets execution mode (sequential or parallel), and targets specific FortiGate devices.
Q2 — MODEL ANSWER
Sequential is right when a later action needs the output of an earlier one — for example, run a diagnostic CLI script, wait 30 seconds for it to finish, then email the output via %%results%%. Parallel is fine when actions are independent — send an email and quarantine an endpoint at the same time.
SECTION 04

Automation stitches — if this, then that

An automation stitch is an if/then workflow that runs inside FortiOS. Stitches are configured on the root FortiGate and pushed to the relevant leaf devices. They do not require the Security Fabric to work — a standalone FortiGate can use stitches — but Fabric membership lets you fire them from Fabric-wide events (compromised host, HA failover, security-rating summary) and target them at any subset of Fabric FortiGates. Once configured, stitches keep running on the leaves even if the root FortiGate is down.

Every stitch has three ingredients. A trigger defines the event to watch: preconfigured options include Compromised Host, FortiOS Event Log, HA Failover, Incoming Webhook, License Expiry, Reboot, Security Rating Summary, High CPU, and Schedule. An action defines the response: preconfigured options include CLI Script, Email, and vendor-specific integrations like FortiClient Quarantine. A stitch ties one trigger to one or more actions and picks how they run.

Use case 1 — high CPU with CLI scripts. The trigger fires when CPU usage crosses a threshold (default 90%, set via set cpu-use-threshold 90). Two custom actions are wired in sequence: Collect_Diagnostics_Action runs a CLI script gathering get system status, get system performance status, and diagnose sys top output; a 30-second delay lets those commands complete; then Email_Diagnostics_Action emails the output as the message body using the %%results%% placeholder. Sequential execution is essential here — the email needs the earlier script's output.

Use case 2 — scheduled configuration backup. A Schedule trigger fires once, hourly, daily, weekly, or monthly. The action is the preconfigured Automatic Backup. The result appears under Configuration > Revisions as Autod backup config by stitch: Backup_Stitch. You can target all Fabric FortiGates or a specific subset, giving you an audit trail you can roll back from without any manual intervention.

FortiOS CLI · tune the High CPU trigger threshold
config system global
  set cpu-use-threshold 90
end
Sequential vs. parallel: pick sequential when one action's output feeds the next. Pick parallel when the actions are independent — for example, send an email and quarantine an endpoint at the same instant.
🧠
MENTAL NOTE
Trigger + action + stitch is the whole vocabulary — memorising those three words unlocks every Fabric automation workflow.

💭 THINK FIRST · IoC & AUTOMATED QUARANTINE

A compromised host is only useful to know about if you can act on it fast. The full-loop IoC pipeline chains FortiClient, FortiAnalyzer, FortiGate, and FortiClient EMS so that the moment a suspicious behaviour is detected, the endpoint is isolated at the network layer without a human clicking anything. Understanding which device does which step tells you exactly what to check when the loop breaks.

  • Trace the five-step IoC quarantine flow — which device triggers each step?
  • What does the diagnose endpoint fctems queue-complete-calls Q-<ip> command do?
Q1 — MODEL ANSWER
1) FortiClient detects a malicious site and sends logs to FortiAnalyzer. 2) FortiAnalyzer discovers the IoC pattern in the logs and notifies FortiGate. 3) FortiGate confirms that FortiClient is connected and that it has credentials for the associated FortiClient EMS, then sends a notification to EMS. 4) FortiClient EMS locates the endpoint and pushes a quarantine message. 5) The endpoint quarantines itself, blocking network traffic, and reports the status change to both FortiGate and EMS.
Q2 — MODEL ANSWER
It's the CLI trigger for quarantining an endpoint at a specific IP address. FortiGate uses it to push the quarantine action into the FortiClient EMS queue, which then propagates the message to the endpoint. It's not supported on FortiClient (Linux).
SECTION 05

IoC detection and automated quarantine

The IoC quarantine use case combines Fabric connectors and stitches into a full detection-to-response loop. It requires four devices: FortiGate, FortiAnalyzer, FortiClient EMS, and FortiClient itself. FortiClient must be connected to both EMS and FortiGate, and both FortiGate and FortiClient must send logs to FortiAnalyzer. On FortiGate you configure the EMS IP address plus administrator login credentials so it can call the EMS REST API on your behalf.

The flow has five well-defined steps. First, FortiClient detects a malicious site (an IoC) and sends the log to FortiAnalyzer. Second, FortiAnalyzer discovers the IoC pattern in the aggregated logs and notifies FortiGate. Third, FortiGate identifies whether FortiClient is connected and whether the local admin has EMS credentials; with that confirmed, it sends a notification to FortiClient EMS. Fourth, EMS searches for the endpoint and pushes a quarantine message. Fifth, FortiClient quarantines itself — blocking all network traffic — and notifies both FortiGate and EMS of the status change.

Under the hood, the trigger is the preconfigured Compromised Host trigger and the action is FortiClient Quarantine. FortiGate uses the CLI command diagnose endpoint fctems queue-complete-calls Q-<endpoint_ip_address> to inject the quarantine action into the EMS queue. This particular quarantine flow is not supported on FortiClient for Linux — a limitation worth remembering.

The whole thing is a stitch: Compromised Host trigger paired with FortiClient Quarantine action. Once configured on the root FortiGate, it will fire against any endpoint FortiAnalyzer flags as compromised, on any FortiGate in the Fabric that has EMS reachability — no daily babysitting required.

MENTAL NOTE
FortiClient sees it, FortiAnalyzer names it, FortiGate coordinates it, EMS orders it, and FortiClient obeys — five devices, one quiet quarantine.
