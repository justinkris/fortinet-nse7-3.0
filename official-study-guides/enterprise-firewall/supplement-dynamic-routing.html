Routers Talking to Routers — OSPF and BGP in Enterprise Firewalls

A concentrated refresher on the two dynamic routing protocols a FortiGate will speak in almost every enterprise deployment.

Static routes scale to the size of one admin's memory. Dynamic routing scales to the size of the internet.

OSPF and BGP are how enterprise networks stay converged when links flap, sites go dark, and topologies grow overnight.

💭 THINK FIRST · OSPF FUNDAMENTALS

OSPF is a link-state protocol — every router builds a full map of the topology and then walks that map to pick the shortest path. That's very different from distance-vector protocols, which trust their neighbor's summary. Why does OSPF's approach converge faster on large networks, and what does "cost" actually measure?

  • What is the LSDB, and why does every OSPF router in the same area maintain an identical copy of it?
  • How does OSPF choose the best route when there are multiple paths to the same destination?
Q1 — MODEL ANSWER
The Link State Database (LSDB) is a per-area topology map — every link, every router, every advertised subnet with its cost. Every router in the same area holds an identical LSDB because that's the only way each of them can independently run Dijkstra's algorithm and arrive at the same shortest-path tree. Consistent LSDB = consistent routing.
Q2 — MODEL ANSWER
OSPF assigns a cost (1–65,535) to each interface and picks the route with the lowest total cost — the sum of interface costs along the path. If multiple paths tie, OSPF can install equal-cost multipath routes. Cost is typically inversely related to link speed, so faster links get lower cost and are preferred.
SECTION 01

OSPF Fundamentals — link-state routing at a glance

OSPF (Open Shortest Path First) is a link-state interior gateway protocol. Every router builds a complete view of the topology, then uses that view to compute optimal paths. Compared to distance-vector protocols, OSPF converges faster and scales further, at the cost of more complex configuration and heavier planning burden. Advantages: scalable to large networks, fast convergence, and quiet during steady state — periodic LSDB refresh only every 30 minutes, with incremental updates in between when topology changes. Disadvantages: requires planning and tuning; can be difficult to troubleshoot in large deployments.

OSPF's operation reduces to four moving parts:

  • Link State Database (LSDB) — each router maintains an identical database describing the area's topology.
  • Dijkstra's shortest-path first (SPF) algorithm — each router walks the LSDB to build a tree rooted at itself.
  • OSPF routes — the tree yields one route per destination.
  • IP routing table — OSPF injects those routes into the router's forwarding table.

The topology data itself travels in Link State Advertisements (LSAs). Network changes generate LSAs; LSDBs are composed of all received LSAs. A link-state update packet consists of an OSPF header plus a string of LSAs — each with its own header — and receiving routers acknowledge every one.

Cost. The OSPF metric is a 16-bit positive number from 1 to 65,535. Routes with the lowest total cost win. Each interface has a cost value; the cost of a route equals the sum of the outgoing-interface costs along the way to the destination. Typically the cost reflects link speed — a Gigabit link is cheaper than a T1.

Exam tip: when an OSPF router advertises its LSDB entries, it's advertising network/cost pairs. R2 in the study guide advertises Net 1/2, Net 2/3, Net 3/1 — those cost values become the building blocks other routers use to compute their own SPF trees.
🧠
MENTAL NOTE
OSPF is a democracy: every router in an area sees the same LSDB, runs the same algorithm, and elects the same shortest paths — no single router dictates the routing table.

💭 THINK FIRST · AREAS & ADJACENCIES

A flat OSPF domain works for a lab. Enterprise networks slice OSPF into areas because a single LSDB for the whole world would be too big, too noisy, and too fragile. Areas force a hub-and-spoke shape onto OSPF and constrain the blast radius of any topology change.

  • Why must every OSPF area connect to the backbone area 0.0.0.0, and what makes an ABR different from an internal router?
  • An adjacency progresses through Down → Init → 2-way → ExStart → Exchange → Loading → Full. Which state means the LSDBs match, and what breaks if two routers stall at 2-way?
Q1 — MODEL ANSWER
The backbone area (0.0.0.0) distributes information between all other areas — it's the transit fabric. Non-backbone areas connect to the backbone via physical or virtual links. An Area Border Router (ABR) has interfaces in multiple areas and maintains a separate LSDB per area, forwarding summarized routing information between them. An internal router has interfaces in only one area and holds one LSDB.
Q2 — MODEL ANSWER
Full state means both routers hold identical LSDB copies — the adjacency is fully formed. Stalling at 2-way (bidirectional hellos but no LSDB exchange) typically happens on multiaccess networks where the router is not the DR or BDR, or when adjacency requirements are unmet (mismatched hello/dead intervals, MTU mismatch, mismatched area type, failed authentication, subnet mismatch).
SECTION 02

OSPF Areas, Router Roles, and Adjacency Formation — the plumbing that keeps OSPF scalable

An OSPF area is a logical grouping of networks and routers, identified by a 32-bit number written either in dotted-quad (0.0.0.10) or single decimal (10) format. Segmenting the network into areas gives smaller LSDBs per router, contains the impact of topology changes to a single area, and enables route summarization at area borders. The trade-off is more complex troubleshooting and design.

Area types. Backbone area (0.0.0.0) is required — all other areas physically or virtually connect to it. Normal areas behave like the backbone but carry a non-zero area ID. Stub areas and NSSAs are also OSPF areas but aren't covered here.

Router types. An OSPF router's role depends on where its interfaces live:

  • Internal router — all connected interfaces belong to one area; maintains one LSDB and one OSPF tree.
  • Area Border Router (ABR) — interfaces in multiple areas; one LSDB and one SPF tree per area; always connected to the backbone.
  • Backbone router — has at least one interface in the backbone area.
  • Autonomous System Boundary Router (ASBR) — imports non-OSPF routes (RIP, BGP, static, connected) into OSPF.

Network types. OSPF classifies each interface's network as one of three shapes: point-to-point (a pair of routers on a dedicated link), broadcast (multiaccess) — supports two or more attached routers and multicast (Ethernet is the canonical example) — or point-to-multipoint, which supports multiple attached routers but does not support multicast.

Designated Router (DR) election. On multiaccess networks, OSPF elects one DR and one BDR to reduce the mesh of adjacencies each router would otherwise maintain. Highest router priority wins (priority 0 disqualifies); router ID breaks ties. If the DR fails, the BDR takes over. Every other router forms Full adjacencies only with the DR and BDR — dramatically reducing the number of adjacencies on a shared segment.

Adjacency states. A new OSPF neighborship transitions through the following states:

StateMeaning
DownInitial state — no hellos received
InitHello packet seen from a non-adjacent neighbor
2-wayBidirectional communication confirmed between the two routers
ExStartPrimary/secondary relationship being negotiated
ExchangeDatabase description (DD) packets exchanged
LoadingDetailed LSA information being requested and sent
FullLSDBs are identical — adjacency established

Adjacency requirements. Two peers must agree on the following to form an adjacency: primary IP addresses in the same subnet with the same mask, matching interface type and OSPF area, matching hello/dead intervals, unique router IDs, matching OSPF IP MTU, and successful OSPF authentication (if enabled). Any mismatch stalls the adjacency below Full.

OSPF destination addresses. On broadcast networks OSPF uses two multicast destinations — 224.0.0.5 (AllSPFRouters, for hellos and DR/BDR-originated LSA updates) and 224.0.0.6 (AllDRouters, for non-DR/BDR LSA updates). On point-to-point networks everything goes to 224.0.0.5. Some packets — like DD exchanges and LSA retransmissions — travel via unicast.

🧠
MENTAL NOTE
Areas cap the LSDB size; DR/BDR caps the adjacency count. Both are ways OSPF trades a little complexity now for a lot of scale later.

💭 THINK FIRST · LSA TYPES

OSPF defines 11 LSA types, but only 5 of them matter day-to-day. Each carries a different piece of information at a different scope. Understanding who generates which type — and where it can travel — is the key to reading an LSDB dump.

  • Type 1 and Type 2 both describe things inside an area. What's the difference and who generates each?
  • Why does an ABR generate Type 3 LSAs but only an ASBR generates Type 5, and when would an OSPF router prefer a Type 1 external metric over a Type 2?
Q1 — MODEL ANSWER
Type 1 (router LSA) is generated by every OSPF router, describes its local links, and stays confined to the area. Type 2 (network LSA) is generated only by the DR on a multiaccess network and lists every other router on that segment. Same area scope, different scopes of information.
Q2 — MODEL ANSWER
An ABR generates Type 3 (summary LSA) to tell one area about networks it can reach in another. An ASBR is defined by redistributing non-OSPF routes into OSPF — so only it can announce external destinations (Type 5). Type 1 external metric adds internal cost to reach the ASBR plus the external cost, so it's preferred (closer to the AS); Type 2 uses only the external cost, so it's "far away." OSPF prefers Type 1 over Type 2.
SECTION 03

LSA Types — the five you must recognize

Of OSPF's 11 LSA types, five carry almost all real-world traffic. Knowing who generates each — and how far each travels — lets you read an OSPF domain like a map.

TypeNameGenerated byScope
1Router LSAEvery OSPF routerConfined to the area — describes the router's links
2Network LSADesignated Router (DR) onlyConfined to the area — lists all routers on a multiaccess network
3Summary LSAArea Border Router (ABR)Advertises networks in one area to another — the ABR sends both directions
4AS Summary LSAABR (about ASBRs)Announces the path to reach an ASBR to areas that don't hold the ASBR
5AS External LSAASBR onlyAdvertises external routes (redistributed from RIP, BGP, static, connected)

Type 1 — router LSAs. Every router advertises the networks connected to it. They flood inside an area but never cross an ABR.

Type 2 — network LSAs. Only the DR on a multiaccess segment produces them; they enumerate the other routers on that segment. Also confined to the area.

Type 3 — summary LSAs. Generated only by ABRs, and can (as the name suggests) represent a range of networks in a single advertisement. The ABR on the left in the study guide diagram sends Type 3 LSAs to Area 1 describing the summarized subnets in Areas 0 and 2, and vice versa. This is how routing information crosses area boundaries without the full LSDB traveling with it.

Type 4 — AS summary LSAs. An ASBR announces itself with a Type 1 LSA carrying the E-bit set in the OSPF header. Type 1s are area-confined, so ABRs generate a Type 4 to tell other areas how to reach that ASBR.

Type 5 — AS external LSAs. Sent only by ASBRs, they carry external destinations. OSPF considers the following to be external: a directly connected interface not running OSPF, a static route, or a route derived from a different routing protocol. Type 5 LSAs reach every standard area (but not stub areas or NSSAs).

External metric types. Each external route gets a metric type — Type 1 or Type 2. Type 1 is close to the AS: metric = external cost + internal cost to reach the ASBR. Type 2 is far away: metric = external cost only, internal cost ignored. If both flavors advertise the same destination, an OSPF router prefers the Type 1 route.

Remember: Type 1 and Type 2 LSAs are area-scoped. Type 3, 4, and 5 cross area boundaries. Type 5 is the only one that carries redistributed (non-OSPF) destinations.
🧠
MENTAL NOTE
Type 1 = "my links." Type 2 = "who's on this LAN." Type 3 = "here's another area's networks." Type 4 = "here's how to reach an ASBR." Type 5 = "here's the outside world."

💭 THINK FIRST · BGP FUNDAMENTALS

OSPF assumes everyone in the domain agrees on cost and cooperates on routing. BGP assumes the opposite — you're peering with a stranger's autonomous system and want strict control over what you announce and what you accept. That's why BGP is path-based, policy-driven, and lives on the boundaries.

  • What is the difference between EBGP and IBGP, and why does the IBGP split-horizon rule exist?
  • Why would a large IBGP mesh be replaced with route reflectors, and what is a "cluster" in that design?
Q1 — MODEL ANSWER
EBGP advertises routing updates between autonomous systems — it's what the internet runs on. IBGP advertises updates within a single AS. Split horizon says a router will not forward IBGP-learned prefixes to other IBGP neighbors. Without it, an AS with more than two IBGP routers would loop routing information indefinitely.
Q2 — MODEL ANSWER
Full-mesh IBGP scales as O(n²) — 100 IBGP speakers need 4,950 peerings. Route reflectors let a subset of routers forward IBGP-learned prefixes to their clients, breaking the split-horizon rule in a controlled way. A cluster = one RR plus its clients; clients talk only to the RR for routing updates; the RR forwards those updates to other RRs and border routers.
SECTION 04

BGP Fundamentals — autonomous systems, sessions, and paths

An autonomous system (AS) is a set of routers and networks under a single administrative authority, identified by a unique AS number and free to choose whichever interior gateway protocol (OSPF, RIP, IS-IS) it wants inside its borders. BGP is the exterior gateway protocol that stitches ASes together. BGP4 is the dominant EGP today — the internet routing table is a BGP4 table.

BGP runs in two modes. EBGP exchanges routing updates across AS boundaries. IBGP exchanges them within one AS. Both use the same protocol but behave very differently: EBGP is typically strict about policy (route filters, prepends, communities), while IBGP mostly propagates whatever EBGP brought in. Two EBGP routers exchange AS-path information for destination prefixes — the full BGP table is exchanged when the session comes up, then only incremental updates are sent.

BGP components are small in number:

  • Speaker (peer) — a router that transmits, receives, and acts on BGP messages.
  • Session — the TCP connection between two peers over which BGP updates flow.

AS types. Three roles show up in real networks:

  • Stub AS — a single exit point, local traffic only. A company running BGP with its own AS number and one ISP connection.
  • Multihomed AS — multiple exit points, local traffic only. Same company, multiple ISPs, no transit.
  • Transit AS — carries local and transit traffic (traffic originating and terminating in other ASes). ISPs are the canonical transit example.

IBGP split horizon. By default, IBGP does not forward prefixes learned from one IBGP neighbor to any other IBGP neighbor. This rule prevents routing loops within the AS. The consequence is that a fully connected IBGP graph — a full mesh — is normally required so every IBGP router learns every prefix directly.

Route reflectors. In large ASes, full-mesh IBGP is unmanageable. Route reflectors (RRs) simplify the topology by allowing a router to forward IBGP-learned routes to its clients — breaking split horizon in a controlled way. An RR and its clients form a cluster. Clients speak IBGP only to the RR, the RR forwards updates to other RRs and border routers, and the number of IBGP peerings drops dramatically. FortiGate can act as an RR or as an RR client.

🧠
MENTAL NOTE
EBGP stitches autonomous systems together; IBGP redistributes what EBGP learned across your own AS. Route reflectors are how you keep IBGP tractable when the AS has more than a handful of speakers.

💭 THINK FIRST · BGP ON FORTIGATE

Configuring BGP on a FortiGate means dealing with RIBs, attributes, tie-breakers, and prefix policy. The defaults are conservative: no prefixes are originated, but all received prefixes are accepted. The two default behaviors have to be reversed for real deployments — announce what you own, filter what you accept.

  • What are RIB-in, Local RIB, and RIB-out, and why does BGP need all three?
  • When two BGP routes to the same prefix have identical attributes down through the AS-path length, which tie-breakers does FortiGate apply next?
Q1 — MODEL ANSWER
RIB-in holds routing information received from peers before any filtering; Local RIB holds what remains after applying inbound policy — this is what BGP considers for selection; RIB-out holds the routes selected to advertise, after outbound policy is applied. Three RIBs let BGP separate "what I heard," "what I use," and "what I tell others" — each stage has its own policy hook.
Q2 — MODEL ANSWER
After highest weight, highest local preference, locally originated, and shortest AS-path have all tied, FortiGate compares lowest origin type, lowest MED, lowest IGP metric to the BGP next hop, prefers EBGP over IBGP, uses ECMP if enabled (up to 10 routes), and finally lowest router ID.
SECTION 05

BGP on FortiGate — RIBs, attributes, selection, and policy

Routing information bases. A BGP router keeps three logical tables. RIB-in is everything the router hears on inbound updates from its peers, before filtering. Local RIB is what survives after the router applies inbound policy — this table drives BGP path selection. RIB-out is what the router chooses to advertise, after applying outbound policy. The end-to-end flow is: BGP RX → RIB-in → In filter → Local RIB → (redistribution in/out) → Out filter → RIB-out → BGP TX.

Attributes. BGP processes routes based on path attributes. Each path — a route to a destination — carries a set of attributes including an AS list, and BGP uses those attributes to select the best route. Attributes fall into four categories:

AttributeCategory
ORIGINWell-known mandatory
AS_PATHWell-known mandatory
NEXT_HOPWell-known mandatory
MULTI_EXIT_DISC (MED)Optional non-transitive
LOCAL_PREFWell-known discretionary
ATOMIC_AGGREGATEWell-known discretionary
AGGREGATOROptional transitive
COMMUNITYOptional transitive

Well-known mandatory attributes must be present in every update. Well-known discretionary attributes may or may not appear. Optional transitive attributes may or may not be accepted but can be passed to other ASes. Optional non-transitive attributes may or may not be accepted and cannot cross AS boundaries.

Route selection tie-breakers. When multiple BGP routes reach the same destination, FortiGate applies these tests in order:

  1. Highest weight
  2. Highest local preference
  3. Prefer the path originated locally
  4. Shortest AS path
  5. Lowest origin type
  6. Lowest multi-exit discriminator (MED)
  7. Lowest IGP metric to the BGP next hop
  8. Prefer external paths (EBGP) over internal paths (IBGP)
  9. If ECMP is enabled, insert up to 10 routes into the routing table
  10. Lowest router ID

FortiGate BGP implementation notes. Scaling capabilities are bounded only by system memory — no hard limits on neighbors, routes, or policies. By default, FortiGate does not originate any prefix — you must enable redistribution or explicitly list prefixes via the network command. Also by default, FortiGate accepts every route it receives; add prefix lists or route maps to filter.

Redistribution. Push non-BGP routes (static, connected, OSPF, RIP) into BGP with:

FortiOS CLI · Redistribute static routes into BGP
# config router bgp
    config redistribute "static"
        set status enable
    end
end

Network command. An alternative to redistribution: explicitly list prefixes to advertise. By default the prefix is announced only when it matches an active route in the routing table. To always advertise the prefix regardless, disable network-import-check:

FortiOS CLI · Explicit prefix advertisement
# config router bgp
    config network
        edit <id>
            set prefix <prefix>
        next
    end
    set network-import-check disable
end

Prefix lists. Filter what BGP advertises to or accepts from each neighbor. By default, any traffic not matching a prefix-list entry is denied — the implicit deny at the end catches everything else. Use set prefix-list-in or set prefix-list-out under config router bgp / config neighbor.

FortiOS CLI · Inbound prefix-list filter
config router prefix-list
    edit filter-subnets
        config rule
            edit 1
                set prefix 10.1.0.0/16
                set action deny
            next
            edit 2
                set prefix 10.0.0.0/8
                set action permit
            next
        end
    next
end

config router bgp
    config neighbor
        edit 10.3.1.254
            set prefix-list-in filter-subnets
        next
    end
end
Common misconception: a prefix list without an explicit permit-all at the end is not an "allow with exceptions" list. Everything not matched is dropped. Order rules carefully and end with a permit if you want a default-allow behavior.
🧠
MENTAL NOTE
BGP's defaults are asymmetric: silent about your own prefixes, promiscuous about your neighbor's. In production you invert both — advertise deliberately, filter aggressively.
