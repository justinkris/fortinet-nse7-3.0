Chapter 08 · NSE7 EF 7.6

Hub-and-spoke that behaves like a full mesh

ADVPN builds spoke-to-spoke shortcuts on demand — full-mesh performance with hub-and-spoke administration.

ADVPN gives you the shape of a full mesh without the maintenance of one.

Configure hub-and-spoke; let the shortcut protocol build direct paths between branches whenever traffic actually needs them.

💭 THINK FIRST · SHORTCUT MECHANICS

Hub-and-spoke is simple to run but ugly to use — every spoke-to-spoke conversation hairpins through the hub. Full mesh is fast but nearly unmanageable at branch scale. ADVPN's insight is that most spoke-to-spoke pairs never talk to each other, so you don't need to preconfigure their tunnels; you just need a signalling protocol that builds them when traffic appears.

  • What are the six ADVPN shortcut messages, and which device sends each one?
  • Why must the initial packet still traverse the hub before a shortcut can be built?
Q1 — MODEL ANSWER
1) Offer — hub tells Spoke 1 a direct path is possible. 2) Query — Spoke 1 asks the hub for Spoke 2's WAN address. 3) Forward — hub relays the query to Spoke 2. 4) Reply — Spoke 2 returns its WAN IP. 5) Forward — hub relays that reply to Spoke 1. 6) Negotiation — Spoke 1 and Spoke 2 build the on-demand IKE tunnel themselves.
Q2 — MODEL ANSWER
The shortcut protocol lives on top of an already-established VPN. The first packet travels Spoke → Hub → Spoke because that's the only path that already exists; only after the hub sees the transit traffic does it fire the Offer message that starts the shortcut negotiation.
SECTION 01

How ADVPN builds a shortcut on demand

In a traditional hub-and-spoke, spoke-to-spoke traffic must transit the hub. That hairpin adds latency, wastes hub CPU, and turns the hub into a bottleneck for every branch conversation. A full mesh eliminates the hairpin but forces every firewall to preconfigure and maintain a tunnel to every other firewall — impossible at branch scale. ADVPN (Auto-Discovery VPN) sits in between: you configure hub-and-spoke, and the hub signals spokes to build direct tunnels only when they actually need them.

ADVPN supports single or multi-hub architectures, NAT for on-demand tunnels, IPv4 and IPv6, and dynamic routing protocols like BGP, OSPF, RIPv2 and RIPng. It relies on IKEv2 and a Fortinet-proprietary network ID attribute that lets multiple IKEv2 tunnels share the same local/remote gateway pair. On the hub, set auto-discovery-sender enable tells FortiGate to fire shortcut messages when transit traffic is seen; on spokes, set auto-discovery-receiver enable tells them to accept those messages and initiate the shortcut.

The message exchange has six steps. When a client behind Spoke 1 sends traffic for a host behind Spoke 2, Spoke 1 encrypts and forwards through the hub. The hub sees the transit and sends a shortcut offer to Spoke 1. Spoke 1 answers with a query; the hub forwards that query to Spoke 2; Spoke 2 replies with its WAN IP; the hub forwards the reply back to Spoke 1; and finally Spoke 1 and Spoke 2 negotiate a direct IKE tunnel using the addresses they just learned. Once the shortcut is up, subsequent packets bypass the hub entirely.

Two networks matter here: the underlay (public internet addresses of each site) and the overlay (private IPs on the IPsec interfaces used by the routing protocol). Tunnels are built on underlay addresses; routing runs over the overlay. If you forget the overlay network, spokes can build tunnels but never learn each other's routes.

Preconditions: IKEv2 must be selected explicitly (set ike-version 2), net-device disable keeps FortiGate from creating a dynamic interface, and add-route disable ensures the routing protocol — not the tunnel — teaches spokes about protected subnets.
🧠
MENTAL NOTE
ADVPN is signalling, not routing — the shortcut is only ever built after a first hairpin through the hub proves the traffic exists.

💭 THINK FIRST · SINGLE-REGION IBGP

A shortcut tunnel is worthless unless the routing table steers the next packet down it. That is the job of the dynamic routing protocol running on the overlay — most commonly IBGP with the hub as a route reflector. If IBGP is misconfigured, ADVPN builds tunnels no one uses.

  • Why must the hub be configured as an IBGP route reflector for ADVPN to work?
  • What is the purpose of setting the IPsec interface IP addresses on the overlay network?
Q1 — MODEL ANSWER
IBGP requires a full mesh of neighbours by default — a route learned from one IBGP peer is not readvertised to another. In a hub-and-spoke overlay, only the hub peers with every spoke. Making the hub a route reflector tells it to forward routes between spokes so they learn each other's networks without needing pairwise IBGP sessions.
Q2 — MODEL ANSWER
The overlay IPs are the addresses BGP uses as next-hop for the routes it advertises. When Spoke 1 learns Spoke 2's subnet, its next-hop is Spoke 2's overlay IP — which resolves recursively to the IPsec virtual interface. Without those IPs, the routing table cannot express "reach that subnet via that tunnel."
SECTION 02

Single-region ADVPN with IBGP

The most common ADVPN topology places one hub, several spokes, and a shared overlay network for routing. Every IPsec phase 1 interface — on the hub and on each spoke — is given a /32 (or matching /24) IP address from the overlay range (e.g. 172.16.1.0/24). Those overlay addresses are what BGP uses as next-hop; they resolve recursively into the IPsec virtual interface so the routing table can say "reach 172.20.3.0/24 via 172.16.1.3, which is on tunnel Hub1-VPN1."

On the hub, IBGP runs with all spokes as neighbours, all inside the same autonomous system. Because IBGP normally doesn't readvertise routes between neighbours, the hub must be configured as a route reflector so it forwards routes learned from Spoke 1 to Spoke 2 and vice versa. Using a neighbor-range with a prefix covering all overlay IPs saves you from defining each spoke individually.

Each spoke's BGP configuration is symmetric and small: define the local AS, pick the hub as the single BGP neighbor, and advertise the local subnet(s) that sit behind the spoke. When both spokes converge, each learns the other's LAN via IBGP; the next-hop resolves to the overlay IP; the packet lands on the IPsec interface; and if a shortcut has been built, it goes directly, otherwise it hairpins via the hub until the shortcut appears.

FortiOS CLI · hub phase 1 for ADVPN
Hub # config vpn ipsec phase1-interface
  edit "VPN1"
    set type dynamic
    set interface "port1"
    set ike-version 2
    set net-device disable
    set add-route disable
    set auto-discovery-sender enable
end

Hub # config vpn ipsec phase2-interface
  edit "VPN1"
    set phase1name "VPN1"
    set proposal aes256-sha256
end
Diagnostic command: get router info routing-table details 172.20.3.62 on a spoke will show the BGP route with its recursive next-hop — proof that the overlay wiring is correct.
🧠
MENTAL NOTE
The overlay network is BGP's map; the underlay is the physical road — ADVPN needs both, doing different jobs, to actually deliver a packet.

💭 THINK FIRST · MULTI-REGION & EBGP

A single hub in one region works fine until a second region joins the picture — different autonomous system, different overlay, its own set of spokes. Now you need two things: IBGP to run inside each region and something else to stitch the regions together. That "something else" is EBGP running between hubs and a new ADVPN role: the auto-discovery forwarder.

  • What does auto-discovery-forwarder do that auto-discovery-sender does not?
  • Why does inter-region routing require EBGP with attribute-unchanged next-hop?
Q1 — MODEL ANSWER
auto-discovery-sender fires shortcut offers between the hub and its own spokes. auto-discovery-forwarder is set on the hub-to-hub tunnel — it tells one hub to relay shortcut messages across the overlay boundary to the other hub, so a spoke in region A can eventually reach a spoke in region B directly.
Q2 — MODEL ANSWER
EBGP normally rewrites the next-hop attribute when routes cross AS boundaries. In ADVPN we want the far spoke's overlay IP preserved so the receiver can build a shortcut to the correct address. attribute-unchanged next-hop tells EBGP to keep the original next-hop; ebgp-enforce-multihop lets EBGP peers reach each other across the overlay rather than requiring direct connection.
SECTION 03

Multi-region ADVPN with IBGP inside, EBGP between

Enterprise deployments rarely stay inside a single overlay. The typical dual-region layout gives each region its own autonomous system, its own overlay network, and its own hub — Hub A anchors AS 65100 with overlay 172.16.1.0/24; Hub B anchors AS 65200 with overlay 172.16.2.0/24. Inside each region, IBGP with the hub as a route reflector handles spoke-to-spoke routing exactly as in the single-region design.

Between the two hubs, EBGP exchanges routing information across regions. Two settings make it work: set attribute-unchanged next-hop preserves the original next-hop so far-side spokes remain reachable at their real overlay IP, and set ebgp-enforce-multihop enable allows the EBGP session to run across the overlay rather than requiring a direct link.

ADVPN gets a third role in this design. On the hub-to-hub IPsec interface, set auto-discovery-forwarder enable. This tells each hub to relay shortcut messages received from one overlay across the boundary to the other overlay. Combined with sender/receiver on the hub-to-spoke tunnels, a shortcut can eventually chain across regions — a spoke in region A negotiates a direct tunnel to a spoke in region B, bypassing both hubs after the initial signalling.

  • Hub-to-hub tunnel: auto-discovery-forwarder enable
  • Hub-to-spoke tunnel: auto-discovery-sender enable plus network-id X so multiple IKEv2 tunnels can share the same gateway pair
  • Spoke-to-hub tunnel: auto-discovery-receiver enable
IPsec templates version: in Use Case 3, FortiManager's hub and branch IPsec templates ship with an Enable ADVPN toggle that wires all three roles automatically. BGP templates offer a matching toggle. Together they let ADVPN be deployed as a preview-then-install workflow rather than a hand-rolled config.
🧠
MENTAL NOTE
Three ADVPN roles — sender at the hub, receiver at the spoke, forwarder between hubs — carry shortcut signalling across every boundary in the design.

💭 THINK FIRST · MIXED & NON-ADVPN SPOKES

ADVPN is a Fortinet-proprietary protocol. If your deployment includes third-party IPsec gateways — or older Fortinet devices that don't yet speak it — you still need those spokes to participate in routing without building shortcuts. The answer is a second phase 2 selector with a very deliberate name.

  • Why does the additional phase 2 on the hub need a name that alphabetically precedes the regular spoke phase 2?
  • What does add-route enable on a phase 2 configuration actually do?
Q1 — MODEL ANSWER
FortiOS looks up phase 2 selectors in alphabetical order and matches the most specific one first. Naming the extra selector Overlay_advertisement makes it precede Spoke, so during migration the hub advertises the overlay's IKE routes before falling through to the generic spoke selector. Reverse the ordering and the specific network never wins the match.
Q2 — MODEL ANSWER
add-route enable injects IKE routes — routes learned via the IPsec SA negotiation itself. That is how non-ADVPN spokes advertise their overlay-side subnet: the hub adds a return route to those subnets via that spoke's tunnel, even though the third-party spoke can't participate in ADVPN's shortcut messaging.
SECTION 04

Mixing ADVPN and non-ADVPN spokes

ADVPN is Fortinet-proprietary. A third-party IPsec gateway sitting as a spoke can participate in the hub-and-spoke design, but it cannot negotiate shortcut messages with other spokes. You still want it to exchange routes with the rest of the network so its LAN is reachable — the trick is a second phase 2 on both the spoke and the hub, using IKE-route injection rather than a dynamic routing protocol.

On the spoke side, phase 1 remains identical to the ADVPN spokes (dynamic type, IKEv2, auto-discovery-receiver enabled where applicable). The change is in phase 2: keep the normal Spoke selector with 0.0.0.0/0 quick-mode selectors, and add a second phase 2 named Overlay_advertisement. Enable add-route enable and set dst-subnet to the overlay subnet that covers all the ADVPN spokes' IPsec interface IPs (for example 172.16.1.0/24). That second phase 2 advertises the overlay via IKE routes and injects reverse routes on the hub so it can route return traffic properly.

Naming matters. FortiOS performs phase 2 lookups in alphabetical order, and the more specific configuration must be matched first. Overlay_advertisement comes alphabetically before Spoke, so diag vpn ike config list shows it in the first position for network-segment lookups. Rename it to something that sorts later and the routing behaviour breaks silently.

FortiOS CLI · non-ADVPN spoke phase 2 for overlay advertisement
config vpn ipsec phase2-interface
  edit "Spoke"
    set phasename "Spoke"
    set proposal aes256-sha256
  next
  edit "Overlay_advertisement"
    set phasename "Spoke"
    set proposal aes256-sha256
    set add-route enable
    set comments "Used by legacy Spokes (non-ADVPN aware) to advertise their overlay IP"
    set dst-subnet 172.16.1.0 255.255.255.0
  next
end
Watch out: the alphabetical-precedence rule is unforgiving. Renaming for aesthetics — e.g. calling the extra selector Z_overlay — will silently move it to the end of the lookup order and break routing.
🧠
MENTAL NOTE
Third-party spokes don't get shortcuts, but they can still share routes via an alphabetically-earlier phase 2 that injects IKE routes for the overlay.

💭 THINK FIRST · SD-WAN & ADVPN 2.0

ADVPN 1.0 made a durable choice: once a shortcut was built via a specific ISP, it stayed on that ISP even if a faster or healthier link appeared. That is overlay stickiness. It's fine for stable environments but bad for SD-WAN, where the whole point is that path selection should react to live link health. ADVPN 2.0 is Fortinet's answer.

  • What is overlay stickiness, and why does classical ADVPN suffer from it?
  • How do edge discovery and path management in ADVPN 2.0 change shortcut selection?
Q1 — MODEL ANSWER
In ADVPN 1.0, the shortcut is built along whichever ISP carried the first data packet through the hub. SD-WAN rules pick that initial path, and once the shortcut exists, it persists on that ISP regardless of whether a different link now has better latency or availability. That persistence is overlay stickiness.
Q2 — MODEL ANSWER
Edge discovery lets spokes learn detailed WAN link info about their peers — WAN IP, transport group, link quality, link cost — via IKE messages exchanged every five seconds. Path management uses that info to compute and re-establish shortcuts on the best current path, independent of the initial hub route. Together they let ADVPN behave like a proper SD-WAN member.
SECTION 05

ADVPN with SD-WAN and ADVPN 2.0

SD-WAN pushes ADVPN in an uncomfortable direction. SD-WAN's promise is that path selection tracks live link health — pick the best ISP right now. But classical ADVPN (1.0) only knows about the ISP that carried the very first transit packet through the hub. Once a shortcut is built on that path, it stays there even if another ISP becomes healthier. The shortcut reply message (step 4) only carries information about the interface that received the packet, not about other links on the same firewall. This behaviour is called overlay stickiness, and it's fine for stable networks but limiting for SD-WAN.

ADVPN 2.0 extends the design with two capabilities. Edge discovery means the shortcut reply carries information about every WAN interface on the responder: WAN IP address, transport group, link quality, link cost, and member configuration order. Spokes update these values every five seconds via IKE messages so they always know the current health of every peer's links. Path management lets the local spoke compute the best shortcut path from all that information and direct IKE to establish (or re-establish) the tunnel on that path — independent of which ISP the hub happened to steer the first packet through.

Practically, ADVPN 2.0 is a natural fit for full-mesh SD-WAN with multiple internet connections. It simplifies connectivity by removing the dependency on specific BGP designs, improves resiliency because shortcuts adjust to changing link status, and enhances routing because the choice of path is now tied to the SD-WAN rules and live parameters rather than to the initial transit ISP.
