Objective

Deploy all five Fortinet VMs on Proxmox VE, build the network bridges, simulate an ISP with internet access, and finish with every device booted, licensed, password-protected, and reachable.

Why this matters

Every failed lab later in this guide will make you doubt your FortiGate configuration — when half the time the real fault will be a VM plugged into the wrong bridge or a missing VLAN tag. Building the environment carefully once means every future symptom is meaningful. This is also your first taste of a core senior-engineer habit: know your physical (here, virtual) layer cold before you blame anything above it.

Think first
  1. A Proxmox bridge behaves like which classic piece of network hardware — a hub, a switch, or a router?
  2. The three FortiGates must all reach the "internet". What single component in this design plays the role of the ISP, and what two functions must it perform?
  3. Why give each site's LAN its own bridge instead of putting everything on one big bridge with VLANs?
  4. FortiManager and FortiAnalyzer sit behind the HQ FortiGate. What must be true on HQ-FGT before the branches can ever reach them?

Stop reading and write your predictions before continuing.

Mental model — the office park

Proxmox is an office park you own. Each bridge is an unmanaged switch bolted to a wall — anything you plug into it can talk to anything else on it. Each VM NIC is a patch cable. The Proxmox host itself, with NAT enabled, is the ISP's street cabinet: it owns the uplink to the real world and hands the tenants a gateway address. If two VMs can't talk, check the patch cables (bridge assignments) before you check the tenants (VM configs).

Step 1 — Create the network bridges

On the Proxmox host, create five internal bridges. vmbr0 (your existing host bridge) stays untouched.

Proxmox GUI: Datacenter → node → System → Network → Create → Linux Bridge (then Apply Configuration)

Or append to /etc/network/interfaces (adjust and reboot or ifreload -a):

# LAB-WAN — the "ISP". Host owns 203.0.113.1 and NATs to the real internet
auto vmbr1
iface vmbr1 inet static
    address 203.0.113.1/24
    bridge-ports none
    bridge-stp off
    bridge-fd 0

auto vmbr10
iface vmbr10 inet manual
    bridge-ports none
    bridge-stp off
    bridge-fd 0
    bridge-vlan-aware yes
    bridge-vids 2-4094
# repeat the vmbr10 stanza (without vlan-aware lines) for vmbr20/30/40/99

Step 2 — Turn the Proxmox host into the ISP

Enable forwarding and NAT so 203.0.113.0/24 can reach the real internet through the host's uplink (assumed vmbr0):

echo 'net.ipv4.ip_forward=1' > /etc/sysctl.d/99-lab.conf
sysctl -p /etc/sysctl.d/99-lab.conf

iptables -t nat -A POSTROUTING -s 203.0.113.0/24 -o vmbr0 -j MASQUERADE
# make persistent:
apt install -y iptables-persistent && netfilter-persistent save
Tip

203.0.113.0/24 is a reserved documentation range (TEST-NET-3), so it will never collide with your home network. The lab treats it as "public" address space.

Step 5 — Create the VMs (example: HQ FortiGate)

Create a shell VM, then import the QCOW2 as its boot disk:

# 1. Create VM 101 with no disk
qm create 101 --name LOF-HQ-FGT01 --memory 2048 --cores 1 \
  --net0 virtio,bridge=vmbr1 \
  --net1 virtio,bridge=vmbr10 \
  --net2 virtio,bridge=vmbr40 \
  --net3 virtio,bridge=vmbr99 \
  --serial0 socket --ostype l26

# 2. Import and attach the FortiGate image
qm importdisk 101 /var/lib/vz/images/fortios.qcow2 local-lvm
qm set 101 --scsi0 local-lvm:vm-101-disk-0 --boot order=scsi0

# 3. Add a 30 GB log disk
qm set 101 --scsi1 local-lvm:30

Repeat for the others, adjusting name/bridges:

Common mistake #1

Proxmox NIC order (net0…net3) maps to FortiGate port1…port4 in creation order. Mixing this up is the #1 cause of "my FortiGate can't see anything". If in doubt, compare MAC addresses: VM → Hardware in Proxmox vs get hardware nic port1 in FortiOS.

Step 6 — First boot & console

Use the serial console (qm terminal 101 or the GUI's xterm.js) — the noVNC "screen" works too, but serial lets you copy-paste. Default login is admin with a blank password; FortiOS immediately forces a change. Set strong passwords and record them.

Step 7 — Base interface config & connectivity test (HQ example)

config system global
    set hostname LOF-HQ-FGT01
end
config system interface
    edit port1
        set alias wan1-isp
        set mode static
        set ip 203.0.113.10/24
        set allowaccess ping https ssh
    next
end
config router static
    edit 1
        set gateway 203.0.113.1
        set device port1
    next
end

execute ping 203.0.113.1        # the ISP (Proxmox host)
execute ping 8.8.8.8            # the real internet through host NAT

Repeat on the branches with .20 and .30. Then license each device:

FortiGate GUI: System → FortiGuard → License (upload eval/VM licence) — FMG/FAZ: System Settings → Dashboard → License Information

Step 8 — Snapshots

qm snapshot 101 lab0-clean --description "Fresh install, WAN up, licensed"
# repeat for 102,103,111,112
Snapshot policy

Take a snapshot named labN-done at the end of every completed lab. When Lab 21 asks you to break things on purpose, you'll be glad you did.

Common deployment mistakes

  • Forgot "Apply Configuration" after creating bridges — they exist in the GUI but not in the kernel.
  • vmbr10 not VLAN-aware — tagged frames from the FortiGate trunk get silently dropped.
  • Firewall enabled on the VM NIC in Proxmox — untick it; Proxmox's own firewall will eat lab traffic mysteriously.
  • Missing MASQUERADE rule — FortiGates ping 203.0.113.1 fine but nothing beyond. That symptom = NAT problem at the ISP, not a FortiGate fault.
  • FMG/FAZ given 2 GB RAM — they boot, then the GUI crawls or database services fail. Give them 8 GB.
