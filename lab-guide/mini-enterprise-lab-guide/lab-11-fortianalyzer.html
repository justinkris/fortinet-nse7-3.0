Objective

Bring LOF-FAZ01 online, point all three FortiGates at it, authorise the devices, verify the log pipeline end-to-end, and build one report — turning your scattered per-device logs into a single source of truth.

Why this matters

Local disk logging dies with the device, fills quickly, and can't correlate. When (not if) a security incident spans HQ and a branch, only centralised logging can tell the story. FortiAnalyzer is also a compliance requirement for a finance company: log retention with integrity. And troubleshooting the log path itself — device to FAZ over OFTP — is a classic exam scenario.

Think first
  1. The branch FortiGates must log to 10.10.99.20, which sits behind HQ-FGT. List everything that must be true (routing, policy, NAT?) for a branch log packet to arrive — today, over the simulated internet.
  2. Why does FAZ require you to authorise each device rather than accepting logs from anyone who sends them?
  3. What's the difference between a log being received and being indexed/searchable? Why might there be a lag?
  4. Realtime vs store-and-upload log transmission: which risks losing recent logs in a crash, and which risks WAN congestion bursts?

Stop reading and write your predictions before continuing.

Mental model — the central archive

Each FortiGate is a branch office scribbling events in a notebook that gets thrown away when full. FortiAnalyzer is the head-office archive: every branch posts certified copies of its records (OFTP, authenticated), archivists file them into a searchable index (SQL insertion — received ≠ searchable yet), and auditors can finally ask questions that span branches: "show me everything this attacker touched, everywhere, in order."

Configuration — FAZ side

# FAZ CLI (first boot):
config system interface
    edit port1
        set ip 10.10.99.20/24
        set allowaccess ping https ssh
    next
end
config system route
    edit 1
        set gateway 10.10.99.1
        set device port1
    next
end
FAZ GUI: Device Manager — unauthorised devices appear here after they first connect; click Authorize

Configuration — FortiGate side (all three)

config log fortianalyzer setting
    set status enable
    set server 10.10.99.20
    set upload-option realtime
    set reliable enable
end
# Log filter: make sure traffic logging is on the policies (logtraffic all — done in Lab 3)
The branch trap

Branches reach 10.10.99.20 across the simulated ISP — but 10.10.x.x is not routable there, and HQ has no inbound policy for it. For now, publish FAZ to the branches with a VIP on HQ (e.g. 203.0.113.90 → 10.10.99.20, OFTP tcp/514) plus a WAN→MGMT policy restricted to the two branch WAN IPs — and point branches at 203.0.113.90. Write in your notes why this is a temporary compromise, and which lab replaces it with the proper answer (Lab 19: logs ride the VPN to the real 10.10.99.20).

Verification — walk the pipeline

# FortiGate side:
diagnose log fortianalyzer-queue stats     # anything stuck in the send queue?
execute log fortianalyzer test-connectivity
FortiAnalyzer Host: ... Registration: registered  Connection: allow  Log: Tx & Rx

# FAZ side:
diagnose test application oftpd 3          # connected devices list
diagnose fortilogd lograte                 # logs/sec arriving
diagnose sql status sql-rebuild-db         # if searches lag: is indexing behind?

Then in FAZ GUI: Log View → Traffic — filter by device, confirm all three FortiGates are represented. Build a quick report: Reports → Templates → run a bandwidth/application summary against the last hour.

Reflection

  • Logs from BRA stop arriving but HQ's keep flowing. Order your checks across the two devices and the path between them — which single FortiGate command halves the search space immediately?
  • Why does reliable enable (TCP) matter more for a finance company than the default? What did it cost?
  • What is the operational difference between FAZ being down for an hour (FortiGates queue/miss?) vs FAZ disk being full? Which FortiGate-side command shows the first case?
Challenge exercise

Configure a FAZ event handler that raises an alert when any FortiGate logs an IPS critical event, and prove it fires by replaying your Lab 8 attack. Where would you send the notification in production, and why is email-to-the-engineer the weakest of the options?
