Concept-first walk-through — build carefully, break deliberately, verify always.

Objective

Protect HQ-WEB-1 with an IPS sensor on the inbound VIP policy, understand signatures vs rate-based detection vs anomalies, walk the monitor→block lifecycle, and handle a false positive with a filter override.

Why this matters

IPS is where security aspiration meets operational reality. Turn everything to block on day one and you will break legitimate business traffic; leave it in monitor forever and you own an expensive log generator. The senior skill is the tuning lifecycle — and, on the exam, correctly diagnosing "IPS is dropping legitimate traffic" scenarios from log evidence.

Think first
  1. AV inspects files; IPS inspects protocol behaviour and exploit patterns. Give one attack IPS catches that AV never would.
  2. Why does an IPS sensor for a web server differ from one protecting user browsing? Think about direction and which side is the victim.
  3. A signature fires 500 times/day, always from the monitoring server's health checks. False positive or true positive with benign intent — and does the distinction change your action?
  4. Where should server-protection IPS attach: the WAN→DMZ policy or the LAN→WAN policy? Why?

Stop reading and write your predictions before continuing.

Mental model — the behavioural spotters

If AV is the X-ray looking for known contraband objects, IPS is the plain-clothes team watching behaviour: someone testing every door handle (port scan), speaking rapid-fire nonsense at a clerk to trip them up (protocol exploit), or a thousand people converging on one counter (flood). Tuning is briefing the spotters on your building's normal quirks — the cleaner who legitimately rattles doors at 3am (your monitoring server) goes on the known-benign list, not the ignore-everything list.

Configuration

config ips sensor
    edit "IPS_DMZ_WEB"
        set comment "Protect HQ-WEB-1 (Linux/Apache)"
        config entries
            edit 1
                set severity high critical
                set os Linux
                set application Apache
                set status enable
                set action block
                set log-packet enable
            next
        end
    next
end
# Attach to the inbound VIP policy:
config firewall policy
    edit 3
        set utm-status enable
        set ips-sensor IPS_DMZ_WEB
        set ssl-ssh-profile certificate-inspection
    next
end
GUI: Security Profiles → Intrusion Prevention — filter signatures by OS/application/severity rather than picking thousands by hand

Verification — trigger and observe

# From the Proxmox host (playing the internet attacker), poke the VIP with
# something signature-worthy — e.g. an oversized/abnormal probe using hping3 or
# a scanner like nikto against http://203.0.113.80/.

# On HQ-FGT:
diagnose ips filter set "policy 3"
get ips session status
# Logs: GUI → Log & Report → Security Events → Intrusion Prevention
action: dropped   severity: high   attack: <signature-name>   src: 203.0.113.1

Handling a false positive

  1. Identify the exact signature ID and the source from the log entry.
  2. Decide scope: is the traffic benign from this source only, or is the signature wrong for your app everywhere?
  3. Add a sensor entry overriding that one signature (action pass, or exempt-ip for the one source) — never disable the sensor, never exempt the whole source from all signatures.
config ips sensor
    edit "IPS_DMZ_WEB"
        config entries
            edit 10
                set rule <signature-id>
                set status enable
                set action pass
                set log enable          # keep visibility even while passing
            next
        end
    next
end

Reflection

  • Why does the override entry keep logging enabled — what future question does that log answer?
  • Rate-based signatures need a threshold and window. What business knowledge do you need before setting them for LoFinance's web server?
  • For encrypted attacks against the server, what additional profile from Lab 7 must join this policy — and whose certificate does inbound inspection use? (Hint: for your own server you hold the real key — investigate SSL inspection's protecting-server mode with the server's own certificate, versus the CA re-sign used for outbound.)
Challenge exercise

Build a second sensor IPS_USERS_OUT for the LAN-to-WAN policy targeting client-side threats (severity medium+, target = client). Explain in your notes why its filter is almost the mirror image of the DMZ sensor's.
