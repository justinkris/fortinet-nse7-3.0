Objective

Convert the hub-and-spoke into ADVPN so spoke-to-spoke traffic dynamically builds direct shortcut tunnels on demand, with the hub acting as the introduction service — and observe a shortcut being born in real time.

Why this matters

Hub-and-spoke scales control but not traffic: every BRA↔BRB packet taxes HQ's WAN and adds a triangle of latency — painful for voice and file transfer. Full mesh scales traffic but not configuration (n² tunnels). ADVPN is the industry's answer: configure a star, behave like a mesh, on demand. It's also a flagship 7.6 exam topic, including how shortcuts interact with routing.

Think first
  1. For 12 branches, count the tunnels in hub-and-spoke vs full mesh. At what business event does the difference change from academic to budget-line?
  2. For BRA to build a direct tunnel to BRB it must learn BRB's WAN address and agree keys. Who is the only party that knows both spokes — and what does that suggest the hub's ADVPN role is?
  3. Once a shortcut exists, BRA needs a route to 10.30.0.0/16 via the shortcut instead of via the hub. Static routes can't do this. What class of mechanism can react to a tunnel appearing at runtime?
  4. Should the first packets of a BRA→BRB flow wait for the shortcut, or flow via the hub while it forms? What does your answer imply users experience?

Stop reading and write your predictions before continuing.

Mental model — the mutual friend

The hub is the mutual friend who knows everyone. At first, all conversation flows through them ("tell Branch B I said…"). ADVPN is the friend saying "actually — Branch B, meet Branch A, here's her number" (a shortcut offer carrying the peer's address and keying help). The two now talk directly, but the friendship with the hub remains: if the direct line goes quiet, conversation falls back through the mutual friend. Dynamic routing is the address book that instantly records the new direct number — and crosses it out again if the line dies.

Configuration

Three moving parts: (1) auto-discovery flags on the phase1s, (2) a routing mechanism over the overlay — we'll use IBGP with the hub as route reflector, the 7.6-era pattern, (3) spoke tunnel addressing that supports shortcuts.

# HQ hub — phase1 additions:
config vpn ipsec phase1-interface
    edit "vpn-bra"
        set auto-discovery-sender enable
    next
    edit "vpn-brb"
        set auto-discovery-sender enable
    next
end

# Spokes — phase1 addition (BRA shown):
config vpn ipsec phase1-interface
    edit "vpn-hub"
        set auto-discovery-receiver enable
    next
end

# BGP — hub as route reflector (AS 65000 everywhere):
config router bgp
    set as 65000
    set router-id 10.255.0.1
    config neighbor-group
        edit "spokes"
            set remote-as 65000
            set route-reflector-client enable
        next
    end
    config neighbor-range
        edit 1
            set prefix 10.255.0.0/16
            set neighbor-group "spokes"
        next
    end
    config network
        edit 1
            set prefix 10.10.0.0/16
        next
    end
end

# Spoke BGP (BRA):
config router bgp
    set as 65000
    set router-id 10.255.1.2
    config neighbor
        edit 10.255.1.1
            set remote-as 65000
        next
    end
    config network
        edit 1
            set prefix 10.20.0.0/16
        next
    end
end
# Remove the static branch-to-branch routes from Lab 19 — BGP owns inter-site now.
# Keep policies zone-based (Z_VPN / vpn-hub) so shortcuts inherit them.
Gotchas that eat afternoons

(1) Shortcut tunnels are spawned as children of the existing phase1 — your policies must reference the parent interface or its zone, or shortcut traffic hits implicit deny. (2) The spokes' WAN addresses must be directly reachable to each other (they are, on vmbr1) — behind separate NAT devices you'd be testing NAT-T behaviour too. (3) If BGP peers via tunnel IPs, the hub must be able to reach spoke tunnel IPs — hence the neighbor-range covering 10.255.0.0/16.

Verification — watch a shortcut be born

# Before: confirm BRA reaches BRB via the hub
get router info bgp summary
get router info routing-table bgp
execute traceroute-options source 10.20.10.1
execute traceroute 10.30.10.1              # expect two overlay hops (via hub)

# Generate sustained spoke-to-spoke traffic (ping BRA-PC→BRB-PC), then on BRA:
diagnose vpn tunnel list                   # a new _0 child tunnel appears — the shortcut
get router info routing-table all          # 10.30/16 next-hop now recursing via the shortcut
execute traceroute 10.30.10.1              # expect ONE overlay hop — direct

# On the hub, the introduction service at work:
diagnose debug application ike -1          # look for shortcut offer/query exchanges

Reflection

  • Narrate the full life of the first BRA→BRB ping: which path did packet 1 take, what messages created the shortcut, and when did the routing table switch? (Being able to tell this story is understanding ADVPN.)
  • Kill the shortcut (bring the child tunnel down). What happens to in-flight traffic and how quickly — and what does that say about the hub's continuing importance?
  • Why route reflection? What specifically breaks in vanilla IBGP with 12 spokes that made the hub-as-RR design necessary?
