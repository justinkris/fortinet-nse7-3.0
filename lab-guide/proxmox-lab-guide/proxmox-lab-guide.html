The LoFinance Home Lab — Proxmox Build Guide

Three FortiGates. One FortiManager. One hypervisor. Everything Señor makes you do at LoFinance, you can now break — and fix — on your own hardware. This guide takes you from an empty Proxmox node to a running miniature of the LoFinance network, then hands you a lab series mapped to the NSE7 Enterprise Firewall 7.6 blueprint.

01 · What You're Building

A Miniature LoFinance

You have 3 FortiGate-VMs and 1 FortiManager-VM. That maps perfectly onto the core of the LoFinance story: the Sydney HQ hub, the Parramatta branch, the DR site, and central management. Chatswood is the one site we can't afford a chassis for — which is fine, because everything Chatswood taught (dial-up spokes) can be practised with FGT-DR wearing a different hat for an afternoon.

VMHostnameRole in the storyRole in the lab
FortiGate #1FGT-HQSydney HQ edge — NGFW, OSPF hub, VPN hubHub for every VPN/routing lab; security-profile labs run here
FortiGate #2FGT-PARRAParramatta branch — spoke, dual-ISP sitePrimary spoke; ADVPN shortcut endpoint
FortiGate #3FGT-DRDR site — spoke, legacy mgmt subnetSecond spoke; redistribution & filtering labs; occasional "Chatswood stand-in"
FortiManagerFMG-LOFICentral management, local FDSADOMs, policy packages, install wizard, revision history, scripts
Licensing: both FortiGate-VM and FortiManager-VM offer a free evaluation license (register on support.fortinet.com, download the KVM images). The FortiGate eval caps you at roughly 1 vCPU / 2 GB RAM per VM and limited crypto — completely fine for these labs. The FortiManager eval supports a small number of managed devices — enough for our three. Size the VMs to the eval limits and the lab runs happily on a modest node (~10–12 GB RAM total including clients).
02 · The Map

Lab Topology

Solid blue = WAN uplinks · amber dashes = IPsec overlay (built in the labs) · green dashes = out-of-band management. Nothing here needs VLANs on Proxmox — one bridge per network keeps it simple.
Design decision worth noticing: the tunnels run over vmbr0, exactly like the real LoFinance tunnels run over the internet. Your home LAN plays the role of every ISP at once. This means IKE, NAT-T, and "the internet is up but the tunnel is down" troubleshooting all behave realistically.

04 · Getting Started

Proxmox Preparation

Everything below assumes a working Proxmox VE node where vmbr0 already exists and gives VMs internet access (the default installer setup). You'll add four isolated internal bridges — no physical NIC attached, no IP on the host needed.

Step 1 — Create the internal bridges

GUI path: Datacenter → your node → System → Network → Create → Linux Bridge. Create each with no ports/slaves and no IP, then click Apply Configuration. Or do all four in one shot from the node's shell:

Proxmox node shell — /etc/network/interfaces additions
auto vmbr1
iface vmbr1 inet manual
    bridge-ports none
    bridge-stp off
    bridge-fd 0
# LAB-MGMT 10.0.99.0/24

auto vmbr2
iface vmbr2 inet manual
    bridge-ports none
    bridge-stp off
    bridge-fd 0
# HQ-LAN 10.1.0.0/24

auto vmbr3
iface vmbr3 inet manual
    bridge-ports none
    bridge-stp off
    bridge-fd 0
# PARRA-LAN 10.2.0.0/24

auto vmbr4
iface vmbr4 inet manual
    bridge-ports none
    bridge-stp off
    bridge-fd 0
# DR-LAN 10.3.0.0/24

Then apply: ifreload -a

Step 2 — Download the images

  • Create a free account at support.fortinet.com → Support → Firmware Download.
  • FortiGate: product FortiGate → version 7.6.x → file like FGT_VM64_KVM-v7.6.x…out.kvm.zip. Unzip → you get fortios.qcow2.
  • FortiManager: product FortiManager → 7.6.x → FMG_VM64_KVM…out.kvm.zip. Unzip → fmg.qcow2.
  • Copy both qcow2 files to the Proxmox node (e.g. scp fortios.qcow2 root@pve:/root/).
Why KVM images? Proxmox is KVM/QEMU under the hood. The FortiGate KVM image boots natively — no conversion needed, just an import.
05 · Build

Creating the Three FortiGate VMs

We'll build FGT-HQ as VM ID 101, then repeat for 102 (PARRA) and 103 (DR) — only the LAN bridge changes. Eval-license sizing: 1 core, 2048 MB.

Proxmox node shell — FGT-HQ (VM 101)
# 1. Create the VM shell with three virtio NICs
qm create 101 --name FGT-HQ --memory 2048 --cores 1 --ostype l26 \
  --net0 virtio,bridge=vmbr0 \
  --net1 virtio,bridge=vmbr2 \
  --net2 virtio,bridge=vmbr1

# 2. Import the FortiOS disk (adjust storage name if not local-lvm)
qm importdisk 101 /root/fortios.qcow2 local-lvm

# 3. Attach it as virtio0 and make it the boot disk
qm set 101 --virtio0 local-lvm:vm-101-disk-0
qm set 101 --boot order=virtio0

# 4. Add a 30 GB log disk (FortiOS formats it on first use)
qm set 101 --virtio1 local-lvm:30

# 5. Serial console works best for FortiOS first-boot
qm set 101 --serial0 socket

# 6. Start it
qm start 101

Now the other two — identical except the name and the LAN bridge on net1:

Proxmox node shell — FGT-PARRA (102) and FGT-DR (103)
qm create 102 --name FGT-PARRA --memory 2048 --cores 1 --ostype l26 \
  --net0 virtio,bridge=vmbr0 --net1 virtio,bridge=vmbr3 --net2 virtio,bridge=vmbr1
qm importdisk 102 /root/fortios.qcow2 local-lvm
qm set 102 --virtio0 local-lvm:vm-102-disk-0 --boot order=virtio0
qm set 102 --virtio1 local-lvm:30 --serial0 socket
qm start 102

qm create 103 --name FGT-DR --memory 2048 --cores 1 --ostype l26 \
  --net0 virtio,bridge=vmbr0 --net1 virtio,bridge=vmbr4 --net2 virtio,bridge=vmbr1
qm importdisk 103 /root/fortios.qcow2 local-lvm
qm set 103 --virtio0 local-lvm:vm-103-disk-0 --boot order=virtio0
qm set 103 --virtio1 local-lvm:30 --serial0 socket
qm start 103
NIC ordering matters: Proxmox net0 → FortiOS port1, net1 → port2, net2 → port3. That's why every device gets WAN on net0, LAN on net1, MGMT on net2 — the config below assumes this order.

Open each VM's console (the xterm.js serial console works well), log in as admin with a blank password, and set a new password when prompted. Then apply the baseline config in section 07.

06 · Build

Creating the FortiManager VM

Proxmox node shell — FMG-LOFI (VM 110)
qm create 110 --name FMG-LOFI --memory 8192 --cores 2 --ostype l26 \
  --net0 virtio,bridge=vmbr0 \
  --net1 virtio,bridge=vmbr1

qm importdisk 110 /root/fmg.qcow2 local-lvm
qm set 110 --virtio0 local-lvm:vm-110-disk-0
qm set 110 --boot order=virtio0

# FortiManager REQUIRES a second data disk for its database/logs (80 GB min)
qm set 110 --virtio1 local-lvm:80

qm set 110 --serial0 socket
qm start 110

If your node is tight on RAM, FMG will limp along on 4 GB for a 3-device lab, but 8 GB is much smoother. Log in on the console (admin, blank password), then:

FMG-LOFI console — initial network
config system interface
    edit port1
        set mode dhcp        # internet-facing, for FortiGuard
        set allowaccess ping https ssh
    next
    edit port2
        set ip 10.0.99.10 255.255.255.0
        set allowaccess ping https ssh
    next
end

config system global
    set hostname FMG-LOFI
end

Browse to https://10.0.99.10 from a machine on vmbr1 (see section 08 for the admin VM), or temporarily to port1's DHCP address from your home LAN. Complete the setup wizard and activate the evaluation license when prompted.

07 · First Boot

Baseline Config — Every FortiGate

Paste this into each console, changing only the hostname and the two IP addresses per the plan in section 03. Shown here for FGT-HQ:

FGT-HQ console — baseline
config system global
    set hostname FGT-HQ
    set timezone "Australia/Sydney"
end

config system interface
    edit port1
        set alias WAN
        set mode dhcp
        set allowaccess ping
    next
    edit port2
        set alias LAN
        set ip 10.1.0.1 255.255.255.0        # PARRA: 10.2.0.1 · DR: 10.3.0.1
        set allowaccess ping
    next
    edit port3
        set alias MGMT
        set ip 10.0.99.1 255.255.255.0       # PARRA: .2 · DR: .3
        set allowaccess ping https ssh fgfm  # fgfm = FortiManager protocol
    next
end

# Basic LAN → internet policy so clients can browse
config firewall policy
    edit 1
        set name "LAN-to-Internet"
        set srcintf port2
        set dstintf port1
        set srcaddr all
        set dstaddr all
        set action accept
        set schedule always
        set service ALL
        set nat enable
    next
end
Sanity checks before moving on — on each FortiGate:
get system status (hostname, version, eval license) · get router info routing-table all (default route learned via DHCP on port1) · execute ping 8.8.8.8 · execute ping 10.0.99.10 (FMG reachable on the mgmt plane).

Register the three FortiGates to FMG-LOFI

  1. On each FortiGate: config system central-management → set type fortimanager → set fmg 10.0.99.10 → end.
  2. On FMG-LOFI: Device Manager → Unauthorized Devices → authorize all three into an ADOM named LoFinance.
  3. Verify tunnel state on a FortiGate: diagnose fdsm central-mgmt-status — you want the FGFM tunnel up.

That's the whole build. From here on, everything is a lab.

08 · Traffic Generators

Test Clients (Cheap Ones)

You need something behind each firewall to generate traffic and prove policies. LXC containers are perfect — a Debian container idles at ~50 MB RAM. Create three via local → CT Templates → download debian-12-standard, then:

Proxmox node shell — one client per LAN
pct create 201 local:vztmpl/debian-12-standard_12.7-1_amd64.tar.zst \
  --hostname lofi-hq-pc --memory 512 --cores 1 \
  --net0 name=eth0,bridge=vmbr2,ip=10.1.0.10/24,gw=10.1.0.1

pct create 202 local:vztmpl/debian-12-standard_12.7-1_amd64.tar.zst \
  --hostname lofi-parra-pc --memory 512 --cores 1 \
  --net0 name=eth0,bridge=vmbr3,ip=10.2.0.10/24,gw=10.2.0.1

pct create 203 local:vztmpl/debian-12-standard_12.7-1_amd64.tar.zst \
  --hostname lofi-dr-pc --memory 512 --cores 1 \
  --net0 name=eth0,bridge=vmbr4,ip=10.3.0.10/24,gw=10.3.0.1

pct start 201 && pct start 202 && pct start 203

Optionally add a fourth container or small desktop VM on vmbr1 (e.g. 10.0.99.50) as your admin workstation — this is where you open the FortiGate GUIs and FMG in a browser, exactly like sitting in the LoFinance NOC. Alternatively, temporarily enable https on each port1 and manage from your home LAN — less tidy, but it works.

09 · The Curriculum

The Lab Series — Mapped to the NSE7 Blueprint

Run these in order. Each lab states its goal, tasks, and — most importantly — the verification and diagnostic commands, because the exam shows you outputs and asks what they mean. Take a snapshot of all four VMs before every lab (section 10).

Lab 1

Build Verification & System Fundamentals

Blueprint: System & Session Mgmt

Goal: prove the build, and learn to read the tools you'll use in every later lab.

  • From lofi-hq-pc, browse the internet through FGT-HQ. Watch the session appear: diagnose sys session list — find your session, identify NAT, timers, state.
  • Run the packet sniffer while pinging: diagnose sniffer packet any 'icmp' 4 — learn verbosity levels 3, 4, 6.
  • Run debug flow for one ping: diagnose debug flow filter addr 8.8.8.8 → diagnose debug flow trace start 10 → diagnose debug enable. Read every line: ingress, route lookup, policy match, NAT.
  • Explore get system performance status and diagnose hardware sysinfo memory (conserve-mode thresholds are exam material).

Done when: you can narrate a packet's full path through the FortiGate from a debug-flow output without looking anything up.

Lab 2

FortiManager Foundations

Blueprint: Central Management

Goal: stop configuring devices directly; manage them the enterprise way.

  • Create the LoFinance ADOM (7.6), authorize all three FortiGates into it.
  • Import each device's existing policy into a policy package per site. Understand what the import wizard does with objects and interface mappings.
  • Make a policy change in FMG, use the Install Wizard, and read the install preview diff before pushing.
  • Break it on purpose: change something directly on FGT-DR, then observe the out-of-sync status in FMG. Practise both resolutions (retrieve vs. re-install).
  • Explore revision history and revert one change.

Verify: diagnose dvm device list (FMG) · diagnose fdsm central-mgmt-status (FGT) · config status Synchronized for all three.

Lab 3

Hub-and-Spoke IPsec

Blueprint: VPN

Goal: rebuild the LoFinance overlay — HQ as hub, PARRA and DR as spokes — using route-based IKEv2 tunnels over vmbr0.

  • Tunnel addressing: HQ↔PARRA 10.255.1.1/.2 /30, HQ↔DR 10.255.2.1/.2 /30. Static routes to remote LANs via tunnel interfaces, plus policies both directions.
  • Deliberately mismatch a phase-2 selector first and watch it fail — then fix it. The failure output is worth more than the success.
  • Test: lofi-parra-pc pings lofi-hq-pc; then PARRA→DR through the hub (this needs routes and policies on HQ — think about why).

Verify & diagnose: get vpn ipsec tunnel summary · diagnose vpn ike gateway list · diagnose vpn tunnel list · IKE debug: diagnose vpn ike log-filter dst-addr4 x.x.x.x → diagnose debug application ike -1.

Lab 4

OSPF Over the Overlay

Blueprint: Routing

Goal: replace Lab 3's static routes with OSPF Area 0 across the tunnels — the Season 1 finale, rebuilt with your own hands.

  • Advertise each LAN; set network-type point-to-multipoint on the hub-side tunnel interface(s) and watch what changes in the neighbor table.
  • Confirm the fingerprint: neighbors show Full/ - — no DR/BDR on p2mp. Know why.
  • Create loopback 10.99.50.1/24 on FGT-DR as a static/connected "legacy subnet," redistribute it, then filter the redistribution with an access-list + route-map so only that prefix leaks (Session 15's MGMT-ONLY lesson).
  • Stretch: adjust OSPF cost to prefer one path, and reason about symmetry before you commit (Session 16's asymmetric-cost trap).

Verify: get router info ospf neighbor · get router info routing-table ospf · get router info ospf database brief · diagnose ip rtcache list.

Lab 5

ADVPN — Shortcuts Between Spokes

Blueprint: VPN

Goal: spoke-to-spoke traffic currently hairpins through HQ. Enable ADVPN so PARRA↔DR builds a dynamic shortcut.

  • Convert the hub to a dynamic (dial-up) phase 1 with auto-discovery-sender enable; spokes get auto-discovery-receiver enable.
  • Start a PARRA→DR ping, then watch the shortcut appear: diagnose vpn tunnel list — look for the child tunnel and the shortcut flag.
  • Understand the routing requirement that makes shortcuts usable (this is where Lab 4's OSPF pays rent).

Verify: traceroute from lofi-parra-pc to 10.3.0.10 — before: two hops through HQ; after: direct. get router info routing-table all on a spoke while the shortcut is up.

Lab 6

BGP Fundamentals

Blueprint: Routing

Goal: run iBGP between HQ and the spokes over the tunnels (own AS, e.g. 65001), alongside or instead of OSPF.

  • Peer HQ↔PARRA and HQ↔DR; advertise the LANs via network statements.
  • HQ as route reflector so the spokes learn each other's routes — observe what breaks without it (iBGP split-horizon, a classic exam distractor).
  • Apply a route-map on one peer to set local-preference and watch path selection change.

Verify: get router info bgp summary · get router info bgp network · get router info routing-table bgp — and be able to say which route wins when OSPF and BGP both offer one (administrative distance).

Lab 7

SSL Inspection, IPS & Web Filtering at HQ

Blueprint: Security Profiles

Goal: rebuild the Season 2 inspection layer on FGT-HQ's LAN→WAN policy.

  • Clone deep-inspection → LoFinance-deep-inspection; export the CA cert and install it on lofi-hq-pc so HTTPS inspection works cleanly (curl -v https://… shows the FortiGate-signed chain).
  • Add a web filter profile; block a category; verify the block page and the log entry.
  • Enable an IPS sensor in monitor first, review logs, then switch to block — the monitor-first discipline from the trading-API story.
  • Test detection safely with the EICAR test file (AV) and curl against a known IPS test signature if available.

Verify: diagnose test application ipsmonitor 1 (engine status) · forward-traffic logs with security events · diagnose debug flow showing the session offloaded to inspection.

Lab 8

FortiManager as Local FDS

Blueprint: Central Management

Goal: the Session 23 build — FortiGates rate URLs and pull updates from FMG-LOFI instead of public FortiGuard.

  • Enable FortiGuard services on FMG (FortiGuard → Settings), let it download packages via port1.
  • Point each FortiGate's FortiGuard config at 10.0.99.10; disable public fallback and understand the risk of doing so.
  • Break it: shut FMG's port1 and observe what happens to web-filter ratings on the FortiGates. Fix, and articulate the failure domain you just created.

Verify: diagnose debug rating on a FortiGate (server list should show 10.0.99.10) · FMG's FortiGuard download status page.

Lab 9

The Troubleshooting Gauntlet

Cowboy Hat

Goal: break/fix under exam-realistic conditions. Snapshot everything first. Then, for each fault: restate the symptom → define scope → build the expected packet path → gather evidence → one hypothesis at a time → verify recovery and root cause.

Faults to plant (do one per sitting; better yet, apply one blind via FMG script a day in advance so you genuinely don't remember):

  • Wrong phase-2 selector on one spoke tunnel.
  • OSPF network-type mismatch on one side (p2p vs p2mp) — neighbors stuck.
  • Missing return-path route on HQ for spoke-to-spoke traffic.
  • A policy above the intended one silently matching first.
  • MTU-induced failure: small pings pass, HTTPS hangs (clamp MSS to fix — Parramatta's open item).
  • FGFM tunnel down: FGT unreachable from FMG — find why.

Done when: you solve a fault you didn't plant that session, using ≤3 diagnostic commands before your first hypothesis — evidence before assumption.

Blueprint coverage check: Labs 1 & 9 → system/sessions/diagnostics · Labs 2 & 8 → central management · Labs 4 & 6 → routing (OSPF + BGP) · Labs 3 & 5 → VPN incl. ADVPN · Lab 7 → security profiles. Every NSE7 EF domain gets hands-on time on this hardware.
10 · Lab Hygiene

Snapshots, Resets & Habits

  • Golden snapshot: the moment section 07 is verified and all devices are registered to FMG, snapshot all four VMs: qm snapshot 101 golden-baseline (repeat for 102, 103, 110). Every lab starts from here or from the previous lab's end-state snapshot.
  • Name snapshots by lab: lab3-complete, lab4-complete… Rolling back is one command: qm rollback 101 lab3-complete.
  • Backup configs too: execute backup config flash on each FortiGate, and use FMG revision history — restoring a config is itself exam-relevant practice.
  • One change at a time. The lab is where the habit is built. If you changed two things and it works, you learned nothing about either.
  • Keep a lab journal: for each fault fixed, write the symptom, the evidence, the root cause, and the one command that revealed it. This becomes personal exam-prep gold.
Resource note: full build ≈ 3×2 GB (FGT) + 8 GB (FMG) + ~1.5 GB (clients) ≈ 12 GB RAM. If tight: FMG at 4 GB, run only the two FortiGates a lab needs, or stop the clients you're not using.
